    CDD: EVA Domain Assistant - Sandbox Spaces¶

    Last Updated: 2025-11-19 Status: Backlog Priority: HIGH Epic: EVA Domain Assistant

    Vision¶

    Provide isolated sandbox environments where PS-A can safely test document uploads, configuration changes, and custom prompts before deploying to production—reducing risk and enabling experimentation without impacting end users.

    Business Value: Prevent production incidents, accelerate iteration cycles by 50%, enable confident experimentation, and reduce rollback frequency.

    Core Capabilities¶

    • Isolated Testing: Sandbox data never affects production
    • Clone Production: Copy production config/documents to sandbox
    • Test Queries: Preview search results and citations
    • Performance Testing: Simulate load, measure latency
    • Compare Configs: Side-by-side before/after analysis
    • Time-Limited: Auto-expire after 7 days (extend on request)
    • Publish Workflow: One-click deploy sandbox → production
    • Rollback Safety: Revert to previous production state

    User Stories¶

    US-1: As a PS-A, I want a sandbox so that I can test changes safely¶

    Acceptance Criteria:

    • Click "Create Sandbox" → isolated environment provisioned <2 minutes
    • Sandbox inherits production config and documents (clone)
    • Test queries return results from sandbox index (not production)
    • Sandbox labeled clearly (banner: "SANDBOX MODE - Not Production")
    • Auto-expire after 7 days unless extended

    US-2: As a PS-A, I want to test document uploads so that I verify quality before production¶

    Acceptance Criteria:

    • Upload documents to sandbox (drag-and-drop, bulk)
    • Documents processed (OCR, chunking, embeddings) in sandbox
    • Preview citations for test queries
    • Compare sandbox results vs. production (side-by-side)
    • Discard sandbox (delete all sandbox data) if unsatisfactory

    US-3: As a PS-A, I want to test custom prompts so that I optimize responses¶

    Acceptance Criteria:

    • Edit system prompt in sandbox
    • Test queries with new prompt
    • Compare responses (sandbox vs. production)
    • Adjust prompt iteratively (A/B test different variants)
    • Publish winning prompt to production

    US-4: As a PS-A, I want performance testing so that I validate production readiness¶

    Acceptance Criteria:

    • Simulate load (100 concurrent queries)
    • Measure P95 latency, error rate, cost
    • Compare sandbox performance vs. production baseline
    • Identify bottlenecks (slow documents, inefficient prompts)
    • Optimize before publishing

    US-5: As a PS-A, I want one-click publish so that I deploy changes confidently¶

    Acceptance Criteria:

    • Review sandbox changes summary (documents added, config changed)
    • Click "Publish to Production" → confirmation modal (risk assessment)
    • Deployment takes <5 minutes (incremental sync)
    • Rollback button (revert to previous state if issues detected)
    • Notification sent when deployment complete

    Functional Requirements¶

    FR-1: Sandbox Creation¶

    PS-A SHALL create sandbox from production (clone config and documents) provisioned in <2 minutes.

    FR-2: Isolated Storage¶

    Sandbox data SHALL be stored separately (dedicated Azure AI Search index, Cosmos DB partition) with no cross-contamination.

    FR-3: Test Queries¶

    PS-A SHALL execute test queries in sandbox with results from sandbox index only.

    FR-4: Document Upload¶

    PS-A SHALL upload documents to sandbox with full processing (OCR, chunking, embeddings, indexing).

    FR-5: Config Testing¶

    PS-A SHALL modify config in sandbox (system prompt, search params, glossary) without affecting production.

    FR-6: Side-by-Side Compare¶

    PS-A SHALL compare sandbox vs. production results (query, citations, latency) side-by-side.

    FR-7: Performance Testing¶

    PS-A SHALL simulate load (1-1000 concurrent queries) and measure latency, error rate, cost.

    FR-8: Publish to Production¶

    PS-A SHALL publish sandbox changes to production (incremental sync) in <5 minutes with confirmation.

    FR-9: Rollback¶

    PS-A SHALL rollback production to previous state (config snapshot) if issues detected post-deployment.

    FR-10: Auto-Expiry¶

    Sandbox SHALL auto-expire after 7 days with email warning at 5 days, extendable to 14 days max.

    FR-11: Discard Sandbox¶

    PS-A SHALL discard sandbox (delete all sandbox data) without affecting production.

    FR-12: Audit Trail¶

    System SHALL log all sandbox activities (create, test, publish, discard) with timestamp and user.

    FR-13: Cost Tracking¶

    System SHALL track sandbox costs separately (Azure AI Search, OpenAI embeddings, storage) for budgeting.

    FR-14: Bilingual UI¶

    Sandbox UI SHALL support bilingual (en-US/fr-FR) with clear "SANDBOX MODE" indicators.

    FR-15: Access Control¶

    Only space admins SHALL access sandbox (not viewers or learners).

    Non-Functional Requirements¶

    Performance¶

    • Sandbox creation: <2 minutes
    • Document upload: <10 seconds per MB
    • Test query: <3 seconds (P95)
    • Publish to production: <5 minutes
    • Support 10 concurrent sandboxes per tenant

    Reliability¶

    • Uptime: 99.9% SLA (sandbox service)
    • Auto-recovery from failed deployments
    • Sandbox data backup (daily snapshots)
    • Rollback success rate: ≥99%

    Security¶

    • Sandbox isolation enforced (no data leakage)
    • Access control (only space admins)
    • Audit logs for all sandbox operations
    • PII masked in sandbox logs

    Accessibility¶

    • WCAG 2.1 AA compliance
    • Clear visual indicators (SANDBOX banner)
    • Keyboard navigation for publish workflow

    i18n¶

    • Bilingual UI (en-US/fr-FR)
    • ICU formatting for dates, numbers

    Observability¶

    • Application Insights (sandbox created, published, discarded)
    • Alerts for failed publish, auto-expiry warnings
    • Dashboard for sandbox usage (active sandboxes, publish rate)

    Scalability¶

    • Support 100+ sandboxes per tenant
    • Horizontal scaling (Azure Functions for sandbox provisioning)

    Technical Architecture¶

    Frontend¶

    • Framework: React 18 + TypeScript
    • UI: Tailwind CSS (sandbox banner, compare views)
    • State: React Query (sandbox data)

    Backend¶

    • API: Azure APIM + Azure Functions
    • Sandbox Storage: Azure AI Search (dedicated index per sandbox), Cosmos DB (HPK /tenantId/spaceId/sandbox)
    • Provisioning: Terraform (create sandbox resources)
    • Publish: Incremental sync (copy sandbox → production)

    Data Models¶

    Sandbox (Cosmos DB, HPK /tenantId/spaceId/sandbox):

    {
  "sandboxId": "uuid",
  "spaceId": "hr-benefits",
  "status": "active",
  "createdAt": "2025-11-19T10:00:00Z",
  "expiresAt": "2025-11-26T10:00:00Z",
  "config": { "systemPrompt": "Test prompt..." },
  "documents": [{ "id": "doc1", "name": "test.pdf" }],
  "metrics": { "queries": 45, "latencyP95": 280, "cost": 12.5 }
}

    Implementation Phases¶

    Phase 1: Sandbox Provisioning (Sprint 4)¶

    • Create sandbox from production (clone)
    • Isolated storage (Azure AI Search index, Cosmos DB partition)
    • Auto-expiry (7 days, extendable to 14)
    • Discard sandbox (cleanup resources)

    Phase 2: Testing Features (Sprint 5)¶

    • Test queries in sandbox
    • Document upload to sandbox
    • Config editing in sandbox
    • Side-by-side compare (sandbox vs. production)

    Phase 3: Publish & Rollback (Sprint 6)¶

    • Publish to production workflow
    • Incremental sync (copy sandbox → production)
    • Rollback to previous state
    • Audit trail and cost tracking

    Phase 4: Performance Testing (Sprint 7)¶

    • Load simulation (1-1000 queries)
    • Performance metrics (latency, error rate, cost)
    • Bottleneck identification
    • UAT with PS-A

    Success Criteria¶

    • ✅ Sandbox creation <2 minutes
    • ✅ Publish to production <5 minutes
    • ✅ Zero production incidents from sandbox testing
    • ✅ PS-A satisfaction: CSAT >4.⅖.0
    • ✅ Rollback success rate ≥99%

    Open Questions¶

    Q1: Sandbox Limits (GOVERNANCE)¶

    Question: How many active sandboxes per space? Impact: Cost vs. flexibility Recommendation: 1 active sandbox per space (delete previous before creating new) Decision Owner: Marco

    Q2: Performance Test Limits (COST)¶

    Question: Max concurrent queries for load testing? Impact: Cost vs. realism Recommendation: 1000 max (warn at 100, 500, 1000) Decision Owner: ai2

    Q3: Publish Approval (GOVERNANCE)¶

    Question: Require approval before publishing to production? Impact: Safety vs. friction Recommendation: No approval for MVP, add optional approval workflow Q2 2026 Decision Owner: Marco

    Q4: Sandbox Sharing (FEATURES)¶

    Question: Allow sharing sandbox with other users (collaboration)? Impact: Collaboration vs. complexity Recommendation: Defer to Phase 2 (Q2 2026) Decision Owner: ai2
