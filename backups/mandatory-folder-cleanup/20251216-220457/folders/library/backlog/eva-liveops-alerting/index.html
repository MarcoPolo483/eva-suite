    CDD: EVA LiveOps Alerting¶

    Last Updated: 2025-11-19 Status: Backlog Priority: MEDIUM Epic: EVA LiveOps (Admin Console)

    Vision¶

    Provide real-time platform health monitoring and incident response capabilities for EVA—detecting anomalies, performance degradation, and system failures proactively, alerting on-call engineers via multiple channels (email, SMS, Teams, PagerDuty), and enabling rapid diagnosis and resolution to ensure 99.9% uptime SLA for enterprise users.

    Business Value: Reduce MTTR (Mean Time To Resolution) by 70%, prevent outages through proactive alerting (save 50+ hours/month), establish EVA as a reliable enterprise platform (SLA compliance), and enable 24/7 monitoring with minimal on-call burden through intelligent alert routing and escalation.

    Core Capabilities¶

    • Real-Time Monitoring: Continuous health checks (API endpoints, databases, AI services)
    • Anomaly Detection: ML-based detection of unusual patterns (latency spikes, error rate increases)
    • Multi-Channel Alerting: Email, SMS, Microsoft Teams, PagerDuty, Slack integrations
    • Alert Routing: Intelligent routing based on severity, component, on-call schedule
    • Escalation Policies: Auto-escalate unacknowledged alerts (15 min → L2, 30 min → L3)
    • Runbook Integration: Link alerts to runbooks (diagnosis steps, remediation actions)
    • Dashboard Integration: Embedded alerts in LiveOps Command Center dashboard
    • Historical Analysis: Alert trends, MTTR, false positive rate, root cause patterns

    User Stories¶

    US-1: As an on-call engineer, I want real-time alerts so that I respond to incidents quickly¶

    Acceptance Criteria:

    • API latency >500ms for 5 minutes → alert triggered
    • Alert sent via: Email (instant), SMS (1 min delay), Teams message (instant)
    • Alert includes: Severity (P1-P4), affected component (API Gateway), metric (P95 latency: 650ms)
    • Acknowledge alert: Click "Ack" in Teams → alert stops escalating
    • Response time: <2 min (alert fired → engineer notified)

    US-2: As a DevOps lead, I want intelligent alert routing so that the right team responds¶

    Acceptance Criteria:

    • Alert routing rules: API Gateway failures → Backend Team, RAG failures → AI Team
    • On-call schedule: Auto-detect current on-call engineer (PagerDuty integration)
    • Business hours: Page on-call engineer (SMS + phone call)
    • After hours: Page secondary on-call (if primary doesn't ack within 15 min)
    • Audit log: All alert deliveries logged (who notified, when, channel)

    US-3: As an engineer, I want runbook links so that I diagnose issues faster¶

    Acceptance Criteria:

    • Alert message includes: "Runbook: API Gateway High Latency"
    • Runbook contains: Symptoms, diagnosis steps, remediation actions, escalation contacts
    • One-click actions: "Restart service", "Scale out", "Rollback deployment" (pre-approved automations)
    • Runbook execution logged: Who ran what, when, outcome (success/failure)
    • Runbook updates: Engineers can edit runbooks (version-controlled, approval required)

    US-4: As a manager, I want alert analytics so that I optimize on-call processes¶

    Acceptance Criteria:

    • Dashboard shows: Alert count (last 7 days), MTTR (average: 12 min), false positive rate (5%)
    • Alert trends: Heatmap by hour/day (identify peak incident times)
    • Component breakdown: Which services trigger most alerts (top 5)
    • Engineer performance: MTTR per engineer (identify training needs)
    • SLA compliance: % of incidents resolved within SLA (target: >95%)

    US-5: As a platform owner, I want proactive alerting so that I prevent outages¶

    Acceptance Criteria:

    • Anomaly detection: ML model detects unusual patterns (latency gradually increasing)
    • Proactive alert: "API latency trending up, projected to exceed SLA in 2 hours"
    • Recommended actions: "Scale out by 2 instances", "Review slow queries"
    • Prevention metrics: Outages avoided (proactive alerts), cost savings (downtime prevented)
    • Feedback loop: Engineers mark alerts as "Helpful" or "False positive" (improve ML model)

    Functional Requirements¶

    1. FR-1: SHALL monitor platform health in real-time (API, databases, AI services, 1-minute intervals)
    2. FR-2: SHALL detect anomalies using ML (latency spikes, error rate increases, resource exhaustion)
    3. FR-3: SHALL trigger alerts based on thresholds (P95 latency >500ms, error rate >2%)
    4. FR-4: SHALL send alerts via multiple channels (email, SMS, Teams, PagerDuty, Slack)
    5. FR-5: SHALL route alerts based on severity, component, on-call schedule
    6. FR-6: SHALL escalate unacknowledged alerts (15 min → L2, 30 min → L3)
    7. FR-7: SHALL link alerts to runbooks (diagnosis steps, remediation actions)
    8. FR-8: SHALL allow one-click runbook actions (pre-approved automations)
    9. FR-9: SHALL log all alert events (triggered, acknowledged, resolved, escalated)
    10. FR-10: SHALL provide alert analytics dashboard (count, MTTR, trends, SLA compliance)
    11. FR-11: SHALL integrate with on-call systems (PagerDuty, VictorOps, Opsgenie)
    12. FR-12: SHALL support alert suppression (maintenance windows, known issues)
    13. FR-13: SHALL allow engineers to snooze alerts (1 hour, 4 hours, 1 day)
    14. FR-14: SHALL provide mobile app integration (iOS, Android push notifications)
    15. FR-15: SHALL generate incident reports (post-mortem templates, timeline reconstruction)

    Non-Functional Requirements¶

    Performance¶

    • Alert Latency: <30s (metric threshold exceeded → alert triggered)
    • Notification Delivery: <2 min (alert triggered → engineer notified)
    • Dashboard Refresh: <5s (real-time alert updates)
    • Anomaly Detection: <1 min (data ingested → anomaly detected)

    Reliability¶

    • Alert Delivery: >99.9% (alerts always delivered, multi-channel redundancy)
    • False Positive Rate: <5% (avoid alert fatigue)
    • Escalation Accuracy: 100% (alerts always escalate if unacknowledged)
    • Monitoring Uptime: 99.99% (monitoring system always available)

    Security¶

    • Alert Privacy: PII redacted from alert messages (no sensitive data in emails/SMS)
    • Audit Logging: All alert events logged to Cosmos DB (1-year retention)
    • Authorization: Only on-call engineers see alerts (RBAC enforcement)
    • Secure Channels: SMS/email encrypted (TLS 1.3, Azure Communication Services)

    Accessibility¶

    • WCAG 2.1 AA: Alert dashboard keyboard-accessible
    • Screen Reader Support: ARIA labels for alert severity, status
    • High Contrast: Alert colors (red: critical, yellow: warning) meet contrast ratios

    Internationalization¶

    • Locales: en-US (default), fr-FR
    • Alert Messages: Translated (e.g., "API latency high" → "Latence API élevée")
    • Runbooks: Bilingual (English and French versions)

    Observability¶

    • Metrics: Application Insights tracks alert count, MTTR, false positives, escalation rate
    • Alerts on Alerts: Meta-alerts (e.g., "Alert delivery failed", "Monitoring system down")
    • Dashboards: Power BI dashboard for alert analytics, MTTR trends, SLA compliance

    Scalability¶

    • Alert Volume: Handle 10K+ alerts/day (peak: 100 alerts/min)
    • Concurrent Incidents: Support 50+ simultaneous incidents (parallel alert processing)
    • Historical Data: Store 1 year of alert history (query performance <3s)

    Technical Architecture¶

    Monitoring Stack¶

    • Metrics Collection: Azure Monitor (Application Insights, Log Analytics)
    • Anomaly Detection: Azure Monitor Anomaly Detection (ML-based)
    • Alerting Engine: Azure Monitor Alerts (metric alerts, log alerts)
    • Notification Routing: Azure Logic Apps (multi-channel delivery, escalation)
    • On-Call Integration: PagerDuty REST API (schedule lookup, incident creation)

    Alert Pipeline¶

    Real-Time Monitoring → Anomaly Detection → Alert Routing → Notification Delivery:

    1. Metrics Ingestion: Application Insights collects telemetry (1-minute intervals)
    2. Threshold Evaluation: Azure Monitor checks thresholds (P95 latency >500ms)
    3. Anomaly Detection: ML model detects unusual patterns (baseline deviation >2σ)
    4. Alert Creation: Alert record created (severity, component, metric, timestamp)
    5. Routing Logic: Logic App routes alert (based on severity, component, on-call)
    6. Notification: Send to multiple channels (email, SMS, Teams, PagerDuty)
    7. Escalation: Auto-escalate if unacknowledged (15 min → L2, 30 min → L3)
    8. Resolution: Engineer acknowledges, resolves, or snoozes alert

    Implementation Phases¶

    Phase 1: Core Monitoring & Alerting (Sprint 8, Week 1-3)¶

    • Azure Monitor setup (Application Insights, Log Analytics)
    • Metric alerts (P95 latency, error rate, CPU, memory)
    • Log alerts (error patterns, exception spikes)
    • Email notifications (SMTP, Azure Communication Services)
    • Alert dashboard (LiveOps Command Center integration)

    Phase 2: Multi-Channel & Routing (Sprint 9, Week 4-6)¶

    • SMS notifications (Azure Communication Services, Twilio)
    • Microsoft Teams integration (webhook, adaptive cards)
    • PagerDuty integration (API, incident creation, schedule lookup)
    • Alert routing logic (severity, component, on-call schedule)
    • Escalation policies (auto-escalate unacknowledged alerts)

    Phase 3: Runbooks & Automation (Sprint 10, Week 7-9)¶

    • Runbook repository (Markdown, version-controlled, GitHub)
    • Runbook linking (alerts include runbook URLs)
    • One-click actions (Azure Automation, pre-approved scripts)
    • Runbook execution logs (who ran what, when, outcome)
    • Runbook feedback (engineers rate helpfulness)

    Phase 4: Analytics & Optimization (Sprint 11, Week 10-12)¶

    • Alert analytics dashboard (count, MTTR, false positive rate)
    • Historical analysis (trends, patterns, root causes)
    • Anomaly detection (ML-based, proactive alerting)
    • SLA compliance tracking (% incidents resolved within SLA)
    • Power BI dashboard for alert analytics, MTTR, SLA trends

    Success Criteria¶

    • ✅ Alert latency <30s (threshold exceeded → alert triggered)
    • ✅ Notification delivery <2 min (alert → engineer notified)
    • ✅ MTTR <15 min (average resolution time)
    • ✅ False positive rate <5% (avoid alert fatigue)
    • ✅ SLA compliance >95% (incidents resolved within SLA)
    • ✅ Escalation accuracy 100% (alerts always escalate if unacked)
    • ✅ On-call satisfaction >80% (engineer survey)
