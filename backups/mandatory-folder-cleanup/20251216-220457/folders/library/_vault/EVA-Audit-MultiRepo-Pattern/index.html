    EVA Audit Trail - Multi-Repo Bootstrap Pattern¶

    Document ID: EVA-AUDIT-MULTI-001
    Created: 2025-11-25
    Status: âœ… Production-ready

    For IT shops & research labs: This pattern enables audit trail deployment across entire development platforms with 23+ repositories. Demonstrates EVA Suite's commitment to operational transparency at scale.

    Executive Summary¶

    The EVA Audit Trail Multi-Repo Pattern is a deployment blueprint for extending automated file change tracking from a single repository (eva-orchestrator) to an entire suite of repositories (23+ EVA repos). This pattern:

    • Standardizes audit infrastructure across repositories
    • Automates nightly change detection via GitHub Actions
    • Centralizes aggregation for cross-repo analytics
    • Adapts to IT shops, research labs, and enterprise development platforms

    Key deliverables: 1. Bootstrap script (eva-audit-bootstrap.ps1) to configure any repo 2. Standard directory structure for audit artifacts 3. GitHub Actions workflow template (nightly + push triggers) 4. Cross-repo aggregation script (future: Phase 3)

    GitHub Actions Workflow Template¶

    File: .github/workflows/eva-audit-update.yml

    name: EVA Audit Update

on:
  schedule:
    - cron: '0 2 * * *'  # Nightly at 2 AM UTC
  push:
    paths:
      - '{repo-name}/**'
      - '.github/workflows/eva-audit-update.yml'
  workflow_dispatch:

jobs:
  audit-update:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v3
        with:
          fetch-depth: 0  # Full history for accurate file tracking

      - name: Run audit update script
        shell: pwsh
        run: |
          .\scripts\eva-audit-update.ps1

      - name: Check for changes
        id: git-check
        run: |
          git diff --exit-code docs/_audit/ || echo "changed=true" >> $GITHUB_OUTPUT

      - name: Commit audit artifacts
        if: steps.git-check.outputs.changed == 'true'
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add docs/_audit/
          git commit -m "chore: update EVA audit trail [skip ci]"

      - name: Push changes with retry
        if: steps.git-check.outputs.changed == 'true'
        run: |
          for i in {1..3}; do
            git pull --rebase && git push && break || sleep 10
          done

      - name: Upload error logs
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: audit-error-logs
          path: docs/_audit/*.log
          retention-days: 30

      - name: Summary
        if: always()
        run: |
          echo "### EVA Audit Update Summary" >> $GITHUB_STEP_SUMMARY
          echo "Repository: {repo-name}" >> $GITHUB_STEP_SUMMARY
          echo "Timestamp: $(date -u)" >> $GITHUB_STEP_SUMMARY

    Customization notes: - Replace {repo-name} with actual repository name (e.g., eva-core) - Adjust cron schedule if needed (default: 2 AM UTC) - Add repo-specific paths to on.push.paths (e.g., exclude docs/archive/**)

    Phase 3: Cross-Repo Aggregation¶

    Security Considerations¶

    1. GitHub Actions Secrets¶

    Store sensitive data in GitHub Secrets (not in workflow YAML):

    - name: Run audit update with secrets
  env:
    AUDIT_API_KEY: ${{ secrets.AUDIT_API_KEY }}
  shell: pwsh
  run: |
    .\scripts\eva-audit-update.ps1

    Use cases: - API keys for cloud cost APIs (Azure, AWS, GCP) - Personal access tokens for cross-repo aggregation - Encryption keys for audit log signing

    2. Audit Log Integrity¶

    Threat: Attacker modifies audit log to hide malicious changes

    Mitigation: 1. Append-only enforcement: File permissions (read-only for most users) 2. SHA-256 checksums: Compute hash of log file, store in separate file 3. Immutable storage: Copy logs to Azure Blob with legal hold (Phase 3)

    Example checksum validation: 

    # Compute checksum
$hash = Get-FileHash docs\_audit\EVA-Audit-Log.jsonl -Algorithm SHA256
$hash.Hash > docs\_audit\EVA-Audit-Log.checksum

# Validate later
$currentHash = Get-FileHash docs\_audit\EVA-Audit-Log.jsonl -Algorithm SHA256
$storedHash = Get-Content docs\_audit\EVA-Audit-Log.checksum
if ($currentHash.Hash -ne $storedHash) {
    throw "Audit log tampered!"
}

    3. Sensitive Path Redaction¶

    Threat: Audit log exposes sensitive file paths (e.g., secrets/api-key.txt)

    Mitigation: 1. Pattern-based redaction: Replace sensitive patterns with [REDACTED] 2. Exclusion filters: Never audit files matching *secret*, *key*, *.pem

    Example redaction: 

    # In eva-audit-update.ps1, before appending to log:
if ($file.Name -match "secret|key|password|token") {
    $filePath = "[REDACTED]"
}

$changedFiles = git diff --name-only HEAD~1

# Scan only changed files
foreach ($file in $changedFiles) {
    # Process file...

    3. EVA Ledger Pattern (Blockchain-Inspired, Not Blockchain)¶

    Decision (Dec 4, 2025): DECLINED full blockchain - using hash-chain audit pattern instead

    Goal: Tamper-evident audit logs with cryptographic integrity (blockchain properties WITHOUT blockchain complexity)

    Architecture - "EVA Ledger" Pattern: - Hash-linked audit tables: previous_hash + current_hash = hash(previous_hash + payload_hash + timestamp + ...) - Append-only: SQL/Cosmos DB with insert-only permissions - Merkle snapshots: Periodic (hourly/daily) Merkle tree root stored in Key Vault - Immutable storage: Azure Blob legal hold for sealed logs - Verification: Chain integrity checks on read, detect tampering if hash chain breaks

    Why NOT blockchain: - EVA has trusted operator (AICOE) - no multi-party trust needed - Hash chains already provide tamper-evidence auditors require - Blockchain adds cost (gas fees), latency (seconds), complexity (nodes/consensus) - Protected B + privacy = awkward with blockchain (only hashes on-chain?)

    Use cases (same as blockchain, simpler implementation): - PIPEDA compliance (tamper-proof audit trails) - Incident investigation (reconstruct events, prove integrity) - RA/IDE evidence ("This action occurred at this time, log not altered") - Cross-department provenance (IF EVA expands to 5+ departments)

    Implementation: Pattern 1 (hash-linked tables) + Pattern 2 (Merkle snapshots) from audit compliance CDD
