    Setting Up EVA Agile Command Team Beta (Second Parallel Team)¶

    Date: November 16, 2025
    Purpose: Scale EVA development by launching a second parallel Agile team
    Owner: Marco Presta (Product Owner)

    üéØ Overview¶

    You now have EVA Agile Command Alpha (your current team) running Sprint 01 on Power BI dashboards. To scale development and tackle multiple moonshots simultaneously, you need EVA Agile Command Beta working in parallel.

    üìê Team Structure (Copy Alpha Model)¶

    EVA Agile Command Beta¶

    Product Owner: Marco Presta (shared across both teams)

    Scrum Master & QA Master: New AI assistant instance (recommend: Claude or ChatGPT) - Facilitates sprint ceremonies - Enforces quality gates - Coordinates dev agents - Reports to Marco

    Dev Master Agents (2 specialists, max 4 issues each): - Agent 3: Focus area TBD (options: Frontend, Backend, Infra, Mobile) - Agent 4: Focus area TBD (options: Security, Testing, Integration, DevOps)

    üöÄ Setup Steps¶

    Step 1: Choose Your Scrum Master Platform¶

    Option A: Claude Projects (Recommended for Beta) - Go to https://claude.ai/ - Create new project: "EVA Agile Command Beta" - Upload context files: - docs/EVA-2.0-MASTER-CONTEXT.md - docs/AGILE-OPERATING-MODE.md - docs/DEFINITION-OF-DONE.md - docs/client-preferences.md - .github/copilot-instructions.md - orchestrator.yml (update with Beta agents) - agents/registry.yaml - Set custom instructions: 

    You are the Scrum Master & QA Master for EVA Agile Command Beta.
You work in parallel with Alpha team (GitHub Copilot).
Your role: Run sprints, enforce quality, coordinate agents, report to Marco.

    Option B: ChatGPT Projects - Go to https://chat.openai.com/ - Create new project: "EVA Agile Command Beta" - Upload same context files as Claude - Set custom instructions (same as above)

    Option C: GitHub Copilot Workspace #2 - Open second VS Code window - Use different workspace folder (e.g., eva-beta-workspace) - Configure separate Copilot context - Coordinate with Alpha via shared docs

    Step 2: Define Beta Team Focus Areas¶

    Recommendation: Let Alpha handle Analytics/Dashboards, Beta handles Platform/Infra

    Beta Team Sprint 01 Focus Options:

    1. Option A: EVA Core Platform (High Priority)
    2. Build eva-core, eva-auth, eva-agent foundational repos
    3. Target: Working authentication + agent orchestration
    4. Duration: 2 weeks

    5. Complexity: High

    6. Option B: EVA Infrastructure (High Priority)

    7. Build eva-infra Terraform landing zone
    8. Target: Networking, Key Vault, APIM, monitoring baseline
    9. Duration: 2 weeks

    10. Complexity: Medium

    11. Option C: EVA Vision Technical Spec (Moonshot)

    12. Design EVA Vision (Power BI Plugin) architecture
    13. Target: Complete technical spec + POC wireframes
    14. Duration: 2 weeks

    15. Complexity: Medium

    16. Option D: EVA OS Integration POC (Moonshot)

    17. Build Windows service prototype for context capture
    18. Target: Win + C hotkey panel, context to EVA API
    19. Duration: 2 weeks
    20. Complexity: High

    Marco's Decision: Choose 1 focus area for Beta Sprint 01

    Step 3: Create Beta Agent Profiles¶

    Update orchestrator.yml to add Beta agents:

    # EVA Agile Command Beta Team
agents:
  - id: agent3
    name: "Agent 3 (Beta Dev Master 1)"
    role: dev-master
    specialization: [frontend, ui-ux]  # or backend, infra, mobile
    max_active_issues: 4

  - id: agent4
    name: "Agent 4 (Beta Dev Master 2)"
    role: dev-master
    specialization: [security, testing]  # or devops, integration
    max_active_issues: 4

# Work Distribution (Beta Team)
work_streams:
  eva-vision-spec: [agent3, agent4]
  eva-os-integration: [agent3, agent4]
  eva-core-platform: [agent3, agent4]
  eva-infra-baseline: [agent3, agent4]

    Step 4: Synchronize Both Teams¶

    Shared Resources (Both teams use): - docs/DEFINITION-OF-DONE.md (single source of truth) - agents/registry.yaml (17 repos, ownership tracked) - docs/master-plan.md (strategic roadmap) - Git repository (coordinate PRs, avoid conflicts)

    Team Coordination: - Daily Sync (5 min): Both SMs report blockers to Marco - Weekly Integration: Teams merge work, resolve conflicts - Sprint Reviews: Joint demo (Alpha + Beta show progress) - Retrospectives: Separate (team-specific improvements)

    Avoid Conflicts: - Alpha owns: dashboards/, analytics work - Beta owns: TBD (based on Sprint 01 focus) - Shared: docs/, agents/, orchestrator config - Rule: Always pull latest before pushing

    Step 5: Launch Beta Sprint 01¶

    Sprint Planning Ceremony (with Beta SM): 1. Review backlog (master-plan.md) 2. Choose Beta focus area (from Step 2 options) 3. Break down into 8-12 issues (~30 story points) 4. Assign issues to agent3 and agent4 (max 4 each) 5. Create docs/sprint-history/2025-11-sprint-01-beta/sprint-plan.md 6. Commit and begin work

    Daily Stand-ups (Both teams): - Alpha SM reports to Marco (morning) - Beta SM reports to Marco (afternoon) - Format: What done? What next? Blockers?

    Sprint Cadence: - Both teams: 2-week sprints - Synchronized end dates (both finish Nov 29) - Joint sprint review (Nov 28-29) - Separate retrospectives (Nov 29)

    üéØ Success Criteria for Beta Team¶

    Sprint 01 (First 2 weeks)¶

    ‚úÖ Team operational (SM + 2 agents working) ‚úÖ 8-12 issues completed (~30 story points) ‚úÖ Quality gates passing (DoD enforced) ‚úÖ No critical blockers escalated ‚úÖ Integration with Alpha smooth (no merge conflicts)

    Sprint 02+ (Ongoing)¶

    ‚úÖ Velocity stable (30-40 story points per sprint) ‚úÖ Both teams delivering in parallel ‚úÖ Marco gets 2x productivity (Alpha + Beta) ‚úÖ Joint demos show integrated progress ‚úÖ Codebase quality maintained (80% coverage, <300ms P95)

    üìä Monitoring Both Teams¶

    Marco's Dashboard (Weekly): - Alpha velocity: X story points - Beta velocity: Y story points - Combined velocity: X + Y (target: 60-80) - Blockers count: Alpha + Beta - Quality gate pass rate: % - Integration issues: count

    Red Flags (Escalate immediately): - Either team velocity drops >30% - Merge conflicts increase >5 per week - Critical bugs in production - Quality gates failing >20% - Agent capacity exceeded (>4 issues)

    üîÑ Scaling to Team Gamma (Third Team)¶

    Once Alpha + Beta are stable (4-6 sprints), consider Team Gamma: - Focus: Mobile, external integrations, or marketplace - Same structure: 1 SM + 2 agents - Same coordination: Daily sync, joint reviews - Target: 3 teams √ó 30 SP = 90 story points per sprint

    Scaling Limits: - Max 3-4 teams before coordination overhead exceeds value - Marco capacity: 3 teams = ~30 min daily sync - Git conflicts: Mitigate with clear ownership boundaries

    üí° Pro Tips for Multi-Team Success¶

    1. Clear Boundaries: Each team owns specific repos/features
    2. Shared DoD: Never compromise quality for velocity
    3. Daily Sync: 5 min with Marco keeps teams aligned
    4. Joint Demos: Show integrated progress, not siloed work
    5. Celebrate Wins: Recognize both teams equally
    6. Learn Together: Share retro insights across teams
    7. Marco as PO: Final arbiter on priorities and conflicts

    üìã Beta Team Launch Checklist¶

    Before starting Beta Sprint 01:

    • Choose SM platform (Claude, ChatGPT, or Copilot #2)
    • Upload context files to SM workspace
    • Set custom instructions for Beta SM
    • Decide Beta focus area (Core, Infra, Vision, or OS)
    • Update orchestrator.yml with agent3 and agent4
    • Create Beta sprint plan (docs/sprint-history/2025-11-sprint-01-beta/)
    • Brief Beta SM on EVA vision and Q1 2026 demo goal
    • Coordinate with Alpha SM on shared resource usage
    • Schedule daily sync times (Alpha AM, Beta PM)
    • Commit all setup docs to git
    • Marco approves Beta Sprint 01 plan
    • Launch! üöÄ
