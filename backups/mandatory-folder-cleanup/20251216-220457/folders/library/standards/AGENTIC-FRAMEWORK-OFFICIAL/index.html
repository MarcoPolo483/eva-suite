    EVA Agentic Framework - Official Standard¶

    Document Type: Official Standard Version: 1.0.0 Status: ACTIVE Effective Date: 2025-12-08 Authority: EVA Suite Core Standards Scope: All EVA repositories and agentic implementations

    üéØ Purpose¶

    This document defines the official agentic framework for EVA Suite, including: - Agentic Maturity Model (L0-L3) - DevTools Agents (P01-P15) - GC Agentic AI Template alignment - Production-ready patterns and personas

    This is THE reference for all agentic capabilities in EVA.

    üìà AGENTIC MATURITY MODEL (Core Framework)¶

    Level Definitions¶

    Level Name Capabilities Tools Planning Production
    L0 Non-Agentic RAG Q&A only None None ‚úÖ Allowed
    L1 Bounded Tool Helpers Run approved tools, short sequences (1-3 steps) Predefined, whitelisted Sequential, no loops ‚úÖ Allowed
    L2 Workflow Agents Plan 3-10 steps, chain tools, human confirmation Dynamic selection from approved set Multi-step with branching ‚úÖ Allowed
    L3 Open Multi-Agent High autonomy, experimentation, agent-to-agent Unrestricted Complex, emergent ‚õî Sandbox/Lab ONLY

    Production Rules¶

    MANDATORY CONSTRAINTS: - ‚úÖ Production: Only L0, L1, L2 allowed - ‚õî Production: L3 is NEVER allowed - üß™ Sandbox/Lab: L3 permitted with explicit approval - üìã All Levels: Full audit trail required - üîí All Levels: Kill switch must exist

    Examples by Level¶

    L0 (Non-Agentic): - EVA DA (Domain Assistant) - RAG Q&A - EVA Chat - Conversational interface - Simple document retrieval

    L1 (Bounded Tools): - "Search jurisprudence and summarize results" ‚Üí Single tool call - "Calculate benefits eligibility" ‚Üí Run approved calculator - "Translate this document" ‚Üí Single translation API call

    L2 (Workflow Agents): - P02 (REQ): Refine idea ‚Üí canvas ‚Üí epic ‚Üí stories ‚Üí tasks (multi-step planning) - P05 (SCA): Analyze requirements ‚Üí scaffold repo ‚Üí generate docs ‚Üí validate structure - P15 (DVM): Route request ‚Üí orchestrate P02 + P05 + P06 ‚Üí synthesize results

    L3 (Open Multi-Agent) - SANDBOX ONLY: - Multiple agents negotiating solutions - Emergent behavior from agent interactions - Research and experimentation scenarios

    üé≠ AGENTIC PATTERNS (Official Implementations)¶

    Pattern 1: Recursive Refinement (P02-REQ)¶

    User Natural Language Request
    ‚Üì
P02: Parse intent + extract entities
    ‚Üì
P02: Generate structured canvas
    ‚Üì
P02: Decompose canvas ‚Üí epics
    ‚Üì
P02: Refine epics ‚Üí stories
    ‚Üì
P02: Break stories ‚Üí atomic tasks
    ‚Üì
Output: requirements.json (machine-readable)

    Characteristics: - Multi-step planning (3-5 iterations) - Schema-driven output - Full traceability - Human review at epic level

    Pattern 2: Orchestration (P15-DVM)¶

    User Goal Input
    ‚Üì
P15: Analyze goal + determine agent chain
    ‚Üì
P15: Route to specialized agents:
     P02 (requirements) ‚Üí
     P05 (scaffold) ‚Üí
     P06 (review) ‚Üí
     P07 (test) ‚Üí
     P08 (CICD)
    ‚Üì
Each agent logs to central DevTools logs
    ‚Üì
P15: Synthesize results ‚Üí return to user

    Characteristics: - Dynamic agent selection - Parallel execution where possible - Centralized logging - Error handling and retry logic

    Pattern 3: Shared Runtime (All Agents)¶

    Standard Agent Execution Pipeline:

    1. Trigger ‚Üí Intent Resolution
    2. GitHub webhook, PR comment, scheduled job, or manual request

    3. Map to agent_id + task_type

    4. Context Gathering

    5. Source control (PR diffs, repo metadata)
    6. Issue tracker (linked issues, epic descriptions)
    7. Docs (README, ADRs, CONTRIBUTING, runbooks)

    8. Policies (DoD, security, a11y, coding standards)

    9. Policy & Guardrail Check

    10. Is agent enabled? (kill switch check)
    11. Is autonomy level appropriate?

    12. Are required approvals in place?

    13. LLM Reasoning

    14. System prompt: Global DevTools rules
    15. Role prompt: Agent-specific behavior
    16. Context: Structured data object

    17. Output schema: Required format

    18. Result Validation

    19. Parse and validate against schema
    20. Enforce limits (max comments, severity levels)

    21. Ensure advisory/assistive scope

    22. Effectors

    23. Post PR comments
    24. Create/update issues
    25. Write draft docs

    26. Produce reports

    27. Logging & Metrics

    28. Log entry per action
    29. Include: timestamp, agent, user, context, result
    30. Store in DevTools index (separate from end-user logs)

    üîë AGENTIC PRINCIPLES (Mandatory)¶

    1. Human Accountability¶

    • Agents assist, not replace accountable decision-makers
    • Critical decisions require human confirmation
    • Humans remain responsible for outcomes

    2. Bounded Autonomy¶

    • Agents operate within project spaces
    • Cannot access data/tools outside their scope
    • Explicit tool whitelists per agent

    3. Observability & Traceability¶

    • Every interaction logged
    • Configuration changes tracked
    • Full audit trail for compliance

    4. Kill Switch & Control¶

    • Each agent can be disabled (per repo, pod, globally)
    • Emergency stop capability
    • Tunable behavior (verbosity, focus areas)

    5. Federal-Grade Security¶

    • Secure, reliable, auditable
    • Accessible (WCAG 2.1 AA)
    • Bilingual (EN/FR)
    • Protected B capable

    6. Explainability¶

    • Agents explain reasoning
    • Users understand why actions were taken
    • Decisions are traceable

    EVA implements the official GC Agentic AI Template (17 documents).

    Template Volumes¶

    Volume Title EVA Mapping
    Vol 0 How-To Guide This document + Agent HELP docs
    Vol 1 Platform & Agentic Requirements EVA architecture + P01-P15 agents
    Vol 2 Engineering & Delivery Guide DevTools runtime + Sprint process
    Vol 3 LiveOps & Governance Model P14 (LVO) + monitoring infrastructure
    Vol 4 Compliance & Security P11 (SEC) + ATO process

    Key Template Concepts¶

    Project Space: - Logical container for agent operations - Defines: business purpose, data sources, tools, RBAC - Maps to: EVA PODs (POD-F, POD-X, POD-O, POD-S)

    Agent Definition: - Configured, semi-autonomous digital assistant - Interprets goals, plans actions, calls tools - Bounded by policies and data scope

    Autonomy Levels (Template C0-C3 maps to L0-L3): - C0/L0: Non-agentic (RAG only) - C1/L1: Bounded tools (single-step) - C2/L2: Workflow agents (multi-step with HITL) - C3/L3: Open multi-agent (sandbox only)

    üö® ESDC STRATEGIC CONTEXT¶

    The Fear¶

    • Leadership perceives "agentic" = uncontrolled, autonomous, unpredictable
    • No clear GoC "AI Safety Playbook" exists yet
    • ITSG-33 doesn't have "agentic annex"

    The Reality¶

    EVA is the safest place for agentic capabilities vs ad-hoc vendor tools because: - ‚úÖ Controlled ATO process - ‚úÖ Full audit trail - ‚úÖ RBAC and access controls - ‚úÖ Kill switches and governance - ‚úÖ Bounded multi-tenant design - ‚úÖ Clear accountability lines - ‚úÖ TBS Guide on Generative AI alignment - ‚úÖ WCAG 2.1 AA, i18n, accessibility

    The Real Choice¶

    Do we want agentic capabilities inside EVA (controlled, auditable) ‚Äî or scattered across vendor black boxes (ABRG, MS Copilot, ServiceNow) outside our governance?

    If ESDC blocks EVA agentic: Risk moves to external stacks where privacy, auditability, and governance are weaker.

    Strategic Positioning¶

    • EVA becomes "ESDC's ITSG-333 model" for GenAI/Agentic Security
    • Agentic Maturity Model resolves ambiguity around "agentic"
    • GC Template provides governance framework for all departments
    • Business Case: $438K savings Year 1, 548% ROI (from template)

    üìã USAGE GUIDELINES¶

    When to Use Agentic Capabilities¶

    ‚úÖ Appropriate Use Cases: - Repetitive multi-step workflows (L2) - Code generation with human review (L2) - Requirements refinement (L2) - Automated testing and analysis (L1-L2) - Documentation synchronization (L1) - Metrics aggregation and insights (L1)

    ‚õî Inappropriate Use Cases: - Autonomous production decisions (L3) - Unrestricted tool access (L3) - Actions without audit trail (any level) - Critical decisions without human confirmation (L3)

    Requesting Agent Assistance¶

    Simple Format: 

    [Agent ID]: [Task description]

Example: P02: Refine requirements for jurisprudence search feature

    Structured Format: 

    [Agent ID] REQUEST:
  TASK: [What needs to be done]
  CONTEXT: [Relevant background]
  CONSTRAINTS: [Limitations]
  DELIVERABLES: [Expected outputs]

Example:
P05 REQUEST:
  TASK: Scaffold new microservice for document processing
  CONTEXT: Part of POD-F, Python FastAPI, Azure Cosmos DB
  CONSTRAINTS: Must follow EVA naming conventions, HPK pattern
  DELIVERABLES: Repo structure, README, basic API skeleton

    Agent Response Format¶

    Agents MUST include: - ‚úÖ Clear explanation of actions taken - ‚úÖ Rationale for decisions - ‚úÖ Any assumptions made - ‚úÖ Required human approvals - ‚úÖ Next steps or follow-up needed

    üîß IMPLEMENTATION CHECKLIST¶

    Before deploying any agentic capability:

    • Maturity level identified (L0/L1/L2/L3)
    • Production approval obtained (if L2 or higher)
    • Project space defined
    • Tool whitelist configured
    • RBAC policies set
    • Audit logging enabled
    • Kill switch tested
    • Human confirmation points identified
    • Privacy assessment completed
    • Security review passed
    • Documentation updated
    • Training provided to users
