    Key Design Decisions¶

    1. Two Event Types (Not One)¶

    Decision: Use separate p02-execution and p02-openai-usage events.

    Rationale: - Separates execution metadata from API usage details - Enables 1:N relationship (one execution, multiple OpenAI calls) - Allows P02 executions without OpenAI (e.g., manual documentation) - Simplifies cost aggregation queries

    2. Mandatory Audit Logging (No Exceptions)¶

    Decision: All P02 implementations MUST log to EVA Audit Trail.

    Rationale: - Consistent observability across all EVA DevTools patterns - Cost transparency as P02 scales across 17+ repos - Compliance requirement for enterprise AI usage - Future-proofs for FinOps integration

    3. GitHub Copilot Exemption (Practical)¶

    Decision: GitHub Copilot usage cannot and does not need to be logged.

    Rationale: - No API access to measure Copilot token usage - Billed separately (GitHub account, not OpenAI API keys) - Focus audit on direct OpenAI API calls we control - Documentation clearly states when logging applies

    4. Run ID Correlation (Queryability)¶

    Decision: Use shared run_id (UUID) to link all events for a single P02 execution.

    Rationale: - Enables cost rollup per P02 run - Supports debugging (trace all OpenAI calls for failed execution) - Aligns with distributed tracing best practices - Simple to implement (generate UUID at start)

    5. Cost Estimate (Optional but Recommended)¶

    Decision: cost_estimate field is required but can be null.

    Rationale: - OpenAI pricing changes over time - Some implementations may not have pricing table - Better to have null than block implementation - Allows retroactive cost calculation if needed

    Questions & Considerations¶

    Open Questions¶

    1. Should all EVA patterns (P01-P14+) adopt similar audit requirements?
    2. Likely yes, but each pattern may have different telemetry needs

    3. Recommend: Create generic "pattern execution" schema, specialized per pattern

    4. Should audit log be per-repo or suite-wide?

    5. Current: per-repo (each repo has its own docs/_audit/EVA-Audit-Log.jsonl)

    6. Phase 3: Consider suite-wide aggregation (Azure Data Lake, Power BI)

    7. Who owns OpenAI pricing updates?

    8. Implementation should fetch pricing from config or API

    9. If hardcoded, document update cadence (monthly review?)

    10. Should P02 audit events be real-time or batch?

    11. Real-time: Emit immediately after OpenAI call (preferred for accuracy)
    12. Batch: Collect in memory, flush at end (risk of loss on crash)

## P02 Requirements Engine Integration

The EVA Audit Trail now tracks P02 Requirements Engine executions and OpenAI usage. This enables cost tracking and observability for AI-powered requirements gathering.

### Query P02 OpenAI costs

```powershell
# Total cost per feature
Get-Content docs/_audit/EVA-Audit-Log.jsonl | 
  ForEach-Object { $_ | ConvertFrom-Json } | 
  Where-Object { $_.type -eq "p02-openai-usage" } | 
  Group-Object feature_key | 
  ForEach-Object {
    [PSCustomObject]@{
      Feature = $_.Name
      TotalCost = ($_.Group | Measure-Object -Property cost_estimate -Sum).Sum
      Calls = $_.Count
    }
  } | 
  Format-Table -AutoSize

    Full documentation: See P02-OpenAI-Usage-Schema.md ```
