    GitHub Spark + EVA-Sovereign-UI Integration Strategy¶

    Date: November 28, 2025
    Context: Autonomous crew expert spawning via Spark
    Source: Marco's GPT conversation + user stories

    üéØ Strategic Positioning¶

    Spark = Accelerator (Upstream)¶

    Role: Fast AI workshop for rapid prototyping - Natural language ‚Üí working Svelte screens - Live preview + instant iteration - 1-click repo creation + deployment - Refactoring assistance for legacy code

    EVA-Sovereign-UI Tool = Governance (Canonical)¶

    Role: Disciplined GC/Five-Eyes maintainer - Layout metadata as source of truth - GC Design System compliance - a11y + i18n enforcement - Sovereign profile management (Canada, Australia, NZ, UK, US)

    Flow: Spark generates fast ‚Üí Tool maintains compliant

    üìã User Stories (Backlog Ready)¶

    Epic 1: Spark-First Development (New Screens)¶

    US-01: Bootstrap new EVA screen using Spark¶

    As a front-end developer
    I want to use GitHub Spark to rapidly generate an initial Svelte screen
    So that I can go from idea to working prototype in minutes

    Acceptance Criteria: - ‚úÖ Spark-generated code compiles in EVA Svelte/Turbo/PNPM workspace - ‚úÖ Layout mappable to Tool's layout schema (sections, components, props) - ‚úÖ No sensitive/prod data in Spark prompts or code

    Estimate: 3 story points
    Labels: spark, eva-ui, greenfield, ready

    US-02: Import Spark prototype into EVA-Sovereign-UI Tool¶

    As a front-end developer
    I want to import Spark-generated screen into the Tool
    So that future changes are driven by metadata and governed layouts

    Acceptance Criteria: - ‚úÖ pageId and layout JSON created and validated - ‚úÖ All Spark visual elements represented as Tool sections/components - ‚úÖ Generated layout passes a11y/i18n checks (or shows warnings)

    Estimate: 5 story points
    Labels: spark, eva-ui, tool-integration, ready

    US-03: Maintain screen only via Tool (post-import)¶

    As a front-end developer
    I want to evolve screen by editing layout/props in Tool
    So that GC/Five Eyes, a11y, i18n rules are always enforced

    Acceptance Criteria: - ‚úÖ All layout changes as metadata diff (JSON/YAML), not hard-coded Svelte - ‚úÖ a11y panel reports no blocking violations - ‚úÖ i18n panel shows all keys resolved (EN/FR) - ‚úÖ Sovereign profile (Canada) remains applied

    Estimate: 3 story points
    Labels: eva-ui, tool-maintenance, gc-design, ready

    Epic 2: Legacy Code Migration¶

    US-04: Bring legacy screen under Tool control¶

    As a developer
    I want to onboard existing hand-coded Svelte screen into Tool
    So that I can maintain it via metadata and benefit from guardrails

    Acceptance Criteria: - ‚úÖ Legacy screen uses LayoutRenderer with pageId - ‚úÖ Layout metadata stored in layouts/{projectId}/{pageId}.json - ‚úÖ Original screen's visible behavior preserved - ‚úÖ Unmapped sections clearly marked with migration plan

    Estimate: 8 story points
    Labels: eva-ui, legacy-migration, tool-integration, ready

    US-05: Refactor legacy code with Spark (Tool remains source of truth)¶

    As a developer
    I want to use Spark to help refactor/modernize legacy Svelte code
    So that I can speed up updates while keeping Tool as canonical source

    Acceptance Criteria: - ‚úÖ Refactored component respects same props contract (Tool-generated layout) - ‚úÖ No divergence in layout (sections/components in JSON ‚Üí runtime 1:1) - ‚úÖ a11y/i18n checks pass in Tool after refactor

    Estimate: 5 story points
    Labels: spark, eva-ui, legacy-refactor, ready

    US-06: Hybrid model for complex custom screens¶

    As a front-end developer
    I want to gradually migrate code-driven screen into hybrid model
    So that we maintain parts via Tool while keeping complex logic in code

    Acceptance Criteria: - ‚úÖ Layout JSON expresses frame and main sections - ‚úÖ Custom component boundaries clear (documented in registry) - ‚úÖ Page works end-to-end; GC, a11y, i18n requirements tracked

    Estimate: 8 story points
    Labels: eva-ui, hybrid-approach, cpp-d, ready

    üìù Architecture Note (for Todd/Jonathan)¶

    Subject: GitHub Spark as EVA UI Accelerator (Not Governance)

    Context: We're evaluating GitHub Spark for rapid UI prototyping in EVA Suite.

    Recommendation: - Use Spark: Fast prototyping, refactoring assistance, expert spawning - NOT for governance: EVA-Sovereign-UI Tool remains source of truth

    Rationale: 1. Speed: Spark reduces idea ‚Üí prototype from days to minutes 2. Compliance: Tool ensures GC Design System, a11y, i18n enforcement 3. Maintainability: Metadata (JSON) easier to govern than ad-hoc code 4. Hybrid approach: Spark upstream, Tool canonical (best of both)

    Risk Mitigation: - All Spark output must import to Tool (validation step) - Layout changes only via Tool post-import - Clear separation: Spark = accelerator, Tool = maintainer

    Approval Request: Proceed with Spark integration for POD 1 proof-of-concept (Week 1)?
