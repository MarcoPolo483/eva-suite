    Chapter 06: Accessibility¶

    EVA Sovereign UI treats accessibility as a non-negotiable architectural constraint, not an optional enhancement. Every component ships with WCAG 2.1 AA compliance built-in, including keyboard navigation, screen reader support, ARIA semantics, and color contrast validation.

    Government of Canada applications must comply with the Accessible Canada Act and Standard on Web Accessibility. EVA Sovereign UI ensures compliance by default.

    WCAG 2.1 AA Compliance¶

    The Four POUR Principles¶

    EVA Sovereign UI implements the four foundational principles of WCAG 2.1:

    1. Perceivable: Information and UI components must be presentable to users in ways they can perceive.

    • Text alternatives: All images, icons, and non-text content have alt text or ARIA labels
    • Adaptable content: Components work with browser zoom (200-400%), browser text sizing, and high contrast modes
    • Distinguishable: Color contrast ratios meet AA standards (4.5:1 for normal text, 3:1 for large text)

    2. Operable: UI components and navigation must be operable.

    • Keyboard accessible: All functionality available via keyboard (Tab, Enter, Space, Escape, Arrows)
    • Enough time: No time limits on user interactions (unless required for security)
    • Seizures and physical reactions: No flashing content faster than 3 times per second
    • Navigable: Skip links, focus indicators, logical tab order, descriptive headings

    3. Understandable: Information and UI operation must be understandable.

    • Readable: Bilingual content (EN-CA/FR-CA), lang attribute set on all text
    • Predictable: Consistent navigation, no automatic context changes
    • Input assistance: Form errors identified, labels/instructions provided, error suggestions offered

    4. Robust: Content must be robust enough to be interpreted by assistive technologies.

    • Compatible: Valid HTML5, semantic elements, ARIA attributes where needed
    • Future-proof: Web Components standard, no vendor lock-in

    Compliance Checklist¶

    Every EVA Sovereign UI component passes this mandatory checklist:

    • 1.1.1 Non-text Content (A): Images/icons have text alternatives
    • 1.3.1 Info and Relationships (A): Semantic HTML (headings, lists, landmarks)
    • 1.4.3 Contrast (AA): 4.5:1 for normal text, 3:1 for large text
    • 1.4.11 Non-text Contrast (AA): 3:1 for UI components and graphical objects
    • 2.1.1 Keyboard (A): All functionality available via keyboard
    • 2.1.2 No Keyboard Trap (A): Focus can be moved away using keyboard
    • 2.4.3 Focus Order (A): Logical tab order
    • 2.4.7 Focus Visible (AA): Visible focus indicator
    • 3.2.1 On Focus (A): No automatic context change on focus
    • 3.3.1 Error Identification (A): Form errors identified and described
    • 3.3.2 Labels or Instructions (A): Labels for all form inputs
    • 4.1.2 Name, Role, Value (A): ARIA attributes for custom components
    • 4.1.3 Status Messages (AA): ARIA live regions for dynamic content

    Keyboard Navigation¶

    All EVA Sovereign UI components support full keyboard control following WAI-ARIA Authoring Practices 1.2.

    Standard Key Bindings¶

    Component Key Action
    Button Enter/Space Activate button
    Link Enter Navigate to href
    Input Tab Move focus to next field
    Input Shift+Tab Move focus to previous field
    Select Arrow Down/Up Navigate options
    Select Enter/Space Open dropdown
    Select Escape Close dropdown
    Radio Group Arrow Down/Right Next option (loops)
    Radio Group Arrow Up/Left Previous option (loops)
    Checkbox Space Toggle checked state
    Modal Escape Close modal
    Modal Tab Cycle through focusable elements (trapped)
    Tabs Arrow Left/Right Navigate tabs
    Tabs Home/End First/last tab
    Menu Arrow Down/Up Navigate menu items
    Menu Escape Close menu
    Table Tab Move to next interactive cell
    Combobox Arrow Down Open suggestions, navigate list
    Combobox Escape Close suggestions
    Combobox Enter Select highlighted suggestion

    Focus Management¶

    EVA Sovereign UI implements intelligent focus management:

    // Modal dialog example
class EvaGCModal extends LitElement {
  private previousFocus: HTMLElement | null = null;

  open() {
    // Save current focus
    this.previousFocus = document.activeElement as HTMLElement;

    // Move focus to modal close button
    this.updateComplete.then(() => {
      const closeBtn = this.shadowRoot?.querySelector('.eva-gc-modal-close');
      (closeBtn as HTMLElement)?.focus();
    });

    // Trap focus within modal
    this.addEventListener('keydown', this.handleFocusTrap);
  }

  close() {
    // Restore focus to previous element
    this.previousFocus?.focus();

    // Remove focus trap
    this.removeEventListener('keydown', this.handleFocusTrap);
  }

  private handleFocusTrap(event: KeyboardEvent) {
    if (event.key !== 'Tab') return;

    const focusableElements = this.shadowRoot?.querySelectorAll(
      'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
    );

    if (!focusableElements || focusableElements.length === 0) return;

    const firstElement = focusableElements[0] as HTMLElement;
    const lastElement = focusableElements[focusableElements.length - 1] as HTMLElement;

    if (event.shiftKey && document.activeElement === firstElement) {
      event.preventDefault();
      lastElement.focus();
    } else if (!event.shiftKey && document.activeElement === lastElement) {
      event.preventDefault();
      firstElement.focus();
    }
  }
}

    Focus Indicators¶

    All interactive elements display a visible focus indicator:

    :host([focused]) {
  outline: 3px solid var(--gc-color-focus);
  outline-offset: 2px;
}

/* High contrast mode support */
@media (prefers-contrast: high) {
  :host([focused]) {
    outline-width: 4px;
    outline-color: CanvasText;
  }
}

    Screen Reader Support¶

    EVA Sovereign UI supports NVDA (Windows), JAWS (Windows), and VoiceOver (macOS/iOS).

    ARIA Landmarks¶

    Components use semantic HTML5 landmarks and ARIA roles:

    <eva-gc-header role="banner">
  <eva-gc-logo></eva-gc-logo>

  <nav role="navigation" aria-label="Main navigation">
    <eva-gc-nav>
      <!-- Navigation items -->
    </eva-gc-nav>
  </nav>
</eva-gc-header>

<main role="main" id="main-content">
  <eva-gc-breadcrumbs role="navigation" aria-label="Breadcrumb">
    <!-- Breadcrumb items -->
  </eva-gc-breadcrumbs>

  <article>
    <!-- Page content -->
  </article>
</main>

<footer role="contentinfo">
  <eva-gc-footer></eva-gc-footer>
</footer>

    ARIA Labels and Descriptions¶

    All interactive elements have accessible names:

    <!-- Icon button with aria-label -->
<eva-gc-icon-button
  icon="close"
  aria-label-en="Close dialog"
  aria-label-fr="Fermer la bote de dialogue"
></eva-gc-icon-button>

<!-- Form input with aria-describedby -->
<eva-gc-input
  id="email"
  label-en="Email Address"
  label-fr="Adresse courriel"
  aria-describedby="email-hint"
  required
></eva-gc-input>
<div id="email-hint" class="eva-gc-hint">
  <span lang="en">Use your @canada.ca email address</span>
  <span lang="fr">Utilisez votre adresse courriel @canada.ca</span>
</div>

    Live Regions¶

    Dynamic content changes are announced to screen readers:

    class EvaGCToast extends LitElement {
  render() {
    return html`
      <div
        role="status"
        aria-live="polite"
        aria-atomic="true"
        class="eva-gc-toast"
      >
        <eva-gc-icon name="${this.variant}" role="img" aria-label="${this.variantLabel}">
        </eva-gc-icon>
        <span>${this.message}</span>
      </div>
    `;
  }
}

    ARIA Live Region Politeness: - aria-live="polite": Announcements wait for user to finish current task (toasts, form validation) - aria-live="assertive": Immediate announcements (critical errors, security alerts)

    ARIA States and Properties¶

    Attribute Usage Example
    aria-label Accessible name when no visible label <eva-gc-icon-button aria-label="Close">
    aria-labelledby References element containing label <input aria-labelledby="label-id">
    aria-describedby References element with description/hint <input aria-describedby="hint-id">
    aria-expanded Whether collapsible element is expanded <button aria-expanded="false">
    aria-hidden Hide element from screen readers <span aria-hidden="true"> (decorative icons)
    aria-live Announce dynamic content changes <div aria-live="polite">
    aria-current Current item in navigation <a aria-current="page">
    aria-required Required form field <input aria-required="true">
    aria-invalid Invalid form field <input aria-invalid="true">
    aria-checked Checkbox/radio state <input type="checkbox" aria-checked="true">
    aria-disabled Disabled state <button aria-disabled="true">
    aria-controls References controlled element <button aria-controls="panel-id">

    Color Contrast¶

    EVA Sovereign UI enforces WCAG AA color contrast ratios:

    Contrast Requirements¶

    • Normal text (< 18pt): 4.5:1 minimum
    • Large text ( 18pt or 14pt bold): 3:1 minimum
    • UI components (buttons, form borders): 3:1 minimum
    • Graphical objects (icons, charts): 3:1 minimum

    GC Design System Tokens¶

    EVA Sovereign UI's design tokens meet contrast requirements:

    :root {
  /* Primary text on white background: 13.9:1 */
  --gc-color-text-primary: #333333;

  /* Secondary text on white background: 7.0:1 */
  --gc-color-text-secondary: #5A5A5A;

  /* Primary button (background + text): 4.8:1 */
  --gc-color-primary: #26374A;
  --gc-color-primary-text: #FFFFFF;

  /* Error text on white background: 5.3:1 */
  --gc-color-error: #D3080C;

  /* Focus indicator: 3.5:1 */
  --gc-color-focus: #0535D2;
}

    Contrast Validation¶

    Components automatically validate color contrast:

    function validateContrast(foreground: string, background: string): boolean {
  const ratio = calculateContrastRatio(foreground, background);
  return ratio >= 4.5; // AA standard for normal text
}

// Example: Validate custom theme
const customButton = document.createElement('eva-gc-button');
customButton.style.setProperty('--eva-button-bg', '#FF5733');
customButton.style.setProperty('--eva-button-text', '#FFFFFF');

if (!customButton.hasValidContrast()) {
  console.error('Insufficient color contrast for button');
}

    Accessibility Testing Tools¶

    Automated Testing¶

    axe DevTools (browser extension): 

    // Install: npm install --save-dev @axe-core/cli
// Run: npx axe http://localhost:3000

// Programmatic usage
import { axe } from 'axe-core';

const results = await axe.run(document.body);
console.log(results.violations); // Array of accessibility issues

    Lighthouse (Chrome DevTools): - Open Chrome DevTools Lighthouse tab - Select "Accessibility" category - Generate report - Target score: 100/100

    Pa11y (CI/CD integration): 

    npm install --save-dev pa11y

# Run accessibility tests
npx pa11y http://localhost:3000

# CI/CD integration
npx pa11y --standard WCAG2AA --reporter json http://localhost:3000

    Manual Testing Checklist¶

    Keyboard Navigation: - [ ] Tab through all interactive elements - [ ] Verify logical tab order - [ ] Verify visible focus indicators - [ ] Test Escape key closes modals/dropdowns - [ ] Test Arrow keys navigate menus/tabs - [ ] Verify no keyboard traps

    Screen Reader Testing: - [ ] All images have alt text - [ ] Form labels announced correctly - [ ] Error messages announced - [ ] Dynamic content changes announced - [ ] Skip links functional - [ ] Landmarks identified

    Visual Testing: - [ ] Zoom to 200% (no horizontal scroll) - [ ] Browser text sizing (no overlap) - [ ] High contrast mode (Windows: Alt+Shift+PrtScn) - [ ] Color blindness simulation (Chrome DevTools) - [ ] Focus indicators visible

    Browser/AT Combinations: - [ ] Chrome + NVDA (Windows) - [ ] Firefox + NVDA (Windows) - [ ] Edge + JAWS (Windows) - [ ] Safari + VoiceOver (macOS) - [ ] Safari + VoiceOver (iOS)

    Automated CI/CD Testing¶

    # .github/workflows/accessibility.yml
name: Accessibility Tests

on: [push, pull_request]

jobs:
  a11y:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3

      - name: Install dependencies
        run: npm ci

      - name: Build application
        run: npm run build

      - name: Start server
        run: npm run preview &

      - name: Wait for server
        run: npx wait-on http://localhost:3000

      - name: Run axe tests
        run: npx axe http://localhost:3000 --exit

      - name: Run Lighthouse
        run: npx lighthouse http://localhost:3000 --only-categories=accessibility --chrome-flags="--headless" --output=json --output-path=./lighthouse-results.json

      - name: Check Lighthouse score
        run: |
          SCORE=$(jq '.categories.accessibility.score * 100' lighthouse-results.json)
          if (( $(echo "$SCORE < 100" | bc -l) )); then
            echo "Accessibility score $SCORE is below 100"
            exit 1
          fi

    Summary¶

    EVA Sovereign UI achieves WCAG 2.1 AA compliance through:

    • Keyboard navigation: Full keyboard support following WAI-ARIA patterns
    • Screen reader support: ARIA landmarks, labels, live regions for NVDA/JAWS/VoiceOver
    • Color contrast: 4.5:1 for text, 3:1 for UI components
    • Testing tools: axe, Lighthouse, Pa11y for automated validation
    • Manual testing: Comprehensive checklist for keyboard, screen readers, visual validation

    All components ship with accessibility built-inno additional configuration required.
