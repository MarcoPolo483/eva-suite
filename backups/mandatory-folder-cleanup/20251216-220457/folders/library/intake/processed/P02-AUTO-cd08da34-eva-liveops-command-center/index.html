    CDD: EVA LiveOps Command Center¶

    Last Updated: 2025-11-19
    Status: Backlog
    Priority: HIGH
    Epic: EVA LiveOps

    Vision¶

    Build a centralized operations dashboard for platform administrators to monitor EVA 2.0 health, manage incidents, track metrics, and ensure SLA compliance. The Command Center provides real-time visibility into platform performance, user activity, cost trends, and quality metricsâ€”enabling proactive issue detection and rapid response.

    Business Value: Reduce MTTR (Mean Time To Resolution), prevent SLA violations, optimize costs, and maintain platform trust through operational excellence.

    Core Capabilities¶

    • Real-Time Metrics: Platform health, active users, sessions, availability
    • Cost Tracking: Azure spend by service, trending, budget alerts
    • Quality Monitoring: CSAT, P95 latency, error rate, safety triggers
    • Alerting: Threshold violations, anomaly detection, PagerDuty integration
    • Incident Management: Create/track/resolve incidents, postmortem templates
    • Space Overview: Active spaces, budgets, usage, top consumers
    • User Activity: Session logs, query patterns, feedback trends
    • System Logs: Live tail, search, filter by severity/service/user

    User Stories¶

    US-1: As a platform operator, I want real-time health metrics so that I can detect issues before users report them¶

    Acceptance Criteria: - Dashboard shows platform availability (99.9% SLA target) - Active user count, session count, requests/sec - P95 latency chart (target: <300ms) - Error rate chart (target: <1%) - Metrics refresh every 30 seconds (auto-refresh)

    US-2: As a cost manager, I want to track Azure spend so that I can optimize resource usage and prevent budget overruns¶

    Acceptance Criteria: - Current month spend displayed with budget progress bar - Cost breakdown by service (Cosmos DB, OpenAI, Search, APIM) - 6-month trend chart showing spend trajectory - Top 5 spaces by cost (drill-down to space details) - Budget alerts trigger at 80%, 90%, 100% thresholds

    US-3: As an incident responder, I want to create and track incidents so that issues are resolved systematically¶

    Acceptance Criteria: - Create incident form (title, severity, service, description) - Assign incidents to team members (ai2, windowsGuy, SM) - Track incident status (open, investigating, resolved) - Postmortem template auto-generated on resolution - PagerDuty integration for high-severity incidents

    US-4: As a quality analyst, I want to monitor user satisfaction so that I can identify quality degradation early¶

    Acceptance Criteria: - CSAT score chart (daily/weekly/monthly aggregation) - Thumbs up/down ratio per space - User feedback table with search/filter (space, date, sentiment) - Drill-down to individual sessions from feedback - Alert if CSAT drops below 4.0/5.0 for 3 consecutive days

    US-5: As a space administrator (PS-A), I want to see my space's operational metrics so that I can optimize usage and stay within budget¶

    Acceptance Criteria: - Space selector dropdown (filter dashboard to single space) - Space-specific metrics (sessions, users, cost, CSAT) - Budget progress bar (allocated vs. consumed) - Top users by cost (optimize heavy consumers) - Export space metrics as CSV

    Functional Requirements¶

    FR-1: Real-Time Metrics¶

    Dashboard SHALL display platform health metrics (availability, users, sessions, latency, errors) with <30 second refresh.

    FR-2: Cost Tracking¶

    Dashboard SHALL show current month Azure spend, cost by service, 6-month trend, and top 5 spaces by cost.

    FR-3: Alerting¶

    Dashboard SHALL trigger alerts when metrics violate thresholds (availability <99.9%, latency >300ms, error rate >1%, cost >budget).

    FR-4: Incident Management¶

    Dashboard SHALL allow operators to create, assign, track, and resolve incidents with postmortem templates.

    FR-5: PagerDuty Integration¶

    Dashboard SHALL integrate with PagerDuty to trigger pages for high-severity incidents.

    FR-6: User Feedback¶

    Dashboard SHALL display user feedback (thumbs up/down, CSAT, comments) with search/filter by space, date, sentiment.

    FR-7: Space Overview¶

    Dashboard SHALL list all active spaces with usage (sessions, users), budgets (allocated, consumed), and CSAT.

    FR-8: User Activity Logs¶

    Dashboard SHALL provide session logs with search/filter by user, space, date, query text.

    FR-9: System Logs¶

    Dashboard SHALL provide live tail of system logs with filter by severity (error, warn, info), service, user.

    FR-10: Export Data¶

    Dashboard SHALL allow export of metrics, feedback, incidents, logs as CSV or JSON.

    FR-11: Role-Based Access¶

    Dashboard SHALL restrict access to platform operators, admins (PS-A see own space only, platform admins see all).

    FR-12: Bilingual UI¶

    Dashboard SHALL support en-US and fr-FR with language toggle.

    FR-13: Anomaly Detection¶

    Dashboard SHALL detect anomalies in metrics (cost spikes, latency spikes, error bursts) and display alerts.

    FR-14: SLA Tracking¶

    Dashboard SHALL track SLA compliance (99.9% uptime, <300ms P95 latency) with monthly reports.

    FR-15: Notification Preferences¶

    Operators SHALL configure notification preferences (email, SMS, PagerDuty) per alert type.

    Non-Functional Requirements¶

    Performance¶

    • Dashboard load time: <2 seconds (P95)
    • Metrics refresh: Every 30 seconds (auto-refresh)
    • Log search: <5 seconds for 1M records (indexed queries)
    • Export: <10 seconds for 100K records

    Reliability¶

    • Uptime: 99.9% SLA (dashboard itself)
    • Graceful degradation if Application Insights unavailable (cached metrics)
    • Retry logic for transient failures (3 attempts)
    • Fallback to static metrics if live data unavailable

    Security¶

    • HTTPS only (TLS 1.3)
    • OAuth 2.0/OIDC authentication (Entra ID)
    • Role-based access control (platform admins, PS-A, operators)
    • Audit logs for all operator actions (create incident, export data)
    • PII masking in logs (SIN, email, phone)

    Accessibility¶

    • WCAG 2.1 AA compliance (keyboard navigation, screen reader)
    • High-contrast mode for charts
    • Focus indicators visible (â‰¥3:1 contrast ratio)
    • ARIA labels for dynamic content (metrics, alerts)

    i18n¶

    • Bilingual support (en-US/fr-FR)
    • ICU formatting for dates, numbers, currencies
    • All UI text and error messages bilingual

    Observability¶

    • Application Insights custom events (dashboard views, alert triggers, incident creation)
    • Distributed tracing (OpenTelemetry)
    • Self-monitoring (dashboard performance metrics)
    • Alerts for dashboard errors or slow loads

    Scalability¶

    • Support 100+ concurrent operators
    • Query optimization for 10M+ log records
    • Horizontal scaling (stateless dashboard, data in Cosmos DB + App Insights)
    • CDN for static assets

    Technical Architecture¶

    Frontend¶

    • Framework: React 18+ with TypeScript
    • Charts: Recharts or Chart.js (accessible, bilingual)
    • Real-Time Updates: SignalR (WebSockets) for live metrics
    • State Management: React Query (caching), Zustand (UI state)
    • Build: Vite (fast dev server, optimized builds)

    Backend¶

    • API Gateway: Azure APIM (routing, rate limiting)
    • Metrics Source: Application Insights (REST API)
    • Cost Source: Azure Cost Management API
    • Incidents: Cosmos DB (HPK: /incidentId)
    • Logs: Application Insights (Kusto queries)
    • Alerts: Azure Monitor + PagerDuty webhooks

    Data Models¶

    • Metrics: Aggregated from Application Insights (no storage)
    • Incidents: Cosmos DB ({ id, title, severity, service, status, assignee, createdAt, resolvedAt })
    • Alerts: Azure Monitor Action Groups
    • User Preferences: Cosmos DB (HPK: /userId)

    Implementation Phases¶

    Phase 1: Core Dashboard (Sprint 2)¶

    • Build dashboard UI (React + Recharts)
    • Integrate Application Insights (metrics API)
    • Display platform health (availability, users, sessions, latency, errors)
    • Add cost tracking (Azure Cost Management API)
    • Deploy to dev environment

    Phase 2: Alerting & Incidents (Sprint 3)¶

    • Configure Azure Monitor alerts (availability, latency, cost)
    • Build incident management UI (create, track, resolve)
    • Store incidents in Cosmos DB
    • Integrate PagerDuty (webhook for high-severity)
    • Add postmortem templates

    Phase 3: Logs & Feedback (Sprint 4)¶

    • Add user feedback dashboard (thumbs up/down, CSAT, comments)
    • Build log viewer (live tail, search, filter)
    • Add user activity logs (session explorer)
    • Export functionality (CSV, JSON)
    • Role-based access control (RBAC)

    Phase 4: Production Launch (Sprint 5)¶

    • Anomaly detection (cost spikes, latency spikes)
    • SLA tracking dashboard (monthly reports)
    • Notification preferences UI
    • WCAG 2.1 AA audit (axe DevTools)
    • Load testing (100+ concurrent operators)

    Open Questions¶

    Q1: Alerting Thresholds (OPERATIONS)¶

    Question: What are the baseline thresholds for alerts (latency, error rate, cost)?
    Impact: Too sensitive = alert fatigue, too loose = missed incidents
    Recommendation: Start conservative (99.9% availability, <300ms P95, <1% errors), tune based on production data
    Decision Owner: Platform Operations Team + Marco

    Q2: PagerDuty Severity Mapping (INCIDENT MANAGEMENT)¶

    Question: Which incident severities trigger PagerDuty pages?
    Impact: On-call burden vs. response time
    Recommendation: Only SEV-1 (platform down) and SEV-2 (degraded performance) trigger pages
    Decision Owner: Platform Operations Team

    Q3: Log Retention (COST)¶

    Question: How long to retain logs in Application Insights? 30 days, 90 days, 1 year?
    Impact: Compliance vs. cost ($2.30/GB/month for hot storage)
    Recommendation: 30 days hot, 1 year cold (archive to Blob Storage)
    Decision Owner: Compliance Officer + Finance

    Q4: Anomaly Detection (FEATURES)¶

    Question: Use Azure Monitor anomaly detection or custom ML models?
    Impact: Accuracy vs. implementation complexity
    Recommendation: Start with Azure Monitor built-in, custom ML in Q2 2026 if insufficient
    Decision Owner: ai2

    Q5: Dashboard Customization (UX)¶

    Question: Allow operators to customize dashboard layout (drag-and-drop widgets)?
    Impact: Personalization vs. development effort
    Recommendation: Fixed layout for MVP, customization in v2 based on user feedback
    Decision Owner: Marco + UX Partner
