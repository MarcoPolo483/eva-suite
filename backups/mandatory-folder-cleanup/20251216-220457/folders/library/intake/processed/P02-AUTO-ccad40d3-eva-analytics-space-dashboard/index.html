    CDD: EVA Analytics Space Dashboard¶

    Last Updated: 2025-11-19 Status: Backlog Priority: HIGH Epic: EVA Analytics

    Vision¶

    Provide per-space performance and budget dashboards that enable space administrators to monitor their team's usage, costs, satisfaction, and performance metrics—empowering decentralized decision-making and accountability.

    Business Value: Enable self-service space management, reduce admin overhead by 40%, improve cost accountability, track team performance, and identify coaching opportunities.

    Core Capabilities¶

    • Space Overview: Active users, queries/day, avg response time, satisfaction score
    • Budget Tracking: Monthly spend vs. budget, cost per query, forecast to month-end
    • User Activity: Top users, inactive users (>30 days), queries per user heatmap
    • Feature Adoption: Which features used most (chat, RAG, tools), adoption trends
    • Performance Metrics: API latency, error rates, slow queries
    • Satisfaction Trends: NPS scores, feedback sentiment, support tickets
    • Comparative Analytics: How space ranks vs. department average, organization top 10%
    • Export & Sharing: Download reports (PDF, Excel), share with team, email digests

    User Stories¶

    US-1: As a space admin, I want budget tracking so that I stay within allocated spend¶

    Acceptance Criteria:

    • Dashboard shows: MTD spend, budget limit, remaining budget, forecast to month-end
    • Cost breakdown: OpenAI tokens (60%), Azure Search (25%), Cosmos DB (10%), Other (5%)
    • Alert when 80% budget consumed
    • Suggest cost optimizations (reduce max tokens, enable caching)
    • Export budget report (Excel, bilingual)

    US-2: As a space admin, I want user activity so that I coach low-engagement members¶

    Acceptance Criteria:

    • Dashboard shows: Active users (30 days), inactive users (0 queries), queries per user
    • Heatmap: User activity by day of week
    • Identify power users (top 10%) and inactive users
    • Send nudges to inactive users (automated reminders)
    • Drill-down to user-level query history

    US-3: As a team lead, I want feature adoption so that I promote underused tools¶

    Acceptance Criteria:

    • Dashboard shows: Feature usage (Chat 90%, RAG 60%, Email Assistant 20%, Code Review 10%)
    • Identify underused features (<20% adoption)
    • Correlate feature usage with satisfaction scores
    • Create targeted training for low-adoption features
    • Track adoption improvement over 90 days

    US-4: As a space admin, I want performance metrics so that I identify slow queries¶

    Acceptance Criteria:

    • Dashboard shows: Avg API latency, P95 latency, slow queries (>5 seconds)
    • List slowest queries with timestamps, users
    • Suggest optimizations (reduce context, enable streaming)
    • Track performance improvement after optimizations
    • Alert when P95 latency >3 seconds

    US-5: As a manager, I want comparative analytics so that I benchmark my team¶

    Acceptance Criteria:

    • Compare my space vs. department average: queries/user, satisfaction, cost per query
    • Identify gaps (my space below average)
    • Show top-performing spaces (anonymized)
    • Set goals (target: match department average)
    • Track progress toward goals (monthly trend)

    Functional Requirements¶

    FR-1: Space Metrics¶

    System SHALL track space-specific metrics: active users, queries/day, avg latency, satisfaction score, monthly cost.

    FR-2: Budget Tracking¶

    System SHALL calculate MTD spend, compare to budget limit, forecast month-end spend using linear regression.

    FR-3: User Activity¶

    System SHALL identify active users (≥1 query in 30 days), inactive users (0 queries in 30 days), and queries per user.

    FR-4: Feature Adoption¶

    System SHALL calculate feature adoption %: (users using feature) / (total active users) × 100 for each feature.

    FR-5: Performance Monitoring¶

    System SHALL track API latency (P50, P95), error rates, slow queries (>5 seconds) per space.

    FR-6: Satisfaction Tracking¶

    System SHALL aggregate NPS scores, feedback sentiment (GPT-4o), support ticket count per space.

    FR-7: Comparative Analytics¶

    System SHALL benchmark space metrics vs. department average and top 10% performers.

    FR-8: Alerts¶

    System SHALL send alerts for: 80% budget consumed, >30% inactive users, P95 latency >3s, NPS drop >10 points.

    FR-9: Export Options¶

    System SHALL allow exporting dashboards as PDF, Excel, PowerPoint with bilingual summaries.

    FR-10: Drill-Down¶

    System SHALL provide drill-down: space → user → session → query → details.

    FR-11: Access Control¶

    System SHALL enforce RBAC: Space Admins see own space only, Users see self only.

    FR-12: Bilingual Support¶

    System SHALL provide dashboards in en-US and fr-FR with ICU formatting.

    FR-13: Data Retention¶

    System SHALL retain space metrics for 24 months (hot), archive to Azure Storage after.

    FR-14: Scheduled Reports¶

    System SHALL allow scheduling reports (weekly, monthly) with email delivery and SharePoint archiving.

    FR-15: Audit Logging¶

    System SHALL log all dashboard views, exports, and alert triggers to Application Insights.

    Non-Functional Requirements¶

    Performance¶

    • Dashboard load time: <3 seconds (P95)
    • Query response time: <2 seconds (P95)
    • Real-time data refresh: <5 minutes latency
    • Support 1000 concurrent dashboard users
    • Export generation: <30 seconds for 10K rows

    Reliability¶

    • Uptime: 99.95% SLA (Business Critical)
    • Data durability: 99.999999999% (Cosmos DB)
    • Auto-retry failed aggregations (3 attempts)
    • Backup metrics daily (Azure Storage snapshots)

    Security¶

    • RBAC: Space Admins see own space, Users see self
    • Encrypt PII at rest (AES-256) and in transit (TLS 1.3)
    • Audit log all data exports and dashboard views
    • PII masking in comparative analytics (anonymized space IDs)
    • Compliance: PIPEDA, TBS Directive on Privacy

    Accessibility¶

    • WCAG 2.1 AA compliance (Power BI dashboards)
    • Keyboard navigation for all interactive elements
    • Screen reader support (ARIA labels, alt text)
    • High-contrast mode for visuals

    i18n¶

    • Bilingual support (en-US/fr-FR)
    • ICU formatting for dates, numbers, currency
    • Localized dashboard titles, labels, legends
    • Language switcher on all dashboards

    Observability¶

    • Application Insights custom events (dashboard viewed, exported, alert sent)
    • Distributed tracing for API calls (OpenTelemetry)
    • Alerts for slow dashboards, failed exports
    • Metrics: dashboard views/day, avg load time

    Scalability¶

    • Horizontal scaling (Azure Synapse for aggregations)
    • Partition Cosmos DB by /tenantId/spaceId (HPK)
    • Azure CDN for static reports
    • Support 10K spaces per tenant

    Technical Architecture¶

    Backend (Analytics Engine)¶

    • Event Ingestion: Azure Event Hubs (real-time event streaming)
    • Storage: Cosmos DB (transactional), Azure Data Lake Gen2 (analytical)
    • Processing: Azure Synapse Analytics (SQL pools for aggregations)
    • API: Azure Functions HTTP triggers (REST API for dashboard queries)

    Frontend (Dashboards)¶

    • Visualization: Power BI Embedded (interactive dashboards)
    • Reports: Power BI Paginated Reports (scheduled PDF/Excel exports)
    • Admin UI: React SPA (configure alerts, set budgets)

    Integration Points¶

    • eva-api: Event ingestion API (POST /events)
    • eva-auth: RBAC for space-level access
    • eva-metering: Cost data per space
    • eva-i18n: Localized dashboard strings

    Data Models¶

    Space Metrics (Cosmos DB, HPK /tenantId/spaceId):

    {
  "tenantId": "ESDC",
  "spaceId": "prod-hr",
  "date": "2025-11-19",
  "activeUsers": 45,
  "totalQueries": 320,
  "avgLatency": 2.1,
  "nps": 72,
  "mtdCost": 1250.0,
  "budgetLimit": 2000.0,
  "forecastCost": 1800.0
}

    User Activity (Azure Synapse view):

    SELECT
  spaceId,
  userId,
  COUNT(*) as queryCount,
  AVG(duration) as avgDuration,
  MAX(timestamp) as lastActive
FROM events
WHERE eventType = 'query_executed'
  AND timestamp > DATEADD(day, -30, GETDATE())
GROUP BY spaceId, userId

    Power BI Dataset (DirectQuery to Synapse):

    • Space Overview Table: date, spaceId, activeUsers, queries, avgLatency, nps, cost
    • Budget Table: spaceId, mtdCost, budgetLimit, remainingBudget, forecastCost
    • User Activity Table: spaceId, userId, queryCount, lastActive, isInactive
    • Feature Adoption Table: spaceId, feature, usageCount, adoptionRate

    Implementation Phases¶

    Phase 1: Space Metrics (Sprint 7)¶

    • Azure Synapse aggregations for space metrics
    • Power BI dataset (DirectQuery to Synapse)
    • Space overview dashboard (active users, queries, latency)
    • Drill-down to user activity
    • RBAC for space-level access

    Phase 2: Budget & Costs (Sprint 8)¶

    • Integrate eva-metering cost data
    • Budget tracking dashboard (MTD, forecast)
    • Cost breakdown (OpenAI, Search, Cosmos DB)
    • Alerts for budget thresholds (80%, 100%)
    • Cost optimization recommendations

    Phase 3: Feature & Satisfaction (Sprint 9)¶

    • Feature adoption metrics
    • NPS and sentiment analysis
    • Comparative analytics (vs. department, top 10%)
    • Performance metrics (slow queries)
    • Scheduled reports (weekly, monthly)

    Phase 4: Polish & UAT (Sprint 10)¶

    • WCAG 2.1 AA audit (dashboards)
    • Load testing (1000 concurrent users)
    • Email delivery (SendGrid) + SharePoint archiving
    • UAT with pilot spaces
    • Training materials for space admins

    Success Criteria¶

    • ✅ Track 10K+ spaces (100% coverage)
    • ✅ Dashboards load in <3 seconds (P95)
    • ✅ Budget forecasts within 10% accuracy
    • ✅ 40% reduction in admin overhead (self-service)
    • ✅ 90% of spaces stay within budget
    • ✅ WCAG 2.1 AA compliance (axe DevTools passes)
    • ✅ 80% space admin satisfaction score

    Open Questions¶

    Q1: Budget Enforcement (POLICY)¶

    Question: Block queries when budget exceeded or warn only? Impact: Cost control vs. user friction Recommendation: Warn at 80%, block at 110% (soft limit) Decision Owner: Marco

    Q2: Inactive User Definition (UX)¶

    Question: 0 queries in 30 days or 60 days? Impact: Sensitivity of alerts Recommendation: 30 days (more proactive), configurable per space Decision Owner: UX Partner

    Q3: Comparative Anonymization (PRIVACY)¶

    Question: Show space names in rankings or anonymize? Impact: Transparency vs. privacy Recommendation: Anonymize by default, allow opt-in for friendly competition Decision Owner: Security Sentinel

    Q4: Forecast Model (TECH)¶

    Question: Linear regression or time-series (ARIMA) for budget forecasting? Impact: Accuracy vs. complexity Recommendation: Linear regression MVP (Sprint 8), ARIMA if needed (Sprint 11) Decision Owner: ai2

    Q5: Report Frequency (SCOPE)¶

    Question: Allow daily reports or weekly minimum? Impact: Usefulness vs. email noise Recommendation: Weekly minimum (configurable), daily opt-in for power users Decision Owner: UX Partner
