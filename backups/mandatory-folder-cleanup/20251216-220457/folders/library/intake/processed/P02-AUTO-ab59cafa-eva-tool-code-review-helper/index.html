    CDD: EVA Code Review Helper Tool¶

    Last Updated: 2025-11-19 Status: Backlog Priority: MEDIUM Epic: EVA Tools Suite

    Vision¶

    Provide AI-powered code review assistance that analyzes pull requests, identifies best practices violations, detects security vulnerabilities, and suggests improvements—enabling faster code reviews and higher code quality.

    Business Value: Reduce code review time by 40%, catch 80% of security issues pre-merge, improve code consistency, and accelerate developer onboarding.

    Core Capabilities¶

    • PR Analysis: Automatically review pull requests on commit or manual trigger
    • Best Practices: Check adherence to coding standards (linting, formatting, naming)
    • Security Checks: Detect SQL injection, XSS, hardcoded secrets, dependency vulnerabilities
    • Performance Hints: Identify inefficient algorithms, N+1 queries, memory leaks
    • Test Coverage: Flag files missing unit tests, recommend test cases
    • Complexity Scoring: Calculate cyclomatic complexity, suggest refactoring
    • Contextual Suggestions: Provide inline comments with explanations and fixes
    • Language Support: TypeScript, Python, C#, Java, Go, JavaScript

    User Stories¶

    US-1: As a developer, I want automated PR reviews so that I catch issues before human review¶

    Acceptance Criteria:

    • Create PR on GitHub/Azure DevOps
    • EVA bot comments within 2 minutes with review summary
    • Issues categorized: Critical, High, Medium, Low
    • Inline comments on specific lines with suggested fixes
    • Approve/reject PR based on configurable rules

    US-2: As a security officer, I want vulnerability detection so that no secrets are committed¶

    Acceptance Criteria:

    • EVA scans PR for hardcoded secrets (API keys, passwords)
    • Detect SQL injection, XSS, CSRF vulnerabilities
    • Flag outdated dependencies with known CVEs
    • Block PR merge if critical security issues found
    • Generate security report (SARIF format)

    US-3: As a tech lead, I want best practices enforcement so that code is consistent¶

    Acceptance Criteria:

    • Check TypeScript code against ESLint rules
    • Verify Python PEP 8 compliance
    • Flag missing JSDoc/docstrings
    • Suggest variable renaming (meaningful names)
    • Enforce team-specific style guide

    US-4: As a developer, I want performance hints so that I optimize code¶

    Acceptance Criteria:

    • Identify O(n²) algorithms, suggest O(n log n) alternatives
    • Detect N+1 query problems (database calls in loops)
    • Flag inefficient regex patterns
    • Suggest caching opportunities
    • Warn about memory leaks (unclosed resources)

    US-5: As a team lead, I want test coverage tracking so that we maintain quality¶

    Acceptance Criteria:

    • Flag files with <80% test coverage
    • Suggest test cases for uncovered branches
    • Identify missing edge case tests
    • Compare coverage before/after PR
    • Block merge if coverage drops >5%

    Functional Requirements¶

    FR-1: PR Integration¶

    System SHALL integrate with GitHub (GitHub Actions) and Azure DevOps (Pipelines) to trigger reviews on PR creation/update.

    FR-2: Code Analysis¶

    System SHALL analyze code using GPT-4o with prompts for best practices, security, performance, and test coverage.

    FR-3: Security Scanning¶

    System SHALL detect hardcoded secrets (regex + entropy analysis), SQL injection, XSS, CSRF, and dependency vulnerabilities (Snyk/Dependabot).

    FR-4: Best Practices¶

    System SHALL check adherence to linting rules (ESLint, Pylint, Checkstyle) and team style guides.

    FR-5: Performance Analysis¶

    System SHALL identify inefficient algorithms (O(n²)), N+1 queries, memory leaks, and suggest optimizations.

    FR-6: Test Coverage¶

    System SHALL calculate coverage per file (Codecov/Istanbul) and flag files below 80% threshold.

    FR-7: Inline Comments¶

    System SHALL post inline comments on PR with issue description, severity, and suggested fix.

    FR-8: Review Summary¶

    System SHALL generate review summary with issue counts (Critical/High/Medium/Low) and overall recommendation (Approve/Request Changes/Block).

    FR-9: Configurable Rules¶

    System SHALL allow teams to customize rules (enable/disable checks, set thresholds) via .eva-review.yml config file.

    FR-10: Language Support¶

    System SHALL support TypeScript, Python, C#, Java, Go, JavaScript with language-specific analyzers.

    FR-11: SARIF Export¶

    System SHALL export security findings as SARIF (Static Analysis Results Interchange Format) for GitHub Advanced Security.

    FR-12: Auto-Fix¶

    System SHALL apply auto-fixes for linting errors (format code, add missing semicolons) via bot commit.

    FR-13: Learning Mode¶

    System SHALL learn from accepted/rejected suggestions and improve recommendations over time (feedback loop).

    FR-14: Diff-Only Analysis¶

    System SHALL analyze only changed lines in PR (not entire codebase) for performance.

    FR-15: Audit Logging¶

    System SHALL log all reviews, issues found, and actions taken to Application Insights.

    Non-Functional Requirements¶

    Performance¶

    • PR review time: <2 minutes for 500-line PR (P95)
    • Inline comment latency: <30 seconds per comment
    • Support 100 concurrent PR reviews
    • Diff analysis: <10 seconds per file
    • Auto-fix application: <1 minute per PR

    Reliability¶

    • Uptime: 99.9% SLA
    • Auto-retry failed API calls (3 attempts)
    • Graceful degradation (skip non-critical checks if timeout)
    • Queue reviews during high load

    Security¶

    • Encrypt code diffs at rest (AES-256) and in transit (TLS 1.3)
    • RBAC: only PR authors and repo admins access review results
    • PII masking in logs (code snippets)
    • Compliance: PIPEDA, TBS Directive on Privacy
    • Delete code diffs after 30 days

    Accessibility¶

    • WCAG 2.1 AA compliance (review dashboard)
    • Keyboard navigation (navigate issues, apply fixes)
    • Screen reader support (issue descriptions)
    • High-contrast mode for dashboard

    i18n¶

    • Bilingual support (en-US/fr-FR)
    • Localized issue descriptions and suggestions
    • ICU formatting for dates, numbers
    • Bilingual review summaries

    Observability¶

    • Application Insights custom events (review started, completed, issues found)
    • Track issue types, languages, auto-fix acceptance rate
    • Alerts for high failure rates or slow reviews
    • Metrics: reviews/day, avg review time, issues per PR

    Scalability¶

    • Horizontal scaling (Azure Functions for analysis)
    • Partition storage by /tenantId/repoId (HPK)
    • Azure CDN for static assets
    • Support 10K PRs/month per tenant

    Technical Architecture¶

    Backend (Analysis Engine)¶

    • Integration: GitHub App or Azure DevOps Service Hook
    • Queue: Azure Queue Storage (PR review jobs)
    • Analysis: Azure Functions (process diffs, run checks)
    • AI: GPT-4o (contextual suggestions, explanations)
    • Security: Snyk API, GitHub Dependabot (vulnerability scanning)

    Frontend (Dashboard)¶

    • UI: React SPA (review history, config management)
    • Visualization: Issue breakdown by severity, language, file
    • Config Editor: YAML editor for .eva-review.yml

    Integration Points¶

    • GitHub: GitHub App (PR webhooks, inline comments API)
    • Azure DevOps: Service Hook (PR webhooks, REST API)
    • eva-openai: GPT-4o for contextual suggestions
    • eva-auth: RBAC for repo access
    • eva-api: REST API for review status, config

    Implementation Phases¶

    Phase 1: Core PR Analysis (Sprint 8)¶

    • GitHub App integration (webhooks, API)
    • Azure Functions for diff analysis
    • GPT-4o prompts for best practices, security
    • Inline comment posting (GitHub API)
    • Review summary generation

    Phase 2: Security & Coverage (Sprint 9)¶

    • Snyk integration (dependency vulnerabilities)
    • Secret detection (regex + entropy)
    • Test coverage calculation (Codecov API)
    • SARIF export for GitHub Advanced Security
    • Block merge on critical issues

    Phase 3: Performance & Auto-Fix (Sprint 10)¶

    • Performance analysis (O(n²), N+1 queries)
    • Auto-fix for linting errors (bot commits)
    • Complexity scoring (cyclomatic complexity)
    • Configurable rules (.eva-review.yml)
    • Learning mode (feedback loop)

    Phase 4: Azure DevOps & Polish (Sprint 11)¶

    • Azure DevOps integration (Service Hook, REST API)
    • Multi-language support (C#, Java, Go)
    • WCAG 2.1 AA audit (dashboard)
    • Load testing (100 concurrent reviews)
    • UAT with pilot repos

    Success Criteria¶

    • ✅ Review 500-line PR in <2 minutes
    • ✅ 40% reduction in human review time (user survey)
    • ✅ 80% of security issues caught pre-merge
    • ✅ Auto-fix acceptance rate >60%
    • ✅ WCAG 2.1 AA compliance (axe DevTools passes)
    • ✅ 1000+ PRs reviewed per month
    • ✅ 95% user satisfaction score

    Open Questions¶

    Q1: Auto-Fix Scope (TECH)¶

    Question: Auto-fix all linting errors or only safe ones? Impact: Speed vs. risk of breaking code Recommendation: Safe fixes only (formatting, missing semicolons), no logic changes Decision Owner: ai2

    Q2: Learning Mode (FEATURES)¶

    Question: Use feedback to improve suggestions or keep static rules? Impact: Accuracy improvement vs. complexity Recommendation: Static rules MVP (Sprint 8), learning mode Sprint 12 Decision Owner: ai2

    Q3: Blocking Merges (POLICY)¶

    Question: Block merges on critical issues or warn only? Impact: Code quality vs. developer friction Recommendation: Configurable per repo, default warn only Decision Owner: Marco

    Q4: Private Repos (SECURITY)¶

    Question: Analyze code in cloud or provide on-premise option? Impact: Convenience vs. data sovereignty Recommendation: Cloud-based MVP, on-premise Sprint 13 Decision Owner: Security Sentinel

    Q5: Language Priority (SCOPE)¶

    Question: Support all languages in MVP or start with TypeScript/Python? Impact: Time-to-market vs. reach Recommendation: TypeScript/Python MVP (Sprint 8), C#/Java Sprint 11 Decision Owner: Marco
