    EVA Data Model & Folder Schema Documentation - P02 Intake¶

    Intake Date: 2025-12-06 Requested By: Marco Presta (PO) Priority: HIGH Intake Type: Research & Documentation (Complex Epic) Status: PENDING P02 PROCESSING

    Request Summary¶

    Comprehensive documentation of folder schemas and EVA Data Model across all EVA Suite repos, focusing on:

    1. Folder structure conventions for each core trunk repo
    2. EVA Data Model schema (Cosmos DB collections, AI Search indexes, data flow)
    3. Mapping of folder schemas to architectural concerns (SDLC, UI, governance, docs, protocols)
    4. Supporting repos where relevant

    Business Context¶

    Problem: After establishing workspace architecture zoning (LESSON-005), need to understand: - How data flows through the EVA system - How folder structures reflect architectural decisions - Where different types of work should be placed - What data entities exist and how they relate

    Value: Enables developers and agents to: - Place new code in correct repos/folders - Understand EVA's data architecture - Navigate workspace efficiently using folder conventions - Make architecture-aligned decisions

    Urgency: HIGH - Required for onboarding new developers and ensuring consistent code placement across 40+ EVA Suite products

    Technical Requirements¶

    1. Folder Schema Documentation¶

    Scope: Document folder structure conventions for:

    Core Trunks (6 repos - PRIMARY FOCUS): - eva-orchestrator - SDLC automation, Agile Crew patterns, workflows - eva-sovereign-ui - UI components, GC Design System, accessibility - eva-meta - Governance, agent registry, SPxx/Pxx definitions - eva-matrix - Public documentation site (MkDocs/Jekyll) - eva-mcp - Protocol integration (MCP, ACP) - Domain pipelines (future) - jurispipeline, assistme-pipeline, cppd-pipeline

    Supporting Repos (22 repos - SECONDARY): - eva-foundation, eva-rag, eva-infra, eva-api, eva-auth, etc. - Document where relevant to data model or architectural patterns

    Deliverables: - docs/architecture/folder-schemas.md - Comprehensive folder conventions guide - Per-repo sections explaining: - Folder structure (docs/, src/, scripts/, etc.) - File naming conventions - Module organization patterns - Where different types of work belong - Examples: "Where do I add APIM docs?" → eva-orchestrator/docs/architecture/apim-architecture.md

    2. EVA Data Model Documentation¶

    Scope: Define complete data model including:

    Cosmos DB Collections: - Container names (e.g., documents, chunks, users, sessions) - Hierarchical Partition Keys (HPK) structure - Document schemas (JSON structure with field types) - RBAC enforcement at database level (security filters, tenant isolation) - Data classification (Protected B, Unclassified)

    Azure AI Search Indexes: - Index names (e.g., jurisprudence-index, assistme-index) - Field mappings (text, vector embeddings) - Vectorization strategy (embedding models, dimensions) - Security filters (RBAC-aware retrieval) - Hybrid search configuration (vector + keyword)

    Data Flow Diagrams: - Document ingestion pipeline (PDF → chunking → embedding → indexing) - RAG retrieval flow (user query → search → context → LLM → response) - APIM request routing (client → APIM → backend → Cosmos/AI Search) - RBAC context propagation (Azure AD → EVA Foundation → UI)

    Deliverables: - docs/architecture/eva-data-model.md - Comprehensive data model documentation - Entity-relationship diagrams (Mermaid format) - Data flow diagrams (ingestion, retrieval, request routing) - Schema reference tables (Cosmos DB, AI Search) - RBAC enforcement patterns

    3. SPxx vs Pxx Mapping¶

    Scope: Explain conceptual split:

    SPxx (Service Personas) - What users see: - SP01-EVA-Chat-Assistant - Conversational Q&A (C1 autonomy) - SP02-EVA-Data-Ingestion - Document ingestion automation (C2 autonomy) - SP03-Jurisprudence-Research-Assistant - Legal research (C1 autonomy) - (See agents/governance-registry.yaml for full list)

    Pxx (Devtools Patterns) - How we build: - P02-REQ - Requirements refinement - P03-SCR - Scrum support - P04-LIB - Repo documentation - P05-SCA - Scaffolding - P12-UXA - UX, copy, accessibility - (See docs/eva-patterns/README.md for full list)

    Deliverables: - docs/architecture/spxx-pxx-mapping.md - Conceptual split explanation - Table mapping SPxx personas to repos and data collections - Table mapping Pxx patterns to folders and deliverables

    Acceptance Criteria¶

    Must Have (V1.0 - Sprint 1):¶

    • docs/architecture/folder-schemas.md created with core trunk repo folder structures
    • docs/architecture/eva-data-model.md created with Cosmos DB + AI Search schemas
    • Entity-relationship diagrams (Mermaid) for key data entities
    • Data flow diagram for RAG ingestion pipeline
    • Data flow diagram for RAG retrieval flow
    • RBAC enforcement pattern documentation

    Should Have (V1.1 - Sprint 2):¶

    • docs/architecture/spxx-pxx-mapping.md created

    • Data flow diagram for APIM request routing

    • Visual architecture diagrams (beyond Mermaid - draw.io/Figma)
    • Developer onboarding guide using folder navigation

    2. docs/architecture/folder-schemas.md¶

    Structure: 

    # EVA Suite Folder Schemas

## Core Trunks

### eva-orchestrator
- docs/ (architecture, patterns, sprint plans)
- scripts/ (automation, workflows)
- _progress/ (task queue, metrics)
- .github/workflows/ (GitHub Actions)

### eva-sovereign-ui
- packages/eva-sovereign-ui-wc/ (Lit web components)
- themes/ (canada-gc theme)
- docs/ (component docs)

(Repeat for eva-meta, eva-matrix, eva-mcp, domain pipelines)

## Supporting Repos
- eva-foundation (Azure Functions backend)
- eva-rag (RAG pipeline)
- eva-infra (IaC templates)

## File Naming Conventions
- kebab-case for files
- PascalCase for TypeScript classes
- camelCase for functions

## Decision Tree: Where Should I Add...?
- SDLC automation → eva-orchestrator
- UI component → eva-sovereign-ui
- Governance doc → eva-meta
- Public narrative → eva-matrix

**SPxx (Service Personas)** - What users see

- Have C0-C3 autonomy levels
- Governed per GC Agentic Framework

**Pxx (Devtools Patterns)** - How we build
- Internal development patterns
- Used by Agile Crew
- Sprint-based delivery

| SP02-EVA-Data-Ingestion | eva-rag | documents, chunks |

| P12-UXA | eva-sovereign-ui | components/, themes/ |
