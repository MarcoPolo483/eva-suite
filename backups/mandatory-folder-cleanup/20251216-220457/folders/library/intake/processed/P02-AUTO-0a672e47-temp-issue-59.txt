P16 ΓÇô Awareness Protocol Pattern

Goal: Every agent run emits awareness metadata so DVM and humans can gauge reliability and risk.

Scope: All Pods & all agents (P01ΓÇôP21).

Output contract (per action):
	ΓÇó	awareness block:
	ΓÇó	confidence (0ΓÇô1 or Low/Med/High)
	ΓÇó	reflection (what might be wrong / missing)
	ΓÇó	risks (security, policy, ambiguity)
	ΓÇó	missingContext
	ΓÇó	suggestedNextAction (retry, ask human, call another agent, stop)

Lifecycle integration:
	ΓÇó	Phase 2 (Build) ΓåÆ test flakiness & code suggestions are explained.
	ΓÇó	Phase 4ΓÇô6 ΓåÆ governance and LiveOps agents explain why they raised alerts.

This pattern reinforces DevMaster Directive 6 ΓÇô Keep everything explainable, but now at per-agent level.  ∩┐╝

P17 ΓÇô Swarm PR Review Pattern

Goal: PRs are reviewed by a swarm of specialized micro-agents, with a single aggregated summary.

Typical swarm per PR:
	ΓÇó	A11y (UXA ΓÇô P12)
	ΓÇó	i18n (UXA ΓÇô P12)
	ΓÇó	GC Design and content checks (UXA / SEC ΓÇô P11/P12)
	ΓÇó	FinOps / APIM cost headers (MET/P14 ΓÇô P10/P14)
	ΓÇó	Security (SEC ΓÇô P11)
	ΓÇó	Testing & QA (TST ΓÇô P07, CICD ΓÇô P08)

SM / DVM responsibilities:
	ΓÇó	Trigger swarm workflows on each PR (Phase 2ΓÇô6).
	ΓÇó	Enforce a single aggregated comment:
	ΓÇó	Blocking issues (must fix)
	ΓÇó	Warnings (should fix)
	ΓÇó	Informational (nice to have)
	ΓÇó	Link each finding to:
	ΓÇó	Files, tests, policies, or design decisions.

This pattern sits directly on top of:
	ΓÇó	P06 (REV), P07 (TST), P08 (CICD), P11 (SEC), P12 (UXA), P14 (LVO).  ∩┐╝

Outputs:

P19 ΓÇô Action Classification & Safety Pattern

Goal: Every agent action gets a risk class and is constrained by what it is allowed to do.

Classes:
	ΓÇó	C0 ΓÇô Read/Analyze only
	ΓÇó	C1 ΓÇô Dev-branch code/config changes
	ΓÇó	C2 ΓÇô Governance / policy touching changes
	ΓÇó	C3 ΓÇô Production / citizen-facing / infra changes

Rules:
	ΓÇó	C0ΓÇôC1 ΓåÆ may be performed autonomously, via PR drafts (never direct merges).
	ΓÇó	C2ΓÇôC3 ΓåÆ require explicit human approval and often governance sign-off.

Integration:
	ΓÇó	Ties directly into GC Template Vol.3 (LiveOps & Governance).  ∩┐╝
	ΓÇó	Extends DevMaster Directive 3/6 (no merging; log everything).  ∩┐╝

P21 ΓÇô Provenance & Fingerprinting Pattern

Goal: Make every agent action reconstructible: who, what, where, when, why, and with what knowledge.

Fingerprint fields (per action):
	ΓÇó	agentFingerprint:
	ΓÇó	agentId (e.g., P07-TST)
	ΓÇó	agentVersion
	ΓÇó	pod (POD-F/X/O/S-*)
	ΓÇó	service / repo (from .eva-pod.yaml)  ∩┐╝
	ΓÇó	environment (sandbox/dev/demo/prod)
	ΓÇó	taskContext:
	ΓÇó	projectId (EVA Suite, Juris, Assist, etc.)
	ΓÇó	sprintId (e.g., Sprint-1B)
	ΓÇó	workItemId (GitHub/ADO issue)
	ΓÇó	awareness (from P16):
	ΓÇó	confidence, reflection, risks, missingContext, suggestedNextAction
	ΓÇó	provenance:
	ΓÇó	promptRef (where the prompt/context snapshot is stored)
	ΓÇó	contextSources (RAG docs, file paths, CDD/ADR references)
	ΓÇó	model (deployment name, temperature, key config flags)
	ΓÇó	changeSummary (if applicable):
	ΓÇó	actionClass (C0ΓÇôC3, from P19)
	ΓÇó	operation (analysis, suggestion, code_change, config_change, report)
	ΓÇó	filesTouched, git.before, git.after
	ΓÇó	humanApproval:
	ΓÇó	Null or object linking to PR/issue review comments.

Storage:
	ΓÇó	Single, append-only JSONL or table:
	ΓÇó	e.g., eva-meta/logs/agents/audit-log.jsonl

This pattern concretizes DevMaster Directive 6 (ΓÇ£keep everything explainableΓÇ¥) and the CharterΓÇÖs requirement that AI-assisted work be traceable and auditable.  ∩┐╝  ∩┐╝
