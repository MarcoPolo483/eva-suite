    CDD: EVA Domain Assistant Core¶

    Last Updated: 2025-11-19
    Status: Backlog
    Priority: HIGH
    Epic: EVA Domain Assistant

    Vision¶

    Build specialized RAG (Retrieval-Augmented Generation) engine for domain-specific knowledge assistants. EVA Domain Assistant enables organizations to create dedicated AI assistants trained on their own documents, policies, and procedures—with per-space configurations, hybrid search, citation tracking, and enterprise governance.

    Business Value: Transform static document repositories into conversational knowledge bases, reduce search time by 80%, improve information accuracy with citations, and enable self-service for complex policy questions.

    Core Capabilities¶

    • Hybrid Search: Vector + keyword search with configurable weights
    • Document Ingestion: PDF, DOCX, PPTX, TXT, HTML with metadata extraction
    • Chunking Strategy: Configurable chunk size, overlap, semantic boundaries
    • Citation Tracking: Link responses to source documents with page numbers
    • Per-Space Configuration: Custom prompts, temperature, top-k, filters per space
    • Multi-Modal RAG: Text + images + tables extraction
    • Reranking: Semantic reranking for top results (Cohere, Azure AI Search)
    • Query Expansion: Synonym expansion, acronym resolution, multilingual queries

    User Stories¶

    US-1: As a policy analyst, I want to ask questions about jurisprudence documents so that I can find relevant precedents quickly¶

    Acceptance Criteria: - Upload jurisprudence PDFs to dedicated space - Ask natural language questions (e.g., "What are precedents for remote work disputes?") - Receive answers with citations (document name + page number) - Bilingual support (en-US/fr-FR queries and responses) - Response time <5 seconds (P95)

    US-2: As a space administrator (PS-A), I want to configure my RAG assistant so that responses match my domain's needs¶

    Acceptance Criteria: - Configure system prompt (tone, style, domain context) - Set search parameters (top-k, vector weight, reranking) - Upload custom glossary (acronyms, terminology) - Preview configuration with sample queries - Changes apply immediately (no redeployment)

    US-3: As an end user, I want cited responses so that I can verify information accuracy¶

    Acceptance Criteria: - Every response includes citations (document name, page, relevance score) - Click citation to open source document at specific page - Hover over citation to see excerpt preview - Filter results by document type, date, author - Export conversation with citations as PDF

    US-4: As a content manager, I want to ingest documents at scale so that my knowledge base stays current¶

    Acceptance Criteria: - Bulk upload via Azure Blob Storage (100+ files) - Automatic OCR for scanned PDFs - Metadata extraction (title, author, date, tags) - Deduplication detection (warn if document already ingested) - Ingestion status dashboard (queued, processing, completed, failed)

    US-5: As a compliance officer, I want audit logs for all queries so that I can track usage and ensure compliance¶

    Acceptance Criteria: - Audit log captures query, response, documents retrieved, user, timestamp - PII masking in audit logs (SIN, email, phone) - Retention policy (30 days hot, 3 years cold storage) - Quarterly compliance reports (query volume, document access, user activity) - Export audit logs as CSV

    Functional Requirements¶

    FR-1: Document Ingestion¶

    System SHALL ingest PDF, DOCX, PPTX, TXT, HTML with metadata extraction (title, author, date, tags).

    FR-2: Chunking Strategy¶

    System SHALL chunk documents with configurable size (default: 1000 tokens), overlap (default: 200 tokens), and semantic boundary detection.

    FR-3: Vector Embeddings¶

    System SHALL generate embeddings using text-embedding-3-large (Azure OpenAI) and store in Azure AI Search.

    FR-4: Hybrid Search¶

    System SHALL combine vector search + BM25 keyword search with configurable weights (default: 70% vector, 30% keyword).

    FR-5: Reranking¶

    System SHALL rerank top-20 search results using semantic reranking (Cohere or Azure AI Search native).

    FR-6: Citation Tracking¶

    System SHALL return document citations with name, page number, relevance score, and excerpt preview.

    FR-7: Per-Space Configuration¶

    System SHALL allow PS-A to configure system prompt, search parameters, custom glossary per space.

    FR-8: Query Expansion¶

    System SHALL expand queries with synonyms, acronyms, and bilingual translations (en-US ↔ fr-FR).

    FR-9: Multi-Modal Extraction¶

    System SHALL extract text from images (OCR) and tables (structure preservation) during ingestion.

    FR-10: Deduplication¶

    System SHALL detect duplicate documents via content hash and warn administrators before ingestion.

    FR-11: Audit Logging¶

    System SHALL log all queries, responses, documents retrieved, user, timestamp to Cosmos DB.

    FR-12: PII Masking¶

    System SHALL mask PII (SIN, email, phone) in audit logs before storage.

    FR-13: Bulk Upload¶

    System SHALL support bulk upload via Azure Blob Storage with ingestion status tracking.

    FR-14: Document Metadata¶

    System SHALL store and allow filtering by document type, date, author, tags, language.

    FR-15: Export Conversations¶

    System SHALL allow users to export conversations with citations as PDF or JSON.

    Non-Functional Requirements¶

    Performance¶

    • Query response time: <5 seconds (P95) including search + LLM generation
    • Ingestion throughput: 100 documents/hour (parallel processing)
    • Search latency: <1 second for top-20 results (Azure AI Search)
    • Embedding generation: <2 seconds per document chunk

    Reliability¶

    • Uptime: 99.9% SLA (8.76 hours downtime/year)
    • Graceful degradation if reranking unavailable (skip reranking step)
    • Retry logic for transient Azure OpenAI failures (3 attempts)
    • Checkpointing for long-running ingestion jobs

    Security¶

    • Document encryption at rest (AES-256) and in transit (TLS 1.3)
    • Role-based access control (PS-A manages space, users query only)
    • PII masking in audit logs (100% coverage)
    • Tenant isolation (no cross-space document leakage)
    • Data residency (documents stored in Canada regions)

    Accessibility¶

    • WCAG 2.1 AA compliance for query UI
    • Screen reader support for citations and search results
    • Keyboard navigation for document filters
    • High-contrast mode for dashboards

    i18n¶

    • Bilingual support (en-US/fr-FR) for queries and responses
    • Automatic language detection from user profile
    • Multilingual embeddings (cross-lingual retrieval)
    • ICU formatting for dates, numbers in citations

    Observability¶

    • Application Insights custom events (queries, ingestion, errors)
    • Distributed tracing (search → rerank → LLM → citation)
    • Token usage tracking per query (cost attribution)
    • Alerts for high error rates, slow queries, ingestion failures

    Scalability¶

    • Support 10,000+ documents per space (horizontal scaling)
    • Azure AI Search auto-scaling (replicas + partitions)
    • Parallel ingestion (10+ documents simultaneously)
    • Rate limiting (100 queries/hour per user)

    Technical Architecture¶

    Ingestion Pipeline¶

    • Blob Storage: Document upload staging
    • Azure Functions: Triggered ingestion (blob trigger)
    • Document Intelligence: OCR for scanned PDFs
    • Chunking: LangChain RecursiveCharacterTextSplitter
    • Embeddings: Azure OpenAI text-embedding-3-large
    • Storage: Azure AI Search (vectors + metadata)

    Query Pipeline¶

    • Query Expansion: Synonym expansion, acronym resolution
    • Hybrid Search: Azure AI Search (vector + BM25)
    • Reranking: Cohere rerank-multilingual-v3 or Azure native
    • RAG Generation: Azure OpenAI GPT-4o with citations
    • Post-Processing: Citation formatting, PII masking

    Data Models¶

    • Documents: Cosmos DB (HPK: /tenantId/documentId)
    • Chunks: Azure AI Search (index per space)
    • Queries: Cosmos DB (HPK: /tenantId/userId/queryId)
    • Space Config: Cosmos DB (HPK: /tenantId)

    Implementation Phases¶

    Phase 1: Ingestion Pipeline (Sprint 2)¶

    • Build document ingestion API (Azure Functions)
    • Integrate Azure Document Intelligence (OCR)
    • Implement chunking strategy (LangChain)
    • Generate embeddings (Azure OpenAI)
    • Store in Azure AI Search

    Phase 2: Query Pipeline (Sprint 3)¶

    • Build hybrid search (vector + keyword)
    • Integrate reranking (Cohere)
    • Implement RAG generation (GPT-4o with citations)
    • Add query expansion (synonyms, acronyms)
    • Deploy to dev environment

    Phase 3: Per-Space Configuration (Sprint 4)¶

    • Build PS-A configuration UI (prompts, search params)
    • Add custom glossary support
    • Implement preview mode (test queries)
    • Store configs in Cosmos DB (per space)
    • Apply configs dynamically (no redeployment)

    Phase 4: Production Launch (Sprint 5)¶

    • Audit logging (queries, responses, citations)
    • PII masking in logs
    • Bulk upload via Blob Storage
    • Export conversations as PDF
    • Load testing (10,000 docs, 1000 concurrent queries)

    Success Criteria¶

    • ✅ Query response time <5 seconds (P95) for 10,000+ docs
    • ✅ Ingestion throughput: 100 documents/hour
    • ✅ Citation accuracy: 95%+ (source document matches context)
    • ✅ Bilingual support (100% feature parity en-US/fr-FR)
    • ✅ WCAG 2.1 AA compliance (axe DevTools passes)
    • ✅ Zero PII leakage incidents (quarterly audit)
    • ✅ User satisfaction: CSAT >4.0/5.0

    Open Questions¶

    Q1: Reranking Provider (VENDOR)¶

    Question: Cohere rerank-multilingual-v3 or Azure AI Search native reranking?
    Impact: Cost ($3/1000 queries Cohere) vs. feature richness
    Recommendation: Start with Azure native (free), upgrade to Cohere if quality insufficient
    Decision Owner: ai2

    Q2: Embedding Model (COST)¶

    Question: text-embedding-3-large (3072 dims) or text-embedding-3-small (1536 dims)?
    Impact: Accuracy vs. cost/storage ($0.13/1M tokens vs. $0.02/1M tokens)
    Recommendation: text-embedding-3-large for HIGH priority spaces, 3-small for sandbox
    Decision Owner: Marco + ai2

    Q3: Chunk Size Strategy (ACCURACY)¶

    Question: Fixed 1000 tokens or adaptive based on document structure?
    Impact: Citation precision vs. implementation complexity
    Recommendation: Start with fixed 1000, adaptive in Phase 2 based on user feedback
    Decision Owner: ai2

    Q4: Multi-Modal Support (SCOPE)¶

    Question: Extract images/tables in MVP or defer to Phase 2?
    Impact: Feature richness vs. OCR complexity
    Recommendation: Phase 2 (text-only MVP is sufficient for 80% use cases)
    Decision Owner: Marco

    Q5: Cross-Space Search (FUTURE)¶

    Question: Allow users to search across multiple spaces?
    Impact: Knowledge sharing vs. tenant isolation complexity
    Recommendation: Defer to Q2 2026 (tenant isolation is paramount in MVP)
    Decision Owner: Security Sentinel + Marco
