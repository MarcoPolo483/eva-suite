    GitHub Actions Monitoring Checklist¶

    Purpose: Prevent silent automation failures by checking workflow health regularly

    Owner: GitHub Copilot (Scrum Master)

    Last Updated: 2025-12-06

    üî¥ MANDATORY: Session Start Health Check¶

    Run EVERY session start BEFORE any other work:

    # 1. Check last 10 workflow runs (look for failures)
gh run list --limit=10

# 2. Check critical workflows specifically
gh run list --workflow=autonomous-queue-executor.yml --limit=3
gh run list --workflow=p02-intake-pipeline.yml --limit=3
gh run list --workflow=nightly-tree-refresh.yml --limit=2

# 3. Check for recent failures (last 24 hours)
gh run list --status=failure --limit=20

# 4. If failures found, investigate immediately
gh run view <run-id>  # Get failure details

    Expected Results: - ‚úÖ autonomous-queue-executor.yml: Last run <2 hours ago, status: success - ‚úÖ p02-intake-pipeline.yml: Last run <7 hours ago, status: success - ‚úÖ nightly-tree-refresh.yml: Last run <26 hours ago, status: success

    Action on Failure: 1. Investigate root cause (gh run view <run-id>) 2. Document in .eva-memory.json (new LESSON if pattern discovered) 3. Fix before continuing with other work 4. Report to Marco with summary and remediation

    üìä Critical Workflows¶

    1. autonomous-queue-executor.yml¶

    Purpose: Process TASK-QUEUE.json (autonomous task execution) Schedule: Hourly (on the hour) Last Run Should Be: <2 hours ago Failure Impact: HIGH - Tasks don't execute, blocking all automation Check: gh run list --workflow=autonomous-queue-executor.yml --limit=5

    2. p02-intake-pipeline.yml¶

    Purpose: Process documents in to upload/ folder (P02-REQ pattern) Schedule: Every 6 hours Last Run Should Be: <7 hours ago Failure Impact: MEDIUM - Intake backlog grows, features delayed Check: gh run list --workflow=p02-intake-pipeline.yml --limit=3

    3. nightly-tree-refresh.yml¶

    Purpose: Update eva-suite-full-tree.txt (folder structure snapshot) Schedule: Daily at 2:00 AM EST Last Run Should Be: <26 hours ago Failure Impact: LOW - Tree becomes stale, file discovery slower Check: gh run list --workflow=nightly-tree-refresh.yml --limit=2

    4. rollout-*.yml (POD Rollouts)¶

    Purpose: Deploy .eva-memory.json to POD repos Schedule: On-demand (manual trigger) Last Run Should Be: N/A (triggered manually) Failure Impact: MEDIUM - Memory not synced to repos Check: gh run list --workflow=rollout-pod-f-memory.yml --limit=1

    üîç Daily Health Check (Morning Routine)¶

    When: First session of the day Duration: 2-3 minutes

    # Check failures in last 24 hours
gh run list --status=failure --created=$(date -d '1 day ago' +%Y-%m-%d)

# Check workflow success rates
gh workflow list

# Verify folder tree is fresh (<48h old)
ls -lh eva-suite-full-tree.txt

    Expected: - 0 failures in last 24h (or documented/resolved) - All critical workflows showing recent runs - eva-suite-full-tree.txt modified within 48h

    üìà Weekly Review (Friday End-of-Sprint)¶

    When: Friday afternoon, sprint review prep Duration: 10 minutes

    Metrics to Check:¶

    1. Workflow Success Rate (target: >95% for critical workflows) 

      # Manual analysis: count successes vs failures
gh run list --workflow=autonomous-queue-executor.yml --limit=50

    2. Flaky Workflows (failures that succeed on re-run)

    3. Document patterns (time-based? resource contention?)

    4. Add to .eva-memory.json if solution found

    5. Task Queue Health 

      # Check for stale tasks (>72h in READY status)
cat _progress/TASK-QUEUE.json | grep -A 5 '"status": "READY"'

    6. Intake Pipeline Throughput 

      # Count processed documents
ls docs/intake/processed/ | wc -l

    üö® Incident Response Procedure¶

    On Workflow Failure:¶

    1. Immediate Triage 

      gh run view <run-id>  # Get failure logs

    2. Root Cause Analysis

    3. Transient failure? (network, GitHub outage) ‚Üí Re-run workflow
    4. Code bug? ‚Üí Fix immediately, add test
    5. Config issue? ‚Üí Update workflow YAML

    6. Dependency issue? ‚Üí Update lockfiles, test locally

    7. Document in Memory 

      // Add to .eva-memory.json lessons[]
{
  "id": "LESSON-XXX",
  "date": "2025-12-06",
  "title": "Workflow Failure: <root cause>",
  "context": "<what happened>",
  "lesson": "<how to prevent>",
  "pattern": "<detection + fix pattern>"
}

    8. Report to Marco

    9. Incident summary (what failed, when, impact)
    10. Root cause
    11. Remediation taken
    12. Prevention plan

    üîß Maintenance Tasks¶

    Monthly:¶

    • Review workflow logs for optimization opportunities
    • Check GitHub Actions usage (minutes consumed)
    • Update workflow dependencies (actions versions)
    • Review alerting strategy (can we automate health checks?)

    Quarterly:¶

    • Evaluate need for monitoring dashboard
    • Review workflow security (secrets, permissions)
    • Benchmark workflow performance (duration trends)

    üìù Monitoring Command Reference¶

    # List all workflows
gh workflow list

# List recent runs (all workflows)
gh run list --limit=20

# List runs for specific workflow
gh run list --workflow=<workflow.yml> --limit=10

# View detailed run logs
gh run view <run-id>

# Watch running workflow in real-time
gh run watch <run-id>

# List failed runs only
gh run list --status=failure --limit=20

# List runs from specific date
gh run list --created=2025-12-06

# Cancel running workflow
gh run cancel <run-id>

# Re-run failed workflow
gh run rerun <run-id>

# Manually trigger workflow
gh workflow run <workflow.yml>
