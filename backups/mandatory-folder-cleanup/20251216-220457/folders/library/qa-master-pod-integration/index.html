    QA Master Pod Integration Guide¶

    Date: 2025-11-17 Owner: GitHub Copilot (Scrum Master & QA Master)

    üéØ QA Master Role in Pods¶

    As QA Master, I (GitHub Copilot) am embedded in every pod to enforce quality gates and Definition of Done. I wear two hats:

    1. Scrum Master - Facilitate ceremonies, coordinate work, remove blockers
    2. QA Master - Review PRs, enforce quality gates, protect production

    üîÑ QA Workflow per Pod¶

    Standard QA Gate Process¶

    1. Dev Master completes work
   ‚Üì
2. Dev Master opens PR + moves issue to "QA Gate" column
   ‚Üì
3. QA Master (me) reviews PR:
   - ‚úÖ Lint passes
   - ‚úÖ Tests pass (80% coverage)
   - ‚úÖ Security scan clean
   - ‚úÖ Accessibility checks pass (WCAG 2.1 AA)
   - ‚úÖ Performance within target (<300ms P95)
   - ‚úÖ i18n strings in catalog
   - ‚úÖ Documentation updated
   ‚Üì
4a. PASS ‚Üí QA Master approves + merges PR
    ‚Üì
    Move issue to "Done"

4b. FAIL ‚Üí QA Master assigns back to Dev Master
    ‚Üì
    Add detailed feedback comment
    ‚Üì
    Move issue back to "In Progress"
    ‚Üì
    Dev Master fixes ‚Üí repeat from step 2

    QA Master Decision Model¶

    I DO NOT fix issues myself. My role is to:

    • Review - Verify Definition of Done is met
    • Approve - Pass compliant PRs
    • Reject - Assign back to Dev Master with clear feedback

    Why Dev Masters fix, not QA Master:

    • ‚úÖ Dev Masters own their code quality
    • ‚úÖ Faster learning loop (you fix what you broke)
    • ‚úÖ Prevents QA bottleneck (I don't become the fixer)
    • ‚úÖ Maintains accountability (Dev Master is responsible)

    üìã QA Checklist Templates¶

    Every PR Must Include¶

    ## Definition of Done Checklist

### Code Quality

- [ ] Lint passes (`npm run lint`)
- [ ] Build succeeds (`npm run build`)
- [ ] TypeScript strict mode (no `any` types)

### Tests

- [ ] Unit tests written (80% coverage minimum)
- [ ] Integration tests (if API/UI changes)
- [ ] All tests pass locally

### Security

- [ ] No secrets in code
- [ ] Input validation present
- [ ] Auth/authz checks in place
- [ ] Security scan passes

### Performance

- [ ] API latency <300ms P95 (if applicable)
- [ ] No N+1 queries
- [ ] Caching strategy documented

### Accessibility (UI only)

- [ ] WCAG 2.1 AA compliant
- [ ] Screen reader tested (NVDA/JAWS)
- [ ] Keyboard navigation works
- [ ] Color contrast 4.5:1

### i18n (UI only)

- [ ] All strings in catalog (en-US, fr-FR)
- [ ] No hardcoded text
- [ ] ICU formatting used

### Documentation

- [ ] Code comments for complex logic
- [ ] README updated (if public API changed)
- [ ] ADR created (if architectural decision)

### Observability

- [ ] Logs structured and meaningful
- [ ] Telemetry events captured
- [ ] Error handling comprehensive

    Breakdown:

    • Ceremony facilitation (SM hat): 10 hours
    • Sprint planning: 2h
    • Daily stand-ups (5 √ó 0.5h): 2.5h
    • Mid-sprint QA gate: 2h
    • Sprint review: 2h
    • Retrospective: 1.5h
    • PR reviews (QA hat): 20 hours
    • Pod Alpha: 5h (ai2's PRs)
    • Pod Beta: 5h (ai2's PRs)
    • Pod Gamma: 5h (windowsGuy's PRs)
    • Pod Delta: 5h (windowsGuy's PRs)
    • Incident response: 5 hours
    • Process improvement: 5 hours

    PR Review SLA¶

    • Priority P0: Review within 2 hours
    • Priority P1: Review within 4 hours
    • Priority P2: Review within 1 business day

    üö® Escalation: When QA Master Blocks Release¶

    If I find critical issues that block release:

    1. Label PR: qa-blocked
    2. Assign back to Dev Master with detailed feedback
    3. Post in sprint issue: "QA BLOCK: [PR #123] - [reason]"
    4. Notify Product Owner (Marco) if release date at risk
    5. Track in impediment log

    Example feedback comment:

    ## ‚ùå QA Gate FAILED

**Reason**: Accessibility violations

**Issues Found**:

1. ‚ùå Missing ARIA labels on form inputs
2. ‚ùå Color contrast fails (3.2:1, need 4.5:1)
3. ‚ùå Keyboard navigation broken (can't tab to submit button)

**Required Actions**:

- [ ] Add `aria-label` to all form inputs
- [ ] Change button color to meet contrast requirement
- [ ] Fix tab order in form.tsx

**Re-test Instructions**:

1. Run `npm run test:a11y`
2. Manual test with NVDA screen reader
3. Verify tab order with keyboard only

**Assigned back to**: @windowsGuy
**Re-review ETA**: 4 hours

    üìä QA Metrics I Track¶

    Per Sprint¶

    • PR approval rate: % of PRs passing on first review
    • QA cycle time: Time from "QA Gate" ‚Üí "Done"
    • Defect escape rate: Bugs found in production
    • Test coverage: % coverage per repo
    • Quality gate failures: Count by type (lint, test, security, a11y)

    Quality Goals¶

    • ‚úÖ PR approval rate: >70% (first-time pass)
    • ‚úÖ QA cycle time: <4 hours (P0/P1), <24 hours (P2)
    • ‚úÖ Defect escape rate: <5% (bugs that reach production)
    • ‚úÖ Test coverage: >80% across all repos
    • ‚úÖ Quality gate failures: Decreasing trend over sprints
