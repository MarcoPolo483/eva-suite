    P17: Swarm PR Review¶

    Pattern Code: P17
    Short Code: SWR
    Name: Swarm PR Review
    Type: DevTools Pattern
    Scope: Code Review (Multi-Perspective)
    Maturity: Alpha

    ğŸ“‹ What It Is¶

    P17 Swarm PR Review (SWR) acts as a multi-perspective reviewer for changes (PRs, patches, MRs). It simulates a "swarm" of specialists using patterns P01â€“P21, providing:

    • Multi-lens review: Security (P11), testing (P07), UX (P12), metrics (P10), ops (P14), etc.
    • Pattern-tagged findings: Each issue labeled with relevant patterns (e.g., "P07-TST: Missing test coverage")
    • Minimal, high-impact suggestions: Top 3-7 issues, not exhaustive lists
    • Merge-readiness judgment: Non-binding status (Looks good / OK with minor fixes / Needs work)

    P17 does NOT replace other agents - it invokes their lenses for comprehensive, structured feedback.

    ğŸ¯ Mission¶

    P17 Swarm PR Review Agent's mission:

    1. Swarm-style PR review: Apply multiple expert perspectives (P06-REV, P07-TST, P11-SEC, P12-UXA, P14-LVO, etc.)
    2. Pattern-tagged feedback: Tag each finding with pattern(s) and severity (BLOCKER / IMPORTANT / NICE-TO-HAVE)
    3. Merge-readiness judgment: Provide advisory status (non-authoritative)
    4. Awareness & safety: Attach P16 awareness block with confidence, assumptions, risks

    ## Swarm Review Checklist (P17 â€“ SWR)

### P06-REV: Code Review Basics
- [ ] Code follows repo conventions (style, naming, structure)
- [ ] Error handling present (try/catch, logging)
- [ ] No obvious bugs or code smells

### P07-TST: Testing
- [ ] Test coverage meets 80% threshold
- [ ] Tests cover happy path + failure cases
- [ ] Tests are maintainable (not brittle)

### P09-RUN: Operations Impact
- [ ] Runbook updated if behavior changes
- [ ] Deployment rollback plan exists
- [ ] Incident response procedures updated if needed

### P10-MET: Metrics & Telemetry
- [ ] Key actions logged with structured fields
- [ ] Alerts configured for failures
- [ ] Cost impact estimated (if applicable)

### P11-SEC: Security & Compliance
- [ ] No secrets in code (use Key Vault)
- [ ] RBAC enforced (principle of least privilege)
- [ ] Data residency respected (Canada-only)
- [ ] Input validation present

### P12-UXA: UX & Accessibility (if UI changes)
- [ ] WCAG 2.1 AA compliant (keyboard nav, screen readers, contrast)
- [ ] Bilingual (en-CA/fr-CA)
- [ ] GC Design System components used

### P14-LVO: LiveOps Readiness
- [ ] SRE can debug failures (sufficient logging)
- [ ] Production impact assessed (downtime, degradation)
- [ ] Rollback procedure clear

### P16-AWR: Awareness
- [ ] Assumptions documented
- [ ] Risks identified
- [ ] Confidence level stated (Low/Medium/High)
- [ ] Next actions clear

### P19-ACS: Action Classification
- [ ] Autonomy level labeled (C0/C1/C2/C3)
- [ ] Safety guardrails appropriate for autonomy level

### P21-PRV: Provenance
- [ ] Key actions generate provenance fingerprints
- [ ] Audit trail sufficient for ATO evidence

    âš ï¸ Important Notes¶

    P17 is Advisory, Not a Gate¶

    • Non-binding: Human reviewers have final authority
    • Merge-readiness is a suggestion: Not a blocker
    • Use as input: Augment human review, don't replace it

    P17 is NOT Exhaustive¶

    • Top 3-7 issues: Focus on high-impact findings
    • Not a checklist auditor: Assumes baseline quality exists
    • Pattern-focused: Calls out pattern-specific issues, not generic nitpicks

    P17 Requires Context¶

    • Provide PR description: What's changing and why
    • Provide repo/pod/phase: Context matters for severity
    • Provide diffs: Can't review without seeing changes

    ğŸ“Š Success Metrics¶

    • Issue detection rate: % of P17 findings that prevent production issues
    • False positive rate: % of P17 findings rejected by human reviewers
    • Review coverage: % of PRs that get swarm review
    • Pattern balance: Are all patterns (P06-P21) represented in findings?
    • Merge time impact: Does P17 speed up or slow down reviews?

    ğŸ¯ Autonomy Level¶

    P17 Itself: C0 (Human-in-the-loop) - Provides feedback, does not merge - All merge decisions require human approval - Advisory only, not enforcement

    ğŸ“– Example Severity Levels¶

    ğŸ”´ BLOCKER¶

    • Severity: Must fix before merge
    • Examples:
    • Secrets committed to code
    • Security vulnerability (SQL injection, XSS)
    • Breaking change without migration plan
    • Critical test failures
    • Data residency violation

    ğŸŸ¡ IMPORTANT¶

    • Severity: Should fix before merge (but negotiable)
    • Examples:
    • Missing test coverage (<80%)
    • No runbook for new failure mode
    • No alerting for new error condition
    • Accessibility issue (WCAG 2.1 AA)
    • Performance regression (>300ms P95)

    • Severity: Can be follow-up PR
