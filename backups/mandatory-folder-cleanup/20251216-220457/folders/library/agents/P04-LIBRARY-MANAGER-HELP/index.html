    Agent Service Catalog - P04 (HELP)¶

    Agent ID: P04 Agent Name: Library Manager POD: POD-F (Foundation) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P04 is the Library Manager agent responsible for shared library development, package management, SDK creation, code reuse, and dependency management across the EVA Suite.

    Core Services¶

    1. Shared Library Development¶

    • Description: Create and maintain shared libraries used across multiple EVA repos
    • Input: Common functionality, reuse requirements, API specifications
    • Output: Packaged library, documentation, usage examples
    • Example: P04: Create eva-core library for shared domain models

    Input Template: 

    P04 REQUEST: Shared Library
  LIBRARY_NAME: [Name, e.g., "eva-core"]
  PURPOSE: [What common functionality this provides]
  CONSUMERS: [Which repos will use this]
  FUNCTIONALITY:
    - [Function 1, e.g., "Document model"]
    - [Function 2, e.g., "Space model"]
  REQUIREMENTS:
    - language: [Python | TypeScript | Both]
    - versioning: [Semantic versioning]
    - distribution: [PyPI | npm | Internal]

    2. Package Management & Dependencies¶

    • Description: Manage dependencies, version conflicts, security updates
    • Input: Dependency requirements, security alerts, version constraints
    • Output: Updated dependency files, conflict resolutions, security patches
    • Example: P04: Resolve Azure SDK version conflicts across eva-api and eva-rag

    Input Template: 

    P04 REQUEST: Dependency Management
  REPOSITORY: [Repo name]
  ISSUE:
    type: [Conflict | Security | Upgrade]
    description: [What's the problem]
  DEPENDENCIES:
    - name: [Package name]
      current_version: [Version]
      desired_version: [Version]
      reason: [Why change needed]
  CONSTRAINTS:
    - [Constraint 1, e.g., "Must maintain Python 3.11 compatibility"]

    3. SDK & API Client Development¶

    • Description: Create SDKs and API clients for EVA services
    • Input: API specification (OpenAPI/Swagger), target languages
    • Output: SDK package, documentation, code examples
    • Example: P04: Create Python SDK for eva-api GraphQL and REST endpoints

    Input Template: 

    P04 REQUEST: SDK Development
  API: [API name and version]
  SPECIFICATION: [OpenAPI/GraphQL schema location]
  TARGET_LANGUAGES:
    - [Language 1, e.g., "Python"]
    - [Language 2, e.g., "TypeScript"]
  FEATURES:
    - [Feature 1, e.g., "Automatic retry"]
    - [Feature 2, e.g., "Type safety"]
  DISTRIBUTION: [PyPI | npm | GitHub]

    4. Code Reuse & Duplication Analysis¶

    • Description: Identify duplicate code and extract to shared libraries
    • Input: Codebase to analyze, duplication threshold
    • Output: Duplication report, refactoring recommendations, shared library proposals
    • Example: P04: Analyze eva-api and eva-rag for duplicate Cosmos DB code

    Input Template: 

    P04 REQUEST: Duplication Analysis
  REPOSITORIES:
    - [Repo 1]
    - [Repo 2]
  SCOPE: [What to analyze - models, utilities, services]
  THRESHOLD: [e.g., "3+ lines duplicated"]
  ACTION: [Report | Refactor | Create Library]

    5. Versioning & Release Management¶

    • Description: Manage library versions, releases, changelogs
    • Input: Changes made, semantic version bump type
    • Output: New version, changelog, release notes, published package
    • Example: P04: Release eva-core v2.1.0 with new HPK support

    Input Template: 

    P04 REQUEST: Library Release
  LIBRARY: [Library name]
  CURRENT_VERSION: [e.g., "2.0.5"]
  CHANGES:
    - type: [BREAKING | FEATURE | FIX]
      description: [What changed]
  NEXT_VERSION: [e.g., "2.1.0" or "auto"]
  RELEASE_NOTES: [Markdown content]
  DISTRIBUTION: [Where to publish]

    P04: Create eva-core library for domain models
- Models: Document, Space, Tenant, User
- Target: Python 3.11+
- Consumers: eva-api, eva-rag, eva-admin
- Distribution: Internal PyPI or GitHub

    Example 2: Resolve Dependency Conflict¶

    P04 REQUEST:
  TASK: Resolve Azure SDK conflict
  CONTEXT: eva-api uses azure-cosmos 4.5, eva-rag uses 4.3, causing issues
  SCOPE: Both repositories
  DELIVERABLES: Updated requirements.txt, migration guide if breaking

    Example 3: Create SDK¶

    P04: Create Python SDK for eva-api
- API: REST + GraphQL endpoints
- Features: Type hints, async support, automatic retry
- Examples: CRUD operations, query, upload
- Publish: PyPI as eva-api-client

    Service Dependencies¶

    Upstream (P04 depends on): - P01 (System Architect): Architecture standards, technology choices - P02 (Data Architect): Data model definitions - P03 (Security Architect): Security requirements for libraries - P08 (API Designer): API specifications for SDK generation

    Downstream (Depends on P04): - All repositories: Depend on shared libraries - P07 (Testing Agent): Test shared libraries - P11 (DevOps): Deploy/publish packages - P12 (Documentation Writer): Document library usage

    Service Standards¶

    Library Standards¶

    • Semantic versioning (MAJOR.MINOR.PATCH)
    • CHANGELOG.md for all releases
    • Comprehensive README with examples
    • Type hints (Python) / TypeScript definitions
    • Unit tests with 80% coverage
    • CI/CD pipeline for releases

    Package Standards¶

    • requirements.txt or package.json for dependencies
    • Pinned versions for production
    • Version ranges for libraries
    • Security scanning in CI (Dependabot, Snyk)
    • License file (MIT or Apache 2.0)

    Code Quality Standards¶

    • Linting (Pylint, ESLint)
    • Formatting (Black, Prettier)
    • Type checking (mypy, TypeScript)
    • Documentation (docstrings, JSDoc)
    • No circular dependencies

    Distribution Standards¶

    • PyPI for Python libraries
    • npm for JavaScript/TypeScript
    • GitHub Releases for all versions
    • Signed releases (GPG)
    • Release notes with migration guides

    Common Patterns¶

    Pattern 1: Semantic Versioning¶

    ## Version Bump Rules

**MAJOR** (breaking changes):
- Remove public API
- Change function signature
- Change data models

**MINOR** (new features):
- Add new function
- Add new parameter (with default)
- Add new class

**PATCH** (bug fixes):
- Fix bugs
- Update dependencies (non-breaking)
- Documentation updates

    Pattern 2: Shared Library Structure¶

    eva-core/
 src/
    eva_core/
        __init__.py
        models/
           document.py
           space.py
           user.py
        utils/
            hash_chain.py
 tests/
    test_models.py
 pyproject.toml
 README.md
 CHANGELOG.md
 LICENSE

    Pattern 3: Dependency Version Pinning¶

    # pyproject.toml (library - use ranges)
[project.dependencies]
azure-cosmos = ">=4.5.0,<5.0.0"
pydantic = ">=2.0.0,<3.0.0"

# requirements.txt (application - pin exactly)
azure-cosmos==4.5.1
pydantic==2.5.3

    [If breaking changes] 

    ### Dependency Analysis Report
**File Naming**:
    P04-DEPENDENCY-ANALYSIS-[REPO]-[YYYYMMDD-HHMMSS].md 
    ### SDK Documentation
**File Naming**:
    P04-SDK-[API-NAME]-[LANGUAGE]-[YYYYMMDD-HHMMSS].md 
    ### File Naming Convention

**All deliverables must follow**:
    P04-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md ```

    Examples: - P04-LIBRARY-EVA-CORE-2.1.0-20251209-120000.md - P04-DEPENDENCY-ANALYSIS-EVA-API-20251209-130000.md - P04-SDK-EVA-API-PYTHON-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P04- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md

    Quality Checklist¶

    Before marking P04 work complete:

    • Library follows semantic versioning
    • CHANGELOG.md updated
    • README.md has clear examples
    • Unit tests written (80% coverage)
    • Type hints / TypeScript definitions included
    • CI/CD pipeline configured
    • Security scanning enabled
    • License file present
    • Published to correct repository (PyPI, npm)
    • Dependent repos notified of release
