    Agent Service Catalog - P02 (HELP)¶

    Agent ID: P02 Agent Name: Data Architect POD: POD-F (Foundation) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P02 is the Data Architect agent responsible for database design, schema management, data modeling, and storage optimization across the EVA Suite.

    Core Services¶

    1. Database Schema Design¶

    • Description: Design and implement database schemas for Cosmos DB, PostgreSQL, etc.
    • Input: Business requirements, data models, performance targets
    • Output: Schema definitions, migration scripts, documentation
    • Example: P02: Design Cosmos DB schema with HPK for multi-tenant document storage

    Input Template: 

    P02 REQUEST: Schema Design
  ENTITY: [Entity name, e.g., "documents", "users"]
  DATABASE: [Cosmos DB | PostgreSQL | MongoDB]
  ACCESS_PATTERNS:
    - [Pattern 1, e.g., "Get document by ID within space"]
    - [Pattern 2, e.g., "List all documents for user"]
  PERFORMANCE_TARGETS:
    - read_latency: [e.g., "<50ms p95"]
    - write_latency: [e.g., "<100ms p95"]
    - throughput: [e.g., "1000 ops/sec"]
  SCALE:
    - expected_volume: [e.g., "10M documents"]
    - growth_rate: [e.g., "20% annually"]
  CONSTRAINTS:
    - [Constraint 1, e.g., "Multi-tenant isolation required"]
    - [Constraint 2, e.g., "Audit trail mandatory"]

    2. Data Migration Planning¶

    • Description: Plan and execute data migrations between schemas or databases
    • Input: Source schema, target schema, data volume, constraints
    • Output: Migration scripts, rollback procedures, validation tests
    • Example: P02: Create migration from legacy partition key to HPK

    Input Template: 

    P02 REQUEST: Migration Planning
  SOURCE:
    database: [Database name]
    container: [Container name]
    schema_version: [e.g., "v1.0"]
    partition_key: [Current partition key]
  TARGET:
    database: [Database name]
    container: [Container name]
    schema_version: [e.g., "v2.0"]
    partition_key: [New partition key, e.g., "[/space_id, /tenant_id, /user_id]"]
  DATA_VOLUME:
    record_count: [e.g., "50000"]
    total_size: [e.g., "2GB"]
  REQUIREMENTS:
    - downtime: [e.g., "Zero downtime" | "Max 1 hour"]
    - rollback: [YES | NO]
    - verification: [YES | NO]
    - backup: [YES | NO]
  TIMELINE: [e.g., "2 weeks"]

    3. Performance Optimization¶

    • Description: Optimize database queries, indexes, and partition strategies
    • Input: Performance metrics, query patterns, throughput requirements
    • Output: Optimization recommendations, index definitions, benchmarking
    • Example: P02: Optimize cross-partition queries for document retrieval

    Input Template: 

    P02 REQUEST: Performance Optimization
  TARGET:
    database: [Database name]
    container: [Container name]
    operation: [e.g., "cross-partition query", "batch insert"]
  CURRENT_METRICS:
    - latency_p50: [e.g., "150ms"]
    - latency_p95: [e.g., "2000ms"]
    - latency_p99: [e.g., "5000ms"]
    - throughput: [e.g., "50 ops/sec"]
    - ru_consumption: [e.g., "20 RU/query"]
  TARGET_METRICS:
    - latency_p95: [e.g., "<500ms"]
    - throughput: [e.g., ">200 ops/sec"]
    - ru_consumption: [e.g., "<10 RU/query"]
  QUERY_PATTERN: |
    [Paste sample query or describe access pattern]
  CONSTRAINTS:
    - [e.g., "Cannot change partition key"]
    - [e.g., "Budget: <$500/month additional cost"]

    4. Data Modeling¶

    • Description: Create conceptual, logical, and physical data models
    • Input: Business entities, relationships, access patterns
    • Output: ERD diagrams, data dictionaries, modeling documentation
    • Example: P02: Model hierarchical space-tenant-user structure with hash chains

    Input Template: 

    P02 REQUEST: Data Modeling
  ENTITIES:
    - name: [Entity name]
      attributes: [List key attributes]
      relationships: [List relationships to other entities]
  BUSINESS_RULES:
    - [Rule 1, e.g., "User must belong to exactly one tenant"]
    - [Rule 2, e.g., "Documents can be shared across spaces"]
  ACCESS_PATTERNS:
    - [Pattern 1, e.g., "Get all documents in a space"]
    - [Pattern 2, e.g., "Find user's recent activities"]
  SPECIAL_REQUIREMENTS:
    - versioning: [YES | NO]
    - soft_delete: [YES | NO]
    - audit_trail: [YES | NO]
    - encryption: [YES | NO]
  OUTPUT_FORMAT: [ERD | UML | JSON Schema | SQL DDL]

    5. Storage Architecture¶

    • Description: Design storage strategies (hot/cold, caching, archival)
    • Input: Data lifecycle, access frequency, retention policies
    • Output: Storage architecture, tiering strategy, cost optimization
    • Example: P02: Design multi-tier storage for documents and embeddings

    Input Template: 

    P02 REQUEST: Storage Architecture
  DATA_TYPES:
    - type: [e.g., "documents"]
      size: [e.g., "100MB avg"]
      access_frequency: [HOT | WARM | COLD]
      retention: [e.g., "7 years"]
    - type: [e.g., "embeddings"]
      size: [e.g., "1KB avg"]
      access_frequency: [HOT]
      retention: [e.g., "90 days"]
  REQUIREMENTS:
    - availability: [e.g., "99.9%"]
    - durability: [e.g., "99.999999999%"]
    - recovery_time: [e.g., "< 1 hour"]
    - backup_frequency: [e.g., "Daily"]
  COST_CONSTRAINTS:
    - monthly_budget: [e.g., "$1000"]
    - cost_per_gb_target: [e.g., "<$0.10"]
  COMPLIANCE:
    - [e.g., "GDPR compliant"]
    - [e.g., "Geo-replication required"]

    P02 REQUEST:
  TASK: Create migration script from legacy to HPK containers
  CONTEXT: 50K documents in production, zero downtime required
  CONSTRAINTS: Must support rollback, verify data integrity
  DELIVERABLES: Python script, migration guide, rollback procedure

    Example 3: Performance Analysis¶

    P02: Analyze query performance for cross-partition document searches
- Current: 2000ms p95 latency
- Target: <500ms p95 latency
- Provide optimization recommendations

    Service Dependencies¶

    Upstream (P02 depends on): - P01 (System Architect): Architecture guidelines, integration patterns - P04 (Library Manager): Data access libraries, SDKs

    Downstream (Depends on P02): - P06 (RAG Engineer): Vector storage, embedding schemas - P08 (API Designer): Data access patterns, query endpoints - P11 (DevOps): Database provisioning, monitoring

    Service Standards¶

    Data Modeling Standards¶

    • Use Cosmos DB HPK for multi-tenant isolation
    • Implement hash chains for audit integrity
    • Follow semantic versioning for schema changes
    • Document all partition key strategies

    Migration Standards¶

    • Always provide dry-run mode
    • Include rollback procedures
    • Verify data integrity with checksums
    • Create DUA documents for production migrations

    Performance Standards¶

    • Single-partition reads: <50ms p95
    • Cross-partition queries: <500ms p95
    • Batch operations: >100 ops/sec
    • RU efficiency: <10 RU per read

    Common Patterns¶

    Pattern 1: HPK Design¶

    # Hierarchical Partition Key Structure
{
  "partition_keys": [
    "/space_id",     # Level 1: Workspace isolation
    "/tenant_id",    # Level 2: Organization isolation
    "/user_id"       # Level 3: User isolation
  ]
}

    Pattern 2: Hash Chain Integrity¶

    # Audit log with hash chain
{
  "sequence_number": 12345,
  "previous_hash": "sha256:...",
  "current_hash": "sha256:...",
  "action": "document_created"
}

    Pattern 3: Dual-Mode Support¶

    # Support both legacy and HPK containers
def get_container(use_hpk: bool = True):
    return hpk_container if use_hpk else legacy_container

    Schema Design Package¶

    File Structure: 

    deliverables/schema-design-[feature]/

 schema.bicep             # Azure IaC

 SCHEMA-DESIGN.md         # Design rationale & decisions
 migration_guide.md       # Implementation instructions
 tests/
     test_schema.py       # Schema validation tests

    Schema Definition Example (schema.json): 

    {
  "container_name": "documents_hpk",
  "partition_key": {
    "paths": ["/space_id", "/tenant_id", "/user_id"],
    "kind": "MultiHash",
    "version": 2
  },
  "indexes": {
    "compositeIndexes": [
      [
        {"path": "/space_id", "order": "ascending"},
        {"path": "/created_at", "order": "descending"}
      ]
    ]
  },
  "throughput": {
    "type": "autoscale",
    "maxThroughput": 4000
  }
}

    Migration Package¶

    File Structure: 

    deliverables/migration-[feature]/
 migrate_[feature].py         # Main migration script
 rollback_[feature].py        # Rollback procedure
 verify_[feature].py          # Post-migration verification
 MIGRATION-GUIDE.md           # Step-by-step procedures
 migration_config.yaml        # Configuration
 DUA-MIGRATION-[DATE].md      # Archive document
 tests/
     test_migration.py        # Migration unit tests

    Migration Config Example (migration_config.yaml): 

    migration:
  source:
    database: eva-rag
    container: documents
    partition_key: /tenant_id
  target:
    database: eva-rag
    container: documents_hpk
    partition_key: [/space_id, /tenant_id, /user_id]
  settings:
    batch_size: 100
    parallel_workers: 4
    verify_checksums: true
    create_backup: true
  thresholds:
    max_failures: 10
    timeout_seconds: 300

    Work Breakdown Package (Epic/Stories/Tasks)¶

    File Naming Convention: 

    P02-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P02-EPIC-TEST-COVERAGE-20251209-122807.md - P02-P07-TESTING-TASKS-20251209-122807.md - P02-P09-QA-TASKS-20251209-122807.md - P02-P11-DEVOPS-TASKS-20251209-122807.md - P02-SM-COORDINATION-PLAN-20251209-122807.md

    File Structure: 

    docs/work-breakdown/
 P02-EPIC-[feature]-[timestamp].md           # Epic with user stories
 P02-P07-TESTING-TASKS-[timestamp].md        # Testing agent tasks
 P02-P09-QA-TASKS-[timestamp].md             # QA agent tasks
 P02-P11-DEVOPS-TASKS-[timestamp].md         # DevOps agent tasks
 P02-SM-COORDINATION-PLAN-[timestamp].md     # Sprint coordination

    Naming Rules: 1. Prefix: Always start with P02- (generator agent ID) 2. Body: Descriptive name indicating document type and target agent 3. Suffix: Timestamp in format -YYYYMMDD-HHMMSS 4. Purpose: Traceability - track when P02 generated each document

    Quality Checklist¶

    Before marking P02 work complete:

    • Schema follows EVA naming conventions
    • Partition keys optimized for access patterns
    • Migration includes dry-run and rollback
    • Performance targets documented and tested
    • All scripts include error handling
    • Documentation includes examples
    • Integration tests written
    • DUA created for production changes
