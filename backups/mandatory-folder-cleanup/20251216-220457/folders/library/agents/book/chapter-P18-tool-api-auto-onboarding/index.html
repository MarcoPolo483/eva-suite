    Chapter P18: Tool/API Auto-Onboarding¶

    Pattern Code: P18 Short Code: TON Name: Tool/API Auto-Onboarding Type: DevTools Pattern Scope: Integration & Onboarding Maturity: Concept Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    Tool/API Auto-Onboarding (P18-TON) transforms the tedious process of integrating external tools and APIs from days of boilerplate coding into hours of template-driven scaffolding. Instead of manually writing authentication logic, error handling, retry mechanisms, rate limiting, and documentation for every new API, P18 generates production-ready integration code with built-in safety guardrails and compliance requirements.

    Without P18, teams reinvent the wheel for every API integration‚Äîwriting similar auth patterns, handling errors inconsistently, forgetting rate limits, and under-documenting contracts. With P18, integrations follow EVA standards automatically: managed identity auth, circuit breakers, cost budgeting, audit logging (P21), and OpenAPI contracts.

    P18 is especially critical for Government of Canada deployments where every external integration must meet strict security, data residency, and audit requirements. By codifying these requirements into templates, P18 ensures new integrations are compliant by default.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P18-TON serves as the API Integration Scaffolder and Compliance-by-Default Enforcer for the EVA Autonomous Agile Command. It ensures:

    1. Rapid Integration: New APIs onboarded in hours instead of days
    2. Standardized Patterns: All integrations follow EVA auth, error handling, and safety patterns
    3. Built-in Safety: Rate limiting, cost budgeting, circuit breakers, audit logging by default
    4. Automatic Documentation: OpenAPI contracts, usage examples, runbooks generated
    5. Compliance-Ready: Managed identity, Key Vault secrets, data residency enforced

    Mission Statement¶

    "Accelerate API integration while enforcing safety and compliance by default through template-driven scaffolding and automated documentation generation."

    Value Proposition¶

    For Developers: - Integrate new APIs in hours instead of days - Focus on business logic instead of boilerplate (auth, retry, error handling) - Learn best practices through generated code templates

    For Integration Engineers: - Ensure consistency across all API integrations - Enforce security patterns (no secrets in code) - Reduce technical debt from ad-hoc integrations

    For Platform Owners: - Monitor integration costs and rate limits centrally - Audit all API calls via P21 provenance - Enforce data residency and compliance requirements

    For Governance Panels: - Every integration has OpenAPI contract (machine-readable) - Every integration has runbook for failures - Every integration has audit trail (P21)

    üöÄ When to Use P18¶

    Primary Use Cases¶

    1. Scaffold New API Integration¶

    Scenario: You need to integrate Azure Cognitive Services for text analytics in EVA.

    Action: Use P18 to scaffold integration with all safety guardrails.

    2. Validate Existing Integration¶

    Scenario: Legacy integration doesn't follow EVA standards (secrets in code, no rate limiting).

    Action: Use P18 validation to audit integration and flag gaps.

    Example: 

    **Request**: "P18 ‚Äì TON, validate this API integration meets EVA standards"

**P18 Validation**:
- ‚ùå **Auth**: API key hardcoded in code (should use Key Vault)
- ‚ùå **Retry**: No retry logic (should have exponential backoff)
- ‚ùå **Circuit Breaker**: None (cascading failures possible)
- ‚úÖ **Timeout**: Configured (5s)
- ‚ùå **Rate Limiting**: None (could exceed quota)
- ‚ùå **Cost Budgeting**: Not tracked
- ‚ùå **Audit Logging**: None (P21 provenance missing)
- ‚ùå **OpenAPI Contract**: Missing
- ‚ùå **Runbook**: Missing

**Recommendation**: Refactor using P18 scaffolding template

    3. Generate API Documentation¶

    Scenario: New Azure OpenAI integration needs comprehensive documentation.

    Action: Use P18 to auto-generate OpenAPI contract and usage guide.

    P18 Generates: - OpenAPI 3.0 contract (machine-readable) - Usage examples (TypeScript + cURL) - Authentication guide (managed identity setup) - Error codes reference (what each error means + how to fix) - Rate limit documentation (per-endpoint limits) - Troubleshooting guide (common issues + solutions)

    When NOT to Use P18¶

    • Don't use P18 for one-time integrations - Reserve for reusable patterns
    • Don't use P18 to bypass security review - Scaffolding doesn't replace SME review
    • Don't use P18 for complex integrations - Highly custom integrations may need manual implementation

    Template 1: TypeScript API Client¶

    // src/integrations/azure-cognitive-services/client.ts

import { DefaultAzureCredential } from "@azure/identity";
import axios, { AxiosInstance, AxiosError } from "axios";
import CircuitBreaker from "opossum";

export interface SentimentRequest {
  documents: Array<{
    id: string;
    language: string;
    text: string;
  }>;
}

export interface SentimentResponse {
  documents: Array<{
    id: string;
    sentiment: "positive" | "negative" | "neutral" | "mixed";
    confidenceScores: {
      positive: number;
      negative: number;
      neutral: number;
    };
  }>;
  errors: Array<{ id: string; error: string }>;
}

export class AzureCognitiveServicesClient {
  private client: AxiosInstance;
  private circuitBreaker: CircuitBreaker;
  private credential: DefaultAzureCredential;
  private scope = "https://cognitiveservices.azure.com/.default";

  constructor(
    private baseURL: string = "https://canadacentral.api.cognitive.microsoft.com",
    private rateLimitPerMin: number = 100,
    private timeout: number = 5000
  ) {
    this.credential = new DefaultAzureCredential();

    this.client = axios.create({
      baseURL: this.baseURL,
      timeout: this.timeout,
      headers: {
        "Content-Type": "application/json",
      },
    });

    // Circuit breaker: open if >50% error rate for 1 minute
    this.circuitBreaker = new CircuitBreaker(this.makeRequest.bind(this), {
      timeout: this.timeout,
      errorThresholdPercentage: 50,
      resetTimeout: 60000, // 1 minute
    });

    this.circuitBreaker.on("open", () => {
      console.error("[Circuit Breaker] OPEN - API unavailable");
      // Alert SRE (P14-LVO)
    });
  }

  private async getAccessToken(): Promise<string> {
    const tokenResponse = await this.credential.getToken(this.scope);
    return tokenResponse.token;
  }

  private async makeRequest<T>(
    method: string,
    path: string,
    data?: any
  ): Promise<T> {
    const token = await this.getAccessToken();

    try {
      const response = await this.client.request<T>({
        method,
        url: path,
        data,
        headers: {
          Authorization: `Bearer ${token}`,
        },
      });

      // Audit logging (P21-PRV)
      this.logProvenanceFingerprint({
        action: `${method} ${path}`,
        status: "success",
        latency: response.headers["x-ms-request-duration-ms"],
      });

      return response.data;
    } catch (error) {
      // Audit logging (P21-PRV)
      this.logProvenanceFingerprint({
        action: `${method} ${path}`,
        status: "error",
        error: (error as AxiosError).message,
      });

      throw error;
    }
  }

  private logProvenanceFingerprint(data: any) {
    // TODO: Integrate with P21 provenance service
    console.log("[Provenance]", JSON.stringify(data));
  }

  public async analyzeSentiment(
    request: SentimentRequest
  ): Promise<SentimentResponse> {
    // Rate limiting check (simple in-memory, production would use Redis)
    // TODO: Implement rate limiting

    // Retry with exponential backoff
    const maxRetries = 3;
    let attempt = 0;

    while (attempt < maxRetries) {
      try {
        return await this.circuitBreaker.fire(
          "POST",
          "/text/analytics/v3.1/sentiment",
          request
        );
      } catch (error) {
        attempt++;
        if (attempt >= maxRetries) throw error;

        // Exponential backoff: 1s, 2s, 4s
        const backoff = Math.pow(2, attempt) * 1000;
        await new Promise(resolve => setTimeout(resolve, backoff));
      }
    }

    throw new Error("Max retries exceeded");
  }
}

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C1 (Human-on-the-Loop)¶

    P18 Itself: C1 - Generates scaffolding automatically - Human reviews generated code before merge - Human approves new integrations

    Integrated APIs: Varies by integration - C1: Most external APIs (human monitors, can intervene) - C2: Internal Azure services (human reviews periodically)

    Safety Rules¶

    Rule 1: No Secrets in Code¶

    Risk: API keys, connection strings, certificates committed to git

    Mitigation: - ‚úÖ All auth uses managed identity (preferred) - ‚úÖ Secrets stored in Azure Key Vault (if managed identity not possible) - ‚úÖ P11-SEC security scan blocks merge if secrets detected - ‚úÖ Template enforces Key Vault integration

    Rule 2: Rate Limiting Enforced¶

    Risk: API quota exhausted, service degradation

    Mitigation: - ‚úÖ Every integration has rate limit configured - ‚úÖ Template includes rate limiting logic (Redis-backed) - ‚úÖ Alert when approaching rate limit (80% threshold)

    Rule 3: Cost Budgeting Tracked¶

    Risk: Unexpected cost overrun from API usage

    Mitigation: - ‚úÖ Every integration has cost budget ($X/month) - ‚úÖ P10-MET tracks actual costs via P21 provenance - ‚úÖ Alert when cost exceeds budget by 20% - ‚úÖ Circuit breaker can be configured to disable at budget threshold

    Rule 4: Audit Trail Required¶

    Risk: No record of API calls for compliance/troubleshooting

    Mitigation: - ‚úÖ Every API call logged via P21 provenance - ‚úÖ Template includes audit logging by default - ‚úÖ 7-year retention per GC requirements

    ‚úÖ End-of-Chapter Checklist¶

    For Integration Authors¶

    • I have used P18 scaffolding template
    • Auth uses managed identity or Key Vault (no secrets in code)
    • Retry logic with exponential backoff implemented
    • Circuit breaker configured (>50% error rate)
    • Timeout configured (<5s for APIs, <30s for batch)
    • Rate limiting enforced
    • Cost budget tracked
    • Audit logging enabled (P21)
    • OpenAPI contract exists
    • Runbook exists for failures

    For Reviewers¶

    • Security scan passed (P11-SEC)
    • No secrets in code
    • Auth pattern correct (managed identity preferred)
    • Error handling comprehensive (retry, circuit breaker, timeout)
    • Tests exist (unit + integration)
    • OpenAPI contract accurate
    • Runbook actionable

    For Operations¶

    • Rate limit appropriate for expected traffic
    • Cost budget reasonable
    • Alerting configured (rate limit, cost, errors)
    • Runbook tested (dry-run in non-prod)
