    Chapter P03: Scrum Agent¶

    Pattern Code: P03 Short Code: SCR Name: Scrum Agent Type: DevTools Pattern Scope: Agile Ceremonies & Sprint Management Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Scrum Agent (P03-SCR) is the heartbeat of the EVA Agile Command, automating the repetitive tasks of sprint management while preserving human oversight for decisions and priorities. It transforms daily stand-ups, sprint boards, and ceremony artifacts from manual chores into auto-generated, consistent, and traceable documents.

    Without P03, sprint management would consume hours of manual effort‚Äîupdating boards, generating summaries, tracking velocity. With P03, the team stays focused on delivery while the agent handles the ceremony mechanics.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P03-SCR serves as the Scrum Assistant and Sprint Automation Engine for the EVA Autonomous Agile Command. It ensures:

    1. Consistency: Sprint artifacts follow standardized formats
    2. Transparency: Real-time visibility into sprint progress
    3. Efficiency: Automated daily summaries and board updates
    4. Traceability: Complete audit trail of sprint execution

    Mission Statement¶

    "Automate sprint mechanics to free teams for delivery, while maintaining transparent, consistent, and traceable agile ceremonies."

    Value Proposition¶

    For Scrum Masters: - Eliminate manual stand-up note-taking - Auto-sync sprint boards without manual drag-and-drop - Generate velocity metrics and burn charts automatically

    For Developers: - Clear visibility into sprint progress - Automatic status tracking (no manual updates) - Focus on coding, not ceremony overhead

    For Product Owners: - Real-time sprint health dashboards - Early warning on blockers and risks - Data-driven sprint planning

    üõ†Ô∏è Responsibilities and Capabilities¶

    Core Responsibilities¶

    1. Daily Stand-Up Automation
    2. Generate daily progress summaries
    3. Extract yesterday/today/blockers from GitHub activity

    4. Publish stand-up notes to sprint docs

    5. Sprint Board Management

    6. Sync GitHub issue states to project board
    7. Update column assignments (Backlog ‚Üí In Progress ‚Üí Done)

    8. Flag stale issues (>72 hours without update)

    9. Velocity Tracking

    10. Calculate burn-down charts
    11. Track completed vs remaining story points

    12. Forecast sprint completion based on velocity

    13. Runbook Maintenance

    14. Auto-update sprint runbooks with progress
    15. Generate risk registers from issue labels

    16. Maintain ceremony artifacts (retros, reviews, planning notes)

    17. Ceremony Support

    18. Pre-generate sprint planning agendas
    19. Auto-populate retrospective templates
    20. Create sprint review presentations

    Capabilities Matrix¶

    Capability Description Autonomy Validation
    Stand-Up Summaries Daily progress reports from GitHub activity C1 Human review
    Board Sync Sync issue states to project board C1 Auto-validated
    Velocity Metrics Calculate burn-down, remaining points C1 Auto-validated
    Runbook Updates Update sprint runbooks with progress C1 Human review

    2. Sprint Board Sync¶

    Automation Workflow: .github/workflows/sprint-board-sync.yml

    Sync Actions: - Pull GitHub issue states from sprint labels - Update project board columns - Move issues: Backlog ‚Üí In Progress ‚Üí In Review ‚Üí Done - Flag stale issues (>72 hours without comment or status change)

    Sync Frequency: Every 6 hours (04:00, 10:00, 16:00, 22:00 UTC)

    Board Structure:

    Column Criteria Auto-Actions
    Backlog No assignee, sprint-N label None
    Ready Has assignee, sprint-N, no PR None
    In Progress Has PR or in-progress label Auto-move from Ready
    In Review Has PR open, needs review Auto-move from In Progress
    Done Closed issue Auto-move from In Review

    Stale Issue Detection: - Issues in "In Progress" >72 hours without update ‚Üí Flag with stale label - Issues in "In Review" >48 hours without review ‚Üí Comment: "@reviewers please review" - Issues in "Backlog" >2 sprints ‚Üí Comment: "Is this still needed?"

    ü§ù Interactions with Other Patterns¶

    P02 (Requirements Engine)¶

    Collaboration: Requirements ‚Üí Sprint Backlog - P02 creates structured requirements - P03 pulls sprint-ready requirements into active sprint - Together: Ensure smooth backlog ‚Üí execution flow

    Handoff: - P02 ‚Üí P03: Sprint-ready stories with clear acceptance criteria - P03 ‚Üí P02: Feedback on requirement clarity, velocity impact

    P06 (Review Agent)¶

    Collaboration: Code Reviews ‚Üí Sprint Progress - P06 reviews PRs and provides feedback - P03 tracks PR review status on sprint board - Together: Ensure code quality gates don't block sprint completion

    Handoff: - P06 ‚Üí P03: PR review status (approved, changes requested) - P03 ‚Üí P06: Priority PRs needing urgent review

    P10 (Metrics Agent)¶

    Collaboration: Sprint Metrics ‚Üí Insights - P03 generates raw sprint metrics (velocity, burn rate) - P10 analyzes trends and provides improvement recommendations - Together: Data-driven sprint retrospectives

    Handoff: - P03 ‚Üí P10: Sprint metrics data (velocity, blockers, completion rate) - P10 ‚Üí P03: Insights (e.g., "Velocity declining, consider smaller stories")

    P15 (Dev Master Orchestrator)¶

    Collaboration: Sprint Execution ‚Üí Multi-Agent Coordination - P03 manages sprint cadence - P15 orchestrates complex multi-pattern workflows - Together: Sprint stories get full agent support

    Handoff: - P03 ‚Üí P15: Sprint priorities and active stories - P15 ‚Üí P03: Workflow completion status, agent assignments

    üö® Common Issues and Troubleshooting¶

    Issue 1: Incorrect Board Column Assignments¶

    Symptoms: - Issues in wrong column (e.g., Done issue in In Progress) - Issues not moving despite PR merge

    Resolution: - Check issue labels (sprint-N, in-progress, etc.) - Verify GitHub API permissions (workflow token scope) - Manually move issue if automation fails, log for P03 improvement

    Issue 2: Missing Stand-Up Data¶

    Symptoms: - Stand-up summary incomplete - No yesterday's progress listed

    Resolution: - Check GitHub Actions workflow logs - Verify issues have activity timestamps - Re-run stand-up generation script manually if needed

    Issue 3: Stale Issue False Positives¶

    Symptoms: - Active issue flagged as stale - Issue with recent discussion flagged

    Resolution: - Adjust stale threshold (currently 72 hours) - Exclude specific issues with "active" label - Remove "stale" label manually if incorrect
