    Chapter P17: Swarm PR Review¶

    Pattern Code: P17 Short Code: SWR Name: Swarm PR Review Type: DevTools Pattern Scope: Code Review (Multi-Perspective) Maturity: Alpha Autonomy Level: C0 (Human-in-the-loop)

    üìñ Chapter Overview¶

    Swarm PR Review (P17-SWR) transforms code review from a single-person bottleneck into a multi-perspective analysis powered by the collective intelligence of all EVA DevTools patterns. Instead of waiting for a human reviewer to manually check security, testing, documentation, accessibility, and operations concerns, P17 orchestrates a "swarm" of specialist agents to provide comprehensive feedback in minutes.

    Without P17, PR reviews are sequential, incomplete, and slow‚Äîone reviewer can't be an expert in security, accessibility, performance, and operations simultaneously. With P17, every PR gets reviewed through multiple lenses (P06-REV, P07-TST, P11-SEC, P12-UXA, P14-LVO, etc.), surfacing issues early and reducing back-and-forth cycles.

    P17 is not a replacement for human reviewers‚Äîit's an augmentation tool that provides structured, pattern-tagged feedback so humans can focus on architectural decisions, business logic validation, and context-specific judgment.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P17-SWR serves as the Multi-Perspective Code Reviewer and Pattern-Tagged Feedback Aggregator for the EVA Autonomous Agile Command. It ensures:

    1. Comprehensive Review: PRs analyzed through multiple specialist lenses (security, testing, UX, ops)
    2. Pattern-Tagged Findings: Each issue labeled with relevant patterns (e.g., "P07-TST: Missing test coverage")
    3. Merge-Readiness Assessment: Advisory judgment (Looks good / OK with minor fixes / Needs work)
    4. Awareness Integration: Every swarm review includes P16 awareness block (assumptions, risks, confidence)
    5. Prioritized Feedback: Top 3-7 issues, not exhaustive lists (signal over noise)

    Mission Statement¶

    "Accelerate PR reviews by orchestrating specialist agents to provide comprehensive, pattern-tagged feedback that helps humans make informed merge decisions."

    Value Proposition¶

    For Developers: - Get comprehensive feedback in minutes instead of hours/days - Understand which patterns are relevant to fix specific issues - Learn from pattern-tagged feedback (educational)

    For Human Reviewers: - Focus on high-level architecture and business logic - Don't need to be expert in all domains (security, accessibility, ops) - Use swarm review as first-pass filter

    For Product Owners: - Reduce PR cycle time (faster feature delivery) - Improve code quality through multi-perspective review - Reduce production issues from overlooked concerns

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Orchestrate Specialist Lenses
    2. Invoke relevant patterns based on PR content (e.g., P12-UXA if UI changes)
    3. Aggregate findings from multiple patterns

    4. Deduplicate and prioritize findings

    5. Generate Pattern-Tagged Findings

    6. Tag each finding with pattern code (P06, P07, P11, etc.)
    7. Classify severity (BLOCKER / IMPORTANT / NICE-TO-HAVE)

    8. Provide actionable suggestions with code examples

    9. Provide Merge-Readiness Assessment

    10. Advisory status: Looks good / OK with minor fixes / Needs work
    11. Non-binding (human has final authority)

    12. Based on severity and count of findings

    13. Include Awareness Block

    14. Every swarm review includes P16 awareness block
    15. Document confidence level (based on patterns invoked)
    16. Surface assumptions and risks

    Supporting Responsibilities¶

    1. Lightweight Review Mode
    2. Quick spot-check for small changes
    3. 1-3 key findings instead of full analysis

    4. Faster feedback for minor PRs

    5. Swarm Review Checklist

    6. Provide checklist for human reviewers to apply swarm-style review manually

    7. Template for teams without automated P17

    8. Educational Feedback

    9. Link findings to pattern documentation
    10. Explain why issue matters (not just what's wrong)
    11. Suggest learning resources

    üöÄ When to Use P17¶

    Primary Use Cases¶

    1. Full Swarm Review for PR¶

    Scenario: You've opened a PR with code changes and want comprehensive multi-perspective feedback.

    Action: Request swarm review from P17.

    Example: 

    **PR**: #123 - Add key rotation to eva-auth
**Repo**: eva-auth
**Pod**: POD-F (Foundation)
**Phase**: 2 (Build / Code / Test)

**Request**: "P17 ‚Äì SWR, perform a swarm review for this PR."

**P17 Output**: (See full example in Templates section)
- **Summary**: Code quality good, needs test coverage + runbook + alerting
- **Merge Readiness**: ‚úÖ OK with minor fixes
- **Findings**: 3 IMPORTANT, 2 NICE-TO-HAVE (no BLOCKERS)
- **Patterns Invoked**: P06, P07, P09, P10, P11, P14, P16, P19, P21

    2. Lightweight Swarm Review (Spot Check)¶

    Scenario: Small code change (e.g., config update, rate limit increase) needs quick sanity check.

    Action: Request lightweight swarm review.

    Example: 

    **Change**: Update APIM rate limit from 100 to 200 req/min

**Request**: "P17 ‚Äì SWR, lightweight swarm review for spot check."

**P17 Output**:
# Lightweight Swarm Review (P17 ‚Äì SWR)

**Change**: Update APIM rate limit from 100 to 200 req/min

**Quick Check**:
- ‚úÖ **[P11-SEC]** Rate limit still protects against abuse (200 is reasonable)
- ‚ö†Ô∏è **[P10-MET]** Cost impact: ~2x OpenAI token usage if traffic scales up (~$1,000 ‚Üí ~$2,000/month)
- ‚ö†Ô∏è **[P14-LVO]** Monitor for 48 hours after deployment (watch cost + latency)

**Merge Readiness**: ‚úÖ Looks good (monitor cost post-deployment)

    3. Swarm Review Checklist for Manual Review¶

    Scenario: Team doesn't have automated P17 yet, but wants to apply swarm-style review manually.

    Action: Request swarm review checklist.

    Example: (See full checklist in Templates section) 

    **Request**: "P17 ‚Äì SWR, provide swarm review checklist template."

**P17 Output**: Checklist covering P06-REV, P07-TST, P09-RUN, P10-MET, P11-SEC, P12-UXA, P14-LVO, P16-AWR, P19-ACS, P21-PRV

    When NOT to Use P17¶

    • Don't use P17 for trivial changes - Config typos, README updates don't need swarm review
    • Don't use P17 as gate without human review - P17 is advisory, not authoritative
    • Don't use P17 to replace architecture review - High-level design decisions still need human judgment

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P17-SWR (Swarm PR Review), the Multi-Perspective Code Reviewer for EVA DevTools.

**Your Capabilities**:
1. Orchestrate specialist patterns (P06-REV, P07-TST, P11-SEC, P12-UXA, P14-LVO, etc.)
2. Generate pattern-tagged findings with severity (BLOCKER / IMPORTANT / NICE-TO-HAVE)
3. Provide merge-readiness assessment (Looks good / OK with minor fixes / Needs work)
4. Include P16 awareness block (assumptions, risks, confidence)
5. Prioritize feedback (top 3-7 issues, not exhaustive lists)

**Your Limitations**:
- You do NOT merge PRs (humans have final authority)
- You do NOT guarantee completeness (edge cases may be missed)
- You do NOT replace architecture review (high-level design needs human judgment)

**Autonomy Level**: C0 (Human-in-the-loop)
- Provide feedback automatically
- Humans make merge decisions
- Non-binding recommendations

**Output Format**:
- Structured markdown with severity-coded findings
- Pattern tags for each finding (e.g., "[P07-TST]")
- Code examples for suggestions
- Merge-readiness assessment at top
- P16 awareness block at bottom

**Safety Rules**:
- Flag security issues (P11-SEC) as BLOCKER
- Flag accessibility violations (P12-UXA) as IMPORTANT for user-facing changes
- Flag missing tests (P07-TST) as IMPORTANT if coverage <80%
- Provide actionable suggestions (not just criticism)

    ## Swarm Review Checklist (P17 ‚Äì SWR)

### P06-REV: Code Review Basics
- [ ] Code follows repo conventions (style, naming, structure)
- [ ] Error handling present (try/catch, logging)
- [ ] No obvious bugs or code smells
- [ ] Functions/methods have clear single responsibility

### P07-TST: Testing
- [ ] Test coverage meets 80% threshold
- [ ] Tests cover happy path + failure cases
- [ ] Tests are maintainable (not brittle)
- [ ] Edge cases tested (null, empty, large data)

### P09-RUN: Operations Impact
- [ ] Runbook updated if behavior changes
- [ ] Deployment rollback plan exists
- [ ] Incident response procedures updated if needed

### P10-MET: Metrics & Telemetry
- [ ] Key actions logged with structured fields
- [ ] Alerts configured for failures
- [ ] Cost impact estimated (if applicable)
- [ ] Performance metrics tracked (latency, throughput)

### P11-SEC: Security & Compliance
- [ ] No secrets in code (use Key Vault)
- [ ] RBAC enforced (principle of least privilege)
- [ ] Data residency respected (Canada-only)
- [ ] Input validation present
- [ ] No SQL injection / XSS vulnerabilities

### P12-UXA: UX & Accessibility (if UI changes)

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C0 (Human-in-the-Loop)¶

    P17 Itself: C0 - Provides feedback automatically - Humans make all merge decisions - Non-binding recommendations

    Why C0? - Code review requires human judgment (context, business logic, architecture) - P17 augments humans, doesn't replace them - Merge decisions have high impact (production deployment)

    Safety Rules¶

    Rule 1: Findings Are Advisory, Not Enforcement¶

    Risk: Developers may ignore findings, leading to production issues

    Mitigation: - ‚úÖ Require human reviewer to acknowledge swarm review - ‚úÖ Track whether findings were addressed (via P21 provenance) - ‚úÖ Retrospective analysis: Did ignored findings cause issues? - ‚úÖ Educate team on value of swarm review

    Rule 2: Severity Must Be Accurate¶

    Risk: Crying wolf (too many BLOCKERS) or missing critical issues (severity too low)

    Mitigation: - ‚úÖ BLOCKER: Security vulnerabilities, data residency violations, breaking changes without migration - ‚úÖ IMPORTANT: Missing tests (<80%), no runbook, accessibility violations, no alerting - ‚úÖ NICE-TO-HAVE: Code style, docs could be clearer, metrics could be richer

    Rule 3: Swarm Review Doesn't Replace Architecture Review¶

    Risk: Approving PRs with swarm review but missing high-level design issues

    Mitigation: - ‚úÖ P17 focuses on code-level concerns (security, testing, ops) - ‚úÖ Architectural decisions still need human review (P02-REQ, P15-DVM) - ‚úÖ Cross-repo impacts need human coordination (P15 impact analysis)

    Rule 4: Human Reviewer Has Final Authority¶

    Risk: Confusion about who decides (P17 vs human reviewer)

    Mitigation: - ‚úÖ P17 merge readiness is advisory, not authoritative - ‚úÖ Human reviewer can override P17 (with rationale documented) - ‚úÖ Merge button controlled by humans, not P17

    ‚úÖ End-of-Chapter Checklist¶

    For PR Authors¶

    • I have requested swarm review for my PR
    • I understand swarm review findings (pattern tags, severity)
    • I have addressed BLOCKER findings before requesting human review
    • I have decided on IMPORTANT findings (address or defer with rationale)
    • I have documented why any findings were deferred

    For Human Reviewers¶

    • I have reviewed swarm review findings
    • I have validated that BLOCKER findings are addressed
    • I have made judgment call on IMPORTANT findings
    • I have considered architecture/business logic (not just swarm review)
    • I have documented my merge decision rationale

    For Teams¶

    • Swarm review is part of PR workflow (automated or checklist)
    • Severity levels are calibrated (not too many BLOCKERS)
    • Findings are tracked (addressed vs deferred)
    • Retrospective analysis: Did swarm review catch real issues?

    üéì Key Takeaways¶

    1. Swarm review is augmentation, not replacement - Humans still decide
    2. Pattern tags educate - Findings linked to patterns help learning
    3. Severity calibration matters - Too many BLOCKERS reduce credibility
    4. Top 3-7 findings - Signal over noise (not exhaustive lists)
    5. Awareness is mandatory - Every swarm review includes P16 block
    6. Multi-perspective is powerful - Security + Testing + UX + Ops = comprehensive
    7. Merge readiness is advisory - Non-binding recommendation
    8. Track findings - Retrospective analysis improves swarm review
