    Chapter P02: Requirements Engine¶

    Pattern Code: P02 Short Code: REQ Name: Requirements Engine Type: DevTools Pattern Scope: Requirements Management & Refinement Maturity: Stable Autonomy Level: C0-C1 (Human-in/on-the-loop)

    üìñ Chapter Overview¶

    The Requirements Engine (P02-REQ) is the entry point for all work in the EVA ecosystem. It transforms raw ideas, feature requests, and problem statements into structured, executable requirements that flow seamlessly into sprint execution. P02 bridges the gap between "what we want to build" and "what the team can execute."

    Without P02, requirements would be scattered across emails, chat messages, and undocumented conversations. With P02, every requirement follows a consistent structure, has clear acceptance criteria, and is traceable from concept to production.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P02-REQ serves as the Requirements Architect and Intake Manager for the EVA Autonomous Agile Command. It ensures:

    1. Standardization: All requirements follow the same structured schema
    2. Completeness: Requirements include context, scope, and success criteria
    3. Traceability: From initial idea through implementation to production
    4. Executability: Clear enough for development teams to act on

    Mission Statement¶

    "Transform ideas into executable requirements through standardized intake, structured refinement, and automated template generation."

    Value Proposition¶

    For Product Owners: - Capture vision and business value clearly - Track requirements from idea to delivery - Ensure alignment between strategy and execution

    For Developers: - Understand what to build and why - Have clear acceptance criteria and QA definitions - Reduce back-and-forth clarification requests

    For Stakeholders: - Visibility into what's being built - Confidence that requirements address real needs - Clear prioritization and sequencing

    üõ†Ô∏è Responsibilities and Capabilities¶

    Core Responsibilities¶

    1. Intake Management
    2. Capture new requirements via standardized schema
    3. Validate completeness of requirement submissions

    4. Route requirements to appropriate PODs

    5. Requirement Refinement

    6. Transform raw ideas into structured CDDs
    7. Break down epics into stories and tasks

    8. Ensure testability and clarity

    9. Template Generation

    10. Auto-generate requirement documents from templates
    11. Populate common fields based on requirement type

    12. Maintain template library for features, bugs, epics

    13. Traceability

    14. Link requirements to implementation artifacts
    15. Track requirement status through lifecycle

    16. Maintain dependency mappings

    17. Quality Assurance

    18. Validate acceptance criteria are testable
    19. Ensure QA definitions are complete
    20. Flag incomplete or ambiguous requirements

    Capabilities Matrix¶

    Capability Description Autonomy Validation
    Schema Validation Check requirement structure completeness C1 Auto-validated
    Template Generation Create requirement docs from templates C1 Human review
    CDD Creation Transform ideas into structured docs C0 Human approval
    Epic Breakdown Decompose epics into stories/tasks C0 Human approval
    Dependency Mapping Identify and document dependencies C1 Human review
    Acceptance Criteria Draft testable acceptance criteria C0 Human approval

    üìã Intake Schema and Structure¶

    Requirement Input Schema¶

    requirement:
  id: REQ-YYYY-MM-DD-NNN
  title: Brief requirement title
  type: feature | bug | epic | story | task
  priority: critical | high | medium | low

  # Context
  problem_statement: What problem are we solving?
  user_story: As a [role], I want [capability] so that [benefit]
  business_value: Why does this matter?

  # Scope
  in_scope:
    - Item 1
    - Item 2
  out_of_scope:
    - Item 1
    - Item 2

  # Success criteria
  acceptance_criteria:
    - Criterion 1 (testable)
    - Criterion 2 (testable)

  qa_definition:
    - Test approach
    - Success metrics
    - Quality gates

  # Dependencies
  depends_on:
    - REQ-ID or TASK-ID
  blocks:
    - REQ-ID or TASK-ID

  # Metadata
  owner: Pod or person
  sprint: Sprint number
  labels:
    - label1
    - label2

    Schema Validation Rules¶

    Required Fields: - id, title, type, priority - problem_statement, business_value - At least one acceptance_criteria - owner (POD assignment)

    Optional Fields: - user_story (recommended for features) - depends_on, blocks (if applicable) - sprint (assigned during planning)

    Validation Checks: - Acceptance criteria must be testable (measurable outcomes) - In-scope items must not overlap with out-of-scope - Dependencies must reference valid requirement IDs - Priority must align with business value (critical = high value)

    üîÑ Requirement Lifecycle¶

    1. Intake ‚Üí Triage (Phase 0)¶

    Inputs: - Raw idea, feature request, or problem statement - Source: Marco (PO), stakeholder, developer, incident

    P02 Actions: - Validate against intake schema - Create initial requirement document - Assign requirement ID (REQ-YYYY-MM-DD-NNN) - Route to appropriate POD

    Outputs: - Structured requirement document - POD assignment - Priority classification

    2. Refinement ‚Üí Elaboration (Phase 1)¶

    Inputs: - Triaged requirement - POD feedback and technical assessment

    P02 Actions: - Collaborate with P03 (Scrum Agent) to refine scope - Work with SMEs to validate business value - Decompose epics into stories - Draft acceptance criteria and QA definitions

    Outputs: - Refined requirement with complete acceptance criteria - Dependency mapping - Effort estimate (T-shirt sizing or story points)

    3. Backlog ‚Üí Sprint Planning (Phase 2)¶

    Inputs: - Refined requirements ready for execution

    P02 Actions: - Validate requirements are sprint-ready - Ensure all required fields populated - Confirm testability of acceptance criteria - Flag any blockers or missing dependencies

    Outputs: - Sprint-ready requirements - Clear task breakdown - Assigned ownership

    4. Execution ‚Üí Validation (Phases 2-6)¶

    Inputs: - In-progress or completed work

    P02 Actions: - Track requirement status - Validate acceptance criteria met - Document any scope changes - Update traceability links

    Outputs: - Requirement status updates - Traceability artifacts - Completion confirmation

    ü§ù Interactions with Other Patterns¶

    P03 (Scrum Agent)¶

    Collaboration: Requirements ‚Üí Sprint Execution - P02 creates structured requirements - P03 manages sprint backlog and ceremonies - Together: Ensure requirements flow smoothly into sprints

    Handoff: - P02 ‚Üí P03: Refined, sprint-ready requirements - P03 ‚Üí P02: Feedback on requirement clarity, scope adjustments

    P05 (Scaffolder)¶

    Collaboration: Requirements ‚Üí Implementation - P02 provides structured requirements - P05 scaffolds code structure based on requirements - Together: Accelerate from idea to working code

    Handoff: - P02 ‚Üí P05: Feature requirements with clear scope - P05 ‚Üí P02: Implementation structure, technical questions

    P07 (Testing Agent)¶

    Collaboration: Requirements ‚Üí Test Cases - P02 defines acceptance criteria - P07 generates test cases from criteria - Together: Ensure testability from the start

    Handoff: - P02 ‚Üí P07: Acceptance criteria, QA definitions - P07 ‚Üí P02: Test coverage gaps, ambiguous criteria

    P15 (Dev Master Orchestrator)¶

    Collaboration: Requirements ‚Üí Multi-Agent Workflows - P02 captures initial requirement - P15 orchestrates multi-pattern execution - Together: Complex requirements get full agent support

    Handoff: - P02 ‚Üí P15: Complex requirement needing orchestration - P15 ‚Üí P02: Workflow plan, agent assignments

    üé≠ Sample Prompts and Workflows¶

    Prompt 1: Create Feature Requirement¶

    User Input: 

    I need bilingual (EN/FR) support for all UI components in EVA Sovereign UI.

    P02 System Prompt: 

    You are P02 (Requirements Engine). Transform this feature request into a structured requirement using the feature template. Include:
- Clear problem statement (why bilingual support matters for GC)
- User story from end-user perspective
- Business value (compliance, accessibility, inclusivity)
- In-scope: UI components, language switcher, i18n service
- Out-of-scope: Backend API translations (separate requirement)
- Acceptance criteria (all UI text available in both languages, language switcher functional, etc.)
- QA definition (manual testing, automated i18n checks)

Output:
- Requirement ID: REQ-2025-12-12-001
- Title: Bilingual UI Support (EN/FR) for EVA Sovereign UI
- Type: feature
- Priority: high (GC compliance requirement)
- Owner: POD-X

    P02 Output: 

    # Feature: Bilingual UI Support (EN/FR) for EVA Sovereign UI

**Requirement ID**: REQ-2025-12-12-001
**Type**: Feature
**Priority**: High
**Owner**: POD-X

## Problem Statement
Government of Canada applications must provide bilingual (English/French) interfaces per Official Languages Act. Currently, EVA Sovereign UI components are English-only, blocking GC deployment.

## User Story
As a bilingual GC employee, I want to switch between English and French in the EVA UI so that I can work in my preferred official language.

## Business Value
- **Compliance**: Satisfies Official Languages Act requirements
- **Accessibility**: Enables all GC employees to use EVA effectively
- **Market Fit**: Unlocks GC-wide deployment (critical for EVA adoption)

## Scope

### In Scope
- All UI components translated (EN/FR)
- Language switcher in header
- i18n service for runtime language switching
- Currency, date, number formatting per locale

### Out of Scope
- Backend API response translations (separate REQ)
- Machine translation of user-generated content
- Languages beyond EN/FR

## Acceptance Criteria
1. All UI text available in both EN and FR
2. Language switcher visible and functional in header
3. Selected language persists across sessions
4. Date/time/currency formatted per selected locale
5. No hardcoded English strings in components
6. WCAG 2.1 AA compliance maintained in both languages

## QA Definition
- **Test Approach**: Manual testing in both languages, automated i18n coverage checks
- **Success Metrics**: 100% UI text coverage, language switch <200ms
- **Quality Gates**: i18n linter passes, bilingual QA sign-off

## Dependencies
- Depends on: GC Design System i18n patterns
- Blocks: REQ-2025-12-12-002 (Bilingual Content Management)

## Metadata
- **Owner**: POD-X (Experience & UI)
- **Priority**: High
- **Sprint**: Sprint 8
- **Labels**: `i18n`, `bilingual`, `pod-x`, `gc-compliance`

    Prompt 2: Break Down Epic¶

    User Input: 

    Epic: EVA Data Ingestion Service - Automate document ingestion for RAG pipelines

    P02 System Prompt: 

    You are P02 (Requirements Engine). Break down this epic into stories and tasks using the epic template. Include:
- Vision (automated, scalable ingestion for all EVA domains)
- Problem statement (manual ingestion is slow, error-prone)
- Success metrics (ingestion time, error rate, throughput)
- Breakdown strategy (Phase 1: Core ingestion, Phase 2: Domain-specific, Phase 3: Automation)
- Dependencies (Azure Storage, Cosmos DB, Search service)
- Risks (data quality, PII handling, scale)

Output structured epic with 3 phases, 8-10 stories total.

    P02 Output: 

    # Epic: EVA Data Ingestion Service

**Epic ID**: EPIC-2025-12-001
**Owner**: POD-F, POD-S (Solution Pods)
**Priority**: Critical
**Timeline**: Sprint 8-12 (5 sprints)

## Vision
Provide automated, scalable, and secure document ingestion for all EVA RAG pipelines, enabling domain-specific assistants to ingest and process documents with minimal manual effort.

## Problem Statement
Current ingestion is manual, slow, and error-prone. Domain teams spend hours preparing documents for ingestion, and there's no standardized process for chunking, metadata extraction, or quality validation. This blocks rapid deployment of new domain assistants.

## Business Value
- **Efficiency**: Reduce ingestion time from hours to minutes
- **Scalability**: Support multiple domains concurrently
- **Quality**: Automated validation and metadata extraction
- **Compliance**: Built-in PII detection and Protected B handling

## Success Metrics
- Ingestion time: <5 minutes for 1000-page document
- Error rate: <1% failed ingestions
- Throughput: 10+ concurrent ingestion jobs
- User satisfaction: 90%+ (measured via surveys)

## Breakdown Strategy

### Phase 1: Core Ingestion (Sprint 8-9)
- **Story 1**: Azure Blob Storage integration for document upload
- **Story 2**: Automatic file format detection (PDF, DOCX, XLSX, etc.)
- **Story 3**: Basic chunking pipeline (fixed-size chunks)
- **Story 4**: Vector embedding generation (Azure OpenAI)

### Phase 2: Domain-Specific Features (Sprint 10-11)
- **Story 5**: Metadata extraction (title, author, date, classification)
- **Story 6**: Domain-specific chunking strategies (semantic, paragraph-based)
- **Story 7**: PII detection and redaction
- **Story 8**: Quality validation (empty chunks, duplicate detection)

### Phase 3: Automation & Monitoring (Sprint 12)
- **Story 9**: Scheduled ingestion (daily, weekly, on-demand)
- **Story 10**: Ingestion dashboard (status, metrics, alerts)
- **Story 11**: Retry logic and error handling
- **Story 12**: Provenance tracking (P21 integration)

## Dependencies
- **Azure Services**: Blob Storage, Cosmos DB, Azure OpenAI, Azure Search
- **EVA Components**: eva-rag, eva-core, eva-api
- **Patterns**: P05 (Scaffolder), P11 (Security), P21 (Provenance)

## Risks
1. **Data Quality**: Poor document quality ‚Üí poor RAG results
   - *Mitigation*: Automated quality gates, validation pipeline
2. **PII Handling**: Accidental PII exposure
   - *Mitigation*: P11 security review, automated PII detection
3. **Scale**: High document volume overwhelms system
   - *Mitigation*: Load testing, autoscaling, queue-based processing

## Timeline
- **Phase 1 Start**: Sprint 8 (Week 1)
- **Phase 1 Complete**: Sprint 9 (Week 4)
- **Phase 2 Complete**: Sprint 11 (Week 8)
- **Phase 3 Complete**: Sprint 12 (Week 10)
- **Milestones**:
  - MVP (Phase 1): Sprint 9
  - Beta (Phase 2): Sprint 11
  - GA (Phase 3): Sprint 12

    Completeness Check¶

    Priority Yes One of: critical, high, medium, low

    Quality Metrics¶

    • Clarity: Can developer understand what to build without asking?
    • Testability: Can QA validate against acceptance criteria?
    • Completeness: Are all required fields populated?
    • Traceability: Can we link back to original request and forward to implementation?

    Success Metrics¶

    • Intake Speed: <30 minutes from raw idea to structured requirement
    • Refinement Accuracy: <10% requirements require major rework
    • Sprint Readiness: 90%+ requirements have complete acceptance criteria
    • Developer Satisfaction: 4.5/5 on "Requirements are clear and actionable"

    üö® Common Issues and Troubleshooting¶

    Issue 1: Incomplete Requirements¶

    Symptoms: - Missing acceptance criteria or QA definitions - Vague problem statements - No business value articulated

    Resolution: - Use P02 template validation - Schedule refinement session with SME - Escalate to Product Owner for prioritization

    Issue 2: Ambiguous Acceptance Criteria¶

    Symptoms: - Acceptance criteria not testable - Subjective language ("looks good", "fast enough") - No measurable success metrics

    Resolution: - Collaborate with P07 (Testing Agent) to make criteria measurable - Replace subjective terms with specific metrics (e.g., "<200ms response time") - Add QA validation steps

    Issue 3: Scope Creep¶

    Symptoms: - Requirement keeps expanding during refinement - New items added after sprint start - "Just one more thing" requests

    Resolution: - Enforce in-scope vs out-of-scope distinction - Create separate requirements for new items - Use P03 (Scrum Agent) to protect sprint boundaries

    Issue 4: Missing Dependencies¶

    Symptoms: - Work blocked mid-sprint due to missing dependency - Circular dependencies discovered late - No clear dependency chain

    Resolution: - Review dependency fields during refinement - Use dependency visualization tools - Escalate to P15 (Orchestrator) for complex dependency resolution

    üìä Metrics and Performance Indicators¶

    Intake Metrics¶

    • Intake Volume: Number of requirements created per sprint

    • Triage Accuracy: % requirements routed to correct POD

    Quality Metrics¶

    • Completeness Rate: % requirements with all required fields
    • Rework Rate: % requirements requiring major revision
    • Acceptance Criteria Clarity: Developer satisfaction score

    Execution Metrics¶

    • Sprint Readiness: % requirements ready for sprint planning
    • Execution Success: % requirements completed as defined
    • Traceability Coverage: % requirements linked to implementation

    üîê Security and Compliance Considerations¶

    Data Classification¶

    • Requirement Content: May contain Protected B information (business logic, user data descriptions)
    • Storage: Requirements stored in GitHub (private repo, SSO-protected)
    • Access Control: RBAC enforced (POD members, Marco as PO)

    Security Checklist (P11 Integration)¶

    • No PII in requirement examples
    • Data classification noted if applicable
    • Security SME review for high-priority requirements
    • Compliance considerations documented (WCAG, OLA, PIPEDA)

    üîß Implementation Guidance¶

    Getting Started with P02¶

    1. Read: Review this chapter and intake schema
    2. Access: Familiarize yourself with requirement templates
    3. Practice: Create a sample requirement using feature template
    4. Validate: Run schema validation against your requirement
    5. Submit: Create GitHub issue using P02 template

    Integration Points¶

    • GitHub: Requirements stored as issues with structured YAML frontmatter
    • Azure DevOps: Optional sync for orgs using ADO boards
    • P03 (Scrum Agent): Backlog management and sprint planning
    • P15 (Orchestrator): Complex requirements with multi-pattern execution

    Best Practices¶

    1. Start Simple: Use templates, don't start from blank page
    2. Be Specific: Vague requirements lead to rework
    3. Test Early: Draft acceptance criteria alongside problem statement
    4. Collaborate: Involve SMEs during refinement
    5. Iterate: Requirements evolve‚Äîupdate as you learn
