    Chapter P07: Testing & Failure Explainer¶

    Pattern Code: P07 Short Code: TST Name: Testing & Failure Explainer Type: DevTools Pattern Scope: Test Quality & Failure Analysis Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Testing & Failure Explainer (P07-TST) transforms cryptic test failures into actionable narratives, suggests test cases for uncovered code, and tracks coverage metrics to ensure EVA Suite maintains 80%+ test coverage. While P06 (Review Agent) checks if tests pass, P07 explains why they fail and what to test next.

    In modern software development, test failures are inevitable‚Äîbut uninformative error messages waste developer time. A message like "Expected: 200, Received: 500" tells you what failed but not why (was it bad input? missing permission? network timeout?). P07 bridges this gap by parsing failures, analyzing context, and generating human-friendly explanations with suggested fixes.

    The Problem P07 Solves: - Cryptic Failures: Error messages like "TypeError: undefined" without context - Debugging Time Waste: Developers spend hours tracing root cause of test failures - Coverage Gaps: Unknown which functions/branches lack test coverage - Test Fatigue: Writing tests feels like chore without guidance on what to test - Regression Ambiguity: Failed tests don't explain if it's a real bug or flaky test

    The Value P07 Delivers: - Fast Diagnosis: Intelligent failure parsing reduces debugging from hours to minutes - Actionable Fixes: Suggested code changes, not just "fix this" - Coverage Visibility: Instant identification of untested code paths - Test Guidance: Auto-generated test templates for uncovered functions - Trend Analysis: Coverage metrics over time highlight quality trajectory

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P07-TST serves as the Test Quality Assurance and Failure Analysis Agent for the EVA Autonomous Agile Command. It ensures:

    1. Intelligent Failure Explanation: Transform raw error logs into developer-friendly narratives
    2. Test Suggestion: Identify coverage gaps and suggest specific test cases
    3. Coverage Tracking: Monitor test coverage metrics (lines, branches, functions) and trends
    4. Quality Enforcement: Block PRs/releases with <80% coverage (configurable threshold)
    5. Regression Analysis: Identify flaky tests and coverage regressions

    Mission Statement¶

    "Accelerate test-driven development and reduce debugging time by providing intelligent test failure analysis, actionable coverage insights, and automated test case suggestions."

    Value Proposition¶

    For Product Owners: - Faster feature delivery (less debugging time) - Higher quality (80%+ coverage enforced) - Predictable releases (test failures explained before production)

    For Developers: - Immediate failure diagnosis (no more hunting through stack traces) - Clear test guidance (know exactly what to test) - Self-service debugging (narratives explain root cause without human help)

    For QA/Test Engineers: - Coverage gap visibility (prioritize testing effort) - Flaky test detection (stabilize test suite) - Metrics for reporting (coverage trends, failure patterns)

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Test Failure Analysis
    2. Parse test output (Vitest, Jest, pytest) to extract failures
    3. Identify error type (null reference, timeout, assertion failure, etc.)
    4. Generate narrative explaining likely cause and context

    5. Suggest specific code fixes with before/after examples

    6. Test Case Suggestion

    7. Analyze coverage reports to find uncovered code
    8. Generate test templates for uncovered functions
    9. Suggest edge cases (null, empty, invalid input)

    10. Recommend integration/e2e tests for complex workflows

    11. Coverage Tracking

    12. Monitor coverage metrics: lines, branches, functions, statements
    13. Track coverage trends over time (improving vs degrading)
    14. Flag coverage regressions (PR reduces coverage)

    15. Generate coverage dashboards (per-file, per-module)

    16. Quality Gate Enforcement

    17. Block PRs with <80% coverage (configurable threshold)
    18. Block releases with failing tests
    19. Alert on flaky tests (inconsistent pass/fail)
    20. Escalate persistent coverage gaps

    Supporting Responsibilities¶

    1. Flaky Test Detection
    2. Re-run tests multiple times to identify flaky behavior
    3. Report tests with <100% pass rate across runs

    4. Suggest stabilization approaches (increase timeout, mock dependencies)

    5. Test Performance Optimization

    6. Identify slow tests (execution time >5s)
    7. Suggest parallelization or mocking for performance

    8. Track test suite execution time trends

    9. Integration Points

    10. Provide failure narratives to P06 (Review Agent) for PR comments
    11. Supply coverage metrics to P10 (Metrics & Insights) for dashboards
    12. Support P08 (CI/CD Guardian) for deployment quality gates

    üöÄ When to Use P07¶

    Primary Use Cases¶

    1. Debugging Test Failures (Automated)¶

    Scenario: Test fails in CI, developer needs to understand why.

    Action: P07 parses failure log and generates narrative with suggested fix.

    Example: 

    // Test failure:
// ‚úï should return user profile
// TypeError: Cannot read property 'name' of undefined

// P07 generates narrative:
## üîç Test Failure Analysis

**Test**: tests/api/users.test.ts - should return user profile
**Error**: TypeError: Cannot read property 'name' of undefined

### Likely Cause
Missing null check or async data not loaded before accessing user.name property.

### Suggested Fix
```typescript
// Before (failing):
export function getUserProfile(userId: string) {
  const user = db.users.find(userId);
  return { name: user.name }; // ‚ùå user might be undefined
}

// After (fixed):
export function getUserProfile(userId: string) {
  const user = db.users.find(userId);
  if (!user) throw new Error(`User not found: ${userId}`);
  return { name: user.name }; // ‚úÖ user guaranteed to exist
}

    When NOT to Use P07¶

    • Don't use P07 to write tests for you - P07 suggests templates, humans write meaningful assertions
    • Don't use P07 for business logic validation - P07 checks coverage, not correctness of tests
    • Don't use P07 to bypass testing - P07 enforces quality, doesn't replace writing tests
    • Don't use P07 for integration test strategy - P07 identifies gaps, architects design integration test plans

    üìê Test Failure Analysis¶

    Intelligent Error Parsing¶

    Script: tools/testing/explain-failure.ts

    interface FailureNarrative {
  title: string;
  description: string;
  likelyCause: string;
  suggestedFix: string;
  relatedDocs: string[];
}

export function explainFailure(error: string, stackTrace: string): FailureNarrative {
  // Pattern matching for common failure types

  // Null/Undefined Reference
  if (error.includes('Cannot read property') || error.includes('undefined')) {
    return {
      title: 'Null/Undefined Reference',
      description: 'Attempting to access property on null or undefined object',
      likelyCause: 'Missing null check or async data not loaded',
      suggestedFix: 'Add optional chaining (?.) or null guard (if/else)',
      relatedDocs: [
        'https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/Optional_chaining',
        'docs/best-practices/error-handling.md'
      ]
    };
  }

  // Timeout
  if (error.includes('timeout')) {
    return {
      title: 'Test Timeout',
      description: 'Test exceeded maximum execution time',
      likelyCause: 'Async operation not completing or infinite loop',
      suggestedFix: 'Increase timeout, optimize code, or fix async handling',
      relatedDocs: [
        'docs/testing/timeouts.md',
        'docs/testing/async-testing.md'
      ]
    };
  }

  // Network/API Failure
  if (error.includes('404') || error.includes('ECONNREFUSED')) {
    return {
      title: 'Network/API Failure',
      description: 'Unable to connect to external service',
      likelyCause: 'Service not running, wrong URL, or network issue',
      suggestedFix: 'Mock external dependencies or ensure test environment is configured',
      relatedDocs: [
        'docs/testing/mocking.md',
        'docs/testing/test-environment-setup.md'
      ]
    };
  }

  // Assertion Failure
  if (error.includes('Expected') && error.includes('Received')) {
    const expected = error.match(/Expected:\s*(.+)/)?.[1];
    const received = error.match(/Received:\s*(.+)/)?.[1];

    return {
      title: 'Assertion Failure',
      description: `Expected ${expected} but received ${received}`,
      likelyCause: 'Logic error or incorrect test expectation',
      suggestedFix: 'Review code logic and verify test assertion is correct',
      relatedDocs: [
        'docs/testing/assertion-best-practices.md'
      ]
    };
  }

  // Async Callback Not Invoked
  if (error.includes('Async callback was not invoked')) {
    return {
      title: 'Async Callback Not Invoked',
      description: 'Test completed without calling done() or returning promise',
      likelyCause: 'Missing await or return statement for async operation',
      suggestedFix: 'Add await before async call or return promise from test',
      relatedDocs: [
        'docs/testing/async-testing.md'
      ]
    };
  }

  // Type Error
  if (error.includes('Type') && error.includes('is not assignable')) {
    return {
      title: 'Type Error',
      description: 'TypeScript type mismatch',
      likelyCause: 'Incorrect type annotation or type inference failure',
      suggestedFix: 'Fix type annotations to match actual values',
      relatedDocs: [
        'https://www.typescriptlang.org/docs/handbook/2/everyday-types.html'
      ]
    };
  }

  // Generic Fallback
  return {
    title: 'Test Failure',
    description: error,
    likelyCause: 'Unknown - review error message and stack trace',
    suggestedFix: 'Debug using VS Code debugger or add console.log statements',
    relatedDocs: ['docs/testing/debugging.md']
  };
}

    Auto-Generated Test Templates¶

    Script: tools/testing/generate-test-template.ts

    export function generateTestTemplate(
  functionName: string,
  params: { name: string; type: string }[],
  returnType: string
): string {
  const paramList = params.map(p => p.name).join(', ');
  const validParams = params.map(p => `valid${capitalize(p.name)}`).join(', ');

  return `
describe('${functionName}', () => {
  it('should return ${returnType} for valid input', () => {
    const result = ${functionName}(${validParams});
    expect(result).toBeDefined();
    // Add specific assertions for ${returnType}
  });

  it('should handle null input gracefully', () => {
    const result = ${functionName}(${params.map(() => 'null').join(', ')});
    // Expect error thrown or null returned
  });

  it('should handle empty input', () => {
    const result = ${functionName}(${params.map(() => "''").join(', ')});
    // Expect error or default value
  });

  it('should handle invalid input', () => {
    const result = ${functionName}(${params.map(() => "'invalid'").join(', ')});
    // Expect validation error
  });

  it('should handle edge cases', () => {
    // Add edge case tests:
    // - Boundary values (min/max)
    // - Special characters
    // - Very large inputs
  });
});
  `.trim();
}

function capitalize(str: string): string {
  return str.charAt(0).toUpperCase() + str.slice(1);
}

    System Prompt (For AI Agents)¶

    You are P07-TST (Testing & Failure Explainer), the Test Quality Agent for EVA Suite.

**Your Capabilities**:
1. Analyze test failures and generate actionable narratives
2. Suggest test cases for uncovered code
3. Track coverage metrics and trends
4. Detect flaky tests
5. Generate test templates for common scenarios

**Your Limitations**:
- You do NOT write complete tests - only templates and suggestions
- You do NOT validate test correctness - only coverage
- You do NOT fix bugs - only explain failures and suggest approaches

**Autonomy Level**: C1 (Human-on-the-loop)
- Automated coverage tracking runs without approval
- Failure narratives generated automatically
- Test suggestions are advisory (human decides which to implement)
- Quality gates (80% coverage) enforced automatically

**Output Format**:
- Failure narratives in Markdown with code snippets
- Test templates in language-specific syntax (TypeScript, Python)
- Coverage metrics in table format
- Trend graphs in text-based visualization

**Safety Rules**:
- Never suggest tests that would pass regardless of implementation
- Flag flaky tests immediately (don't assume they're real bugs)
- Escalate persistent coverage gaps to Tech Lead
- Don't block releases for trivial coverage gaps (<1%)

    ‚úÖ Definition of Done¶

    For Failure Narrative Generation¶

    • Error type identified (null ref, timeout, assertion, etc.)
    • Narrative includes: title, description, likely cause, suggested fix
    • Suggested fix includes before/after code snippet
    • Related documentation links provided
    • Narrative is developer-friendly (avoid jargon)
    • Confidence level indicated (high/medium/low)

    For Test Suggestion¶

    • Coverage gap identified (file, function, branch)
    • Current coverage percentage reported
    • Test template generated for uncovered code
    • Template includes valid/invalid/edge case scenarios
    • Template is syntactically correct (compiles/lints)
    • Placeholders clearly marked for customization

    For Coverage Tracking¶

    • Coverage metrics collected (lines, branches, functions, statements)
    • Metrics saved to persistent storage (docs/metrics/)
    • Trend analysis performed (comparing to previous weeks)
    • Coverage regression flagged (alert if drops >2%)
    • Dashboard updated with latest metrics
    • Per-file breakdown available for drill-down

    üéØ Quality Gates and Success Criteria¶

    Quality Gates¶

    Gate 1: Minimum Coverage Threshold¶

    Criteria: ‚â•80% line coverage, ‚â•75% branch coverage

    Implementation: 

    - name: Enforce Coverage Threshold
  run: |
    lineCoverage=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
    branchCoverage=$(cat coverage/coverage-summary.json | jq '.total.branches.pct')

    if [ $(echo "$lineCoverage < 80" | bc) -eq 1 ]; then
      echo "‚ùå Line coverage $lineCoverage% below 80%"
      exit 1
    fi

    if [ $(echo "$branchCoverage < 75" | bc) -eq 1 ]; then
      echo "‚ö†Ô∏è Branch coverage $branchCoverage% below 75%"
      # Warning but don't fail (branches harder to cover)
    fi

    Severity: Blocker for production releases

    Gate 2: No Failing Tests¶

    Criteria: 100% tests pass

    Implementation: 

    - name: Enforce Test Success
  run: npm test
  # No continue-on-error - fail workflow if any test fails

    Severity: Blocker for all PRs and releases

    üö® Common Issues and Troubleshooting¶

    Issue 1: False Positive - Coverage Drop from Refactor¶

    Symptom: Coverage drops after refactoring even though functionality unchanged

    Cause: Lines moved to new files not covered by existing tests

    Solution: 

    # Identify newly uncovered files
npm run test:coverage -- --coverage-diff

# Add tests for newly created files
tools/testing/suggest-tests.ps1 -File "src/new-module.ts"

    Prevention: Run coverage before/after refactor, ensure net-zero coverage change

    Issue 2: Flaky Test Masking Real Bug¶

    Symptom: Test fails intermittently, assumed to be flaky but actually reveals race condition

    Diagnosis: 

    # Run test many times to confirm flakiness
tools/testing/detect-flaky.ps1 -Test "tests/api/users.test.ts" -Runs 100

# If pass rate <95%, investigate root cause (don't just skip)

    Solution: Fix underlying race condition, don't just increase timeout

    Issue 3: Coverage Metric Misleading¶

    Symptom: 80% coverage but tests are shallow (no meaningful assertions)

    Cause: Tests execute code but don't validate behavior

    Solution: 

    // Bad test (executes code but doesn't verify):
it('should process user', () => {
  processUser(testUser);
  // ‚ùå No assertions - coverage increased but no value
});

// Good test (validates behavior):
it('should process user', () => {
  const result = processUser(testUser);
  expect(result.status).toBe('processed');
  expect(result.timestamp).toBeDefined();
  // ‚úÖ Verifies expected behavior
});

    Prevention: Code review focuses on assertion quality, not just coverage %

    üîê Security and Compliance Considerations¶

    Test Data Security¶

    Risk: Tests may expose sensitive data (PII, credentials) in logs

    Mitigation: - Use synthetic/mock data in tests (never real user data) - Sanitize test output logs (redact sensitive values) - Encrypt test fixtures containing sensitive patterns

    Example: 

    // ‚ùå Bad: Real data in test
it('should authenticate user', () => {
  login('john.doe@example.com', 'RealPassword123');
});

// ‚úÖ Good: Synthetic data
it('should authenticate user', () => {
  login('testuser@example.test', 'TestPassword');
});

    Coverage Metrics Privacy¶

    Risk: Coverage reports may reveal proprietary business logic structure

    Mitigation: - Store coverage reports in private repositories only - Redact file paths in public-facing reports - Exclude coverage badges for closed-source products
