    Chapter P01: Agent Patterns Toolbox¶

    Pattern Code: P01 Short Code: APT Name: Agent Patterns Toolbox Type: DevTools Pattern Scope: Pattern Discovery & Standardization Maturity: Stable Autonomy Level: C0 (Human-in-the-loop)

    üìñ Chapter Overview¶

    The Agent Patterns Toolbox (P01-APT) is the foundation of the EVA DevTools ecosystem. It provides the catalog, metamodel, and discovery workflow that enables any pod to spin up new agents from standardized patterns. Think of it as the "library card catalog" for all EVA agents‚Äîboth DevTools patterns (P01-P21) and Service Personas (SP00-SPxx).

    Without P01, the EVA ecosystem would be a collection of ad-hoc scripts and undocumented processes. With P01, every agent follows a consistent structure, has discoverable documentation, and integrates seamlessly with the broader EVA platform.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P01-APT serves as the Pattern Registry Manager and Discovery Engine for the EVA Autonomous Agile Command. It ensures:

    1. Consistency: All agents follow the same metamodel structure
    2. Discoverability: Developers can quickly find the right pattern for their use case
    3. Reusability: Proven patterns are documented and ready to spin up
    4. Quality: Templates and validation ensure high standards

    Mission Statement¶

    "Enable rapid, consistent, and quality-assured agent creation through standardized patterns, discoverable documentation, and validated templates."

    Value Proposition¶

    For Product Owners: - Accelerate feature delivery by reusing proven patterns - Reduce risk through standardized, validated approaches - Improve transparency with consistent documentation

    For Developers: - Find the right pattern quickly without searching scattered docs - Bootstrap new agents with confidence using validated templates - Understand dependencies and relationships between patterns

    For Operations: - Predictable agent behavior through standardization - Easier troubleshooting with consistent structure - Clear autonomy levels (C0-C3) for governance

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Maintain Pattern Registry (docs/agents/registry.yaml)
    2. Keep catalog up-to-date with all patterns (P01-P21, SP00-SPxx)
    3. Track maturity levels (stable, alpha, beta, concept, deprecated)

    4. Document autonomy levels (C0-C3) for each pattern

    5. Define Metamodel (Pattern structure specification)

    6. Pattern code, name, short code
    7. Type (devtools vs service_persona)
    8. Scope, description, home path
    9. Primary users, invocation methods
    10. Inputs, outputs, limitations
    11. Safety rules (C0-C3 constraints)

    12. Related repos, patterns, frameworks

    13. Provide Discovery Workflow

    14. Browsing catalog by type, maturity, scope
    15. Reading pattern documentation
    16. Checking dependencies and prerequisites

    17. Understanding autonomy level constraints

    18. Enable Pattern Creation

    19. Template for new pattern documentation
    20. Registration process in catalog
    21. Validation checklist
    22. Version control and change tracking

    Supporting Responsibilities¶

    1. Documentation Quality
    2. Ensure all patterns have complete, accurate docs
    3. Validate links and references

    4. Maintain consistency in tone and structure

    5. Lifecycle Management

    6. Track patterns from concept ‚Üí alpha ‚Üí beta ‚Üí stable
    7. Manage deprecation and migration paths

    8. Archive obsolete patterns appropriately

    9. Integration Points

    10. Coordinate with P02 (Requirements Engine) for new pattern requests
    11. Support P15 (Dev Master Orchestrator) with pattern recommendations
    12. Provide metadata to P21 (Provenance & Fingerprinting) for audit trails

    üöÄ When to Use P01¶

    Primary Use Cases¶

    1. Discovering an Existing Pattern¶

    Scenario: You need to automate something (e.g., testing, security scanning) but aren't sure if a pattern already exists.

    Action: Browse the registry or search by scope/keywords to find relevant patterns.

    Example: 

    # Search for testing-related patterns
grep "scope: testing" docs/agents/registry.yaml

# Result: P07 (Testing & Failure Explainer)

    2. Creating a New Pattern¶

    Scenario: You're building a new automation that will be reused across multiple repos/pods and want to formalize it.

    Action: Follow P01's pattern creation workflow to register and document your new pattern.

    Example: Creating P22 for "Contract Testing Agent" - Choose next available code (P22) - Draft registry entry with metamodel fields - Create documentation using P01 template - Register in registry.yaml

    3. Validating Pattern Compliance¶

    Scenario: You've inherited an undocumented automation and want to bring it into compliance.

    Action: Use P01 metamodel to audit the automation and fill gaps.

    Example: Reviewing a legacy script - Does it have a pattern code? (No ‚Üí assign one) - Is it documented? (No ‚Üí create docs) - What's its autonomy level? (Unknown ‚Üí analyze and classify C0-C3)

    4. Understanding Pattern Dependencies¶

    Scenario: You're implementing P09 (Runbook Agent) and need to know what other patterns it depends on.

    Action: Check P09's registry entry and documentation for dependencies.

    Example: 

    # From registry.yaml
- code: P09
  related_patterns:
    - P07  # Testing hooks for failure detection
    - TASK-034  # Telemetry instrumentation

    When NOT to Use P01¶

    • Don't use P01 to execute patterns - P01 is for discovery and registration, not execution
    • Don't use P01 for one-off scripts - Only formalize patterns that will be reused
    • Don't use P01 to bypass governance - Autonomy levels (C0-C3) must be respected

    Template 3: Pattern Discovery Query¶

    Use Case: Searching for patterns by criteria

    # PowerShell script: tools/p01/find-pattern.ps1

param(
    [string]$Scope,      # e.g., "testing", "security"
    [string]$Maturity,   # e.g., "stable", "alpha"
    [string]$Type,       # "devtools" or "service_persona"
    [string]$Keyword     # Free-text search
)

$registry = Get-Content "docs/agents/registry.yaml" -Raw | ConvertFrom-Yaml

$results = $registry.patterns | Where-Object {
    ($null -eq $Scope -or $_.scope -like "*$Scope*") -and
    ($null -eq $Maturity -or $_.maturity -eq $Maturity) -and
    ($null -eq $Type -or $_.type -eq $Type) -and
    ($null -eq $Keyword -or
     $_.name -like "*$Keyword*" -or
     $_.description -like "*$Keyword*")
}

$results | ForEach-Object {
    Write-Host "`n$($_.code): $($_.name)" -ForegroundColor Cyan
    Write-Host "  Scope: $($_.scope)"
    Write-Host "  Maturity: $($_.maturity)"
    Write-Host "  Docs: $($_.home)"
}

    Example Usage: 

    # Find all stable testing patterns
.\tools\p01\find-pattern.ps1 -Scope "testing" -Maturity "stable"

# Output:
# P07: Testing & Failure Explainer
#   Scope: testing
#   Maturity: stable
#   Docs: docs/agents/devtools/P07-Testing-Failure-Explainer.md

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C0 (Human-in-the-Loop)¶

    P01 Itself: C0 - All registry changes require human review and commit - Pattern discovery is advisory only (humans choose which pattern to use) - Documentation updates must be validated by pattern owner

    Why C0? - Pattern registry is foundational infrastructure‚Äîerrors cascade - Autonomy level assignments have governance implications - Incorrect pattern recommendations could waste team effort

    Safety Rules¶

    Rule 1: Registry Integrity¶

    Risk: Corrupted registry breaks pattern discovery for entire team

    Mitigation: - ‚úÖ Validate YAML syntax before committing changes - ‚úÖ Require PR review for all registry.yaml edits - ‚úÖ Run automated validation in CI/CD pipeline - ‚úÖ Keep backup of registry in version control history

    Rule 2: Autonomy Level Accuracy¶

    Risk: Misclassified autonomy level leads to governance violations

    Mitigation: - ‚úÖ Require Security SME review for C1+ patterns - ‚úÖ Document rationale for autonomy level assignment - ‚úÖ Include safety rules in pattern documentation - ‚úÖ Periodic audit of autonomy level classifications

    Rule 3: Maturity Level Honesty¶

    Risk: Marking alpha/beta patterns as "stable" causes production issues

    Mitigation: - ‚úÖ Start all new patterns as "concept" or "alpha" - ‚úÖ Require successful pilot usage before "stable" promotion - ‚úÖ Document maturity promotion criteria - ‚úÖ Flag experimental patterns prominently in docs

    Rule 4: Dependency Tracking¶

    Risk: Pattern used without satisfying dependencies causes failures

    Mitigation: - ‚úÖ Explicitly list all dependencies in registry entry - ‚úÖ Validate dependencies before pattern invocation - ‚úÖ Warn users when dependencies are alpha/beta maturity - ‚úÖ Document dependency installation/configuration steps

    Autonomy Level Definitions (Reference)¶

    Level Name Description P01 Usage
    C0 Human-in-the-loop AI suggests, human approves every action Pattern discovery recommendations
    C1 Human-on-the-loop AI acts, human monitors Not applicable (P01 is read-only)
    C2 Human-over-the-loop AI acts with constraints, periodic review Not applicable (P01 is read-only)
    C3 Full autonomy AI acts autonomously within boundaries Not applicable (P01 is read-only)

    • I have validated YAML syntax before committing
    • I have requested PR review from POD-F lead

    For Pattern Maintainers (Ongoing)¶

    • Pattern documentation is up-to-date with latest changes
    • All links in documentation are valid and working
    • Maturity level reflects actual usage (alpha ‚Üí beta ‚Üí stable)
    • Autonomy level is still accurate (re-audit annually)
    • Dependencies are current and maintained
    • Related patterns list is complete
    • Version history is maintained
    • Deprecation path documented if applicable
    • Usage examples are clear and tested
    • Security/safety considerations are current

    For Governance (Periodic Audit)¶

    • All patterns have valid autonomy level classifications
    • C1+ patterns have Security SME sign-off
    • No patterns stuck in "concept" >3 months without progress
    • Deprecated patterns have migration plans
    • Registry YAML schema validation passes
    • No orphaned pattern documentation (docs without registry entries)
    • No broken inter-pattern dependencies
    • Maturity levels align with actual usage metrics

    üéì Key Takeaways¶

    1. P01 is the foundation - All other patterns depend on a well-maintained registry
    2. Standardization accelerates - Consistent structure reduces cognitive load
    3. Discovery is critical - Teams can't use patterns they don't know exist
    4. Maturity matters - Be honest about alpha/beta vs stable

    6. Documentation is code - Treat pattern docs with same rigor as source code
    7. Dependencies compound - Validate the full dependency tree before usage
    8. Deprecation is kind - Provide clear migration paths, not abrupt removals
