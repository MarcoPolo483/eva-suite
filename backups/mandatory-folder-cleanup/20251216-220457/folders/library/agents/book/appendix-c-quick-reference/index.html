    Appendix C: Quick Reference Guide¶

    EVA Agentic Framework - Pattern Catalog Version 2.0 | December 2025

    How to Use This Guide¶

    This appendix provides one-page summaries of all 21 EVA DevTools Patterns (P01-P21), organized for quick lookup during development work. Each pattern entry includes:

    • Purpose: What the pattern does
    • When to Use: Trigger conditions and scenarios
    • Key Commands: Common prompts and CLI commands
    • Inputs/Outputs: What the pattern consumes and produces
    • Quality Gates: Success criteria and validation steps
    • Related Patterns: Common combinations and workflows

    Pattern Index: - P01-P05: Foundation & Setup (Architecture, Requirements, Scrum, Documentation, Scaffolding) - P06-P10: Code Quality & Operations (Review, Testing, CI/CD, Runbooks, Metrics) - P11-P15: Cross-Cutting Concerns (Security, UX/Accessibility, Onboarding, LiveOps, Dev Master) - P16-P21: Advanced Capabilities (RAG, AI Safety, FinOps, Governance, Knowledge, Integration)

    P01: Architecture Decision Records (ADR)¶

    Purpose: Document significant architectural decisions with context, alternatives, and consequences.

    When to Use: - Selecting between competing technologies or approaches - Making infrastructure or platform decisions - Establishing patterns that affect multiple teams - Introducing breaking changes or major refactors

    Key Commands: 

    # Create new ADR
gh copilot suggest "Create ADR for [decision topic]"

# List existing ADRs
gh copilot explain "Show me all architectural decisions for [domain]"

# Update ADR status
gh copilot suggest "Mark ADR-XXX as superseded by ADR-YYY"

    Common Prompts: - "Create an ADR for choosing between Azure Cosmos DB and PostgreSQL for EVA Core" - "Document the decision to adopt GC Design System for all UI components" - "Update ADR-012 with consequences discovered during implementation"

    Inputs: - Decision context and background - Alternatives considered - Stakeholder requirements - Technical constraints

    Outputs: - docs/architecture/decisions/ADR-NNN-title.md - Updated docs/architecture/decisions/README.md index - Traceability links to related issues/PRs

    Quality Gates: - ✅ Title clearly states the decision - ✅ Context section explains why decision is needed - ✅ At least 2 alternatives documented - ✅ Consequences (positive and negative) identified - ✅ Status is one of: Proposed, Accepted, Deprecated, Superseded

    Related Patterns: - P02-REQ: ADRs inform CDD development - P04-LIB: ADRs are part of repo documentation standards - P05-SCA: Scaffolding implements ADR decisions

    P02: Requirements Engineering (REQ)¶

    Purpose: Refine raw requirements into structured, testable user stories with clear acceptance criteria.

    When to Use: - Converting Product Owner requests into sprint-ready stories - Refining epics into implementable features - Creating Collaborative Design Documents (CDDs) - Validating requirements against business value

    Key Commands: 

    # Generate CDD from epic
gh copilot suggest "Create CDD for epic: [epic description]"

# Refine story acceptance criteria
gh copilot suggest "Write acceptance criteria for story: [story title]"

# Generate test scenarios
gh copilot suggest "Create test scenarios for requirement: [requirement]"

    Common Prompts: - "Convert this feature request into a CDD with user personas and success metrics" - "Refine this epic into 3-5 implementable user stories" - "Generate acceptance criteria for WCAG 2.1 AA compliance requirement"

    Inputs: - Product Owner vision/goals - User research and personas - Business constraints - Technical dependencies

    Outputs: - Collaborative Design Documents (CDDs) in docs/requirements/ - User stories with acceptance criteria - Test scenarios and edge cases - Traceability matrix linking requirements to tests

    Quality Gates: - ✅ Requirements follow INVEST criteria (Independent, Negotiable, Valuable, Estimable, Small, Testable) - ✅ Acceptance criteria are measurable and binary (pass/fail) - ✅ User personas clearly defined - ✅ Success metrics identified (e.g., "Users can complete task in <30s")

    Related Patterns: - P01-ADR: Requirements may trigger architectural decisions - P03-SCR: Stories are input to sprint planning - P07-TST: Test scenarios drive test case generation

    P03: Scrum Support (SCR)¶

    Purpose: Automate sprint ceremonies, backlog management, and agile metrics tracking.

    When to Use: - Planning sprints and estimating capacity - Running daily stand-ups and identifying blockers - Conducting retrospectives and tracking action items - Generating sprint reports and burn charts

    Key Commands: 

    # Create sprint plan
gh copilot suggest "Create sprint plan for Sprint N with capacity X"

# Generate stand-up summary
gh copilot suggest "Summarize yesterday's progress and today's plan"

# Run retrospective
gh copilot suggest "Generate retro board with what went well/what needs improvement"

    Common Prompts: - "Plan Sprint 12 with 80 story points capacity across 3 developers" - "Generate daily stand-up summary from git commits and issue activity" - "Create sprint retrospective and identify top 3 action items" - "Show burn-down chart and predict sprint completion date"

    Inputs: - Sprint backlog (prioritized stories) - Team velocity and capacity - Issue tracking data (GitHub Issues, ADO) - Git commit history

    Outputs: - Sprint plans in docs/sprint-kit/sprints/sprint-NN/ - Daily stand-up summaries - Retrospective notes and action items - Metrics dashboards (burn charts, velocity trends)

    Quality Gates: - ✅ Sprint backlog prioritized by Product Owner - ✅ Team capacity matches committed work - ✅ Daily updates captured (no stale work >72 hours) - ✅ Retrospective action items assigned and tracked

    Related Patterns: - P02-REQ: Requirements feed sprint backlog - P10-MET: Metrics track sprint health - P15-DVM: Dev Master orchestrates sprint execution

    P04: Librarian (LIB)¶

    Purpose: Maintain consistent, high-quality repository documentation and knowledge artifacts.

    When to Use: - Creating or updating README files - Standardizing documentation structure across repos - Generating API documentation and code comments - Auditing documentation quality and coverage

    Key Commands: 

    # Generate README from code
gh copilot suggest "Create README for this repository based on package.json and src/"

# Update API docs
gh copilot suggest "Generate API documentation for all exported functions"

# Audit documentation
gh copilot suggest "Check documentation coverage and identify gaps"

    Common Prompts: - "Create a README with quick start, architecture overview, and contribution guide" - "Generate JSDoc comments for all public API functions" - "Audit this repository's documentation and suggest improvements" - "Standardize CONTRIBUTING.md across all EVA Suite repositories"

    Inputs: - Source code and project structure - Existing documentation fragments - Repository templates and standards - Code comments and type definitions

    Outputs: - README.md with standard structure - CONTRIBUTING.md and CODE_OF_CONDUCT.md - docs/ folder with architecture, API, and guides - Auto-generated API reference (JSDoc, Sphinx, etc.)

    Quality Gates: - ✅ README includes: purpose, quick start, architecture, contribution guide - ✅ All public APIs documented with parameters and return types - ✅ Code examples tested and executable - ✅ Links verified (no 404s)

    Related Patterns: - P01-ADR: ADRs are core documentation artifacts - P05-SCA: Scaffolding creates initial documentation structure - P13-ONB: Documentation supports onboarding workflows

    P05: Scaffolding (SCA)¶

    Purpose: Bootstrap new repositories, modules, and components with consistent structure and tooling.

    When to Use: - Creating new microservices or libraries - Adding new features or modules to existing projects - Setting up CI/CD pipelines and development environments - Implementing architectural patterns from ADRs

    Key Commands: 

    # Create new repo from template
gh copilot suggest "Scaffold new TypeScript library with Jest and ESLint"

# Add new module
gh copilot suggest "Create new module for [domain] with standard structure"

# Setup CI/CD
gh copilot suggest "Add GitHub Actions workflow for build, test, and deploy"

    Common Prompts: - "Scaffold a new Next.js app with TypeScript, Tailwind, and Vitest" - "Create a new Python FastAPI service with Poetry, Pytest, and pre-commit hooks" - "Add a new React component library with Storybook and Chromatic" - "Setup GitHub Actions workflow with lint, test, security scan, and deploy stages"

    Inputs: - Repository templates (e.g., eva-seed, language-specific starters) - Architecture decisions (ADRs) - Technology stack preferences - Project naming conventions

    Outputs: - Fully initialized repository with: - package.json/pyproject.toml with dependencies - .gitignore, .editorconfig, linter configs - CI/CD pipelines (.github/workflows/) - Initial test structure - README and contribution guides

    Quality Gates: - ✅ All dependencies pinned with lock file - ✅ Linter and formatter configured and passing - ✅ Initial test suite passes (100% coverage for scaffolding code) - ✅ CI/CD pipeline runs successfully - ✅ Security scan passes (no vulnerabilities in dependencies)

    Related Patterns: - P01-ADR: Scaffolding implements architectural decisions - P04-LIB: Scaffolding creates documentation structure - P08-CICD: Scaffolding includes CI/CD pipeline setup

    P06: Code Review (REV)¶

    Purpose: Automate first-pass code review for style, quality, security, and best practices.

    When to Use: - Reviewing pull requests before human review - Identifying common code smells and anti-patterns - Enforcing coding standards and conventions - Suggesting refactoring opportunities

    Key Commands: 

    # Review pull request
gh copilot suggest "Review PR #123 for code quality and security issues"

# Explain code changes
gh copilot explain "What does this PR change and why?"

# Suggest improvements
gh copilot suggest "Suggest refactoring for this function to improve readability"

    Common Prompts: - "Review this PR for security vulnerabilities, code smells, and style violations" - "Explain the purpose of this code change in simple terms" - "Suggest improvements to this function's error handling" - "Check if this PR follows the repository's coding standards"

    Inputs: - Pull request diff - Coding standards and style guides - Security policies (OWASP, CWE) - Test coverage reports

    Outputs: - PR review comments with severity (Info, Warning, Error) - Suggested code improvements with diffs - Security findings (e.g., "Potential SQL injection on line 42") - Test coverage impact analysis

    Quality Gates: - ✅ No high-severity security findings - ✅ Code style violations resolved - ✅ Test coverage maintained or improved - ✅ No commented-out code or debug statements - ✅ Documentation updated for public API changes

    Related Patterns: - P07-TST: Code review checks test quality - P11-SEC: Security review is part of code review - P15-DVM: Dev Master routes PRs to reviewers

    P07: Testing (TST)¶

    Purpose: Generate, execute, and explain test cases with focus on coverage and failure diagnosis.

    When to Use: - Writing unit, integration, and E2E tests - Diagnosing test failures and flaky tests - Improving test coverage - Generating test data and mocks

    Key Commands: 

    # Generate test cases
gh copilot suggest "Generate unit tests for function [name] with edge cases"

# Explain test failure
gh copilot explain "Why is this test failing? [paste error]"

# Improve coverage
gh copilot suggest "Add tests to increase coverage from 60% to 80%"

    Common Prompts: - "Generate unit tests for this React component with all user interactions" - "Create integration tests for this API endpoint with success and error cases" - "Explain why this test is failing and suggest a fix" - "Generate test data for a user with 10 orders and 3 addresses"

    Inputs: - Source code to test - Existing test suite - Coverage reports - Test failure logs

    Outputs: - Test files (.test.ts, _test.py, etc.) - Test data fixtures and mocks - Coverage reports (HTML, JSON) - Test failure explanations and fixes

    Quality Gates: - ✅ Unit test coverage ≥80% - ✅ Integration test coverage ≥60% - ✅ All tests pass in CI/CD - ✅ No flaky tests (100% pass rate in last 10 runs) - ✅ Test naming follows AAA pattern (Arrange, Act, Assert)

    Related Patterns: - P02-REQ: Test scenarios derived from acceptance criteria - P06-REV: Code review checks test quality - P08-CICD: Tests run in CI/CD pipeline

    P08: CI/CD Pipeline (CICD)¶

    Purpose: Design, implement, and maintain continuous integration and deployment pipelines.

    When to Use: - Setting up build, test, and deployment automation - Implementing blue-green or canary deployments - Configuring multi-stage pipelines (dev, staging, prod) - Troubleshooting pipeline failures

    Key Commands: 

    # Create pipeline
gh copilot suggest "Create GitHub Actions workflow for Node.js app with deploy to Azure"

# Debug pipeline failure
gh copilot explain "Why did this workflow fail? [paste logs]"

# Optimize pipeline
gh copilot suggest "Reduce pipeline duration from 15min to <5min"

    Common Prompts: - "Create a CI/CD pipeline with lint, test, security scan, build, and deploy stages" - "Setup blue-green deployment with automated rollback on health check failure" - "Optimize Docker build to use layer caching and multi-stage builds" - "Configure deployment approvals for production environment"

    Inputs: - Source code repository - Deployment targets (Azure, AWS, GCP) - Environment configurations - Security and compliance policies

    Outputs: - .github/workflows/ or .azure-pipelines.yml - Deployment scripts and Dockerfiles - Environment configuration files - Pipeline metrics (duration, success rate)

    Quality Gates: - ✅ Pipeline runs in <10 minutes - ✅ Success rate ≥95% (excluding flaky tests) - ✅ All security scans pass before deployment - ✅ Deployments are atomic (rollback on failure) - ✅ Production deployments require approval

    Related Patterns: - P05-SCA: Scaffolding creates initial pipeline - P07-TST: Tests run in CI/CD pipeline - P11-SEC: Security scans integrated in pipeline

    P09: Runbooks (RUN)¶

    Purpose: Create operational runbooks and SRE playbooks for incident response and maintenance.

    When to Use: - Documenting operational procedures - Creating incident response playbooks - Automating routine maintenance tasks - Onboarding on-call engineers

    Key Commands: 

    # Create runbook
gh copilot suggest "Create runbook for handling database connection failures"

# Generate incident playbook
gh copilot suggest "Create playbook for high CPU usage alerts"

# Automate task
gh copilot suggest "Script routine log rotation and archival"

    Common Prompts: - "Create runbook for deploying hotfixes to production" - "Generate incident response playbook for API latency spikes" - "Document procedure for rotating Azure Key Vault secrets" - "Create automation script for weekly database backup verification"

    Inputs: - System architecture and dependencies - Historical incident reports - Monitoring and alerting configuration - SLA and SLO definitions

    Outputs: - Runbooks in docs/runbooks/ (Markdown or Jupyter notebooks) - Incident playbooks with decision trees - Automation scripts (scripts/ops/) - On-call rotation schedules

    Quality Gates: - ✅ Runbooks tested in staging environment - ✅ Clear success criteria and rollback procedures - ✅ Contact information and escalation paths included - ✅ Automation scripts have error handling and logging - ✅ Reviewed by on-call engineers

    Related Patterns: - P10-MET: Metrics trigger runbook execution - P14-LVO: LiveOps uses runbooks during incidents - P15-DVM: Dev Master orchestrates runbook automation

    P10: Metrics & Insights (MET)¶

    Purpose: Track, visualize, and analyze system and team metrics for data-driven decisions.

    When to Use: - Setting up dashboards for system health - Tracking sprint velocity and team performance - Analyzing user behavior and product usage - Creating alerts and anomaly detection

    Key Commands: 

    # Create dashboard
gh copilot suggest "Create dashboard for API latency, error rate, and throughput"

# Analyze metrics
gh copilot explain "Why did latency spike at 14:00? [link to metrics]"

# Setup alerts
gh copilot suggest "Create alert for 5xx errors >1% over 5 minutes"

    Common Prompts: - "Create Application Insights dashboard for EVA API with RED metrics" - "Analyze sprint velocity trends and predict completion date" - "Setup anomaly detection for unusual user activity patterns" - "Generate weekly metrics report for Product Owner"

    Inputs: - Application logs and traces - System metrics (CPU, memory, disk, network) - Business metrics (user signups, feature usage) - Sprint data (velocity, burn rate, cycle time)

    Outputs: - Dashboards (Azure Monitor, Grafana, Datadog) - Alert rules and notification channels - Metrics reports (weekly, monthly) - Anomaly detection models

    Quality Gates: - ✅ Key metrics have defined SLIs and SLOs - ✅ Dashboards load in <2 seconds - ✅ Alerts have clear actionable next steps - ✅ False positive rate <5% - ✅ Metrics retained per retention policy (7 days hot, 90 days warm, 7 years cold)

    Related Patterns: - P03-SCR: Sprint metrics track team performance - P09-RUN: Metrics trigger runbook execution - P20-FNO: FinOps metrics track cost attribution

    P11: Security & Compliance (SEC)¶

    Purpose: Enforce security best practices and compliance requirements (OWASP, CWE, Protected B).

    When to Use: - Scanning code for security vulnerabilities - Implementing authentication and authorization - Ensuring data encryption and privacy - Preparing for security audits and ATOs

    Key Commands: 

    # Security scan
gh copilot suggest "Scan this code for OWASP Top 10 vulnerabilities"

# Implement auth
gh copilot suggest "Add Azure AD authentication with RBAC"

# Encryption
gh copilot suggest "Encrypt data at rest using Azure Key Vault CMK"

    Common Prompts: - "Scan this repository for secrets, hardcoded credentials, and API keys" - "Implement Protected B encryption for Cosmos DB and Blob Storage" - "Add input validation to prevent SQL injection and XSS" - "Generate security audit report for ATO submission"

    Inputs: - Source code and dependencies - Security policies (OWASP, CWE, NIST) - Compliance requirements (Protected B, WCAG 2.1 AA) - Threat model and attack vectors

    Outputs: - Security scan reports (SARIF format) - Remediation recommendations with code samples - Compliance checklists - Security audit artifacts (PIAs, TRAs, ATOs)

    Quality Gates: - ✅ No high or critical severity vulnerabilities - ✅ Secrets managed in Azure Key Vault (no hardcoded credentials) - ✅ TLS 1.2+ enforced for all connections - ✅ RBAC implemented with principle of least privilege - ✅ Audit logging enabled for all sensitive operations

    Related Patterns: - P06-REV: Security review is part of code review - P08-CICD: Security scans run in CI/CD pipeline - P19-GOV: Governance ensures compliance controls

    P12: UX & Accessibility (UXA)¶

    Purpose: Ensure WCAG 2.1 AA compliance, bilingual support (EN-CA/FR-CA), and GC Design System adherence.

    When to Use: - Designing UI components and interactions - Implementing accessibility features (keyboard nav, screen readers) - Supporting bilingual content and language switching - Auditing UX quality and usability

    Key Commands: 

    # Accessibility audit
gh copilot suggest "Audit this component for WCAG 2.1 AA compliance"

# Bilingual support
gh copilot suggest "Add i18n keys for EN-CA and FR-CA to this component"

# GC Design System
gh copilot suggest "Convert this component to use GC Design System"

    Common Prompts: - "Audit this page for keyboard navigation and screen reader compatibility" - "Add bilingual labels and help text to this form" - "Create a component using GC Design System with canada-gc theme" - "Fix color contrast issues to meet WCAG 2.1 AA standards"

    Inputs: - UI mockups and wireframes - User research and personas - WCAG 2.1 AA requirements - GC Design System component library

    Outputs: - Accessible React components (ARIA labels, keyboard handlers) - i18n JSON files (en-CA.json, fr-CA.json) - Lighthouse accessibility audit reports - Design system compliance reports

    Quality Gates: - ✅ Lighthouse accessibility score ≥95 - ✅ All interactive elements keyboard accessible - ✅ Color contrast ratio ≥4.5:1 for text - ✅ All content available in EN-CA and FR-CA - ✅ GC Design System components used (canada-gc theme)

    Related Patterns: - P04-LIB: Documentation includes accessibility guidelines - P07-TST: Accessibility tests included in test suite - P11-SEC: Compliance includes WCAG 2.1 AA

    P13: Onboarding (ONB)¶

    Purpose: Guide new developers through sandbox environments and EVA Suite onboarding.

    When to Use: - Onboarding new team members - Creating sandbox environments for training - Generating onboarding checklists and tutorials - Troubleshooting setup issues

    Key Commands: 

    # Create onboarding guide
gh copilot suggest "Create onboarding guide for new frontend developer"

# Setup sandbox
gh copilot suggest "Create sandbox environment for testing EVA API"

# Generate checklist
gh copilot suggest "Generate Day 1 checklist for new engineer"

    Common Prompts: - "Create onboarding tutorial for EVA Suite with step-by-step setup instructions" - "Generate Day 1, Week 1, and Month 1 checklists for new developers" - "Setup sandbox environment with sample data and test accounts" - "Troubleshoot Node.js version mismatch during local development setup"

    Inputs: - Repository README and documentation - Development environment requirements - Team processes and workflows - Training materials and tutorials

    Outputs: - Onboarding guides in docs/onboarding/ - Sandbox environment scripts (scripts/sandbox/) - Checklists and progress trackers - Troubleshooting FAQs

    Quality Gates: - ✅ New developer can complete setup in <1 hour - ✅ Sandbox data reset script works reliably - ✅ All required tools documented with versions - ✅ Onboarding feedback collected and incorporated

    Related Patterns: - P04-LIB: Documentation supports onboarding - P05-SCA: Scaffolding creates sandbox environments - P15-DVM: Dev Master coordinates onboarding tasks

    P14: LiveOps (LVO)¶

    Purpose: Support live operations with real-time monitoring, incident response, and hotfix deployment.

    When to Use: - Responding to production incidents - Deploying emergency hotfixes - Monitoring system health during releases - Conducting post-incident reviews

    Key Commands: 

    # Incident response
gh copilot suggest "Diagnose production issue: [error message]"

# Deploy hotfix
gh copilot suggest "Create hotfix branch for critical bug in production"

# Post-incident review
gh copilot suggest "Generate post-incident report for [incident ID]"

    Common Prompts: - "Diagnose 500 errors affecting 10% of users on /api/chat endpoint" - "Create hotfix for SQL injection vulnerability discovered in production" - "Monitor deployment of v2.1.5 and prepare rollback plan" - "Generate post-incident report with timeline, root cause, and action items"

    Inputs: - Real-time logs and metrics - Incident reports and alerts - Deployment pipelines - On-call engineer assignments

    Outputs: - Incident response notes - Hotfix branches and PRs - Post-incident reviews (PIRs) - System health dashboards

    Quality Gates: - ✅ Incident acknowledged within 5 minutes - ✅ Mitigation deployed within SLA (15-60 minutes) - ✅ PIR completed within 48 hours - ✅ Action items assigned and tracked - ✅ Lessons learned incorporated into runbooks

    Related Patterns: - P09-RUN: Runbooks guide incident response - P10-MET: Metrics detect anomalies and incidents - P15-DVM: Dev Master coordinates hotfix deployment

    P15: Dev Master Orchestrator (DVM)¶

    Purpose: Orchestrate complex multi-pattern workflows and coordinate team assignments (Jarvis mode).

    When to Use: - Coordinating work across multiple patterns - Managing dependencies between tasks - Automating complex workflows - Delegating work to specialist patterns

    Key Commands: 

    # Orchestrate workflow
gh copilot suggest "Orchestrate: Design API → Generate tests → Setup CI/CD → Deploy"

# Coordinate team
gh copilot suggest "Assign story to P06-REV for code review, then P08-CICD for deployment"

# Automate release
gh copilot suggest "Automate release workflow: version bump → changelog → tag → deploy → notify"

    Common Prompts: - "Orchestrate end-to-end feature development from CDD to production deployment" - "Coordinate sprint planning: P02-REQ refines stories, P03-SCR plans sprint" - "Automate pull request workflow: P06-REV reviews, P07-TST checks coverage, P08-CICD deploys" - "Delegate incident response: P14-LVO triages, P09-RUN executes playbook, P10-MET monitors"

    Inputs: - High-level goals and workflows - Team assignments and availability - Pattern capabilities and dependencies - Automation templates

    Outputs: - Workflow orchestration plans - Task delegation and tracking - Automation scripts for multi-step workflows - Coordination dashboards

    Quality Gates: - ✅ Workflow steps clearly sequenced with dependencies - ✅ Each pattern's inputs satisfied by previous outputs - ✅ Error handling and rollback procedures defined - ✅ Progress tracking and notifications configured

    Related Patterns: - ALL PATTERNS: DVM orchestrates all other patterns - Commonly coordinates: P02+P03 (planning), P06+P07+P08 (quality), P09+P10+P14 (operations)

    P16: RAG Engineering (RAG)¶

    Purpose: Design, implement, and optimize Retrieval-Augmented Generation (RAG) pipelines for document search and context retrieval.

    When to Use: - Building document ingestion pipelines - Implementing semantic search and chunking strategies - Optimizing embedding models and vector stores - Troubleshooting retrieval quality issues

    Key Commands: 

    # Design RAG pipeline
gh copilot suggest "Design RAG pipeline for legal documents with citation tracking"

# Optimize chunking
gh copilot suggest "Optimize chunk size and overlap for 100-page PDF documents"

# Troubleshoot retrieval
gh copilot explain "Why is semantic search returning irrelevant results?"

    Common Prompts: - "Create ingestion pipeline for 10,000 PDF documents with metadata extraction" - "Implement hybrid search combining BM25 and vector similarity" - "Optimize chunk size (current: 1000 tokens) to improve retrieval precision" - "Setup Azure AI Search with semantic ranking and custom skills"

    Inputs: - Source documents (PDFs, DOCX, HTML) - Metadata schemas (author, date, category) - User queries and search patterns - Performance requirements (latency, precision, recall)

    Outputs: - Ingestion pipelines (scripts/ingestion/) - Vector store configurations (Azure AI Search, Pinecone) - Chunking and embedding strategies - Retrieval quality metrics (MRR, NDCG)

    Quality Gates: - ✅ Ingestion throughput ≥100 docs/minute - ✅ Retrieval latency <500ms (P95) - ✅ Precision@10 ≥0.8 - ✅ Citation accuracy 100% (links to source chunks) - ✅ Embedding model versioned and reproducible

    Related Patterns: - P10-MET: Metrics track retrieval quality - P11-SEC: Security ensures data access controls - P21-KNO: Knowledge management structures content

    P17: AI Safety (SAF)¶

    Purpose: Implement content moderation, jailbreak prevention, and responsible AI guardrails.

    When to Use: - Filtering harmful or inappropriate content - Preventing prompt injection and jailbreak attacks - Implementing content moderation pipelines - Auditing AI outputs for bias and fairness

    Key Commands: 

    # Content moderation

    Common Workflows¶

    Feature Development (Happy Path)¶

    P02-REQ → P01-ADR → P05-SCA → P07-TST → P06-REV → P08-CICD → P14-LVO
    1. P02-REQ: Refine epic into user stories with acceptance criteria 2. P01-ADR: Document architectural decisions (if needed) 3. P05-SCA: Scaffold new module or component 4. P07-TST: Generate tests from acceptance criteria 5. P06-REV: Review code for quality and security 6. P08-CICD: Deploy via CI/CD pipeline 7. P14-LVO: Monitor production deployment

    Sprint Planning & Execution¶

    P02-REQ → P03-SCR → P15-DVM → P10-MET → P03-SCR
    1. P02-REQ: Refine backlog items 2. P03-SCR: Plan sprint with capacity and priorities 3. P15-DVM: Coordinate work assignments across patterns 4. P10-MET: Track progress and burn-down 5. P03-SCR: Conduct retrospective and plan next sprint

    Security & Compliance Pipeline¶

    P11-SEC → P17-SAF → P19-GOV → P08-CICD → P10-MET
    1. P11-SEC: Scan code for vulnerabilities 2. P17-SAF: Test content moderation and safety filters 3. P19-GOV: Validate compliance controls (AIIA, PIA) 4. P08-CICD: Gate deployment on security approval 5. P10-MET: Monitor security metrics and alerts

    RAG Pipeline Development¶

    P21-KNO → P16-RAG → P17-SAF → P18-MCP → P10-MET
    1. P21-KNO: Curate and structure source documents 2. P16-RAG: Design ingestion and retrieval pipeline 3. P17-SAF: Add content moderation and safety filters 4. P18-MCP: Expose RAG as MCP tool for agents 5. P10-MET: Track retrieval quality and latency

    Incident Response¶

    P10-MET → P14-LVO → P09-RUN → P11-SEC → P03-SCR
    1. P10-MET: Alert detects anomaly (e.g., latency spike) 2. P14-LVO: On-call engineer triages incident 3. P09-RUN: Execute runbook for mitigation 4. P11-SEC: Assess security implications (if applicable) 5. P03-SCR: Document in retrospective and create action items

    Knowledge Base Refresh¶

    P04-LIB → P21-KNO → P16-RAG → P07-TST → P10-MET
    1. P04-LIB: Update documentation and guides 2. P21-KNO: Curate and version knowledge base 3. P16-RAG: Re-ingest and re-index content 4. P07-TST: Test retrieval quality with sample queries 5. P10-MET: Track search metrics (precision, recall)
