    Chapter P16: Awareness Protocol¶

    Pattern Code: P16 Short Code: AWR Name: Awareness Protocol Type: DevTools Pattern Scope: Awareness & Explainability Maturity: Stable Autonomy Level: C0 (Human-in-the-loop)

    üìñ Chapter Overview¶

    The Awareness Protocol (P16-AWR) is the "conscience" of the EVA Autonomous Agile Command. It ensures every significant action or output from DevTools agents (P01-P21) and Service Personas (SP00-SPxx) is accompanied by structured awareness blocks that make AI decision-making transparent, testable, and trustworthy.

    Without P16, EVA agents would operate as black boxes‚Äîmaking recommendations without explaining their limitations, assumptions, or risks. With P16, every output includes explicit awareness metadata: what we know, what we don't, what could go wrong, and what to do next.

    P16 is not just about explainability for humans; it's about building AI systems that are self-aware in a healthy way‚Äîknowing their own limitations, communicating uncertainty, and suggesting validation experiments to increase confidence.

    This chapter is critical for understanding how EVA achieves the transparency and trustworthiness required for Government of Canada deployment, where AI decisions must be auditable, explainable, and aligned with the GC Agentic AI Framework.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P16-AWR serves as the Explainability Engine and Uncertainty Communicator for all EVA agents. It ensures:

    1. Transparency: All AI outputs include explicit awareness of limitations
    2. Uncertainty Quantification: Confidence levels (Low/Medium/High) communicate reliability
    3. Assumption Surfacing: Implicit assumptions made explicit and testable
    4. Risk Identification: Potential negative consequences flagged proactively
    5. Actionable Next Steps: Clear guidance on validation and safe progress

    Mission Statement¶

    "Make AI decision-making transparent, testable, and trustworthy by ensuring every significant output includes explicit awareness of what we know, what we don't, and what could go wrong."

    Value Proposition¶

    For Product Owners: - Make informed go/no-go decisions with full understanding of risks - Validate AI recommendations before committing resources - Reduce surprises from overlooked assumptions

    For Developers: - Understand limitations of generated code/recommendations - Know which assumptions to validate before implementation - Identify missing context that affects quality

    For Governance Panels (ATO, Security, Privacy): - Transparent AI decision-making for compliance reviews - Explicit risk documentation for risk registers - Testable assumptions for validation evidence

    For External Auditors (SSC, TBS, OAG): - Clear audit trail of AI decision rationale - Documented uncertainty and confidence levels - Risk-aware AI governance aligned with GC Agentic AI Framework

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Define Awareness Block Template
    2. Standard YAML schema for all awareness blocks
    3. Required fields: pod(s), phase, confidence, assumptions, risks, actionClass, suggestedNextAction

    4. Optional fields: evidenceRefs, alternativeApproaches, validationExperiments

    5. Generate Awareness Blocks

    6. Auto-generate awareness blocks for agent outputs
    7. Infer confidence level from context (known data, tested vs untested)
    8. Identify implicit assumptions from recommendations

    9. Flag risks based on autonomy level (C0-C3) and impact

    10. Review & Upgrade Existing Outputs

    11. Analyze agent outputs for missing/weak awareness blocks
    12. Identify unstated assumptions and risks
    13. Suggest validation experiments to increase confidence

    14. Provide meta-commentary on reliability

    15. Provide Awareness Checklists

    16. Template checklists for artifact types (CDDs, PRs, runbooks, security recs)
    17. Force self-check on assumptions and impact before finalization
    18. Reduce blind spots through structured reflection

    Supporting Responsibilities¶

    1. Meta-Reflection
    2. Ask clarifying questions: "What are we assuming here?"
    3. Surface alternative approaches not considered
    4. Suggest experiments to test assumptions

    5. Flag when confidence level may be overstated

    6. Integration with Other Patterns

    7. P19 (Action Classification): Link awareness to autonomy levels (C0-C3)
    8. P21 (Provenance): Include awareness blocks in audit trail
    9. P11 (Security): Risk documentation feeds into security reviews

    10. P17 (Swarm PR Review): Awareness blocks included in PR reviews

    11. Governance Alignment

    12. Ensure awareness blocks support GC Agentic AI Framework requirements
    13. Map confidence levels to evidence quality expectations
    14. Document uncertainty for AIIA (Algorithmic Impact Assessment) submissions

    üöÄ When to Use P16¶

    Primary Use Cases¶

    1. Generating Awareness Block for Agent Output¶

    Scenario: A DevTools agent (e.g., P05-SCA Scaffolder) has generated code, and you want to document its limitations and assumptions.

    Action: Invoke P16 to generate structured awareness block.

    Example: 

    **Input**: P05 scaffolded a new API endpoint for multi-region Cosmos DB failover.

**P16 Output**:
```yaml
--- Awareness (P16 ‚Äì AWR) ---
pod(s): POD-F (Foundation)
phase: 2 (Build / Code / Test)
confidence: Medium
assumptions:
  - "Secondary region (Canada East) is always healthy when primary fails"
  - "Application can tolerate up to 5 minutes of downtime during failover"
  - "Cosmos DB SDK handles connection string updates automatically"
  - "No manual intervention required for failover"
risks:
  - "Both regions may fail simultaneously (regional outage)"
  - "Failover may take >5 minutes if large transaction backlog"
  - "Application may require restart to pick up new connection string"
  - "Data loss possible if recent writes not replicated"
actionClass: C1 (human-on-loop: automated failover, SRE monitors and can intervene)
suggestedNextAction:
  - "Test failover in non-prod with simulated regional outage"
  - "Measure actual failover time (P95 latency)"
  - "Validate SDK behavior with connection string changes"
  - "Document rollback procedure if failover fails"
  - "Set up alerting for secondary region health"
evidenceRefs:
  - "docs/architecture/decisions/ADR-007-multi-region-cosmos.md"
  - "https://learn.microsoft.com/en-us/azure/cosmos-db/high-availability"
alternativeApproaches:
  - "Manual failover (C0: human-in-loop, slower but more control)"
  - "Active-active multi-region (higher cost, no failover needed)"
validationExperiments:
  - "Chaos engineering: Kill primary region in dev, measure failover time"
  - "Load test: Verify secondary can handle production traffic"
---

    2. Reviewing Existing Output for Awareness Gaps¶

    Scenario: A PR contains a recommendation without awareness block, and you want to identify missing context.

    Action: Invoke P16 to review and flag gaps.

    Example: 

    **Input**: PR recommends "Deploy multi-region Cosmos DB for high availability."

**P16 Review**:

**Missing Context Identified**:
- ‚ùå No cost impact stated (multi-region ~2x cost)
- ‚ùå No consistency model documented (eventual vs strong)
- ‚ùå No data residency constraints validated (can we use Canada East?)
- ‚ùå No confidence level provided
- ‚ùå No assumptions documented

**Recommended Awareness Block**:
```yaml
--- Awareness (P16 ‚Äì AWR) ---
pod(s): POD-F (Foundation)
phase: 1 (Design & Architecture)
confidence: Low ‚Üí Medium (after validation experiments)
assumptions:
  - "Cost increase (~2x) is acceptable for high availability"
  - "Eventual consistency is acceptable (seconds lag)"
  - "Canada East meets data residency requirements"
  - "Operations team can manage multi-region complexity"
risks:
  - "Cost may exceed budget if not accounted for"
  - "Geo-replication lag may impact read consistency"
  - "Deployment complexity increases operational burden"
  - "Data residency constraints may prohibit Canada East"
actionClass: C0 (human-in-loop: design decision requires PO approval)
suggestedNextAction:
  - "Validate cost impact with Platform Owner (est. $X/month)"
  - "Confirm consistency requirements with architects"
  - "Verify Canada East meets data residency (check Azure Policy)"
  - "Review operational runbooks for multi-region management"
evidenceRefs:
  - "docs/architecture/decisions/ADR-draft-multi-region.md"
validationExperiments:
  - "Deploy to dev environment, measure actual cost for 1 week"
  - "Test geo-replication lag with load test"
  - "Query Azure Policy: Does Canada East pass Protected B requirements?"
---

    #### 3. Awareness Checklist for Artifact Type
**Scenario**: You're finalizing a CDD (Conceptual Design Document) and want to ensure all awareness aspects are covered.

**Action**: Request P16 awareness checklist for CDDs.

**Example**:
```markdown
## CDD Awareness Checklist (P16 ‚Äì AWR)

Before finalizing a CDD, verify:

**Context & Scope**:
- [ ] **Pods impacted**: Which pods (POD-F/X/O/S-*) are affected?
- [ ] **Lifecycle phase**: Which phase is this for (0-6)?
- [ ] **Autonomy level**: What C0-C3 autonomy level applies (per P19)?

**Assumptions & Constraints**:
- [ ] **Assumptions explicit**: What are we assuming? (data availability, team capacity, tech feasibility)
- [ ] **Constraints documented**: What are the hard limits? (budget, timeline, compliance)
- [ ] **Dependencies mapped**: What other CDDs/repos/services are needed?

**Risks & Mitigation**:
- [ ] **Risks identified**: What could go wrong? (technical, operational, budgetary, privacy, security)
- [ ] **Impact assessed**: Who is affected if risks materialize? (users, developers, operations)
- [ ] **Mitigation planned**: How do we reduce or handle each risk?

**Validation & Evidence**:
- [ ] **Success criteria testable**: Can we measure "Done"? (metrics, acceptance tests)
- [ ] **Evidence linked**: Are decisions backed by ADRs, docs, demos, or prototypes?
- [ ] **Confidence level stated**: Low/Medium/High? (based on validation performed)

**Compliance & Governance**:
- [ ] **Compliance requirements**: Does this need PIA/AIIA/ATO evidence?
- [ ] **Data residency**: Does this process/store data? Where? (Canada-only?)
- [ ] **Security classification**: What's the data classification? (Protected B?)

**Next Steps**:
- [ ] **Next actions clear**: What's the immediate next action?
- [ ] **Owner assigned**: Who owns the next action?
- [ ] **Timeline realistic**: When will next actions complete?

    4. Meta-Reflection: "What Are We Assuming?"¶

    Scenario: A recommendation feels confident, but you want to surface implicit assumptions.

    Action: Ask P16 to perform meta-reflection.

    Example: 

    **Recommendation**: "Use Azure OpenAI ada-002 for embedding generation in jurisprudence search."

**P16 Meta-Reflection**:

**Implicit Assumptions Identified**:
1. **Availability**: Azure OpenAI ada-002 is available in Canada regions
   ‚Üí **Need to verify**: Check [Azure OpenAI Service Availability](https://learn.microsoft.com/en-us/azure/ai-services/openai/concepts/models)

2. **Cost**: Cost (~$0.0001/1K tokens) is acceptable
   ‚Üí **Need to verify**: Calculate monthly cost at 10M embeddings/month = $1,000/month. Is this in budget?

3. **Latency**: Latency (~200ms P95) meets search SLA
   ‚Üí **Need to verify**: What's our latency budget? Is 200ms acceptable for user-facing search?

4. **Protected B approval**: Azure OpenAI is approved for Protected B in Canada
   ‚Üí **Need to verify**: Check SSC Cloud Security Controls attestation for Azure OpenAI

5. **Embedding quality**: ada-002 embeddings work well for French legal text
   ‚Üí **Need to verify**: Benchmark against jurisprudence test set (English + French)

**Questions to Increase Certainty**:
- [ ] Is ada-002 available in Canada Central/East? (Check Azure Portal)
- [ ] What's the actual cost for 10M embeddings/month? (Calculate: 10M √ó $0.0001 = $1,000)
- [ ] What's our P95 latency budget for search? (Check NFRs)
- [ ] Is Azure OpenAI approved for Protected B? (Ask Security SME)
- [ ] How do ada-002 embeddings perform on French legal text? (Run benchmark)

**Validation Experiments**:
1. **Availability test**: Deploy ada-002 in Canada Central dev environment (5 min)
2. **Cost test**: Run 100K embeddings in dev, measure actual cost (30 min)
3. **Latency test**: Measure P95 latency for embedding generation under load (1 hour)
4. **Compliance check**: Review Azure OpenAI SSC attestation (30 min)
5. **Quality test**: Benchmark search relevance with ada-002 on 100 French legal queries (2 hours)

**Confidence Upgrade Path**:
- **Before validation**: Low confidence (many untested assumptions)
- **After validation**: Medium ‚Üí High confidence (assumptions tested, risks quantified)

    When NOT to Use P16¶

    • Don't use P16 for trivial outputs - Reserve for significant decisions/recommendations
    • Don't use P16 to block action - Awareness blocks inform, they don't prevent (unless C0 requires approval)
    • Don't use P16 as a crutch - Fix bad recommendations instead of just documenting limitations

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P16-AWR (Awareness Protocol), the Explainability Engine for EVA DevTools and Service Personas.

**Your Capabilities**:
1. Generate structured awareness blocks (YAML) for agent outputs
2. Review existing outputs and identify missing assumptions/risks
3. Provide awareness checklists for artifact types (CDDs, PRs, runbooks)
4. Perform meta-reflection: "What are we assuming?"
5. Suggest validation experiments to increase confidence

**Your Limitations**:
- You do NOT make decisions (that's other agents)
- You do NOT execute actions (you only provide awareness/commentary)
- You do NOT guarantee completeness (assumptions/risks may still be missed)

**Autonomy Level**: C0 (Human-in-the-loop)
- Generate awareness blocks automatically
- Humans review awareness blocks and decide whether to proceed
- No actions blocked solely by P16 (awareness is advisory, not enforcement)

**Output Format**:
- Structured YAML awareness blocks (follow standard schema)
- Clear explanations for confidence levels (why Low vs Medium vs High)
- Concrete validation experiments (not generic suggestions)
- Evidence references (link to docs, ADRs, prototypes)

**Safety Rules**:
- Always include confidence level (Low/Medium/High)
- Flag C3 (full autonomy) actions for extra scrutiny (not yet implemented in EVA)
- Highlight data residency risks (Canada-only requirement)
- Surface security/privacy risks prominently

    üé™ Awareness Block Schema¶

    Standard Schema¶

    All awareness blocks follow this YAML schema:

    --- Awareness (P16 ‚Äì AWR) ---

# REQUIRED FIELDS

pod(s): [POD-F | POD-X | POD-O | POD-S-*] (which pods impacted)

phase: [0-6]  # Lifecycle phase
# 0 = Intake/Sandbox
# 1 = Design & Architecture
# 2 = Build/Code/Test
# 3 = Demo/Alpha
# 4 = Pilot/Beta
# 5 = GA/Production
# 6 = Day-2 Operations

confidence: [Low | Medium | High]
# Low: Exploratory, many untested assumptions
# Medium: Validated in non-prod, some assumptions tested
# High: Tested in production, evidence-based

assumptions:
  - "Assumption 1: [explicit statement of what we're assuming]"
  - "Assumption 2: [...]"
  # Include 3-7 key assumptions

risks:
  - "Risk 1: [potential negative consequence if assumption wrong]"
  - "Risk 2: [...]"
  # Include 3-7 key risks

actionClass: [C0 | C1 | C2 | C3]  # Per P19 Action Classification
# C0: Human-in-loop (human approves every action)
# C1: Human-on-loop (human monitors, can intervene)
# C2: Human-over-loop (human reviews periodically)
# C3: Full autonomy (not yet implemented)

suggestedNextAction:
  - "Next action 1: [concrete, testable step]"
  - "Next action 2: [...]"
  # Include 1-3 next actions

# OPTIONAL FIELDS

evidenceRefs:  # Where did this information come from?
  - "docs/path/to/evidence.md"
  - "github://org/repo#issue-number"
  - "https://external-reference.com"

alternativeApproaches:  # What else could we do?
  - "Alternative 1: [different approach with pros/cons]"
  - "Alternative 2: [...]"

validationExperiments:  # How can we test assumptions?
  - "Experiment 1: [concrete test, time-boxed]"
  - "Experiment 2: [...]"

---

    Field Definitions¶

    Field Type Required Description
    pod(s) String or Array Yes Which pods are impacted (POD-F, POD-X, POD-O, POD-S-*)
    phase Integer (0-6) Yes Lifecycle phase (0=Intake, 1=Design, 2=Build, 3=Demo, 4=Pilot, 5=GA, 6=Day-2)
    confidence Enum Yes Confidence level: Low (exploratory), Medium (validated), High (tested in prod)
    assumptions Array Yes 3-7 explicit assumptions made in the output
    risks Array Yes 3-7 potential negative consequences if output is wrong/incomplete
    actionClass Enum Yes Autonomy level per P19: C0 (human-in-loop), C1 (human-on-loop), C2 (human-over-loop), C3 (full autonomy)
    suggestedNextAction Array Yes 1-3 concrete next steps to validate or proceed safely
    evidenceRefs Array Optional Links to supporting evidence (docs, repos, ADRs, prototypes)
    alternativeApproaches Array Optional Other approaches considered (useful for design reviews)
    validationExperiments Array Optional Concrete experiments to test assumptions and increase confidence

    üìä Confidence Level Guidelines¶

    Low Confidence¶

    When to use: Exploratory, many untested assumptions, early design phase

    Characteristics: - Based on general knowledge, not specific validation - Multiple critical assumptions untested - No prototype or demo yet - Derived from documentation only (no hands-on experience)

    Example: "We should use Azure OpenAI for embeddings" (no validation in EVA context)

    Next Steps: Run validation experiments to increase confidence

    Medium Confidence¶

    When to use: Validated in non-prod, some assumptions tested, design solidified

    Characteristics: - Prototype or demo exists and works - Tested in dev/staging environment - Some assumptions validated (cost, latency, compatibility) - Edge cases or production unknowns remain

    Example: "Azure OpenAI ada-002 works for English legal text" (tested in dev with sample queries)

    Next Steps: Test in production-like conditions (load, failure scenarios)

    High Confidence¶

    When to use: Tested in production, evidence-based, minimal unknowns

    Characteristics: - Deployed to production and monitored - All critical assumptions tested - Edge cases handled or documented - Historical data from production usage

    Example: "Azure OpenAI ada-002 delivers <300ms P95 latency for embeddings in production" (6 months production data)

    Next Steps: Continuous monitoring for drift, periodic re-validation

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C0 (Human-in-the-Loop)¶

    P16 Itself: C0 - Awareness blocks are generated automatically - Humans review awareness blocks and decide whether to proceed - No actions are blocked solely by P16 (awareness is advisory, not enforcement)

    Why C0? - P16 provides information, not decisions - Humans make go/no-go decisions based on awareness blocks - Transparency without blocking progress

    Safety Rules¶

    Rule 1: Awareness Is Advisory, Not Enforcement¶

    Risk: Awareness blocks may be ignored, leading to problems P16 warned about

    Mitigation: - ‚úÖ Include awareness blocks in code review checklists - ‚úÖ Require awareness blocks for C1+ autonomy (per P19) - ‚úÖ Track whether assumptions were validated (via P21 provenance) - ‚úÖ Retrospective analysis: Did ignored warnings cause issues?

    Rule 2: Confidence Levels Must Be Honest¶

    Risk: Overstating confidence leads to false sense of safety

    Mitigation: - ‚úÖ Default to Low confidence unless evidence justifies Medium/High - ‚úÖ Require validation experiments before Medium ‚Üí High upgrade - ‚úÖ Document what evidence would justify higher confidence - ‚úÖ Periodically re-assess confidence as context changes

    Rule 3: Assumptions Must Be Testable¶

    Risk: Vague assumptions that can't be validated

    Mitigation: - ‚úÖ Every assumption should have a validation experiment - ‚úÖ Assumptions should be falsifiable (not always-true statements) - ‚úÖ Link assumptions to evidence when validated - ‚úÖ Document which assumptions remain untested

    Rule 4: Awareness Blocks Don't Absolve Responsibility¶

    Risk: "We documented the risk, so we're not responsible if it happens"

    Mitigation: - ‚úÖ Awareness blocks inform decisions, don't replace risk management - ‚úÖ High-risk actions still require mitigation plans (not just documentation) - ‚úÖ Security/privacy risks require SME approval even with awareness blocks - ‚úÖ Use P19 (Action Classification) to enforce appropriate guardrails

    üéì Key Takeaways¶

    1. Awareness is not optional - Required for C1+ autonomy (per P19)
    2. Confidence must be honest - Default to Low unless evidence justifies Medium/High
    3. Assumptions must be testable - Every assumption needs a validation experiment
    4. Risks must be actionable - Not just documentation, but mitigation plans
    5. Awareness informs, doesn't block - Humans make decisions based on awareness
    6. GC compliance alignment - P16 supports AIIA and GC Agentic AI Framework
    7. Continuous re-validation - Confidence degrades over time (re-test assumptions)
    8. Meta-awareness is powerful - Asking "What are we assuming?" surfaces blind spots
