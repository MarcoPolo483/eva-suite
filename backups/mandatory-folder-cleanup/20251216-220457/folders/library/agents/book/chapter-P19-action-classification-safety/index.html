    Chapter P19: Action Classification & Safety¶

    Pattern Code: P19 Short Code: ACS Name: Action Classification & Safety Type: DevTools Pattern Scope: Safety & Autonomy Governance Maturity: Concept Autonomy Level: C0 (Human-in-the-loop)

    üìñ Chapter Overview¶

    Action Classification & Safety (P19-ACS) is the governance backbone of the EVA Autonomous Agile Command. It implements the Government of Canada Agentic AI Framework by classifying every agent action into one of four autonomy levels (C0-C3) and enforcing appropriate safety guardrails for each level.

    Without P19, EVA agents would operate with ambiguous autonomy‚Äîteams wouldn't know which actions require human approval (C0), monitoring (C1), periodic review (C2), or are fully autonomous (C3). With P19, every action is explicitly classified, safety guardrails are enforced, and compliance evidence is automatically generated for ATO (Authority to Operate) and AIIA (Algorithmic Impact Assessment) submissions.

    P19 is not just about documentation‚Äîit's about building AI systems that are safe by design. By forcing explicit autonomy classification and appropriate controls for each level, P19 ensures EVA operates within GC policy boundaries and societal expectations for responsible AI.

    This chapter is critical for understanding how EVA achieves compliance with TBS (Treasury Board Secretariat) and SSC (Shared Services Canada) requirements for AI governance.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P19-ACS serves as the Autonomy Classifier and Safety Guardrail Enforcer for all EVA agents. It ensures:

    1. Explicit Autonomy Classification: Every action labeled C0/C1/C2/C3
    2. Appropriate Safety Guardrails: Controls enforced per autonomy level
    3. Human Oversight Mapping: Clear triggers for human intervention
    4. Risk Assessment: High-risk actions flagged for additional validation
    5. Compliance Evidence: AIIA and ATO documentation auto-generated

    Mission Statement¶

    "Enable safe AI autonomy by classifying every action's autonomy level (C0-C3) and enforcing appropriate safety guardrails aligned with the GC Agentic AI Framework."

    Value Proposition¶

    For Product Owners: - Make informed go/no-go decisions with clear autonomy levels - Reduce AI risk through appropriate safety controls - Demonstrate responsible AI governance to stakeholders

    For Developers: - Understand what level of autonomy their code operates at - Know what safety guardrails are required - Build compliant AI systems by default

    For Governance Panels (ATO, Security, Privacy): - Clear classification for every AI action - Evidence of appropriate controls - Alignment with GC Agentic AI Framework

    For External Auditors (SSC, TBS, OAG): - Transparent autonomy governance - Documented safety guardrails - Compliance with GC AI policy requirements

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Define Autonomy Levels (C0-C3)
    2. C0: Human-in-the-loop (human approves every action)
    3. C1: Human-on-the-loop (human monitors, can intervene)
    4. C2: Human-over-the-loop (human reviews periodically)

    5. C3: Full autonomy (human sets boundaries only)

    6. Classify Actions by Autonomy Level

    7. Analyze action characteristics (approval needed? monitoring? review cadence?)
    8. Assign autonomy level based on GC Agentic AI Framework criteria

    9. Document rationale for classification

    10. Define Safety Guardrails per Level

    11. C0: Human approval UI, no auto-execution
    12. C1: Real-time monitoring, pause/stop mechanisms, 24-hour review
    13. C2: Strict constraints, auto-rollback, weekly/monthly review

    14. C3: Extensive safety validation, governance panel approval, continuous monitoring

    15. Validate Guardrails Implementation

    16. Check if required controls are implemented
    17. Block actions if guardrails missing (BLOCKER)
    18. Provide implementation guidance

    Supporting Responsibilities¶

    1. Risk Assessment
    2. Assess risk factors (cost, data, availability, autonomy level)
    3. Calculate overall risk (low/medium/high)

    4. Identify required approvals (Platform Owner, Data Steward, Security SME)

    5. Human Oversight Mapping

    6. Define when human approval required (C0)
    7. Define monitoring requirements (C1: real-time dashboard, alerts)
    8. Define review cadence (C2: weekly/monthly)

    9. Define boundary setting (C3: quarterly review)

    10. Compliance Documentation

    11. Generate AIIA evidence (autonomy levels, safety controls)
    12. Generate ATO evidence (governance alignment)
    13. Track autonomy level changes over time (via P21)

    üöÄ Autonomy Levels (C0-C3)¶

    Per GC Agentic AI Framework:

    C0: Human-in-the-Loop¶

    Description: AI suggests, human approves every action

    Human Role: Active decision-maker for each action

    Examples: - P06 Review Agent (suggests changes, human merges) - SP02 ATO Auditor (generates report, human decides) - P02 Requirements Engine (suggests user stories, PO approves)

    Safety Guardrails: - Human approval required before execution - Clear explanation of proposed action - Explicit accept/reject mechanism - No time-based auto-approval

    When to Use C0: - High-impact decisions (architecture, budget, compliance) - Novel situations (first-time actions) - Low confidence in AI recommendation - High cost or risk

    Example (P06 Review Agent): 

    action:
  name: "Merge PR #123"
  agent: "P06-REV (Review Agent)"
  human_approval: true  # Required for C0
  approval_ui: "GitHub PR merge button"
  explanation: "Code review passed: security ‚úÖ, tests ‚úÖ, docs ‚úÖ"

autonomy_classification:
  level: "C0"
  rationale: "Merging code to production has high impact (affects all users). Human judgment required for context-specific decisions (business logic, architecture)."

safety_guardrails:
  - "Human reviews PR findings before merge"
  - "Human validates test coverage (80%+)"
  - "Human confirms no breaking changes"
  - "No auto-merge (even if all checks pass)"

    C1: Human-on-the-Loop¶

    Description: AI acts autonomously, human monitors and can intervene

    Human Role: Monitor, can pause/stop/override

    Examples: - P04 Librarian (auto-updates docs, human reviews) - P10 Metrics Agent (auto-generates reports) - P15 Dev Master Orchestrator (auto-coordinates agents, human can override)

    Safety Guardrails: - Real-time monitoring dashboard - Clear pause/stop mechanism - Alert on anomalous behavior - Human review within 24 hours

    When to Use C1: - Routine operations with proven track record - Medium confidence in AI recommendations - Medium cost/risk - Human expertise available for intervention

    Example (P15 Dev Master Orchestrator): 

    action:
  name: "Execute new-feature workflow"
  agent: "P15-DVM (Dev Master Orchestrator)"
  human_approval: false  # Not required (C1)
  human_monitoring: true  # Real-time monitoring

autonomy_classification:
  level: "C1"
  rationale: "Workflow automates multi-step process (P02 ‚Üí P05 ‚Üí P07 ‚Üí P06). Human monitors progress and can pause/stop if issues detected."

safety_guardrails:
  - "Real-time dashboard shows workflow progress"
  - "Human can pause workflow at any step"
  - "Alert if step takes >2x expected time"
  - "Alert if any step fails"
  - "Human reviews final PR before merge (C0)"

monitoring_requirements:
  - dashboard: "tools/jarvis/dashboard.html"
  - alerts:
      - "Workflow step failure"
      - "Workflow duration >30 minutes"
  - review_cadence: "Within 24 hours of workflow completion"

    C2: Human-over-the-Loop¶

    Description: AI acts autonomously with constraints, human reviews periodically

    Human Role: Review periodically (weekly/monthly), adjust constraints

    Examples: - P07 Testing Agent (generates tests, runs in CI) - SP01 Data Ingestion (auto-processes approved sources) - P08 CI/CD Guardian (auto-optimizes pipelines)

    Safety Guardrails: - Strict constraints (approved sources, safe operations) - Periodic human review (weekly/monthly) - Automatic rollback on failure - Audit trail for all actions (P21)

    When to Use C2: - Highly routine operations - High confidence in AI (tested in production) - Low risk (constrained scope) - Clear rollback procedures

    Example (SP01 Data Ingestion): 

    action:
  name: "Ingest Jurisprudence Q4 2025"
  agent: "SP01 (Data Ingestion)"
  human_approval: false  # Not required (C2)
  human_monitoring: false  # No real-time monitoring
  constraints:
    - "Only ingest from approved sources (SharePoint jurisprudence site)"
    - "Only index Supreme Court decisions (filter other courts)"
    - "Auto-rollback if ingestion errors >5%"
    - "Cost budget: $500/month (auto-pause if exceeded)"

autonomy_classification:
  level: "C2"
  rationale: "Data ingestion from approved source with strict constraints. Proven track record (6 months production use). Low risk (read-only operation)."

safety_guardrails:
  - "Constraint: Only approved sources (SharePoint URL whitelist)"
  - "Constraint: Only Supreme Court decisions (filter rule)"
  - "Auto-rollback if error rate >5%"
  - "Auto-pause if cost exceeds $500/month"
  - "Audit trail for all ingestions (P21)"
  - "Weekly human review of ingestion logs"

review_cadence:
  frequency: "Weekly"
  reviewer: "Data Steward"
  review_artifacts:
    - "Ingestion logs (success/failure counts)"
    - "Cost report (actual vs budget)"
    - "Quality metrics (duplicate rate, PII detection rate)"

    C3: Full Autonomy¶

    Description: AI acts fully autonomously within well-defined boundaries

    Human Role: Set boundaries, review outcomes quarterly

    Examples: - Not yet implemented in EVA Suite - Requires extensive safety validation - Requires governance panel approval

    Safety Guardrails: - Extensive safety validation required - Governance panel approval mandatory - Continuous monitoring with automatic kill switch - Quarterly human review of outcomes

    When to Use C3: - Highly mature AI systems (years of production use) - Very high confidence (proven at scale) - Very low risk (no user/data/cost impact possible) - Clear boundaries and kill switch

    Example (Hypothetical - Auto-Scaling): 

    action:
  name: "Auto-scale Cosmos DB based on load"
  agent: "Hypothetical Auto-Scaler"
  human_approval: false
  human_monitoring: false
  boundaries:
    - "Min: 400 RU/s (never scale below)"
    - "Max: 10,000 RU/s (never scale above)"
    - "Cost limit: $2,000/month (kill switch)"
    - "Only scale during business hours (9 AM - 5 PM EST)"

autonomy_classification:
  level: "C3"
  rationale: "Auto-scaling proven at scale (2 years production use). Strict boundaries prevent runaway cost. Kill switch at $2,000/month."

safety_guardrails:
  - "Boundary: Min 400 RU/s, Max 10,000 RU/s"
  - "Boundary: Only scale during business hours"
  - "Kill switch: Auto-disable if cost >$2,000/month"
  - "Kill switch: Auto-disable if error rate >10%"
  - "Continuous monitoring (Application Insights)"
  - "Quarterly human review of scaling decisions"
  - "Governance panel approval required (initial + annual)"

approval_requirements:
  - "Governance Panel: Initial approval"
  - "Security SME: Annual re-approval"
  - "Platform Owner: Quarterly review of outcomes"

compliance_requirements:
  - "AIIA: Updated annually"
  - "PIA: Updated if data flows change"
  - "Risk Register: Updated quarterly"

**NOTE**: C3 is not currently implemented in EVA. All patterns are C0-C2.

    üöÄ When to Use P19¶

    Primary Use Cases¶

    1. Classify Action by Autonomy Level¶

    Scenario: New pattern (e.g., P22-Contract-Testing-Agent) needs autonomy classification.

    Action: Use P19 to analyze and classify.

    Example: 

    action:
  name: "Run contract tests on API changes"
  agent: "P22 (Contract Testing Agent)"
  description: "Automatically run contract tests when API schema changes"
  human_approval: false
  human_monitoring: false
  constraints:
    - "Only run tests in CI/CD pipeline (not production)"
    - "Auto-fail build if contracts broken"

**P19 Classification**:
autonomy_level: "C2"
rationale: "Contract tests run automatically in CI with strict constraints (non-prod only). Human reviews test failures weekly. Auto-rollback (fail build) prevents broken contracts from reaching production."

safety_guardrails:
  - "Constraint: Tests run in CI only (not production)"
  - "Constraint: Build fails if contracts broken (prevents merge)"
  - "Audit trail for test results (P21)"
  - "Weekly human review of test failures"

required_validations:
  - "Contract tests tested in dev environment"
  - "False positive rate <5%"
  - "Test execution time <5 minutes"

    2. Validate Safety Guardrails for Autonomy Level¶

    Scenario: Existing C2 action needs validation that required guardrails are implemented.

    Action: Use P19 to audit and flag gaps.

    Example: 

    **Action**: "Auto-deploy multi-region Cosmos DB on primary region failure"

**P19 Validation**:
- ‚úÖ Strict constraints defined (only on primary region failure)
- ‚úÖ Automatic rollback on failure (restore to primary)
- ‚ö†Ô∏è Human review schedule: Missing (should be weekly)
- ‚ùå Audit trail enabled: Missing (P21 not configured)
- ‚ùå Real-time alerting: Missing (SRE not notified)
- ‚ùå Runbook for manual override: Missing

**Result**: **Action cannot be classified C2 until gaps resolved**

**Required Actions**:
1. Set up weekly review of failover logs (assign Data Steward)
2. Enable P21 audit trail for failover events
3. Configure alerting (SRE notified immediately on failover)
4. Write runbook for manual rollback (P09-RUN)

    3. Risk Assessment for High-Risk Actions¶

    Scenario: Proposed action has high cost/data/availability impact.

    Action: Use P19 to assess risk and identify required approvals.

    Example: 

    action:
  name: "Increase Azure OpenAI quota from 100K ‚Üí 1M tokens/min"
  cost_impact: "$10,000/month increase"
  data_impact: "none"
  availability_impact: "improved (reduce rate limit errors)"
  autonomy_level: "C1"

**P19 Risk Assessment**:
overall_risk: "medium"
risk_factors:
  - factor: "cost_impact"
    severity: "high"
    rationale: "$10,000/month is 20% of monthly AI budget"
    mitigation: "Cost alert at $8,000/month (80% threshold)"
  - factor: "autonomy_level"
    severity: "medium"
    rationale: "C1 autonomy requires real-time monitoring"
    mitigation: "Real-time dashboard for token usage, SRE can pause"

required_approvals:
  - "Platform Owner (budget impact >$1,000/month)"
  - "Finance Team (budget reallocation needed)"

monitoring_requirements:
  - "Alert: Token usage >800K/min (80% of new quota)"
  - "Alert: Cost >$8,000/month (80% of new budget)"
  - "Dashboard: Real-time token usage by API endpoint"
  - "Weekly review: Actual usage vs forecast"

    When NOT to Use P19¶

    • Don't use P19 for read-only documentation - Informational docs don't need autonomy classification
    • Don't use P19 to bypass security review - Classification doesn't replace SME review
    • Don't use P19 for one-off actions - Reserve for reusable patterns with long-term impact

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P19-ACS (Action Classification & Safety), the Autonomy Classifier and Safety Guardrail Enforcer for EVA.

**Your Capabilities**:
1. Classify actions by autonomy level (C0/C1/C2/C3 per GC Agentic AI Framework)
2. Define required safety guardrails for each level
3. Validate that guardrails are implemented
4. Assess risk (low/medium/high) and identify required approvals
5. Generate AIIA/ATO compliance evidence

**Your Limitations**:
- You do NOT execute actions (you only classify and validate)
- You do NOT approve autonomy levels (Security SME has final authority)
- You do NOT guarantee safety (validation is best-effort)

**Autonomy Level**: C0 (Human-in-the-loop)
- Classify actions automatically
- Human Security SME reviews classifications
- Human approves C2/C3 autonomy levels

**Output Format**:
- Autonomy level (C0/C1/C2/C3)
- Rationale (why this level?)
- Safety guardrails (required controls)
- Risk assessment (if applicable)
- Required approvals (if applicable)

**Safety Rules**:
- Default to C0 unless evidence justifies C1/C2/C3
- Require Security SME approval for C2+ autonomy
- Require governance panel approval for C3 autonomy
- Flag data residency violations as BLOCKER

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C0 (Human-in-the-Loop)¶

    P19 Itself: C0 - Classifies actions automatically - Human Security SME reviews classifications - Human approves C2/C3 autonomy levels

    Why C0? - Autonomy classification has governance implications - Incorrect classification could lead to unsafe AI autonomy - Security SME expertise required for final approval

    Safety Rules¶

    Rule 1: Default to C0¶

    Risk: Overstating autonomy leads to unsafe AI

    Mitigation: - ‚úÖ Default to C0 unless evidence justifies C1/C2/C3 - ‚úÖ Require testing in non-prod before C1+ classification - ‚úÖ Require Security SME approval for C2+ autonomy - ‚úÖ Require governance panel approval for C3 autonomy

    Rule 2: Guardrails Are Mandatory¶

    Risk: Classifying action as C1/C2 without implementing guardrails

    Mitigation: - ‚úÖ P19 validation blocks deployment if guardrails missing - ‚úÖ Security scan (P11-SEC) checks for guardrail implementation - ‚úÖ Human reviewer validates guardrails before merge - ‚úÖ Audit trail (P21) tracks whether guardrails were bypassed

    Rule 3: Autonomy Levels Can Degrade¶

    Risk: C2 action becomes riskier over time (e.g., cost increases, errors increase)

    Mitigation: - ‚úÖ Periodic re-classification (quarterly for C2, annually for C3) - ‚úÖ Auto-degrade if metrics worsen (e.g., C2 ‚Üí C1 if error rate >10%) - ‚úÖ Alert if action approaches degradation threshold - ‚úÖ Document degradation triggers in classification

    ‚úÖ End-of-Chapter Checklist¶

    For Action Authors¶

    • I have classified my action's autonomy level (C0/C1/C2/C3)
    • I understand the rationale for this classification
    • I have implemented required safety guardrails
    • I have tested guardrails in non-prod
    • I have documented required approvals
    • I have updated AIIA/PIA if needed

    For Security SMEs¶

    • I have reviewed autonomy classification
    • I agree with classification rationale
    • I have validated safety guardrails are implemented
    • I have identified any missing controls
    • I have approved C2+ autonomy (or flagged for governance panel)

    For Governance Panels¶

    • C3 autonomy has been extensively validated
    • Safety guardrails are comprehensive
    • Kill switch mechanisms are tested
    • Quarterly review cadence is scheduled
    • AIIA and risk register are updated

    üéì Key Takeaways¶

    1. Every action needs autonomy level - C0/C1/C2/C3 classification mandatory
    2. Default to C0 - Prove C1/C2/C3 justified, don't assume
    3. Guardrails are mandatory - Classification without controls is meaningless
    4. Autonomy can degrade - Re-classify quarterly/annually
    5. C3 requires governance approval - Not a team-level decision
    6. AIIA alignment - P19 supports Algorithmic Impact Assessment
    7. Audit trail required - P21 tracks autonomy decisions
    8. Human judgment final - Security SME overrides P19 if needed
