    Chapter P21: Provenance & Fingerprinting¶

    Pattern Code: P21 Short Code: PRO Name: Provenance & Fingerprinting Type: DevTools Pattern Scope: Traceability & Audit Maturity: Concept Autonomy Level: C2 (Human-over-the-loop)

    üìñ Chapter Overview¶

    Provenance & Fingerprinting (P21-PRO) is the audit backbone of the EVA Autonomous Agile Command. It creates immutable fingerprints for every significant action (code changes, deployments, AI decisions, cost allocations) and maintains a 7-year audit trail required for Government of Canada ATO (Authority to Operate) and compliance requirements.

    Without P21, EVA's AI decisions are opaque black boxes‚Äîauditors can't trace why a deployment happened, who approved it, or what data it touched. Cost attribution is manual guesswork. Incident investigations lack forensic evidence. With P21, every action has a cryptographic fingerprint linking it to originating requirements, approving humans, and downstream impacts.

    P21 is not about surveillance‚Äîit's about accountability and explainability. By maintaining provenance chains, P21 enables EVA to answer critical audit questions: Who decided this? Why? What data justified it? What was the outcome?

    This chapter is critical for understanding how EVA achieves ATO compliance and meets TBS/SSC audit requirements.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P21-PRO serves as the Audit Trail Generator and Provenance Chain Maintainer for all EVA operations. It ensures:

    1. Immutable Fingerprints: Every significant action gets a unique, tamper-proof ID
    2. Provenance Chains: Actions linked to originating requirements and approvals
    3. Cost Attribution: Every Azure resource traced to owning product/feature
    4. 7-Year Retention: Audit trail retained per GC requirements
    5. ATO Evidence: Compliance reports auto-generated for auditors

    Mission Statement¶

    "Enable transparent, accountable AI governance by maintaining immutable provenance chains linking every EVA action to originating requirements, approving humans, and downstream outcomes."

    Value Proposition¶

    For Auditors (SSC, TBS, OAG): - Clear audit trail for every AI decision - Cryptographic proof of data lineage - Evidence of human oversight (C0-C3 classifications)

    For Security Teams: - Forensic evidence for incident investigations - Tamper-proof logs for compliance - Data lineage for Protected B handling

    For Finance Teams: - Accurate cost attribution by product/feature - Budget variance analysis (planned vs actual) - FinOps metrics for cloud spending

    For Development Teams: - Traceability from requirement ‚Üí code ‚Üí deployment - Impact analysis (which products affected by change?) - Rollback evidence (when/why/by whom)

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Generate Provenance Fingerprints
    2. Create unique IDs for actions (SHA-256 hash of action metadata)
    3. Include who/what/when/why/how in fingerprint

    4. Store fingerprints in Cosmos DB (append-only, no deletes)

    5. Maintain Provenance Chains

    6. Link actions to originating requirements (CDD ‚Üí epic ‚Üí story ‚Üí commit ‚Üí deployment)
    7. Link AI decisions to supporting data (which docs informed RAG response?)

    8. Link cost to originating product/feature

    9. Cost Attribution

    10. Tag all Azure resources with product/feature/tier
    11. Track resource usage by tag

    12. Generate cost reports (actual vs budget, variance analysis)

    13. 7-Year Audit Trail Retention

    14. Store provenance fingerprints for 7 years (GC requirement)
    15. Prevent deletion/modification (immutable log)
    16. Backup to Azure Archive Storage (low-cost long-term retention)

    Supporting Responsibilities¶

    1. ATO Evidence Generation
    2. Generate compliance reports for ATO auditors
    3. Provide evidence of C0-C3 autonomy controls (P19)

    4. Demonstrate data lineage for Protected B handling

    5. Incident Forensics

    6. Provide audit trail for incident investigations
    7. Identify what changed before incident

    8. Trace blast radius (what was affected?)

    9. Impact Analysis

    10. Identify which products/features affected by code change
    11. Predict cost impact of scaling operations
    12. Trace dependency chains across repos

    üöÄ When to Use P21¶

    Primary Use Cases¶

    1. Generate Provenance Fingerprint for Action¶

    Scenario: Deploy new feature to Pilot tier, need audit trail.

    Action: Use P21 to create fingerprint and store in Cosmos DB.

    Example: 

    provenance_fingerprint:
  id: "sha256:a3f9c7b2..."  # SHA-256 hash of action metadata
  timestamp: "2025-12-12T14:30:00Z"
  action_type: "deployment"

  who:
    human_initiator: "Marco Presta (PO)"
    ai_agent: "P15-DVM (Dev Master Orchestrator)"
    approvers:
      - "Marco Presta (PO) - Sprint approval"
      - "Security SME - Security review"

  what:
    artifact: "eva-da-2:v1.5.0"
    target: "Pilot tier (Canada Central)"
    components:
      - "eva-api:v1.5.0"
      - "eva-rag:v1.2.3"
      - "eva-ui:v1.4.0"

  when:
    initiated: "2025-12-12T14:00:00Z"
    completed: "2025-12-12T14:30:00Z"
    duration: "30 minutes"

  why:
    originating_requirement: "CDD-DA2-001 (Multi-language support)"
    epic: "EPIC-42 (Bilingual RAG)"
    user_story: "US-123 (As a French user, I want to search in French)"
    sprint: 42

  how:
    workflow: "new-feature (P15-DVM)"
    pipeline: "GitHub Actions (.github/workflows/deploy-pilot.yml)"
    approvals:
      - "PR #456 (P06 review passed)"
      - "Sprint planning (PO approved for Sprint 42)"
      - "Security review (P11-SEC passed)"

  data_lineage:
    input_data:
      - "Jurisprudence corpus (fr-CA)"
      - "Training data (bilingual embeddings)"
    data_classification: "Protected B"
    data_residency: "Canada Central"

  cost_attribution:
    product: "EVA Domain Assistant 2.0"
    feature: "Bilingual RAG"
    tier: "Pilot"
    estimated_cost: "$120/month"
    budget_variance: "+$20/month vs planned"

  compliance:
    autonomy_level: "C1 (Human-on-the-loop)"
    safety_guardrails:
      - "Real-time monitoring (Application Insights)"
      - "24-hour review (PO validates Pilot usage)"
    ato_evidence: true
    pia_updated: true
    aiia_updated: true

  storage:
    cosmos_db_id: "prov-a3f9c7b2"
    retention: "7 years"
    backup: "Azure Archive Storage (monthly snapshot)"

    2. Trace Provenance Chain from Requirement to Deployment¶

    Scenario: Auditor asks, "Why was this feature deployed? Who approved it? What data does it use?"

    Action: Use P21 to trace provenance chain.

    Example: 

    **Audit Query**: "Why was Bilingual RAG deployed to Pilot on 2025-12-12?"

**P21 Provenance Chain**:

1. **Originating Requirement (CDD-DA2-001)**
   - Fingerprint: `sha256:1a2b3c4d...`
   - Created: 2025-10-01
   - Author: Marco Presta (PO)
   - Rationale: "GC users require French-language search (Official Languages Act compliance)"

2. **Epic (EPIC-42: Bilingual RAG)**
   - Fingerprint: `sha256:5e6f7g8h...`
   - Created: 2025-10-15
   - Linked to: CDD-DA2-001
   - Scope: "Implement bilingual embeddings, French chunking, language detection"

3. **User Story (US-123)**
   - Fingerprint: `sha256:9i0j1k2l...`
   - Created: 2025-11-01
   - Linked to: EPIC-42
   - Acceptance Criteria:
     - "French user can search in French"
     - "RAG retrieves French documents with 90%+ precision"
     - "Language detection auto-switches UI to French"

4. **Code Changes (PR #456)**
   - Fingerprint: `sha256:3m4n5o6p...`
   - Created: 2025-11-20
   - Linked to: US-123
   - Files Changed:
     - `eva-rag/src/embeddings/bilingual.py` (added French embeddings)
     - `eva-api/src/language-detection.ts` (added language switch)
   - Reviews: P06-REV passed, P11-SEC passed

5. **Deployment (Pilot Tier)**
   - Fingerprint: `sha256:a3f9c7b2...` (main fingerprint above)
   - Created: 2025-12-12
   - Linked to: PR #456, US-123, EPIC-42, CDD-DA2-001
   - Approvals: Marco (PO), Security SME
   - Data: Jurisprudence corpus (fr-CA, Protected B)
   - Cost: $120/month (Pilot tier)

**Audit Answer**:
- **Why**: Official Languages Act compliance (CDD-DA2-001)
- **Who Approved**: Marco Presta (PO), Security SME
- **What Data**: Jurisprudence corpus (French, Protected B, Canada Central)
- **Evidence**: 5 linked fingerprints forming provenance chain

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P21-PRO (Provenance & Fingerprinting), the Audit Trail Generator and Provenance Chain Maintainer for EVA.

**Your Capabilities**:
1. Generate provenance fingerprints for actions (SHA-256 hash)
2. Maintain provenance chains (requirement ‚Üí code ‚Üí deployment)
3. Attribute cost to products/features via Azure tags
4. Store audit trail for 7 years (Cosmos DB + Azure Archive)
5. Generate ATO compliance reports

**Your Limitations**:
- You do NOT prevent actions (you only record them)
- You do NOT guarantee fingerprints are forensically sound (best-effort)
- You do NOT validate cost allocations (Finance reviews)

**Autonomy Level**: C2 (Human-over-the-loop)
- Auto-generate fingerprints for all actions
- Auto-store in Cosmos DB (immutable log)
- Human reviews compliance reports monthly
- Human validates cost attribution quarterly

**Output Format**:
- Provenance fingerprint (YAML with who/what/when/why/how)
- Provenance chain (linked fingerprints from requirement to deployment)
- Cost attribution report (Azure resources by product/feature)
- ATO evidence report (compliance artifacts)

**Safety Rules**:
- Never delete fingerprints (7-year retention mandatory)
- Never modify fingerprints (append-only log)
- Backup to Azure Archive monthly (disaster recovery)
- Flag data residency violations as BLOCKER

    Task Prompts (For Humans)¶

    Prompt 1: Generate Provenance Fingerprint¶

    **Task**: Generate provenance fingerprint for this action

**Context**:
- **Action Type**: [deployment | code change | AI decision | cost allocation]
- **Who**: [human initiator, AI agent, approvers]
- **What**: [artifact, target, components]
- **When**: [initiated, completed, duration]
- **Why**: [originating requirement, epic, user story]
- **How**: [workflow, pipeline, approvals]

**Expected Output**:
- Provenance fingerprint (YAML)
- Fingerprint ID (SHA-256 hash)
- Cosmos DB storage confirmation
- Provenance chain links (if applicable)

    Prompt 2: Trace Provenance Chain¶

    **Task**: Trace provenance chain from requirement to deployment

**Context**:
- **Starting Point**: [CDD | epic | user story | commit | deployment]
- **End Point**: [where should the chain stop?]
- **Audit Question**: [what does the auditor want to know?]

**Expected Output**:
- Provenance chain (linked fingerprints)
- Who approved at each step
- What data was used
- Evidence of compliance (autonomy level, safety guardrails)

    Prompt 3: Cost Attribution Report¶

    **Task**: Generate cost attribution report for product/feature

**Context**:
- **Product**: [EVA DA 2.0 | EVA Chat 2.0 | etc.]
- **Feature**: [specific feature or "all features"]
- **Tier**: [Sandbox | Pilot | Production]
- **Time Period**: [month or date range]

**Expected Output**:
- Azure resources by tag (product/feature/tier)
- Usage metrics (RU, tokens, storage, queries)
- Cost breakdown (by resource)
- Budget variance (actual vs planned)
- Provenance links (which fingerprints caused these costs?)

    awareness_block:
  agent: "P21-PRO"

    - "Shared resource allocation is fair (proportional by usage)"

  recommendations:
    - "Validate shared resource allocation with Finance team"
    - "Consider dedicated resources for high-cost features (more accurate attribution)"
    - "Track developer time separately (not included in Azure cost)"

  human_review_required: true
  human_review_reason: "Cost variance is +20% vs budget. Finance team should review and approve mitigation plan."

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C2 (Human-over-the-Loop)¶

    P21 Itself: C2 - Auto-generates fingerprints for all actions - Auto-stores in Cosmos DB (immutable log) - Human reviews compliance reports monthly - Human validates cost attribution quarterly

    Why C2? - Provenance generation is highly routine (proven at scale) - Low risk (recording actions, not executing them) - Periodic human review ensures accuracy - Strict constraints (append-only, no deletes, 7-year retention)

    Safety Rules¶

    Rule 1: Fingerprints Are Immutable¶

    Risk: Tampering with audit trail to hide evidence

    Mitigation: - ‚úÖ Cosmos DB configured as append-only (no deletes/updates) - ‚úÖ SHA-256 hash prevents fingerprint forgery - ‚úÖ Monthly backup to Azure Archive Storage (disaster recovery) - ‚úÖ Alert if fingerprint count decreases (potential tampering)

    Rule 2: 7-Year Retention Mandatory¶

    Risk: Deleting audit trail before GC retention period ends

    Mitigation: - ‚úÖ Cosmos DB retention policy enforced (no auto-delete) - ‚úÖ Azure Archive Storage backup monthly - ‚úÖ Alert if retention policy changed - ‚úÖ Governance panel approval required to change retention

    Rule 3: Data Residency Violations Are BLOCKER¶

    Risk: Storing Protected B data outside Canada

    Mitigation: - ‚úÖ P21 validates data residency (Canada Central/Canada East only) - ‚úÖ Block action if data residency violation detected - ‚úÖ Alert Security SME immediately - ‚úÖ Audit trail includes data residency for every action

    • I have validated provenance chains support ATO requirements

    For Auditors (SSC, TBS, OAG)¶

    • I have accessed Cosmos DB audit trail
    • I have traced provenance chains for sampled actions
    • I have validated approvals at each step
    • I have confirmed 7-year retention policy is enforced
