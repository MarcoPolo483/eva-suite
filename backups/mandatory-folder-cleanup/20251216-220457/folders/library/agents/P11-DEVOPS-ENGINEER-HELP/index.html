    Agent Service Catalog - P11 (HELP)¶

    Agent ID: P11 Agent Name: DevOps Engineer POD: POD-D (Delivery) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P11 is the DevOps Engineer agent responsible for CI/CD pipelines, infrastructure automation, deployment orchestration, monitoring, and production operations across the EVA Suite.

    Core Services¶

    1. CI/CD Pipeline Development¶

    • Description: Build and maintain CI/CD pipelines for automated testing and deployment
    • Input: Repository, test suite, deployment target, quality gates
    • Output: GitHub Actions/Azure Pipelines workflow, deployment scripts
    • Example: P11: Create CI/CD pipeline for eva-api with pytest, coverage, and Azure deployment

    Input Template: 

    P11 REQUEST: CI/CD Pipeline
  REPOSITORY: [Repo name]
  PLATFORM: [GitHub Actions | Azure Pipelines]
  STAGES:
    - build: [Install dependencies, compile]
    - test: [Run pytest with coverage]
    - quality: [Quality gates to enforce]
    - deploy: [Deploy to Azure App Service]
  TRIGGERS:
    - [On push to main]
    - [On pull request]
  ENVIRONMENTS: [dev | test | staging | prod]

    2. Infrastructure Provisioning & Management¶

    • Description: Provision and manage cloud infrastructure using IaC
    • Input: Infrastructure requirements, Azure services needed, environments
    • Output: Bicep/Terraform templates, provisioning scripts, documentation
    • Example: P11: Provision Azure App Service, Cosmos DB, Blob Storage for eva-api dev environment

    Input Template: 

    P11 REQUEST: Infrastructure Provisioning
  ENVIRONMENT: [dev | test | staging | prod]
  SERVICES:
    - type: [Azure App Service]
      sku: [B1 | S1 | P1v2]
      region: [East US]
    - type: [Cosmos DB]
      throughput: [Autoscale, 1000-4000 RU/s]
    - type: [Blob Storage]
      redundancy: [LRS | GRS]
  IaC_TOOL: [Bicep | Terraform]
  COST_BUDGET: [e.g., "$500/month"]

    3. Deployment Orchestration¶

    • Description: Execute deployments to environments with rollback capability
    • Input: Application package, target environment, deployment strategy
    • Output: Deployed application, deployment logs, validation results
    • Example: P11: Deploy eva-api v1.2.0 to dev environment with blue-green strategy

    Input Template: 

    P11 REQUEST: Deployment
  APPLICATION: [App name and version]
  ENVIRONMENT: [dev | test | staging | prod]
  STRATEGY: [Rolling | Blue-green | Canary]
  PRE_DEPLOYMENT:
    - [Check 1, e.g., "P09 sign-off"]
    - [Check 2, e.g., "All tests passing"]
  POST_DEPLOYMENT:
    - [Validation 1, e.g., "Health check"]
    - [Validation 2, e.g., "Smoke tests"]
  ROLLBACK_PLAN: [How to rollback if deployment fails]

    4. Monitoring & Observability Setup¶

    • Description: Configure monitoring, logging, and alerting
    • Input: Application, metrics to track, alert thresholds
    • Output: Application Insights config, dashboards, alert rules
    • Example: P11: Set up Application Insights monitoring for eva-api with custom events

    Input Template: 

    P11 REQUEST: Monitoring Setup
  APPLICATION: [App name]
  MONITORING_TOOL: [Application Insights | Azure Monitor]
  METRICS:
    - response_time: [Track p50, p95, p99]
    - error_rate: [Track 4xx, 5xx]
    - custom: [Document uploads, queries]
  ALERTS:
    - name: [Alert name]
      condition: [e.g., "Error rate >5% for 5 min"]
      severity: [P0 | P1 | P2]
      notification: [Email | Teams | PagerDuty]
  DASHBOARDS: [Real-time operations, daily summary]

    5. Secrets & Configuration Management¶

    • Description: Manage application secrets and environment configurations
    • Input: Secrets needed, configuration per environment
    • Output: Azure Key Vault setup, config files, access policies
    • Example: P11: Configure secrets in Azure Key Vault for eva-api (Cosmos keys, OpenAI keys)

    Input Template: 

    P11 REQUEST: Secrets Management
  APPLICATION: [App name]
  SECRETS:
    - name: [Secret name, e.g., "COSMOS_DB_KEY"]
      type: [Connection string | API key | Certificate]
      rotation: [Manual | Automatic]
  KEY_VAULT: [Key Vault name or create new]
  ACCESS_POLICIES:
    - identity: [Managed identity or service principal]
      permissions: [Get | List | Set]
  ENVIRONMENTS: [dev | test | prod]

    Service Dependencies¶

    Upstream (P11 depends on): - P01 (System Architect): Architecture, infrastructure requirements - P03 (Security Architect): Security configs, access policies - P09 (QA Engineer): Deployment approval after quality validation - SM (Scrum Master): Deployment schedule, sprint timeline

    Downstream (Depends on P11): - All agents: Deployed environments for development/testing - PO (Product Owner): Production deployment status - P10 (Performance Engineer): Performance monitoring infrastructure

    Service Standards¶

    CI/CD Standards¶

    • All repos have CI/CD pipelines
    • Pipelines run on every push and PR
    • Quality gates enforced (coverage, tests, security)
    • Failed pipelines block deployment
    • Pipeline runs complete in <10 minutes

    Infrastructure Standards¶

    • Infrastructure as Code (IaC) for all resources
    • All resources tagged (Environment, Owner, Cost Center)
    • Dev environments auto-shutdown nights/weekends
    • Production resources have redundancy (HA)
    • All resources in same region for latency

    Deployment Standards¶

    • Zero-downtime deployments (blue-green or canary)
    • Rollback plan tested before deployment
    • Health checks after every deployment
    • Smoke tests validate critical paths
    • Deployment logs retained for 90 days

    Monitoring Standards¶

    • Application Insights enabled for all apps
    • Custom events for business metrics
    • Alerts configured for P0/P1 issues
    • Dashboards accessible to team
    • Log retention: 90 days (dev), 1 year (prod)

    Common Patterns¶

    Pattern 1: GitHub Actions Workflow¶

    name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest --cov=src --cov-report=xml
      - name: Check coverage
        run: |
          coverage report --fail-under=80

  deploy:
    if: github.ref == 'refs/heads/main'
    needs: test
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Azure
        uses: azure/webapps-deploy@v2
        with:
          app-name: eva-api-dev
          publish-profile: ${{ secrets.AZURE_WEBAPP_PUBLISH_PROFILE }}

    Pattern 2: Bicep Infrastructure Template¶

    param appName string = 'eva-api'
param environment string = 'dev'
param location string = resourceGroup().location

resource appServicePlan 'Microsoft.Web/serverfarms@2022-03-01' = {
  name: '${appName}-${environment}-plan'
  location: location
  sku: {
    name: 'B1'
    tier: 'Basic'
  }
  kind: 'linux'
  properties: {
    reserved: true
  }
}

resource appService 'Microsoft.Web/sites@2022-03-01' = {
  name: '${appName}-${environment}'
  location: location
  properties: {
    serverFarmId: appServicePlan.id
    siteConfig: {
      linuxFxVersion: 'PYTHON|3.11'
      appSettings: [
        {
          name: 'COSMOS_DB_ENDPOINT'
          value: '@Microsoft.KeyVault(SecretUri=${keyVault.properties.vaultUri}secrets/COSMOS-DB-ENDPOINT/)'
        }
      ]
    }
  }
}

    Pattern 3: Deployment Checklist¶

    ## Deployment Checklist: [App] to [Env]

**Pre-Deployment** (P11 verifies):
- [ ] P09 sign-off received
- [ ] All tests passing in CI/CD
- [ ] Coverage 80%
- [ ] Security scan clean (no P0/P1 vulnerabilities)
- [ ] Rollback plan documented and tested

**Deployment** (P11 executes):
- [ ] Backup current version
- [ ] Deploy new version to staging slot
- [ ] Run smoke tests on staging
- [ ] Swap staging  production
- [ ] Verify health endpoint returns 200

**Post-Deployment** (P11 validates):
- [ ] Application Insights showing telemetry
- [ ] No errors in logs (first 15 minutes)
- [ ] Smoke tests pass in production
- [ ] Performance baseline maintained (p95 latency)
- [ ] Alerts configured and tested

**Rollback Trigger**:
- Error rate >5%
- Health check fails
- Smoke tests fail
- Performance degradation >50%

    File Naming Convention¶

    All deliverables must follow: 

    P11-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P11-CICD-PIPELINE-EVA-API-20251209-120000.md - P11-INFRASTRUCTURE-DEV-20251209-130000.md - P11-DEPLOYMENT-REPORT-EVA-API-V1.2-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P11- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md
