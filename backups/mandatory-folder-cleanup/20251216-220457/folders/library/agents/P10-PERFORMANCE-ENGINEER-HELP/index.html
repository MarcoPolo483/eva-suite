    Agent Service Catalog - P10 (HELP)¶

    Agent ID: P10 Agent Name: Performance Engineer POD: POD-Q (Quality) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P10 is the Performance Engineer agent responsible for load testing, performance optimization, benchmarking, scalability analysis, and ensuring EVA Suite applications meet performance targets under production load.

    Core Services¶

    1. Load & Stress Testing¶

    • Description: Execute load tests to validate performance under expected and peak traffic
    • Input: Application endpoint, expected load, performance targets
    • Output: Load test results, bottlenecks identified, capacity recommendations
    • Example: P10: Load test eva-api document upload (100 concurrent users, 1000 uploads/hour)

    Input Template: 

    P10 REQUEST: Load Testing
  APPLICATION: [App name and endpoint]
  TEST_SCENARIOS:
    - name: [Scenario 1, e.g., "Document upload"]
      load: [e.g., "100 concurrent users"]
      duration: [e.g., "30 minutes"]
      ramp_up: [e.g., "5 minutes"]
  PERFORMANCE_TARGETS:
    - latency_p95: [e.g., "<2 seconds"]
    - throughput: [e.g., "1000 req/hour"]
    - error_rate: [e.g., "<1%"]
  TOOLS: [JMeter | Locust | k6 | Azure Load Testing]
  DELIVERABLES: [Test results, bottleneck analysis, recommendations]

    2. Performance Optimization¶

    • Description: Identify and fix performance bottlenecks in code, queries, or infrastructure
    • Input: Performance issue, current metrics, codebase access
    • Output: Optimized code/queries, performance improvement metrics
    • Example: P10: Optimize Cosmos DB query in eva-api (currently 5s, target <500ms)

    Input Template: 

    P10 REQUEST: Performance Optimization
  ISSUE: [Performance problem description]
  CURRENT_METRICS:
    - [Metric 1, e.g., "Latency: 5 seconds (p95)"]
    - [Metric 2, e.g., "CPU: 80% utilization"]
  TARGET_METRICS:
    - [Target 1, e.g., "Latency: <500ms (p95)"]
    - [Target 2, e.g., "CPU: <60%"]
  SCOPE:
    - [Area 1, e.g., "Cosmos DB queries"]
    - [Area 2, e.g., "Embedding generation"]
  CONSTRAINTS: [Budget | Time | Breaking changes]

    3. Scalability Analysis¶

    • Description: Analyze application scalability and recommend architecture changes
    • Input: Current architecture, growth projections, bottlenecks
    • Output: Scalability assessment, architecture recommendations, cost analysis
    • Example: P10: Assess eva-api scalability for 10x growth (1K → 10K users, 10K → 100K docs)

    Input Template: 

    P10 REQUEST: Scalability Analysis
  APPLICATION: [App name]
  CURRENT_SCALE:
    - users: [e.g., "1,000 active users"]
    - documents: [e.g., "10,000 documents"]
    - traffic: [e.g., "100 req/hour"]
  PROJECTED_SCALE:
    - users: [e.g., "10,000 active users"]
    - documents: [e.g., "100,000 documents"]
    - traffic: [e.g., "1,000 req/hour"]
  BOTTLENECKS:
    - [Bottleneck 1, e.g., "Cosmos DB throughput"]
    - [Bottleneck 2, e.g., "Embedding generation"]
  DELIVERABLES: [Scalability report, architecture changes, cost estimate]

    4. Performance Benchmarking¶

    • Description: Establish performance baselines and track changes over time
    • Input: Application, key user flows, benchmark frequency
    • Output: Benchmark suite, baseline metrics, performance trends
    • Example: P10: Create benchmark suite for eva-api (upload, query, download) and run weekly

    Input Template: 

    P10 REQUEST: Performance Benchmarking
  APPLICATION: [App name]
  USER_FLOWS:
    - [Flow 1, e.g., "Upload 10MB document"]
    - [Flow 2, e.g., "Query 100 documents"]
  METRICS:
    - [Metric 1, e.g., "Latency (p50, p95, p99)"]
    - [Metric 2, e.g., "Throughput (req/sec)"]
    - [Metric 3, e.g., "Resource usage (CPU, memory)"]
  FREQUENCY: [Weekly | After each deployment | Monthly]
  BASELINE: [Establish new | Compare to existing]

    5. Monitoring & Alerting Setup¶

    • Description: Configure performance monitoring and alerts for production
    • Input: Application, key metrics, alert thresholds
    • Output: Monitoring dashboards, alert rules, runbooks
    • Example: P10: Set up Application Insights performance monitoring for eva-api with p95 latency alerts

    Input Template: 

    P10 REQUEST: Performance Monitoring
  APPLICATION: [App name]
  METRICS:
    - name: [Metric name, e.g., "API latency"]
      target: [e.g., "<500ms p95"]
      alert_threshold: [e.g., ">1s p95 for 5 minutes"]
      severity: [P0 | P1 | P2]
  DASHBOARDS:
    - [Dashboard 1, e.g., "Real-time performance"]
    - [Dashboard 2, e.g., "Daily summary"]
  TOOLS: [Application Insights | Azure Monitor | Grafana]
  DELIVERABLES: [Dashboards, alert rules, runbooks]

    Service Standards¶

    Load Testing Standards¶

    • Test environments mirror production (same SKUs, configs)
    • Ramp-up load gradually (avoid cold-start effects)
    • Run tests for ≥30 minutes (identify memory leaks)
    • Test realistic user scenarios (not just max throughput)
    • Monitor all tiers (API, database, storage, external APIs)

    Performance Targets¶

    • API latency: <500ms (p95), <1s (p99)
    • Query latency: <2s (p95)
    • Upload latency: <5s for 10MB file
    • Throughput: ≥1000 req/hour per instance
    • Error rate: <1% under load
    • Resource usage: CPU <70%, memory <80%

    Optimization Standards¶

    • Profile before optimizing (use data, not guesses)
    • Measure before/after (prove improvements)
    • Optimize critical paths first (Pareto principle)
    • Document trade-offs (performance vs maintainability)
    • Regression test (ensure no functionality broken)

    Monitoring Standards¶

    • Application Insights enabled for all apps
    • Custom metrics for business operations
    • Dashboards for real-time and historical trends
    • Alerts for P0/P1 performance degradations
    • Performance reviewed weekly (trend analysis)

    Common Patterns¶

    Pattern 1: Load Test Script (k6)¶

    // k6 load test for eva-api document upload
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '5m', target: 100 },  // Ramp-up to 100 users
    { duration: '30m', target: 100 }, // Stay at 100 users
    { duration: '5m', target: 0 },    // Ramp-down
  ],
  thresholds: {
    http_req_duration: ['p(95)<2000'], // 95% < 2s
    http_req_failed: ['rate<0.01'],    // Error rate < 1%
  },
};

export default function () {
  const url = 'https://api.eva-suite.com/v1/documents';
  const file = open('./sample.pdf', 'b');
  const payload = {
    file: http.file(file, 'sample.pdf'),
  };

  const res = http.post(url, payload, {
    headers: {
      'Authorization': `Bearer ${__ENV.API_TOKEN}`,
    },
  });

  check(res, {
    'status is 201': (r) => r.status === 201,
    'response time < 2s': (r) => r.timings.duration < 2000,
  });

  sleep(1); // Think time between requests
}

    Pattern 2: Performance Optimization Checklist¶

    ## Performance Optimization Checklist

**Database**:
- [ ] Queries use partition key (avoid cross-partition)
- [ ] Indexes created for frequently filtered fields
- [ ] No SELECT * (only select needed fields)
- [ ] Pagination implemented (limit result sets)
- [ ] Connection pooling configured

**Application**:
- [ ] Caching implemented (Redis, in-memory)
- [ ] Async operations for I/O (non-blocking)
- [ ] Batch operations where possible (reduce round-trips)
- [ ] Compression enabled (gzip, Brotli)
- [ ] Static assets served from CDN

**Infrastructure**:
- [ ] Auto-scaling enabled (scale out under load)
- [ ] Resource SKUs appropriate (not under-provisioned)
- [ ] Geo-distribution for global users
- [ ] Health checks configured (detect unhealthy instances)

    File Naming Convention¶

    All deliverables must follow: 

    P10-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P10-LOAD-TEST-REPORT-EVA-API-20251209-120000.md - P10-OPTIMIZATION-REPORT-COSMOS-QUERY-20251209-130000.md - P10-SCALABILITY-ASSESSMENT-EVA-API-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P10- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md
