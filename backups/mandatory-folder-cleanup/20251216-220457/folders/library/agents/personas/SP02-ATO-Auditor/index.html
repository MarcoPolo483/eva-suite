    SP02: ATO Auditor - Service Persona Home¶

    Persona Code: SP02
    Name: ATO Auditor
    Type: Service Persona (Compliance Audit)
    Maturity: Concept
    Autonomy Level: C0 (Human-in-the-loop)

    üìã What It Is¶

    The ATO Auditor is an automated compliance auditor that validates evidence for Authority to Operate (ATO) in Government of Canada environments. It:

    • Scans logs, provenance, and documentation for compliance evidence
    • Generates compliance reports against TBS, SSC, and departmental standards
    • Performs gap analysis identifying missing controls or documentation
    • Creates evidence inventory for governance panels
    • Updates risk registers with new findings

    The ATO Auditor operates in advisory mode (C0) - it provides recommendations and evidence summaries, but all ATO decisions remain with human authorities.

    üöÄ How to Invoke¶

    Invocation Method: On-demand audit request ‚Üí automated compliance report

    Process: 1. Submit audit request via P11 (Security & Compliance Helper) or manual request 2. Define audit scope (repos, services, time period, compliance requirements) 3. ATO Auditor scans evidence sources (logs, provenance, docs) 4. Automated analysis generates compliance report 5. Human Security SME / AI Steward reviews report 6. Governance panel uses report for ATO decision (human authority)

    Example Audit Request: 

    audit_type: ato_compliance
scope:
  repos: ["eva-api", "eva-ui", "eva-rag"]
  services: ["Azure OpenAI", "Azure AI Search", "Cosmos DB"]
  time_period: "2025-01-01 to 2025-12-31"
compliance_frameworks:
  - "TBS Policy on Service and Digital"
  - "SSC Cloud Security Controls"
  - "GC Agentic AI Framework (C0-C3)"
  - "ESDC Departmental Standards"
evidence_sources:
  - provenance_store: "cosmos://eva-meta/provenance"
  - logs: "log-analytics://eva-logs"
  - docs: "github://eva-orchestrator/docs/compliance"
output_format: "detailed_report"

    ‚ö†Ô∏è Limitations¶

    1. Advisory Only¶

    • No ATO authority: Cannot grant or deny Authority to Operate
    • Human decision required: Governance panel has final authority
    • Recommendations only: Cannot enforce remediation

    2. Evidence Scope¶

    • Only automated evidence: Cannot validate manual processes or interviews
    • Document-based: Relies on written policies, logs, and provenance
    • Point-in-time: Audit reflects specific time period, not real-time

    3. Framework Coverage¶

    • Predefined frameworks: Only supports explicitly programmed compliance frameworks
    • No custom standards: Cannot interpret new or organization-specific standards without updates
    • Manual interpretation required: Complex edge cases require human Security SME review

    4. Risk Assessment¶

    • No severity scoring: Cannot determine business impact of risks
    • No cost-benefit analysis: Cannot recommend prioritization of remediation
    • Human judgment required: Security SME must interpret findings in business context

    üéØ Compliance Frameworks¶

    1. TBS Policy on Service and Digital¶

    Key controls validated: - Digital services must be accessible (WCAG 2.1 AA) - Use open standards and solutions by default - Use cloud-first approach - Secure by design - Privacy by design

    2. SSC Cloud Security Controls¶

    INFxx (Infrastructure): - INF01: Private endpoints only - INF02: TLS 1.2+ enforced - INF03: Azure AD authentication - INF04: SCED-routed traffic - INF05: Multi-region deployment

    SECxx (Security): - SEC01: Protected B encryption (CMK, AES-256) - SEC02: RBAC/PoLP enforced - SEC03: Key rotation (90-day automatic) - SEC04: Audit logging (7-year retention) - SEC05: Data residency (Canada-only)

    3. GC Agentic AI Framework¶

    4 Pillars: 1. Ethical AI (fairness, transparency, accountability) 2. Safety & Security (content safety, adversarial robustness) 3. Privacy & Data Governance (PIA, data minimization) 4. Continuous Oversight (monitoring, incident response)

    C0-C3 Autonomy Levels: - C0: Human-in-the-loop (every action approved) - C1: Human-on-the-loop (monitor, can intervene) - C2: Human-over-the-loop (review periodically) - C3: Full autonomy (within boundaries)

    Required Documentation: - Privacy Impact Assessment (PIA) for all C1+ services - Algorithmic Impact Assessment (AIIA) for all C2+ services - Risk register with autonomy-related risks

    üë• Primary Users¶

    • Security SMEs: Review compliance reports and prioritize remediation
    • AI Steward: Ensure AI/ML services meet GC Agentic AI Framework
    • Platform Owner: Authorize ATO submissions and budget remediation
    • Governance Panels: Use reports as evidence for ATO decisions
    • Auditors: External auditors (SSC, TBS, OAG) reviewing compliance

    üîó Related Patterns & Repos¶

    DevTools Patterns: - P11 (SEC) - Security & Compliance Helper (triggers audits) - P21 (PRV) - Provenance & Fingerprinting (evidence source) - P09 (RUN) - Runbook & SRE Agent (incident evidence) - P10 (MET) - Metrics & Insight Agent (operational evidence)

    Related Repos: - eva-meta - Governance registry, provenance store, compliance docs - eva-ops - LiveOps dashboard, incident reports - eva-audit - Audit logs, compliance evidence (future) - eva-infra - Terraform configs, Azure Policy definitions
