    SP01 ‚Äì Data Ingestion Persona¶

    EVA Data Pipeline Architecture Specification¶

    Version: 1.0.0
    Date: 2025-12-11
    Status: Active
    Pod: POD-F (Foundation & Platform)
    Owner: EVA Suite Architecture Team

    Executive Summary¶

    EVA Data Pipeline must support heterogeneous enterprise content ingestion across all Government of Canada departments and agencies. This specification defines the canonical architecture for ingesting, processing, and preparing documents for RAG-based retrieval and AI-assisted workflows.

    Supported File Types: PDF, DOCX, PPTX, HTML/HTM, JSON, TXT, EML, MSG
    Canonical Ingestion Profiles: 5 (A‚ÄìE)
    Pipeline Capabilities: 5 major layers (parsing, chunking, embedding, retrieval, traceability)

    1. Consolidated File Type Universe¶

    EVA must handle exactly six primary document categories:

    Primary Document Types¶

    1. PDF ‚Äì Portable Document Format
    2. DOCX / PPTX ‚Äì Microsoft Office OOXML formats
    3. HTML / HTM ‚Äì Web/hypertext content

    Text & Structured Data¶

    1. JSON / TXT ‚Äì Structured and unstructured text

    Email Formats¶

    1. EML / MSG ‚Äì Email messages and threads

    Edge Case¶

    1. "Not Applicable" ‚Äì Metadata-only submissions (no content files)

    Complete Supported Set: PDF, DOCX, PPTX, HTML/HTM, JSON, TXT, EML, MSG

    2. Canonical Ingestion Profiles (A‚ÄìE)¶

    To reduce complexity and enable consistent processing, EVA internally maps file types into five canonical ingestion profiles:

    üìÑ Profile A ‚Äî Static Documents¶

    File Types: PDF, DOCX, PPTX
    Processing Pipeline: - Layout-aware parsing (multi-column, headers/footers, page numbers) - Table extraction and reconstruction - Heading/section detection - Image and diagram handling - Footnote and reference extraction

    Tools Required: - PDFMiner / PyMuPDF / PDFPlumber - Docx2Python or Aspose (if licensed) - Table canonicalization engine

    Output: Structured markdown with preserved layout semantics

    üåê Profile B ‚Äî Web/Hypertext¶

    File Types: HTML, HTM
    Processing Pipeline: - DOM cleaning (remove scripts, styles, navigation) - Semantic block extraction - Link and reference preservation - Heading hierarchy detection - Content chunking

    Tools Required: - HTML5 parser (BeautifulSoup, lxml) - Custom DOM cleaner - Markdown converter

    Output: Clean semantic markdown with preserved links

    üóÇÔ∏è Profile C ‚Äî Structured Data¶

    File Types: JSON, TXT
    Processing Pipeline: - Schema detection and validation - Structure type inference (list, table, content) - Inline annotation - Field extraction - Chunking with structure preservation

    Tools Required: - JSON schema validator - Structure classifier - Custom annotator

    Output: Annotated structured content with metadata

    ‚úâÔ∏è Profile D ‚Äî Communications¶

    File Types: EML, MSG
    Processing Pipeline: - Header extraction (from, to, cc, subject, date) - Body content extraction - Attachment handling - Thread deduplication - Timestamp normalization

    Tools Required: - Email parsers (email, mailparser packages) - MIME extractors - Thread analyzer

    Output: Structured email record with extracted attachments

    √ò Profile E ‚Äî Metadata Only¶

    File Types: "Not Applicable"
    Processing Pipeline: - Index-only ingestion - Metadata capture and validation - Reference linking - Catalog entry creation

    Tools Required: - Metadata schema validator - Catalog indexer

    Output: Catalog entry without content body

    3. Pipeline Architecture: Five Major Capabilities¶

    EVA Data Pipeline implements five production-ready capability layers:

    3.1 Multi-Format Parsing Layer¶

    Purpose: Convert heterogeneous input formats into normalized intermediate representation

    Capabilities: - PDF Parsing: Complex layout, multi-column, tables, headers/footers - Office Documents: Tables, charts, lists, heading hierarchies - HTML Parsing: Script/style removal, semantic block extraction - JSON/TXT Processing: Auto-tagging (list vs table vs content) - Email Parsing: Attachments, timestamps, threading

    Implementation Requirements: - Pluggable parser architecture - Error handling and fallback strategies - Format detection and validation - Metadata extraction pipeline

    3.2 Table & Object Reconstruction Layer¶

    Purpose: Preserve structured information critical for government content

    Use Cases: - Pay grids and salary schedules - Rate tables and fee structures - Benefit schedules - Service level agreements - Case examples in structured blocks

    Output Formats: 1. Markdown tables (preferred for RAG) 2. JSON canonical table model (for structured queries) 3. Hybrid metadata block + text (for complex tables)

    Quality Requirements: - Column header preservation - Row relationship maintenance - Cell content accuracy - Multi-level header support

    3.3 Semantic Chunking Engine¶

    Purpose: Intelligent content segmentation optimized for RAG retrieval

    Chunking Strategies: - Layout-aware segmentation (respect document structure) - Semantic chunking (title ‚Üí paragraph ‚Üí table ‚Üí footnotes) - Cross-reference detection (attachments, annexes, images) - Multi-modal block support (text + metadata + citations)

    Chunk Metadata: 

    {
  "chunkId": "doc-123-chunk-05",
  "documentId": "doc-123",
  "chunkType": "paragraph",
  "parentSection": "Section 3.2",
  "pageNumber": 15,
  "profile": "A",
  "tokens": 512,
  "semanticWeight": 0.85
}

    Optimization Goals: - Optimal chunk size for embedding models (256-1024 tokens) - Semantic coherence within chunks - Minimal context loss at chunk boundaries - Efficient retrieval performance

    3.4 Embeddings Layer¶

    Purpose: High-quality vector representations for semantic search

    Requirements: - Chunk ‚Üí Embed ‚Üí Store pipeline - Metadata Attachment: - File type and profile (A‚ÄìE) - Layout type (header, paragraph, table, footnote) - Page/section location - Canonical profile classification - Ingestion timestamp - Classification/clearance level - Vector Database Integration: - Azure AI Search - Qdrant / Weaviate / Pinecone (alternatives) - Hybrid search support (dense + sparse)

    Embedding Model Selection: - Primary: Azure OpenAI text-embedding-ada-002 or text-embedding-3-large - Bilingual Support: Models supporting EN-CA / FR-CA - Domain Adaptation: Fine-tuning for government terminology

    3.5 Retrieval / Reranking / Traceability Layer¶

    Purpose: Accurate, explainable, and auditable search and retrieval

    Capabilities: - Hybrid Search: - BM25 keyword search (traditional) - Dense vector search (semantic) - Metadata filtering (clearance, date, source) - Multi-Hop Retrieval: Follow references and citations - Reranking: Cross-encoder reranking for citation accuracy - Hallucination Detection: Evidence verification before response generation - Provenance Tracking: Full audit trail from query ‚Üí retrieval ‚Üí response

    Quality Metrics: - Precision@K (top-K accuracy) - Recall (coverage of relevant documents) - Mean Reciprocal Rank (MRR) - Latency (p50, p95, p99) - Hallucination rate (false positive citations)

    4. Capability Checklist¶

    Core Parsing¶

    • ‚úÖ PDF parsing (text + tables + layout reconstruction)
    • ‚úÖ DOCX parsing (lists, headings, tables)
    • ‚úÖ PPTX parsing (slides, speaker notes, shapes)
    • ‚úÖ HTML parsing (cleaned DOM ‚Üí semantic blocks)
    • ‚úÖ JSON/TXT structural interpretation
    • ‚úÖ EML/MSG extraction (header + body + attachments)

    Normalization¶

    • ‚úÖ Canonical ingestion profiles (A‚ÄìE)
    • ‚úÖ Table canonicalization
    • ‚úÖ Structured metadata extraction

    Chunking + Embeddings¶

    • ‚úÖ Hybrid chunker (layout + semantic)
    • ‚úÖ Layout-aware segmentation
    • ‚úÖ Section labeling
    • ‚úÖ Embeddings with metadata tagging

    Retrieval Quality¶

    • ‚úÖ Hybrid search (BM25 + vectors)
    • ‚úÖ Reranking
    • ‚úÖ Citations with page/section accuracy
    • ‚úÖ Confidence scoring / hallucination guardrails

    5. Implementation Roadmap¶

    Phase 1: Foundation (Weeks 1-2)¶

    • Profile A parser (PDF + DOCX)
    • Basic chunking engine
    • Embedding pipeline
    • Vector storage integration

    Phase 2: Expansion (Weeks 3-4)¶

    • Profile B parser (HTML)
    • Profile C parser (JSON/TXT)
    • Profile D parser (EML/MSG)
    • Advanced chunking (semantic + layout-aware)

    Phase 3: Quality & Scale (Weeks 5-6)¶

    • Table reconstruction
    • Reranking integration
    • Hallucination detection
    • Performance optimization

    Phase 4: Production Readiness (Weeks 7-8)¶

    • Monitoring and observability
    • Error handling and retry logic
    • Scalability testing
    • Documentation and runbooks

    8. Governance & Compliance¶

    Data Classification¶

    • Public: No restrictions
    • Protected A: Controlled access
    • Protected B: Encrypted at rest and in transit
    • Protected C: Enhanced controls + audit trail

    Retention Policy¶

    • Raw documents: 7 years
    • Processed chunks: As long as source exists
    • Embeddings: Regenerate on model upgrade
    • Audit logs: 10 years (compliance)

    Privacy & Security¶

    • PII detection and redaction
    • Encryption (AES-256)
    • Access control (RBAC)
    • Audit trail (all operations logged)

    9. Performance Requirements¶

    Metric Target Measurement
    Ingestion Throughput 1000 docs/hour Per worker node
    Chunk Generation <5 sec/doc P95 latency
    Embedding Generation <2 sec/chunk P95 latency
    Search Latency <100ms P95 (hybrid search)
    Reranking Latency <200ms P95 (top-20 results)
    Hallucination Rate <2% False citation rate
    Uptime 99.9% Monthly SLA

    10. Monitoring & Observability¶

    Key Metrics¶

    • Documents ingested (per hour/day)
    • Ingestion failures (by profile type)
    • Chunk quality score (semantic coherence)
    • Embedding drift (model version tracking)
    • Search relevance (MRR, NDCG@10)
    • Hallucination detection rate

    Alerts¶

    • Ingestion failure >5% in 1 hour
    • Search latency P95 >500ms
    • Vector index lag >1 hour
    • Storage capacity <20% remaining
