    Agent Service Catalog - P06 (HELP)¶

    Agent ID: P06 Agent Name: RAG Engineer POD: POD-E (Experience) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P06 is the RAG Engineer (Retrieval-Augmented Generation) agent responsible for vector search, document embeddings, retrieval strategies, chunking optimization, and ensuring high-quality context retrieval for AI queries across the EVA Suite.

    Core Services¶

    1. Document Chunking & Embedding Strategy¶

    • Description: Design chunking strategies and embedding approaches for document types
    • Input: Document types, content structure, query patterns
    • Output: Chunking strategy, embedding model selection, configuration
    • Example: P06: Design chunking strategy for legal PDFs (contracts, regulations) with semantic preservation

    Input Template: 

    P06 REQUEST: Chunking Strategy
  DOCUMENT_TYPES:
    - [Type 1, e.g., "Legal contracts (50-200 pages)"]
    - [Type 2, e.g., "Technical reports (10-50 pages)"]
  CONTENT_CHARACTERISTICS:
    - [Char 1, e.g., "Numbered sections"]
    - [Char 2, e.g., "Tables and charts"]
  QUERY_PATTERNS:
    - [Pattern 1, e.g., "Find specific clauses"]
    - [Pattern 2, e.g., "Summarize sections"]
  CONSTRAINTS:
    - chunk_size: [e.g., "512-1024 tokens"]
    - overlap: [e.g., "10-20%"]
  EMBEDDING_MODEL: [text-embedding-ada-002 | text-embedding-3-small | Custom]

    2. Vector Search Implementation¶

    • Description: Implement and optimize vector search for document retrieval
    • Input: Document corpus, query requirements, performance targets
    • Output: Vector search implementation, index configuration, query optimization
    • Example: P06: Implement Azure AI Search with hybrid search (vector + keyword) for eva-rag

    Input Template: 

    P06 REQUEST: Vector Search
  CORPUS_SIZE: [e.g., "10,000 documents"]
  VECTOR_STORE: [Azure AI Search | Pinecone | Weaviate | Cosmos DB]
  SEARCH_STRATEGY:
    - [Strategy 1, e.g., "Dense vector (embeddings)"]
    - [Strategy 2, e.g., "Sparse vector (BM25)"]
    - [Strategy 3, e.g., "Hybrid (combined)"]
  PERFORMANCE:
    - latency: [e.g., "<500ms p95"]
    - recall: [e.g., ">90% at k=5"]
  FILTERS: [Metadata filters needed, e.g., "by date, by category"]

    3. Retrieval Quality Analysis¶

    • Description: Evaluate retrieval quality and identify improvement opportunities
    • Input: Query set, retrieved results, ground truth (if available)
    • Output: Quality metrics, failure analysis, improvement recommendations
    • Example: P06: Analyze retrieval quality for 100 sample queries, identify low-recall cases

    Input Template: 

    P06 REQUEST: Retrieval Analysis
  QUERY_SET:
    - [Query 1, e.g., "What are the contract terms?"]
    - [Query 2, e.g., "Summarize financial data"]
  METRICS:
    - [Metric 1, e.g., "Precision@5"]
    - [Metric 2, e.g., "Recall@5"]
    - [Metric 3, e.g., "MRR (Mean Reciprocal Rank)"]
  GROUND_TRUTH: [Available | Not available]
  FAILURE_CRITERIA: [When retrieval is considered failed]
  DELIVERABLES: [Metrics report, failure cases, recommendations]

    4. Query Rewriting & Optimization¶

    • Description: Design query transformation strategies to improve retrieval
    • Input: User queries, retrieval failures, query patterns
    • Output: Query rewriting rules, expansion strategies, implementation
    • Example: P06: Implement query expansion for ambiguous questions (add synonyms, context)

    Input Template: 

    P06 REQUEST: Query Optimization
  PROBLEM:
    - [Issue 1, e.g., "Ambiguous queries return wrong docs"]
    - [Issue 2, e.g., "Synonym mismatches"]
  STRATEGIES:
    - [Strategy 1, e.g., "Query expansion (add synonyms)"]
    - [Strategy 2, e.g., "Query rewriting (rephrase)"]
    - [Strategy 3, e.g., "Multi-query (ask multiple ways)"]
  QUERY_EXAMPLES:
    - original: [Original query]
      rewritten: [How it should be rewritten]
  IMPLEMENTATION: [LLM-based | Rule-based | Hybrid]

    5. Context Window Management¶

    • Description: Optimize context selection for LLM to fit token limits
    • Input: Retrieved chunks, LLM token limit, ranking strategy
    • Output: Context selection algorithm, reranking implementation
    • Example: P06: Implement reranking to fit top 5 chunks in 8K token window for GPT-4

    Input Template: 

    P06 REQUEST: Context Management
  LLM_MODEL: [GPT-4 | GPT-4-turbo | Claude]
  TOKEN_LIMIT: [e.g., "8000 tokens"]
  RETRIEVAL_COUNT: [e.g., "Top 20 chunks"]
  RERANKING_STRATEGY:
    - [Strategy 1, e.g., "Semantic reranking (Cohere)"]
    - [Strategy 2, e.g., "Diversity selection"]
    - [Strategy 3, e.g., "Recency weighting"]
  FINAL_CONTEXT: [e.g., "Top 5 chunks after reranking"]

    Service Standards¶

    Embedding Standards¶

    • Model: text-embedding-3-small (default), text-embedding-ada-002 (fallback)
    • Dimensions: 1536 (ada-002), 512 or 1536 (text-3-small)
    • Batch size: 16 documents per API call
    • Rate limiting: Respect OpenAI rate limits
    • Caching: Cache embeddings to avoid recomputation

    Chunking Standards¶

    • Chunk size: 512-1024 tokens (fits most LLM contexts)
    • Overlap: 10-20% (preserves context across boundaries)
    • Preserve structure: Keep section headers, numbered lists intact
    • Metadata: Store chunk index, parent document, section title
    • Deduplication: Remove near-duplicate chunks (>95% similarity)

    Retrieval Standards¶

    • Recall@5: 85% (user's answer in top 5 chunks)
    • Latency: <500ms p95 (fast user experience)
    • Hybrid search: Combine vector (semantic) + keyword (exact match)
    • Reranking: Apply semantic reranker for top 5-10 results
    • Filtering: Support metadata filters (date, category, etc.)

    Quality Monitoring Standards¶

    • Log all queries and retrieved chunks
    • Track retrieval metrics (precision, recall, latency)
    • Monthly quality review (analyze failure cases)
    • A/B test retrieval changes (before/after comparison)

    Common Patterns¶

    Pattern 1: Chunking Configuration¶

    # Chunking strategy for legal documents
CHUNKING_CONFIG = {
    "chunk_size": 1024,  # tokens
    "chunk_overlap": 150,  # tokens (15% overlap)
    "separators": ["\n\n", "\n", ". ", " "],  # Try in order
    "preserve_structure": True,  # Keep section headers
    "metadata": {
        "chunk_index": "auto",  # 0, 1, 2, ...
        "parent_doc_id": "required",
        "section_title": "extract",  # Extract from content
    }
}

# Example chunk metadata
{
    "chunk_id": "doc123_chunk5",
    "parent_doc_id": "doc123",
    "chunk_index": 5,
    "section_title": "Article 3: Payment Terms",
    "chunk_text": "...",
    "embedding": [0.123, -0.456, ...]
}

    Pattern 2: Hybrid Search Query¶

    # Azure AI Search hybrid query
search_query = {
    "search": user_query,  # Keyword search (BM25)
    "vectors": [
        {
            "value": query_embedding,  # Vector search
            "k": 20,  # Retrieve top 20
            "fields": "embedding"
        }
    ],
    "queryType": "semantic",  # Semantic ranking
    "semanticConfiguration": "default",
    "filter": f"category eq '{category}' and date ge {start_date}",
    "select": "chunk_id,chunk_text,section_title,parent_doc_id",
    "top": 5  # Return top 5 after reranking
}

    Pattern 3: Retrieval Quality Report¶

    ## Retrieval Quality Report: [Date]

**Corpus**: 10,000 documents, 150,000 chunks
**Query Set**: 100 queries (user logs)
**Evaluation Period**: Week of Dec 2-8, 2025

### Metrics
| Metric | Target | Actual | Status |
|--------|--------|--------|--------|
| Precision@5 | 80% | 87% |  Exceeds |
| Recall@5 | 85% | 82% |  Below |
| MRR | 0.70 | 0.74 |  Exceeds |
| Latency (p95) | <500ms | 420ms |  Meets |

### Failure Analysis
**Low Recall Cases** (18% of queries):
1. **Ambiguous queries** (8%): "What's the process?"  Needs context
   - Fix: Implement query clarification prompt
2. **Multi-document queries** (6%): "Compare contracts A and B"
   - Fix: Multi-document retrieval strategy
3. **Date-specific queries** (4%): "What changed in 2024?"
   - Fix: Temporal reranking

### Recommendations
1. **P0**: Implement query clarification for ambiguous queries
2. **P1**: Add temporal weighting to reranker
3. **P2**: Experiment with larger chunk overlap (20%)

**Next Review**: Dec 16, 2025

    File Naming Convention¶

    All deliverables must follow: 

    P06-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P06-CHUNKING-STRATEGY-LEGAL-DOCS-20251209-120000.md - P06-VECTOR-SEARCH-IMPL-20251209-130000.md - P06-RETRIEVAL-QUALITY-WEEK50-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P06- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md
