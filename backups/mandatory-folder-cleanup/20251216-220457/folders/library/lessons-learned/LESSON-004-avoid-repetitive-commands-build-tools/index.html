    [LESSON-004] Avoid Repetitive Commands - Build Reusable Tools¶

    Pod: Cross-Pod (applies to all pods) Role: Dev Master & Scrum Master Sprint: Sprint-01 (Week 0 - Planning) Date: 2025-11-17 Tags: automation, productivity, tooling, dx, best-practices

    ğŸ“‹ Context¶

    During EVA 2.0 development, agents (Dev Masters, Scrum Master) frequently execute repetitive terminal commands:

    • Git operations: git add -A; git commit -m "..."; git push origin main
    • File searches: Finding specific files across 17 repos
    • Code generation: Creating similar files with templates
    • Status checks: Reading sprint board, issue status, PR reviews
    • Dependency management: Installing packages, updating lockfiles

    Problem recognized: Marco (Product Owner) observed agents repeating manual commands instead of creating reusable tools.

    Key insight: "Create your own tools and leverage other peers' tools from the shared POD Tools box"

    âŒ Problem¶

    Symptom: Repetitive Manual Work¶

    Example 1: Git workflow repeated 10+ times per sprint:

    git add -A
git commit -m "Update X"
git push origin main

    Time wasted: 30 seconds Ã— 10 commits = 5 minutes/sprint

    Example 2: Searching for files across repos:

    grep -r "pattern" eva-core/
grep -r "pattern" eva-api/
grep -r "pattern" eva-ui/
# ... repeat for all 17 repos

    Time wasted: 2 minutes Ã— 5 searches = 10 minutes/sprint

    Example 3: Creating similar markdown files:

    # Manually copy template
# Manually fill in placeholders
# Manually update metadata

    Time wasted: 3 minutes Ã— 8 files = 24 minutes/sprint

    Total waste: ~40 minutes/sprint on repetitive tasks

    Root Cause¶

    1. No automation mindset: Agents default to manual execution instead of building tools
    2. No shared toolbox: Each agent builds tools independently (duplication)
    3. No discoverability: Existing tools not documented or shared
    4. No incentive: Faster to do manual work than build tool (short-term thinking)

    Impact¶

    • âŒ Developer experience suffers (tedious, error-prone work)
    • âŒ Velocity reduced (time wasted on repetitive tasks)
    • âŒ Quality issues (manual work = typos, missed steps)
    • âŒ Knowledge silos (each agent reinvents the wheel)
    • âŒ No compounding productivity (same manual work every sprint)

    âœ… Solution¶

    Principle: "Automate on Third Repetition"¶

    Rule: If you do the same task 3 times, stop and build a reusable tool.

    Example decision tree:

    Task needed?
  â†“

  â†“

  â†“

  â†“

    1. Quick Commit (Git)¶

    2. Find Across Repos (Search)¶

    File: tools/file-tools/find-across-repos.ps1

    # Usage: .\find-across-repos.ps1 "search pattern"
param(
    [Parameter(Mandatory=$true)]
    [string]$Pattern
)

$repos = @(
    "eva-core", "eva-auth", "eva-agent",
    "eva-api", "eva-mcp", "eva-openai",
    "eva-ui", "eva-i11y", "eva-i18n",
    "eva-infra", "eva-ops", "eva-metering",
    "eva-safety", "eva-seed", "eva-rag", "eva-enterprise"
)

Write-Host "ğŸ” Searching for: $Pattern" -ForegroundColor Cyan

foreach ($repo in $repos) {
    $repoPath = Join-Path $PSScriptRoot "../../$repo"
    if (Test-Path $repoPath) {
        Write-Host "`nğŸ“¦ $repo" -ForegroundColor Yellow
        git -C $repoPath grep -n $Pattern 2>$null
    }
}

    Before (manual): 17 grep commands, 2 minutes After (tool): 1 command, 15 seconds Savings: 1.75 minutes per search Ã— 5 searches = 9 minutes/sprint

    3. Create From Template (File Generation)¶

    File: tools/file-tools/create-from-template.ps1

    # Usage: .\create-from-template.ps1 -Template "lesson" -Output "LESSON-005.md"
param(
    [Parameter(Mandatory=$true)]
    [string]$Template,

    [Parameter(Mandatory=$true)]
    [string]$Output
)

$templatePath = "docs/templates/$Template-template.md"
$outputPath = "docs/lessons-learned/$Output"

if (-not (Test-Path $templatePath)) {
    Write-Error "Template not found: $templatePath"
    exit 1
}

# Copy template
Copy-Item $templatePath $outputPath

# Auto-fill metadata
$date = Get-Date -Format "yyyy-MM-dd"
(Get-Content $outputPath) `
    -replace 'YYYY-MM-DD', $date `
    -replace 'LESSON-NNN', ($Output -replace '\.md$', '') |
    Set-Content $outputPath

Write-Host "âœ… Created: $outputPath" -ForegroundColor Green
code $outputPath  # Open in VS Code

    Before (manual): Copy, paste, find/replace, 3 minutes After (tool): 1 command, 10 seconds Savings: 2.8 minutes per file Ã— 8 files = 22 minutes/sprint

    When to Create a Tool¶

    1. Third repetition of the same task
    2. Complex multi-step workflow (>3 commands)
    3. Error-prone manual process (typos, missed steps)
    4. Cross-repo operation (needs to touch multiple repos)

    How to Add a Tool¶

    1. Create script in appropriate category folder
    2. Add usage example + description to this README
    3. Test tool (dry-run mode if destructive)
    4. Share in sprint retro (demo + celebrate)
    5. Update agent memory files (link to tool)

    Tool Quality Standards¶

    • âœ… Clear parameters (with help text)
    • âœ… Error handling (exit codes, validation)
    • âœ… Dry-run mode (for destructive operations)
    • âœ… Progress feedback (Write-Host with colors)
    • âœ… Documentation (usage examples in header)

## ğŸ“š Lesson Learned

### Key Principles

1. **"Automate on Third Repetition"**
   - First time: Learn (manual OK)
   - Second time: Validate (manual OK)
   - Third time: **BUILD TOOL** (stop wasting time)
   - Fourth+ time: Use tool (profit forever)

2. **"Shared POD Tools Box"**
   - All tools in `tools/` directory
   - Discoverable via README catalog
   - Reusable across all agents
   - Versioned with git (rollback if broken)

3. **"Build for Peers, Not Just Yourself"**
   - Document usage clearly
   - Handle edge cases
   - Provide examples
   - Celebrate tool creation (retro demos)

4. **"Compose, Don't Duplicate"**
   - Call existing tools from new tools
   - Build pipelines (tool A â†’ tool B â†’ tool C)
   - DRY principle (Don't Repeat Yourself)

### When to Apply

**High-value automation targets**:
- âœ… Git workflows (commit, push, merge, cleanup)
- âœ… Cross-repo operations (search, status, tests)
- âœ… File generation (from templates)
- âœ… Sprint ceremonies (bootstrap, retro, burn chart)
- âœ… Quality gates (lint, test, coverage, security)

**Low-value automation** (not worth tooling):
- âŒ One-off tasks (used once, never again)
- âŒ Simple single commands (no composition needed)
- âŒ Highly variable tasks (too many edge cases)
