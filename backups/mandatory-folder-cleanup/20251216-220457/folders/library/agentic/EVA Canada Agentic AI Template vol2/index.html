    Government of Canada Agentic AI Template¶

    Volume 2 – Engineering & Delivery Guide (Template)¶

    Subtitle: How We Build and Evolve AI Agent Platforms for Federal Use
    Version: Draft 0.2
    Date: YYYY-MM-DD

    A. Instantiation Header (to be filled by each department)¶

    • Primary Code Hosting Platform: {{e.g., GitHub Enterprise, Azure DevOps, GitLab}}

    1. Purpose¶

    This Volume defines the engineering and delivery model for building and evolving a departmental AI agent platform (the “Platform”) suitable for use in the Government of Canada.

    Where:

    • Volume 1 describes what the Platform and agents must do,
    • Volume 2 describes how we build it in a repeatable, auditable, and secure way.

    It is intended to:

    • Provide a standard engineering playbook for teams building Platform components and agents.
    • Ensure consistency across:
    • Repositories and branching.
    • Development environments.
    • CI/CD and quality gates.
    • AI-assisted development practices.

    2. Scope¶

    2.1 In Scope¶

    • Engineering practices and workflows for the Platform and its services:
    • Repository structure and organization.
    • Development environments (dev containers, remote dev, local).
    • Branching strategy and pull request workflow.
    • CI/CD pipelines and environments.
    • Testing, security scanning, and quality guardrails.
    • AI-assisted development (e.g., Copilot-type tools) guidelines.
    • Documentation, architecture decisions, and knowledge sharing.
    • Team structure (pods / squads) for delivering Platform features.

    2.2 Out of Scope¶

    • Platform requirements and governance (Volume 1).
    • Operational and LiveOps processes (Volume 3).
    • Use case patterns and agent archetypes (Volume 4).

    3. Audience¶

    • Developers & Engineers – building Platform services and agents.
    • DevOps / Cloud / Infrastructure Teams – managing pipelines and environments.
    • Technical Leads & Architects – enforcing patterns and reviewing designs.
    • AI / Data Engineers – integrating models and data pipelines.
    • Scrum Masters / Delivery Managers – organizing work for pods/squads.

    4. Engineering Principles¶

    Each principle is tagged as [CORE] (recommended minimum) or [ADV] (for more mature teams).

    1. Single Product Mindset
      The Platform is treated as one product, even if composed of multiple services/repos.
      Tag: [CORE]

    2. Small, Frequent, Tested Changes
      Use short-lived branches and pull requests with automated tests and checks.
      Tag: [CORE]

    3. Standardized Development Environments
      All developers use shared dev environment definitions (e.g., Dev Containers) so onboarding is fast and “works on my machine” issues are minimized.
      Tag: [CORE]

    4. Automation First
      Builds, tests, security scans, and deployments are automated via CI/CD pipelines.
      Tag: [CORE]

    5. AI-Assisted, Human-Governed Development
      AI tools accelerate coding and documentation, but humans remain accountable for correctness, security, and compliance.
      Tag: [CORE]

    6. Security and Compliance by Design
      Security scanning, secrets management, and dependency hygiene are built into the SDLC, not bolted on.
      Tag: [CORE]

    7. Documentation as a First-Class Artifact
      Each repo includes clear documentation (README, architecture notes, ADRs) so new team members can onboard without relying on institutional memory.
      Tag: [CORE]

    8. Inner Source & Reuse
      Shared components (auth, logging, UI, RAG building blocks) are designed for reuse by other teams within the department.
      Tag: [ADV]

    5. Repository & Organization Structure¶

    5.1 Code Hosting Platform¶

    • R-Repo.1 [CORE] – The Platform’s source code SHALL be hosted in a managed, departmental-approved code hosting platform (e.g., GitHub Enterprise, Azure DevOps, GitLab) with:
    • Enterprise authentication (SSO where possible).
    • Organization/team-based access controls.
    • Audit logging for repository access and changes.

    5.2 Organization-Level Configuration¶

    • R-Repo.2 [CORE] – A dedicated organization/project space SHOULD be created for Platform repositories (e.g., gc-ai-platform) with:
    • Default branch protection rules.
    • Default security policies.

    • Shared templates for issues, pull requests, and workflows.

    • R-Repo.3 [ADV] – A .github or equivalent “meta” repository SHOULD provide:

    • Shared issue templates (bug, feature, tech debt).
    • Shared pull request template with quality checklist.
    • Organization-wide CONTRIBUTING and SECURITY guidelines.

    5.3 Example Repository Layout¶

    Departments may adapt this, but a typical structure could be:

    • platform-orchestrator – central orchestration / API composition.
    • platform-core – shared domain models, utilities.
    • platform-rag – retrieval-augmented generation pipeline components.
    • platform-agent-runtime – orchestration and tools for agents.
    • platform-api – external-facing APIs.
    • platform-ui – web UIs (chat, domain assistant, internal dashboards).
    • platform-auth – authentication/authorization integration.
    • platform-i18n – localization resources and tooling.
    • platform-a11y – shared accessibility and personalization components.
    • platform-infra – infrastructure-as-code and environment definitions.
    • platform-ops – scripts, monitoring configurations, runbook helpers.

    R-Repo.4 [CORE] – Each repo SHOULD have:

    • README.md – purpose, quick start, how to run tests.
    • CONTRIBUTING.md (or common org-level link).
    • /src – source code.
    • /tests – automated tests.
    • /.devcontainer – dev container configuration (see Section 6).
    • /.github/workflows (or equivalent) – CI/CD workflows.
    • .editorconfig or equivalent – common formatting rules.

    6. Development Environments¶

    6.1 Dev Containers as Canonical Definition¶

    • R-Env.1 [CORE] – Each repo SHOULD define a Dev Container configuration (e.g., .devcontainer/devcontainer.json) specifying:
    • Base image (language/runtime + tooling).
    • Required OS packages (e.g., build-essential, curl).
    • Language-level tools (e.g., Node, .NET, Python, Java).
    • Recommended editor extensions (linting, formatting, test runners).

    • Exposed ports (e.g., 3000 for frontend, 5000 for APIs).

    • R-Env.2 [CORE] – The Dev Container definition SHALL be the single source of truth for the dev environment used by:

    • Remote dev environments (e.g., GitHub Codespaces).
    • Local dev using Dev Containers (VS Code / other supported tools).

    6.2 Remote Dev Environments (e.g., Codespaces)¶

    • R-Env.3 [ADV] – The Platform SHOULD support remote development environments (e.g., Codespaces, remote hosted containers) for:
    • Rapid onboarding.
    • Thin-client or restricted-device scenarios.
    • Training and workshops.

    Recommendations:

    • Provide project-level Codespaces/remote env presets for:
    • Backend development.
    • Frontend/UI development.
    • Infrastructure/Terraform work.

    6.3 Local Environments¶

    • R-Env.4 [CORE] – Developers who run code locally SHOULD:
    • Use the same dev container configuration (via Dev Containers).
    • Avoid bespoke, undocumented local setups.

    6.4 Config & Secrets¶

    • R-Env.5 [CORE] – All repo configuration SHALL:
    • Use templates (e.g., .env.example, appsettings.template.json).

    • Avoid committing real secrets or environment-specific values.

    • R-Env.6 [CORE] – Secrets for CI/CD and runtime MUST be stored in:

    • Department-approved secret management systems (e.g., Key Vault, Secret Manager).
    • Code hosting platform secrets feature for pipeline-only uses.

    7. Branching Strategy & Workflow¶

    7.1 Trunk or Trunk+Develop¶

    • R-Branch.1 [CORE] – The Platform SHALL use a clear, documented branching strategy, typically one of:

    • Trunk-based development:

      • main branch is always releasable.
      • Short-lived feature branches merge frequently; feature flags used where needed.

    • Short Git Flow variant:

      • main (production).
      • develop (integration).
      • Short-lived features off develop.

    Departments may choose based on team maturity, but complexity should be minimized.

    7.2 Short-Lived Feature Branches¶

    • R-Branch.2 [CORE] – Changes SHALL be made via short-lived feature branches, named consistently, for example:
    • feature/<short-description>
    • bugfix/<short-description>
    • chore/<short-description>

    7.3 Pull Requests (PRs)¶

    • R-Branch.3 [CORE] – All changes to main/develop MUST be merged via pull requests that:
    • Link to at least one issue or work item.
    • Include a description of intent and impact.

    • Use a PR template with a quality checklist (tests, docs, a11y, security considerations).

    • R-Branch.4 [CORE] – Branch protection rules SHALL require:

    • At least one approving review.

    • All required CI checks passing (see Section 8).

    • R-Branch.5 [ADV] – A CODEOWNERS file SHOULD define per-path code owners to ensure the right reviewers are involved.

    8. CI/CD Pipelines¶

    8.1 CI: Continuous Integration¶

    • R-CI.1 [CORE] – Each repo SHALL have CI workflows that run on PRs and pushes to main/develop, including:
    • Checkout.
    • Dependency restoration.
    • Build/compile.
    • Unit tests.

    • Linting/formatting checks.

    • R-CI.2 [ADV] – CI SHOULD also include:

    • Code coverage measurement.
    • Static code analysis.
    • Security scanning where available (e.g., CodeQL, language-specific scanners).

    8.2 CD: Continuous Deployment¶

    • R-CD.1 [CORE] – Deployments to shared non-prod environments (e.g., Dev, Test) SHOULD be automated via pipelines triggered by:
    • Branch merges.

    • Tag creation (e.g., vX.Y.Z).

    • R-CD.2 [CORE] – Deployments to Production SHALL:

    • Be driven by an automated pipeline.
    • Require explicit approvals from authorized roles.
    • Use versioned artifacts (e.g., containers, packages).

    8.3 Environments & Pipelines¶

    Typical pipeline structure:

    • ci.yml – build, test, lint on PR + push.
    • deploy-dev.yml – deploy to Dev environment.
    • deploy-test.yml – deploy to Test environment.
    • deploy-prod.yml – deploy to Prod (with approval gates).

    R-CD.3 [ADV] – Infrastructure SHOULD be deployed via IaC (Infrastructure-as-Code), integrated into pipelines (e.g., Terraform/Bicep/CloudFormation jobs).

    9. Quality & Security Guardrails¶

    9.1 Testing¶

    • R-Qual.1 [CORE] – New features SHALL be accompanied by appropriate tests (unit and/or integration) covering at least:
    • Happy paths.

    • Key failure modes.

    • R-Qual.2 [ADV] – Critical components (agent runtime, API gateway integrations, RAG pipeline) SHOULD:

    • Maintain a minimum test coverage threshold (e.g., 70% for core logic).
    • Include end-to-end tests for major flows.

    9.2 Linting & Formatting¶

    • R-Qual.3 [CORE] – Repos SHOULD:
    • Use standard linters (ESLint, Pylint, etc.) and formatters (Prettier, dotnet format).
    • Enforce lint/format checks in CI.

    9.3 Dependency Management¶

    • R-Sec.1 [CORE] – Automated dependency update tooling (e.g., Dependabot, Renovate) SHOULD be enabled to:
    • Detect and propose updates to third-party libraries.

    • Particularly for security patch versions.

    • R-Sec.2 [ADV] – Teams SHOULD define a cadence and policy for reviewing and merging dependency updates.

    9.4 Security & Code Scanning¶

    • R-Sec.3 [CORE] – For security-sensitive repositories (APIs, agent runtime, infra), code scanning SHOULD be enabled (e.g., SAST tools like CodeQL).

    • R-Sec.4 [CORE] – Identified high/critical vulnerabilities MUST be triaged and addressed with:

    • Fixes, or
    • Documented, time-bound exceptions approved by security.

    9.5 Secrets Management¶

    • R-Sec.5 [CORE] – Secrets SHALL NOT be committed to code repositories.
    • R-Sec.6 [CORE] – Pre-commit hooks or CI checks SHOULD be used to detect accidental secrets in commits.
    • R-Sec.7 [CORE] – Runtime secrets MUST be stored in departmental secret management tooling and injected via environment/configuration.

    10. AI-Assisted Development (e.g., Copilot-Type Tools)¶

    10.1 Use Principles¶

    • R-AI-Dev.1 [CORE] – Developers MAY use AI-assisted coding tools, subject to departmental policy, provided:
    • No prohibited data (e.g., classified, highly sensitive) is sent to external services.

    • They understand and review the generated code.

    • R-AI-Dev.2 [CORE] – AI-generated code MUST:

    • Be treated like any other code for review and testing.
    • Not be merged solely because “the tool suggested it”.

    10.2 Good Practices¶

    • Use AI tools to:
    • Scaffold boilerplate code.
    • Suggest tests and documentation.

    • Generate initial versions of functions, which are then refined.

    • Avoid using AI tools to:

    • Introduce unknown or unvetted third-party dependencies.
    • Copy proprietary code from unlicensed sources.

    10.3 Documentation & Traceability¶

    • R-AI-Dev.3 [ADV] – For critical components, teams MAY record where AI tools assisted major designs or algorithms (e.g., ADR note: “Initial design drafted with AI assistant; finalized and reviewed by X”).

    11. Environments & Configuration Management¶

    11.1 Environment Tiers¶

    Recommended set:

    • Dev – High churn, internal use by Platform team.
    • Test / QA – Integration and regression testing.
    • Stage / UAT – Pre-production validation, mirrors Production as closely as feasible.

    • Prod – Production environment used by real users.

    • R-EnvTier.1 [CORE] – Environment purposes and promotion rules SHALL be documented and respected.

    11.2 Configuration Strategy¶

    • R-Config.1 [CORE] – Application configuration MUST be externalized from code (env variables, config files).
    • R-Config.2 [CORE] – Each environment SHALL have properly separated configuration/secrets.

    11.3 Feature Flags¶

    • R-Config.3 [ADV] – Feature flagging SHOULD be used for:
    • Gradual rollout of new features.
    • Enabling internal testing for small user groups.

    12. Documentation & Knowledge Practices¶

    12.1 Minimal Documentation Set¶

    • R-Doc.1 [CORE] – Each repo MUST provide:
    • README.md – purpose, architecture overview, how to run/build/test.

    • “How to contribute” guidance (directly or via link to org-level CONTRIBUTING).

    • R-Doc.2 [ADV] – Repos SHOULD maintain:

    • /docs directory for deeper design notes.
    • Architecture diagrams (even if simple text-based or ASCII).

    12.2 Architecture Decision Records (ADRs)¶

    • R-Doc.3 [ADV] – Significant decisions (e.g., choice of agent orchestration pattern, RAG library, logging approach) SHOULD be recorded as ADRs under /docs/adr.

    12.3 API & Contract Documentation¶

    • R-Doc.4 [CORE] – APIs SHOULD:
    • Be documented via OpenAPI/Swagger (or equivalent).
    • Provide examples for common integration scenarios.

    13. AI Agile Pods / Squads¶

    13.1 Pod Structure¶

    • R-Pod.1 [CORE] – Delivery teams SHOULD be cross-functional, including:
    • Product owner or product representative.
    • Tech lead / senior developer.
    • Backend and/or frontend developers.
    • QA/test role (dedicated or shared).
    • DevOps/Cloud engineer (dedicated or shared).
    • Optional: AI/ML specialist where needed.

    13.2 Pod Responsibilities¶

    • Deliver Platform features end-to-end, including:
    • Design and implementation.
    • Tests and documentation.

    • Support for deployment and initial operational handover.

    • R-Pod.2 [CORE] – Pods SHALL adhere to the engineering standards in this Volume, regardless of their working methodology (Scrum, Kanban, etc.).

    14. Definition of Ready (DoR) & Definition of Done (DoD)¶

    14.1 Definition of Ready (Example)¶

    A backlog item is Ready when:

    • Business goal and acceptance criteria are clear.
    • Impacted services/repos are identified.
    • Dependencies and risks are known or documented.
    • Security/privacy/accessibility implications are noted (even “none”).

    14.2 Definition of Done (Example)¶

    A backlog item is Done when:

    • Code changes are implemented and build successfully.
    • Relevant tests are added/updated and pass.
    • Linting and security checks pass.
    • Documentation updated where applicable (README, API docs, ADR).
    • PR reviewed and approved; CI pipeline green.

    • Deployed or ready for deployment to the appropriate environment.

    • R-DoD.1 [CORE] – Teams SHALL define and adopt a DoD that includes at least tests, security checks, and documentation updates and may extend this baseline as maturity grows.

    15. Tailoring Guidance for Departments¶

    When adopting this Volume, departments SHOULD:

    1. Fill the Instantiation Header (Section A) to contextualize this guide.
    2. Confirm or adjust repo structure to match internal standards.
    3. Define a canonical dev environment (Dev Containers + remote dev option).
    4. Pick a branching strategy (trunk or trunk+develop) and document it.
    5. Establish CI/CD templates for common project types (API, UI, infra).
    6. Clarify AI-assisted development policy based on departmental rules.
    7. Align with Volume 1:
    8. Ensure engineering practices support observability, trust posture, and autonomy enforcement.
    9. Align with Volume 3:
    10. Ensure deployment and logging enable LiveOps monitoring, incident handling, and audits.

    16. Relationship to Other Volumes¶

    • Volume 1 – Platform & Agentic Requirements
      Provides the functional and non-functional requirements the Platform must satisfy. Volume 2 shows how to build against those requirements.

    • Volume 3 – LiveOps & Governance Model
      Relies on the engineering practices here (logging, CI/CD, environment practices) to:

    • Monitor health and usage.
    • Support incidents and kill switch operations.

    • Demonstrate compliance and trust.

    • Volume 4 – Use Cases, Patterns & Templates
      Feeds the engineering backlog with concrete:

    • Agent archetypes,
    • Patterns, and
    • Example journeys from idea to pilot to production.
