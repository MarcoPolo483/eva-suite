    EVA Jarvis DevMaster – GitHub & Copilot Implementation Guide¶

    Version: 0.3 Audience: Jarvis (EVA DevMaster), GitHub Copilot Pro+, EVA AI Agile Dev Crew Scope: EVA Suite repos hosted on GitHub (local orchestrator + cloud workflows)

    1. Purpose & Context¶

    This guide explains how Jarvis (EVA DevMaster) should use:

    • GitHub (Projects, Issues, Actions, Environments, Codespaces)
    • GitHub Copilot Pro+ (Copilot Chat + Copilot coding agent)
    • Existing EVA orchestrator assets (local Python agents, CDDs, MD docs)

    to run, as autonomously as possible, an AI Agile Dev Crew that:

    1. Starts from CDDs (Conceptual Design Documents).
    2. Recursively refines requirements into sprintable stories.
    3. Plans sprints via YAML + GitHub Projects.
    4. Executes work using Copilot coding agent + CI/CD.
    5. Reaches “finished, tested, UAT-ready code.”

    Jarvis must treat GitHub as the primary control plane, and the local EVA orchestrator (info-assistant-reference folder) as a companion/bridge, not a separate world.

    3. High-Level Architecture in GitHub¶

    3.1 Roles¶

    • Jarvis (EVA DevMaster) A set of GitHub Actions + scripts (and optionally local orchestrator hooks) that:

    • Refine requirements (from CDDs → backlog).

    • Plan sprints (YAML + Projects).
    • Orchestrate execution (Pods, coding agents, CI/CD, UAT).

    • Generate reports for Scrum ceremonies.

    • Copilot Pro+ & Copilot coding agent The “hands” that:

    • Implement code and tests.

    • Modify MD/YAML/JSON files.

    • Open PRs based on issues and instructions.

    • EVA DevTools Agents (conceptual) Patterns like Requirements Agent, Review Agent, Testing Agent, Security Helper, etc., realized as:

    • Copilot prompt profiles.

    • Scripts invoked by Actions.
    • Config rules in EVA DevTools runtime.

    3.2 Two Main Pipelines¶

    1. Planning Pipeline (Workflow A: Jarvis-Plan)

    2. CDD → epics/features → recursive refinement → sprintable stories/tasks → sprints/sprint-XXX.yaml → sprint board.

    3. Execution Pipeline (Workflow B: DevMaster Orchestration)

    4. Sprint plan → assignments → coding agent + CI/CD + QA → UAT → finished, tested code.

    4.3 GitHub Projects, Labels, and Fields¶

    • Project(s): Create an EVA Suite project with fields:

    • Level = initiative / epic / feature / story / task

    • Status = idea / needs-refinement / ready-for-sprint / in-progress / in-review / done / blocked
    • Pod = POD-F, POD-X, POD-O, POD-S-Juris, POD-S-Assist, etc.
    • Repo = eva-api, eva-core, etc.
    • Sprint = numeric (12, 13, etc.)
    • CDD-ID = e.g., CDD-04

    • Estimate = points or t-shirt size

    • Labels:

    • level:epic, level:feature, level:story, level:task

    • needs:refinement
    • ready-for-sprint
    • type:feature, type:bug, type:infra, type:doc
    • pod:POD-F, pod:POD-X, pod:POD-O, pod:POD-S-...
    • sprint:012 (optional if not using Project field)

    Jarvis will rely heavily on these fields and labels.

    5. Process 1 – From CDD to Sprint Plan (Planning Pipeline)¶

    5.1 Levels of Refinement¶

    Jarvis must use a consistent “ladder”:

    • level:initiative (optional, top)
    • level:epic
    • level:feature
    • level:story
    • level:task

    A work item is sprintable when:

    • It is level:story or level:task.
    • It has Repo, Pod, Estimate.
    • It has acceptance criteria.
    • It can be delivered by one pod in one sprint as one or a small number of PRs.

    Jarvis’s recursive refinement rule:

    If an item cannot be delivered by one pod in one sprint as ~1–2 PRs, or lacks repo, pod, estimate, or acceptance criteria, you must refine it into smaller child items.

    5.2 CDD → Epics/Features¶

    Inputs:

    • CDD files in eva-meta/docs/cdd/ or project repos.
    • Example: eva-meta/docs/cdd/cdd-eva-suite.txt.

    Jarvis (Requirements Agent P02) should:

    1. Read the CDD.

    2. Create issues for epics and features:

    3. level:epic or level:feature

    4. needs:refinement
    5. Link back to the CDD file and section.

    6. Label issues with:

    7. type:* (feature/infra/doc)

    8. pod:* if obvious, or leave for later.

    5.3 Recursive Refinement (Workflow jarvis-refine)¶

    Workflow name: jarvis-refine.yml in eva-meta.

    Trigger:

    • Manual: /jarvis refine comment, or
    • workflow_dispatch with optional filters (pod, repo, CDD-ID).

    Behaviour:

    1. List issues with:

    2. needs:refinement and level:epic or level:feature.

    3. For each parent issue:

    4. Read issue body and referenced CDD section.

    5. Read EVA Agentic Pods Map.md to know pods and repos.

    6. Call a Requirements Agent (via EVA DevTools / Azure OpenAI) with:

      • Parent text.
      • CDD excerpt.
      • Desired target level (features for epics, stories/tasks for features).
      • From AI output:

    7. Create child issues with:

      • Title, description, acceptance criteria.
      • level:*, type:*, pod:* labels.
      • Repo hints in body.
      • Link children to parent (child “blocks” parent).
      • Decide if parent still needs refinement:

    8. If coverage is insufficient → keep needs:refinement.

    9. If children cover parent scope → remove needs:refinement.

    10. For any child still vague/big:

    11. Add needs:refinement for later recursion.

    Jarvis stops recursion when there is a sufficient pool of issues with:

    • level:story or level:task
    • ready-for-sprint
    • Repo, Pod, Estimate, Acceptance criteria

    5.4 Sprint Config (sprints/sprint-XXX.yaml)¶

    Jarvis-Plan uses sprint YAML files, for example:

    sprint: 12
team: pod-f-platform
start_date: 2025-11-25
end_date: 2025-12-09
goal: >
  Stand up EVA APIM backbone and DevTools runtime.

items:
  - issue: https://github.com/MarcoPolo483/eva-api/issues/23
    type: feature
    pod: POD-F
    estimate: 5
  - issue: https://github.com/MarcoPolo483/eva-core/issues/17
    type: infra
    pod: POD-F
    estimate: 3

    Copilot Chat should help generate these from “ready-for-sprint” items:

    “From EVA backlog, propose a 2-week sprint (30 points) for pod-f-platform and generate sprints/sprint-012.yaml using the standard format.”

    5.5 jarvis-plan Workflow¶

    Workflow name: jarvis-plan.yml in eva-meta.

    Trigger:

    • Comment: /jarvis plan sprint 12, or
    • Push/PR that changes sprints/sprint-012.yaml.

    Behaviour:

    1. Read sprints/sprint-012.yaml.

    2. For each items[i]:

    3. Verify issue exists.

    4. Ensure labels:

      • sprint:012 or Sprint field in Project.
      • pod:*, type:*, ready-for-sprint.
      • Add to EVA Suite Project with fields set (Sprint, Pod, Repo, Estimate).
      • Set milestone Sprint 12 if using milestones.
      • Create/update:

    5. orchestration/sprint-012-devmaster-log.txt in eva-meta.

    6. Optional repo-local logs under orchestration/.

    7. Post summary comment in a “Sprint 12 – EVA” tracking issue:

    8. Items list, total points, pods involved, dates and goal.

    Result: Sprint 12 is planned and ready for execution.

    6. Process 2 – Sprint Execution (DevMaster Orchestration)¶

    6.1 Preconditions¶

    Jarvis-Execute only starts when:

    • sprints/sprint-XXX.yaml exists and is valid.

    • All referenced issues:

    • Are level:story or level:task.

    • Have ready-for-sprint.
    • Have Pod, Repo, Estimate, acceptance criteria.

    • Service repos:

    • Have .eva-pod.yaml.

    • Have .github/copilot-instructions.md.
    • Have working CI for PRs.

    6.2 jarvis-execute – Start Sprint Execution¶

    Workflow name: jarvis-execute.yml in eva-meta.

    Trigger:

    • Comment: /jarvis start sprint 12.

    Behaviour:

    1. Read sprints/sprint-012.yaml.

    2. For each item:

    3. Confirm repo, Pod and labels.

    4. Ensure status:todo / board column “Todo”.

    5. Assign:

      • Copilot coding agent as assignee.
      • Human owner (pod lead, you) as co-assignee.
      • Create/update:

    6. orchestration/sprint-012-devmaster-log.txt with:

      • Sprint metadata.
      • List of issues per repo/pod.
      • Known cross-pod dependencies.
      • Repo-local logs: orchestration/sprint-012-log.txt (optional).

    This is the moment where the sprint “goes live”.

    6.3 Copilot Coding Agent – Implementation Mode¶

    For each assigned issue, Copilot coding agent should:

    1. Start a GitHub-hosted VM.
    2. Clone the repo.

    3. Read:

    4. .github/copilot-instructions.md.

    5. Relevant docs and CDD references.

    6. Implement:

    7. Code changes.

    8. Tests.
    9. Document updates where indicated.
    10. Create one or more PRs referencing the issue.
    11. Respond to review comments and test failures, iterating until ready.

    Jarvis does not micromanage these edits; it coordinates at the issue/PR level.

    6.4 jarvis-daily – Daily Scrum Automation¶

    Workflow name: jarvis-daily.yml in eva-meta.

    Trigger:

    • Scheduled (e.g. every weekday morning).
    • Manual: /jarvis daily sprint 12.

    Behaviour:

    1. Query EVA Project for Sprint = 12.

    2. For each pod:

    3. Count items by status: Todo, In Progress, In Review, Done, Blocked.

    4. List blocked items and reasons (if available).
    5. List PRs waiting review > N days.

    6. Write:

    7. reports/sprint-012/daily-YYYYMMDD.txt with:

      • Per-pod “yesterday/today/blockers”-style bullets.
      • Key alerts (stale PRs, repeated test failures).
      • Optionally comment on the Sprint 12 tracking issue with a short summary.

    Humans and AI agents use this as the Daily Scrum input.

    6.5 jarvis-review – Sprint Review Summary¶

    Workflow name: jarvis-review.yml in eva-meta.

    Trigger:

    • Comment: /jarvis review sprint 12.
    • Or milestone/Project closure event.

    Behaviour:

    1. Identify Sprint 12 issues:

    2. Done.

    3. Not-done (Todo/In Progress/Blocked).

    4. For Done issues:

    5. Link merged PRs.

    6. Briefly summarize CI results (#passes, #failures).

    7. Compute metrics:

    8. Planned vs completed (stories/points).

    9. Average cycle time.
    10. Defects introduced and fixed during sprint.

    11. Write:

    12. reports/sprint-012/review.txt:

      • Sprint goal and whether met.
      • Delivered items (by repo/pod).
      • Slipped items.
      • Metrics.
      • Comment summary to Sprint 12 tracking issue.

    This feeds the Sprint Review and decision on UAT/release.

    7. UAT and Release (jarvis-release, jarvis-uat)¶

    7.1 jarvis-release – Build UAT Candidate¶

    Workflow name: jarvis-release.yml in eva-meta (or infra repo).

    Trigger:

    • /jarvis release sprint 12.

    Behaviour:

    1. Determine commit(s) that include all Sprint 12 Done items.

    2. Create UAT branch/tag:

    3. Example: release/sprint-012-uat.

    4. Trigger deployment to UAT environment using existing CI/CD.

    5. Initialize:

    6. uat/uat-plan-sprint-012.txt listing each story:

      • Story ID and link.
      • Acceptance criteria.
      • Space for UAT steps, result, notes.

    7.2 jarvis-uat – UAT Support and Tracking¶

    Workflow name: jarvis-uat.yml in eva-meta.

    Trigger:

    • /jarvis uat sprint 12.

    Behaviour:

    1. For each sprint story:

    2. Ensure it appears in uat/uat-plan-sprint-012.txt with acceptance criteria.

    3. Optionally generate draft UAT steps (via agent).

    4. As UAT runs:

    5. New defects → create type:bug issues:

      • Linked to original story and UAT release entry.
      • Prioritized (blocker/high/medium/low).
      • Define UAT acceptance:

    6. A story is UAT-accepted when:

      • No open blocking bugs remain (fixed or explicitly deferred).
      • UAT result is “Pass” or “Accepted with minor issues.”
      • Write:

    7. uat/uat-report-sprint-012.txt with:

      • Accepted stories.
      • Rejected/deferred stories and reasons.
      • Defects summary and status.
      • Recommendations (next sprint, production readiness).

    When conditions are met:

    • Mark Sprint 12 as “Ready for production release” in EVA Project.
    • Optionally trigger production release pipeline and generate docs/releases/release-notes-sprint-012.txt.

    8. jarvis-retro – Retrospective Input¶

    Workflow name: jarvis-retro.yml in eva-meta.

    Trigger:

    • /jarvis retro sprint 12.

    Behaviour:

    1. Read:

    2. reports/sprint-012/review.txt.

    3. uat/uat-report-sprint-012.txt.
    4. All daily-*.txt reports for Sprint 12.

    5. Derive:

    6. Candidate “What went well” items.

    7. “What could be improved” (e.g., long-lived PRs, repeated CI failures, heavy UAT bug load).
    8. Suggested experiments/changes for next sprint.

    9. Write:

    10. reports/sprint-012/retro-input.txt.

    Humans use this as structured input for the Sprint Retrospective.

    “Implement jarvis-release.yml and jarvis-uat.yml workflows in eva-meta to create UAT candidate branches/tags for Sprint 12, generate uat/uat-plan-sprint-012.txt, track UAT results, and produce uat/uat-report-sprint-012.txt.”

    10. Readiness Checklist – AI Agile Dev Crew “Go Live”¶

    Jarvis can consider the AI Agile Dev Crew “ready” when:

    1. Repos and Metadata

    2. eva-meta exists with this guide, lifecycle map, pods map, recipes, CDDs, sprints/, reports/, uat/.

    3. Service repos have .eva-pod.yaml, .github/copilot-instructions.md, standard folders, and CI.

    4. Backlog & Projects

    5. EVA Project is configured with Level, Status, Pod, Repo, Sprint, CDD-ID, Estimate.

    6. Issues from CDDs are created and classified (epics, features, stories, tasks).

    7. Recursive refinement can run via jarvis-refine.

    8. Sprint Mechanics

    9. Sprint YAML files exist and are accepted by jarvis-plan.

    10. jarvis-execute can start a sprint and assign issues to coding agents + humans.

    11. Execution & Reporting

    12. jarvis-daily, jarvis-review, jarvis-release, jarvis-uat, jarvis-retro workflows exist and run successfully.

    13. Reports and logs are written to eva-meta as described.

    14. Governance & Safety

    15. Branch protections and required checks are enabled for main/release branches.

    16. Security and secret scanning are active.
    17. Environments and approvals are configured for UAT and production.

    At that point, Jarvis (backed by GitHub + Copilot Pro+) can run the full EVA AI Agile Dev Crew loop: CDD → refined requirements → planned sprint → coded + tested → UAT-accepted → released, with all steps traceable and auditable.
