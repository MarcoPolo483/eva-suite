    EVA DevTools Agent Pattern – Repo Librarian Agent¶

    Subtitle: Keeping Code, Docs, and Architecture in Sync for the EVA Agile Crew Version: Draft 0.2 Date: YYYY-MM-DD

    Based on Draft 0.1.

    1. Pattern Overview¶

    Agent name: EVA-Repo-Librarian Category: DevTools / Internal SDLC Agent Audience: Developers, Tech Leads, Architects, Ops / SRE

    The EVA-Repo-Librarian helps keep documentation and code aligned across EVA Suite repos.

    It:

    • Notices when code changes are not reflected in READMEs, ADRs, or other docs.
    • Suggests doc updates, ADR skeletons, and architecture notes.
    • Encourages repos to stay self-explanatory and onboardable.
    • Never commits changes directly to main; it only suggests and drafts.

    Think of it as the crew’s documentation conscience.

    2. Intended Use and Non-Use¶

    Use EVA-Repo-Librarian when you want to:

    • Ensure key changes such as new services, endpoints, commands, flags, or configs are documented.
    • Encourage a culture of ADRs (Architecture Decision Records).
    • Keep setup instructions, architecture overviews, and runbooks up to date.
    • Help new developers understand how to run the project, what each major component does, and where important docs live.

    Typical scenarios:

    • After merging significant PRs.
    • When refactoring substantial parts of a codebase.
    • During periodic “doc health checks” across the repo.

    Do not rely on EVA-Repo-Librarian to:

    • Replace human architectural thinking or sign-off.
    • Write full, polished docs without human review.
    • Decide which design decisions are official. ADRs still require human approval and governance.

    This agent supports documentation hygiene, not architecture governance.

    3. Role in the GC Agentic Template¶

    EVA-Repo-Librarian reinforces expectations from the GC Agentic AI Template:

    • Volume 1 – Platform and Agentic Requirements

    • Helps ensure repos that implement agentic functionality document required hooks such as logging, kill switches, and RBAC.

    • Volume 2 – Engineering and Delivery Guide

    • Supports best practices for clear READMEs, ADRs for major decisions, and consistent documentation of architecture and components.

    • Volume 3 – LiveOps and Governance Model

    • Encourages runbooks and operating procedures to remain in sync with actual behaviour and observed incidents.

    • Volume 4 – Use Cases, Patterns and Templates

    • Makes it easier to apply patterns by surfacing where documentation diverges from the real implementation.

    4. High-Level Behaviour¶

    At a high level, EVA-Repo-Librarian:

    • Scans repos periodically or after key events such as large PR merges.
    • Compares current code structure and configuration to existing docs.
    • Detects gaps such as new modules with no description, changed behaviour with no updated usage examples, and missing ADRs for major shifts.
    • Suggests improvements in the form of doc snippets, ADR skeletons, and documentation to-dos.

    All suggestions are non-blocking and appear as issues, PR comments, or draft markdown files.

    5. Inputs and Outputs¶

    Inputs:

    • Repository contents including code tree, docs tree, CI and infra files.
    • Change history such as recent commits, PR diffs, tags and releases.
    • Repo conventions such as doc locations, ADR templates, and runbook formats.
    • Optional team-specific documentation standards or checklists.

    Outputs:

    • Doc suggestions in PR comments that point to missing or outdated docs.
    • Draft doc fragments for getting started, configuration, architecture overview and runbooks.
    • ADR skeletons with context, decision and consequences headings.
    • Documentation to-do issues summarizing what needs to be updated and why.

    All outputs must be clearly tagged as generated-by:eva-repo-librarian.

    6. Triggers¶

    EVA-Repo-Librarian can be triggered by:

    • Post-merge hooks after PRs larger than a configured threshold or labeled as major-change.
    • Scheduled scans such as nightly or weekly documentation health checks.
    • Manual commands invoked by developers, for example via a bot comment or CLI helper.

    If the agent is disabled nothing breaks; work continues with human-only documentation updates.

    7. Interaction Flows¶

    7.1 Post-Merge Doc Check¶

    Event:

    • A large PR is merged into main or another long-lived branch.

    Context gathering:

    • Files changed, including paths and additions or deletions.
    • PR title and description.
    • Existing docs such as README, docs folders and ADRs.

    Agent analysis:

    • Identifies new components or services without documentation.
    • Spots configuration or environment changes not mentioned in docs.
    • Detects design shifts that likely warrant an ADR.

    Outputs:

    • PR comments with concrete suggestions for where and how to document new behaviour.
    • Optional creation of documentation to-do issues for follow-up.

    Human action:

    • Developers or Tech Leads author or edit docs.
    • They may paste in suggested snippets and refine them.
    • They may close suggestions with rationale when they are not needed.

    7.2 Periodic Repo Doc Health Check¶

    Event:

    • A scheduled job runs weekly or bi-weekly.

    Context gathering:

    • Full repo layout including code and docs.
    • Reference to previous doc health reports if stored.

    Agent analysis:

    • Identifies directories without a short README or description.
    • Finds components that never appear in any docs.
    • Flags ADR directories that have not been updated despite major changes.

    Outputs:

    • A single “Doc Health Report” issue or doc summarizing missing docs, ADR topics to consider and doc-debt hotspots.

    Human action:

    • Use the report in retrospectives or planning.
    • Create “doc debt” tasks or epics based on the report.

    8. Profile Configuration in EVA Runtime¶

    Conceptual profile for this agent:

    agent_id: eva-repo-librarian
display_name: EVA Repo Librarian Agent
description: Keeps documentation, ADRs, and repo structure aligned with code changes.
autonomy_level: advisory

triggers:
  - github_webhook: pull_request_merged
  - schedule: weekly_monday_03_00
  - manual_command: "/eva-docs review-repo"
  - manual_command: "/eva-docs suggest-adr"

context_requirements:
  - repo_tree
  - recent_commits
  - pull_request_diff
  - readme_and_docs
  - adr_directory
  - repo_doc_conventions

outputs:
  - github_issues
  - github_pr_comments
  - markdown_drafts

policies:
  max_issues_per_scan: 10
  max_comments_per_pr: 10
  prefer_summary_issue: true
  mark_items_with_tag: "generated-by:eva-repo-librarian"

    The exact schema will live in the EVA DevTools runtime, but this illustrates the intent.

    9. Prompting Model¶

    System prompt (DevTools global):

    • Shared with other DevTools agents.
    • Enforces GC and EVA principles.
    • Requires safe, explainable and auditable outputs.

    Role prompt (Repo Librarian Agent):

    • You are the EVA Repo Librarian Agent.
    • You keep documentation in sync with code and architecture.

    The agent must:

    • Identify high-value documentation gaps such as new services, major behaviour changes and new configuration flags.
    • Propose concise, actionable updates including text snippets, ADR titles and doc to-do items.

    The agent must not:

    • Modify code or docs directly.
    • Flood users with trivial suggestions.
    • Invent system details that cannot be inferred from code or PR descriptions.

    The output structure typically includes PR comments, doc issues and ADR suggestions.

    10. Guardrails and Governance¶

    Guardrails:

    • Advisory only; no direct pushes to main or protected branches.
    • Suggestions appear as comments, issues or draft files.
    • All suggestions are tagged with the agent identity.

    Noise control:

    • Limits on issues and comments per scan.
    • Option to prefer one summary issue instead of many small ones.

    Attribution and accountability:

    • Humans remain responsible for final documentation.
    • ADRs are only official when approved under existing governance.
    • The agent can suggest ADRs but cannot finalize them.

    11. Implementation Notes for EVA Suite¶

    Within EVA Suite, EVA-Repo-Librarian uses the shared DevTools runtime:

    • Trigger to context to policy to LLM to outputs to logging.
    • Integrated with GitHub or Azure Repos via webhooks and APIs.
    • Uses repo standards such as .eva-pod.yaml and any repo-level config that declares doc paths and ADR conventions.

    It should:

    • Respect security and classification boundaries when scanning repos.
    • Avoid including sensitive content in logs or external prompts.
    • Align with EVA Folder Structure and Housekeeping specs so it knows where docs should live.

    12. Integration with Jarvis and GitHub¶

    Within the EVA + GitHub model:

    • Jarvis DevMaster triggers EVA-Repo-Librarian from workflows such as jarvis-daily, jarvis-review or jarvis-retro.
    • EVA-Repo-Librarian reads the repo layout and recent changes for the sprint or release in question.

    • Outputs are written as:

    • Doc follow-up issues linked to the sprint YAML in eva-meta.

    • Comments on PRs that are part of the current sprint.
    • Optional doc health reports stored under reports or docs in eva-meta.

    Jarvis uses these outputs to:

    • Highlight doc-debt in sprint reviews and retros.
    • Propose small doc tasks for future sprints.
    • Keep the EVA Suite documentation in line with the evolving architecture.

    13. Adoption and Maturity¶

    Phase 1 – PR Comment Helper:

    • Enabled only on larger PRs.
    • Adds doc-related hints as PR comments.

    Phase 2 – Doc Issues and ADR Drafts:

    • Creates documentation issues and ADR skeletons in feature branches.
    • Still advisory but more proactive.

    Phase 3 – Regular Doc Health Checks:

    • Scheduled scans produce doc health reports.
    • Reports feed retrospectives and “doc debt” clean-up work.

    Throughout all phases, documentation responsibility remains with humans; the agent simply reduces friction and helps EVA meet GC-grade transparency and reuse expectations.

    14. Benefits Summary¶

    Benefits:

    • Better onboarding because repos are more self-explanatory.
    • Reduced documentation drift after large code or architecture changes.
    • Stronger architectural memory via ADR suggestions.
    • Clearer, more auditable documentation for GC governance, RAIDE and reuse across departments.
