    EVA DevTools Agent Pattern – P13 Onboarding & Sandbox Coach Agent¶

    Subtitle: Helping Humans and Teams Get Productive Safely From Day 0 Version: Draft 0.2 Date: YYYY-MM-DD

    1. Pattern Overview¶

    Agent name: EVA-Onboarding-Agent Short label: ONB Category: DevTools / Enablement / Learning Audience: New developers, product owners, internal clients, partner teams, GC departments onboarding to EVA and AI sandboxes

    The EVA-Onboarding-Agent is the day-0 to day-30 guide for people and projects entering the EVA ecosystem. It:

    • Explains what EVA is and how their work fits into EVA Suite.
    • Guides sandbox and dev environment setup using approved patterns.
    • Suggests role-aware learning paths and “next best steps”.
    • Standardizes onboarding so AICOE does not have to re-explain basics for every person or project.

    It complements, not replaces, formal HR onboarding, mandatory training, and governance processes.

    2. Intended Use and Non-Use¶

    2.1 Intended Use¶

    Use EVA-Onboarding-Agent to:

    • Help individuals

    • Understand EVA components and where their team fits.

    • Set up a dev sandbox (local, Codespaces, EVA Dev Sandbox) following standard templates.

    • Discover the right documentation, policies, and training for their role and experience.

    • Help project teams

    • Run a guided project onboarding checklist from “idea” to “sandbox ready”.

    • Generate tailored Getting Started docs and checklists checked into their repo.

    • Help partner GC departments

    • Produce Canada-grade sandbox guidance aligned with GC AI and cloud policies.

    2.2 Non-Use¶

    Do not use EVA-Onboarding-Agent to:

    • Grant or revoke access, change IAM or create cloud subscriptions.
    • Replace formal security, privacy, or HR onboarding.
    • Perform destructive environment actions such as deleting sandboxes or rotating secrets.

    The agent coaches and structures the work; humans and existing processes remain in charge of approvals and access.

    3. Role in the EVA and GC Agentic Model¶

    The agent supports the GC Agentic Template and EVA lifecycle as follows:

    • Volume 1 – Requirements and platform

    • Turns high-level EVA and client requirements into simple onboarding journeys for people and projects.

    • Volume 2 – Engineering and delivery

    • Promotes EVA Dev Sandbox patterns, standard repo layouts, and GC-compliant engineering practices from day one.

    • Volume 3 – LiveOps and governance

    • Surfaces data classification, Terms of Use, AI usage rules, and accountability expectations early during onboarding.

    • Volume 4 – Patterns and templates

    • Encapsulates reusable “onboarding recipes” that can be exported to other GC departments.

    In the EVA lifecycle map, ONB is most active in Phase 0 (Idea / Intake / Sandbox), Phase 1 (Design), and Phase 6 (Day-2 continuous learning for new teams).

    4. High-Level Behaviour¶

    The agent behaves as a structured coach:

    • Asks a small number of questions about role, project, experience level, and environment.
    • Builds a tailored onboarding path with steps, links, commands, and expected outcomes.
    • Tracks progress in a simple, human-readable way (checklists, “what’s next” summaries).
    • Routes users to humans, official docs, or training when something needs human judgment or approval.

    It does not maintain HR records or training portfolios; it only produces artefacts and guidance that can be stored in repos or onboarding spaces.

    5. Inputs and Outputs¶

    5.1 Inputs¶

    The agent can safely use:

    • User-provided profile information such as role, team, project, technical background.
    • Project metadata such as project name, sponsor, target repos, expected environments.
    • Sandbox pattern catalogue (Codespaces templates, EVA Dev Sandbox templates).
    • Governance and training indexes (links to AI guidelines, security, RAIDE, MTI, etc.).

    Only minimal personal data is requested and used; names and identifiers are optional and not required in file names.

    5.2 Outputs¶

    Typical outputs include:

    • Personalized onboarding journeys

    • Day-0, day-1–3, and first-sprint steps.

    • Quickstart instructions to clone repos and run apps or sandboxes.

    • Project onboarding checklists

    • Steps to register the project in EVA Directory or equivalent.

    • Repo creation and template selection.

    • CI/CD, logging, and data-classification tasks.

    • Short explanations and nudges

    • Why APIM headers are used.

    • How AI governance guidance applies to their project.

    • Markdown artefacts committed to repos or stored centrally, such as:

    • onboarding/USER-journey.md

    • onboarding/PROJECT-setup-checklist.md

    All artefacts are tagged with a marker such as generated-by:eva-onboarding-agent for transparency.

    6. Triggers¶

    The agent is triggered explicitly; it does not run autonomously in the background.

    Examples:

    • New member joins a pod

    • Command such as /eva-onboard me or /eva-onboard new-member.

    • New project or sandbox

    • Command such as /eva-onboard new-project from an intake ticket or repo.

    • Partner sandbox

    • Command such as /eva-onboard partner-sandbox initiated by an AICOE coordinator.

    Optional opt-in follow-ups can be offered such as a weekly “do you want an updated what’s next?” prompt.

    7. Core Interaction Flows¶

    7.1 Individual Onboarding¶

    • User invokes the agent from chat, a portal, or a repo.
    • Agent asks a few focused questions about role, experience, and project.

    • Agent composes a role-aware journey, for example:

    • Read a short EVA overview.

    • Accept Terms of Use and AI guidance.
    • Set up sandbox access and tools.
    • Run the standard dev environment and complete a first simple ticket.
    • Agent writes a markdown checklist file and optionally posts a summary to a team channel or issue.
    • User can ask the agent to show remaining steps or mark items as done.

    7.2 Project or Sandbox Onboarding¶

    • Product owner or tech lead invokes the agent with a short description of the initiative.
    • Agent collects domain, data classification, expected users, and goals.
    • Agent generates a project setup checklist aligned with EVA Directory, standard repos, CI/CD, logging, and AI governance.
    • Output is stored as onboarding/PROJECT_NAME-setup.md or added to the intake record.
    • Human leads own execution; the agent is used to view progress or regenerate updated checklists.

    7.3 Partner GC Sandbox¶

    • AICOE coordinator invokes the agent for a partner department.
    • Agent produces a Canada-grade sandbox guide including high-level architecture, expected governance elements, and recommended tech stack patterns.
    • Output is reviewed by a human before being shared externally.

    8. Runtime Profile (Conceptual)¶

    Example conceptual configuration for EVA DevTools runtime:

    agent_id: eva-onboarding-agent display_name: EVA Onboarding and Sandbox Coach Agent description: Guides people and projects through EVA onboarding and sandbox setup in a safe, structured way. autonomy_level: advisory

    triggers:

    • manual commands such as /eva-onboard me, /eva-onboard new-member, /eva-onboard new-project, /eva-onboard partner-sandbox

    context_requirements:

    • user profile context
    • optional project metadata
    • sandbox pattern catalogue
    • governance documentation index
    • training resources index

    outputs:

    • markdown guides
    • checklists
    • short explanations
    • progress snapshots

    llm_profile:

    • system_prompt_ref: devtools_global
    • role_prompt_ref: onboarding_agent
    • output_schema_ref: onboarding_and_sandbox_v1

    policies:

    • require human review flag true for project-level artefacts shared beyond the team
    • minimize personal data in context and outputs
    • tag all artefacts with generated-by:eva-onboarding-agent

    9. Prompting Model¶

    System prompt (shared DevTools global) ensures the agent:

    • Acts as an enablement tool, not an access management system.
    • Encourages minimal use of personal data and high transparency.
    • Aligns with GC values of openness, inclusion, and accountable AI use.

    Role prompt (onboarding_agent) emphasises that the agent:

    • Tailors journeys to role and context using just enough questions.
    • Provides clear, ordered steps with links and expected outcomes.
    • Clearly distinguishes between steps a user can perform and steps needing approvals or help from specific roles such as cloud ops or security.
    • Never claims that access has been granted or that governance steps are complete; instead it points to the right processes and humans.

    Outputs can be structured internally as JSON for logging and UI, but are exposed to users mainly as markdown checklists and guides.

    10. Guardrails and Governance¶

    Key guardrails:

    • No access management or IAM changes.
    • Minimal personal data, with preference for generic identifiers in long-lived artefacts.
    • Clear marking of AI-generated content and assumptions.
    • Human review required for artefacts that affect cross-team onboarding, external partners, or GC-wide guidance.

    This pattern must be wired into EVA’s overall governance so that onboarding journeys always reference the latest official policies and Terms of Use rather than embedding outdated copies.

    11. Implementation Notes¶

    • The agent runs on the EVA DevTools runtime and uses the same logging, metering, and kill-switch mechanisms as other DevTools agents.

    • It integrates with:

    • EVA Directory or registry for project metadata.

    • Documentation portals such as SharePoint, wikis, or GitHub Pages.
    • Internal training catalogues.

    • Onboarding artefacts can be stored:

    • In the project repo under onboarding/.

    • In a central EVA Onboarding space for cross-project learning.

    Initial rollout can start as a simple “cheat-sheet generator” and gradually evolve into a full project onboarding companion and GC sandbox coach as patterns stabilize.
