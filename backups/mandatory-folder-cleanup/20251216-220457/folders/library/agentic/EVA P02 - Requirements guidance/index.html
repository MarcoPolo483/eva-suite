    EVA P02 Requirements guidance

    EVA JARVIS – REQUIREMENTS AGENT AND SENIOR ADVISOR GUIDE Version: 0.1 Audience: Jarvis (EVA DevMaster), Copilot Pro+, EVA DevTools, AICOE devs Goal: Tell Jarvis how to use a Requirements Agent and Senior Advisor, which backends to call, and how to keep everything safe and traceable.

    1. Jarvis’ Job In Requirements Refinement¶

    Jarvis does not “invent” requirements. Jarvis orchestrates a pipeline:

    • Read high-level inputs (CDD, canvases, idea notes) from GitHub.
    • Decide which Requirements backend to call.
    • Receive structured proposals (epics, stories, NFRs).
    • Turn those into draft GitHub issues and docs.
    • Tag and log everything so humans stay in control.

    Backends can be:

    • Internal EVA DevTools Requirements Agent (inside GC tenant).
    • External Senior Advisor (ChatGPT / OpenAI) in lab-only mode.
    • Human-only refinement using Copilot and local docs (no LLM calls by Jarvis).

    Jarvis must never bypass the safety rules in this document.

    2. Requirements Backends And When To Use Them¶

    Jarvis knows three “modes”.

    Mode 1 – Internal EVA DevTools Requirements Agent

    • Backend is an EVA DevTools API in the GC environment.
    • Backed by Azure OpenAI or other in-tenant models.
    • Used for real EVA work, up to the allowed security classification.
    • This is the default for departmental / AICOE projects.

    Mode 2 – External Senior Advisor (Lab Only)

    • Backend is an external LLM (ChatGPT / OpenAI).
    • Only allowed for content explicitly marked Unclassified.
    • Only used in personal / lab environments or explicit sandbox setups.
    • Never used for Protected B or above.

    Mode 3 – Human and Copilot Only

    • Jarvis does not call any LLM backend.
    • Jarvis only manages structure (folders, labels, sprint YAML) and leaves all requirement text to humans and Copilot in the editor.
    • This is the fallback mode if no backend is configured or allowed.

    Jarvis must treat the backend as a pluggable component, selected by configuration and classification, not hard-coded.

    3. Advisor Configuration (Conceptual)¶

    Jarvis reads an advisors configuration file (for example advisors.yaml in eva-meta/config).

    The file defines entries such as:

    • id (for example eva-req-internal, senior-advisor-external, human-only)
    • type (internal, external, none)
    • endpoint or adapter name (for internal / external)
    • allowed_classification values (for example unclassified, protected_b)

    Jarvis uses:

    • advisor_id from script arguments or workflow inputs.
    • classification from the CDD or canvas (for example unclassified, protected_b).

    If classification is not in allowed_classification for that advisor, Jarvis must refuse to call that backend and fall back to human-only mode.

    4. Inputs Jarvis Uses For Refinement¶

    Jarvis-refine operates mainly inside eva-meta and reads:

    • CDD files in docs/cdd (for example cdd-eva-suite, cdd-jurisprudence, cdd-assistme).
    • Concept canvases or intake notes (for example docs/intake or docs/canvases).
    • Existing issues labeled needs:refinement.
    • Pods and repos mapping from config/pods-map.
    • DevMaster rules from config/devmaster-rules if present.

    Jarvis expects CDDs or canvases to carry at least:

    • A clear problem or opportunity statement.
    • A classification tag (for example classification: unclassified or classification: protected_b).
    • Any known constraints that affect refinement.

    If classification is missing, Jarvis should assume the safest option and not call an external backend.

    5. Behaviour Of jarvis-refine¶

    Jarvis-refine is the main refinement pipeline Jarvis runs.

    High-level behaviour:

    1. Discover “parents” that need refinement

    2. Issues labeled needs:refinement and level:epic or level:feature.

    3. Or direct instructions to refine a specific CDD or canvas.

    4. Gather context

    5. Read the associated CDD and any linked design files.

    6. Read pods-map to understand which pods and repos are relevant.

    7. Decide backend

    8. Read advisor_id from input or default.

    9. Check project classification against advisors.yaml.

    10. If not allowed, switch to human-only mode and log why.

    11. Call backend (if any)

    12. For internal EVA DevTools: call the Requirements Agent API with CDD text, parent issue description, and any config flags.

    13. For external Senior Advisor: call OpenAI API with a clear system prompt that describes the EVA Requirements Agent pattern and expected JSON structure.

    14. For human-only mode: do not call any LLM; only create placeholders.

    15. Receive and interpret response

    16. Expected structure: epics, stories, tasks, NFRs or a similar list.

    17. Each item should contain title, description, suggested repo/pod, estimate, and acceptance criteria when possible.

    18. Create or update GitHub issues

    19. Create child issues linked to the parent (blocked/by or parent/child link).

    20. Apply labels for level (epic, feature, story, task).
    21. Apply type labels (feature, bug, infra, doc) when suggested.
    22. Apply pod labels if known.

    23. Mark all created items as draft and ready-for-review, not ready-for-sprint.

    24. Update parent issue state

    25. If children cover the scope, remove needs:refinement from the parent.

    26. If more work is needed, keep or re-apply needs:refinement on specific children.

    27. Log what happened

    28. Record in orchestration/sprint-XXX-devmaster-log or a similar log which parents were refined, which backend was used, and how many children were created.

    29. Tag issues with generated-by:eve-req-agent or generated-by:senior-advisor when applicable.

    Jarvis never silently overwrites human content. It only adds new issues or clearly tagged updates.

    6. Safety And Classification Rules¶

    Jarvis must follow these rules regardless of backend:

    • Never send Protected B or higher content to an external backend.
    • Never send explicit personal information, credentials, secrets, or full production logs to any backend.
    • For internal EVA DevTools, respect the maximum classification configured for that environment.
    • Always log the advisor_id and classification in the orchestration log when a refinement run is performed.
    • Always mark agent-generated content as draft and require human review before use in sprint planning.

    If Jarvis is uncertain about classification, it must:

    • Assume it is not safe for external backends.
    • Use internal EVA DevTools only if explicitly configured for that project.
    • Otherwise fall back to human-only mode.

    7. Integration With Sprint Planning¶

    Jarvis-plan must only consume items that have been:

    • Fully refined to level:story or level:task.
    • Marked ready-for-sprint by humans (for example label ready-for-sprint).

    Jarvis-refine’s job is to help produce candidate stories and tasks. Jarvis-plan’s job is to select from that pool for a specific sprint.

    Jarvis-refine and Jarvis-plan must stay decoupled:

    • Refinement can be run multiple times on the same epic or feature.
    • Planning picks only those items that humans have accepted as sufficiently clear and small.

    8. How Copilot Should Use This Guide¶

    When this guide is present in eva-meta docs, Copilot can be instructed to:

    • Implement advisors configuration “Create config/advisors.yaml and a Python script scripts/jarvis_refine.py that follow the EVA Jarvis – Requirements Agent and Senior Advisor Guide. The script should support three advisor modes (internal, external, human-only), enforce classification rules, and create draft GitHub issues for epics, stories, and NFRs tagged with the generating advisor.”

    • Wire jarvis-refine workflow “Create .github/workflows/jarvis-refine.yml in eva-meta that runs jarvis_refine.py with advisor_id and classification parameters, using GITHUB_TOKEN and environment variables for internal or external backends.”

    • Enforce draft and logging rules “Update jarvis_refine.py so that any agent-generated issues are labeled draft and generated-by:, and write a summary line to orchestration logs for each refinement run.”

    Jarvis and Copilot together must treat this document as the contract for how Requirements Agent and Senior Advisor are integrated into EVA’s development lifecycle.
