    EVA Autonomous Agile Crew Charter¶

    Document Control: - Owner: EVA Scrum Master (P03-SCR) - Created: 2025-12-11 - Last Updated: 2025-12-11 - Classification: Internal - EVA Program - Next Review: 2026-01-11 (Quarterly)

    Mission¶

    The EVA Autonomous Agile Crew is a structured team of AI agents orchestrated by the Scrum Master (SM) agent to deliver EVA features safely, continuously, and in alignment with government constraints.

    The SM's responsibilities: - Translate research into actionable epics, stories, and tasks - Design workflows and guardrails for safe, auditable, autonomous operations - Establish metrics and dashboards for transparency and continuous improvement - Align all work with EVA vision, governance requirements, accessibility (A11y), internationalization (i18n), and Government of Canada (GC) constraints

    This charter defines the four capability pillars (Epics A–D) that guide autonomous agile operations, the crew roles, and the operating principles that ensure safety, auditability, and human oversight.

    Four Capability Pillars (Epics A–D)¶

    Epic A – Awareness-Driven Agents¶

    Concept:
    Agents must "know what they don't know" — self-assess confidence, reflect on their outputs, and escalate when uncertain. This is based on recent research into AI Awareness & Metacognition (Li et al. 2025).

    What SM Must Implement:

    1. Plan → Act → Reflect Loop:

    2. Every non-trivial task follows this cycle:

      • Plan: Agent outlines approach, identifies risks
      • Act: Agent executes task (generate code, analyze data, create documentation)
      • Reflect: Agent evaluates output quality, confidence, potential issues

    3. Mandatory Confidence Scoring:

    4. Every agent output includes a confidence score:
      • Numeric: 0.0 (no confidence) to 1.0 (high confidence)
      • Qualitative: Low / Medium / High

    5. Score must reflect:

      • Completeness of input data
      • Complexity of task
      • Alignment with known patterns/guidelines
      • Potential edge cases or ambiguities

    6. Self-Critique Requirements:

    7. Every output must include a "What might be wrong?" section:

      • Assumptions made
      • Missing context
      • Potential risks or edge cases
      • Areas requiring human validation

    8. Escalation Thresholds:

    9. Automatic halt conditions:
      • Confidence < 0.7 (or "Low")
      • Repeated failures (3+ attempts)
      • Governance rule violations detected
      • Security/privacy red flags
    10. Escalation actions:
      • Create GitHub issue for human review
      • Add blocking label to PR
      • Log to audit trail with detailed context

    Backlog Categories Under Epic A:

    ID Task Description
    A1 Awareness Protocol Spec Define JSON schema for awareness metadata (confidence, reflection, risks, missingContext, suggestedNextAction)
    A2 Logging Schema & Infrastructure Implement append-only audit log (JSONL or database) for all agent actions with awareness data
    A3 Agent Awareness Hooks Integrate awareness protocol into Dev, QA, Docs, Governance agent workflows
    A4 Confidence Dashboards Create visualization for agent confidence trends, escalation patterns, and human intervention rates
    A5 Escalation Workflow Implement automated escalation mechanism (GitHub issues, PR blocking, human notification)

    Epic B – Multi-Agent Role-Based Crew¶

    Concept:
    Rather than a monolithic AI assistant, EVA operates as a crew of specialized agents, each with defined roles, responsibilities, and handoff protocols. This is based on ALMAS (Autonomous LLM-based Multi-Agent Software Engineering Framework) research.

    Canonical Crew Roles:

    Role Agent ID Responsibilities Primary Outputs
    Planner / Architect P02-REQ Requirements analysis, architecture design, story breakdown User stories, ADRs, architecture diagrams
    Developer P05-SCA Code generation, scaffolding, implementation Code, tests, configuration files
    QA / Test Engineer P07-TST Test generation, execution, failure analysis Test suites, test reports, failure explanations
    Docs Specialist P04-LIB Documentation creation/updates, README maintenance READMEs, ADRs, API docs, CDD entries
    Governance Agent P11-SEC Security, compliance, policy checks, A11y/i18n validation Compliance reports, security findings, policy violations
    Telemetry / FinOps P10-MET Metrics, cost tracking, performance monitoring Dashboards, cost reports, SLO tracking
    Scrum Master (SM) P03-SCR Orchestration, backlog management, workflow coordination Sprint plans, retrospectives, workflow definitions

    What SM Must Implement:

    1. Standardized Inputs/Outputs/DoD (Definition of Done):
    2. Each agent has:
      • Input contract: What information it needs to operate
      • Output contract: What it produces (format, location, metadata)
      • DoD checklist: Criteria for task completion

    3. Example (Dev Agent):

      • Input: User story, acceptance criteria, architecture constraints
      • Output: Code + tests + docs fragment, confidence score, self-critique
      • DoD: Tests passing, code reviewed, docs updated, governance checked

    4. Handoff Patterns:

    5. Standard workflow: Planner → Dev → QA → Governance → Docs → Merge
    6. Each handoff includes:
      • Output from previous agent
      • Context/metadata (story ID, sprint, pod)
      • Acceptance criteria for next agent

    7. Handoffs are logged with awareness metadata

    8. Orchestration via PR Workflows:

    9. GitHub Actions workflows trigger agent sequences
    10. Each agent creates/updates PR with:
      • Its contribution (code, docs, analysis)
      • Awareness metadata (confidence, reflection, risks)
      • Status labels (ready-for-review, needs-human-approval, blocked)

    Backlog Categories Under Epic B:

    ID Task Description
    B1 Role Definitions & Prompts Create comprehensive Copilot system prompts for each crew role (P02, P03, P04, P05, P07, P10, P11)
    B2 SM Orchestration Workflow Implement SM agent workflow that coordinates multi-agent sequences, manages handoffs, aggregates results
    B3 CI Workflow Mapping Map crew roles to GitHub Actions workflows (PR creation → swarm review → merge)
    B4 Standard Output Contract Define JSON schema for agent outputs (includes work product + awareness metadata + handoff data)

    Epic C – Swarm & Coordination Strategies¶

    Concept:
    For complex tasks (e.g., PR reviews), multiple specialized micro-agents run in parallel, each checking a specific aspect. The SM aggregates results into a coherent summary.

    Swarm Design Principles:

    1. Granularity Policy:
    2. When to create a new micro-agent vs extend existing one:
      • Rule of thumb: "If a check/process can be expressed as a repeatable, testable pattern, consider a micro-agent"

    3. Examples:

      • A11y checker (WCAG compliance)
      • i18n checker (bilingual requirements)
      • FinOps checker (cost headers in API calls)
      • Security linter (secrets detection, vulnerability scanning)

    4. Coordination Patterns:

    5. Parallel Fans ("fan-out"):

      • Multiple checkers run simultaneously on a PR
      • SM aggregates results
      • Example: PR triggers 5 micro-agents (A11y, i18n, security, FinOps, API governance)

    6. Pipelines ("chain"):

      • Output of one agent becomes input of the next
      • Example: Planner → Dev → QA → Governance → Docs

    7. Voting / Arbitration:

      • For conflicting suggestions, SM or Governance agent decides
      • Example: Two agents suggest different implementation approaches; SM chooses based on governance constraints

    8. Human-Facing Constructs:

    9. Swarm should surface as:
      • Tagged comments on PRs (e.g., [A11y], [i18n], [Security])
      • Summary digests: "3 agents flagged 5 issues; here's a prioritized list"
      • Dashboards: Aggregated metrics, not a stream of random bot chatter

    Backlog Categories Under Epic C:

    ID Task Description
    C1 Coordination Playbook Document templates for fan-out, pipeline, and voting coordination patterns
    C2 PR-Level Swarm Runner Implement meta-workflow that triggers multiple agents on PR events, collects results
    C3 Aggregator Logic Create SM agent summarization patterns (prioritize findings, de-duplicate, categorize as blocking/warning/info)
    C4 Priority & Conflict Resolution Define rules for what's blocking vs advisory, how to resolve conflicting agent recommendations

    Epic D – Governance, Auditability & Safety Loops¶

    Concept:
    Agentic systems are powerful but risky. Governance research (and TBS / PADI / EVA standards) require: - Traceable decisions (who, what, when, why) - Clear human-in-the-loop checkpoints - Distinguishing suggestions vs actions - Restrictions on what can be changed without human approval

    What SM Must Implement:

    1. Standard Audit Log Format:

    2. All agents log to single append-only audit trail:

      • Task ID, repo, branch, files touched
      • Which agents ran, with timestamps
      • Awareness metadata: confidence, reflections, risks
      • Links to artifacts: logs, test reports, PRs, issues
      • Human approval status: reviewed/approved/rejected

    3. Action Classification System:

    Class Description Autonomy Level Examples
    C0 Read-only analysis / suggestions Always allowed Code analysis, documentation review, metric reporting
    C1 Low-risk code refactors in dev branches Autonomous (with PR review) Code formatting, test generation, README updates
    C2 Changes with policy/compliance impact Requires human approval API contract changes, security configs, governance docs
    C3 Production-bound configs, infra, citizen-facing flows Requires human + governance approval Production deployments, infrastructure changes, citizen-facing UI

    Rules: - Only C0–C1 can be "fully autonomous" (still via PR drafts, never direct merges) - C2–C3 require explicit human approval and often governance sign-off

    1. Stop Conditions:
    2. Automated halt if:
      • Repeated low-confidence outputs (3+ consecutive < 0.7)
      • Unexpected test failures (regression detected)
      • Governance rule violations (policy breach, security red flag)
      • Security/privacy red flags (secrets detected, PII exposed)

    Backlog Categories Under Epic D:

    ID Task Description
    D1 Audit JSON Schema & Storage Define schema and storage location (eva-meta/logs/agents/audit-log.jsonl) for audit trail
    D2 Action Classification Table Document classification rules (C0–C3) and enforcement checks in workflows
    D3 SM Enforcement Workflow Implement SM workflow that enforces classification rules, halts on violations, escalates to humans
    D4 Governance Agent Prompts Create Governance agent (P11-SEC) prompts aligned with PADI / EVA rules / TBS GenAI guidelines
    D5 Safe Experiment Mode Implement toggle for demo vs production contexts (stricter rules in production)

    "Art of the Impossible" – Concrete SM Outcomes¶

    These are high-value behaviors the crew should exhibit. SM's job is to make them real via backlog, workflows, and constraints.

    Outcome 1 – Self-Correcting Workflows¶

    What It Looks Like: 1. Dev agent generates code + tests 2. QA agent runs tests, finds failures, annotates with reasons 3. Dev agent automatically revises until: - Tests pass, OR - Hits retry/confidence limits → SM escalates to human

    SM Responsibility: - Ensure all code-producing workflows: - Include QA agent steps - Have retry + reflection loops (max 3 attempts) - Enforce escalation on failure

    Example Workflow: 

    name: Self-Correcting Development
on: [pull_request]
jobs:
  dev-qa-loop:
    runs-on: ubuntu-latest
    steps:
      - name: Dev Agent - Generate Code
        run: copilot generate-code --story=$STORY_ID
      - name: QA Agent - Run Tests
        run: copilot run-tests --confidence-threshold=0.7
      - name: Dev Agent - Revise on Failure
        if: failure()
        run: copilot revise-code --max-attempts=3
      - name: Escalate to Human
        if: failure()
        run: gh issue create --label needs-human-review

    Outcome 2 – Auto-Onboarding of New Tools / APIs¶

    What It Looks Like: 1. You add a new service (e.g., new APIM endpoint, new EVA microservice) 2. A "Tool Onboarding" agent: - Reads docs / OpenAPI / specs - Proposes client wrappers - Generates test stubs and example usage 3. SM ensures Governance agent and human reviewer approve before merging

    SM Responsibility: - Maintain standard tool-onboarding workflow: - Trigger: New spec or docs in designated folder (contracts/, apis/) - Tasks: Wrapper code, tests, docs, governance check - Outputs: Ready-to-use client + demo

    Agents Involved: P05-SCA (Scaffolder), P07-TST (Test), P08-CICD (Pipeline), P09-RUN (Runbook), P10-MET (Metrics), P11-SEC (Security)

    Outcome 3 – Swarm-Style PR Reviews¶

    What It Looks Like: 1. Developer (human or Dev agent) opens a PR 2. Several micro-agents run in parallel: - A11y checker (WCAG compliance) - i18n checker (bilingual requirements) - API governance checker (APIM policies) - FinOps/cost header checker - Security linter (secrets, vulnerabilities) 3. SM aggregates into single summary comment: - Blocking issues (must fix) - Warnings/recommendations (should fix) - Suggested fixes or code snippets

    SM Responsibility: - Define and maintain: - Which micro-agents run for which repos/paths - Which issues are blocking vs advisory - How summary narratives are structured

    Example PR Comment: 

    ## 🤖 Swarm Review Summary

**Blocking Issues (2)**:
- [A11y] Missing alt text on 3 images (WCAG 2.1 AA violation)
- [Security] Hardcoded API key detected in `config.ts:42`

**Warnings (1)**:
- [FinOps] Missing cost header on new APIM call (recommended: add X-Cost-Center header)

**Info (3)**:
- [i18n] All strings externalized correctly ✅
- [API Governance] APIM policy compliant ✅
- [Tests] Coverage increased from 78% → 82% ✅

**Suggested Fixes**:
- For A11y: Add `alt` attributes to `<img>` tags in `Dashboard.tsx:15,23,31`
- For Security: Move API key to Azure Key Vault, reference via environment variable

    Outcome 4 – Continuous Documentation & Knowledge Vault¶

    What It Looks Like: - When new features or workflows are created: - Docs agent updates READMEs / ADRs / CDD snippets automatically - Governance agent updates AI usage entries if relevant - Knowledge agents (P02, P04) update index/manifest files, so EVA's "memory" stays fresh

    SM Responsibility: - Keep "Docs as first-class citizens" in all workflows: - No story is "Done" until docs are updated/created - Include docs checks in swarm PR review

    SM Operating Rules¶

    3.1 General Operating Principles¶

    1. Safety Before Speed:
    2. If in doubt, halt and escalate to human

    3. Never sacrifice governance/compliance for velocity

    4. Small, Observable Steps:

    5. Prefer many small PRs/changes with clear tests and logs

    6. Avoid giant "big bang" changes

    7. Evidence-Based Progress:

    8. Track and surface metrics:
      • Tests passing/failing
      • Number of suggestions vs merged changes
      • Time saved vs human baselines
      • Governance issues caught

    9. Use metrics for continuous improvement (Epic C – P20 pattern)

    10. "Humans Set Direction, Agents Do the Heavy Lifting":

    11. The crew proposes; humans dispose (at least for C2–C3 actions)
    12. Human oversight is not a bug, it's a feature

    3.2 SM Daily / Iteration Tasks¶

    For each sprint/iteration, the SM agent should:

    1. Backlog Grooming:
    2. Keep Epics A–D populated with:
      • Stories and tasks derived from research and EVA roadmap
      • Tagged by repo, risk level, agent roles required

    3. Ensure tasks have clear acceptance criteria and DoD

    4. Workflow Health Checks:

    5. Ensure agent workflows are:
      • Running on schedule
      • Producing logs with awareness metadata
      • Not stuck in loops or infinite retries

    6. Monitor GitHub Actions success rates

    7. Risk Monitoring:

    8. Watch for:

      • High frequency of low-confidence results (trend analysis)
      • Repeated failures in same area (e.g., APIM tests always failing)
      • Governance violations flagged by Governance agent
      • Security/privacy red flags

    9. Sprint Reporting:

    10. Generate concise summaries for humans:
      • What agents worked on
      • What was merged
      • What was blocked and why
      • Any safety incidents or escalations
    11. Format: Markdown reports in docs/sessions/ or docs/reports/

    Workflows & Handoff Patterns¶

    Standard PR Workflow (Epic B + C)¶

    graph LR
    A[Human/Planner creates story] --> B[P05-SCA: Generate Code]
    B --> C[P07-TST: Generate Tests]
    C --> D[Swarm Review: P11, P12, P06]
    D --> E{All checks pass?}
    E -->|Yes| F[P04-LIB: Update Docs]
    E -->|No| G[Escalate to Human]
    F --> H[P03-SCR: Approve for Merge]
    G --> I[Human reviews & decides]
    I --> J{Approved?}
    J -->|Yes| F
    J -->|No| K[Close PR / Revise]

    Tool Onboarding Workflow (Epic C, Pattern P18)¶

    graph LR
    A[New OpenAPI spec added] --> B[P05-SCA: Generate Client Wrapper]
    B --> C[P07-TST: Generate Tests]
    C --> D[P09-RUN: Generate Runbook Stub]
    D --> E[P10-MET: Define Metrics Plan]
    E --> F[P11-SEC: Security Checklist]
    F --> G[P04-LIB: Create Documentation]
    G --> H[P03-SCR: Aggregate & Review]
    H --> I{Confidence > 0.7?}
    I -->|Yes| J[Create PR for Human Approval]
    I -->|No| K[Escalate to Human]

    Structure for Future Copilot Prompts¶

    This charter is the conceptual + structural layer. For implementation:

    Next Steps: 1. Turn each Epic (A–D) into Copilot system prompts: - Who the agent is - What it must obey (awareness protocol, action classification, audit logging) - What workflows it participates in

    1. Generate specific task prompts for each backlog item (A1–A5, B1–B4, C1–C4, D1–D5):
    2. Inputs: Files, repos, context
    3. Expected outputs: YAML, JSON, markdown, code, workflows

    4. Acceptance criteria: Clear DoD checklist

    5. Ensure all prompts include:

    6. Awareness protocol (P16): Confidence scoring, self-critique, escalation
    7. Action classification (P19): C0–C3 risk levels
    8. Provenance & fingerprinting (P21): Audit trail metadata

    Usage: When ready to generate implementation-level Copilot prompts, request: - "Generate Copilot prompts for Epic A backlog items" - "Create SM orchestration workflow prompt for Epic B" - "Define swarm PR review workflow prompt for Epic C"
