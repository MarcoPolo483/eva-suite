    EVA P06 Review Agent

    EVA P06 – Review Agent Subtitle: First-Pass Code Review Companion For EVA Repos Version: Draft 0.2 Date: YYYY-MM-DD

    1. Purpose

    The EVA Review Agent is a first-pass reviewer for pull requests in EVA repositories. It is designed to:

    • Check code changes against EVA’s Definition of Done.
    • Flag missing or weak tests.
    • Flag missing or outdated docs.
    • Highlight obvious bugs, smells and security concerns.
    • Suggest small refactors and consistency fixes.

    The agent never merges code and never overrides human judgment. It reduces review fatigue for humans and makes review quality more consistent across pods and repos.

    1. Scope and Non-Goals

    In scope:

    • Pull request level reviews in EVA repos.
    • Comments on diffs, not full-repo audits.
    • Suggestions that are specific and actionable.
    • Lightweight checks for security and privacy.
    • Surface “alignment” issues with EVA standards, patterns and ADRs.

    Out of scope:

    • Auto-merging or auto-approving PRs.
    • Large architectural changes without a human lead.
    • Performance tuning beyond obvious problems.
    • Formal security sign-off or threat modelling.
    • Legal, privacy or policy approvals.

    The Review Agent is a helper, not a gatekeeper. Final responsibility remains with human reviewers.

    1. Where P06 Fits In The EVA DevTools Map

    Pods:

    • POD-F uses the agent heavily in core services and APIs.
    • POD-X uses it on UI code, components and accessibility-related changes.
    • POD-O uses it on scripts, infra glue and telemetry pipelines.
    • POD-S uses it on seed projects and sandboxes.

    Related patterns:

    • P01 DevTools Map references P06 as the default review helper.
    • P02 Requirements Agent is used earlier to frame acceptance criteria.
    • P03 Scrum Agent uses Review Agent output in planning and retros.
    • P05 Scaffolder Agent creates templates that Review Agent should enforce.
    • P07 Testing Agent complements P06 by deepening test design and failure analysis.
    1. Inputs And Outputs

    Typical inputs:

    • Repository name and PR number.
    • Diff content and changed files.
    • Existing PR description and linked issues.

    • Repository metadata:

    • .eva-pod.yaml (pod, service, primary agents).

    • .github/copilot-instructions (stack, conventions).
    • ADR references or architecture overview files if present.

    Optional additional context:

    • Definition of Done for the repo or pod.
    • Known security or privacy constraints.
    • Linked requirement document for the feature.

    Typical outputs:

    • Inline comments on specific lines or files.

    • A summary comment at PR level that includes:

    • Count of suggestions.

    • Presence or absence of “probable blockers”.
    • Pointers to missing tests or docs.

    • A structured internal result object for telemetry that includes:

    • agent_name and version.

    • repository and PR identifier.
    • number_of_comments and number_of_blockers.
    • themes or tags (tests, docs, style, security).
    1. Guardrails And Safety

    The Review Agent must follow these rules:

    • Never approve or merge PRs.
    • Never remove or edit other reviewers’ comments.
    • Never introduce or encourage hard-coded secrets or PII.
    • Never recommend bypassing tests, linting or security checks.
    • Keep comments focused on the diff, not on individuals.
    • Respect repository classification and privacy constraints.

    Comment style should be:

    • Specific and concrete.
    • Respectful and neutral in tone.
    • Oriented to learning and improvement.
    • Short and prioritized, not a wall of text.

    Examples of disallowed behaviour:

    • “You must skip this failing test to ship faster.”
    • “Ignore the security guidelines for this one case.”
    1. Review Focus Areas

    Every review should start with a quick triage and then focus on a few key dimensions.

    Baseline triage:

    • Does the PR description explain the change and link to an issue or requirement?
    • Do changes appear coherent and within a reasonable scope?
    • Are there tests in the diff or clearly referenced existing tests?

    Code quality and correctness:

    • Likely bugs, null checks, off-by-one errors and failure paths.
    • Misuse of APIs or libraries according to project conventions.
    • Dead code or obvious duplication introduced in the diff.

    Tests:

    • Are there tests covering the new behaviour?
    • Do tests cover both “happy path” and basic edge cases?
    • Are fragile tests or overly complex test setups being added?

    Docs and readability:

    • Is there a need to update README, API docs, runbooks or comments?
    • Are variable names, function names and comments clear?
    • Does the change create or remove public endpoints or commands that need documentation?

    Security and privacy (lightweight pass):

    • Obvious injection, insecure defaults, logging of sensitive data.
    • New configuration flags that might need privilege checks.
    • Use of deprecated or insecure patterns already noted in EVA docs.

    Consistency with EVA patterns:

    • Alignment with ADRs and architecture documents where relevant.
    • Respect for pod-specific conventions and file layouts.
    • Reuse of existing helpers instead of re-inventing them.
    1. Behaviour Template (How The Agent Should Act)

    High-level behaviour:

    • Read context before commenting.
    • Scan the diff once, then focus on the most impactful issues.
    • Prefer three to eight solid comments over dozens of minor nits.
    • Always add a closing summary with clear next steps.

    When reviewing a PR, the Review Agent should:

    • Read .eva-pod.yaml to understand pod and service.
    • Read .github/copilot-instructions to learn stack and conventions.
    • Read the PR description and linked issues.

    • Scan the diff and classify concerns into themes:

    • Tests.

    • Docs.
    • Style and readability.
    • Correctness.
    • Security and privacy.

    Then the agent should:

    • Add inline comments on lines where a specific suggestion is useful.

    • Add a final summary comment that:

    • States whether the agent sees likely blockers.

    • Groups suggestions by theme.
    • Points to any missing tests or documentation.

    Tone guidelines:

    • Use “suggestion” instead of “requirement”.
    • Provide examples when suggesting refactors or tests.
    • Use neutral language, for example “Consider adding” or “It may be safer to”.
    1. Integration With Jarvis And GitHub

    Jarvis workflows use the Review Agent in a predictable way.

    Triggers:

    • On pull_request opened or synchronized for eligible repos.
    • On a manual command such as “/jarvis review” in a PR comment.

    Jarvis responsibilities:

    • Identify whether the PR belongs to an EVA repo with .eva-pod.yaml.
    • Check devmaster-rules to see if P06 applies for this repo and change type.
    • Collect context: repo, PR, pod, service, instructions, relevant docs.
    • Call the Review Agent endpoint in EVA DevTools or an equivalent LLM.
    • Post the agent’s comments back into GitHub using a bot identity.
    • Log the interaction in EVA telemetry for LiveOps and audits.

    Minimum data Jarvis should send to the Review Agent:

    • Repository identifier and pod metadata.
    • Pull request identifier and diff content.
    • High-level descriptor of change type if available (feature, bugfix, infra).
    • Pointers to repo conventions and ADRs if present.

    Minimum data Jarvis should persist after the review:

    • Date and time of the review.
    • PR identifier and repo name.
    • Agents involved (P06 and any others).
    • Count of comments and blockers.
    • Any severe security tags the agent raised.

    Jarvis must not store raw sensitive code or full diffs in long-term logs. It should store identifiers and summary statistics.

    1. Example Agent Profile (Conceptual)

    Agent identity:

    • agent_id: eva-review-agent
    • display_name: EVA Review Agent
    • description: First-pass reviewer for pull requests against EVA standards.

    Capabilities:

    • Reads PR descriptions, diffs and linked issues.
    • Understands pod and service from .eva-pod.yaml.
    • Uses repository instructions for style and stack knowledge.
    • Produces inline comments and a summary review.

    Inputs:

    • repo_name
    • pr_number
    • pod and service information
    • diff_text or PR patch
    • optional: related requirement or ADR identifiers

    Outputs:

    • list of inline_comments with file, line and message.

    • summary_comment with:

    • overview.

    • suggestions_count.
    • blockers_count.
    • key themes.
    • telemetry_snapshot for logging.

    This profile can be implemented as:

    • An EVA DevTools API endpoint.
    • A GitHub Copilot Custom Agent profile.
    • A local script harnessed by jarvis-review workflows.
    1. Evolution And Future Work

    The Review Agent should improve over time as EVA standards mature. Potential enhancements include:

    • Deeper coupling with the Testing Agent to auto-suggest specific test cases.
    • Tighter integration with security tooling for better vulnerability hints.
    • Per-pod specialization so the agent learns about EVA-specific patterns, for example APIM policies or RAG query flows.
    • Configurable “strictness levels” per repo or branch.
    • Better explanations for junior developers to support learning, not only compliance.

    Changes to P06 should be coordinated with:

    • P01 DevTools Map.
    • P02 Requirements Agent.
    • P03 Scrum Agent.
    • P05 Scaffolder Agent.
    • P07 Testing Agent.

    Whenever Definition of Done, coding standards or CI rules change for a repo or pod, the Review Agent configuration and prompts must be updated so advice remains aligned with EVA’s actual practices.
