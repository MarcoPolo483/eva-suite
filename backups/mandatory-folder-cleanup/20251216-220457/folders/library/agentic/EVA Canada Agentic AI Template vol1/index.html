    Volume 1 – Platform & Agentic Requirements (Template)¶

    Subtitle: How AI Agents Should Work to Support Federal Government Requirements

    1. Purpose¶

    This Volume defines platform and agentic requirements for a departmental AI assistant and agent platform (“the Platform”) that is suitable for use in the Government of Canada.

    It is intended to be:

    • A template any GC department can adopt and tailor.
    • A basis for:
    • Architecture and design.
    • Governance and approvals.
    • Development roadmaps.

    This Volume answers what the Platform and agents must do to be considered:

    • Federal-grade (secure, reliable, auditable, accessible, bilingual).
    • Human-centred (supporting, not replacing, accountability).
    • Governable (traceable, explainable, controllable).

    Technical implementation details are expanded in Volume 2 – Engineering & Delivery Guide and operational details in Volume 3 – LiveOps & Governance Model.

    2. Scope¶

    2.1 In Scope (Template Content)¶

    This Volume covers functional and non-functional requirements for:

    • Agentic AI capabilities (goal-driven, tool-calling agents).
    • Project spaces and data boundaries.
    • Core platform services:
    • Agent runtime and orchestration.
    • Policy / configuration directory.
    • Logging, metrics, and trust posture (conceptual).
    • UX, explainability, and AI literacy requirements.
    • Solution lifecycle (Sandbox → Pilot → Production).
    • Agent catalogue and discovery.
    • Security, privacy, accessibility, bilingualism, observability, and extensibility.

    2.2 Out of Scope¶

    • Implementation details (tooling, cloud provider) → see Volume 2.
    • Operational processes (incident management, change approval) → see Volume 3.

    3. Audience¶

    • Program / Product Owners – defining agent use cases and scope.
    • Architects & Technical Leads – designing and integrating the Platform.
    • AI Governance, Privacy, Security, Legal – assessing compliance and risk.
    • Platform & DevOps Teams – implementing requirements in code and infra.

    4. Key Concepts & Definitions¶

    4.1 Agentic AI / Agent¶

    An agent is a configured, semi-autonomous digital assistant that can:

    • Interpret a goal expressed by a user or system.
    • Plan and execute multi-step actions.
    • Call tools / APIs / search / RAG services.
    • Ask clarification questions when needed.
    • Log relevant steps for oversight and audit.

    Agents:

    • Operate within a project space.
    • Are bounded by policies, tools, autonomy level, and data scope.
    • Are not free-running decision-makers.

    4.2 Project Space¶

    A project space is a logical container within the Platform that defines:

    • Business purpose and objectives.
    • Data sources and security classification (e.g., Unclassified, Protected B).
    • Tools/APIs and RAG indices available to agents.
    • Role-based access control (RBAC) for users and services.
    • One or more agents configured to operate within these constraints.
    • References to governance artefacts:
    • Privacy assessments, AI impact assessments.
    • Risk registers, decision logs, approvals.

    4.3 Platform Components (Abstract)¶

    Each department will map these to its own architecture. Typical components include:

    • Foundation Services
      Core capabilities: RAG pipeline, tool registry, agent runtime, safety/guardrails, usage metering, API gateway integration.

    • Assistant UIs

    • General-purpose conversational UI (“Chat”).

    • Domain / project-aware assistant UI (“Domain Assistant”).

    • LiveOps & Monitoring Interface
      Dashboards and tools to:

    • See health, usage, incidents.

    • Manage kill switches and autonomy settings.

    • Directory / Registry
      System of record for:

    • Projects, agents, tools.

    • Policies, autonomy levels, approvals.

    • Shared Services
      Authentication, authorization, internationalization (i18n), accessibility personalization, notifications, logging, and metrics.

    5. Design Principles¶

    Each requirement below is tagged [CORE] or [ADV].

    1. Platform, Not Point Solutions
      The Platform hosts multiple assistants and agents. New use cases are implemented on the Platform, not as standalone stacks.
      Tag: [CORE]

    2. Human-Centred & Human-in-the-Loop
      Agents assist; they do not replace accountable decision-makers.
      Autonomy is configured and enforced.
      Tag: [CORE]

    3. Outcome-Driven
      Each project space and agent has clear objectives and KPIs (e.g., time saved, quality, satisfaction).
      Tag: [CORE]

    4. Governed & Auditable
      Behaviour is observable; interactions and configuration changes can be traced.
      Tag: [CORE]

    5. Modular & Reusable
      RAG, tools, agents, safety, and UX are reusable building blocks, not one-off code.
      Tag: [ADV]

    6. Accessible & Bilingual
      Interfaces follow WCAG 2.1 AA and support English and French.
      Tag: [CORE]

    7. Secure & Policy-Aligned
      The Platform complies with:

    8. GC security classifications,
    9. data residency constraints,
    10. AI, privacy, and digital policies.
      Tag: [CORE]

    6. High-Level Architecture Expectations¶

    Requirements¶

    • R-Arch.1 [CORE] – The Platform SHALL expose capabilities via well-defined APIs and/or service interfaces (e.g., API gateway), with:
    • Project / application / feature identifiers on requests.

    • Authentication and authorization integrated with departmental identity systems.

    • R-Arch.2 [CORE] – The Platform SHALL enforce project-space boundaries:

    • Tools and data sources are explicitly associated with project spaces.

    • Agents cannot access tools or data outside their project space.

    • R-Arch.3 [CORE] – The Platform SHALL log:

    • Interactions (prompts, responses, high-level metadata).
    • Tool calls and outcomes.

    • Configuration and model version changes.

    • R-Arch.4 [ADV] – The Platform SHOULD support multiple deployment “faces”:

    • Web UIs.
    • Embedded widgets in line-of-business systems.
    • API clients (e.g., low-code tools, automations).

    Implementation Notes (Non-Normative)¶

    • Use any API gateway technology (APIM, API Gateway, etc.) that can:
    • Inject headers (project, app, feature).
    • Enforce authentication (OAuth, JWT, mTLS).
    • Produce logs for observability.
    • Represent project/app/feature IDs consistently in headers or claims so they can be:
    • Used for routing.
    • Used for cost tracking and analytics.
    • Consider using a central logging pipeline (e.g., cloud logging + SIEM) to aggregate runtime and governance data.

    7. Epics & Requirements¶

    EPIC 1 – Agent Runtime & Orchestration¶

    Goal: Provide a reusable runtime for agentic behaviour across project spaces.

    Functional Requirements¶

    1. Multi-Step Workflows

    2. R1.1 [CORE] – The runtime SHALL support agents that:

    3. Plan and execute multi-step actions (e.g., plan → execute → summarize).
    4. Call multiple tools/APIs in one workflow.

    5. Ask clarification questions when context is insufficient.

    6. Project-Scoped Tools

    7. R1.2 [CORE] – For each project space, authorized admins SHALL be able to:

    8. Register tools/APIs with metadata (description, owner, auth type, classification).
    9. Assign tools to one or more agents in that project.

    10. Enable/disable tools and adjust configuration without code changes.

    11. Autonomy Levels

    The Platform SHALL support at least three levels:

    • Advisory –
      May retrieve and explain information, and draft neutral/descriptive content;
      MUST NOT recommend or decide actions.

    • Assistive –
      May propose recommendations, next steps, drafts;
      Human must review and approve before use in decisions or client communications.

    • Semi-Autonomous –
      May perform low-risk, reversible tasks within clearly defined constraints;
      MUST NOT make final decisions about rights, benefits, or obligations.

    • R1.3 [CORE] – The Platform SHALL allow agents to be configured at a specific autonomy level and SHALL enforce behaviour consistent with that level.

    • API Gateway Routing

    • R1.4 [CORE] – All tool calls initiated by agents MUST pass through a managed interface (e.g., API gateway) that:

    • Attaches project/app/feature identifiers.
    • Propagates authenticated user identity/role where permitted.

    • Applies rate limiting and access controls.

    • Metric Hooks

    • R1.5 [CORE] – The runtime SHALL emit events/logs for:

    • Task start and end (with timestamps).
    • Tools used and outcomes (success/failure).

    • Escalations (e.g., hand-off to human).

    • Configurable Memory

    • R1.6 [ADV] – The runtime SHALL support:

    • Session-level memory (conversation context).

    • Optional user/project-level memory (preferences, reusable summaries), only where aligned with privacy and classification policies.

    • Traceable Decision Paths

    • R1.7 [CORE] – The runtime SHALL record metadata (input summary, tools, key intermediate steps, model version, correlation IDs) sufficient to:

    • Reconstruct how an output was produced.
    • Support audits and investigations.

    Implementation Notes (Non-Normative)¶

    • Start with “simple agents” that orchestrate:
    • Chat model + RAG + 1–3 tools (search, retrieval, basic APIs).
    • Represent autonomy levels as configuration fields in the Directory (e.g., autonomy: "advisory" | "assistive" | "semi-autonomous").
    • Use correlation IDs at:
    • API gateway.
    • Agent runtime.
    • Tool calls.

    So logs can be joined later.

    EPIC 2 – Trust, Oversight & Monitoring (Conceptual LiveOps)¶

    Goal: Provide visibility and governance over agents in line with GC risk expectations.

    Functional Requirements¶

    1. Central Logging

    2. R2.1 [CORE] – The Platform SHALL provide aggregated, searchable logs of:

    3. Interactions (subject to classification/retention policy).
    4. Tool calls and results (metadata level).

    5. Configuration and model version changes.

    6. Trust Posture

    7. R2.2 [ADV] – The Platform SHOULD maintain a trust posture per project/agent based on:

    8. Governance completeness (assessments, approvals).
    9. Monitoring coverage and alerting.
    10. Incident history.

    11. Autonomy level and human-in-the-loop design.

    12. Alerts & Kill Switch

    13. R2.3 [CORE] – The Platform MUST support:

    14. Alerts for significant anomalies (usage spikes, error spikes, policy violations where detectable).

    15. A kill switch to quickly disable an agent or feature.

    16. Reporting

    17. R2.4 [CORE] – The Platform SHALL provide:

    18. Usage dashboards (volume, trends).
    19. Per-agent metrics (usage, feedback, error rates).
    20. Exportable reports to support departmental oversight bodies.

    Implementation Notes¶

    • Start with existing department monitoring tools (cloud monitoring + SIEM).
    • Define a minimal “Agent Health Dashboard”:
    • Requests per project/agent.
    • Error rate.
    • Kill switch state.
    • Link incidents and changes (from ITSM/ITIL tools) to agents and project spaces.

    EPIC 3 – Directory: Policy & Configuration Registry¶

    Goal: Provide a single, governed registry for projects, agents, tools, roles, and policies.

    Functional Requirements¶

    1. Project Registry

    2. R3.1 [CORE] – The Directory SHALL maintain a record for each project space:

    3. Name, description, business domain.
    4. Owner, sponsor, contact.
    5. Data sources and classification.

    6. Governance artefact links (privacy/AI assessments, risk logs).

    7. Agent Registry

    8. R3.2 [CORE] – For each project, the Directory SHALL store:

    9. Agent name and description.
    10. Autonomy level.
    11. Allowed tools and data scopes.

    12. Intended use cases and KPIs.

    13. Role & Access Mapping

    14. R3.3 [CORE] – The Directory SHALL map:

    15. Roles (platform admin, project owner, agent owner, end-user, auditor, etc.).

    16. Permissions (configure agents, view logs, use agents, etc.).

    17. Policy as Configuration

    18. R3.4 [ADV] – Behavioural policies (e.g., “no decision support using personal information”) SHOULD be expressed as configuration wherever feasible.

    19. Configuration Versioning

    20. R3.5 [CORE] – The Directory SHALL version:

    21. Project records.
    22. Agent configurations (prompts, tools, autonomy).

    23. Role/permission mappings.

    24. Approval Workflows

    25. R3.6 [ADV] – High-risk changes (e.g., increasing autonomy, broadening data scope) SHOULD require:

    26. Explicit approval from designated roles (program, AI governance, privacy, security).
    27. Logged rationale and outcomes.

    Implementation Notes¶

    • The Directory can initially be:
    • A database + admin UI, or
    • A well-structured configuration repo (e.g., YAML/JSON) with a simple management tool.
    • Over time, evolve to:
    • An internal self-serve portal for registering project spaces and agents.

    EPIC 4 – UX, Explainability & AI Literacy¶

    Goal: Make agents understandable and safe for non-technical public servants.

    Functional Requirements¶

    1. Capability Cards

    2. R4.1 [CORE] – For each agent, user interfaces SHALL show:

    3. What it does (capabilities).
    4. What it does not do (limitations).
    5. Data sources / types it uses.

    6. How the user remains responsible (e.g., “always verify before acting”).

    7. Explainability

    8. R4.2 [CORE] – For each response, the UI SHOULD:

    9. Explain at a high level how the answer was produced (e.g., “based on these documents/tools”).

    10. Provide links to underlying documents or key references where feasible.

    11. Embedded AI Literacy

    12. R4.3 [CORE] – The Platform SHALL include contextual guidance:

    13. Responsible use tips.
    14. Prohibited uses (e.g., making final eligibility decisions).

    15. Pointers to departmental AI guidelines.

    16. Feedback

    17. R4.4 [CORE] – Users SHALL be able to provide:

    18. Quick feedback (thumbs up/down, flag).
    19. Optional comments (incorrect, incomplete, biased, harmful).

    Implementation Notes¶

    • Reuse corporate design systems and accessibility guidelines.
    • Embed short “info” panels and tooltips:
    • “What this agent can help with”
    • “How to supervise its outputs”
    • Surface feedback in LiveOps dashboards for continuous improvement.

    EPIC 5 – Solution Lifecycle (Sandbox → Pilot → Production)¶

    Goal: Provide a standard path for experiments to become governed solutions.

    Functional Requirements¶

    1. Maturity Tiers

    2. R5.1 [CORE] – The Platform SHALL support at least:

    3. Sandbox – experimental; clearly labelled; restricted to low-risk data.
    4. Pilot – limited audience; documented governance; monitored.

    5. Production – integrated into normal work; full governance evidence.

    6. Gating Criteria

    7. R5.2 [CORE] – For each tier, the Platform (and governance process) SHALL define minimal criteria, including:

    8. Documentation completed.
    9. Monitoring in place.
    10. Training/AI literacy for intended users.

    11. Approvals from relevant bodies.

    12. Migration Support

    13. R5.3 [ADV] – The Platform SHOULD provide patterns/templates for moving from Sandbox to Pilot to Production with minimal rework.

    14. Governance Links

    15. R5.4 [CORE] – The Directory SHALL record governance status (Sandbox/Pilot/Production) and link to key assessments and approvals.

    Implementation Notes¶

    • Start with a simple checklist per tier that program teams must fill.
    • Use tags/fields in the Directory and LiveOps to:
    • Filter agents by tier.
    • Generate lists of “pilots ready to scale”.

    EPIC 6 – Agent Catalogue & Discovery¶

    Goal: Make agents discoverable and reusable, within RBAC boundaries.

    Functional Requirements¶

    1. Catalogue

    2. R6.1 [ADV] – The Platform SHOULD provide a catalogue listing:

    3. Agents by program area, project, function.

    4. Owners, sponsors, lifecycle tier.

    5. Access-Aware Listing

    6. R6.2 [CORE] – Users SHALL only see agents they:

    7. Are permitted to use, or

    8. Can request access to.

    9. Reusable Patterns

    10. R6.3 [ADV] – The Platform SHOULD support cloning/template-based creation of agents to reuse successful patterns.

    Implementation Notes¶

    • Initially, a simple internal web page or dashboard may serve as the catalogue.
    • Over time, integrate catalogue into the Platform UI itself.

    EPIC 7 – Safety, Guardrails & Testing¶

    Goal: Ensure agents behave within policy boundaries, especially around personal information and decisions.

    Functional Requirements¶

    1. Guardrail Policies

    2. R7.1 [CORE] – The Platform SHALL support configuration of guardrail policies at project/agent level (e.g., allowed data types, prohibited behaviours).

    3. Input/Output Controls

    4. R7.2 [ADV] – Where personal or sensitive information may appear, the Platform SHOULD:

    5. Offer input warnings/blocks for obvious high-risk patterns (if feasible).

    6. Limit or mask output where required by policy.

    7. Pre-Production Testing

    8. R7.3 [CORE] – Before moving to higher tiers (Pilot/Production), agents SHALL be tested using:

    9. Representative “golden path” prompts.
    10. Edge cases and abuse scenarios.

    11. Documented results and known limitations.

    12. Policy Versioning

    13. R7.4 [ADV] – Guardrail configurations MUST be versioned and auditable (who changed what, when, and why).

    Implementation Notes¶

    • Start with process-level guardrails (intake, training, checklists).
    • Add technical guardrails over time (pattern detection, restricted tools).
    • Use test scripts stored alongside code/config to exercise agents before promotion.

    8. Non-Functional Requirements (NFRs)¶

    8.1 Security & Privacy¶

    • NFR-Sec.1 [CORE] – The Platform SHALL operate within approved GC security classifications and must not process data beyond its authorization.
    • NFR-Sec.2 [CORE] – Data residency and data-in-transit/at-rest protections SHALL align with GC and departmental security standards.
    • NFR-Sec.3 [ADV] – The Platform SHOULD support multiple model backends and record which model/version was used per interaction.
    • NFR-Sec.4 [CORE] – RBAC SHALL apply to all key functions: agent use, configuration, log access.

    8.2 Accessibility & Official Languages¶

    • NFR-Acc.1 [CORE] – All user-facing interfaces SHALL meet WCAG 2.1 AA (or applicable GC standard).
    • NFR-Acc.2 [CORE] – The Platform SHALL support English and French UI and, where applicable, content assistance.
    • NFR-Acc.3 [ADV] – The Platform SHOULD offer personalization controls (contrast, font size, motion, etc.).

    8.3 Reliability & Performance¶

    • NFR-Rel.1 [CORE] – Define and monitor SLOs for key flows (e.g., P95 latency).
    • NFR-Rel.2 [CORE] – Design for high availability consistent with departmental expectations for internal services.
    • NFR-Rel.3 [ADV] – Implement degraded modes (fallback behaviours) for major dependencies.

    8.4 Observability¶

    • NFR-Obs.1 [CORE] – The Platform SHALL emit structured logs and metrics consumable by departmental monitoring.
    • NFR-Obs.2 [CORE] – Correlation IDs SHALL be used across requests, agent steps, and tool calls.
    • NFR-Obs.3 [ADV] – The Platform SHOULD enable replay of execution traces (metadata only) for audits.

    8.5 Extensibility¶

    • NFR-Ext.1 [CORE] – New agents, tools, and project spaces MUST be addable without major redesign.
    • NFR-Ext.2 [ADV] – Integration points (APIs, events) SHOULD be designed for potential cross-department reuse.

    9. Tailoring Guidance for Departments¶

    When adopting this Volume, departments SHOULD:

    1. Fill in the Instantiation Header (Section A).
    2. Review each requirement and:
    3. Confirm adoption of [CORE] items, or document justified deviations.
    4. Decide which [ADV] items to adopt now versus later.
    5. Map internal roles to:
    6. Platform Service Owner
    7. Platform Ops Team
    8. Project/Agent Owners
    9. AI Governance, Privacy, Security contacts.
    10. Cross-reference:
    11. Departmental AI guidelines.
    12. Privacy and security assessment processes.
    13. Digital and data strategies.

    • Monitor, operate, and govern the Platform in production.

      Provides:

    • Archetypes and patterns for applying these requirements to real federal use cases.
