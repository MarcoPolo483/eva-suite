    EVA Requirements Engine (P02) Overview¶

    P02 is the EVA Requirements Engine that keeps the Copilot Design Document (CDD) landscape organized without touching source repositories. It reads the authoritative EVA Suite tree snapshot stored under _SeniorAdv-office, inventories CDD activity, detects missing anchors, and produces demo-friendly reports that product and engineering leads can consume quickly.

    Today the engine is considered feature-complete. The priority is clarity—not adding new moving parts—so the focus is on consistent reporting, lightweight automation, and guidance for other pods that need to understand CDD coverage before shipping Unified EVA Chat Frame improvements.

    Tip: all task JSON definitions live under _SeniorAdv-office/p02-task-queue/. The naming convention matches the IDs above, so P02-005...json maps directly to p02-005-cdd-gap-detection.ps1.

    Running P02 locally¶

    1. Refresh the EVA Suite tree snapshot whenever repositories change: 
      pwsh -File .\scripts\refresh-eva-suite-tree.ps1 -EvaSuiteRoot "C:\Users\marco\Documents\_AI Dev\EVA Suite" -LwoPath "C:\Users\marco\Documents\_AI Dev\EVA Suite\_SeniorAdv-office"
    2. Queue or edit the desired task JSON in _SeniorAdv-office/p02-task-queue/.
    3. Execute the local agent so it picks the correct handler: 
      pwsh -File .\scripts\eva-local-agent.ps1 -PathToTaskJson "C:\Users\marco\Documents\_AI Dev\EVA Suite\_SeniorAdv-office\p02-task-queue\P02-006-cdd-dependency-graph.json"
    4. Inspect _SeniorAdv-office/p02-reports/ for Markdown, JSON, or GraphViz outputs. The agent also writes a per-run audit file such as p02-006-cdd-dependency-graph-agent-report.json that records success state, handler path, and any missing outputs.

    Interpreting the main reports¶

    • Intake inventory (P02-001) – Provides repo-level file counts so you can sanity-check that the tree snapshot captured every workspace. Use it when enabling new repos or validating CI sync jobs.
    • CDD coverage (P02-002) – Highlights which repos contain cdds/ folders or individual *cdd*.md files. Repos with 0/0 coverage should appear in the CDD TODO list; those with files but no directories (for example eva-suite) indicate ad-hoc documentation that should migrate into canonical anchors.
    • Gap detection (P02-005) – Cross-references each cdds/ directory against the canonical anchors (README.md, requirements.md, architecture.md, tests.md). Missing anchors surface as a table so pods can quickly backfill.
    • Template generator (P02-007) – Produces a timestamped starter CDD and a JSON inventory of all templates currently staged. Use it to seed new docs without hunting for older examples.
    • Ecosystem report (P02-008) – Aggregates CDD counts, file sizes, and recent edits. It is the go-to artifact for demos because it shows breadth (repos covered) and freshness (last updated UTC timestamps) in a single page.

    For a living backlog of missing anchors and candidate docs, see docs/cdd-todo-list.md.

    When to use P02¶

    • You need to brief another pod (POD-X or POD-F) on the current CDD footprint before a design review.
    • You are about to demo Unified EVA Chat Frame and want to cite up-to-date documentation stats.
    • You plan to author a new CDD and want a compliant template plus a checklist of missing anchor files.
    • You are grooming intake items and must prove that sprint-ready artifacts exist (use P02-003 and P02-005 for evidence).

    Future Enhancements (Not Implemented)¶

    • CDD Quality Heatmap – Idea: combine linting + anchor completeness to show readiness tiers.
    • Model Drift Watcher – Idea: compare historical CDD anchors with recent telemetry to flag stale docs.
    • Unified CDD Dashboard UI – Idea: surface the Markdown/JSON artifacts in an interactive dashboard instead of static files.
    • Automated Reminders – Idea: notify repo owners when coverage drops or gap detection fails for their CDD root.
