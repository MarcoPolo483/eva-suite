    GitHub Spark Integration Strategy - EVA 2.0¶

    Date: November 28, 2025
    Product Owner: Marco Presta
    Scrum Master: Copilot (GitHub Copilot)
    Context: Spark as accelerator, EVA-Sovereign-UI Tool as canonical maintainer

    Executive Summary¶

    GitHub Spark: AI-powered rapid prototyping tool for building micro-apps with natural language prompts. Announced July 23, 2025

    Integration Philosophy: - Spark = Upstream Accelerator: Fast prototyping, code generation, refactoring - EVA-Sovereign-UI Tool = Downstream Governance: Maintains layouts, enforces GC/Five Eyes compliance, ensures a11y+i18n

    Marco's Reaction:

    "Yes, Spark fits perfectly as an accelerator—upstream of EVA-Sovereign-UI—and your Siebel-style Tool becomes the canonical maintainer of the code afterwards."

    What GitHub Spark Provides¶

    1. Natural Language Prototyping¶

    Capability: Describe UI in plain English/French, get working code instantly

    Example Prompt:

    "Create a Svelte page for EVA Sovereign UI that shows a GC header, bilingual hero banner, and a card grid for Jurisprudence quick actions. Use Svelte + Tailwind-style utility classes. Keep content placeholders and no real data."

    Spark Output: - Working Svelte component file - Basic layout with sample data - Responsive styling (mobile, tablet, desktop) - Live preview in browser

    Benefit: Go from idea to prototype in minutes (vs hours of hand-coding)

    2. Live Preview & Iteration¶

    Capability: See changes instantly, iterate with follow-up prompts

    Example Iteration:

    "Move the hero to the top"
    "Add another card row"
    "Make the header sticky on scroll"

    Benefit: Rapid experimentation without touching code editor

    3. One-Click Deployment¶

    Capability: Deploy prototype to GitHub Pages or create new GitHub repo

    Workflow: 1. Prototype in Spark (natural language + live preview) 2. Click "Create repo" → New GitHub repo created 3. Clone to Codespaces or local machine 4. Merge into EVA-Sovereign-UI monorepo

    Benefit: No manual repo setup, instant shareable demo

    Why Spark ≠ Canonical Maintainer¶

    What Spark Can't Do (By Design)¶

    1. GC/Five Eyes Compliance Enforcement¶

    Missing: - No sovereign profile awareness (Canada GC, USA.gov, GOV.UK) - No mandatory component validation (e.g., "GC header required") - No bilingual content checks (EN/FR parity)

    Tool Advantage: Enforces profile requirements at save time (can't ship non-compliant layout)

    2. Accessibility Governance¶

    Missing: - No WCAG 2.1 AA validation (can generate non-accessible code) - No screen reader testing integration (NVDA, JAWS) - No keyboard navigation enforcement (tab order, focus management)

    Tool Advantage: Real-time a11y panel, blocks deployment if violations found

    3. Internationalization Completeness¶

    Missing: - No i18n key coverage tracking (may miss translations) - No locale-specific formatting (dates, numbers, currency) - No grammatical gender support (French conjugation)

    Tool Advantage: I18n panel shows % coverage, highlights missing keys

    4. Layout Metadata Ownership¶

    Missing: - Spark outputs code, not metadata (no layouts/pageId.json) - No component registry integration (what components are available?) - No props schema validation (can pass invalid props)

    Tool Advantage: Layout JSON is source of truth, code is generated FROM metadata

    Integration Workflow: Spark → Tool → Production¶

    Phase 1: Rapid Prototyping (Spark)¶

    User: Front-end developer (or Marco himself)

    Steps: 1. Open GitHub Spark (spark.github.com or via Codespaces) 2. Describe page in natural language: 

    "Create a Svelte page for EVA Sovereign UI that shows:
- GC header with language toggle
- Hero banner with title 'Welcome to EVA' in EN/FR
- 3-column card grid for quick actions
- GC footer with social media links
Keep it simple, no backend data yet."
    3. Review live preview, iterate with follow-up prompts 4. When satisfied, click "Create repo" or "Export code"

    Output: - Spark-generated Svelte component (code, not metadata) - Sample data hardcoded in component - Basic styling (Tailwind or inline CSS)

    Time: 5-10 minutes (vs 1-2 hours hand-coding)

    Phase 2: Import to Tool (EVA-Sovereign-UI Tool)¶

    User: Same developer, now using WYSIWYG Tool

    Steps: 1. Clone Spark repo or copy code from Spark export 2. Open EVA-Sovereign-UI Tool in VS Code 3. Choose "Import Spark prototype" (new feature to build) 4. Tool analyzes Spark-generated code: - Identifies components (GC header, hero, cards, footer) - Maps to registered Sovereign components (from ComponentRegistry.ts) - Generates initial layouts/spark-import-{timestamp}.json 5. Developer reviews mapping in visual canvas: - Confirms: "Yes, this is a GCHeader component" - Adjusts: "No, this is a CustomHero, not standard HeroBanner" 6. Developer adds metadata: - Set sovereign profile: canada_gc_intranet - Map hardcoded strings to i18n keys: "Welcome to EVA" → hero.title - Add ARIA labels where missing 7. Save layout metadata

    Output: - Layout JSON file: layouts/eva-da/spark-import-dashboard.json - Svelte component refactored to: 

    <script lang="ts">
  import { LayoutRenderer } from '@eva-suite/eva-sovereign-ui';
  export let data;
</script>

<LayoutRenderer pageId="spark_import_dashboard" {data} />

    Time: 15-30 minutes (mapping + metadata enrichment)

    Phase 3: Governance & Compliance (Tool)¶

    User: QA Master (autonomous agent) or Marco (manual review)

    Steps: 1. Tool runs automated checks: - A11y Panel: Runs Axe audit, reports violations - I18n Panel: Checks i18n key coverage (100% required) - Profile Validator: Ensures sovereign profile requirements met (e.g., GC header present) 2. Developer fixes violations: - Add missing alt text to images - Translate remaining hardcoded strings - Adjust color contrast (if failing WCAG AA) 3. Tool re-runs checks, verifies all green 4. Developer commits layout JSON + refactored Svelte

    Output: - Compliant layout metadata (passes all checks) - Production-ready code (no a11y/i18n violations)

    Time: 10-20 minutes (fix violations, re-test)

    Phase 4: Deployment (CI/CD)¶

    User: Automated (GitHub Actions)

    Steps: 1. Developer pushes to dev branch 2. GitHub Actions triggers: - Build: npm run build (generates static assets) - Test: npm test (runs unit tests, a11y tests) - Deploy: Pushes to Azure Static Web App /dev slot 3. Smoke tests run (see AZURE-INFRA-PLAN.md) 4. If all pass, deploy to test branch → repeat 5. If all pass, deploy to main branch → production

    Output: - Page live in production: https://eva-suite.marcopolo483.dev/dashboard

    Time: 10-15 minutes (automated, no manual steps)

    User Stories (6 Stories)¶

    US-01: Bootstrap a new EVA screen using Spark¶

    As a front-end developer
    I want to use GitHub Spark to rapidly generate an initial Svelte screen wired to EVA-Sovereign-UI
    So that I can go from idea to working prototype in minutes, then bring it under GC-compliant governance.

    Acceptance Criteria: - [ ] Spark-generated code compiles in EVA Svelte/Turbo/PNPM workspace - [ ] Layout is expressed in a way that can be mapped to Tool's layout schema (sections, components, props) - [ ] No sensitive/prod data in Spark prompts or code

    Narrative: 1. Dev opens GitHub Spark, describes page in natural language 2. Spark generates working micro app with sample data, styling 3. Dev tests UI in Spark's live preview, iterates with more prompts 4. When satisfied, dev clicks "Create repo" from Spark 5. Clones or opens in Codespaces 6. Moves/merges files into eva-sovereign-ui / eva-ui monorepo

    Time Estimate: 5-10 minutes (Spark prototyping)

    US-02: Import Spark prototype into EVA-Sovereign-UI Tool¶

    As a front-end developer
    I want to import the Spark-generated screen into the EVA-Sovereign-UI WYSIWYG Tool
    So that future changes are driven by metadata and governed layouts, not ad-hoc code.

    Acceptance Criteria: - [ ] A pageId and layout JSON are created and validated against schema - [ ] All visual elements present in Spark's prototype are represented as Tool sections/components - [ ] Generated layout passes basic a11y/i18n checks (or shows actionable warnings)

    Narrative: 1. Dev opens Tool, chooses "Import Spark prototype" 2. Pastes or points to Spark-generated Svelte component 3. Tool scans component, maps known patterns to registered Sovereign components 4. Generates initial layouts/{projectId}/{pageId}.json 5. Dev tweaks in visual canvas (re-order, rename IDs), saves

    Time Estimate: 15-30 minutes (import + mapping)

    US-03: Maintain the screen only via Tool (post-import)¶

    As a front-end developer
    I want to evolve the screen by editing the layout and props in the Tool
    So that GC/Five Eyes, a11y, and i18n rules are always enforced.

    Acceptance Criteria: - [ ] All layout changes are represented as diff in metadata (JSON/YAML), not hard-coded Svelte changes - [ ] A11y panel reports no blocking violations for new elements - [ ] I18n panel shows all keys resolved for en-CA and fr-CA - [ ] Sovereign profile (e.g., Canada) remains applied and required GC elements still present

    Narrative: 1. Dev needs to add "Voice input" mic button, adjust hero copy, add new quick-action card 2. Opens page in Tool (not Svelte file) 3. Adds VoiceInputButton component from palette to hero section 4. Adds new item to QuickActionsGrid via props editor 5. Tool updates layout JSON, ensures button has labels/tooltip keys/ARIA roles 6. Validates i18n keys exist in EN/FR, applies sovereign profile 7. Dev commits updated layout file; Svelte runtime picks it up automatically

    Time Estimate: 10-20 minutes (per change)

    US-04: Bring legacy screen (hand-coded Svelte) under Tool control¶

    As a developer
    I want to onboard an existing Svelte screen into the EVA-Sovereign-UI Tool
    So that I can maintain it via metadata and benefit from GC/a11y/i18n guardrails.

    Acceptance Criteria: - [ ] Legacy screen now uses LayoutRenderer with a pageId - [ ] Layout metadata stored in layouts/{projectId}/{pageId}.json - [ ] Original screen's visible behaviour is preserved - [ ] Any sections not yet mapped are clearly marked, with migration plan

    Narrative: 1. There's an older Svelte screen in apps/eva-da/src/routes/jp/search/+page.svelte built manually before Tool existed 2. Dev opens Tool, selects "Onboard existing screen", points at jp/search 3. Tool analyzes Svelte template, identifies what it can map to Sovereign components 4. Suggests draft layout JSON 5. Dev confirms mappings, manually resolves custom components 6. Once layout saved, runtime screen refactored to: 

    <script lang="ts">
  import LayoutRenderer from '$lib/layout/LayoutRenderer.svelte';
  export let data;
</script>
<LayoutRenderer pageId="jp_search" {data} />

    Time Estimate: 30-60 minutes (per legacy screen, depends on complexity)

    US-05: Developer safely refactors legacy code with Spark, but Tool remains source of truth¶

    As a developer
    I want to use GitHub Spark to help refactor or modernize legacy Svelte code
    So that I can speed up updates while keeping the Tool & metadata as the canonical layout source.

    Acceptance Criteria: - [ ] Refactored component still respects same props contract used by Tool-generated layout - [ ] No divergence in layout: sections/components in JSON still map 1:1 to runtime components - [ ] A11y/i18n checks pass in Tool after refactor

    Narrative: 1. Legacy Svelte component has complex, messy logic (e.g., results table) 2. Dev copies component or opens repo in Spark (via "Open in Codespaces / repo") 3. In Spark, prompts:

    "Refactor this Svelte component to separate presentation and data-loading logic. Keep all props and output identical." 4. Spark suggests cleaned-up diff 5. Dev reviews and applies changes locally (or via PR) 6. Importantly: - Does NOT change layout metadata in Spark - Layout structure still comes from Tool's layout JSON - Spark used only for component-level refactoring (performance, readability)

    Time Estimate: 20-40 minutes (per component refactor)

    US-06: Tool-led update of a legacy screen authored only in code¶

    As a front-end developer
    I want to gradually migrate a purely code-driven screen into a hybrid model
    So that we can maintain parts of it through the Tool while keeping complex custom logic in code.

    Acceptance Criteria: - [ ] Layout JSON expresses at least the frame and main sections - [ ] Custom component boundaries are clear (documented in component registry) - [ ] Page still works end-to-end; GC, a11y, and i18n requirements clearly tracked

    Narrative: 1. Very custom EVA DA page with niche CPP-D logic (full conversion risky in one shot) 2. Dev uses Tool to define only high-level layout regions (header, main, side panel, footer) 3. Keeps inner sections (e.g., CPPDComplexForm) as opaque custom Svelte components 4. Tool owns overall frame, GC header/footer, language toggle, etc. 5. Applies a11y/i18n checks to what it can "see" (labels, headings, global structure) 6. Over time, more of custom component decomposed into Sovereign components and moved into metadata

    Time Estimate: 1-2 hours (initial hybrid setup), then gradual migration over multiple sprints

    Positioning Spark vs Tool¶

    Aspect GitHub Spark EVA-Sovereign-UI Tool
    Purpose Rapid prototyping, code generation Layout governance, compliance enforcement
    Input Natural language prompts Component palette, visual canvas, props editor
    Output Svelte/React/Vue code Layout metadata JSON + generated code
    Governance None (generates any code you ask for) GC/Five Eyes profiles, a11y, i18n enforced
    Iteration Live preview + follow-up prompts Visual editor + real-time validation
    Use Case Greenfield prototypes, explorations, refactors Production layouts, compliant screens
    Lifecycle Stage Upstream (idea → prototype) Downstream (prototype → production)
    Maintainability Code-first (manual edits) Metadata-first (layout JSON drives code)
    Collaboration Share Spark link, iterate together Commit layout JSON, version control
    Testing Manual (dev tests in browser) Automated (a11y, i18n, profile validation)
    Deployment One-click to GitHub Pages (demo) CI/CD to Azure Static Web Apps (production)

    Complementary Roles: - Spark: Helps you get code fast and reshape ugly code - Tool: Ensures everything that ships is consistent, compliant, and maintainable

    Technical Implementation (Tool Features)¶

    Feature 1: "Import Spark Prototype" (NEW)¶

    UI: - New button in Tool: "Import from Spark" - File picker: Select Spark-generated Svelte file - Or paste Spark code directly into text area

    Backend Logic: 1. Parse Svelte component AST (using svelte/compiler) 2. Identify component hierarchy: - Find <script> tag → extract props, data - Find <template> → parse HTML structure - Map HTML tags/components to Sovereign component registry 3. Generate draft layout JSON: 

    interface SparkImporter {
  parseComponent(svelteCode: string): ComponentAST;
  mapToSovereignComponents(ast: ComponentAST): SectionMetadata[];
  generateLayoutJSON(sections: SectionMetadata[]): LayoutMetadata;
}
    4. Display in visual canvas, allow developer to confirm/adjust mappings 5. Save layout JSON to layouts/{projectId}/{pageId}.json

    Acceptance Criteria: - [ ] Can parse Spark-generated Svelte (React → Svelte conversion manual for now) - [ ] Recognizes common patterns (header, hero, cards, footer) - [ ] Generates valid layout JSON (passes schema validation) - [ ] Developer can manually resolve unmapped components

    Time Estimate: 2-3 sprints to build (Sprint 13-15)

    Feature 2: "Onboard Legacy Screen" (NEW)¶

    UI: - New button in Tool: "Onboard existing screen" - File picker: Select existing Svelte page (e.g., +page.svelte) - Or specify route path (e.g., /jp/search)

    Backend Logic: 1. Same parsing as Spark import, but for existing EVA code 2. Detect legacy component imports: 

    import { GCHeader } from '@eva-suite/eva-gc-look-n-feel'; // OLD
// Map to:
import { GCHeader } from '@eva-suite/eva-sovereign-ui'; // NEW
    3. Generate layout JSON + refactored Svelte file (using LayoutRenderer) 4. Show side-by-side diff (before/after) 5. Developer confirms, tool commits both files

    Acceptance Criteria: - [ ] Can parse existing EVA DA / EVA Chat Svelte pages - [ ] Detects legacy imports, suggests replacements - [ ] Generates layout JSON + refactored Svelte - [ ] Visual regression test passes (before/after screenshots identical)

    Time Estimate: 2-3 sprints to build (Sprint 13-15)

    Feature 3: "Spark Refactor Assistant" (FUTURE)¶

    Vision: Right-click on component in Tool, select "Refactor with Spark"

    Workflow: 1. Tool exports component code to Spark API 2. Sends refactor prompt: "Simplify this component, improve performance, keep props unchanged" 3. Spark returns refactored code 4. Tool shows diff, developer approves or rejects 5. If approved, replaces component file, re-validates layout

    Status: Future enhancement (not in Sprint 1-16 scope)

    Benefits Summary¶

    For Developers¶

    • ⚡ Faster prototyping: Spark → prototype in 5-10 minutes
    • 🎨 Visual iteration: Spark live preview → instant feedback
    • 🛡️ Safety net: Tool catches compliance issues before production
    • 📦 Metadata-first: Layout JSON easier to maintain than code
    • 🔄 Flexible: Use Spark for exploration, Tool for production

    For Marco (Product Owner)¶

    • 💰 Cost-effective: Developers spend less time on boilerplate (more features per sprint)
    • 🇨🇦 GC-compliant: Tool enforces Canada.ca Web Experience Toolkit automatically
    • 🌍 Five Eyes ready: Sovereign profiles (USA, UK, AUS, NZ) built into Tool
    • ♿ Accessible by default: A11y panel prevents WCAG violations
    • 🌐 Bilingual out-of-box: I18n panel ensures EN/FR parity

    For QA (Autonomous Agent)¶

    • 🤖 Automated checks: No manual WCAG audits (Tool runs Axe automatically)
    • 📊 Metrics: Track i18n coverage, a11y score, profile compliance per sprint
    • 🚨 Early detection: Catch violations in dev, not production
    • 🔁 Regression testing: Visual regression (Playwright screenshots) built into CI/CD

    Risks & Mitigations¶

    Risk 1: Spark generates non-compliant code¶

    Impact: Developer imports Spark prototype with WCAG violations, incorrect color contrast

    Mitigation: - Tool's a11y panel catches violations at import time - Developer must fix before saving layout metadata - QA Master (autonomous agent) runs Axe audit in CI/CD (blocks merge if failing)

    Risk 2: Spark imports create "zombie" metadata¶

    Impact: Layout JSON references components that don't exist in registry

    Mitigation: - Tool validates all component IDs against ComponentRegistry.ts at save time - Shows error: "Component 'CustomWidget' not found in registry. Register it first." - Developer must either: - Register new component in registry, OR - Map to existing component

    Risk 3: Developers bypass Tool, edit Svelte directly¶

    Impact: Code diverges from metadata, layout JSON becomes stale

    Mitigation: - CI/CD pre-commit hook: npm run validate-layouts - Checks: Does Svelte file still use LayoutRenderer? Does layout JSON exist? - If validation fails, blocks commit with error message:

    "⚠️ Direct Svelte edits detected. Please update via EVA-Sovereign-UI Tool to maintain metadata."

    Risk 4: Spark API rate limits or downtime¶

    Impact: Developers can't use Spark for prototyping during outage

    Mitigation: - Spark is upstream (nice-to-have, not critical path) - Developers can still hand-code Svelte, import to Tool afterwards - Document fallback workflow: "If Spark unavailable, create skeleton Svelte manually"

    Success Criteria¶

    Spark Integration Successful When: - ✅ 80% of new EVA screens start with Spark prototype (vs 0% today) - ✅ Average time to prototype drops from 1-2 hours → 10-15 minutes - ✅ 100% of Spark imports pass Tool validation (a11y, i18n, profile) - ✅ Zero production incidents from Spark-generated code (Tool catches issues)

    Tool Adoption Successful When: - ✅ 100% of new screens use LayoutRenderer (no hand-coded layouts) - ✅ 80%+ of legacy screens onboarded to Tool (by end of Sprint 16) - ✅ A11y/i18n violations drop to <5% (from current ~30%) - ✅ Marco can visually edit pages without touching code
