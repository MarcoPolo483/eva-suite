    ADR-004: Terraform State Management¶

    Status¶

    Accepted

    Context¶

    Terraform state tracks resource mappings and metadata for EVA 2.0 infrastructure. State management must address: - Concurrency: Multiple engineers/CI pipelines modifying infrastructure simultaneously - Security: State contains sensitive data (connection strings, IDs, resource ARNs) - Disaster Recovery: State loss would require full resource recreation - Audit Trail: Track who changed what and when - Encryption: State data must be encrypted at rest and in transit

    Terraform offers several state backends: 1. Local: File on disk (terraform.tfstate) 2. Azure Storage: Blob storage with lease-based locking 3. Terraform Cloud: Managed service with versioning, RBAC, collaboration 4. S3 + DynamoDB: AWS-native backend (not applicable for Azure)

    Decision¶

    Use Azure Storage Backend with: - Blob container: tfstate in Storage Account stevatfstatecac - Resource Group: rg-eva-tfstate-cac (Canada Central) - Authentication: Azure AD RBAC (no access keys) - Locking: Blob lease (automatic 20-minute timeout) - Versioning: Enabled (automatic backup of previous states) - Soft Delete: 90-day retention (accidental deletion protection)

    Rationale¶

    Why Azure Storage?¶

    1. Native Integration: Same cloud provider as infrastructure (no cross-cloud dependencies)
    2. Cost-Effective: ~$0.01/month per state file (vs. $70/month for Terraform Cloud)
    3. Security: Azure AD RBAC eliminates access key management
    4. Locking: Blob lease prevents concurrent writes (no external DynamoDB-like service)
    5. Versioning: Automatic state history with rollback capability
    6. Compliance: Data residency in Canada Central (meets PIPEDA requirements)

    Why Not Local State?¶

    • ❌ No concurrency control (multiple users = state conflicts)
    • ❌ No backup (disk failure = state loss)
    • ❌ No collaboration (state file on one machine)
    • ❌ Security risk (unencrypted file with secrets)

    Why Not Terraform Cloud?¶

    • ❌ Cost: $70/month per organization (5 users)
    • ❌ Vendor lock-in: Migration back to self-hosted requires re-initialization
    • ❌ Data residency: State stored in Terraform Cloud US region (compliance risk)
    • ❌ Overkill: Team of 1-2 doesn't need collaboration features yet

    Consequences¶

    Positive¶

    • ✅ Concurrent access with automatic locking (prevents state corruption)
    • ✅ State versioning for rollback (automatic snapshots)
    • ✅ Encrypted at rest (AES-256) and in transit (TLS 1.2+)
    • ✅ Azure AD RBAC (no hardcoded access keys)
    • ✅ Audit logs via Storage Analytics (who accessed state and when)
    • ✅ Low cost (~$0.01/month per environment)

    Negative¶

    • ❌ Manual setup required (Storage Account, Container, RBAC roles)
    • ❌ State lock timeout: 20 minutes (long-running applies may timeout)
    • ❌ No built-in collaboration UI (must use terraform state pull to inspect)
    • ❌ Lease conflicts during CI/CD failures (requires manual unlock)

    Mitigation¶

    • Automate Storage Account setup via Azure CLI script (one-time operation)
    • Document lock timeout handling in runbook (terraform force-unlock)
    • Use CI/CD pipeline logs for audit trail (GitHub Actions captures plan/apply output)
    • Monitor state file size (alert if >10 MB, indicates excessive resources)

    Implementation¶

    Backend Configuration¶

    # env/dev/backend.tf
terraform {
  backend "azurerm" {
    resource_group_name  = "rg-eva-tfstate-cac"
    storage_account_name = "stevatfstatecac"
    container_name       = "tfstate"
    key                  = "eva-dev.terraform.tfstate"
    use_azuread_auth     = true
  }
}

    Prerequisites (Manual Setup)¶

    # Create resource group
az group create --name rg-eva-tfstate-cac --location canadacentral

# Create storage account (globally unique name)
az storage account create \
  --name stevatfstatecac \
  --resource-group rg-eva-tfstate-cac \
  --location canadacentral \
  --sku Standard_LRS \
  --encryption-services blob \
  --allow-blob-public-access false

# Enable versioning
az storage account blob-service-properties update \
  --account-name stevatfstatecac \
  --enable-versioning true

# Enable soft delete (90 days)
az storage account blob-service-properties update \
  --account-name stevatfstatecac \
  --enable-delete-retention true \
  --delete-retention-days 90

# Create container
az storage container create \
  --name tfstate \
  --account-name stevatfstatecac \
  --auth-mode login

# Grant RBAC roles to engineers
az role assignment create \
  --assignee <user-object-id> \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-eva-tfstate-cac/providers/Microsoft.Storage/storageAccounts/stevatfstatecac

# Grant RBAC role to GitHub Actions service principal
az role assignment create \
  --assignee <sp-object-id> \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-eva-tfstate-cac/providers/Microsoft.Storage/storageAccounts/stevatfstatecac

    State File Naming Convention¶

    • dev: eva-dev.terraform.tfstate
    • stg: eva-stg.terraform.tfstate
    • prd: eva-prd.terraform.tfstate

    Terraform Initialization¶

    cd env/dev
terraform init
# Output:
# Initializing the backend...
# Successfully configured the backend "azurerm"!

    State Locking Behavior¶

    Lock Acquisition¶

    1. User runs terraform apply
    2. Terraform acquires blob lease (20-minute timeout)
    3. State file locked with lease ID (prevents concurrent writes)
    4. Apply operation modifies state
    5. Lease released on completion

    Lock Conflict¶

    Error: Error acquiring the state lock
│ Error message: ConditionalCheckFailedException: The conditional check failed
│ Lock Info:
│   ID:        abc123-def456-ghi789
│   Path:      tfstate/eva-dev.terraform.tfstate
│   Operation: OperationTypeApply
│   Who:        github-actions@runner-1
│   Version:   1.9.0
│   Created:   2025-11-20 14:32:15.123456 +0000 UTC

    Resolution: 

    # Wait for lease expiration (20 minutes) OR
terraform force-unlock abc123-def456-ghi789

    Security Best Practices¶

    Access Control¶

    • Use Azure AD RBAC roles (never use Storage Account access keys)
    • Grant least privilege: Storage Blob Data Contributor for engineers, Storage Blob Data Reader for auditors
    • Rotate RBAC assignments quarterly (audit access logs)

    Encryption¶

    • State encrypted at rest with Microsoft-managed keys (AES-256)
    • Consider customer-managed keys (Azure Key Vault) for stricter compliance
    • TLS 1.2+ enforced for all blob operations

    Audit Trail¶

    • Enable Storage Analytics logging (track all state file access)
    • Send logs to Log Analytics workspace for alerting
    • Alert on:
    • State file deletion attempts
    • Access from unexpected IPs
    • Failed authentication attempts

    Backup & Recovery¶

    Automatic Versioning¶

    Azure Storage versioning creates snapshot on every state update: 

    # List state versions
az storage blob list \
  --container-name tfstate \
  --account-name stevatfstatecac \
  --include v \
  --auth-mode login

# Restore previous version
az storage blob copy start \
  --source-container tfstate \
  --source-blob eva-dev.terraform.tfstate \
  --source-version-id <version-id> \
  --destination-container tfstate \
  --destination-blob eva-dev.terraform.tfstate \
  --account-name stevatfstatecac \
  --auth-mode login

    Manual Backup¶

    # Pull current state to local file
terraform state pull > backup-$(date +%Y%m%d-%H%M%S).tfstate

# Push state from backup (use with caution)
terraform state push backup-20251120-143015.tfstate

    Disaster Recovery Scenarios¶

    Scenario 1: State File Corrupted¶

    1. Stop all Terraform operations
    2. List state versions: az storage blob list --include v
    3. Restore last known good version (see Automatic Versioning above)
    4. Test with terraform plan (should show no changes)
    5. Document incident in runbook

    Scenario 2: State File Deleted¶

    1. Check soft delete retention: az storage blob undelete
    2. Restore deleted blob (90-day retention window)
    3. If beyond retention: restore from manual backup
    4. Last resort: terraform import all resources (high effort)

    Scenario 3: State Lock Stuck¶

    1. Identify lock holder from error message
    2. Verify no active Terraform operations
    3. Force unlock: terraform force-unlock <lock-id>
    4. Investigate cause (crashed CI job, network timeout)

    Migration Path (Future)¶

    If team scales >10 engineers or requires collaboration features:

    Option 1: Terraform Cloud¶

    1. Export state: terraform state pull > state-export.tfstate
    2. Create Terraform Cloud workspace
    3. Configure backend: 
      terraform {
  cloud {
    organization = "eva-platform"
    workspaces {
      name = "eva-dev"
    }
  }
}
    4. Migrate state: terraform init -migrate-state

    Option 2: Azure Container Instance (Terraform Enterprise)¶

    1. Deploy self-hosted Terraform Enterprise on ACI
    2. Retain Azure Storage backend (data residency)
    3. Gain collaboration UI without SaaS dependency
