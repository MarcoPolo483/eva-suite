    EVA Orchestrator - Project Journal¶

    Date: November 24, 2025
    Session: 20251124-063033 (Active)
    Project: EVA 2.0 Demo
    Task: Implementing ChatGPT Plan - GitHub Actions + propose-patch mode

    Session Summary¶

    This session focused on implementing and debugging the GitHub Actions workflow for the EVA Orchestrator's new propose-patch mode, which integrates OpenAI API to generate code proposals from YAML plan files.

    Work Completed¶

    1. GitHub Actions Workflow Debugging (Multiple Iterations)¶

    Issue 1: Incorrect working-directory¶

    Problem: Workflow had working-directory: ./eva-orchestrator but the repo root IS eva-orchestrator
    Fix: Removed working-directory directives from all steps
    Commit: 42f6144 - "fix: remove incorrect working-directory from workflow - repo IS eva-orchestrator"

    Issue 2: Environment variable scope¶

    Problem: EVA_OPENAI_API_KEY was set at workflow level instead of job level
    Fix: Moved env: block from workflow level to job level
    Added: Debug step to verify key presence without exposing value
    Commit: 5dd53c2 - "feat: move env to job level and add EVA_OPENAI_API_KEY presence check"

    Issue 3: Missing PYTHONPATH¶

    Problem: ModuleNotFoundError: No module named 'eva_suite_orchestrator'
    Root Cause: Package lives under src/ but Python didn't know to look there
    Fix: Added PYTHONPATH: src to job-level env block
    Commit: a685519 - "fix: add PYTHONPATH=src to workflow env for package imports"

    Final Workflow Configuration: 

    jobs:
  propose_patch:
    runs-on: ubuntu-latest
    env:
      PYTHONPATH: src
      EVA_OPENAI_API_KEY: ${{ secrets.EVA_OPENAI_API_KEY }}
    steps:
      - Checkout repository
      - Set up Python 3.11
      - Install dependencies (pip install -r requirements.txt)
      - Check EVA_OPENAI_API_KEY presence (debug step)
      - Run propose-patch mode
      - Upload proposals as artifact

    2. CLI Implementation - propose-patch Mode¶

    File: src/eva_suite_orchestrator/cli.py

    Implementation: - Extended argparse to accept 'propose-patch' in --mode choices - Requires --task parameter (exits with error if missing) - Builds prompt using existing build_task_prompt(plan, task) - Calls call_openai(prompt) to get AI-generated proposal - Creates out/proposals/ directory structure - Generates timestamped filename: {plan-id}__{task-id}__{YYYYMMDD_HHMMSS}.md - Writes response to disk with UTF-8 encoding - Prints success message with file path

    Commit: fe427e8 - "feat: implement propose-patch mode in CLI with OpenAI integration"

    3. OpenAI Agent Enhancements¶

    File: src/eva_suite_orchestrator/openai_agent.py

    Changes Made:¶

    A. Custom Exception Classes: 

    class OpenAIConfigError(RuntimeError):
    """Raised when OpenAI API key configuration is missing or invalid."""

class OpenAIRateLimitError(RuntimeError):
    """Raised when the OpenAI API returns a 429 rate limit error."""

class OpenAIAPIError(RuntimeError):
    """Raised for other non-successful OpenAI HTTP responses."""

    B. Dual API Key Support: - Now checks EVA_OPENAI_API_KEY first, falls back to OPENAI_API_KEY - Raises OpenAIConfigError (not generic ValueError) when missing

    C. Retry Logic with Exponential Backoff: - Max retries: 3 attempts - Base backoff: 5 seconds (doubles each retry: 5s → 10s → 20s) - 429 Rate Limit Handling: - Attempts < max: logs retry message, sleeps, retries - Final attempt: raises OpenAIRateLimitError with friendly message - Other HTTP errors: Raises OpenAIAPIError with status code - Network errors: Retries with backoff, raises OpenAIAPIError if all fail - Success path: Returns choices[0].message.content as before

    D. Updated CLI Error Handling: 

    try:
    response = call_openai(prompt)
except OpenAIConfigError as e:
    print(f"[ERROR] OpenAI configuration error: {e}")
    sys.exit(1)
except OpenAIRateLimitError as e:
    print(f"[WARN] Rate limit reached while generating proposal. No file written.")
    print(f"[WARN] {e}")
    sys.exit(1)
except OpenAIAPIError as e:
    print(f"[ERROR] OpenAI API call failed: {e}")
    sys.exit(1)

    Commit: 5d940bc - "feat: add retry logic with exponential backoff and friendly rate limit handling"

    5. Rate Limit Intelligence & Offline Mode¶

    Files: src/eva_suite_orchestrator/openai_agent.py, src/eva_suite_orchestrator/cli.py

    A. Rate Limit Debug Helper¶

    Added _debug_rate_limit(model: str, response: requests.Response) function that: - Inspects OpenAI response headers on 429 errors - Prints diagnostic information: - Requests: limit, remaining, reset time - Tokens: limit, remaining, reset time - Provides intelligent hints: - "Likely hitting REQUESTS PER MINUTE (RPM) limit" when remaining_requests == 0 - "Likely hitting TOKENS PER MINUTE (TPM) limit" when remaining_tokens == 0 - "Could be daily quota or mixed limits" for other cases - Called automatically when status_code == 429

    Output Example: 

    ========== RATE LIMIT DEBUG ==========
Model: gpt-4.1-mini
Requests: limit=500, remaining=0, reset_in=12s
Tokens:   limit=200000, remaining=150000, reset_in=45s
[RATE] Likely hitting REQUESTS PER MINUTE (RPM) limit.
======================================

    Technical Challenges & Solutions¶

    Challenge 1: GitHub Actions Path Context¶

    Issue: Workflow assumed nested folder structure (./eva-orchestrator/) but repo IS eva-orchestrator
    Learning: GitHub Actions checks out the repo to the root of the runner's workspace
    Solution: Remove all working-directory directives, run everything from repo root

    Challenge 2: Python Module Import in CI¶

    Issue: Local dev had PYTHONPATH configured, CI didn't
    Learning: Always explicitly set PYTHONPATH when package isn't installed via pip
    Solution: Add PYTHONPATH: src to job-level environment variables

    Challenge 3: OpenAI Rate Limits¶

    Issue: Hit 429 error during testing
    Learning: Need graceful degradation and retry logic
    Solution: Exponential backoff (3 retries, 5s → 10s → 20s), friendly error messages

    Challenge 4: Secret Visibility in Logs¶

    Issue: Don't want API keys in workflow logs
    Learning: GitHub automatically masks secrets, but be explicit about not printing them
    Solution: Debug step checks if variable is set without printing value
