    SM Job Runbook¶

    Owner: GitHub Copilot (Scrum Master & QA Master)
    Maintainer: Marco Presta (Product Owner)
    Last Updated: 2025-11-19
    Version: 1.0.0

    Purpose¶

    The SM Job workflow automates Sprint 0 deliverables for the EVA Agile Command:

    1. Board Sync: Synchronize GitHub Project board with repository issues
    2. Sprint Metrics: Generate metrics snapshot (metrics/latest-metrics.json)
    3. Onboarding Pack: Assemble latest checklists and bootstrap instructions
    4. Executive Digest: Produce Markdown/HTML briefing for stakeholders

    Goal: Ensure continuous telemetry and artifact generation during Sprint 0 without manual intervention.

    Triggers¶

    Automatic¶

    • Schedule: Daily at 05:00 UTC (nightly run via cron: 0 5 * * *)

    Manual¶

    • workflow_dispatch: On-demand execution via GitHub Actions UI
    • Navigate to: Actions → SM Job → Run workflow → Run workflow

    Inputs¶

    Secrets (Required)¶

    Secret Purpose Scope Rotation Cadence
    PROJECT_SYNC_TOKEN GitHub PAT for project board API repo, project 90 days
    METRICS_TOKEN GitHub PAT for metrics queries repo (read-only) 90 days
    TEAMS_WEBHOOK_URL Teams/Slack webhook for alerts N/A As needed

    Environment Variables¶

    • GITHUB_RUN_ID: Correlation ID for observability
    • GITHUB_RUN_NUMBER: Sequential run counter
    • GITHUB_ACTOR: User who triggered the workflow

    File Inputs¶

    • docs/sprint-history/YYYY-MM-sprint-XX/sprint-plan.md: Current sprint plan
    • sm-qa-toolbox/project/project-sync.config.template.json: Board sync configuration template
    • docs/sprint-kit/: Source material for onboarding pack

    Outputs¶

    Artifacts (GitHub Actions)¶

    1. metrics-latest: metrics/latest-metrics.json (sprint metrics snapshot)
    2. onboarding-pack-YYYYMMDD-HHMMSSZ.zip: Timestamped onboarding kit
    3. executive-digest-YYYYMMDD-HHMMSSZ.zip: Markdown + HTML digest bundle
    4. sm-job-logs: Transcript logs for all steps

    Committed Files (Auto-commit on success)¶

    • metrics/latest-metrics.json (if changed)
    • reports/executive-digest.md (if changed)
    • reports/executive-digest.html (if changed)
    • docs/onboarding/latest-pack/**/* (if changed)

    Notifications¶

    • Failure Alert: Posted to Teams/Slack webhook within 10 minutes (includes run link, actor, timestamp)

    Execution Flow¶

    graph TD
    A[Trigger: Schedule/Manual] --> B[Checkout Repository]
    B --> C[Validate Secrets]
    C --> D[Run Pester Tests]
    D --> E[Sync Project Board]
    E --> F[Generate Sprint Metrics]
    F --> G[Generate Onboarding Pack]
    G --> H[Generate Executive Digest]
    H --> I[Upload Artifacts]
    I --> J[Auto-commit Changes]
    J --> K{Success?}
    K -->|Yes| L[Complete]
    K -->|No| M[Send Failure Alert]
    M --> N[Upload Logs]

    Estimated Duration: 3-5 minutes (typical)
    SLA: Complete within 10 minutes or alert stakeholders

    Failure Modes & Remediation¶

    1. Secret Validation Failure¶

    Symptom: Workflow fails at "Validate required secrets" step
    Cause: Missing or expired PROJECT_SYNC_TOKEN or METRICS_TOKEN
    Remediation: 1. Navigate to: Settings → Secrets and variables → Actions 2. Verify secrets are present and not expired 3. Regenerate PAT if expired (GitHub → Settings → Developer settings → Personal access tokens) 4. Update secret value in repository settings 5. Re-run workflow

    Prevention: Set calendar reminder for 90-day rotation

    2. Pester Test Failure¶

    Symptom: Workflow fails at "Run SM toolbox unit tests" step
    Cause: Regression in generate-sprint-metrics.ps1 or generate-executive-digest.ps1
    Remediation: 1. Review test output in workflow logs 2. Fix failing script locally: pwsh sm-qa-toolbox/sprint-metrics/tests/generate-sprint-metrics.Tests.ps1 3. Commit fix, push to main 4. Next scheduled run will succeed

    Prevention: Run tests locally before pushing changes to SM scripts

    3. Board Sync Failure¶

    Symptom: Workflow fails at "Synchronize project board" step
    Cause: GitHub API rate limit, network timeout, or malformed project config
    Remediation: 1. Check GitHub API rate limit: curl -H "Authorization: token $PROJECT_SYNC_TOKEN" https://api.github.com/rate_limit 2. If rate-limited, wait for reset (check X-RateLimit-Reset header) 3. If timeout, re-run workflow (transient network issue) 4. If config issue, validate project-sync.config.json schema

    Prevention: Spread workflow runs across time zones to avoid rate limit clustering

    4. Metrics Generation Failure¶

    Symptom: Workflow fails at "Generate sprint metrics JSON" step
    Cause: Malformed sprint plan, missing sprint directory, or parsing error
    Remediation: 1. Validate sprint plan exists: docs/sprint-history/YYYY-MM-sprint-XX/sprint-plan.md 2. Check sprint plan formatting matches expected structure (see LESSON-001 for schema) 3. Run generator locally: pwsh sm-qa-toolbox/sprint-metrics/generate-sprint-metrics.ps1 4. Fix sprint plan or script, commit, push

    Prevention: Use sprint plan template to ensure consistent formatting

    5. Auto-commit Failure¶

    Symptom: Workflow succeeds but metrics not committed (artifact uploaded only)
    Cause: Git push conflict (parallel commit raced), or no changes detected
    Remediation: 1. Check auto-commit step logs for "nothing to commit" or "non-fast-forward" error 2. If conflict, manually pull latest, merge, push: git pull --rebase; git push 3. If no changes, ignore (metrics unchanged since last run)

    Prevention: Use manage-session.ps1 guardrail for parallel work

    6. Alert Delivery Failure¶

    Symptom: Workflow fails but no Teams/Slack notification received
    Cause: Invalid or expired TEAMS_WEBHOOK_URL
    Remediation: 1. Test webhook manually: curl -X POST -H "Content-Type: application/json" -d '{"text":"Test alert"}' $TEAMS_WEBHOOK_URL 2. If 404 or 401, regenerate webhook in Teams/Slack admin console 3. Update TEAMS_WEBHOOK_URL secret in repository settings

    Prevention: Document webhook regeneration process in team wiki

    Health Checks¶

    Dashboard: SM Job Run History¶

    Location: sm-qa-toolbox/observability/sm-job-dashboard.ps1

    Usage: 

    pwsh sm-qa-toolbox/observability/sm-job-dashboard.ps1

    Output: Last 7 runs with status, duration, artifacts, run links

    Metrics Captured: - Success rate (%) - Average duration (seconds) - Failure reasons (aggregated) - Artifact upload success rate

    Refresh: Daily (automated), on-demand (manual)

    Escalation Paths¶

    Level 1: Self-Service (SM/QA Master)¶

    • Review workflow logs
    • Check runbook failure modes
    • Re-run workflow if transient failure

    Level 2: Product Owner (Marco Presta)¶

    • Persistent failures (>3 consecutive)
    • Secret rotation required
    • Workflow architecture changes needed

    Level 3: Engineering Leadership¶

    • GitHub platform issues
    • Architectural rework required
    • Budget/resource constraints
