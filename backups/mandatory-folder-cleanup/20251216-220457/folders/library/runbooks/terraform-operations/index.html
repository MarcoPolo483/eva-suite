    Terraform Operations Runbook¶

    This runbook provides step-by-step procedures for common Terraform operations in the EVA 2.0 infrastructure project.

    Table of Contents¶

    1. Prerequisites
    2. Local Development Setup
    3. Remote State Access
    4. Plan/Apply Workflow
    5. Rollback Procedures
    6. Troubleshooting

    Prerequisites¶

    Required Tools¶

    Tool Version Installation
    Terraform 1.9.0+ Download
    Azure CLI 2.50.0+ winget install Microsoft.AzureCLI
    Git 2.40.0+ winget install Git.Git
    VS Code Latest winget install Microsoft.VisualStudioCode

    Azure Permissions¶

    You need the following RBAC role assignments:

    • Contributor on Resource Group rg-eva-dev-* (dev environment)
    • Storage Blob Data Contributor on Storage Account stevatfstatecac (Terraform state)
    • Key Vault Administrator on Key Vault kv-eva-dev-001 (manage secrets)

    Verify permissions: 

    # Check your role assignments
az role assignment list --assignee $(az ad signed-in-user show --query id -o tsv) --output table

    Azure Authentication¶

    # Login to Azure
az login

# Set default subscription
az account set --subscription "<subscription-id>"

# Verify logged-in identity
az account show

    Local Development Setup¶

    1. Clone Repository¶

    cd "C:\Users\<your-username>\Documents\_AI Dev\Repos"
git clone https://github.com/MarcoPolo483/eva-infra.git
cd eva-infra

    2. Initialize Terraform¶

    # Initialize dev environment (downloads providers, configures backend)
terraform -chdir=env/dev init

    Expected output: 

    Initializing the backend...
Successfully configured the backend "azurerm"!

Initializing provider plugins...
- Finding hashicorp/azurerm versions matching "~> 4.0"...
- Installing hashicorp/azurerm v4.x.x...

Terraform has been successfully initialized!

    3. Verify Backend Connection¶

    # Pull current state (should show empty or existing resources)
terraform -chdir=env/dev state list

    4. Validate Configuration¶

    # Format all Terraform files
terraform fmt -recursive

# Validate syntax
terraform -chdir=env/dev validate

    Remote State Access¶

    View State¶

    # List all resources in state
terraform -chdir=env/dev state list

# Show details of specific resource
terraform -chdir=env/dev state show azurerm_virtual_network.main

# Pull full state to local file (for inspection)
terraform -chdir=env/dev state pull > state-backup.json

    State Locking¶

    Terraform uses blob lease locking (20-minute timeout). If another user or CI pipeline is running terraform apply, you'll see:

    Error: Error acquiring the state lock
Lock Info:
  ID:        abc123-def456
  Who:       github-actions@runner-1
  Created:   2025-11-20 14:32:15 +0000 UTC

    Actions: 1. Wait: Lease expires after 20 minutes 2. Coordinate: Ask team if anyone is running Terraform 3. Force Unlock (use with caution): 

    terraform -chdir=env/dev force-unlock abc123-def456

    Plan/Apply Workflow¶

    Standard Workflow¶

    # 1. Create feature branch
git checkout -b feat/add-cosmos-module

# 2. Make changes to Terraform code
# Edit modules/cosmos-db/main.tf, env/dev/main.tf

# 3. Format and validate
terraform fmt -recursive
terraform -chdir=env/dev validate

# 4. Preview changes (dry run)
terraform -chdir=env/dev plan

# 5. Review plan output carefully
# Look for:
# - Resource additions (+)
# - Resource modifications (~)
# - Resource deletions (-)
# - Unexpected changes (review carefully)

# 6. Save plan to file (optional, for manual apply)
terraform -chdir=env/dev plan -out=tfplan

# 7. Apply changes
terraform -chdir=env/dev apply tfplan
# OR (auto-approve, use with caution)
terraform -chdir=env/dev apply -auto-approve

# 8. Commit changes
git add .
git commit -m "Add Cosmos DB module"
git push origin feat/add-cosmos-module

# 9. Create PR, review CI plan output, merge

    Targeted Apply (Use Sparingly)¶

    Apply changes to specific resources only:

    # Plan for specific resource
terraform -chdir=env/dev plan -target=module.networking.azurerm_virtual_network.main

# Apply to specific resource
terraform -chdir=env/dev apply -target=module.networking.azurerm_virtual_network.main

    Warning: Targeted applies can cause drift. Use only for debugging or emergency fixes.

    Destroy Resources¶

    # Destroy specific resource
terraform -chdir=env/dev destroy -target=module.cosmos_db

# Destroy entire environment (use with extreme caution)
terraform -chdir=env/dev destroy

    Warning: Destruction is permanent. Soft-delete enabled for Key Vault (90 days), but data loss is immediate for Cosmos DB.

    Rollback Procedures¶

    Scenario 1: Bad Apply (Non-Breaking)¶

    Example: Applied wrong NSG rule, need to revert.

    Steps: 1. Revert Terraform code to previous commit: 

    git revert HEAD
git push origin main
    2. Let CI/CD pipeline auto-apply reverted config: - Monitor GitHub Actions workflow run - Verify resources restored to previous state

    Recovery Time: 5-10 minutes (CI pipeline + apply)

    Scenario 2: Bad Apply (Breaking)¶

    Example: Deleted Key Vault by accident, apps can't access secrets.

    Steps: 1. Restore from state version (if state corrupted): 

    # List state versions
az storage blob list \
  --container-name tfstate \
  --account-name stevatfstatecac \
  --include v \
  --auth-mode login

# Restore previous version
az storage blob copy start \
  --source-container tfstate \
  --source-blob eva-dev.terraform.tfstate \
  --source-version-id <previous-version-id> \
  --destination-container tfstate \
  --destination-blob eva-dev.terraform.tfstate \
  --account-name stevatfstatecac \
  --auth-mode login

    1. Recover deleted Key Vault (soft delete enabled): 

      # List deleted Key Vaults
az keyvault list-deleted --resource-type vault

# Recover Key Vault
az keyvault recover --name kv-eva-dev-001 --location canadacentral

    2. Re-import resource to state (if state out of sync): 

      terraform -chdir=env/dev import \
  module.key_vault.azurerm_key_vault.main \
  /subscriptions/<subscription-id>/resourceGroups/rg-eva-dev-net/providers/Microsoft.KeyVault/vaults/kv-eva-dev-001

    Recovery Time: 10-30 minutes (depends on resource type)

    Scenario 3: State File Corrupted¶

    Example: Manual edit broke state, terraform plan fails.

    Steps: 1. Restore from automatic versioning: 

    # Download current state (corrupted)
terraform -chdir=env/dev state pull > state-corrupted.json

# List versions
az storage blob list \
  --container-name tfstate \
  --account-name stevatfstatecac \
  --include v \
  --auth-mode login

# Restore last known good version
az storage blob copy start \
  --source-version-id <good-version-id> \
  # ... (same as above)

    1. Verify restoration: 
      terraform -chdir=env/dev state list
terraform -chdir=env/dev plan
# Should show no changes

    Recovery Time: 5-15 minutes

    Troubleshooting¶

    1. State Lock Conflicts¶

    Symptom: 

    Error: Error acquiring the state lock
Lock Info:
  ID:        abc123-def456
  Who:       github-actions@runner-1
  Created:   2025-11-20 14:32:15 +0000 UTC

    Causes: - Another user running terraform apply - CI/CD pipeline job in progress - Crashed Terraform process (lock not released) - Network timeout during apply

    Resolution: 1. Check team chat: Is anyone running Terraform? 2. Check GitHub Actions: Is a workflow running? 3. Wait: Lease expires after 20 minutes 4. Force unlock (last resort): 

    terraform -chdir=env/dev force-unlock abc123-def456

    Prevention: - Coordinate with team before running apply - Use CI/CD pipeline for all production applies - Monitor workflow runs in GitHub Actions

    2. Provider Version Mismatches¶

    Symptom: 

    Error: Failed to query available provider packages
 Could not retrieve the list of available versions for provider hashicorp/azurerm

    Causes: - Outdated provider version constraint - Network issues downloading provider - Corrupted Terraform cache

    Resolution: 

    # Clear Terraform plugin cache
Remove-Item -Recurse -Force $env:APPDATA\terraform.d\plugins

# Reinitialize with latest providers
terraform -chdir=env/dev init -upgrade

# Verify provider versions
terraform -chdir=env/dev version

    Prevention: - Pin provider versions in main.tf: 

    terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}

    3. RBAC Permission Errors¶

    Symptom: 

    Error: authorization failed
 Status=403 Code="AuthorizationFailed"
 Message="The client does not have authorization to perform action"

    Causes: - Missing RBAC role assignment - Role assignment not yet propagated (5-minute delay) - Wrong scope (resource vs. resource group vs. subscription) - Identity not authenticated

    Resolution: 1. Verify role assignments: 

    az role assignment list \
  --assignee $(az ad signed-in-user show --query id -o tsv) \
  --scope <resource-id> \
  --output table

    1. Grant required role (if missing): 

      az role assignment create \
  --assignee <user-object-id> \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-eva-dev-net

    2. Wait for propagation: RBAC changes take 5-10 minutes

    Prevention: - Document required RBAC roles in onboarding guide - Use Azure Policy to auto-assign roles for new team members - Test RBAC with az role assignment list before running Terraform

    4. Resource Quota Limits¶

    Symptom: 

    Error: creating/updating Resource
 Code="QuotaExceeded"
 Message="Operation could not be completed as it results in exceeding approved quota"

    Causes: - Subscription quota limit reached (e.g., 10 VNets per region) - Region capacity exhausted (rare) - Temporary throttling (API rate limit)

    Resolution: 1. Check current quotas: 

    az vm list-usage --location canadacentral --output table
az network vnet list-usage --location canadacentral --output table

    1. Request quota increase:
    2. Navigate to Azure Portal Subscriptions Usage + quotas
    3. Select quota type (e.g., "Virtual Networks")
    4. Click "Request Increase"

    5. Provide justification and new limit

    6. Alternative: Deploy to different region or use smaller SKUs

    Prevention: - Monitor quota usage monthly - Request increases proactively based on growth projections - Use Azure Policy to prevent resource sprawl

    5. Terraform Init Failures¶

    Symptom: 

    Error: Failed to get existing workspaces: storage: service returned error
 StatusCode=403, ErrorCode=AuthenticationFailed

    Causes: - Not authenticated to Azure (az login required) - Missing Storage Blob Data Contributor role - Incorrect backend configuration in backend.tf - Storage account doesn't exist

    Resolution: 1. Authenticate: 

    az login
az account set --subscription "<subscription-id>"

    1. Verify backend config: 

      cat env/dev/backend.tf
# Should show:
# resource_group_name  = "rg-eva-tfstate-cac"
# storage_account_name = "stevatfstatecac"
# container_name       = "tfstate"
# key                  = "eva-dev.terraform.tfstate"
# use_azuread_auth     = true

    2. Check Storage Account exists: 

      az storage account show --name stevatfstatecac --resource-group rg-eva-tfstate-cac

    3. Verify RBAC role: 

      az role assignment list \
  --assignee $(az ad signed-in-user show --query id -o tsv) \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-eva-tfstate-cac/providers/Microsoft.Storage/storageAccounts/stevatfstatecac

    Prevention: - Document backend setup in Terraform State Management - Automate backend provisioning with Azure CLI script - Test backend access before onboarding new team members

    6. Plan Shows Unexpected Changes¶

    Symptom: 

    Terraform will perform the following actions:
  # module.networking.azurerm_virtual_network.main will be replaced
  - destroy
  + create

    Causes: - Manual changes made in Azure Portal (drift) - Provider version upgrade changed resource behavior - Incorrect Terraform code (e.g., wrong resource ID) - State file out of sync with reality

    Resolution: 1. Check for drift: 

    terraform -chdir=env/dev refresh
terraform -chdir=env/dev plan

    1. Review Azure Portal: Did someone make manual changes?

    2. Investigate provider changes:

    3. Check azurerm provider changelog

    4. Look for breaking changes in recent versions

    5. Reimport resource (if state out of sync): 

      terraform -chdir=env/dev import \
  module.networking.azurerm_virtual_network.main \
  /subscriptions/<subscription-id>/resourceGroups/rg-eva-dev-net/providers/Microsoft.Network/virtualNetworks/vnet-eva-dev-001

    Prevention: - Use Azure Policy to deny manual changes (drift prevention) - Enable drift detection in CI/CD pipeline (terraform plan on schedule) - Document "Do NOT make manual changes in Portal" in team guidelines

    Best Practices¶

    1. Always run terraform plan before apply: Review changes carefully
    2. Use CI/CD pipeline for production: Avoid local applies to stg/prd
    3. Never force-unlock without coordination: Risk of state corruption
    4. Backup state before risky operations: terraform state pull > backup.json
    5. Test changes in dev first: Validate before promoting to stg/prd
    6. Document all manual fixes: Update this runbook with new scenarios
    7. Monitor state file size: Alert if >10 MB (indicates excessive resources)
    8. Rotate RBAC roles quarterly: Remove unused permissions
