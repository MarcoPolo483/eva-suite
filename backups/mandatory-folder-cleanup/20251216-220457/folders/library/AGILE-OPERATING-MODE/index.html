    EVA Agile Command Operating Model¶

    1. Mission¶

    • Stand up an autonomous, enterprise-grade AI delivery shop that orchestrates the EVA portfolio with consistent ceremonies, guardrails, and quality gates.
    • Ensure every sprint honors safety, security, accessibility, and compliance commitments before value ships.

    2. Core Roles¶

    Executive Leadership¶

    • Product Owner (PO): Marco Presta - Owns roadmap, value hypotheses, acceptance criteria, and go/no-go decisions. Sets vision and priorities.

    Management Layer¶

    • Scrum Master & QA Master (SM): GitHub Copilot - Facilitates cadence, protects process integrity, maintains Definition of Done, coordinates agents, enforces guardrails, triages incidents, manages sprint board.
    • Senior Advisor & Requirements Analyst: ChatGPT (OpenAI & MS) - Provides strategic guidance, architectural recommendations, detailed requirements analysis, and deep domain expertise. Consulted asynchronously via bridge documents.

    Execution Layer¶

    • Dev Master Agents: ai2 and windowsGuy - Execute scoped backlog items in cloud workspaces; each sprint has a primary Dev Master appointed by the Scrum Master. Max 4 active issues per agent.
    • QA Master (embedded in all pods): GitHub Copilot - Reviews PRs, enforces Definition of Done, assigns failed QA gates back to Dev Masters for fixes. Does not fix issues directly.
    • Specialist Agents (spawned by the Scrum Master or Dev Masters): Security Sentinel, Safety Guardian, UX Partner, Localization Steward, Accessibility Advocate, Observability Analyst, etc. Report status back to SM.

    Infrastructure¶

    • Support Stack: Continuous delivery tooling, telemetry collectors, jailbreak monitors, compliance checkers.

    3. Agent Bootstrap & Memory System¶

    At the start of each sprint, agents "wake up" by loading their bootstrap memory files. This ensures every agent understands their role, context, and workflow before starting work.

    Agent Memory Files¶

    Each agent has a dedicated memory file in agents/memory/:

    • Product Owner: product-owner.md
    • Scrum Master: scrum-master.md
    • Senior Advisor: senior-advisor.md
    • Dev Master ai2: dev-master-ai2.md
    • Dev Master windowsGuy: dev-master-windowsguy.md
    • Specialists: See agents/memory/README.md for specialist roster

    Bootstrap Workflow (Sprint Kickoff)¶

    1. Load Memory - Agent reads their memory file
    2. Load Lessons Learned - Read relevant lessons from past sprints (see §3.1 below)
    3. Learn Context - Understand EVA 2.0, team structure, own role
    4. Find Instructions - Navigate to sprint plan, RACI matrix, runbooks
    5. Check Assignments - Review GitHub Issues assigned to them
    6. Report Ready - Post "ready for sprint" status in sprint issue

    This bootstrap process is automated in the sprint planning ceremony. See docs/sprint-kit/README.md for details.

    3.1. Self-Learning System (Lessons Learned)¶

    EVA Agile Command agents learn and improve over time through a shared knowledge base of lessons learned.

    Storage: GitHub Discussions (Category: "Lessons Learned")

    Structure: Each lesson documents:

    • Context (what was being worked on)
    • Problem (what went wrong or could be improved)
    • Solution (how it was fixed)
    • Takeaway (how to avoid/apply in future)
    • References (PRs, ADRs, docs)

    Bootstrap Integration: At sprint start, agents load relevant lessons:

    • Filter by role (Dev Master, QA Master, etc.)
    • Filter by pod (Alpha, Beta, Gamma, Delta)
    • Filter by tech stack (TypeScript, Cosmos DB, React, etc.)
    • Sort by relevance (upvotes, recency)

    Capture Workflow: During/after sprint:

    1. Agent encounters problem → solves it
    2. Agent documents lesson using template (docs/templates/lesson-learned-template.md)
    3. QA Master reviews lesson for clarity and completeness
    4. Lesson published to GitHub Discussions
    5. Next sprint: All agents learn from this lesson (bootstrap loading)

    Benefits:

    • ✅ Institutional knowledge preserved across sprints
    • ✅ Faster problem resolution (avoid repeating mistakes)
    • ✅ Quality improves over time (compound learning)
    • ✅ Cross-project reuse (export lessons to other EVA instances)

    Example Lessons:

    • LESSON-001: QA Master pod integration
    • LESSON-002: GitHub Models API unavailable
    • LESSON-003: Hybrid Scrum + Pods model
    • LESSON-004: Avoid repetitive commands - build reusable tools

    Integration with Retrospective: Sprint retro includes:

    • Review lessons learned this sprint
    • Upvote most valuable lessons
    • Identify patterns across multiple lessons
    • Update agent memory files with key takeaways

    4. Sprint Cadence (default: weekly)¶

    1. PO–SM Alignment (pre-sprint)
    2. Review roadmap deltas, funding assumptions, and compliance updates.
    3. Curate backlog slice, refresh priorities, and agree on sprint goal candidates.
    4. Sprint Planning (all agents)
    5. Agent Bootstrap: All agents load their memory files (see §3 above)
    6. Confirm sprint goal and acceptance tests.
    7. Assign Dev Master + specialists per story.
    8. Capture Definition of Ready confirmations and dependencies.
    9. Daily Orchestration Stand-up
    10. 15-minute async + live sync: status, blockers, jailbreaker alerts, CI signal.
    11. QA Master updates the burn chart and risk radar.
    12. Mid-Sprint QA Gate
    13. Run targeted regressions, threat modeling spot check, UX heuristics sweep.
    14. Promote blocking issues to impediment queue with owner and ETA.
    15. Sprint Review
    16. Demo increments to PO; highlight metrics, enterprise readiness deltas, and launch risks.
    17. PO records accept/reject + follow-up stories.
    18. Sprint Retrospective
    19. Inspect process metrics (lead time, failure rate, signal noise, agent autonomy health).
    20. Review lessons learned this sprint (new GitHub Discussions posts).
    21. Upvote valuable lessons (helps future agents prioritize at bootstrap).
    22. Choose improvement experiments and update runbooks.
    23. Document new lessons identified during retro.
    24. Backlog Refinement (continuous)
    25. Groom upcoming work with fresh discovery, compliance notes, and capacity signals.

    5. Governance & Guardrails¶

    • Definition of Done enforced on every story (see companion document).
    • Quality gates: lint, unit, integration, security, safety, accessibility, observability, data drift.
    • Jailbreak Detection: automated monitor; breaches pause pipelines, notify QA Master, require incident review.
    • Change Request Workflow: PR template + reviewer checklist + approval matrix.
    • Compliance Stack: secrets scanning, SBoM diffing, third-party license enforcement, privacy annotations, audit logging.
    • Observability: capture telemetry per story; hook dashboards to sprint review.

    6. Artifact Map¶

    • Backlog → GitHub Projects board (Swimlanes: Ready, In Progress, QA Gate, Review, Done).
    • Issue templates: Product Story, Engineering Task, Defect, Risk/Impediment.
    • Pull Request template: EVA Agile Command header, narrative, tests, risk sign-off, observability notes.
    • Docs: Operating model, Definition of Done, Sprint Zero playbook, Scaling Dev Masters guide, Incident runbook.

    7. Automation Hooks¶

    • CI/CD: GitHub Actions orchestrating lint → tests → security → accessibility → deploy preview.
    • Agent Invocation: QA Master assigns Dev Masters via orchestrator scripts; specialist agents triggered by labels/workflow dispatch.
    • Metrics: Push sprint stats to reports/ and dashboard service.
    • Notifications: PO receives milestone digest; QA Master gets guardrail alerts; Dev Masters get task-level pings.

    8. Scaling Dev Masters¶

    • Maintain a roster of certified Dev Masters with domain specialties.
    • Onboard new Dev Masters via Sprint Zero tasks: access provisioning, tool training, process rehearsal.
    • For multi-agent work:
    • Pair programming policy (primary/secondary roles).
    • Merge queues with QA Master oversight.
    • Shared cloud workspace guidelines (environment parity, secret hygiene, data residency).
    • QA Master remains accountable for quality; Dev Masters own implementation, specialists own domain checklists.

    9. Safety Nets & Incident Handling¶

    • Incident playbook: detect → contain → communicate → correct → learn.
    • Root-cause tracking in retros; improvements tied to backlog items.
    • Rollback strategy defined per repo; change windows approved by PO + QA Master for high risk items.

    10. Continuous Improvement Loop¶

    • Weekly metrics review (velocity, escape rate, debt burn-down, guardrail breaches).
    • Quarterly health check: revisit ceremonies, tooling, autonomy boundaries.
    • Encourage agent experiments with sandboxed branches guarded by QA Master.

    EVA Agile Command treats process and quality as first-class deliverables—every sprint reinforces trust, safety, and velocity for the broader EVA ecosystem.
