    Blockchain Decision: DECLINED (Dec 4, 2025)¶

    TL;DR¶

    Decision: Use hash-chain audit pattern instead of blockchain for EVA audit trails.

    Rationale: EVA has trusted operator + central governance. Blockchain adds cost/complexity without solving multi-party trust problem EVA doesn't have.

    Alternative: EVA Ledger Pattern (blockchain-inspired, not blockchain)

    Decision Context¶

    Question¶

    Should EVA use blockchain for tamper-evident audit trails?

    Participants¶

    • Marco Presta (Product Owner)
    • Senior Advisor (ChatGPT) - provided deep analysis
    • Scrum Master (GitHub Copilot) - documented decision

    Date¶

    December 4, 2025 (5:45 AM - morning analysis session)

    Analysis Summary¶

    Where Blockchain COULD Help (Theoretically)¶

    Blockchain is good at: 1. Tamper-evident logs - entries can't be silently altered 2. Multi-party trust - orgs that don't trust each other share ledger 3. Provenance - cryptographic evidence of state at time T 4. Dispute resolution - "who changed what, when, under which authority"

    For EVA, interesting ones are (1) tamper-evident audit and (3) provenance.

    EVA's Reality (Why Blockchain Doesn't Fit)¶

    EVA Context: - Trusted operator: AICOE + IITB - Central governance: Single department (ESDC) deployment - Protected B environment: Azure Canada, Entra ID, APIM - Problems are: Good audit design, linking events across services, RA/IDE evidence, explaining "who did what" to auditors

    NOT the classic "zero trust between parties" blockchain use case.

    Pros vs Cons¶

    Pros of Blockchain for EVA: - ✅ Cryptographically strong "can't rewrite history" - ✅ Looks nice in governance decks ("immutable ledger") - ✅ Could serve as cross-department proof (IF EVA becomes GC-wide)

    Cons of Blockchain for EVA: - ❌ Extra infrastructure (nodes, consensus, upgrades, skills) - ❌ Performance/latency hit (seconds vs <100ms for high-volume logs) - ❌ Awkward with Protected B + privacy (what to store on-chain? Only hashes?) - ❌ Auditors ask "Why blockchain? Why not WORM logs, signed hashes, Key Vault?"

    Verdict: Cost, complexity, governance overhead outweigh benefit for ESDC.

    Selected Alternative: EVA Ledger Pattern¶

    Concept¶

    Get "blockchain-grade" audit without blockchain complexity.

    Pattern 1 - Hash-Linked Audit Log¶

    Implementation: "EVA Ledger" table with: - id (GUID) - timestamp_utc - actor (user/service) - project_space, feature, agent - action_type - payload_hash - previous_hash - current_hash = SHA-256(previous_hash + payload_hash + timestamp_utc + ...)

    Benefits: - Append-only, hash-linked chain per project_space - If someone edits/deletes row, chain breaks - Periodically seal head hash in Key Vault - Same "blockchain property" auditors care about (tamper evidence) - Just SQL + hashes, no miners/nodes, 100% under EVA control

    Pattern 2 - Periodic Merkle Snapshot / Notarization¶

    Implementation: Every X minutes/hours: 1. Build Merkle tree of recent audit entries 2. Store root hash in tamper-resistant location: - Azure Key Vault - Separate subscription - Secured log file

    Benefits: - Proof log contents haven't changed since last snapshot - Simple, infrastructure-friendly

    Pattern 3 - Siebel-Style History Tables + Hashing¶

    Siebel Inspiration: Every core table had history/audit table with row_id, created, created_by, last_updated

    EVA Application: For each critical entity (project_space, RAG corpus, UI strings, agents, policies): - Base table - History table (insert-only) with hash of record state

    Benefits: - Familiar Siebel-like model - Cryptographic integrity layered on top - Easy to reason about, very "enterprise"

    Siebel Data Model vs Blockchain¶

    What Siebel Already Gave Us (90% of What We Need)¶

    Siebel Strengths: - Strong normalization + metadata - Row-id, created, updated, who, when - Config-driven behavior

    Combined with Modern Security: - Append-only history tables - Hash-chaining or Merkle-style snapshots - Proper RBAC and logging

    Result: Governed ledger without blockchain buzzword.

    Blockchain Doesn't Add Functionally: - ❌ Distributed trust (don't need inside AICOE) - ❌ Change management pain (really don't need)

    When Blockchain MIGHT Make Sense (Future)¶

    Revisit blockchain IF:

    1. EVA becomes pan-GC platform with:
    2. Multiple departments operating own infrastructure

    3. Some not fully trusting each other operationally

    4. Publicly verifiable provenance needed:

    5. "This citizen-facing decision explanation is provably derived from X legal sources at time T"

    Then consider: Permissioned ledger (Hyperledger Fabric-style) for: - Provenance proofs (hashes only, no PB data) - Cross-department "event notarization"

    Even then: Start with hash-chained audit + Merkle roots, only move to real blockchain if absolutely necessary.

    Recommended Implementation for EVA (Next Steps)¶

    Step 1: Define EVA Ledger Schema¶

    Inspired by Siebel + blockchain: - Append-only, hash-linked audit tables per project_space

    Step 2: Add Merkle Snapshot Jobs¶

    Daily/hourly jobs: - Store snapshot hashes in Key Vault or separate storage

    Step 3: Expose Read-Only Audit Views¶

    In EVA LiveOps: - RA/IDE, security, program owners can inspect trails

    Step 4: Document as "Tamper-Evident Audit"¶

    In ATO/RA/IDE packages: - "Immutable, tamper-evident logs" story without blockchain complexity

    Benefits Summary¶

    ✅ Story people want to hear: "Immutable, tamper-evident logs"
    ✅ Without complexity: No blockchain infrastructure
    ✅ GC compliance: Meets ITSG-33/NIST 800-53 requirements
    ✅ Cost effective: Azure native (Cosmos DB + Key Vault + Blob legal hold)
    ✅ Performance: <100ms latency (vs seconds for blockchain)
    ✅ Skills: Existing team capabilities (no blockchain expertise needed)
    ✅ Protected B friendly: All data stays in Azure Canada
