**Governance:** Existing ITSG-33 controls and TBS GenAI guidance.

---

## Level 1 ‚Äî Tool-Bound Assistants

**Description**

Systems where an AI model can call **a small, fully whitelisted set of tools**.

**Characteristics**

- Tools strictly registered and approved (e.g., ‚Äúsearch jurisprudence‚Äù, ‚Äúretrieve case summary‚Äù).
- No loops, self-modifying prompts, or dynamic tool discovery.
- Every tool invocation is logged and traceable. :contentReference[oaicite:5]{index=5}

**Examples**

- A jurisprudence Q&A assistant invoking a single ‚Äúsearch decisions‚Äù API.
- A CPP-D assistant that:
  - fetches a file,
  - summarizes it,
  - returns a draft explanation to a human.

**Risk Level:** Low ‚Üí Moderate  
**Controls:**  
- Tool whitelisting  
- APIM mediation  
- RBAC on each tool  
- Basic monitoring and logging  

---

## Level 2 ‚Äî Workflow Agents (Governed Autonomy)

**Description**

Agents can:

- break tasks into multiple steps,
- call several tools in sequence,
- propose a plan,
- and generate outputs that may influence decisions ‚Äî **but always under human approval and guardrails**. 

**Characteristics**

- Multi-step planning (e.g., ‚Äúingest ‚Üí retrieve ‚Üí compare ‚Üí summarize ‚Üí recommend‚Äù).
- Outputs remain **advisory** until a human validates them.
- High-impact actions require **explicit human authorization**.
- Every step, tool call, and choice is logged and replayable.

**Examples**

- Jurisprudence case analyzer that:
  1. pulls all relevant cases,
  2. groups them by topic,
  3. summarizes precedents,
  4. proposes a reasoning outline for the adjudicator.
- CPP-D triage agent that:
  1. parses incoming documents,
  2. extracts metadata,
  3. suggests next steps to a decision-maker.

**Risk Level:** Moderate ‚Üí High  
**Controls:**

- All Level 1 controls  
- Human-in-the-loop checkpoints  
- Workflow traceability and replay  
- Policy guardrails (no PII, no cross-domain data leakage)  
- Kill-switch at orchestrator level  
- Continuous monitoring and periodic re-certification  

---

## Level 3 ‚Äî Multi-Agent Systems (Sandbox-Only)

**Description**

Multiple agents collaborating under an orchestrator, for example:

- Planner, Researcher, Reviewer, Builder agents interacting,
- but **in a sandboxed environment only**, never directly connected to production systems.

**Characteristics**

- Complex behaviour emerges from coordination across agents.
- Used only in lab environments or tightly controlled pilots.
- Strong isolation, strict data boundaries, enhanced logging. 

**Risk Level:** High  
**Controls:**

- All Level 2 controls  
- Network and environment isolation  
- Explicit sandbox designation  
- Restricted datasets and no production data  

**Production Rule**

> **Level 3 agents are not permitted to directly operate on live production systems.**

---

## Level 4 ‚Äî Fully Autonomous Systems (Prohibited)

**Description**

Systems that:

- self-modify policies or prompts,
- discover new tools autonomously,
- access new data sources without approval,
- execute high-impact actions without human oversight.

**GC Policy Position**

> **Level 4 autonomy is prohibited.  
> No GC system may operate at this level.** :contentReference[oaicite:8]{index=8}

---

## Summary Table

| Level | Name                    | Allowed in Prod? | Typical Context                |
|------:|-------------------------|------------------|--------------------------------|
| 0     | Non-Agentic AI         | Yes              | Basic RAG, summarization       |
| 1     | Tool-Bound Assistants   | Yes              | Jurisprudence search, CPP-D    |
| 2     | Workflow Agents         | Yes (with ATO)   | Jurisprudence, KM, CPP-D       |
| 3     | Multi-Agent Systems     | Sandbox only     | Labs, PoCs, internal research  |
| 4     | Fully Autonomous Agents | No               | Explicitly prohibited          |

    # üîí Security & Governance Controls for Agentic AI

This section translates the GC Agentic Framework into **concrete control requirements** mapped to:

- **ITSG-33**
- **NIST AI RMF 1.0**
- **ISED Voluntary AI Code**
- **TBS Guide on Generative AI** 

---

## 1. Core Control Themes

All agentic systems MUST address:

- **Access Control (AC)** ‚Äì who can trigger agents and tools  
- **Tool Security (SC, SA, SI)** ‚Äì how tools are restricted and secured  
- **Logging & Audit (AU)** ‚Äì full trace of each action and decision  
- **Human Oversight (AT, PE)** ‚Äì explicit human checkpoints  
- **Boundary Protection (SC)** ‚Äì network and data domain controls  
- **Testing & Evaluation (CA, RA)** ‚Äì red-teaming, bias, robustness tests  
- **Incident Response & Kill-Switches (IR, CP)** ‚Äì clear stoppage and rollback paths  

---

## 2. Control Requirements by Level

### Level 0 ‚Äî Non-Agentic AI

- Apply existing **ITSG-33** and **TBS GenAI** controls.
- Ensure:
  - prompt logging,
  - transparency statement (‚ÄúYou are interacting with AI‚Äù),
  - content filters where appropriate.

---

### Level 1 ‚Äî Tool-Bound Assistants

In addition to Level 0:

- Define a **Tool Registry**:
  - each tool has an owner,
  - documented purpose,
  - security classification. :contentReference[oaicite:10]{index=10}
- Enforce **APIM mediation** for all tool calls.
- Implement **RBAC** at tool level (who can invoke what).
- Log:
  - user identity,
  - tool name,
  - parameters (subject to privacy),
  - timestamp,
  - outcome.
- Conduct targeted **red-teaming** before production release.

---

### Level 2 ‚Äî Workflow Agents

Add the following on top of Level 1:

- **Workflow Traceability**:
  - store the plan (steps),
  - store actual executed steps,
  - make workflows replayable.
- **Human-in-the-loop**:
  - define ‚Äúhigh-impact‚Äù actions,
  - require explicit user approval for these actions.
- **Policy Guardrails**:
  - prohibitions on PII and sensitive data in prompts,
  - domain-specific constraints (e.g., no legal advice to the public).
- **Kill-Switches** built at:
  - agent level,
  - tool level,
  - orchestrator/platform level. 
- **Periodic Re-Certification**:
  - review workflows,
  - re-run security and bias tests,
  - update documentation (system card, data card, risk register).

---

### Level 3 ‚Äî Multi-Agent Systems (Sandbox Only)

Add additional controls:

- Network **segmentation and isolation** for sandbox.
- No connection to production systems or live classified datasets.
- Strict data minimization ‚Äî test or synthetic data where possible.
- Enhanced monitoring and logging, including:
  - inter-agent messages,
  - tool calls,
  - abnormal behaviour alerts.

---

    5. operations.md ‚Äî Operational Patterns & Guardrails¶

    # ‚öôÔ∏è Operational Patterns & Guardrails

This section describes **how agents must operate in practice** to remain safe, governable, and auditable. It is drawn from the vNext framework and the ‚ÄúAgentic Fear Resolver‚Äù narrative. 

---

## 1. All Agents Run Through a Secure Orchestrator

- Agents MUST execute through a **GC-approved orchestrator**, such as **EVA**.
- The orchestrator is responsible for:
  - enforcing policy guardrails,
  - routing tool calls via APIM,
  - injecting audit metadata (correlation IDs, request IDs),
  - applying safety filters and content boundaries,
  - enforcing rate limits and cost ceilings.

---

## 2. Tool Whitelisting (No ‚ÄúPlugin Wild West‚Äù)

- All tools must be:
  - explicitly registered in a **Tool Registry**,
  - security-reviewed,
  - tied to an owner and a business purpose.
- Agents **cannot discover tools dynamically**.
- Decommissioned tools must be removed from the registry and blocked at APIM.

---

## 3. Human-in-the-loop by Design

For Level 2 workflows:

- Define **which actions are advisory** vs **which require approval**.
- Provide:
  - clear UI indicators when an agent suggests vs when it attempts to act,
  - explicit ‚ÄúApprove / Reject / Edit‚Äù controls for users.
- Capture approver identity and decision in logs.

---

## 4. Dry-Run / ‚ÄúExplain First‚Äù Mode

Before executing a workflow, agents should:

1. Present a **draft plan** (steps, tools, expected outputs).  
2. Allow the user to:
   - confirm the plan,
   - edit or constrain it,
   - cancel it. :contentReference[oaicite:14]{index=14}  

This feature is strongly recommended for Jurisprudence, CPP-D, and KM use cases.

---

## 5. Reversibility & Rollback

- Where agents may trigger downstream changes, systems must:
  - support rollback (technical or procedural),
  - keep sufficient logs to understand what happened,
  - document how to revert actions in runbooks.

**No ‚Äúirreversible‚Äù actions may be delegated to an agent without a clear human-controlled rollback strategy.**

---

## 6. Monitoring, Telemetry, and Alerts

Operational monitoring must cover:

- volume and type of tool calls,
- anomaly detection (e.g., sudden spikes, unusual tool combinations),
- error rates and failure modes,
- policy violations (e.g., disallowed content, blocked data).

Alerts must be routed to:

- platform operators (EVA),
- security operations when thresholds are exceeded,
- product owners/domain stewards when domain-specific rules are hit. 

---

## 7. Kill-Switch Patterns

**Three kill-switch levels:**

1. **Agent kill-switch** ‚Äì disables a particular agent configuration.
2. **Tool kill-switch** ‚Äì blocks a specific tool everywhere.
3. **Platform kill-switch** ‚Äì pauses all agentic workflows in EVA.

Runbooks MUST specify:

- who can trigger which kill-switch,
- under what conditions,
- how to recover safely.

---

## 8. Change Management for Agentic Workflows

Any change to:

- prompts,
- policies,
- tool registry,
- underlying models,

must go through a **change management process** consistent with:

- ATO,
- RA/IDE,
- ESDC change management policies.

Document changes, version them, and record approvals.
