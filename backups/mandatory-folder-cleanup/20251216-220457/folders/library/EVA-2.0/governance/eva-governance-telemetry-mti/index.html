    EVA Governance Telemetry & Trust Index – Technical Design v0.1¶

    Author: Marco Presta (Product Owner) + GitHub Copilot (Scrum Master) Date: 2025-11-17 Status: Draft for Review Related: Machine Trust Index (MTI) briefing, NuEnergy partnership, EDFP governance platform

    1. Purpose¶

    This document defines how EVA exposes governance-ready telemetry so that any AI governance platform (e.g., NuEnergy's Machine Trust Platform, EDFP, or a future internal solution) can:

    • Compute a Machine Trust Index (MTI)-style score (0–100) per AI solution
    • Track category scores over time (Compliance, Performance, Security, etc.)
    • Drive dashboards, alerts, and deployment gates for AI solutions built on EVA

    Key principle: EVA itself does not compute governance scores. EVA emits structured events and metadata that allow external governance services to do so in a consistent, vendor-neutral way.

    2. Conceptual Model¶

    2.1 Entities¶

    Solution¶

    Represents a governed AI-enabled solution (e.g., AssistMe, EI Jurisprudence, EVA DA project).

    Fields:

    • solution_id (GUID)
    • name (string)
    • owner_org (string)
    • owner_contact (string/email)
    • classification_level (enum: ULL, PB, PC, Secret)
    • status (enum: planned, dev, test, prod, retired)

    SolutionVersion¶

    Represents a specific deployed version of a solution.

    Fields:

    • version_id (GUID)
    • solution_id (FK → Solution)
    • version_label (string, e.g., v1.3.2 or date tag)
    • environment (enum: dev, test, prod)
    • model_family (string, e.g., gpt-4.1, o3-mini, custom model id)
    • rag_index_ids (array of strings)
    • deployment_config_ref (URI to IaC / APIM config)

    TrustAssessment¶

    Represents one MTI-style assessment for a given solution version at a point in time.

    Fields:

    • assessment_id (GUID)
    • solution_version_id (FK → SolutionVersion)
    • assessed_at (timestamp)
    • assessed_by (enum/string: NuEnergy, EDFP, Manual, etc.)
    • mt_index (0–100)
    • notes (string)
    • source_report_uri (URI to full report)

    CategoryScore¶

    Represents a per-category score under a given assessment.

    Fields:

    • category_score_id (GUID)
    • assessment_id (FK → TrustAssessment)
    • category_name (enum):
    • Compliance
    • Performance
    • Security
    • Transparency
    • DataGovernance
    • Ethics
    • Privacy
    • Fairness
    • Explainability
    • Bias
    • score (0–100)
    • weight (0–1)
    • threshold_min (optional; minimum acceptable score)
    • key_findings (short text)
    • key_actions (short text)

    3. Governance Telemetry Contract¶

    EVA emits governance-relevant events into a central log/event pipeline. Any governance platform can subscribe to these events to compute MTI-style scores.

    3.1 Event Envelope¶

    All events share a common envelope:

    {
  "solution_id": "assistme",
  "solution_version": "1.3.2",
  "environment": "prod",
  "timestamp": "2025-11-17T16:34:00Z",
  "event_type": "eval_run",
  "actor": "system",
  "payload": {}
}

    Notes:

    • solution_id, solution_version, and environment map events back to Solution/SolutionVersion.
    • event_type drives downstream scoring logic.
    • payload is type-specific.

    Storage: Events are written to a central store (e.g., Azure Log Analytics, Event Hub → DFP/Cosmos).

    4. Category-Specific Telemetry¶

    Below are minimal payloads EVA should emit to support MTI-style scoring per category.

    Event type: user_feedback_submitted

    Metrics derived:

    • % negative feedback by category (content quality signals)
    • Time-to-resolution for negative feedback (responsiveness)

    5. Trust Scoring Service (External to EVA)¶

    A separate Trust Scoring Service (in EDFP or a governance platform) is responsible for:

    1. Ingesting events from EVA (via Event Hub, Kafka, etc.).
    2. Aggregating metrics per SolutionVersion over a time window.
    3. Computing per-category scores and the overall MTI.
    4. Writing TrustAssessment and CategoryScore records for dashboards and APIs.

    5.1 Scoring Algorithm (Pseudocode)¶

    def calculate_mti(solution_version_id, time_window):
    events = load_events(solution_version_id, time_window)

    # Compute category scores
    perf_score = score_performance(events)
    sec_score = score_security(events)
    priv_score = score_privacy(events)
    trans_score = score_transparency(events)
    data_gov_score = score_data_governance(events)
    ethics_score = score_ethics(events)
    fairness_score = score_fairness(events)
    explain_score = score_explainability(events)
    bias_score = score_bias(events)
    comp_score = score_compliance(events)

    # Build scores map
    scores = {
        "Compliance": comp_score,
        "Performance": perf_score,
        "Security": sec_score,
        "Transparency": trans_score,
        "DataGovernance": data_gov_score,
        "Ethics": ethics_score,
        "Privacy": priv_score,
        "Fairness": fairness_score,
        "Explainability": explain_score,
        "Bias": bias_score
    }

    # Apply weights (configurable per department/policy)
    weighted_scores = []
    total_weight = 0
    for cat, score in scores.items():
        weight = get_weight(cat)
        weighted_scores.append(score * weight)
        total_weight += weight

    # Compute overall MTI
    mti = sum(weighted_scores) / total_weight

    # Persist assessment
    persist_assessment(solution_version_id, mti, scores)

    return mti, scores

    5.2 Weights and Thresholds¶

    Weights and thresholds are stored in a configuration table (DB or config file) and can be aligned with:

    • Departmental AI policy
    • NIST AI RMF mappings
    • NuEnergy/MTI defaults
    • Project-specific risk profiles

    Example weights (sum to 1.0):

    Category Weight Rationale
    Compliance 0.15 High priority for government AI
    Performance 0.12 User satisfaction depends on accuracy
    Security 0.15 Critical for classified data
    Transparency 0.10 Required for explainability
    DataGovernance 0.10 Data quality = answer quality
    Ethics 0.08 Responsible AI principles
    Privacy 0.12 PIPEDA compliance
    Fairness 0.08 Demographic parity
    Explainability 0.05 User trust
    Bias 0.05 Language parity (en-US, fr-FR)

    Example thresholds (minimum acceptable scores):

    Category Minimum Action if Below
    Security 60 Block prod deployment
    Privacy 70 Block prod deployment
    Compliance 50 Require manual approval
    Performance 40 Warning, not blocking

    6. EVA Integration Points¶

    6.1 EVA Foundation¶

    Provide a shared governance telemetry library for EVA apps and services.

    Location: eva-core/src/governance/telemetry.ts

    Usage:

    import { GovernanceTelemetry } from "@eva/core/governance";

const telemetry = new GovernanceTelemetry({
  solutionId: "assistme",
  solutionVersion: "1.3.2",
  environment: "prod",
});

// Emit performance eval event
telemetry.emitEvalRun({
  evalSuiteId: "assistme-core-2025-09",
  metrics: {
    accuracy: 0.82,
    hallucination_rate: 0.07,
    latency_p95_ms: 2200,
  },
  sampleSize: 500,
});

// Emit user feedback event
telemetry.emitUserFeedback({
  feedbackId: "feedback-uuid",
  messageId: "msg-uuid",
  tenantId: "tenant-a",
  rating: "negative",
  categories: ["incorrect"],
});

    Ensure all EVA-backed solutions (EVA DA, AssistMe, Jurisprudence, etc.) use the same event envelope and category-specific payloads.

    6.2 EVA LiveOps / Project Registry¶

    Expose governance data per project:

    Location: eva-ops/src/dashboard/governance-tab.tsx

    Features:

    • Chart of MTI over time (last 90 days)
    • Table of category scores with red/yellow/green indicators
    • Last assessment source (NuEnergy, EDFP, Manual)
    • Top 3 open technical actions (e.g., "Add bias tests on disability axis")
    • Drill-down to raw events (for auditing)

    Data source: Read-only from external Trust Scoring Service API.

    Example API call:

    const assessment = await governanceApi.getLatestAssessment(solutionVersionId);

console.log(assessment.mt_index); // 72
console.log(assessment.category_scores);
// [
//   { category: 'Security', score: 65, threshold: 60, status: 'pass' },
//   { category: 'Privacy', score: 68, threshold: 70, status: 'fail' },
//   ...
// ]

    6.3 CI/CD Gates¶

    Integrate trust scores into deployment decisions:

    Location: .github/workflows/deploy-prod.yml (or Azure DevOps pipeline)

    Logic:

    - name: Check Governance Scores
  run: |
    # Call Trust Scoring Service API
    MTI=$(curl -s "$GOVERNANCE_API/assessments/$SOLUTION_VERSION_ID/latest" | jq '.mt_index')
    SECURITY=$(curl -s "$GOVERNANCE_API/assessments/$SOLUTION_VERSION_ID/latest" | jq '.category_scores[] | select(.category=="Security") | .score')
    PRIVACY=$(curl -s "$GOVERNANCE_API/assessments/$SOLUTION_VERSION_ID/latest" | jq '.category_scores[] | select(.category=="Privacy") | .score')

    # Block if thresholds not met
    if [ "$SECURITY" -lt 60 ]; then
      echo "❌ Security score ($SECURITY) below threshold (60). Blocking deployment."
      exit 1
    fi

    if [ "$PRIVACY" -lt 70 ]; then
      echo "❌ Privacy score ($PRIVACY) below threshold (70). Blocking deployment."
      exit 1
    fi

    echo "✅ Governance checks passed. MTI: $MTI"

    Actions:

    • Block promotion to prod if:
    • Security < 60
    • Privacy < 70
    • Compliance < 50
    • Require manual approval if:
    • Overall MTI < 50 (but solution is UAT-only)
    • Any category in "yellow" zone (below ideal, above minimum)

    9. Summary¶

    EVA's role: Emit consistent, rich governance telemetry.

    External governance platforms: Compute MTI-style scores using this telemetry.

    EVA LiveOps and CI/CD: Consume those scores to support:

    • Operational visibility
    • Risk-based approvals
    • Alignment with departmental AI governance strategy.

    Key differentiator: Vendor-neutral, standards-aligned (NIST AI RMF), extensible to any governance platform.
