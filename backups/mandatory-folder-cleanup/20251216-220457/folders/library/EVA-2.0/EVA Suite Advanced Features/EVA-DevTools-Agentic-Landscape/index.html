    EVA DevTools – Agentic Landscape & Feature Catalog (v2)¶

    Version: 0.5 Audience: Scrum Master, Agile Crew & Humans Date: 2025-11-23

    Product name: EVA DevTools
    Motto: Developer’s Tools Box that can be created and maintained at AI speed.

    EVA DevTools is EVA’s internal delivery optimization layer for developers, BAs, architects, testers, and governance teams. It takes the ABRG / EI agentic concept (Orchestrator, Super, Utility agents) and reimplements it inside EVA 2.0 for software delivery, modernization, and governance.

    1. Agent Taxonomy¶

    We use a three-tier agent design:

    1. Orchestrator Agents
    2. Primary entry points for humans.
    3. Understand user intent (“what is this person trying to do?”).
    4. Route to the right Super Agents.

    5. Manage multi-step flows, tool selection, and user interaction.

    6. Super Agents

    7. Own a domain or capability area (e.g., backlog quality, testing, legacy analysis).
    8. Orchestrate multiple Utility Agents and tools to accomplish complex tasks.

    9. Provide opinionated workflows and domain-specific logic.

    10. Utility Agents

    11. Small, reusable building blocks.
    12. Provide focused capabilities (parse code, call ADO, generate tests, summarize logs, etc.).
    13. Reused across Orchestrators and Super Agents.

    Runtime model

    All agents are implemented as EVA 2.0 “tools”:

    • APIs exposed through APIM.
    • RAG/LLM workflows where needed.
    • Callable by:
    • EVA DevTools web UI,
    • EVA Chat (e.g., /devtools … commands),
    • Future agent frameworks (MCP, Semantic Kernel, etc.).

    2. EVA DevTools – Orchestrator Agent¶

    2.1 EVA DevTools Orchestrator¶

    Purpose
    Be the main entry point for developers and delivery teams.

    Primary users

    • Developers and Tech Leads
    • Testers and QA
    • Business Analysts
    • Scrum Masters and Release Train Engineers
    • Architects and Governance teams

    Example intents

    • “Clean up my ADO backlog for the next PI.”
    • “Generate test scripts for this feature branch.”
    • “Summarize mainframe module X and generate documentation.”
    • “Create a first-cut process map for this feature.”
    • “Give me a sprint health report for the last 3 sprints.”

    Responsibilities

    • Classify incoming requests into capability areas:
    • Backlog, Testing, Legacy, Governance, Planning, Skills, Observability.
    • Select the appropriate Super Agent(s).
    • Orchestrate multi-step flows (for example: backlog → impact analysis → release planning).
    • Maintain conversational state so users can iterate (“now refine this”, “show me the risks only”).

    EVA Backlog Guardian ADO Feature Consistency Checker, Dependency Checker POs, BAs, Dev Leads
    EVA Process Mapper Product Process Map Generator BAs, Architects
    EVA Test Architect Test Suite Optimizer, Automated Test Script Creator Testers, Devs
    EVA Legacy Analyzer Mainframe Code Documentation Generator, Cúram Capability Checker Modernization teams, Architects

    Below are concise specs for each Super Agent.

    3.1 EVA Backlog Guardian¶

    Goal
    Improve backlog quality, consistency, and traceability.

    Key functions

    • Analyze ADO boards for:
    • INVEST-quality of user stories.
    • Missing or weak acceptance criteria.
    • Inconsistent naming, tags, or taxonomy.
    • Detect dependencies between features/stories:
    • Cross-team impacts.
    • Potential blockers and sequencing issues.
    • Suggest improvements:
    • Merge or split stories.
    • Add missing links and tags.
    • Standardize wording using agreed vocabularies.

    Inputs

    • ADO project / team / query (e.g., “current PI backlog”).
    • Optional domain-specific vocab/ontology (EI, BDM, EVA).

    Outputs

    • Backlog quality report (score and findings).
    • Suggested changes (per-item recommendations).
    • Optional machine-readable export (JSON/CSV) for bulk updates.

    3.2 EVA Process Mapper¶

    Goal
    Accelerate creation of product process maps, user journeys, and system flows.

    Key functions

    • Generate candidate process maps from:
    • Natural language descriptions.
    • Existing user stories and features.
    • Auto-structure flows:
    • Steps, decisions, outcomes, actors, systems.
    • Produce dual artifacts:
    • Diagram spec (Mermaid/BPMN JSON or similar).
    • Human-readable narrative for documentation.

    Inputs

    • Free-form text describing the process.
    • Optional links to ADO items or policy documents.

    Outputs

    • Machine-readable process map definition.
    • Text description (for Confluence, SharePoint, EVA DA, etc.).
    • Suggested swimlanes (actors, systems) and boundaries.

    3.3 EVA Test Architect¶

    Goal
    Improve test coverage and reduce redundant / low-value tests.

    Key functions

    • Analyze existing test suites:
    • Redundancy and overlaps.
    • Coverage gaps vs. code or requirements.
    • Generate new tests from:
    • Legacy or modern code.
    • User stories and acceptance criteria.
    • API contracts (OpenAPI/Swagger).
    • Propose optimized suite execution:
    • Prioritization by risk, frequency, and impact.

    Inputs

    • Test repository / configuration.
    • Target module or service.
    • Reference stories / features.

    Outputs

    • Test coverage summary & heatmap.
    • Suggested new test cases (ready-to-adapt code stubs).
    • Optimized regression suite definition.

    3.4 EVA Legacy Analyzer¶

    Goal
    Demystify legacy systems and help teams modernize them.

    Key functions

    • Parse legacy code and configuration:
    • Programs, data structures, key entry points.
    • Produce:
    • Plain-language system summaries.
    • Rule descriptions and decision paths.
    • Assist modernization:
    • Identify candidate user stories.
    • Suggest “quick-win” extractions into modern services.

    Inputs

    • Source code repositories or extracts.
    • Optional platform metadata (mainframe, Cúram, etc.).

    Outputs

    • High-level architecture overview.
    • Module-level documentation for critical components.
    • Candidate backlog items for modernization.

    3.5 EVA Impact Analyzer¶

    Goal
    Assess the impact of new or changed policies/requirements on systems and flows.

    Key functions

    • Compare:
    • Old vs. new text (policy, regulations, requirements).
    • Identify:
    • Affected rules, services, UIs, data structures, test cases.
    • Suggest:
    • Change tasks.
    • Impacted documentation and artefacts.

    Inputs

    • Old and new policy/requirement documents (or marked sections).
    • Optional mapping to existing systems or code.

    Outputs

    • Impact analysis report.
    • List of impacted components, flows, and tests.
    • Suggested change backlog.

    3.6 EVA Release Navigator¶

    Goal
    Support release planning, scope selection, and rough estimation.

    Key functions

    • Use WSJF-like logic and historical data to:
    • Prioritize features.
    • Propose sensible release groupings.
    • Provide:
    • Rough effort and timeline estimates.
    • Highlight risks and dependencies.

    Inputs

    • Feature / story list with basic value & complexity estimates.
    • Historical velocity data (where available).
    • Known dependencies and constraints.

    Outputs

    • Suggested release plan (sprint / PI breakdown).
    • WSJF-style priority scores.
    • Risk and dependency summary.

    3.7 EVA Agile Coach¶

    Goal
    Improve team execution and Agile maturity.

    Key functions

    • Analyze past sprints:
    • Velocity trends, spillover, defects.
    • Work type breakdown (features vs bugs vs tech debt).
    • Suggest:
    • Topics for retrospectives.
    • Improvements to estimation, WIP limits, and flow.
    • Provide:
    • Just-in-time guidance on ceremonies (“how to run a better retro”).

    Inputs

    • ADO sprint data and work item history.
    • Optional qualitative feedback from teams.

    Outputs

    • Sprint / PI health reports.
    • Actionable improvement suggestions.
    • Retro prompts and checklists.

    3.8 EVA Skills Mentor¶

    Goal
    Align team skills with program needs and surface learning paths.

    Key functions

    • Analyze:
    • Skills inventories, CVs, or self-assessments.
    • Project / program requirements and tech stacks.
    • Identify:
    • Skill gaps and emerging needs.
    • Opportunities for cross-training and pairing.
    • Suggest:
    • Individual and team learning plans.

    Inputs

    • Skills profile data (anonymized where appropriate).
    • Role / position requirements for projects.

    Outputs

    • Skill gap analysis by person/team.
    • Prioritized upskilling / reskilling recommendations.

    3.9 EVA Governance Reporter¶

    Goal
    Automate governance and reporting deliverables.

    Key functions

    • Generate:
    • RAID logs and summaries.
    • Steering deck content (key metrics, decisions, risks).
    • AI governance reports (tool usage, evaluations, incidents).
    • Pull from:
    • ADO, EVA telemetry, cost dashboards, logs, etc.
    • Help with compliance:
    • Populate mandatory templates for committees and boards.

    Inputs

    • Time window and project/portfolio scope.
    • Target report type (PMO, AI Governance, DG/ADM deck, etc.).

    Outputs

    • Draft reports in chosen format (Markdown, PPT skeleton, JSON).
    • Data extracts for dashboards (Power BI, etc.).

    3.10 EVA Observability Wizard¶

    Goal
    Enhance observability and streamline production diagnostics.

    Key functions

    • Suggest:
    • Log statements and metrics to add to code (based on patterns and incidents).
    • Analyze:
    • Existing logs for anomalies, error patterns, and root causes.
    • Provide:
    • Natural-language incident narratives.
    • Recommendations for self-healing hooks and alerts.

    Inputs

    • Code snippets or repositories.
    • Log collections and telemetry.

    Outputs

    • PR-ready suggestions for improved logging and metrics.
    • Diagnostic summaries and probable root-cause analysis.

    4. EVA DevTools – Utility Agent Layer¶

    Utility Agents are shared capabilities used by many Super Agents:

    • ADO Connector Agent
      Read/write work items, queries, sprints, tags.

    • Repo & Code Parser Agent
      Build ASTs, find dependencies, detect patterns and smells.

    • Test Framework Agent
      Generate test stubs in Jest/xUnit/PyTest, etc.

    • Policy & Document RAG Agent
      Retrieve and summarize policy, design, and architecture standards.

    • Metrics & Telemetry Agent
      Pull metrics from Application Insights, log stores, and monitoring tools.

    • Diagram Generator Agent
      Produce Mermaid/BPMN specs from structured descriptions.

    • Report Template Agent
      Fill governance/report templates based on structured data.

    These live as generic EVA Core / Matrix services and are composed into higher-level flows by Super Agents.

    5. Integration with EVA 2.0¶

    • EVA DevTools is an EVA Product:
    • UI exposed via DevTools web app and EVA Chat commands.

    • Backend built on EVA Core and EVA Matrix libraries.

    • Agents are exposed as:

    • APIM endpoints for Orchestrator and Super Agents.

    • Tools usable by agent frameworks (MCP, Semantic Kernel, etc.).

    • Governance alignment:

    • Usage is logged with project, team, and tool context.
    • Complies with NIST-flavoured requirements: REQ-GOV, REQ-TRC, REQ-XAI, etc.
    • Supports cost and usage reporting through EVA’s metering services.

    EVA DevTools is EVA’s AI-powered toolbox for delivery optimization—built so that new tools can be created, evolved, and retired at AI speed.
