    2. Rules as Code (RaC) Overview¶

    File: EVA RAC - Rule as Code.txt Length: 341 lines (17 pages) Read Time: 15 minutes

    What It Contains: - RaC Definition: Legislation/policy encoded as machine-readable logic, exposed as API - Current State: OpenFisca Canada instance + APIs (CERB, post-CERB benefits, EI Act sections) - Strategic Framing: RaC = guardrail/single source of truth that LLMs sit on top of - Vision: AI + RaC enables continuous program review, scenario testing, auditability (digital twin of rules)

    EVA Rules Service (New Core Service): Treat RaC as backend capability alongside RAG:

    APIM Endpoints: 1. POST /rules/evaluate - Input: program, ruleSetVersion, variables (client situation) - Output: Eligibility decisions, amounts, reasons, trace metadata

    1. POST /rules/trace
    2. Input: Same as evaluate

    3. Output: Decision tree showing which conditions fired, defaults used

    4. POST /rules/scenario

    5. Input: Baseline + list of "what-if" changes

    6. Output: Comparative results (current vs proposed rules)

    7. GET /rules/metadata

    8. Output: Available programs, rule sets, versions, owners, source (Act/regulation)

    Internal Implementation: - Wrap OpenFisca Canada for EI/CERB experiments - Support other RaC engines (CPP/OAS, GIS, BDM CÃºram rule exports) - Register each rule set as Project Space artifact (versioning, tests, governance)

    EVA DA Integration (Neurosymbolic AI): Example: "Am I eligible for benefit X if I worked 600 hours and was laid off due to shortage of work?"

    Pipeline: 1. RAG retrieves relevant policy/guidance/jurisprudence documents 2. EVA Rules Service evaluates eligibility (machine-readable rules) 3. LLM composes answer: - Decision (from RaC - no hallucination) - Explanation citing law/policy (from RAG) - Optional trace ("because conditions A, B, C were true")

    Result: Less hallucination on rules (decision from RaC), human-friendly explanations (LLM wording)

    OpenFisca-Style Rules Example: 

    # benefit_x_eligibility.yaml
parameters:
  benefit_x:
    min_hours:
      value: 420

variables:

    formula: |
      hours_ok = hours_worked >= 420
      reason_ok = separation_reason in allowed_reasons
      return hours_ok and reason_ok

    Benefits: - Consistency: Same rules across all channels (web, phone, in-person) - Auditability: Every decision traceable to specific rule version - Explainability: Trace shows which conditions fired - Testability: Scenario testing for policy changes (what-if analysis) - Maintainability: Rules versioned, tested, deployed like code

    Use When: - Building benefits eligibility assistants (EI, CPP, OAS, GIS) - Implementing policy change impact analysis - Creating "what-if" simulators for citizens/caseworkers - Ensuring LLM answers comply with legislation (guardrails)

    Business Value: - Compliance: 100% rule fidelity (no LLM hallucination on eligibility) - Speed: 95% faster policy impact analysis (automated scenario testing) - Transparency: Full audit trail for regulatory reviews - Reusability: One rule set, multiple channels (EVA Chat, EVA DA, legacy systems)

    3. RaC Implementation Example¶

    File: EVA RaC - RaC-example-benefit-x.md Length: 555 lines (28 pages) Read Time: 30 minutes

    What It Contains: Complete worked example showing Rules as Code implementation from YAML rules through API calls to LLM integration.

    Scenario: User asks: "Am I eligible for benefit X if I worked 600 hours and was laid off due to shortage of work?"

    Business Rules (Benefit X - simplified): - Eligible if: - Worked â‰¥420 insurable hours in last 52 weeks AND - Separation reason in: ["shortage_of_work", "layoff", "business_closure"] - If eligible: Flat weekly amount of $500

    Machine-Readable Rules (OpenFisca YAML): 

    # benefit_x_eligibility.yaml

entities:
  person:
    plural: persons

parameters:
  benefit_x:
    min_hours:
      description: Minimum insurable hours required in qualifying period
      values:
        2025-01-01:
          value: 420
    allowed_separation_reasons:
      description: Reasons for job separation that preserve eligibility
      values:
        2025-01-01:
          value:
            - "shortage_of_work"
            - "layoff"
            - "business_closure"

variables:
  hours_worked_last_52_weeks:
    entity: person
    label: "Total insurable hours in last 52 weeks"
    value_type: float

  separation_reason:
    entity: person
    label: "Reason for job separation from last employer"
    value_type: string

  benefit_x_eligible:
    entity: person
    label: "Eligibility for Benefit X"
    value_type: bool
    definition_period: month
    formula: |
      hours_ok = hours_worked_last_52_weeks >= parameters.benefit_x.min_hours
      reason_ok = separation_reason in parameters.benefit_x.allowed_separation_reasons
      return hours_ok and reason_ok

  benefit_x_amount:
    entity: person
    label: "Weekly Benefit X amount"
    value_type: float
    definition_period: month
    formula: |
      if not benefit_x_eligible:
          return 0.0
      return 500.0

    API Request (to EVA Rules Service): 

    POST /rules/evaluate
{
  "program": "benefit_x",
  "ruleSetVersion": "2025-01-01",
  "variables": {
    "person_1": {
      "hours_worked_last_52_weeks": 600,
      "separation_reason": "shortage_of_work"
    }
  },
  "calculate": ["benefit_x_eligible", "benefit_x_amount"]
}

      "hours_ok": true,

          "shortage_of_work",

      }
    }
  },
  "version": "2025-01-01",
