    CDD: EVA API Gateway & Integration Layer¶

    Last updated: 2025-11-15
    Source: docs/EVA-2.0/ESDC/EVA Backlog/EVA Backlog/CDD EVA API Gateway & Integration Layer.docx
    Status: Backlog
    Priority: HIGH

    Vision¶

    Build enterprise-grade API Gateway as single entry point for all EVA services, providing:

    • Unified authentication and authorization
    • Rate limiting and throttling
    • API versioning and routing
    • Request/response transformation
    • Observability and analytics

    Core Capabilities¶

    1. Authentication & Authorization¶

    • Azure AD B2C integration for citizens
    • Entra ID for employees
    • OAuth 2.0 / OIDC flows
    • JWT token validation
    • Role-based access control (RBAC)
    • Tenant isolation

    2. Traffic Management¶

    • Rate limiting (per user, per tenant, per API)
    • Throttling and queueing
    • Circuit breaker for downstream failures
    • Retry policies with exponential backoff
    • Load balancing across backend services

    3. API Management¶

    • API versioning (v1, v2, v3)
    • Request routing based on headers
    • GraphQL and REST endpoint support
    • WebSocket/SSE for real-time chat
    • OpenAPI/Swagger documentation generation

    4. Security¶

    • TLS 1.2+ enforcement
    • CORS policy management
    • IP allowlisting/blocklisting
    • Request size limits (prevent DoS)
    • Content type validation
    • SQL injection / XSS protection

    5. Observability¶

    • Request/response logging
    • Distributed tracing (OpenTelemetry)
    • Metrics (latency, throughput, errors)
    • Analytics dashboard (Application Insights)
    • Alerting on SLA violations

    Technical Architecture¶

    Option A: Azure API Management (APIM)¶

    Pros:

    • Managed service (no infrastructure management)
    • Built-in developer portal
    • Comprehensive policy engine
    • Native Azure integration
    • Application Insights integration

    Cons:

    • Higher cost (~\(700-\)2800/month)
    • Vendor lock-in
    • Limited customization

    Option B: Kong Gateway (Open-Source)¶

    Pros:

    • Open-source, portable
    • Rich plugin ecosystem
    • Kubernetes-native
    • Lower cost (self-hosted)

    Cons:

    • Infrastructure management overhead
    • Less Azure-native tooling
    • Plugin compatibility testing needed

    Recommendation: Azure APIM for EVA 2.0¶

    Given ESDC's Azure-first strategy and enterprise support requirements, APIM provides best fit.

    Implementation Phases¶

    Phase 1: APIM Foundation¶

    • Provision Azure APIM instance (Developer tier for dev, Standard for prod)
    • Configure VNet integration (secure backend access)
    • Set up Entra ID authentication
    • Define base API policies (CORS, rate limiting)

    Phase 2: API Onboarding¶

    • Onboard eva-api (REST/GraphQL endpoints)
    • Onboard eva-agent (chat completion, tool invocation)
    • Onboard eva-rag (document search, ingestion)
    • Onboard eva-ops (dashboard, metrics)
    • Generate OpenAPI specs for each backend

    Phase 3: Security Hardening¶

    • Implement JWT validation policies
    • Configure tenant-scoped rate limits
    • Add IP allowlisting for admin APIs
    • Enable threat protection (OWASP Top 10)
    • Set up Web Application Firewall (WAF)

    Phase 4: Observability¶

    • Wire Application Insights logging
    • Configure distributed tracing (W3C Trace Context)
    • Build analytics dashboard (latency, errors, top APIs)
    • Set up alerts (5xx errors, P95 latency >500ms)

    Phase 5: Developer Experience¶

    • Enable APIM developer portal
    • Generate API documentation (Swagger UI)
    • Create code samples (Python, Node.js, .NET)
    • Publish versioned SDKs (npm, PyPI, NuGet)

    API Structure¶

    Base URL¶

    • Dev: https://eva-dev.azure-api.net
    • Test: https://eva-test.azure-api.net
    • Prod: https://eva.esdc-edsc.gc.ca

    API Namespaces¶

    • /v1/chat - Chat completion endpoints
    • /v1/agent - Agent orchestration and tools
    • /v1/rag - Document search and ingestion
    • /v1/auth - Authentication and token refresh
    • /v1/ops - Dashboards and metrics
    • /v1/admin - Tenant and user management

    Versioning Strategy¶

    • URI path versioning: /v1/, /v2/
    • Support N-1 versions (e.g., v2 released, v1 still supported)
    • 6-month deprecation notice for old versions

    User Stories¶

    Story 1: Developer API Access¶

    As a backend developer building EVA-powered applications
    I want a single API gateway endpoint with consistent authentication
    So that I don't have to manage multiple service URLs and auth mechanisms

    Acceptance: Can authenticate once via Entra ID and access all EVA APIs through gateway; OpenAPI spec available; code samples provided

    Story 2: Platform Operator Rate Limiting¶

    As a platform operator
    I want configurable rate limits per tenant and per user
    So that no single tenant can monopolize resources and impact other tenants

    Acceptance: Can set tenant-level quotas (req/min); can set user-level quotas; 429 errors returned with Retry-After headers; quota metrics visible in ops dashboard

    Story 3: Compliance Auditor Traceability¶

    As a compliance auditor
    I want complete audit logs of all API requests including who, what, when, from where
    So that I can track data access and investigate security incidents

    Acceptance: Every API request logged with user ID, tenant ID, IP, timestamp, endpoint, response code; logs retained per policy; searchable via Log Analytics

    Story 4: Product Owner API Analytics¶

    As a product owner
    I want analytics on API usage patterns (top endpoints, latency trends, error rates)
    So that I can prioritize improvements and capacity planning

    Acceptance: Application Insights dashboard shows P50/P95/P99 latency, request volume by endpoint, error rates, top consumers; exportable to Power BI

    Story 5: External Partner Integration¶

    As an external partner organization
    I want secure API access with scoped permissions
    So that I can integrate EVA capabilities into my applications without full tenant access

    Acceptance: Can issue API keys with limited scopes; partner requests isolated from internal tenant traffic; usage metered separately

    Functional Requirements¶

    FR-1: Authentication & Authorization¶

    All API requests MUST be authenticated via Entra ID (employees) or Azure AD B2C (citizens) using OAuth 2.0/OIDC flows. Anonymous access is prohibited except for /health and /docs endpoints.

    FR-2: JWT Token Validation¶

    APIM MUST validate JWT tokens on every request, checking signature, expiration, issuer, and audience claims. Invalid tokens return 401 Unauthorized with error details.

    FR-3: Tenant Isolation¶

    APIM MUST extract tenant ID from JWT claims and enforce tenant-scoped access. Cross-tenant requests are blocked with 403 Forbidden.

    FR-4: Rate Limiting (Tenant-Level)¶

    APIM MUST enforce configurable rate limits per tenant (default: 1000 req/min). Exceeded requests return 429 Too Many Requests with Retry-After header.

    FR-5: Rate Limiting (User-Level)¶

    APIM MUST enforce configurable rate limits per user (default: 100 req/min). Exceeded requests return 429 Too Many Requests with Retry-After header.

    FR-6: API Versioning¶

    APIM MUST support multiple API versions via URI path (/v1/, /v2/). Version headers are optional but URI path takes precedence.

    FR-7: Request Routing¶

    APIM MUST route requests to correct backend services based on API namespace (/v1/chat eva-api, /v1/rag eva-rag, etc.).

    FR-8: Circuit Breaker¶

    APIM MUST implement circuit breaker pattern: if backend returns 5xx errors >50% over 1 minute, open circuit for 30 seconds. Return 503 Service Unavailable during open state.

    FR-9: Distributed Tracing¶

    APIM MUST propagate W3C Trace Context headers (traceparent, tracestate) to backend services. Generate correlation ID if not provided.

    FR-10: Response Transformation¶

    APIM MUST transform backend errors into consistent API error format (RFC 7807 Problem Details): {"type": "...", "title": "...", "status": 400, "detail": "...", "instance": "..."}.

    FR-11: Audit Logging¶

    APIM MUST log every request/response to Log Analytics with: timestamp, user ID, tenant ID, endpoint, method, status code, latency, IP address, user agent.

    FR-12: CORS Policy¶

    APIM MUST enforce CORS policy: allow origins from approved domains only (configurable per environment). Return proper CORS headers for preflight requests.

    FR-13: TLS Enforcement¶

    APIM MUST enforce TLS 1.2+ for all connections. HTTP requests are redirected to HTTPS. Self-signed certificates are rejected.

    FR-14: OpenAPI Documentation¶

    APIM MUST auto-generate OpenAPI 3.0 specification from backend services. Developer portal displays interactive Swagger UI.

    FR-15: Health Checks¶

    APIM MUST expose /health endpoint (unauthenticated) returning gateway status and backend service health. Used by Azure Front Door health probes.

    Non-Functional Requirements (Expanded)¶

    Performance¶

    • Gateway Overhead: P95 latency <50ms (excluding backend processing time)
    • Throughput: Support 10,000 requests/second per APIM unit
    • Cold Start: APIM policies execute in <10ms
    • Caching: Support response caching with configurable TTL (reduces backend load)

    Security¶

    • Authentication: 100% of API requests authenticated (except /health, /docs)
    • Authorization: RBAC enforced at gateway level (no backend bypass possible)
    • Secrets Management: API keys and certificates stored in Azure Key Vault
    • Threat Protection: OWASP Top 10 mitigations via WAF policies
    • Audit: All API access logged with immutable audit trail

    Accessibility¶

    • Developer Portal: WCAG 2.1 AA compliant (keyboard navigation, screen reader support, color contrast)
    • API Documentation: Accessible HTML format with semantic structure
    • Error Messages: Clear, actionable error descriptions (no cryptic codes)

    Internationalization¶

    • API Documentation: Available in en-US and fr-FR
    • Error Messages: Localized based on Accept-Language header (default: en-US)
    • Developer Portal: UI strings externalized, bilingual navigation

    Observability¶

    • Distributed Tracing: 100% of requests traced via OpenTelemetry
    • Metrics: Request rate, latency (P50/P95/P99), error rate, quota usage
    • Logs: Structured JSON logs with correlation IDs
    • Alerting: Automated alerts for P95 latency >500ms, error rate >1%, quota breaches

    Reliability¶

    • Availability: 99.9% SLA (3-nines target)
    • Failover: Multi-region active-passive deployment (manual failover <1 hour)
    • Disaster Recovery: RPO <1 hour, RTO <4 hours
    • Backup: APIM configuration backed up daily to Blob Storage

    Scalability¶

    • Horizontal Scaling: Auto-scale APIM units based on CPU >70% threshold
    • Rate Limiting: Graceful degradation under load (throttle non-critical APIs first)
    • Caching: Reduce backend load via response caching (configurable per API)

    Acceptance Criteria¶

    AC-1: Authentication Enforcement¶

    All API requests (except /health, /docs) require valid JWT token
    Invalid tokens return 401 with error details
    Expired tokens return 401 with token refresh instructions

    AC-2: Rate Limiting Active¶

    Tenant-level rate limit enforced (1000 req/min default)
    User-level rate limit enforced (100 req/min default)
    429 responses include Retry-After header
    Quota metrics visible in ops dashboard

    AC-3: Audit Logging Complete¶

    Every API request logged to Log Analytics
    Logs include user ID, tenant ID, IP, endpoint, status, latency
    Logs searchable via KQL queries
    Audit trail immutable (write-only, no deletion)

    AC-4: API Versioning Works¶

    /v1/ routes to v1 backends
    /v2/ routes to v2 backends
    Unsupported versions return 404 with deprecation notice

    AC-5: Circuit Breaker Functional¶

    Backend failures trigger circuit breaker after threshold
    503 returned during open circuit state
    Circuit closes after recovery period
    Metrics track circuit breaker state changes

    AC-6: OpenAPI Docs Generated¶

    Developer portal displays interactive Swagger UI
    OpenAPI spec auto-generated from backend services
    Code samples provided (Python, Node.js, .NET)
    Documentation available in en-US and fr-FR

    AC-7: Performance Targets Met¶

    P95 gateway overhead <50ms (measured via distributed tracing)
    No degradation under 10,000 req/s load
    Auto-scaling triggers at 70% CPU

    AC-8: Security Hardened¶

    TLS 1.2+ enforced (TLS 1.0/1.1 rejected)
    CORS policy blocks unauthorized origins
    WAF policies active (OWASP Top 10 mitigations)
    Zero security incidents in penetration testing

    Technical Design Notes¶

    Azure Services & Integration¶

    Azure API Management (APIM): - Tier: Standard for dev/test, Premium for production (multi-region support) - Capacity: Start with 2 units, auto-scale to 10 units max - Networking: Deployed in VNet with private IP (secure backend access) - Custom Domain: eva.esdc-edsc.gc.ca with TLS certificate from Key Vault

    Entra ID Integration: - OAuth 2.0 Provider: Configure Entra ID as OAuth provider in APIM - Scopes: Define API scopes (chat.read, rag.write, admin.manage) - Claims Mapping: Extract tenant ID, user ID, roles from JWT claims

    Application Insights: - Request Telemetry: APIM sends request/response data to App Insights - Custom Metrics: Quota usage, rate limit hits, circuit breaker state - Distributed Tracing: W3C Trace Context propagated to backends

    Azure Key Vault: - TLS Certificates: Store custom domain certificates - API Keys: Store partner API keys (encrypted at rest) - Backend Credentials: Store backend service credentials

    Policy Configuration¶

    Inbound Policies (applied to all APIs): 

    <policies>
    <inbound>
        <!-- Validate JWT -->
        <validate-jwt header-name="Authorization">
            <issuer-signing-keys>...</issuer-signing-keys>
        </validate-jwt>

        <!-- Extract tenant ID from claims -->
        <set-variable name="tenantId" value="@(context.Request.Headers.GetValueOrDefault("Authorization").AsJwt().Claims["tid"])" />

        <!-- Rate limiting -->
        <rate-limit-by-key calls="1000" renewal-period="60" counter-key="@(context.Variables["tenantId"])" />
        <rate-limit-by-key calls="100" renewal-period="60" counter-key="@(context.Request.Headers.GetValueOrDefault("Authorization").AsJwt().Claims["sub"])" />

        <!-- Distributed tracing -->
        <set-header name="traceparent" exists-action="skip">
            <value>@(context.RequestId)</value>
        </set-header>
    </inbound>
</policies>

    Backend Policies: - Circuit Breaker: Open circuit after 50% error rate over 1 min - Retry: 3 retries with exponential backoff (1s, 2s, 4s) - Timeout: 30s timeout for backend calls

    Outbound Policies: - CORS: Add CORS headers based on origin whitelist - Response Transformation: Convert backend errors to RFC 7807 format

    Routing Strategy¶

    API-to-Backend Mapping: - /v1/chat/* https://eva-api.internal.esdc.gc.ca/chat - /v1/agent/* https://eva-agent.internal.esdc.gc.ca/agent - /v1/rag/* https://eva-rag.internal.esdc.gc.ca/rag - /v1/auth/* https://eva-auth.internal.esdc.gc.ca/auth - /v1/ops/* https://eva-ops.internal.esdc.gc.ca/ops

    Version Migration Path: 1. Deploy v2 backends alongside v1 (no breaking changes yet) 2. Release v2 API gateway routes (both v1 and v2 active) 3. Announce v1 deprecation (6-month notice) 4. Monitor v1 usage (track laggards via analytics) 5. Sunset v1 after grace period (return 410 Gone)

    • Requirement: OpenAPI specs from each backend service
    • Timing: Sprint 3-5 (incremental onboarding)
    • Owner: Dev Masters (per service)
    • Risk: Backend teams must publish OpenAPI specs (may need scaffolding support)

    Monitoring (eva-ops)¶

    • Requirement: Application Insights workspace and dashboards
    • Timing: Sprint 1 (parallel with APIM provisioning)
    • Owner: QA Master (observability)

    Networking (Azure)¶

    • Requirement: VNet, subnet, NSG rules for APIM
    • Timing: Sprint 1 (before APIM deployment)
    • Owner: windowsGuy (networking)

    • Requirement: eva.esdc-edsc.gc.ca DNS record and TLS certificate
    • Timing: Sprint 4 (after dev/test validation)
    • Owner: Marco (coordinate with IT networking team)
    • Risk: DNS propagation delay (24-48 hours), certificate issuance lead time

    Open Questions (Updated)¶

    Q1: APIM Tier Selection (DECISION NEEDED)¶

    Question: Standard (\(700/mo, 2 regions max) vs. Premium (\)2800/mo, multi-region, higher throughput)?
    Impact: Budget vs. availability/performance tradeoff
    Recommendation: Start with Standard for Sprints 1-3; upgrade to Premium before Q1 2026 demo if multi-region is required
    Decision Owner: Marco (PO) + Finance

    Q2: Multi-Region Strategy (DECISION NEEDED)¶

    Question: Active-active (complex, automatic failover) or active-passive (simpler, manual failover)?
    Impact: RTO (active-passive: ~1 hour, active-active: <1 minute) vs. cost/complexity
    Recommendation: Active-passive for MVP; active-active if SLA requires <1 min RTO
    Decision Owner: Marco (PO) + Tech Leads

    Q3: API Key vs. JWT for M2M (DESIGN DECISION)¶

    Question: Machine-to-machine (M2M) access: API keys (simpler) or service principal JWT (more secure)?
    Impact: Developer experience vs. security/auditability
    Recommendation: JWT for internal services (full traceability), API keys for external partners (scoped, revocable)
    Decision Owner: ai2 (Security Sentinel)

    Q4: Rate Limit Configuration (POLICY DECISION)¶

    Question: Default quotas: 100 req/min per user, 1000 req/min per tenantare these appropriate?
    Impact: User experience (too low = frustration) vs. cost control (too high = runaway usage)
    Recommendation: Start conservative (100/1000), monitor usage in Sprint 2-3, adjust based on P95 usage patterns
    Decision Owner: Marco (PO) + Product Team

    Q5: External Partner Access Model (BUSINESS DECISION)¶

    Question: Separate APIM instance for partners (full isolation) or same instance with strict policies?
    Impact: Cost (2x APIM instances) vs. security risk (shared infrastructure)
    Recommendation: Same instance with strict policies for MVP; separate instance if partner volume >10% of total traffic
    Decision Owner: Marco (PO) + Security Team

    Q6: Caching Strategy (PERFORMANCE)¶

    Question: Which APIs should have response caching enabled? TTL values?
    Impact: Backend load reduction vs. data freshness
    Recommendation: Enable caching for /v1/ops/health (30s), /v1/rag/search (5s for identical queries), disable for /v1/chat (always fresh)
    Decision Owner: ai2 (Performance Optimization)

    Q7: WAF Integration (SECURITY)¶

    Question: Azure Front Door WAF (full DDoS protection, $250/mo) or APIM built-in policies (basic)?
    Impact: Security posture vs. cost
    Recommendation: APIM built-in for MVP; Front Door WAF if penetration testing reveals gaps
    Decision Owner: Security Sentinel + Marco (PO)

    Success Criteria¶

    • 99.9% API availability (SLA)
    • P95 latency <50ms for gateway overhead
    • Rate limiting enforced (no bypass)
    • 100% authenticated requests (no anonymous access)
    • Zero security incidents (auth bypass, injection)
    • API documentation 100% up-to-date (automated generation)

    Dependencies¶

    • eva-infra: APIM provisioning via Terraform
    • eva-auth: JWT token issuance and validation
    • eva-api: Backend REST/GraphQL services
    • eva-ops: Monitoring and alerting
    • Azure Entra ID: Identity provider
    • Application Insights: Observability backend
