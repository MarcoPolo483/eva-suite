    1. Cost Analysis & Local Model Evaluation¶

    Request:

    "please save the cost analysis you just did so we can revisited it later. Perhaps we can look into adding a local model in my machine (or a powerful new one i would buy-- up to CAD 3k, or we can use that in cloud services.. that is a huge budget increase."

    Preference Extracted: - Budget Flexibility: Up to $3,000 CAD available for hardware OR increased cloud services - Local Model Interest: Open to self-hosted Llama 3.2 70B (if cost-effective) - Cloud Services Priority: Willing to increase cloud budget significantly if it's more effective than local hardware

    Decision Made: - Hybrid Burst Strategy RECOMMENDED (by Copilot, pending Marco's approval) - Keep $100/month baseline (Azure $36 + OpenAI $50 + buffer $14) - Use $3K for burst capacity: $200/month OpenAI + $50/month monitoring when needed - Provides 20 months of burst runway for sprint deadlines / production incidents - Defer local GPU until Llama 4 or Mistral Large 3 matches GPT-4 reasoning (expected Q2 2026)

    Rationale: - Offline resilience concern (see below) argues AGAINST local GPU (machine might be offline during internet outages) - Cloud burst strategy more flexible (scale up/down as needed) - No upfront hardware commitment or maintenance overhead

    Follow-up Needed: - [ ] Marco: Approve hybrid burst strategy OR choose alternative (local GPU, pure cloud)

    Statement:

    "Regarding Azure infrastructure, it's better that Windows Guy do it as imagine the IaC won't be tested before applying it in Azure."

    Preference Extracted: - Owner: Windows Guy (human), NOT Copilot (AI agent) - Reason: Concern about untested Terraform being applied directly to Azure (risk of breaking prod) - Trust Level: Marco trusts Windows Guy more than AI for infrastructure provisioning

    Follow-up Question (Marco):

    "Unless you confirm it can be tested in some terraform best practices workbench."

    Answer Provided (Copilot): - YES, Terraform can be tested before applying: - terraform validate (syntax check) - terraform plan (dry-run, shows what will be created/modified/deleted) - Apply to dev environment first (safe testing ground) - Smoke tests after dev deploy (validate before promoting to test/prod)

    Decision: - Windows Guy still owns provisioning (Marco's preference respected) - Copilot can generate draft Terraform (for Windows Guy to review and test) - Testing strategy documented in AZURE-INFRA-PLAN.md

    Follow-up Needed: - [ ] Copilot: Generate draft Terraform files (for Windows Guy review) - [ ] Windows Guy: Review, test with terraform plan, apply to dev

    4. Rollback Concerns (Patches > Public Webpage Deployment)¶

    Statement:

    "Rollback options: i am more concerned on the possibility to have to rollback patches and minor releases than the possibility of public webpage deployment not working. is that a valid concern?"

    Preference Extracted: - Primary Concern: Rollback of patches and minor releases (business logic, API changes, database schema) - Secondary Concern: Public webpage deployment failures (less critical, easier to fix) - Valid Concern: YES - patches are higher risk (data corruption, breaking changes, security regressions)

    Answer Provided (Copilot): - Why Patches Are Higher Risk: - Modify data in place (hard to undo) - Downstream services depend on old behavior (breaking changes) - Performance regressions not caught in testing - Security regressions (fix may introduce new vulnerability)

    • Rollback Strategy (By Change Type): | Change Type | Risk | Rollback Method | Time | Approval Needed? | |-------------|------|-----------------|------|------------------| | UI-only | Low | Slot swap | 2 min | No (automated) | | Minor release (no schema) | Medium | Slot swap + feature flag | 5 min | No (PSM) | | Patch (with schema) | HIGH | Slot swap + migration rollback | 10-30 min | YES (Marco) | | Major release (breaking API) | Critical | Full rollback + comms | 30-60 min | YES (Marco + stakeholders) |

    Mitigation Provided: - Blue/green deployment for ALL releases (not just UI) - Feature flags for patches (disable without rollback) - Automated rollback on failed health checks (within 5 minutes) - Canary releases (5% traffic, monitor, then 100%) - Database migration up/down scripts (versioned, tested)

    Follow-up Needed: - [ ] Implement blue/green deployment (all Azure environments) - [ ] Create database migration tooling (Flyway or Liquibase for Cosmos DB) - [ ] Test rollback procedure monthly (automated GitHub Actions workflow)

    5. Offline Resilience (Cloud Team Advantage)¶

    Statement:

    "I woken up this morning and i had no internet access. That's one of advantages of having the team in the cloud."

    Preference Extracted: - Cloud Resilience: HIGH PRIORITY ✅ - Observation: Autonomous AI crew (Master Scrum Master, PSM-1 to PSM-4, DevMaster, QA) continues working 24/7 even when Marco is offline - Value: Team doesn't stop when Marco's internet goes down

    Architectural Benefits (Marco's Realization): 1. GitHub Actions run in cloud: No dependency on Marco's machine being online 2. Agent spawns happen in cloud: OpenAI API calls work regardless of Marco's internet 3. Daily standups posted automatically: MSM workflow runs on schedule, Marco sees results when reconnected 4. Deployments happen automatically: Merge to main triggers deployment, no manual intervention

    Trade-off for Local GPU Model: - Local Llama 3.2 70B would run on Marco's machine - If Marco's internet goes down → machine may be powered off → agents can't work - Argues FOR cloud-based OpenAI API (resilience over cost savings)

    Implication for Cost Decision: - This offline resilience observation STRENGTHENS the recommendation for hybrid burst cloud strategy - WEAKENS the case for local GPU hardware (would create dependency on Marco's machine uptime)

    Follow-up Needed: - [ ] Marco: Re-evaluate local GPU option given offline resilience priority - [ ] Consider: If internet outages frequent, invest in Starlink backup ($120/month) instead of local GPU
