    Markdown to PowerShell Template Converter¶

    Purpose¶

    Prevents backtick escaping issues when converting markdown specifications (like TRANSFORMATION-ROADMAP.md) into PowerShell templates.

    The Backtick Problem¶

    Root Cause: Markdown uses backticks (`) for inline code highlighting. PowerShell uses backticks as escape characters in here-strings (@"..."@). When markdown content is copied into PowerShell here-strings, backticks must be doubled ( ``) to render correctly.

    Solution: External Templates¶

    Approach: Store templates as external .md.template files instead of embedded PowerShell here-strings.

    Benefits¶

    • ✅ No backtick escaping required
    • ✅ Templates editable as normal markdown
    • ✅ Version control-friendly
    • ✅ Reusable across scripts

    Template System¶

    Template Location: templates/*.md.template

    Placeholder Format: {{VARIABLE_NAME}}

    Example Template: 

    # Requirements - {{DESCRIPTION}}

**Feature**: {{FEATURE_KEY}}
**Slug**: {{FEATURE_SLUG}}
**Generated**: {{TIMESTAMP}}

Use `backticks` freely for code references!

    PowerShell Usage: 

    function Get-TemplateContent {
    param(
        [string]$TemplateName,
        [hashtable]$Replacements
    )

    $templatePath = Join-Path $PSScriptRoot "..\templates\$TemplateName.template"

    if (-not (Test-Path $templatePath)) {
        return $null  # Fallback to inline template
    }

    $content = Get-Content $templatePath -Raw

    # Replace all placeholders
    foreach ($key in $Replacements.Keys) {
        $content = $content -replace "\{\{$key\}\}", $Replacements[$key]
    }

    return $content
}

# Usage
$replacements = @{
    "DESCRIPTION" = "My Feature"
    "FEATURE_KEY" = "P02-FEAT-001"
    "FEATURE_SLUG" = "my-feature"
    "TIMESTAMP" = (Get-Date).ToString()
}

$content = Get-TemplateContent -TemplateName "requirements.md" -Replacements $replacements

    Intake Pattern: Markdown Sanitization¶

    When ingesting markdown from external sources (e.g., Senior Advisor specifications), use this sanitization function:

    function ConvertTo-TemplateFile {
    <#
    .SYNOPSIS
    Converts markdown content to external template file format

    .DESCRIPTION
    Sanitizes markdown for PowerShell compatibility by:
    1. Extracting variable references (e.g., $FeatureSlug)
    2. Replacing them with placeholders (e.g., {{FEATURE_SLUG}})
    3. Preserving backticks (no escaping needed)

    .PARAMETER MarkdownContent
    Raw markdown content to convert

    .PARAMETER OutputPath
    Path to save the .template file

    .PARAMETER VariableMap
    Hashtable mapping PowerShell variables to placeholder names
    #>
    param(
        [Parameter(Mandatory)]
        [string]$MarkdownContent,

        [Parameter(Mandatory)]
        [string]$OutputPath,

        [hashtable]$VariableMap = @{
            '$FeatureKey' = '{{FEATURE_KEY}}'
            '$FeatureSlug' = '{{FEATURE_SLUG}}'
            '$Description' = '{{DESCRIPTION}}'
            '$runId' = '{{RUN_ID}}'
            '$Caller' = '{{CALLER}}'
            '$timestamp' = '{{TIMESTAMP}}'
        }
    )

    $sanitized = $MarkdownContent

    # Replace PowerShell variables with placeholders
    foreach ($var in $VariableMap.Keys) {
        $placeholder = $VariableMap[$var]
        $sanitized = $sanitized -replace [regex]::Escape($var), $placeholder
    }

    # Write to template file
    Set-Content -Path $OutputPath -Value $sanitized -Encoding UTF8

    Write-Host "✅ Template created: $OutputPath" -ForegroundColor Green
    Write-Host "   Backticks preserved: No escaping needed!" -ForegroundColor Cyan
}

    Usage Example: 

    # Read markdown from Senior Advisor specification
$markdown = Get-Content "docs/TRANSFORMATION-ROADMAP.md" -Raw

# Extract a section (e.g., BOLT 6 template spec)
$templateSection = $markdown -split "## BOLT 6" | Select-Object -Last 1

# Convert to external template
ConvertTo-TemplateFile `
    -MarkdownContent $templateSection `
    -OutputPath "templates/some-new-artifact.md.template" `
    -VariableMap @{
        '$FeatureSlug' = '{{FEATURE_SLUG}}'
        '$FeatureKey' = '{{FEATURE_KEY}}'
    }

    Prevention at Source¶

    When creating TRANSFORMATION-ROADMAP.md (or similar specification documents):

    1. Write markdown normally - use backticks freely for code
    2. Use placeholder syntax - {{VARIABLE}} instead of $variable in examples
    3. Document variables separately - list variable names in a table

    Good Practice: 

    ## Template Specification

Template file: `requirements.md.template`

Variables to replace:
| Placeholder | PowerShell Variable | Description |
|-------------|---------------------|-------------|
| `{{FEATURE_SLUG}}` | `$FeatureSlug` | URL-friendly feature name |
| `{{TIMESTAMP}}` | `$timestamp` | UTC + ET dual timestamp |

Template content:
- Use `backticks` for inline code
- Use \`\`\`powershell for code blocks
- NO escaping needed!

    Migration Strategy¶

    Existing Scripts (like p02-run.ps1):

    1. Extract here-string templates to templates/ directory
    2. Add Get-TemplateContent helper function
    3. Replace $content = @"..."@ with $content = Get-TemplateContent
    4. Keep fallback for artifacts without external templates

    New Scripts:

    1. Create template in templates/ first
    2. Use Get-TemplateContent from the start
    3. No here-strings with backticks ever again!

    Testing¶

    Validate Template Rendering: 

    # Test template loading
$replacements = @{
    "FEATURE_SLUG" = "test-feature"
    "TIMESTAMP" = "2025-12-07 14:00:00 UTC (9:00:00 AM ET)"
}

$content = Get-TemplateContent -TemplateName "requirements.md" -Replacements $replacements

# Verify backticks rendered correctly
if ($content -match '`') {
    Write-Host "✅ Backticks preserved in output" -ForegroundColor Green
} else {
    Write-Warning "⚠️  Backticks may be missing"
}
