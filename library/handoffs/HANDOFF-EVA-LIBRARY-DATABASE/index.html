    HANDOFF: EVA Meta - Unified Metadata Registry¶

    To: Backend/Database Team From: Scrum Master (GitHub Copilot) Date: 2025-12-12 Pattern: P22-DPB (Documentation Publishing Bot) Repository: eva-meta (metadata registry for entire EVA Suite)

    ğŸ¯ Mission¶

    Create EVA Meta - the unified metadata registry for the entire EVA Suite. This is the single source of truth for:

    1. EVA Library - Books registry (documentation portal)
    2. EVA Data Pipeline - Data sources, raw data locations, transformations
    3. EVA Data Model - Schema definitions, entity relationships
    4. EVA DAL - Data Access Layer configuration
    5. Service Registry - EVA services, endpoints, health status

    Architecture Decision: EVA Meta = Central metadata hub. EVA Library is one consumer.

    ğŸ“Š Database Schema¶

    EVA Meta database contains 5 core tables + relationships:

    1. Table: eva_books¶

    Purpose: Books registry for EVA Library portal (documentation)

    CREATE TABLE eva_books (
    book_id VARCHAR(100) PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    description TEXT NOT NULL,
    privacy VARCHAR(20) NOT NULL CHECK (privacy IN ('public', 'private')),
    category VARCHAR(100) NOT NULL,
    url VARCHAR(500) NOT NULL,
    repo VARCHAR(500),
    tags JSON,
    status VARCHAR(50) NOT NULL CHECK (status IN ('production', 'alpha', 'beta', 'concept', 'deprecated')),
    last_updated DATE NOT NULL,
    nav_priority INTEGER DEFAULT 999,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_privacy (privacy),
    INDEX idx_category (category),
    INDEX idx_status (status),
    INDEX idx_nav_priority (nav_priority)
);

    2. Table: eva_data_sources¶

    Purpose: Data catalog for EVA Data Pipeline (raw data locations, ingestion metadata)

    CREATE TABLE eva_data_sources (
    source_id VARCHAR(100) PRIMARY KEY,
    source_name VARCHAR(255) NOT NULL,
    source_type VARCHAR(50) NOT NULL CHECK (source_type IN ('sharepoint', 'blob-storage', 'sql-database', 'api', 'file-share', 'web-scrape')),
    description TEXT NOT NULL,
    connection_string TEXT,  -- Encrypted, Azure Key Vault reference
    data_classification VARCHAR(50) NOT NULL CHECK (data_classification IN ('unclassified', 'protected-a', 'protected-b', 'protected-c')),
    owner_email VARCHAR(255) NOT NULL,
    owner_department VARCHAR(255),
    storage_location VARCHAR(500) NOT NULL,  -- Azure Blob container URL, SharePoint site URL, etc.
    raw_data_path VARCHAR(500) NOT NULL,     -- Path within storage (e.g., /raw/jurisprudence/)
    ingestion_frequency VARCHAR(50) CHECK (ingestion_frequency IN ('real-time', 'hourly', 'daily', 'weekly', 'manual')),
    last_ingested TIMESTAMP,
    total_documents INTEGER DEFAULT 0,
    total_size_mb DECIMAL(10, 2) DEFAULT 0,
    status VARCHAR(50) NOT NULL CHECK (status IN ('active', 'paused', 'deprecated', 'error')),
    pipeline_id VARCHAR(100),  -- FK to eva_pipelines (future)
    tags JSON,
    metadata JSON,  -- Custom metadata (file formats, schemas, etc.)
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_source_type (source_type),
    INDEX idx_data_classification (data_classification),
    INDEX idx_status (status),
    INDEX idx_owner (owner_email)
);

    3. Table: eva_data_model¶

    Purpose: Schema definitions and entity relationships for EVA Suite

    CREATE TABLE eva_data_model (
    entity_id VARCHAR(100) PRIMARY KEY,
    entity_name VARCHAR(255) NOT NULL,
    entity_type VARCHAR(50) NOT NULL CHECK (entity_type IN ('table', 'view', 'document', 'api-resource', 'message-queue')),
    description TEXT NOT NULL,
    schema_definition JSON NOT NULL,  -- JSON Schema or SQL DDL
    database_name VARCHAR(100),
    service_id VARCHAR(100),  -- FK to eva_services (which service owns this entity)
    data_classification VARCHAR(50) NOT NULL CHECK (data_classification IN ('unclassified', 'protected-a', 'protected-b', 'protected-c')),
    relationships JSON,  -- FK relationships to other entities
    indexes JSON,  -- Index definitions
    constraints JSON,  -- Constraints (PK, FK, unique, check)
    sample_data JSON,  -- Sample records for docs/testing
    tags JSON,
    version VARCHAR(20) NOT NULL DEFAULT '1.0.0',
    status VARCHAR(50) NOT NULL CHECK (status IN ('active', 'deprecated', 'draft')),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_entity_type (entity_type),
    INDEX idx_service_id (service_id),
    INDEX idx_status (status)
);

    Example Record: 

    {
  "entity_id": "chat-message",
  "entity_name": "ChatMessage",
  "entity_type": "table",
  "description": "Chat messages exchanged between users and EVA assistants",
  "schema_definition": {
    "type": "object",
    "properties": {
      "message_id": {"type": "string", "format": "uuid"},

      "timestamp": {"type": "string", "format": "date-time"},
      "metadata": {"type": "object"}
    },
    "required": ["message_id", "session_id", "message_text", "role", "timestamp"]
  },
  "database_name": "eva-chat-db",
  "service_id": "eva-chat",
  "data_classification": "protected-b",
  "relationships": {
    "session_id": "chat-session.session_id"
  },
  "indexes": [
    {"name": "idx_session_id", "columns": ["session_id"]},
    {"name": "idx_timestamp", "columns": ["timestamp"]}
  ],
  "version": "1.0.0",
  "status": "active"
}

    4. Table: eva_services¶

    Purpose: Service registry for all EVA microservices (APIs, UIs, pipelines)

    CREATE TABLE eva_services (
    service_id VARCHAR(100) PRIMARY KEY,
    service_name VARCHAR(255) NOT NULL,
    service_type VARCHAR(50) NOT NULL CHECK (service_type IN ('api', 'ui', 'pipeline', 'worker', 'function', 'database')),
    description TEXT NOT NULL,
    endpoint_url VARCHAR(500),  -- API endpoint or UI URL
    health_check_url VARCHAR(500),
    repo VARCHAR(500),
    documentation_url VARCHAR(500),
    owner_pod VARCHAR(50),  -- POD-O, POD-Q, POD-F, POD-X
    owner_email VARCHAR(255),
    deployment_environment VARCHAR(50) NOT NULL CHECK (deployment_environment IN ('sandbox', 'pilot', 'production')),
    data_classification VARCHAR(50) NOT NULL CHECK (data_classification IN ('unclassified', 'protected-a', 'protected-b', 'protected-c')),
    dependencies JSON,  -- List of service_id this service depends on
    tags JSON,
    status VARCHAR(50) NOT NULL CHECK (status IN ('active', 'degraded', 'down', 'maintenance', 'deprecated')),
    last_health_check TIMESTAMP,
    version VARCHAR(20) NOT NULL DEFAULT '1.0.0',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_service_type (service_type),
    INDEX idx_deployment_environment (deployment_environment),
    INDEX idx_status (status),
    INDEX idx_owner_pod (owner_pod)
);

    5. Table: eva_dal_connections¶

    Purpose: Data Access Layer configuration for database connections and ORMs

    CREATE TABLE eva_dal_connections (
    connection_id VARCHAR(100) PRIMARY KEY,
    connection_name VARCHAR(255) NOT NULL,
    database_type VARCHAR(50) NOT NULL CHECK (database_type IN ('postgresql', 'cosmosdb', 'sqlserver', 'mysql', 'sqlite', 'mongodb')),
    description TEXT NOT NULL,
    connection_string_key VARCHAR(500) NOT NULL,  -- Azure Key Vault secret name (never store actual connection string)
    database_name VARCHAR(100) NOT NULL,
    schema_name VARCHAR(100),
    service_ids JSON NOT NULL,  -- List of service_id that use this connection
    data_classification VARCHAR(50) NOT NULL CHECK (data_classification IN ('unclassified', 'protected-a', 'protected-b', 'protected-c')),
    orm_type VARCHAR(50) CHECK (orm_type IN ('sqlalchemy', 'typeorm', 'prisma', 'mongoose', 'none')),
    orm_config JSON,  -- ORM-specific configuration
    pool_size INTEGER DEFAULT 10,
    max_overflow INTEGER DEFAULT 20,
    connection_timeout_seconds INTEGER DEFAULT 30,
    tags JSON,
    status VARCHAR(50) NOT NULL CHECK (status IN ('active', 'degraded', 'down', 'maintenance')),
    last_health_check TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_database_type (database_type),
    INDEX idx_data_classification (data_classification),
    INDEX idx_status (status)
);

    Example Record:

  "connection_id": "eva-chat-cosmosdb",

    Data Pipeline â†’ Data Model¶

    -- A data source (SharePoint) feeds into a data model (jurisprudence schema)
SELECT
  ds.source_id,
  ds.source_name,
  ds.raw_data_path,
  dm.model_id,
  dm.entity_name,
  dm.schema_definition
FROM eva_data_sources ds
JOIN eva_data_model dm ON ds.pipeline_id = dm.pipeline_id
WHERE ds.pipeline_id = 'jurisprudence-pipeline';

    Services â†’ DAL Connections¶

    -- A service (EVA Chat API) uses a DAL connection (CosmosDB)
SELECT
  s.service_id,
  s.service_name,
  s.endpoint_url,
  d.connection_id,
  d.connection_name,
  d.database_type
FROM eva_services s
JOIN eva_dal_connections d ON s.service_id = d.service_id
WHERE s.service_id = 'eva-chat-api';

    Books â†’ Services¶

    -- A Book (EVA-Sovereign-UI) documents a service (EVA UI Library)
SELECT
  b.book_id,
  b.title,
  s.service_id,
  s.service_name,
  s.endpoint_url
FROM eva_books b
LEFT JOIN eva_services s ON b.book_id = s.documented_in
WHERE b.book_id = 'eva-sovereign-ui';

    Note: The documented_in column in eva_services table links services to their documentation Books.

    Architecture Principles¶

    1. Single Source of Truth: EVA Meta is the canonical registry for all Books, data sources, data models, services, and DAL connections
    2. Privacy-Aware: Books table enforces public/private separation; data sources enforce Protected B classification
    3. Service Discovery: Services table enables dynamic service discovery (health checks, dependencies, endpoints)
    4. DAL Abstraction: DAL connections table abstracts database connections with Key Vault encryption (no secrets in metadata)
    5. Cross-Product Integration: All EVA Suite products query EVA Meta for metadata (no duplication)

    Repository Architecture Options¶

    Option 1: EVA Meta Standalone (Recommended) - eva-meta repo becomes the unified metadata registry - Contains database schema, API endpoints, Web Components - All EVA products import from eva-meta (Books, data sources, services) - Pros: Clear ownership, single repo for metadata, easy to version - Cons: New repo to manage

    Option 2: EVA API Integration - EVA Meta database lives in eva-api backend - API endpoints exposed via eva-api - Pros: Leverages existing backend infrastructure - Cons: Couples metadata to backend deployment

    Option 3: EVA Orchestrator Extension - EVA Meta lives in eva-orchestrator as metadata subsystem - Pros: Already here, close to SDLC automation - Cons: Mixes metadata with orchestration concerns

    Marco's Decision Required: Which option for EVA Meta repository architecture?

    SELECT * FROM eva_books
    WHERE privacy = ? OR ? = 'all'
    ORDER BY nav_priority ASC
  `, [privacyFilter === 'all' ? 'public' : privacyFilter, privacyFilter]);
}

    What Changed¶

    Expanded from Books-only registry to comprehensive EVA Meta metadata system with 5 core tables:

    1. eva_books - Documentation Books (public/private)
    2. eva_data_sources - Data catalog for RAG pipelines (SharePoint, Blob Storage, etc.)
    3. eva_data_model - Schema definitions (JSON Schema, SQL DDL)
    4. eva_services - Microservices registry (APIs, UIs, workers)
    5. eva_dal_connections - Database connection pool config (Key Vault secured)

    Why This Matters¶

    EVA Meta becomes the single source of truth for: - EVA Library: Public documentation portal (Books) - EVA Data Pipeline: Data source metadata (where raw data lives) - EVA Services: Service discovery (health checks, dependencies, endpoints) - EVA DAL: Abstracted database connections (no secrets in code)
