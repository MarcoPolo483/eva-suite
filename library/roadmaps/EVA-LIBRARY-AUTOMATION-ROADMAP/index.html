    EVA Library Automation Roadmap¶

    Vision: "I don't want to worry about EVA Library content anymore. P22-DPB takes care of it all as features are designed, developed, and implemented. At the end, we get the working feature AND its live documentation in markdown, HTML, and RAG (with all the whistles and bells)."

    Owner: P22-DPB (Documentation Publishing Bot) Status: In Progress Updated: 2025-12-13

    The Dream State (Maturity Level 5)¶

    Developer builds eva-gc-button component → P22-DPB automatically generates:

    1. Markdown Documentation:
    2. Component API reference (props, events, methods)
    3. Usage examples (code snippets)
    4. Accessibility notes (WCAG 2.1 AA compliance)

    5. Bilingual descriptions (EN-CA/FR-CA)

    6. HTML Site (MkDocs):

    7. Live component catalog
    8. Interactive demos (Storybook integration)

    9. Search-optimized (tags, categories)

    10. RAG Text (with metadata):

    11. RBAC controls (privacy: public, roles: Anonymous/Authenticated)
    12. Citation pointers (URL, tags, version)
    13. Usage instructions for AI assistants

    Zero Manual Work: Developer only maintains code. Documentation auto-syncs on every commit.

    Current State (Maturity Level 2)¶

    ✅ What Works Today¶

    1. Manual Markdown → HTML + RAG pipeline:
    2. Developer writes .md files in docs/eva-library/
    3. Push to main triggers .github/workflows/publish-docs.yml
    4. MkDocs builds HTML → site/eva-library/
    5. extract-rag-text.ps1 generates RAG → site/rag-output/eva-library/

    6. GitHub Pages deploys all formats

    7. RBAC Metadata embedded in RAG outputs:

    8. Privacy levels (public/admin)
    9. Required roles (Anonymous, Authenticated, Developer, etc.)

    10. Citation URLs for attribution

    11. Manifest Contract for ingestion:

    12. _manifest.json with rbac{}, citation{}, files[] metadata
    13. EVA Data Pipeline can read from predictable URLs

    ❌ What's Manual Today¶

    1. Writing Documentation: Developer must write markdown manually
    2. Keeping Docs in Sync: No automation to detect code changes and update docs
    3. API Reference Generation: No auto-extraction from TypeScript/JSDoc comments
    4. Example Code: Manually copy-pasted into markdown (can drift from actual code)
    5. Bilingual Content: Manual translation (no i18n workflow)
    6. Version Management: No automated versioning or changelog generation

    Maturity Levels (5-Stage Evolution)¶

    Level 1: Manual Everything (Pre-P22)¶

    • Developer writes code
    • Developer manually writes docs
    • Developer manually deploys HTML
    • No RAG extraction

    Level 2: Automated Publishing ✅ (CURRENT)¶

    • Developer writes code + markdown
    • Automated: HTML + RAG generation on push
    • Automated: GitHub Pages deployment
    • Automated: RBAC metadata embedding

    Level 3: Code-Driven Documentation (Target: Q1 2026)¶

    • Developer writes code with JSDoc comments
    • Automated: Markdown API reference from code
    • Automated: Example snippets from test files
    • Manual: Narrative documentation (guides, tutorials)

    Level 4: Living Documentation (Target: Q2 2026)¶

    • Developer writes code
    • Automated: API reference + examples + guides
    • Automated: Detect code changes → update affected docs
    • Automated: Version bump + changelog generation

    Level 5: Zero-Touch Documentation (Target: Q3 2026)¶

    • Developer writes code
    • Automated: Everything from Level 4 + bilingual translation
    • Automated: Interactive demos (Storybook auto-deployed)
    • Automated: Accessibility audit reports in docs
    • Automated: Usage analytics in RAG metadata

    Roadmap (6 Sprints)¶

    Sprint 1 (2025-12-13 ✅ DONE): Foundation¶

    • P22-DPB pattern specification
    • MkDocs + GitHub Actions workflow
    • RAG extraction with RBAC metadata
    • Manifest contract for EVA Data Pipeline
    • ADR-007: RAG storage strategy

    Sprint 2 (2025-12-20): API Reference Automation¶

    • Tool: TypeScript AST parser → extract component props/events/methods
    • Template: Generate markdown tables from parsed API
    • Workflow: extract-api-reference.ps1 step in publish-docs.yml
    • Example: eva-gc-button.md auto-generated from src/components/eva-gc-button/eva-gc-button.ts

    Acceptance Criteria: - Developer adds JSDoc comment to component → API reference markdown auto-updates on push - Markdown includes: Props table, Events table, Methods table, CSS custom properties

    Sprint 3 (2025-12-27): Example Extraction¶

    • Tool: Extract code blocks from .stories.ts (Storybook) → markdown examples
    • Validation: Ensure examples are runnable (no drift from actual code)
    • Workflow: extract-examples.ps1 step in publish-docs.yml

    Acceptance Criteria: - Developer writes Storybook story → code snippet appears in markdown docs - Examples include: Basic usage, props variations, accessibility examples

    Sprint 4 (2026-01-03): Change Detection¶

    • Tool: Git diff analyzer → detect modified components
    • Workflow: Only regenerate docs for changed components (not all)
    • Notification: Create GitHub Issue if docs drift detected (code changed but docs not updated)

    Acceptance Criteria: - Developer changes eva-gc-button.ts → only eva-gc-button.md regenerated (faster builds) - If developer deletes component → docs auto-archived with deprecation notice

    Sprint 5 (2026-01-10): Bilingual Automation¶

    • Tool: Azure AI Translator API integration
    • Workflow: Auto-translate EN-CA → FR-CA for component descriptions
    • Review: Human review step for translations (flag machine-translated content)

    Acceptance Criteria: - Developer writes EN-CA description → FR-CA auto-generated in same markdown (side-by-side tables) - Flagged for human review before deploying to Production tier

    Sprint 6 (2026-01-17): Interactive Demos¶

    • Integration: Embed Storybook iframe in MkDocs HTML
    • Workflow: Deploy Storybook to /storybook/ subdirectory
    • Linking: Auto-link component docs to live demo

    Acceptance Criteria: - Developer writes Storybook story → live demo embedded in HTML docs - Users can interact with component directly in documentation page

    Quality Gates (Per Maturity Level)¶

    Level 2 (Current)¶

    • ✅ All markdown files render without errors
    • ✅ RAG outputs include RBAC metadata
    • ✅ Manifest validates against schema
    • ✅ GitHub Pages deployment succeeds

    Level 3 (Code-Driven Docs)¶

    • ✅ All above +
    • ✅ API reference matches code (props/events/methods)
    • ✅ Examples extracted from tests (no manual copy-paste)
    • ✅ JSDoc coverage >80% for public APIs

    Level 4 (Living Docs)¶

    • ✅ All above +
    • ✅ Change detection triggers selective regeneration
    • ✅ Version bump on breaking changes (semver)
    • ✅ Changelog auto-generated from commits

    Level 5 (Zero-Touch)¶

    • ✅ All above +
    • ✅ Bilingual content (EN-CA/FR-CA) auto-translated
    • ✅ Interactive demos embedded and working
    • ✅ Accessibility audit reports published
    • ✅ Usage analytics in RAG metadata

    Success Metrics¶

    Developer Experience (DX)¶

    • Time to Document: Developer time spent on docs (target: <5% of dev time)
    • Doc Staleness: % of components with out-of-sync docs (target: 0%)
    • Update Lag: Time from code commit to docs live (target: <5 minutes)

    Content Quality¶

    • API Accuracy: % of API reference matching actual code (target: 100%)
    • Example Validity: % of code examples that run without errors (target: 100%)
    • Translation Quality: Human review score for machine translations (target: >⅘)

    Operational¶

    • Build Time: Time to regenerate all docs (target: <2 minutes)
    • Storage Cost: GitHub Pages + CDN cost (target: $0 free tier)
    • Pipeline Failures: Failed workflow runs per month (target: <2%)

    Risks & Mitigations¶

    Risk Impact Mitigation
    JSDoc coverage low API reference incomplete Lint rule enforces JSDoc for public APIs
    Translations inaccurate Bilingual users confused Human review gate before Production
    Build time too slow Developer friction Selective regeneration (only changed components)
    Schema drift RAG ingestion breaks Versioned manifest schema + validation tests
    GitHub Pages quota Deployment failures Monitor bandwidth, fallback to Azure CDN
