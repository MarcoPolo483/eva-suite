    Document Owner: Marco Presta (Product Owner) Created: November 17, 2025 Last Updated: November 17, 2025 Status: Living Document (Updated per Sprint)

    Executive Summary¶

    EVA 2.0 follows the Government of Canada's Guide on the Use of Generative AI and the FASTER principles (Fair, Accountable, Secure, Transparent, Educated, Relevant) as outlined by the Treasury Board of Canada Secretariat.

    Primary Reference: Canada.ca - Guide on the Use of Generative AI

    The FASTER Principles - EVA 2.0 Implementation¶

    üéØ F - Fair¶

    Canada Requirement: "Ensure that content from these tools does not include or amplify biases and that it complies with human rights, accessibility, and procedural and substantive fairness obligations; engage with affected stakeholders before deployment."

    EVA 2.0 Approach:¶

    Component Implementation Evidence
    eva-safety Pre/post filters, Azure Content Safety moderation, prompt injection defense Repo: eva-safety/
    eva-i11y Accessibility library (WCAG 2.1 AA), audit CLI, keyboard navigation, screen reader support Repo: eva-i11y/
    eva-i18n Bilingual support (en-US, fr-FR), ICU formatting, catalog management Repo: eva-i18n/
    Bias Testing GBA Plus analysis planned Sprint 12, user testing with diverse groups Sprint 12 backlog
    Stakeholder Engagement Community feedback loops, accessibility audits, user acceptance testing Sprint 15+

    Key Safeguards:

    • Moderation filters prevent harmful/biased content generation
    • Accessibility-first design ensures WCAG 2.1 AA compliance
    • Bilingual support (English/French) ensures language equity
    • User feedback loops to detect and address bias in responses

    üîí A - Accountable¶

    Canada Requirement: "Take responsibility for the content generated by these tools and the impacts of their use. This includes making sure generated content is accurate, legal, ethical, and compliant with the terms of use; establish monitoring and oversight mechanisms."

    EVA 2.0 Approach:¶

    Component Implementation Evidence
    eva-metering Event pipeline, cost calculation, usage tracking per user/tenant Repo: eva-metering/
    eva-ops Live Ops dashboard, quota management, feature flags, alerts Repo: eva-ops/
    Audit Trail All AI interactions logged with metadata (user, tenant, model, tokens, citations) eva-core audit service
    Human Oversight Dashboard for reviewing flagged interactions, manual intervention capability eva-ops dashboard
    Quality Gates Pre-commit hooks, test coverage ‚â•80%, lint pass, security scans .github/workflows/ + sm-qa-toolbox/

    Key Safeguards:

    • Every AI interaction is logged and traceable
    • Live Ops dashboard provides real-time monitoring
    • Quality gates enforce code standards before deployment
    • Human operators can review/override AI decisions

    üõ°Ô∏è S - Secure¶

    Canada Requirement: "Ensure that the infrastructure and tools are appropriate for the security classification of the information and that privacy and personal information are protected; assess and manage cyber security risks and robustness when deploying a system."

    EVA 2.0 Approach:¶

    Component Implementation Evidence
    eva-auth Entra ID integration, role-based access control (RBAC), tenant isolation Repo: eva-auth/
    eva-infra Terraform landing zone: VNet, Key Vault, APIM, Private Endpoints, NSGs Repo: eva-infra/
    Data Residency Azure Canada Central/East, no cross-border data transfer Terraform config
    Encryption TLS 1.3 in transit, Azure Storage encryption at rest, Key Vault for secrets Azure security baseline
    Privacy No PII in prompts (policy enforcement), synthetic data for testing eva-safety filters
    Security Scans Dependabot, Snyk, OWASP ZAP, CodeQL, secret scanning .github/workflows/security.yml

    Key Safeguards:

    • Entra ID authentication prevents unauthorized access
    • All data encrypted in transit (TLS 1.3) and at rest
    • Private networking (VNet, Private Endpoints) isolates resources
    • Secret management via Azure Key Vault (no hardcoded secrets)
    • Automated security scanning in CI/CD pipeline

    üì¢ T - Transparent¶

    Canada Requirement: "Identify content that has been produced using generative AI; notify users that they are interacting with an AI tool; provide information on institutional policies, appropriate use, training data and the model when deploying these tools; document decisions and be able to provide explanations if tools are used to support decision-making."

    EVA 2.0 Approach:¶

    Component Implementation Evidence
    AI Disclosure UI displays "Powered by Azure OpenAI" badge, every response marked as AI-generated eva-ui chat interface
    Citations RAG responses include source documents with page numbers eva-rag citation system
    Model Info UI shows active model (GPT-4o, GPT-4 Turbo), token usage, response time eva-ui metadata panel
    Documentation Public roadmap (README), architecture docs (docs/EVA-2.0/), API specs Repository root
    Audit Logs Full conversation history with metadata (model, tokens, latency, citations) eva-metering logs
    Explainability Memory context visible to users, tool call steps shown in UI eva-agent transparency

    Key Safeguards:

    • Users always know they're interacting with AI (no deception)
    • Citations provide traceability to source documents
    • Full audit trail enables post-hoc explainability
    • Public documentation explains capabilities and limitations

    üéì E - Educated¶

    Canada Requirement: "Learn about the strengths, limitations and responsible use of the tools; learn how to create effective prompts and to identify potential weaknesses in the outputs."

    EVA 2.0 Approach:¶

    Component Implementation Evidence
    User Training Onboarding tutorials, prompt engineering guides, best practices docs docs/user-guides/ (Sprint 15+)
    Team Training Community Board introductions, lessons learned documentation, tool creation exercises docs/eva-agile-command-community-board.md
    Prompt Templates Pre-built prompt templates for common tasks (summarization, Q&A, code generation) eva-agent prompt library
    Limitations Docs Clear documentation of model limitations, hallucination risks, bias potential README.md + user guides
    Feedback Loops Users can report inaccurate responses, thumbs up/down on every message eva-ui feedback system

    Key Safeguards:

    • Onboarding process educates users on AI capabilities/limits
    • Prompt templates reduce likelihood of poor-quality outputs
    • Feedback mechanism allows continuous improvement
    • Documentation sets realistic expectations

    ‚úÖ R - Relevant¶

    Canada Requirement: "Make sure the use of generative AI tools supports user and organizational needs and contributes to better outcomes for clients; consider the environmental impacts when choosing to use a tool; identify appropriate tools for the task; AI tools aren't the best choice in every situation."

    EVA 2.0 Approach:¶

    Component Implementation Evidence
    Use Case Validation User stories define clear business value, acceptance criteria validate outcomes docs/backlog-status.md
    Environmental Impact Azure Carbon Neutral commitment, efficient prompts (token optimization), caching Azure sustainability docs
    Appropriate Tooling Hybrid search (vector + keyword), tool calling only when needed, human escalation eva-rag + eva-agent
    Alternatives Static FAQ, keyword search, manual workflows still available (AI optional) eva-ui navigation
    ROI Tracking Metrics on time saved, accuracy improved, user satisfaction eva-ops dashboard (Sprint 15+)

    Key Safeguards:

    • Every feature justified by user need (not AI for AI's sake)
    • Carbon-neutral Azure infrastructure reduces environmental impact
    • Users can choose non-AI alternatives when appropriate
    • ROI metrics ensure AI adds actual value

    Directive on Automated Decision-Making Compliance¶

    Does EVA 2.0 fall under the Directive?

    Answer: Partially Yes (depends on deployment context)

    Scenario Directive Applies? EVA 2.0 Approach
    Internal productivity (drafting emails, brainstorming, research) ‚ùå No Low-risk use, no AIA required
    Client support chatbot (answering questions, troubleshooting) ‚ö†Ô∏è Maybe If informing decisions: AIA required
    Automated decisions (eligibility, benefits, assessments) ‚úÖ Yes NOT RECOMMENDED - EVA 2.0 is advisory, not decisional

    EVA 2.0 Position: Advisory Only, Not Decisional

    • EVA assists humans, does not make administrative decisions
    • All high-stakes decisions require human review and approval
    • If deployed for decision support: Algorithmic Impact Assessment (AIA) will be completed
    • Terms of Use will prohibit use for high-impact decisions (employment, benefits, credit, law enforcement)

    Privacy Considerations (Privacy Act Compliance)¶

    Personal Information Handling¶

    Risk Mitigation Evidence
    PII in prompts Policy enforcement: users warned not to enter PII, input filters scan for PII patterns eva-safety filters
    PII in training data Azure OpenAI models do NOT train on customer data (Microsoft contractual guarantee) Azure OpenAI ToS
    PII in logs Audit logs anonymized (user IDs only, no PII content), access restricted to admins eva-metering anonymization
    PII in RAG Document ingestion pipelines strip PII before indexing, Access control per tenant eva-rag de-identification
    Data residency All data stored in Azure Canada Central/East, no cross-border transfer Terraform config

    Key Safeguard: NO personal information enters Azure OpenAI (enforced by input filters)

    Best Practices Implementation Status¶

    Protection of Information¶

    • ‚úÖ No sensitive data in public tools: Policy enforced via input filters
    • ‚úÖ Understand data usage: Azure OpenAI ToS reviewed, no customer data used for training
    • ‚úÖ Security classification: Unclassified data only (no Protected A/B/C)
    • ‚úÖ Chief Security Officer approval: Pending (Sprint 10 security audit)
    • ‚úÖ Data residency: Canada Central/East (no US/EU cross-border)
    • üöß Opt-out feature: Azure OpenAI does not use customer data (opt-out not needed)

    Bias Mitigation¶

    • üöß Bias training: Sprint 12 (GBA Plus analysis, diversity training)
    • ‚úÖ Review generated content: Human-in-the-loop for all outputs
    • üöß Prompt formulation: Prompt engineering guide (Sprint 08)
    • ‚ö†Ô∏è Training data transparency: Azure OpenAI training data not fully disclosed (vendor limitation)
    • ‚úÖ AI disclosure: All AI-generated content marked as such

    Quality Assurance¶

    • ‚úÖ Not authoritative: UI warns users to verify AI responses
    • ‚úÖ Verify personal info: No PII generation (enforced by policy)
    • ‚úÖ Impact assessment: Low-risk use (advisory only, no decisions)
    • üöß Training data quality: Azure OpenAI responsible (vendor trust required)
    • ‚úÖ Grounding/prompt engineering: RAG citations, prompt templates
    • ‚úÖ User notification: "Powered by Azure OpenAI" badge in UI
    • ‚úÖ Performance monitoring: Live Ops dashboard (Sprint 15+)

    Public Servant Autonomy¶

    • ‚úÖ Need assessment: Every feature justified by user story
    • ‚úÖ AI literacy: Team training (Community Board, lessons learned)
    • ‚úÖ AI as aid: EVA assists, does not replace human judgment
    • ‚úÖ Form own views: Users encouraged to think before prompting
    • ‚úÖ Neutral prompts: Prompt engineering guide emphasizes neutrality (Sprint 08)
    • ‚úÖ Always review: UI requires user acknowledgment of AI limitations

    Legal Risk Mitigation¶

    • ‚ö†Ô∏è Legal counsel: Pending (Sprint 10 legal review of ToS)
    • ‚úÖ Directive compliance: Positioned as advisory only (no AIA unless deployment changes)
    • üöß Copyright check: UI will warn about potential IP issues (Sprint 08)
    • üöß Crown copyright: Pending (Sprint 10 Crown copyright policy review)
    • ‚úÖ Output evaluation: Human review required for all public-facing content

    Human/Machine Distinction¶

    • ‚úÖ Communicate AI use: "Powered by Azure OpenAI" badge in UI
    • ‚úÖ Inform users: Every message marked as AI-generated
    • üöß Alternative means: Phone/email support available (Sprint 15+)
    • üöß Watermarks: Pending (Sprint 08 AI-generated content watermarks)
    • ‚úÖ System description: README + user guides explain how EVA works

    Environmental Impact¶

    • ‚úÖ Net-zero data centers: Azure Carbon Neutral commitment (Microsoft 2030 goal)
    • ‚úÖ Relevant use: Every feature justified by business value (not AI for AI's sake)
    • ‚ö†Ô∏è GHG reduction targets: Microsoft public commitment (Azure infrastructure)
    • üöß Environmental impact assessment: Pending (Sprint 20 sustainability review)
    • ‚ö†Ô∏è Carbon awareness: Token optimization reduces compute (indirect benefit)

    Key Differences: EVA 2.0 vs. Canada Guide Assumptions¶

    Canada Guide Assumes EVA 2.0 Reality Implication
    Government employees using public AI tools (ChatGPT, Copilot) Custom enterprise platform on Azure with private infrastructure ‚úÖ Lower risk (controlled environment, no public tool risks)
    Ad-hoc use (drafting emails, brainstorming) Structured workflows (RAG, citations, memory, tools) ‚úÖ Higher quality (grounded in documents, traceable)
    Limited transparency (black-box models) Full audit trail (logs, citations, metadata) ‚úÖ More accountable (explainability built-in)
    Training on public data (Internet scraping) Azure OpenAI models (no customer data training) ‚úÖ More secure (privacy-preserving)
    English-first bias Bilingual by design (en-US, fr-FR) ‚úÖ More equitable (language parity)

    Conclusion: EVA 2.0 exceeds Canada Guide baseline requirements due to enterprise architecture, private infrastructure, and governance-first design.
