    Core Philosophy¶

    Marco's Vision:

    "I am designing the EVA Agile Dev Crew to be autonomous. That is my secret sauce to develop anything... I am sure other people will have the same idea... I hope I am the first in Canada."

    Autonomy Level: 90% (AI decisions) + 10% (Marco strategic escalation)

    Key Principles: 1. Scrum Master drives work (not developers pulling tasks) 2. Self-monitoring at all levels (Master SM → POD SMs → Dev Masters) 3. Peer-to-peer health checks (no single point of failure) 4. Living product always available (blue/green deployments) 5. Budget-conscious (optimize AI API costs) 6. Marco's breakfast = daily standup (GitHub Discussions sync)

    Crew Structure¶

    Master Scrum Master (MSM)
├── POD 1: Scrum Master → Dev Master → [Frontend Expert, Backend Expert, QA Expert]
├── POD 2: Scrum Master → Dev Master → [Frontend Expert, Backend Expert, QA Expert]
├── POD 3: Scrum Master → Dev Master → [Frontend Expert, Backend Expert, QA Expert]
└── POD 4: Scrum Master → Dev Master → [Frontend Expert, Backend Expert, QA Expert]

Cross-Cutting Roles:
├── Senior Advisor (P02 - ChatGPT API with 3-year memory)
├── Master Solution Expert (MSM wearing architecture hat)
└── Marco (Product Owner - strategic decisions only)

    Role Definitions:

    Master Scrum Master (MSM)¶

    • Spawned: Once (persistent via GitHub Actions cron)
    • Responsibilities:
    • Backlog prioritization across all PODs
    • Sprint planning (2-week sprints)
    • Work distribution to 4 POD Scrum Masters
    • Monitoring POD health (peer-to-peer heartbeats)
    • Daily report generation for Marco (GitHub Discussion)
    • Architecture decisions (as Master Solution Expert)
    • Senior Advisor consultation (P02 ChatGPT API)
    • Marco escalation (strategic decisions only)
    • Monitoring: POD Scrum Masters monitor MSM (peer health checks)
    • Recovery: Any surviving Scrum Master can respawn MSM

    POD Scrum Master (PSM)¶

    • Spawned: 4 instances (one per POD)
    • Responsibilities:
    • POD sprint backlog management
    • Dev Master task assignment
    • Expert utilization tracking (cost optimization)
    • Progress monitoring (every 2 hours)
    • Impediment removal
    • MSM reporting (daily)
    • Peer health checks (MSM + other PSMs)
    • Monitoring: MSM + peer PSMs monitor each PSM
    • Recovery: MSM or peer PSM respawns failed PSM

    Dev Master (DM)¶

    • Spawned: 4 instances (one per POD)
    • Responsibilities:
    • Spawn experts on-demand (Frontend, Backend, QA)
    • Code review and merge
    • Test execution (80%+ coverage)
    • Technical decision-making (within autonomy)
    • Architecture escalation to PSM → MSM
    • Expert cost tracking
    • Monitoring: PSM monitors DM (every 2 hours)
    • Recovery: PSM respawns failed DM

    Experts (Frontend, Backend, QA)¶

    • Spawned: On-demand by Dev Master (ephemeral)
    • Responsibilities:
    • Implement specific tasks (user stories, bugs, tests)
    • Follow coding standards (80%+ coverage, WCAG AAA)
    • Self-contained work (no handoffs between experts)
    • Monitoring: Dev Master monitors task completion
    • Recovery: Dev Master respawns if task hangs (timeout 4 hours)

    Recovery Procedures¶

    Scenario 1: MSM Fails 1. Any PSM detects MSM heartbeat missing (>30 min) 2. PSM-1 (primary backup) triggers MSM respawn 3. If PSM-1 also failed, PSM-2 takes over (cascade) 4. MSM reads last heartbeat log + task queue 5. MSM resumes from last known state 6. MSM notifies Marco via GitHub Discussion (incident report)

    Scenario 2: PSM Fails 1. MSM detects PSM heartbeat missing (>30 min) 2. MSM triggers PSM respawn 3. PSM reads last heartbeat log + POD task queue 4. PSM resumes from last known state 5. PSM notifies MSM (incident report)

    Scenario 3: Dev Master Fails 1. PSM detects DM progress stalled (>2 hours no updates) 2. PSM triggers DM respawn 3. DM reads last task state from git commits 4. DM resumes or reassigns task to new expert 5. DM notifies PSM (incident report)

    Scenario 4: Expert Hangs 1. DM detects task timeout (>4 hours no commits) 2. DM terminates expert (GitHub Actions timeout) 3. DM spawns new expert with same task 4. New expert reads previous expert's commits 5. New expert continues or restarts (idempotent tasks)

    Scenario 5: All Scrum Masters Fail (Catastrophic) 1. GitHub Actions cron triggers MSM spawn (failsafe) 2. MSM detects all PSMs missing 3. MSM spawns all 4 PSMs from last known state 4. MSM notifies Marco immediately (critical alert) 5. Marco reviews logs, approves recovery plan 6. MSM executes recovery (restore from git state)

    Recovery State Sources: - Git commits: Last code state (idempotent recovery) - Heartbeat logs: Last known health state - Task queue files: Last assigned work (eva-lwo/task-queues/) - Sprint board: Last sprint state (GitHub Projects API)

    Estimation & Time Tracking¶

    How Scrum Master Estimates Efforts¶

    Estimation Method: T-shirt sizing + historical velocity

    Marco's Question:

    "How are you estimating the efforts and how long that would take?"

    Answer:

    Step 1: T-shirt Sizing (Sprint Planning) 

    XS: 1-2 hours   (bug fix, small refactor)
S:  2-4 hours   (component update, test addition)
M:  4-8 hours   (new feature, API endpoint)
L:  8-16 hours  (complex feature, integration)
XL: 16-32 hours (major module, architecture change)

    Step 2: Historical Velocity (After Sprint 1) - Track actual time per task (git commit timestamps) - Calculate POD velocity: completed_points / sprint_duration - Adjust future estimates: estimated_time * (actual_avg / estimated_avg)

    Step 3: Expert Cost Estimation 

    Frontend Expert: $0.10-0.30 per task (OpenAI API)
Backend Expert:  $0.10-0.30 per task
QA Expert:       $0.05-0.15 per task
Total per task:  $0.25-0.75 average

    Step 4: Capacity Planning 

    1 POD capacity: 80 hours per 2-week sprint
  - Dev Master overhead: 10 hours (monitoring, reviews)
  - Expert actual work: 70 hours
  - Buffer (blockers): 10 hours
  = Net capacity: 60 hours per POD per sprint

4 PODs capacity: 240 hours per 2-week sprint

    Step 5: Sprint Planning Algorithm 

    def plan_sprint(backlog_items, pod_count=4, sprint_capacity=60):
    """
    Scrum Master's sprint planning algorithm
    """
    # Sort by priority (Business Value × Dependency Impact × Completeness)
    sorted_items = sort_by_priority(backlog_items)

    # Assign to PODs (load balancing)
    pod_assignments = {1: [], 2: [], 3: [], 4: []}
    pod_loads = {1: 0, 2: 0, 3: 0, 4: 0}

    for item in sorted_items:
        # Find least loaded POD
        target_pod = min(pod_loads, key=pod_loads.get)

        # Check capacity
        if pod_loads[target_pod] + item.estimate <= sprint_capacity:
            pod_assignments[target_pod].append(item)
            pod_loads[target_pod] += item.estimate
        else:
            # Item too large, split or defer
            if item.estimate > sprint_capacity:
                # Split into sub-tasks
                sub_tasks = split_item(item, max_size=sprint_capacity/2)
                backlog_items.extend(sub_tasks)
            else:
                # Defer to next sprint
                next_sprint_backlog.append(item)

    return pod_assignments

    Red Flag Detection (Automated): 

    def detect_red_flags(task):
    """
    Scrum Master's task health check
    """
    red_flags = []

    # Over-estimate threshold: 125% of estimate
    if task.elapsed > task.estimate * 1.25:
        red_flags.append("over_estimate")
        # Action: PSM escalates to MSM (impediment)

    # No commits threshold: 2 hours
    if now() - task.last_commit > timedelta(hours=2):
        red_flags.append("stalled_no_commits")
        # Action: DM checks expert health

    # Blocked state threshold: 4 hours
    if task.status == "blocked" and task.blocked_since > timedelta(hours=4):
        red_flags.append("blocked_too_long")
        # Action: PSM escalates to MSM (impediment removal)

    # Test coverage below 80%
    if task.test_coverage < 0.80:
        red_flags.append("insufficient_coverage")
        # Action: DM assigns QA expert

    return red_flags

    Decision Escalation¶

    When Does Scrum Master Escalate to Marco?¶

    Marco's Question:

    "Which kind of decisions? The only one I thought would be about architecture design."

    Answer: 10% Strategic Decisions

    Escalate to Marco (via GitHub Discussion):

    1. Architecture Changes
    2. Example: "Should we switch from Cosmos DB to PostgreSQL?"
    3. Trigger: Dev Master proposes major tech stack change

    4. Process:

      1. PSM escalates to MSM
      2. MSM (as Master Solution Expert) generates recommendation
      3. MSM consults Senior Advisor (P02 ChatGPT API)
      4. MSM compares answers, proposes best approach
      5. MSM posts decision request to Marco (GitHub Discussion)
      6. Marco approves/rejects (expected: <24 hours)

    5. Budget Overruns (>$100/month)

    6. Example: "We need $120/month for additional Cosmos DB throughput"
    7. Trigger: Cost tracking exceeds budget threshold

    8. Process:

      1. MSM analyzes cost drivers
      2. MSM generates cost optimization recommendations
      3. MSM posts decision request to Marco
      4. Marco approves new budget or requests alternatives

    9. Scope Additions (New EVA Products)

    10. Example: "Customer requests EVA-Compliance product"
    11. Trigger: New product request from external stakeholder

    12. Process:

      1. MSM logs request
      2. MSM estimates effort (ROM: 100-200 hours)
      3. MSM posts decision request to Marco
      4. Marco prioritizes against roadmap

    13. Timeline Adjustments (>2 weeks delay)

    14. Example: "Sprint 8 will overrun by 3 weeks due to Azure infrastructure delays"
    15. Trigger: Sprint velocity drops below 50% of target

    16. Process:

      1. MSM analyzes root cause
      2. MSM generates recovery plan options
      3. MSM posts decision request to Marco
      4. Marco selects option (defer features, extend sprint, add resources)

    17. Technology Adoption (New Dependencies)

    18. Example: "Should we adopt Azure AI Search for semantic search?"
    19. Trigger: Dev Master identifies new tech need
    20. Process:
      1. PSM escalates to MSM
      2. MSM researches cost, benefits, alternatives
      3. MSM consults Senior Advisor
      4. MSM posts decision request to Marco
      5. Marco approves/rejects

    Do NOT Escalate (90% Autonomous): - Code refactoring (performance, maintainability) - Bug fixes (even critical production bugs) - Test additions (coverage improvements) - Documentation updates - Process improvements (workflow optimizations) - Cost optimizations within budget (<$100/month) - Technology updates (dependency upgrades, security patches) - UI/UX improvements (within design system)

    CI/CD Strategy¶

    Marco's Question:¶

    "Would we need a team to do the CI/CD?"

    Answer: No separate team. CI/CD is fully automated via GitHub Actions.

    CI/CD Architecture:

    Developer Activity (AI Expert commits code)
    ↓
GitHub Actions Trigger (on push to branch)
    ↓
CI Pipeline (Build, Test, Lint)
    ├── Build (Vite/SvelteKit)
    ├── Test (Vitest - must pass 80%+ coverage)
    ├── Lint (ESLint - must pass)
    └── Security Scan (npm audit)
    ↓
Dev Master Review (automated PR review)
    ├── Code quality check
    ├── Test coverage validation (≥80%)
    └── Auto-merge if passing
    ↓
CD Pipeline (Deploy to environment)
    ├── Dev: Auto-deploy on merge to main
    ├── Test: Manual trigger (PSM approval)
    └── Prod: Manual trigger (Marco approval)
    ↓
Living Product Updated (Blue/Green Deployment)
    └── Zero downtime

    GitHub Actions Workflows:

    1. CI: Build & Test 

    # .github/workflows/ci.yml
name: CI - Build and Test
on:
  push:
    branches: [main, develop, 'feature/**']
