    Architectural Decision Records – EVA Artifacts Generator¶

    Feature ID: P02-ARTIFACTS-001
    Last updated: 2025-11-25

    Context¶

    EVA Artifacts Generator automates detection and generation of documentation across EVA Suite repos. Key challenges: - Data source selection: What file activity data should drive gap detection? - Architecture complexity: How to balance simplicity vs extensibility? - AI quality vs cost: When to use templates vs OpenAI? - Auditability: How to track AI usage for cost attribution? - Quality control: How to prevent low-quality AI-generated docs from polluting repos? - Multi-repo scaling: How to extend generator to 23 EVA Suite repos without fragmentation?

    These ADRs document architectural decisions made during P02 pattern application.

    Decision drivers¶

    1. Cost efficiency: Minimize OpenAI API costs (<$100 USD/month)
    2. Quality assurance: Prevent hallucinations and low-quality docs
    3. Auditability: Full traceability of AI usage (P02 telemetry compliance)
    4. Maintainability: Simple, testable, modular PowerShell scripts
    5. Extensibility: Support new doc types, rules, and repos incrementally
    6. User experience: Intuitive CLI, fast performance, clear feedback
    7. Integration: Leverage existing EVA Audit Trail infrastructure

    ADR-001: Use EVA-New-Artifacts-Map.md as primary data source¶

    Status: ✅ Accepted
    Date: 2025-11-25
    Deciders: Marco (Product Owner, Architect)

    Decision¶

    Use EVA-New-Artifacts-Map.md (maintained by EVA Audit Trail) as the primary data source for file activity. Optionally complement with EVA-Audit-Log.jsonl for enriched metadata.

    Rationale¶

    1. Already maintained: EVA Audit Trail updates map nightly (zero additional maintenance burden)
    2. 21-day window: Sufficient for gap detection (captures recent activity without noise)
    3. Human-readable: Markdown format easy to parse and debug
    4. Proven reliability: Map used successfully by P02-AUDIT-001
    5. Lightweight: Single file read vs querying Git log (faster startup)

    Consequences¶

    Pros: - Zero additional data collection infrastructure - Reuses proven EVA Audit Trail patterns - Fast parsing (<2 seconds for 1,000 files) - Easy troubleshooting (inspect map manually)

    Cons: - Dependency on EVA Audit Trail (must be installed first) - 21-day window may miss older files (acceptable trade-off) - Map format changes break parser (mitigated by schema validation)

    Mitigations: - Validate map structure before parsing (fail gracefully if invalid) - Document supported EVA Audit Trail versions - Provide fallback: Parse Git log if map unavailable (Phase 3)

    Alternatives considered¶

    1. Git log directly: git log --name-only --since="21 days ago"
    2. Rejected: Slower (5-10s for large repos), harder to parse, Git dependency
    3. EVA-Audit-Log.jsonl only: JSONL events as single source
    4. Rejected: Requires parsing entire log (slower), less human-readable
    5. GitHub API: Query commit activity via API
    6. Rejected: Rate limits, authentication complexity, online-only

    ADR-002: Modular 4-layer design for testability and extensibility¶

    Status: ✅ Accepted
    Date: 2025-11-25
    Deciders: Marco (Architect)

    Decision¶

    Implement generator using 4-layer modular architecture: 1. Layer 1 (Scanner): Parse data sources (Markdown, JSONL) 2. Layer 2 (Analyzer): Detect gaps, categorize, prioritize 3. Layer 3 (Generator): Load templates, call OpenAI, write output 4. Layer 4 (Telemetry): Log P02 events, track costs

    Each layer has isolated functions with clear interfaces.

    Rationale¶

    1. Testability: Each function unit-testable in isolation (mock dependencies)
    2. Extensibility: Add new rules (Layer 2) or doc types (Layer 3) without refactoring
    3. Separation of concerns: Scanner doesn't know about OpenAI, Analyzer doesn't know about telemetry
    4. Maintainability: Functions <50 lines, single responsibility (NFR-8)
    5. Clarity: New developers quickly understand system boundaries

    Consequences¶

    Pros: - High test coverage achievable (>80% target) - Easy to add new gap rules (extend Detect-Gaps only) - Easy to add new doc types (extend Load-Template only) - Parallel development possible (different devs work on different layers)

    Cons: - More files/functions than monolithic script (acceptable trade-off) - Requires discipline to maintain layer boundaries

    Mitigations: - Document layer responsibilities in architecture-notes.md - Code review checklist: Verify layer boundaries respected - Integration tests validate end-to-end flows

    Alternatives considered¶

    1. Monolithic script: Single eva-artifacts-scan.ps1 with inline logic
    2. Rejected: Hard to test, hard to extend, hard to understand
    3. 3-layer design: Merge Analyzer + Generator
    4. Rejected: Violates separation of concerns (gap detection ≠ doc generation)
    5. 5+ layers: Add orchestrator layer, config layer, etc.
    6. Rejected: Over-engineering for Phase 1 scope

    ADR-003: Template-based (Phase 1) → AI-assisted (Phase 2) progressive enhancement¶

    Status: ✅ Accepted
    Date: 2025-11-25
    Deciders: Marco (Product Owner, FinOps)

    Decision¶

    Implement templates first (Phase 1, Sprint 02), add AI assistance later (Phase 2, Sprint 03). Users explicitly opt-in to AI via --UseOpenAI flag.

    Rationale¶

    1. Cost management: Validate value before incurring OpenAI costs
    2. Quality baseline: Templates provide known-good structure (fallback if AI fails)
    3. Incremental delivery: Phase 1 delivers value (gap detection + stubs) in 1 sprint
    4. Risk mitigation: Test AI quality on small scale before full rollout
    5. User choice: Developers can use templates only if AI quality insufficient

    Consequences¶

    Pros: - Phase 1 zero AI costs (budget-friendly) - Early feedback on gap detection accuracy (most critical feature) - Templates serve as fallback if OpenAI API unavailable - Incremental cost increase (easier to monitor and adjust)

    Cons: - Users must run generator twice (once for stubs, again for AI if desired) - Template maintenance burden (mitigated by versioning, ADR-003)

    Mitigations: - Clear CLI messaging: "Phase 1: Templates only. Phase 2: Add --UseOpenAI for AI assistance" - Template quality high enough to be useful standalone - AI flag optional (backwards compatible)

    Alternatives considered¶

    1. AI-first: Implement OpenAI in Phase 1, skip templates
    2. Rejected: High cost risk, no fallback if AI quality poor
    3. AI-only: No templates, always use OpenAI
    4. Rejected: Expensive for simple stubs (README for 1-line script)
    5. Hybrid: Always generate templates, optionally enhance with AI
    6. Deferred: Consider for Phase 3 based on Phase 2 learnings

    ADR-004: P02 telemetry integration mandatory (100% compliance)¶

    Status: ✅ Accepted
    Date: 2025-11-25
    Deciders: Marco (FinOps, Compliance)

    Decision¶

    All OpenAI API calls must be logged via P02 telemetry (NFR-2). No exceptions. Feature key: P02-ARTIFACTS-001.

    Rationale¶

    1. Cost attribution: Track costs per feature, repo, doc type
    2. Dogfooding: EVA Artifacts Generator uses EVA patterns (credibility)
    3. Auditability: Compliance with EVA Suite observability standards
    4. Observability: Power BI dashboards show generator usage alongside other P02 features
    5. Budgeting: Monthly cost rollups enable proactive budget management

    Consequences¶

    Pros: - Full cost transparency (no surprise bills) - Integration with existing P02 usage reporting infrastructure - Enables cost optimization (identify expensive doc types, optimize prompts) - Builds trust with stakeholders (verifiable AI usage)

    Cons: - Additional code complexity (wrap all OpenAI calls with telemetry) - Slight performance overhead (~10ms per call to log event)

    Mitigations: - Import scripts/p02-telemetry.ps1 helpers (reuse existing functions) - Automated tests verify 100% compliance (T-017) - Document integration clearly in README and architecture notes

    Alternatives considered¶

    1. Optional telemetry: --EnableTelemetry flag
    2. Rejected: Undermines cost tracking (users may disable)
    3. Manual logging: Developer logs costs in spreadsheet
    4. Rejected: Error-prone, not scalable, no automation
    5. Post-hoc analysis: Parse OpenAI usage reports monthly
    6. Rejected: Delayed visibility, can't track per-feature attribution

    ADR-005: Human-in-the-loop review (no auto-commit until Phase 3)¶

    Status: ✅ Accepted
    Date: 2025-11-25
    Deciders: Marco (Product Owner, Quality)

    Decision¶

    All AI-generated docs have "DRAFT - REVIEW REQUIRED" header. Developer must review and approve before committing. No auto-commit to repo in Phase 1-2.

    Rationale¶

    1. Quality control: Prevent hallucinations and inaccuracies from polluting repo
    2. Trust building: Validate AI quality before enabling automation
    3. Learning: Developers learn prompt patterns by reviewing outputs
    4. Compliance: Ensures docs meet EVA Suite standards (tone, structure, accuracy)
    5. Risk mitigation: Human oversight catches edge cases (e.g., sensitive data in prompts)

    Consequences¶

    Pros: - High confidence in committed docs (human-verified) - Incremental trust building (quality improves over time) - Developer education (understand AI strengths/weaknesses) - Compliance with quality standards

    Cons: - Manual step required (slows workflow vs auto-commit) - Developer time investment (~5-10 minutes per P02 package review)

    Mitigations: - Provide review checklist (10 items) for efficient review - Track quality ratings (good/fair/poor) to measure readiness for auto-commit - Phase 3: Enable auto-commit for high-quality doc types (e.g., 90%+ "good" ratings)

    Alternatives considered¶

    1. Auto-commit immediately: No human review
    2. Rejected: High risk of poor-quality docs, trust issues
    3. Auto-PR: Generate PR for review instead of direct commit
    4. Deferred: Consider for Phase 3 GitHub Actions integration
    5. Spot-check: Review 10% of docs randomly
    6. Rejected: Insufficient quality control for Phase 1-2

    ADR-006: Multi-repo via bootstrap pattern (reuse EVA Audit Trail approach)¶

    Status: ✅ Accepted
    Date: 2025-11-25
    Deciders: Marco (Product Owner, DevOps)

    Decision¶

    Extend generator to other EVA Suite repos using bootstrap script (eva-artifacts-bootstrap.ps1), similar to EVA Audit Trail's eva-audit-bootstrap.ps1.

    Rationale¶

    1. Proven pattern: Bootstrap successfully onboarded 5+ repos for EVA Audit Trail
    2. Standard conventions: Consistent structure across repos (easier troubleshooting)
    3. Autonomy: Each repo has local copy of scripts (no fragile cross-repo dependencies)
    4. Flexibility: Repos can customize rules/templates while maintaining core compatibility
    5. Incremental rollout: Onboard 1-2 repos per sprint, validate before scaling

    Consequences¶

    Pros: - Reuses proven EVA Audit Trail multi-repo pattern - Each repo self-sufficient (no monorepo complexity) - Clear documentation: Multi-repo guide similar to EVA-Audit-MultiRepo-Pattern.md - Gradual adoption (low risk)

    Cons: - Script copies diverge over time (version drift) - Updates require re-bootstrap or manual sync - 23 repos = 23 copies to maintain (acceptable trade-off)

    Mitigations: - Version headers in all scripts (e.g., # Version: 1.0) - Inventory script to track versions across repos - Phase 3: Optional automated sync workflow (GitHub Actions) - Clear documentation: When to re-bootstrap vs manual update

    Alternatives considered¶

    1. Monorepo: Single generator repo, all EVA repos call remotely
    2. Rejected: Fragile (network dependency), breaks if remote unavailable
    3. Git submodules: Link to central generator repo
    4. Rejected: Complex, developers unfamiliar with submodules
    5. Package manager: Publish as PowerShell module, Install-Module eva-artifacts
    6. Deferred: Consider for Phase 3 if version drift becomes problematic
