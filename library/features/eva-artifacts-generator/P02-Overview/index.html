    Requirements gathering insights¶

    What we learned¶

    1. EVA-New-Artifacts-Map.md is the goldmine: 21-day rolling window provides perfect context for recent activity without overwhelming data volume

    2. Gap detection requires heuristics: No universal rule for "missing docs"—need configurable rules (e.g., "scripts/ without tests/", "features/ without P02 packages")

    3. AI assistance is force multiplier: Generating P02 requirements.md manually takes 2-3 hours; AI could reduce to 30 minutes + review time

    4. Telemetry is non-negotiable: Generator MUST log its own OpenAI usage to maintain EVA Suite's "dogfooding" principle (we audit ourselves)

    5. Templates balance consistency and flexibility: Too rigid = unusable, too flexible = inconsistent. Solution: Markdown templates with placeholder blocks

    Gaps identified¶

    1. No standard doc structure: EVA Suite has inconsistent patterns for README vs docs/features/ vs docs/EVA-2.0/. Generator must handle variance.

    2. Context building is expensive: To generate high-quality docs, need to read multiple related files (30-50 KB of context) = higher OpenAI costs

    3. Review workflow undefined: Who approves AI-generated docs? How to track "draft" vs "reviewed" status?

    4. Multi-repo coordination: Each repo has different conventions (eva-core uses TypeScript doc comments, eva-orchestrator uses PowerShell help blocks)

    5. Version control integration: Should generator commit directly or create PRs? Security implications.

    Architectural decisions made¶

    See adr-eva-artifacts-generator.md for full ADRs. Key decisions:

    • ADR-001: Use EVA-New-Artifacts-Map.md as primary data source (21-day window sufficient)
    • ADR-002: Modular 4-layer design (Scanner, Analyzer, Generator, Telemetry)
    • ADR-003: Template-based generation for Phase 1, AI-assisted in Phase 2
    • ADR-004: P02 telemetry integration (log ALL OpenAI usage)
    • ADR-005: Human-in-the-loop review (no auto-commit in Phase 1-2)
    • ADR-006: Multi-repo via bootstrap pattern (reuse eva-audit-trail approach)

    Assumptions & open questions¶

    Assumptions¶

    1. EVA-New-Artifacts-Map.md format stable: Script assumes directory grouping with "Last modified" timestamps. Breaking changes require parser updates.

    2. P02 telemetry helpers available: Assumes scripts/p02-telemetry.ps1 exists with New-P02OpenAIUsageEvent and Write-P02AuditEvent functions.

    3. OpenAI API access: Phase 2 requires OpenAI API key (gpt-4.1 model preferred). Cost estimates: \(0.03-\)0.06 per 1K tokens.

    4. Git workflow: Assumes developer will manually review and commit generated docs (no auto-commit in Phase 1-2).

    5. Template maintenance: Someone must update templates when EVA Suite doc standards evolve.

    6. 21-day window sufficient: Assumes most documentation needs arise from recent work (not years-old orphaned files).

    Open questions¶

    1. Q: Should generator auto-create GitHub issues for detected gaps?
      A: Defer to Phase 3. Phase 1 focuses on local reports, Phase 3 adds GitHub API integration.

    2. Q: How to handle confidential/sensitive files?
      A: Add exclude patterns (e.g., .env, secrets/, private/). Use .gitignore as heuristic.

    3. Q: What's the approval process for AI-generated docs?
      A: TBD. Options: PR review, dedicated "doc-review" label, Marco's manual inspection.

    4. Q: Should generator run automatically in GitHub Actions?
      A: Yes, but only for gap detection (Phase 2). AI generation remains manual (Phase 3) due to cost/quality concerns.

    5. Q: How to measure documentation quality?
      A: Metrics TBD. Candidates: Flesch reading score, completeness checklist (sections present), peer review rating.

    6. Q: Can generator update existing docs or only create new ones?
      A: Phase 1-2: Create only (append mode). Phase 3: Explore update mode with diff tracking.

    7. Q: How to handle multi-language support (beyond English)?
      A: Out of scope for Phase 1-2. French translation could be Phase 4 (leverage existing i18n patterns from eva-suite).

    • ⏳ OpenAI API Key: Required for Phase 2 AI-assisted generation

    • P02 Telemetry (scripts/p02-telemetry.ps1): Must support logging generator's OpenAI usage

    Phase 2 (Sprint 03): AI-Assisted Generation - [ ] OpenAI integration working (gpt-4.1 model) - [ ] Context builder reads existing files for style/tone matching - [ ] P02 telemetry logs ALL generator OpenAI usage (100% compliance) - [ ] Generated docs pass human review (80% usable without edits) - [ ] p02-usage-report.ps1 includes P02-ARTIFACTS-001 data - [ ] Power BI dashboard shows generator usage trends

    Phase 3 (Sprint 04+): Multi-Repo & Automation - [ ] Bootstrap script extends generator to 5+ EVA Suite repos - [ ] GitHub Actions workflow runs nightly scans (gap detection only) - [ ] Cross-repo aggregation (suite-wide gap report) - [ ] Real-time suggestions (file watcher + notifications) - [ ] Knowledge graph links docs to code
