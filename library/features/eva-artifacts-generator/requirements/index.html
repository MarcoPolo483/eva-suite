    Requirements – EVA Artifacts Generator¶

    Feature ID: P02-ARTIFACTS-001
    Last updated: 2025-11-25

    Functional requirements¶

    FR-1: Parse EVA-New-Artifacts-Map.md¶

    Priority: MUST
    Status: ⏳ Not implemented

    The system must parse EVA-New-Artifacts-Map.md to extract list of recently modified files (21-day window) with their last modified timestamps.

    Acceptance criteria: - [ ] Reads Markdown file at docs/_vault/EVA-New-Artifacts-Map.md - [ ] Parses directory headers (e.g., ## eva-orchestrator, ## eva-orchestrator\.github) - [ ] Extracts file paths and timestamps (format: - path *(Last modified: YYYY-MM-DD HH:mm)*) - [ ] Returns structured data (array of objects: {path, lastModified, directory}) - [ ] Handles malformed entries gracefully (logs warning, continues parsing) - [ ] Completes parsing in <2 seconds for 1,000 file entries

    Schema (output): 

    @(
    @{
        path = "eva-orchestrator\scripts\eva-audit-update.ps1"
        lastModified = [datetime]"2025-11-25 05:28"
        directory = "eva-orchestrator\scripts"
    },
    @{
        path = "eva-orchestrator\docs\features\eva-audit-trail\README.md"
        lastModified = [datetime]"2025-11-24 14:30"
        directory = "eva-orchestrator\docs\features\eva-audit-trail"
    }
)

    Test scenarios: See tests.md #T-001, #T-002

    FR-2: Parse EVA-Audit-Log.jsonl (optional)¶

    Priority: SHOULD
    Status: ⏳ Not implemented

    The system should optionally parse EVA-Audit-Log.jsonl for additional metadata (change type, file size).

    Acceptance criteria: - [ ] Reads JSONL file at docs/_audit/EVA-Audit-Log.jsonl - [ ] Parses each line as JSON object - [ ] Filters to events within specified date range (default: last 21 days) - [ ] Returns structured data: {scanTime, filePath, lastWriteTime, sizeBytes, changeType} - [ ] Handles corrupted lines gracefully (logs warning, skips line) - [ ] Completes parsing in <5 seconds for 10,000 events

    Use case: Complementary data to EVA-New-Artifacts-Map.md (e.g., detect "new" vs "modified" files).

    Test scenarios: See tests.md #T-003

    FR-3: Detect missing documentation gaps¶

    Priority: MUST
    Status: ⏳ Not implemented

    The system must analyze parsed file lists to identify documentation gaps based on configurable rules.

    Acceptance criteria: - [ ] Rule 1: Scripts in scripts/ without accompanying README.md or test files - [ ] Rule 2: Feature directories in docs/features/ missing P02 package files (requirements.md, backlog.md, tests.md, etc.) - [ ] Rule 3: CDDs in docs/EVA-2.0/CDDs/ without status tracking (missing "Status:" header) - [ ] Rule 4: PowerShell scripts missing help blocks (no .SYNOPSIS, .DESCRIPTION) - [ ] Rule 5: TypeScript files missing JSDoc comments (configurable per-repo) - [ ] Rule 6: New features (last 7 days) without corresponding backlog entries - [ ] Rules configurable via JSON or YAML file (e.g., eva-artifacts-rules.json) - [ ] Reports gaps in structured format (Markdown table + JSON export)

    Gap report schema: 

    {
  "scanDate": "2025-11-25T14:30:00Z",
  "totalFiles": 127,
  "gaps": [
    {
      "category": "MissingREADME",
      "filePath": "scripts/eva-audit-bootstrap.ps1",
      "severity": "medium",
      "recommendation": "Create scripts/README-eva-audit-bootstrap.md"
    },
    {
      "category": "IncompleteP02",
      "filePath": "docs/features/eva-safety/",
      "severity": "high",
      "missingFiles": ["tests.md", "backlog.md"],
      "recommendation": "Complete P02 package (8/10 files present)"
    }
  ]
}

    Test scenarios: See tests.md #T-004, #T-005, #T-006

    FR-4: Generate documentation stubs from templates¶

    Priority: MUST
    Status: ⏳ Not implemented

    The system must generate documentation stubs using predefined Markdown templates.

    Acceptance criteria: - [ ] Template library at scripts/templates/ with 3+ doc types: - README.md.tmpl (script documentation) - P02-Package.tmpl (10-file skeleton) - CDD.tmpl (Copilot Design Document outline) - [ ] Templates use placeholder syntax (e.g., {{FILE_NAME}}, {{DATE}}, {{AUTHOR}}) - [ ] Generator replaces placeholders with actual values - [ ] Outputs stub file to specified location (e.g., scripts/README-eva-audit-bootstrap.md) - [ ] Does NOT overwrite existing files (fails with error if target exists) - [ ] Logs generation event to stdout (timestamp, template used, output path)

    Template example (README.md.tmpl): 

    # {{FILE_NAME}}

**Created**: {{DATE}}  
**Author**: {{AUTHOR}}  
**Feature**: {{FEATURE_KEY}}

## Purpose

[TODO: Describe what this script does]

## Usage

\```powershell
{{COMMAND_EXAMPLE}}
\```

## Parameters

- **Parameter1**: [TODO: Description]

## Examples

[TODO: Add examples]

## Related

- [Feature docs](./docs/features/{{FEATURE_SLUG}}/)
- [Backlog](./docs/backlog/{{FEATURE_SLUG}}.md)

    Test scenarios: See tests.md #T-007, #T-008

    FR-5: AI-assisted documentation generation (Phase 2)¶

    Priority: SHOULD
    Status: ⏳ Not implemented (Phase 2)

    The system should use OpenAI API (gpt-4.1) to generate comprehensive documentation based on code analysis.

    Acceptance criteria: - [ ] Reads target file(s) to build context (e.g., script source code, feature folder contents) - [ ] Constructs OpenAI prompt with: - Doc type (README, P02 requirements, CDD) - Target file content (max 50 KB) - Style guide / example docs for tone matching - [ ] Calls OpenAI API with appropriate parameters (temperature=0.7, max_tokens=4000) - [ ] Parses API response and formats as Markdown - [ ] Writes output to specified location - [ ] Logs OpenAI usage via P02 telemetry (see NFR-2) - [ ] Handles API errors gracefully (retry with backoff, fallback to template stub)

    Context building strategy: - For README: Read target script + related test files - For P02 package: Read feature folder contents + related code files - For CDD: Read architecture docs + existing CDDs for style

    Test scenarios: See tests.md #T-009, #T-010

    FR-6: Generate gap analysis report¶

    Priority: MUST
    Status: ⏳ Not implemented

    The system must generate human-readable Markdown report summarizing detected gaps.

    Acceptance criteria: - [ ] Report saved to docs/_reports/EVA-Artifacts-Gaps-YYYY-MM-DD.md - [ ] Includes header with scan timestamp, total files analyzed, summary statistics - [ ] Groups gaps by category (MissingREADME, IncompleteP02, MissingTests, etc.) - [ ] Sorts gaps by severity (high → medium → low) - [ ] For each gap, includes: file path, severity, recommendation, estimated effort - [ ] Includes summary table (gap count by category) - [ ] Optionally exports to JSON for programmatic consumption (--json flag)

    Report template: 

    # EVA Artifacts Gap Analysis

    FR-7: P02 package bootstrap¶

    Priority: SHOULD
    Status: ⏳ Not implemented (Phase 2)

    The system should generate complete P02 package (10 files) for a new feature using AI assistance.

    Acceptance criteria: - [ ] User specifies target feature (e.g., "eva-safety") - [ ] Generator scans related files (feature folder, code, tests) - [ ] Generates all 10 P02 files: - README.md - P02-Overview.md - requirements.md - feature-map.md - backlog.md - use-cases.md - risks.md - architecture-notes.md - adr-[feature].md - tests.md - [ ] Uses OpenAI for content generation (logs usage via P02 telemetry) - [ ] Maintains consistent style/tone with existing P02 packages (references P02-AUDIT-001 as example) - [ ] Outputs "DRAFT" marker in each file header - [ ] Generates review checklist (10 items, one per file)

    Test scenarios: See tests.md #T-012

    FR-8: Executive summary generation¶

    Priority: COULD
    Status: ⏳ Not implemented (Phase 3)

    The system could generate executive summaries of recent EVA Suite activity for stakeholder updates.

    Acceptance criteria: - [ ] Summarizes last 7/14/30 days of activity (configurable) - [ ] Includes metrics: files changed, features added, tests written, docs updated - [ ] Groups by repo (multi-repo support) - [ ] Highlights top 5 most active areas - [ ] Generates narrative summary (2-3 paragraphs) using AI - [ ] Formats as PowerPoint-ready Markdown or HTML

    Use case: Sprint review prep, executive reporting.

    Test scenarios: See tests.md #T-013

    FR-9: Multi-repo scanning (Phase 3)¶

    Priority: COULD
    Status: ⏳ Not implemented (Phase 3)

    The system could extend scanning to all 23 EVA Suite repositories.

    Acceptance criteria: - [ ] Configurable repo list (JSON/YAML) - [ ] Scans each repo's EVA-New-Artifacts-Map.md (if exists) - [ ] Aggregates gaps across repos - [ ] Generates suite-wide gap report - [ ] Identifies cross-repo documentation inconsistencies

    Test scenarios: See tests.md #T-014

    FR-10: GitHub Actions integration (Phase 3)¶

    Priority: COULD
    Status: ⏳ Not implemented (Phase 3)

    The system could run automatically via GitHub Actions for nightly gap detection.

    Acceptance criteria: - [ ] Workflow triggers: nightly cron (2 AM UTC), manual dispatch - [ ] Runs scanner (gap detection only, no AI generation) - [ ] Commits gap report to docs/_reports/ - [ ] Creates GitHub issue if high-severity gaps detected (optional) - [ ] Uploads gap report as workflow artifact

    Test scenarios: See tests.md #T-015

    Non-functional requirements¶

    NFR-1: Performance¶

    Priority: MUST
    Status: ⏳ Not implemented

    Requirements: - Scanner parses 1,000 files from EVA-New-Artifacts-Map.md in <2 seconds - Gap detection analyzes 1,000 files in <5 seconds - Template-based stub generation completes in <1 second per file - AI-assisted generation completes in <30 seconds per document (depends on OpenAI API latency) - Total end-to-end scan + report generation: <15 seconds for eva-orchestrator

    Rationale: Must be fast enough for interactive use (developer runs before commit).

    Test scenarios: See tests.md #T-016

    NFR-2: Auditability (P02 telemetry)¶

    Priority: MUST
    Status: ⏳ Not implemented (Phase 2)

    Requirements: - ALL OpenAI API calls logged via scripts/p02-telemetry.ps1 - Use New-P02OpenAIUsageEvent helper function - Log schema matches P02 standard: 

    {
  "type": "p02-openai-usage",
  "project": "eva-orchestrator",
  "feature_key": "P02-ARTIFACTS-001",
  "run_id": "uuid-v4",
  "model": "gpt-4.1",
  "input_tokens": 8500,
  "output_tokens": 2400,
  "total_tokens": 10900,
  "cost_estimate": 0.381,
  "caller": "cli",
  "timestamp": "2025-11-25T14:30:00.000Z"
}
    - Events written to docs/_audit/EVA-Audit-Log.jsonl via Write-P02AuditEvent - Generator appears in p02-usage-report.ps1 output (feature_key: P02-ARTIFACTS-001) - Zero OpenAI calls bypassing telemetry (100% compliance)

    Rationale: EVA Suite "dogfoods" its own audit/telemetry infrastructure.

    Test scenarios: See tests.md #T-017

    NFR-3: Traceability¶

    Priority: MUST
    Status: ⏳ Not implemented (Phase 2)

    Requirements: - Each generator run has unique run_id (UUID v4) - Generated files include comment header with: - Generation timestamp - run_id - feature_key (P02-ARTIFACTS-001) - Generator version - Template or AI model used - Example header: 

    <!-- 
Generated by EVA Artifacts Generator
Feature: P02-ARTIFACTS-001
Run ID: a1b2c3d4-e5f6-7890-abcd-ef1234567890
Timestamp: 2025-11-25T14:30:00Z
Method: template (README.md.tmpl v1.0)
-->
    - run_id links generator execution to P02 telemetry events - Enables "which docs were generated by which AI run?" queries

    Rationale: Audit trail for generated content, especially AI-generated.

    Test scenarios: See tests.md #T-018

    NFR-4: Extensibility¶

    Priority: SHOULD
    Status: ⏳ Not implemented

    Requirements: - Modular design: Scanner, Analyzer, Generator, Telemetry as separate components - Plugin architecture for gap detection rules (add new rule types without modifying core) - Template engine supports multiple formats (Markdown, HTML, LaTeX) - Multi-repo support via configuration file (not hardcoded paths) - AI provider abstraction (support OpenAI, Azure OpenAI, local models in future)

    Rationale: EVA Suite grows to 23+ repos; different teams have different doc standards.

    Test scenarios: See tests.md #T-019

    NFR-5: Usability¶

    Priority: SHOULD
    Status: ⏳ Not implemented

    Requirements: - CLI interface with intuitive parameters (e.g., --scan, --generate, --report) - Colorized console output (green=success, yellow=warning, red=error) - Progress indicators for long operations (e.g., "Parsing 1000 files... 50% complete") - Comprehensive help text (--help flag) with examples - Error messages include troubleshooting hints (e.g., "File not found: Check if EVA Audit Trail is running") - Dry-run mode (--dry-run) shows what would be generated without writing files

    Rationale: Developer-friendly tool; must be easy to adopt.

    Test scenarios: See tests.md #T-020

    NFR-6: Security¶

    Priority: MUST
    Status: ⏳ Not implemented (Phase 2)

    Requirements: - OpenAI API key stored securely (environment variable or Azure Key Vault, NOT hardcoded) - Exclude patterns for sensitive files (.env, secrets/, *.key) - AI-generated content includes "DRAFT - REVIEW REQUIRED" warning - No auto-commit of AI-generated docs (human review mandatory) - Rate limiting on OpenAI API calls (max 10 calls/minute to prevent runaway costs) - Validate file paths (prevent directory traversal attacks)

    Rationale: Generator handles code/docs; must not leak secrets or auto-commit untrusted content.

    Test scenarios: See tests.md #T-021

    NFR-7: Cost efficiency¶

    Priority: SHOULD
    Status: ⏳ Not implemented (Phase 2)

    Requirements: - Minimize OpenAI API calls (cache context, reuse prompts) - Provide cost estimates BEFORE running AI generation (e.g., "Estimated cost: $0.45 for 3 docs. Proceed? [Y/n]") - Track cumulative monthly costs via P02 usage report - Alert if monthly cost exceeds threshold (e.g., $100 USD) - Prefer template-based generation over AI when quality difference is minimal

    Rationale: OpenAI costs can escalate; need guardrails.

    Test scenarios: See tests.md #T-022

    NFR-8: Maintainability¶

    Priority: SHOULD
    Status: ⏳ Not implemented

    Requirements: - Code commented with PowerShell help blocks (.SYNOPSIS, .DESCRIPTION, .EXAMPLE) - Modular functions (max 50 lines per function) - Unit tests for core logic (parser, gap detector, template engine) - Integration tests for end-to-end flows - Comprehensive README with architecture diagrams - Template version tracking (update templates without breaking existing docs)

    Rationale: Long-term feature; must be maintainable by future team members.

    Test scenarios: See tests.md #T-023

    Out of scope¶

    The following are explicitly out of scope for Phase 1-3:

    1. Real-time file watching: Generator runs on-demand or scheduled, not triggered by file system events
    2. Auto-commit to Git: All generated docs require human review before commit
    3. Multi-language support: English only (French translation deferred to Phase 4)
    4. Visual documentation: No diagram generation (Mermaid, PlantUML, etc.)
    5. Code generation: Only documentation generation (no source code creation)
    6. Interactive wizard: Command-line only (no GUI/TUI)
    7. Cloud storage: Local files only (no Azure Blob, S3, etc.)
    8. Notification integrations: No Slack, Teams, email alerts (Phase 4)
