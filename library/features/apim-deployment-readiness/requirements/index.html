    Requirements: Azure API Management Deployment¶

    Feature: P02-APIM-DEPLOYMENT Document Type: Requirements Specification Version: 1.0 Last Updated: 2025-12-09 02:17:09 UTC Status: ‚úÖ APPROVED - 100% Production Ready Pattern: Three Concepts (Context Engineering + Housekeeping + Directory Mapping)

    üß© CONTEXT¶

    Mission: Define functional and non-functional requirements for Azure API Management (APIM) Consumption tier deployment to enable per-user cost attribution, rate limiting, and enterprise-grade API gateway capabilities for EVA Suite demo sandbox (25 users), preparing for ESDC-scale deployment (15,000 users).

    Constraints: - Must use Azure Consumption tier (FREE for <1M calls/month) - Must not modify existing backend APIs (eva-auth, eva-api) - Must maintain <50ms latency overhead (p95) - Must support bilingual (EN/FR) developer portal - Must comply with WCAG 2.1 AA standards - Must support ProtectedB data classification

    Reuses: - Existing Azure OpenAI endpoints (canadacentral.api.cognitive.microsoft.com) - Existing Cosmos DB instance (eva-suite-cosmos-dev) - Existing Azure Entra ID tenant - Existing App Services (4 running: eva-auth-dev-app, eva-api-marco-5346, eva-api-container, eva-api-marco-prod)

    Validates: - Backend services running and healthy (DEPLOYMENT-REPORT.md) - Azure subscription has APIM quota available - Entra ID app registrations configured - OpenAPI specifications available for import

    üìë Functional Requirements (FR)¶

    FR-1: API Gateway Deployment ‚≠ê P0¶

    Category: Infrastructure Stakeholders: Marco (PO), POD-F DevOps, P02 (Architect)

    Description: Deploy Azure API Management Consumption tier instance in Canada Central region with enterprise-grade configuration.

    Acceptance Criteria: - [ ] APIM instance created: eva-apim-demo or eva-apim-{environment} - [ ] Location: Canada Central (sovereignty requirement) - [ ] Tier: Consumption (serverless, pay-per-execution) - [ ] Deployment time: <15 minutes - [ ] Gateway URL accessible: https://eva-apim-demo.azure-api.net - [ ] Management API accessible via Azure Portal - [ ] Status endpoint returns HTTP 200

    Test Scenarios (T-FR1-*): 

    Scenario: T-FR1-1 - APIM Instance Deployment
  Given Azure subscription has APIM quota
  When deploying APIM Consumption tier via ARM template
  Then instance created within 15 minutes
  And gateway URL accessible
  And health check returns HTTP 200

Scenario: T-FR1-2 - Region Compliance
  Given APIM instance deployed
  When checking instance location
  Then location equals "Canada Central"
  And data residency compliant with Canadian sovereignty

Scenario: T-FR1-3 - Tier Validation
  Given APIM instance deployed
  When checking pricing tier
  Then tier equals "Consumption"
  And billing model equals "Pay-per-execution"

    Related Docs: APIM-DEMO-SANDBOX-SETUP.md ¬ß3.1

    FR-2: OpenAPI Import ‚≠ê P0¶

    Category: API Management Stakeholders: POD-F DevOps, P12 UI Team

    Description: Import OpenAPI specifications from eva-api and eva-auth services with automatic endpoint discovery.

    Acceptance Criteria: - [ ] eva-api OpenAPI spec imported (endpoints: /graphql, /health, /api/v1/) - [ ] eva-auth OpenAPI spec imported (endpoints: /auth/, /token) - [ ] All endpoints visible in APIM developer portal - [ ] API versioning configured (v1 support, v2 future-ready) - [ ] Import time: <10 minutes per API - [ ] Swagger UI accessible via developer portal - [ ] Interactive API testing enabled

    Test Scenarios (T-FR2-*): 

    Scenario: T-FR2-1 - Eva-API Import
  Given eva-api FastAPI service running
  When exporting OpenAPI spec from /openapi.json
  And importing to APIM
  Then all endpoints listed in APIM portal
  And endpoint count matches source API
  And Swagger documentation accessible

Scenario: T-FR2-2 - Endpoint Validation
  Given APIM import complete
  When querying APIM endpoints list
  Then /graphql endpoint present
  And /health endpoint present
  And /api/v1/* endpoints present
  And WebSocket upgrade supported for /graphql

Scenario: T-FR2-3 - Developer Portal Access
  Given OpenAPI imported
  When accessing developer portal
  Then API documentation visible
  And interactive console enabled
  And code samples available (Python, JS, PowerShell, cURL)

    Source Files: - eva-api/openapi.json (auto-generated from FastAPI) - eva-auth/openapi.json (auto-generated from FastAPI)

    FR-3: Backend Service Configuration ‚≠ê P0¶

    Category: Integration Stakeholders: POD-F DevOps, P15 DVM

    Description: Connect APIM to existing 4 Azure App Services without modifying backend code. Support health checks, timeouts, retry logic, and circuit breakers.

    Acceptance Criteria: - [ ] Backend URLs configured for 4 App Services: - eva-auth-dev-app.azurewebsites.net (auth service) - eva-api-marco-5346.azurewebsites.net (API primary) - eva-api-container.azurewebsites.net (API container) - eva-api-marco-prod.azurewebsites.net (API production) - [ ] Health check endpoints monitored: /health every 30 seconds - [ ] Backend timeout: 30 seconds (configurable per endpoint) - [ ] Retry policy: 3 attempts with exponential backoff (1s, 2s, 4s) - [ ] Circuit breaker: 5 consecutive failures ‚Üí open circuit for 30 seconds - [ ] Load balancing: Round-robin across healthy backends

    Test Scenarios (T-FR3-*): 

    Scenario: T-FR3-1 - Backend Routing
  Given APIM configured with 4 backend services
  When sending request to /api/v1/spaces
  Then request routed to healthy backend
  And response time <500ms p95

Scenario: T-FR3-2 - Health Check Monitoring
  Given APIM health checks enabled
  When backend /health returns HTTP 500
  Then backend marked unhealthy
  And traffic routed to healthy backends
  And alert triggered to POD-F team

Scenario: T-FR3-3 - Timeout Handling
  Given backend timeout set to 30 seconds
  When backend takes >30 seconds to respond
  Then APIM returns HTTP 504 Gateway Timeout
  And request retried on different backend

Scenario: T-FR3-4 - Circuit Breaker
  Given 5 consecutive backend failures
  When circuit breaker opens
  Then new requests fail fast (HTTP 503)
  And circuit remains open for 30 seconds
  And circuit closes after successful health check

    FR-4: Cost Attribution & Usage Tracking ‚≠ê P0¶

    Category: FinOps Stakeholders: Marco, P13 RISK, POD-F DevOps

    Description: Track API usage per user and per department for accurate cost attribution to support chargeback billing.

    Acceptance Criteria: - [ ] User ID captured from JWT token (sub claim) - [ ] Department ID captured from custom claim or X-Department-ID header - [ ] All API calls logged to Application Insights with: - Timestamp (UTC) - User ID - Department ID - Endpoint path - HTTP method - Response status code - Response time (ms) - Tokens consumed (for OpenAI calls - extracted from response) - Estimated cost ($USD) - [ ] Data retention: 90 days in hot storage, 1 year in cold storage - [ ] Export capability: CSV/JSON/Excel for billing - [ ] Real-time dashboard: Cost per user/department

    Test Scenarios (T-FR4-*): 

    Scenario: T-FR4-1 - User Attribution
  Given authenticated user with JWT token
  When making API call to /graphql
  Then user ID logged to Application Insights
  And user ID matches JWT sub claim

Scenario: T-FR4-2 - Department Attribution
  Given request with X-Department-ID: "justice"
  When making API call
  Then department ID logged as "justice"
  And cost allocated to justice department

Scenario: T-FR4-3 - Token Tracking
  Given /chat endpoint call
  When response contains thought_chain.openai_tokens: 1250
  Then token usage extracted and logged
  And cost calculated: tokens * $0.03/1000 = $0.0375

Scenario: T-FR4-4 - Billing Export
  Given 30 days of usage data
  When exporting billing report
  Then CSV contains all required fields
  And totals match Application Insights query
  And grouped by department and user

    Related Docs: APIM-COST-ATTRIBUTION-ESDC.md

    FR-5: Rate Limiting & Quotas ‚≠ê P0¶

    Category: Security / Performance Stakeholders: P13 RISK, POD-F DevOps

    Description: Enforce rate limits to prevent abuse and ensure fair usage across all users.

    Acceptance Criteria: - [ ] Per-user rate limit: 100 requests/minute - [ ] Per-user daily quota: 10,000 requests/day (demo tier) - [ ] Per-IP rate limit: 200 requests/minute (DDoS protection) - [ ] Rate limit headers included in all responses: - X-RateLimit-Limit: 100 - X-RateLimit-Remaining: 47 - X-RateLimit-Reset: 1701234567 (Unix timestamp) - [ ] HTTP 429 (Too Many Requests) returned when exceeded - [ ] Retry-After header provided (seconds until reset) - [ ] Graceful degradation: Queue overflow returns 503

    Test Scenarios (T-FR5-*): 

    Scenario: T-FR5-1 - Rate Limit Enforcement
  Given user rate limit 100 req/min
  When user sends 101 requests in 1 minute
  Then first 100 requests succeed (HTTP 200)
  And 101st request fails (HTTP 429)
  And Retry-After header present

Scenario: T-FR5-2 - Rate Limit Headers
  Given user makes 1st request
  When checking response headers
  Then X-RateLimit-Limit equals 100
  And X-RateLimit-Remaining equals 99
  And X-RateLimit-Reset is future timestamp

Scenario: T-FR5-3 - Daily Quota
  Given user daily quota 10,000 requests
  When user sends 10,001st request in 24 hours
  Then request fails (HTTP 429)
  And error message: "Daily quota exceeded"
  And quota resets at midnight UTC

Scenario: T-FR5-4 - IP-Based Rate Limiting
  Given unauthenticated requests from IP 1.2.3.4
  When IP sends 201 requests in 1 minute
  Then first 200 succeed
  And 201st fails (HTTP 429)
  And applies to all users from that IP

    FR-8: Authentication & Authorization ‚≠ê P0¶

    Category: Security Stakeholders: P13 RISK, POD-F DevOps

    Description: Integrate with Azure Entra ID for OAuth 2.0 / OpenID Connect authentication with role-based access control (RBAC).

    Acceptance Criteria: - [ ] OAuth 2.0 / OpenID Connect configured - [ ] JWT token validation enforced on all authenticated endpoints - [ ] Required claims validated: - sub (user ID) - aud (audience - must match APIM app ID) - iss (issuer - must match Entra ID tenant) - exp (expiration - must be future timestamp) - roles (custom claim for RBAC) - [ ] Role-based access control (RBAC): - Reader: Read-only access (GET endpoints only) - User: Full API access (GET, POST, PUT) - Admin: Portal + API access + analytics dashboard - [ ] Unauthorized requests return HTTP 401 with WWW-Authenticate header - [ ] Forbidden requests return HTTP 403 with clear error message - [ ] Token refresh supported (refresh tokens valid 90 days)

    Test Scenarios (T-FR8-*): 

    Scenario: T-FR8-1 - Valid Token
  Given user with valid JWT token (exp in future)
  When making API call with Authorization: Bearer {token}
  Then request authenticated
  And user ID extracted from sub claim

Scenario: T-FR8-2 - Expired Token
  Given user with expired JWT token
  When making API call
  Then HTTP 401 returned
  And error message: "Token expired"
  And WWW-Authenticate header present

Scenario: T-FR8-3 - RBAC Enforcement
  Given user with role "Reader"
  When making POST request to /api/v1/spaces
  Then HTTP 403 returned
  And error message: "Insufficient permissions"

Scenario: T-FR8-4 - Token Validation
  Given malformed JWT token
  When making API call
  Then HTTP 401 returned
  And error message: "Invalid token signature"

    üîí Non-Functional Requirements (NFR)¶

    NFR-1: Performance ‚≠ê P0¶

    Priority: P0 (Critical) Owner: POD-F DevOps, P02

    Requirements: - Latency overhead: <50ms (p95) - APIM must add <50ms to backend response time - Throughput: 1,000 requests/second sustained (demo tier) - Availability: 99.9% SLA (Consumption tier guarantee) - Backend timeout: 30 seconds (configurable per endpoint) - Connection pooling: Enabled (100 connections per backend) - Caching: Enabled for GET endpoints (5-minute TTL)

    Validation: - Load test with 100 concurrent users ‚Üí measure p95 latency - Verify p95 latency <500ms total (backend 450ms + APIM 50ms) - Verify no dropped requests under 1,000 RPS load - Verify 99.9% uptime over 30 days

    Test Scenarios (T-NFR1-*): 

    Scenario: T-NFR1-1 - Latency Overhead
  Given backend response time measured at 450ms
  When routing through APIM
  Then total response time <500ms p95
  And APIM overhead <50ms

Scenario: T-NFR1-2 - Throughput
  Given load test with 1,000 RPS for 5 minutes
  When measuring request success rate
  Then 100% requests successful
  And error rate <0.1%

    NFR-2: Security ‚≠ê P0¶

    Priority: P0 (Critical) Owner: P13 RISK, POD-F DevOps

    Requirements: - TLS 1.2+ enforced (no TLS 1.0/1.1) - HTTPS only (HTTP auto-redirects to HTTPS) - JWT token validation on all authenticated endpoints - API keys transmitted via Ocp-Apim-Subscription-Key header only (not query string) - Rate limiting enforced per user (100 req/min) - DDoS protection: Azure Front Door (future enhancement) - Secrets stored in Azure Key Vault (not in APIM configuration) - ProtectedB data classification support

    Validation: - SSL Labs scan: A+ rating - OWASP ZAP security scan: Zero critical/high vulnerabilities - Penetration test: Zero exploitable vulnerabilities - Verify expired JWT tokens rejected

    Test Scenarios (T-NFR2-*): 

    Scenario: T-NFR2-1 - TLS Version
  Given APIM gateway endpoint
  When connecting with TLS 1.1 client
  Then connection refused
  And only TLS 1.2+ accepted

Scenario: T-NFR2-2 - Secret Management
  Given APIM configuration
  When inspecting policy XML
  Then zero plaintext secrets present
  And all secrets reference Key Vault

    NFR-3: Scalability ‚≠ê P1¶

    Priority: P1 (High) Owner: P02, POD-F DevOps

    Requirements: - Support 25 users (demo) ‚Üí 15,000 users (ESDC scale) with no architecture changes - Auto-scaling enabled (Consumption tier serverless) - No manual capacity planning required - Linear cost scaling (<$500/month for 15K users) - Support 1M+ requests/month without performance degradation

    Validation: - Load test: 100 ‚Üí 1,000 ‚Üí 10,000 concurrent users - Verify auto-scaling triggers correctly - Verify no performance degradation under increasing load - Verify cost scales linearly with usage

    NFR-4: Compliance ‚≠ê P0¶

    Priority: P0 (Critical) Owner: P13 RISK, Official Languages Compliance

    Requirements: - WCAG 2.1 AA compliance (developer portal) - Bilingual support (EN/FR) - 100% translation coverage - Official Languages Act compliance (Canada) - ProtectedB data classification support - Audit trail for all API calls (90-day retention) - Role-based access control (RBAC) enforced - Privacy by Design principles

    Validation: - WCAG scan: Zero violations (axe, Pa11y) - Translation coverage: 100% verified - Audit trail: 100% API calls logged - RBAC: Verified via security audit
