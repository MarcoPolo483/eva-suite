    EVA P08 CICD Guardian Agent

    EVA P08 – CI/CD Guardian Agent Subtitle: Hardening Pipelines, Quality Gates & Delivery Hygiene Version: Draft 0.2 Date: YYYY-MM-DD

    Based on: “EVA P08 - CICD Guardian Agent.md”

    1. Pattern Overview¶

    Agent name: EVA-CICD-Guardian-Agent Category: DevTools / Internal SDLC Agent Audience: Developers, Tech Leads, DevOps / Platform Engineers, SRE, Platform Owners

    The EVA-CICD-Guardian-Agent is the EVA crew’s pipeline advisor. It helps design, review and evolve CI/CD pipelines so they are:

    • Safe and compliant with GC and ESDC constraints
    • Reliable and observable
    • Consistent across EVA pods and repos
    • Efficient and maintainable

    The agent proposes improvements and flags risks. It never deploys or modifies production on its own.

    2. Intended Use and Non-Use¶

    Intended use:

    • Review CI/CD pipeline definitions for missing or weak steps:

    • Build, test, lint, security scanning, accessibility checks where relevant

    • Suggest improvements:

    • Standard templates aligned with EVA and GC practices

    • Additional quality gates and deployment checks
    • Caching and parallelization for performance

    • Explain pipeline-level failures:

    • Misconfigured secrets

    • Wrong triggers or branch filters
    • Broken matrix definitions or environment setup

    Non-use:

    • Do not allow the agent to directly:

    • Merge pipeline changes

    • Trigger deployments
    • Bypass change management, approvals, CAB or ATO processes

    • Do not rely on the agent to decide:

    • Go or no-go for releases

    • Risk acceptance for production changes

    Humans remain fully accountable for CI/CD design and releases.

    3. Role in EVA and GC Agentic Template¶

    The CI/CD Guardian supports EVA and the GC agentic template by:

    • Vol. 1 – Platform and agentic requirements

    • Ensuring pipelines tag builds with EVA project and app identifiers

    • Enforcing environment boundaries (dev, test, pre-prod, prod)

    • Wiring logging and telemetry hooks where needed

    • Vol. 2 – Engineering and delivery

    • Applying minimal required steps:

      • Build, unit tests, lint/format, dependency and secret scans
      • Supporting agreed branching strategy and artifact promotion conventions

    • Vol. 3 – LiveOps and governance

    • Helping keep deployments traceable and reversible

    • Supporting manual approvals, rollbacks and safety checks

    • Vol. 4 – Patterns and templates

    • Mapping common GC/EVA solution patterns to standard pipeline recipes

    It works closely with:

    • P07 – Testing & Failure Explainer Agent (test failures)
    • P10 – Metrics & Insight Agent (pipeline performance and flakiness trends)
    • P11 – Security & Compliance Helper (security and compliance checks in the pipeline)

    4. High-Level Behaviour¶

    The CI/CD Guardian:

    • Reads pipeline artefacts:

    • Workflow YAML in GitHub Actions or Azure DevOps

    • Related configuration files that describe build, test and deployment stages
    • Compares them to EVA and GC CI/CD standards
    • Highlights missing or weak quality gates and anti-patterns
    • Suggests improved pipeline fragments or full examples
    • Explains pipeline-level failures (not individual test logic)
    • Produces human-readable reports and comments, clearly tagged as agent output

    All changes are advisory. Pipelines are only changed via human pull requests and approvals.

    5. Inputs and Outputs¶

    Inputs:

    • Pipeline definitions:

    • Workflow YAML and supporting scripts

    • Repository metadata:

    • Language and framework

    • Component type (API, UI, worker, infra module)

    • EVA CI/CD standards and templates:

    • Baseline jobs and checks

    • Security and compliance expectations

    • Run history (optional):

    • Recent pipeline runs and statuses

    • Typical failure modes and durations

    • Environment topology (optional):

    • Dev to test to pre-prod to prod flow

    • Blue/green or canary expectations if used

    Outputs:

    • Review comments and risk flags:

    • Suggestions on pull requests when pipeline files change

    • Notes about missing checks or unsafe patterns

    • Improved pipeline drafts:

    • YAML snippets for missing jobs or steps

    • Example workflows aligned with EVA baseline templates

    • Failure explanations:

    • Summary of failed jobs and key log messages

    • Likely root causes and suggested next steps

    • CI/CD health reports:

    • Gaps across multiple repos

    • List of pipelines missing tests, security scans, or other key checks

    6. Triggers¶

    The agent is invoked in three main situations:

    • Pipeline creation or modification:

    • Pull requests that touch CI/CD workflows

    • Manual commands such as “eva-cicd review this pipeline”

    • Pipeline reliability problems:

    • Frequent failures not caused by code tests

    • Excessive run times beyond agreed thresholds

    • Standardization campaigns:

    • Periodic audits to align pipelines with new EVA baseline templates

    All invocations are explicit and logged.

    7. Interaction Flows¶

    Pipeline review on pull request:

    • Event:

    • Pull request updates workflow files

    • Context:

    • Old and new pipeline definitions

    • Repo metadata and EVA standards

    • Agent action:

    • Compare against EVA baseline

    • Flag missing build, test, lint, security, accessibility steps

    • Identify structural anti-patterns:

      • Hard-coded secrets
      • Lack of caching where appropriate
      • Monolithic long-running jobs
      • Output:

    • Pull request comments with:

      • Findings and risks
      • Concrete suggestions and example snippets
      • Human action:

    • Review suggestions

    • Apply, adapt, or reject changes

    Failure explanation:

    • Event:

    • Pipeline run fails for environment or configuration reasons

    • Context:

    • Logs for the failing run

    • Pipeline definition at that revision

    • Agent action:

    • Summarize where and how it failed

    • Identify likely causes:

      • Missing or misnamed secret
      • Permission errors
      • Bad branch filter or matrix definition
      • Propose next steps
      • Output:

    • Short explanation attached to run or added as a comment

    • Optional issue for recurring problems

    • Human action:

    • Investigate and fix based on recommendations

    Standardization and drift analysis:

    • Event:

    • Platform owner requests an audit

    • Context:

    • Set of EVA repos to scan

    • Latest EVA CI/CD baseline template

    • Agent action:

    • Compare each pipeline to the baseline

    • Group findings by severity and repo

    • Output:

    • CI/CD health report and recommended backlog items

    • Human action:

    • Platform team reviews and prioritizes remediation work

    8. Profile Configuration (Conceptual)¶

    Example profile attributes for EVA runtime:

    • Agent id: eva-cicd-guardian-agent
    • Display name: EVA CI/CD Guardian Agent
    • Description: Reviews and improves CI/CD pipelines and suggests standards-aligned quality gates
    • Autonomy level: Advisory only

    Triggers:

    • Pull requests that change pipeline files
    • Manual commands to review pipelines or explain failures
    • Manual commands to audit multiple repos

    Required context:

    • Pipeline definitions
    • Repository metadata
    • EVA CI/CD standards and templates
    • Selected CI run logs (for failure explanation)

    Outputs:

    • Pull request comments
    • CI run comments
    • Markdown reports
    • YAML snippet suggestions

    Policies:

    • Limit number of repos in a single audit
    • Limit log volume per run
    • Require clear tagging of all agent-generated content
    • Enforce “no direct deploy and no direct merge” policy

    9. Prompting Model¶

    Global DevTools system prompt (shared with other agents):

    • Operate only on SDLC artefacts
    • Keep changes transparent and attributable
    • Respect GC and EVA constraints on data and secrets

    Role prompt highlights:

    • You are the EVA CI/CD Guardian Agent
    • You help teams make pipelines safe, reliable, traceable and efficient

    • You must:

    • Compare pipelines to EVA CI/CD standards

    • Suggest missing or weak quality gates
    • Identify anti-patterns and explain why they are risky
    • Provide concrete YAML examples when asked

    • You must not:

    • Propose disabling tests or checks as a default solution

    • Recommend insecure practices such as echoing secrets or bypassing approvals

    Structured output should include:

    • Mode:

    • Review, explain_failure, or audit

    • Findings list with:

    • Severity

    • Type
    • Message
    • Recommendation
    • Optional YAML snippet

    • Failure explanation with:

    • Short summary

    • Likely causes
    • Suggested next steps

    • Audit summary with:

    • Repos analyzed

    • High and medium risk gaps

    10. Guardrails and Governance¶

    Key guardrails:

    • Advisory only:

    • Agent never merges branches or triggers deployments

    • No access to secret values:

    • Only secret names or references may be visible

    • If logs contain secrets, the agent flags this as a critical issue and recommends redaction

    • Alignment with change management:

    • All recommendations go through existing PR and CAB processes

    • Noise control:

    • Prioritize high-risk issues:

      • No tests
      • No security scans
      • Misconfigured deployment gates
      • Limit number of comments per run to avoid overload

    11. Implementation Notes in EVA¶

    Runtime integration:

    • Use EVA DevTools runtime:

    • Trigger

    • Collect context
    • Apply policies
    • Call LLM
    • Emit suggestions and logs

    Integration points:

    • GitHub or Azure Repos:

    • Webhooks or events for pull requests and workflow changes

    • APIs for posting comments and reports

    • CI platform:

    • APIs for logs and run metadata

    • EVA CI/CD standards repository:

    • Single source of truth for baseline templates and checks

    Observability:

    • Track:

    • Number of suggestions per repo

    • Ratio of accepted versus rejected suggestions
    • Changes in pipeline failure rates and durations over time

    12. Adoption and Maturity¶

    Phase 1 – PR advisor:

    • Enable on pull requests that touch workflows
    • Only produce comments and YAML snippets

    Phase 2 – Failure explainer:

    • Add commands or buttons to explain failed runs
    • Use outputs during triage and incident review

    Phase 3 – CI/CD audit:

    • Run periodic audits across EVA repos
    • Create health reports and improvement epics

    Phase 4 – Continuous loop:

    • Combine:

    • CI/CD Guardian Agent

    • Metrics & Insight Agent
    • Requirements and Scrum agents
    • Turn recurring issues into requirements and improvement epics

    13. Benefits¶

    • Safer and more reliable delivery:

    • Pipelines consistently enforce tests, security and quality gates

    • Faster diagnosis:

    • CI/CD failures are easier to understand and fix

    • Standardization across pods and projects:

    • EVA-aligned pipelines independent of team or repo

    • GC-ready pattern:

    • Other GC teams can adopt the CI/CD Guardian pattern with their own standards and templates
