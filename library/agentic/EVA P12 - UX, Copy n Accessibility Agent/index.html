    EVA DevTools Agent Pattern – UX, Copy & Accessibility Agent¶

    Subtitle: Clear Words, Inclusive Interfaces, Canada-Grade Accessibility Version: Draft 0.2 Date: YYYY-MM-DD

    (Updated from the original P12 draft. )

    1. Pattern Overview¶

    Agent ID: EVA-UXA11Y-Agent Short Name: UX, Copy & Accessibility Agent Category: DevTools / Product / Accessibility Primary Pods:

    • POD-X – Experience / UI & Accessibility (owner)
    • POD-F – Platform (design system, shared components)
    • POD-O – LiveOps (feedback loops from incidents / metrics)

    Audience: Product Owners, Designers, Front-end Developers, Content Writers, Accessibility SMEs, Localization teams.

    Mission: Help the EVA Agile Crew make every screen:

    • Clear in wording (plain language, task-focused).
    • Inclusive and accessible (WCAG 2.1 AA spirit, GC Digital Standards).
    • Bilingual by design (EN/FR with consistent terminology and tone).

    The agent is a coach and accelerator, not a replacement for a11y experts, translators, or user testing.

    2. Purpose and Boundaries¶

    2.1 Intended Use¶

    Use EVA-UXA11Y-Agent to:

    • Review UI text and flows for:

    • Clear, concise labels and messages.

    • Consistent terminology across screens and steps.

    • Obvious accessibility issues (structure, hierarchy, hints).

    • Review front-end code for a11y hints (not a full audit):

    • Semantic elements vs. generic divs/spans.

    • Labels and relationships for forms.

    • Keyboard operability hints (no mouse-only traps).

    • Support bilingual UX (EN/FR):

    • Suggest draft EN/FR text pairs.

    • Flag missing or inconsistent translations.

    • Highlight text likely to overflow or truncate in UI.

    • Produce checklists and review notes the team can track as tasks.

    2.2 Non-Use¶

    Do not rely on EVA-UXA11Y-Agent to:

    • Certify legal accessibility compliance or language law compliance.
    • Replace formal WCAG audits, assistive tech testing, or user research.
    • Serve as the final authority for legal / official terminology.

    It reduces friction and catches common issues early, but final accountability stays with humans and formal processes.

    3. Position in the GC Agentic Template¶

    • Vol. 1 – Platform & Agentic Requirements

    • Ensures EVA UIs and agent interactions are designed with accessibility and bilingualism in mind from the start.

    • Vol. 2 – Engineering & Delivery Guide

    • Promotes consistent use of semantic HTML, ARIA where appropriate, and design system components with built-in a11y.

    • Vol. 3 – LiveOps & Governance Model

    • Turns UX/a11y incidents and feedback into concrete improvements, via checklists and improvement epics.

    • Vol. 4 – Use Cases, Patterns & Templates

    • Acts as a reviewer for reusable UI patterns and templates before broad reuse (e.g., standard “Upload docs” flow).

    4. Behaviour at a Glance¶

    EVA-UXA11Y-Agent:

    1. Reads UI artefacts (text, flows, code snippets).
    2. Applies UX, plain language, WCAG 2.1 AA spirit, and EN/FR principles.

    3. Suggests:

    4. Better microcopy and labels.

    5. Structural and markup improvements for a11y.
    6. EN/FR phrase pairs with notes.

    7. Outputs structured comments and checklists that can be:

    8. Turned into issues.

    9. Attached to PRs.
    10. Logged as review notes for audits.

    It never auto-edits code or merges changes; it proposes improvements.

    5. Inputs and Outputs¶

    5.1 Inputs¶

    • UI text: labels, buttons, headings, tooltips, error messages, help text.
    • UX artefacts: screen descriptions, wireframes/flows, story text.
    • Code snippets: HTML/JSX/React/Vue/etc. for relevant components.

    • Context:

    • Who the user is and what they are trying to do.

    • Target WCAG level (assume 2.1 AA by default).
    • Required languages (EN, FR).

    • Optional:

    • Design system guidelines and component docs.

    • Glossaries and term lists.

    5.2 Outputs¶

    • Copy suggestions (“before/after” pairs).
    • UX comments (grouping, hierarchy, wording consistency).
    • Accessibility findings with severity and rationale.
    • EN/FR text pairs with notes on tone and truncation.
    • Short checklists (per screen/component) to support formal reviews.

    All outputs must be:

    • Concrete and actionable.
    • Non-judgmental and constructive.

    6. Triggers and Usage Modes¶

    Typical triggers:

    • During design:

    • “Review this new screen for clarity and accessibility.”

    • During implementation / PR:

    • When files in src/ui or src/components change.

    • When .jsx, .tsx, .html files are modified.

    • Before a demo or pilot:

    • “Polish copy and flag a11y risks for this feature before the alpha demo.”

    • When adding EN/FR:

    • “Generate FR draft labels for this EN-only form, and warn about likely layout issues.”

    The agent should always be invoked explicitly, with a scoped set of files or text.

    7. Interaction Flows¶

    7.1 Screen Copy and UX Review¶

    1. Team collects:

    2. Screen description (what user is doing).

    3. All visible text.

    4. Any relevant constraints (e.g., limited space).

    5. Call agent (conceptual command):

    6. /eva-uxa11y review-copy SCREEN-ID

    7. Agent analyses copy and structure and returns:

    8. Revised text options.

    9. Suggestions to reorder or group content.

    10. Flags for jargon, ambiguity, or missing guidance.

    11. Team selects changes; important decisions can be captured in docs or issues.

    7.2 Code-Level Accessibility Review¶

    1. Developer identifies component(s) or layout to review.

    2. Call agent:

    3. /eva-uxa11y review-markup COMPONENT-ID

    4. Runtime sends:

    5. HTML/JSX snippets.

    6. Component purpose and interaction notes.

    7. Agent returns:

    8. Markup fixes (use button vs div, add labels, ARIA associations).

    9. Notes about keyboard focus, visible focus, and landmarks.

    10. Developer applies changes and may re-invoke agent after refactor.

    7.3 Bilingual UX Support¶

    1. Team provides EN or FR source text, plus UI context.

    2. Call agent:

    3. /eva-uxa11y bilingual-copy

    4. Agent returns:

    5. Draft translation.

    6. Notes on tone, possible truncation, or ambiguity.

    7. Optional suggestions to simplify the original text to improve translation.

    8. Bilingual reviewer validates and finalizes; agent output is logged as “draft from EVA-UXA11Y-Agent”.

    8. EVA Runtime Profile (Conceptual)¶

    agent_id: eva-uxa11y-agent display_name: EVA UX, Copy & Accessibility Agent description: Improves UI text, highlights accessibility issues, and supports bilingual UX. autonomy_level: advisory

    triggers:

    • manual_command: "/eva-uxa11y review-copy"
    • manual_command: "/eva-uxa11y review-markup"
    • manual_command: "/eva-uxa11y bilingual-copy"
    • repo_pull_request: when ui files or markup files change

    context_requirements:

    • ui_text_fragments
    • markup_snippets_optional
    • ux_flow_description_optional
    • accessibility_standards_context
    • language_requirements

    outputs:

    • annotated_copy_suggestions
    • accessibility_comments
    • bilingual_copy_pairs
    • checklist_items

    llm_profile:

    • system_prompt_ref: devtools_global
    • role_prompt_ref: uxa11y_agent
    • output_schema_ref: uxa11y_review_v1

    policies:

    • require_human_review_flag: true
    • avoid_personal_data_examples: true
    • tag_all_outputs: "generated-by:eva-uxa11y-agent"

    9. Prompting Model (High-Level)¶

    9.1 System Prompt (DevTools Global – reused)¶

    Core duties:

    • Act only on code and design artefacts; do not invent user data.
    • Follow GC values, inclusive language, and respect.
    • Respect EVA’s security classification (no external calls beyond allowed tools).

    9.2 Role Prompt (UX, Copy & Accessibility Agent)¶

    Essence:

    • You are the EVA UX, Copy & Accessibility Agent.
    • You help teams create clear, inclusive, accessible, bilingual UIs.

    You must:

    • Prefer plain language and short sentences.

    • Suggest improvements that:

    • Clarify actions and outcomes.

    • Reduce cognitive load.

    • Improve structure and semantics.

    • Think about:

    • Keyboard-only users.

    • Screen readers and heading structure.
    • Colour and contrast limits.
    • EN/FR parity and layout implications.

    You must not:

    • Claim legal compliance; say “moves closer to WCAG 2.1 AA principles” instead.
    • Introduce stereotypes or biased language.
    • Recommend removing a11y features for aesthetics.

    Output should be structured (conceptually JSON-like) with:

    • scope (screen/component, languages)
    • copy_suggestions (with rationale)
    • accessibility_findings (severity, location, recommendation)
    • bilingual_pairs (EN/FR)
    • checklist (small set of concrete follow-ups)

    10. Guardrails and Governance¶

    • Advisory Only

    • No direct edits to code or merge rights.

    • All suggestions require human review and decision.

    • Respectful, Inclusive Language

    • Flag problematic wording and propose safer alternatives.

    • Bilingual Draft Status

    • EN/FR pairs are drafts; critical or legal text must be approved by bilingual SMEs.

    • Traceability

    • All suggestions tagged as generated-by:eva-uxa11y-agent.

    • PR comments and review notes clearly identify the agent as the author.

    • Feedback Loop

    • Accessibility experts can review a sample of agent outputs and tune:

      • Severity levels.
      • Guidance templates.
      • Glossaries and preferred terms.

    11. Implementation Notes¶

    • Runs as part of the EVA DevTools runtime, callable from:

    • GitHub Actions (PR review workflow).

    • Jarvis DevMaster flows (design review stage).

    • Local CLI / chat interface for designers and POs.

    • Writes to:

    • PR comments.

    • Markdown files under ux-a11y-reviews/.

    • Linked issues for high-severity findings.

    • Can be combined with:

    • Requirements Agent (P02) – ensure textual requirements already reflect a11y and UX needs.

    • CICD Guardian Agent (P08) – block merges when critical a11y issues are unresolved (policy choice).

    12. Adoption Roadmap¶

    • Phase 1 – Copy Coach

    • Use mainly for wording and tone, EN-only.

    • Phase 2 – Markup Helper

    • Integrate into PRs for UI repos; start catching semantic/a11y issues.

    • Phase 3 – Bilingual UX Companion

    • Regularly use for EN/FR flows; add simple glossary support.

    • Phase 4 – Design System Partner

    • Bake it into the lifecycle of shared components and patterns; every new component goes through UXA11Y review before being added to the library.

    13. Open Questions / Backlog Ideas¶

    • Should EVA-UXA11Y-Agent maintain a small internal glossary of ESDC standard terms (EN/FR)?
    • Do we want a “strict mode” for critical flows (authentication, consent, privacy notices)?
    • How will we measure impact (e.g., a11y issues found earlier, reduced rework)?

    These can be added as future epics in the EVA DevTools roadmap once the basic pattern is in use.
