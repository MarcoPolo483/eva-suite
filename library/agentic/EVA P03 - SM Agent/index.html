    EVA P03 SM Agent

    EVA DevTools Agent Pattern – Scrum Agent Subtitle: Backlog Refinement and Ceremony Support for the EVA Agile Crew Pattern ID: P03 – Scrum Agent Version: Draft 0.3 Date: YYYY-MM-DD

    1. Pattern Overview¶

    Agent name: EVA Scrum Agent Category: DevTools internal SDLC agent Audience: Product Owners, Scrum Masters or Flow Facilitators, Tech Leads, Jarvis DevMaster, developers

    The EVA Scrum Agent supports the EVA Agile Crew in running Scrum or flow-based delivery. It:

    • Helps draft and refine epics, user stories and acceptance criteria.
    • Helps prepare sprint or iteration goals, summaries and release notes.
    • Keeps backlog items aligned with EVA platform standards and GC agentic templates.
    • Integrates with Jarvis workflows (refine, plan, review) as the “text brain” for backlog quality.

    The agent only proposes text. It never changes issue status, priority, assignments or sprint membership by itself.

    2. Intended Use and Non-Use¶

    Intended use:

    • Turn rough notes into clean backlog items.
    • Refine stories so they meet Definition of Ready.
    • Propose acceptance criteria, test ideas and edge cases.
    • Draft sprint summaries, release notes and stakeholder-friendly descriptions.
    • Keep story wording consistent with EVA and GC context, including human-in-the-loop, privacy, safety, accessibility and bilingual requirements.
    • Support Jarvis when it runs refinement and planning workflows.

    Non-use:

    • Deciding priorities, scope, estimates or sprint commitments.
    • Running ceremonies autonomously or assigning work.
    • Changing item status, sprint membership or due dates.
    • Overriding decisions from governance bodies such as AI Review panels or architecture boards.

    Humans stay accountable for what is built, when it is delivered and how trade-offs are made.

    3. Link to GC Agentic Template and EVA Suite¶

    The Scrum Agent is one of the DevTools patterns in the P01 toolbox and connects the GC Agentic volumes to daily work.

    • Volume 0 (overview): it helps translate high-level adoption guidance into concrete backlog items.
    • Volume 1 (platform and agentic requirements): it ensures stories referencing agents and EVA platform features include governance hooks such as logging, kill switches, auditability, RBAC and classification tags where relevant.
    • Volume 2 (engineering and delivery): it reinforces DoR and DoD, and prompts for technical and non-functional stories.
    • Volume 3 (LiveOps and governance): it reminds teams to include monitoring, metrics, safety and lifecycle considerations.
    • Volume 4 (patterns and templates): it turns pattern descriptions (P01 to P15) into actionable stories and tasks.

    Within EVA Suite, the Scrum Agent:

    • Works alongside the Requirements Agent (P02 REQ) and DevMaster Orchestrator (P15 DVM).
    • Produces refined text that is stored in eva-meta sprint files, reports and orchestration logs.
    • Helps keep backlog quality consistent across pods (POD-F, POD-X, POD-O, POD-S) and repos.

    4. High-Level Behaviour¶

    The Scrum Agent:

    • Reads backlog items and notes such as epics, stories, tasks, labels, basic metadata and linked documents.
    • Proposes clearer titles, descriptions, acceptance criteria and test ideas.
    • Suggests how a large epic could be broken down into smaller stories.
    • Drafts sprint summaries, review notes and release highlights.
    • Identifies questions, risks and missing information and calls them out explicitly.
    • Marks all output as draft content that requires human review.

    Typical operations:

    • Refine a single story.
    • Split an epic into candidate stories.
    • Polish a set of stories selected for a sprint.
    • Summarize a sprint based on completed items.

    5. Inputs and Outputs¶

    Inputs:

    • Backlog items: title, description, labels, type, status, links to epics or related issues.
    • Team standards: Definition of Ready, Definition of Done, story templates and naming conventions.
    • EVA standards: guidance on human-in-the-loop, privacy, security, accessibility, EN and FR, LiveOps expectations.
    • Context documents: CDDs, architecture notes, UX flows, metrics plans.
    • Sprint configuration: when Jarvis runs, the sprint YAML file listing items in scope.
    • Optional meeting notes: rough text from grooming, planning or retrospectives.

    Outputs:

    • Refined story descriptions and titles.
    • Acceptance criteria lists.
    • Test ideas and edge cases.
    • Suggestions for child stories for large epics.
    • Sprint or release summaries for stakeholders.
    • Lists of questions and risks to be resolved.

    These outputs appear as:

    • Draft edits in story descriptions and acceptance criteria fields.
    • Comments on issues with suggested text.
    • Markdown files in eva-meta reports or docs folders.
    • Structured payloads for Jarvis to log and link back to.

    6. Triggers¶

    The Scrum Agent can be triggered in several ways.

    Manual in backlog tools:

    • A user adds a comment such as “Refine this story” or presses a “Refine with EVA” action.
    • A user requests “Split this epic into candidate stories.”
    • A user asks “Summarize Sprint N” on a sprint board.

    During ceremonies:

    • During backlog refinement, the facilitator highlights a set of items and calls the agent to propose improved wording and acceptance criteria.
    • During sprint demo or review, the facilitator calls the agent to generate a first draft of the sprint summary.

    Jarvis workflows:

    • jarvis-refine: runs Scrum Agent and Requirements Agent against issues marked as needing refinement in a sprint YAML file.
    • jarvis-plan: uses Scrum Agent to propose a sprint goal statement and a summary text.
    • jarvis-review: uses Scrum Agent to prepare sprint review text for the reports folder.

    All triggers are explicit and visible; there are no hidden background changes.

    7. Interaction Flows¶

    Story refinement flow:

    • Human selects a story that is unclear or incomplete.
    • Scrum Agent is invoked for that story.
    • The runtime gathers context from the backlog item, linked epic, design notes and standards.
    • The agent proposes improved title and description, acceptance criteria, test ideas, questions and risks.
    • The Product Owner or developer reviews and edits these suggestions and then confirms the final text.

    Epic splitting flow:

    • Human selects an epic that is too large or vague.
    • Scrum Agent is invoked for that epic.
    • Context is gathered from epic description, checklists, linked patterns and constraints.
    • The agent proposes a small set of candidate stories with short descriptions and optional hints about pods or repos.
    • The team reviews, renames, merges or discards proposals and creates final stories.

    Sprint summary flow:

    • Human or Jarvis selects a sprint with a list of done items.
    • Scrum Agent is invoked to create a summary.
    • The runtime gathers done items, labels and relevant notes.
    • The agent generates a short sprint narrative, a list of key changes and optionally a “what is next” section.
    • Scrum Master or Product Owner edits the summary and publishes it.
    • Jarvis stores the final text in the sprint reports.

    8. Configuration in EVA Runtime¶

    Configuration goals:

    • The agent is registered under an identifier such as eva-scrum-agent.
    • Autonomy level is advisory only; it cannot change backlog fields directly.
    • Triggers include backlog commands and Jarvis workflow invocations.
    • Required context includes backlog item details, linked items, team standards and sprint metadata.
    • Outputs are configured as drafts for backlog fields, comments, markdown fragments and entries in orchestration logs.
    • A global DevTools system prompt applies EVA policies, and a Scrum-specific role prompt defines its behaviour.

    Key policy points in configuration:

    • Maximum number of items processed per call to keep runs manageable.
    • Limits on acceptance criteria length so stories stay readable.
    • A requirement that all output is flagged as generated by the Scrum Agent.
    • A flag that always requires human review before changes are applied.

    You can store these settings in a DevTools configuration file or service rather than embedding them in code.

    9. Prompting Model¶

    System-level rules (shared DevTools system prompt):

    • Operate only on SDLC artefacts and EVA documentation.
    • Respect ESDC and GC policies on data classification and AI use.
    • Do not invent policies or commitments; highlight assumptions and uncertainties.
    • Provide clear, concise, practically useful text.

    Role-specific rules for the Scrum Agent:

    • Focus on clarity, testability and alignment with Definition of Ready.
    • Use the team’s story template consistently.
    • Always propose acceptance criteria when refining a story.
    • Call out missing information, questions and risks rather than guessing.
    • Highlight missing governance, monitoring, safety, accessibility or bilingual aspects when they are obviously relevant.
    • Never change items’ status, priority or assignment and never promise dates or capacity.

    The runtime ensures outputs follow a simple structure that Jarvis can parse, for example a list of refined items plus a short summary, without requiring any specific serialization format in this document.

    10. Guardrails and Governance¶

    Guardrails:

    • The Scrum Agent only produces draft text. Any change to backlog fields must be applied by a human or by a separate, explicit “apply changes” action.
    • All suggestions are clearly attributed, using a visible marker such as a tag, label or comment line indicating that content was generated by the Scrum Agent.
    • The agent does not call APIs that change workflow state, such as transitions, assignments or sprint moves.
    • The agent cannot approve or merge code; that belongs to Review and DevMaster patterns.

    Audit and logging:

    • Every call records who invoked the agent, which items were processed and when.
    • Jarvis workflows copy a high-level summary of agent activity into orchestration logs.
    • Logs avoid unnecessary personal or sensitive data and focus on IDs, timestamps and high-level actions.

    Accountability:

    • Scrum Masters and Product Owners are responsible for final backlog content.
    • They must especially review stories that touch privacy, security, LiveOps or external stakeholder communication.

    11. Implementation in EVA and GitHub¶

    Within EVA and GitHub:

    • The Scrum Agent runs inside the DevTools runtime and is called from:

    • Backlog tools such as GitHub Issues or Azure Boards.

    • Jarvis workflows in eva-meta.

    • Optionally an EVA DA “DevTools panel” for teams.

    • Integration patterns:

    • Backlog comments or slash-style commands trigger a small backend service that passes context to DevTools.

    • Jarvis workflows load sprint YAML, fetch referenced issues through the GitHub API, pass them to DevTools and then post comments or draft text back.

    • Orchestration logs and reports in eva-meta link to the items that were refined.

    • Traceability across repos and pods:

    • The agent includes repo and pod information in its logs when available (for example from .eva-pod.yaml).

    • Jarvis uses this to build cross-pod views of backlog quality and refinement progress.

    12. Adoption and Maturity Levels¶

    Maturity level 1: text helper

    • Use the Scrum Agent as a simple writing assistant.
    • Developers and Product Owners paste stories into it and reuse improved wording and acceptance criteria manually.

    Maturity level 2: backlog companion

    • Integrate the agent into GitHub or the backlog tool so it can act on issues in place.
    • Use it during grooming and planning to keep stories aligned with DoR.
    • Jarvis leverages it during jarvis-refine to prepare sprint-ready stories.

    Maturity level 3: ceremony companion and reporting

    • The agent is used live during refinement and review sessions.
    • Sprint summaries and release notes are routinely drafted by the agent and then edited by humans.
    • Jarvis regularly calls the agent to keep sprint reports and orchestration logs up to date.

    In all stages, humans make the decisions; the agent accelerates the writing and standardization work.

    13. Benefits¶

    • Improved backlog quality: clearer, more testable stories across EVA pods.
    • Time savings: less effort spent on wordsmithing and more time for design and delivery.
    • Better alignment: backlog items naturally reflect GC and EVA requirements for safety, governance, accessibility and bilingualism.
    • Scalability: consistent backlog practices across many pods and repos, supported by Jarvis and DevTools rather than manual policing.
