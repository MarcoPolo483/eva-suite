    EVA DevTools Agents – Runtime, Catalog & Profiles¶

    Subtitle: Internal AI Agents Supporting the EVA Agile Crew
    Version: Draft 0.2
    Date: YYYY-MM-DD

    1. Purpose & Context¶

    This document defines the EVA DevTools Agents – a family of internal AI agents that assist the EVA Agile Crew in designing, building, and operating the EVA Platform.

    They are:

    • For developers and platform teams only (not business users / case workers).
    • Intended to improve speed, consistency, and quality of EVA development.
    • Governed under the same GC-grade principles as other EVA agents (Vol. 0–4 + EVA Agile Crew Charter).

    DevTools agents are helpers inside the SDLC. They:

    • Read from repos, backlogs, logs, and docs.
    • Write suggestions, comments, and drafts – never merge code or change production config autonomously.
    • Are fully traceable and controllable (logs, kill switch, scopes).

    All DevTools agents share a single runtime pipeline (“dev agent routine”) and differ only by their role profile (Review, Scrum, Scaffolder, etc.).

    2. Common Principles & Constraints¶

    All EVA DevTools Agents share these properties:

    1. Internal Only
    2. Accessible only to EVA Agile Crew and designated technical collaborators.

    3. Operate on development artefacts (code, docs, tickets, logs).

    4. Advisory / Assistive Autonomy Only

    5. They draft content, comment on PRs, propose structures and fixes.

    6. They do not:

      • Merge to main branches.
      • Change production config.
      • Modify security or access control.

    7. Visible & Traceable Actions

    8. Every action leaves a visible footprint:
      • PR comments under an agent identity.
      • Draft doc files / suggestions clearly marked.

    9. All actions are logged (see Section 8).

    10. Scope-Limited Access

    11. Each agent has the minimum access needed for its job:

      • Repo read-only or read/write.
      • Issue tracker read/write for specific projects.
      • Observability platform read-only.

    12. Human Accountability

    13. Devs and leads remain accountable for:

      • Code correctness.
      • Security and privacy.
      • Adhering to departmental standards.

    14. Kill Switch & Tuning

    15. Each agent can be:
      • Disabled per repo/pod.
      • Globally paused if misbehaving.
    16. Behaviour can be tuned via configuration (verbosity, max comments, etc.).

    3. Shared EVA DevTools Runtime – “Dev Agent Routine”¶

    All DevTools agents (Scrum, Review, Scaffolder, etc.) are roles on top of a single runtime.

    3.1 Standard Pipeline¶

    Every DevTools agent follows this routine:

    1. Trigger → Intent Resolution

    A trigger arrives:

    • GitHub webhook (e.g., pull_request.opened).
    • PR comment command (e.g., /eva-review).
    • Scheduled job (e.g., weekly metrics).
    • Manual chat request.

    The runtime maps the trigger to:

    • agent_id (e.g., eva-review-agent).

    • task_type (e.g., review_pull_request).

    • Context Gathering

    Using agent_id + task_type + configuration, the runtime fetches context:

    • Source control: PR diff, changed files, repo metadata.
    • Issue tracker: linked issues, epic descriptions.
    • Docs: README, ADRs, CONTRIBUTING, runbooks.
    • Policies: Definition of Done, security/a11y notes, coding standards.

    Example structured context object:

       {
     "repo": "eva-ui",
     "pull_request": {
       "number": 123,
       "title": "Add agent catalogue view",
       "diff": "<diff or structured changes>",
       "author": "dev-user"
     },
     "linked_issues": [
       { "id": 456, "title": "Implement agent catalogue view" }
     ],
     "repo_policies": {
       "definition_of_done": "...",
       "languages": ["typescript", "react"],
       "testing_expectations": "unit + basic e2e",
       "accessibility_expectations": "WCAG 2.1 AA"
     },
     "docs": {
       "readme_excerpt": "...",
       "contributing_excerpt": "..."
     }
   }
    1. Policy & Guardrail Check

    Before calling the LLM, the runtime checks:

    • Is DevTools allowed on this repo/project?
    • Is this agent_id enabled (kill switch)?
    • Is the requested task_type within this agent’s autonomy level (advisory/assistive)?

    If not allowed: exit gracefully (no action) and log the reason.

    1. LLM Reasoning (via EVA)

    The runtime calls EVA (agent/LLM endpoint) with:

    • System prompt = global DevTools rules and constraints.
    • Role prompt = specific behaviour for this agent (reviewer vs scrum vs scaffolder).
    • Context = structured context object.
    • Task template = required output format (JSON schema).

    Example call payload:

       {
     "system_prompt": "devtools_global_system_prompt_v1",
     "role_prompt": "review_agent_role_prompt_v1",
     "task": "review_pull_request",
     "context": { ... },
     "output_schema": "review_comments_v1"
   }
    1. Result Validation & Action Planning

    LLM output is parsed and validated against the output schema, for example:

       {
     "comments": [
       {
         "file": "src/components/AgentList.tsx",
         "line": 42,
         "severity": "warning",
         "message": "This component lacks basic unit tests. Consider adding tests for filtering behaviour."
       }
     ],
     "summary": "1 warning, 0 blockers.",
     "overall_severity": "info"
   }

    The runtime enforces:

    • Limits on number of comments.
    • Allowed severity levels.

    • That no actions exceed advisory/assistive scope.

    • Effectors – Writing Outputs

    Based on the agent profile, the runtime:

    • Posts comments on PRs.
    • Creates or updates issues.
    • Writes draft docs (branch or suggestion text).
    • Produces summary reports.

    Outputs are tagged with:

    • author = eva-<agent-name> (bot identity).

    • A text marker such as “(Generated by EVA-Review-Agent)” where appropriate.

    • Logging & Metrics

    The runtime writes a DevTools log entry per action, for example:

       {
     "timestamp": "YYYY-MM-DDTHH:MM:SSZ",
     "agent_name": "eva-review-agent",
     "task_type": "review_pull_request",
     "context_ref": "github:eva-ui#pr:123",
     "result": "success",
     "invoked_by": "github-webhook",
     "num_comments": 3
   }

    Logs go to a DevTools-specific index/table for auditing, measuring usefulness, and tuning.

    3.2 Global DevTools System Prompt (Conceptual)¶

    All DevTools agents share a common system prompt encoding:

    • GC + departmental expectations:
    • Only operate on SDLC artefacts.
    • No hidden changes; actions must be visible and explainable.
    • EVA platform principles:
    • Advisory/assistive only; no autonomous deployments or config changes.
    • Respect DoR/DoD, coding standards, accessibility, bilingualism.
    • Behavioural rules:
    • Be concise and specific.
    • Prefer actionable suggestions with clear “what/where/why”.
    • When unsure, ask for more context instead of fabricating.

    Each agent then adds a role prompt defining its specific function.

    4. DevTools Agent Catalog – Overview¶

    Each DevTools agent is a profile on the shared runtime.

    Agent Name Short Description Primary Users Autonomy
    EVA-Scrum-Agent Helps refine epics, stories, acceptance criteria, notes PO, SM, Tech Lead Advisory
    EVA-Requirements-Agent Recursively refines requirements from idea → stories PO, Tech Lead Advisory
    EVA-Repo-Librarian Keeps READMEs, ADRs, and docs in sync with code All devs, Tech Leads Assistive
    EVA-Scaffolder-Agent Bootstraps repos, services, UI modules, pipelines Devs, Tech Leads Assistive
    EVA-Review-Agent Pre-reviews PRs; flags missing tests/docs/security Devs, Reviewers Advisory
    EVA-Runbook-Agent Drafts/updates runbooks from logs and incidents Ops, SRE, Devs Advisory
    EVA-Metrics-Agent Summarizes metrics; suggests tech/process improvements Platform Owner, Leads Advisory

    5. Agent Profiles – General Structure¶

    An agent profile is a configuration object plus a role prompt.

    Minimal fields:

    • agent_id
    • display_name
    • description
    • autonomy_level (advisory/assistive)
    • triggers
    • context_requirements
    • outputs
    • llm_profile (system/role prompts, output schema)
    • policies (limits and knobs)

    Conceptual example:

    agent_id: eva-some-agent
display_name: "EVA Some Agent"
description: "What this agent does."
autonomy_level: advisory  # or assistive

triggers:
  - type: github_webhook        # or github_comment_command | schedule | manual
    event: pull_request_opened
  - type: github_comment_command
    command: "/eva-some-agent"

context_requirements:
  - pull_request_diff
  - issues
  - repo_policies

outputs:
  - type: github_pr_comments
  - type: github_issues

llm_profile:
  system_prompt_ref: devtools_global
  role_prompt_ref: some_agent_role
  output_schema_ref: schema_v1

policies:
  max_comments: 20
  comment_style: "specific_actionable"

    6. Example Agent Profile – EVA Review Agent¶

    6.1 Purpose¶

    The EVA Review Agent is a first-pass reviewer for pull requests:

    • Checks code changes against EVA’s Definition of Done.
    • Flags missing tests, docs, or obvious issues.
    • Suggests improvements or refactors.
    • Does not approve or merge PRs; it only comments.

    6.2 Configuration (Conceptual YAML)¶

    agent_id: eva-review-agent
display_name: "EVA Review Agent"
description: "First-pass reviewer for pull requests against EVA standards."
autonomy_level: advisory   # Comments only, never merges.

triggers:
  - type: github_webhook
    event: pull_request_opened
  - type: github_webhook
    event: pull_request_synchronize   # optional – when PR is updated
  - type: github_comment_command
    command: "/eva-review"            # manual trigger by a dev

context_requirements:
  - pull_request_diff
  - pull_request_metadata
  - linked_issues
  - repo_policies         # DoD, languages, test strategy, security notes
  - docs_readme
  - docs_contributing

outputs:
  - type: github_pr_comments
  - type: github_pr_summary_comment

llm_profile:
  system_prompt_ref: devtools_global
  role_prompt_ref: review_agent
  output_schema_ref: review_comments_v1

policies:
  max_comments: 20
  min_severity_to_comment: "info"     # info | warning | blocker
  comment_style: "specific_actionable"
  allow_blocker_label: false          # cannot mark "blocker" formally

    6.3 Role Prompt (Conceptual)¶

    The review_agent role prompt defines behaviour such as:

    • You are the EVA Review Agent.
    • You review pull requests for:
    • Missing tests.
    • Missing documentation updates.
    • Violations of coding standards and basic security/accessibility practices.
    • You MUST:
    • Be respectful and concise.
    • Suggest concrete improvements (what to change, where, and why).
    • Respect the repository’s Definition of Done.
    • You MUST NOT:
    • Approve or merge pull requests.
    • Suggest actions outside advisory scope.
    • Expose or fabricate secrets.

    Output must match the review_comments_v1 JSON schema (comments + summary).

    6.4 Runtime Execution Flow for EVA Review Agent¶

    When a PR is opened:

    1. GitHub webhook fires; DevTools Runtime receives pull_request.opened.
    2. Runtime resolves:
    3. agent_id = eva-review-agent

    4. task_type = review_pull_request.

    5. Runtime loads the Review Agent profile and validates that the agent is enabled; it reads context_requirements, outputs, and policies.

    6. Runtime fetches context (PR diff, metadata, linked issues, repo policies, docs excerpts).

    7. Runtime calls EVA with:

    8. system_prompt_ref = devtools_global

    9. role_prompt_ref = review_agent
    10. output_schema_ref = review_comments_v1

    11. Structured context and task type.

    12. EVA returns structured review comments; runtime validates and enforces:

    13. max_comments

    14. min_severity_to_comment

    15. comment_style.

    16. Runtime posts comments on the PR as eva-review-agent:

    17. Inline file comments where possible.

    18. Optional summary comment (e.g., “EVA-Review-Agent summary: 3 suggestions, 0 blockers.”).

    19. Runtime logs the interaction:

    20. agent_name, task_type, context_ref (repo + PR number), result, comment counts.

    Developers and human reviewers then read the agent comments, accept or ignore suggestions, and proceed with standard human review and merge processes.

    7. Other DevTools Agents (Profiles on the Same Runtime)¶

    Other DevTools agents reuse the same pipeline, with different profiles.

    7.1 EVA-Scrum-Agent¶

    • Purpose: Supports backlog refinement and ceremonies.
    • Triggers: Manual commands (e.g., /eva-scrum refine epic-123), planning-tool actions.
    • Context: Epics, stories, notes, architecture docs.
    • Outputs: Refined descriptions, acceptance criteria, draft release notes.

    7.2 EVA-Requirements-Agent¶

    • Purpose: Implements recursive requirements refinement from idea → canvas → epics → stories.
    • Triggers: During discovery / early design.
    • Context: Agent Concept Canvas, platform requirements, governance constraints.
    • Outputs: Epics & capabilities, implementation-ready stories and tasks (tagged as drafts).

    7.3 EVA-Repo-Librarian¶

    • Purpose: Keeps documentation in sync with code and architecture.
    • Triggers: After significant PR merges; scheduled repo scans.
    • Context: Repo structure, docs, ADRs, commit history.
    • Outputs: Doc suggestions, ADR skeletons, README updates (drafts).

    7.4 EVA-Scaffolder-Agent¶

    • Purpose: Provides standardized scaffolding for new repos/modules.
    • Triggers: Explicit commands or UI actions (e.g., “create new EVA UI module”).
    • Context: Chosen templates/blueprints, parameters (name, type, language, pod).
    • Outputs: Folder structures, devcontainer, CI workflows, starter components or endpoints.

    7.5 EVA-Runbook-Agent¶

    • Purpose: Converts operational experience into updated runbooks.
    • Triggers: After incidents, or on schedule.
    • Context: Incidents, logs, alert descriptions, existing runbooks.
    • Outputs: Updated or new runbook sections (draft PRs).

    7.6 EVA-Metrics-Agent¶

    • Purpose: Summarizes delivery and platform health to support improvement.
    • Triggers: Weekly/monthly scheduled jobs or on-demand.
    • Context: PR/issue stats, CI results, incidents.
    • Outputs: Narrative reports, tables, charts, suggested focus areas.

    8. Logging & Observability for DevTools Agents¶

    For each agent action, log at minimum:

    • timestamp
    • agent_name
    • user (who invoked it, if applicable)
    • task_type
    • context_type (PR, issue, doc, incident)
    • context_id (PR number, issue ID, file path)
    • action_type (comment, suggestion, draft-doc, scaffold, report)
    • result (success/failure + error if any)

    These logs:

    • Live in a DevTools-specific table/index.
    • Are used to:
    • Debug behaviour.
    • Evaluate usefulness (e.g., which agents help the most).
    • Provide evidence for governance reviews.

    9. Agent Interaction Patterns¶

    DevTools agents do not maintain private, invisible conversations between themselves.

    Instead, they “interact” via shared artefacts:

    • EVA-Requirements-Agent outputs epics/stories → EVA-Scrum-Agent refines acceptance criteria.
    • EVA-Scaffolder-Agent sets up new repo → EVA-Repo-Librarian suggests doc improvements once code lands.
    • EVA-Review-Agent repeatedly flags missing tests → EVA-Metrics-Agent includes that pattern in its reports.

    This preserves:

    • Transparency – humans see the whole chain.
    • Traceability – we can follow how an idea became code, docs, and runbooks.

    10. Adoption & Maturity Levels¶

    DevTools agents can be adopted gradually.

    Level A – Tool-Assisted (Today)¶

    • Personal use of Copilot/ChatGPT; no persistent agents.

    Level B – Assistive Agents in One Pod¶

    • Introduce EVA-Review-Agent, EVA-Scrum-Agent, and EVA-Repo-Librarian for a single pod (e.g., EVA UI).
    • Evaluate impacts and adjust.

    Level C – Platform-Wide DevTools Agents¶

    • Extend agents across pods.
    • Introduce EVA-Requirements-Agent and EVA-Scaffolder-Agent.

    Level D – Metrics-Driven Improvement¶

    • Add EVA-Runbook-Agent and EVA-Metrics-Agent.
    • Use their outputs in retrospectives and quarterly reviews.

    11. Next Steps¶

    1. Implement the DevTools Runtime as a single service:

    2. Trigger handling, context gathering, LLM calls via EVA, outputs, logging.

    3. Start with 2–3 agent profiles:

    4. EVA-Review-Agent
    5. EVA-Scrum-Agent

    6. EVA-Repo-Librarian

    7. Wire to GitHub / boards in read/comment/draft mode only.

    8. Capture feedback in retrospectives and tune noise level, prompts, and schemas.

    9. Gradually add remaining agent profiles as the EVA Agile Crew’s maturity increases.
