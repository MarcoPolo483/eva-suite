    Pattern Index P01–P21: EVA Agile Crew DevTools Catalog¶

    Document Control: - Owner: EVA Dev Master Orchestrator (P15-DVM) + P01-PAT (Pattern Agent) - Created: 2025-12-11 - Last Updated: 2025-12-11 - Classification: Internal - EVA Program - Next Review: 2026-01-11 (Quarterly)

    Purpose¶

    This Pattern Index is the master reference for the EVA Autonomous Agile Crew. It defines:

    1. How Pods collaborate across the EVA Agile Agentic Lifecycle (Phases 0–6)
    2. Full pattern catalog P01–P21 (DevTools agents and capabilities)
    3. Explainability, traceability, provenance & fingerprinting rules
    4. Role of Dev Master Orchestrator (P15/DVM) and local Scrum Master
    5. Alignment with GC Agentic AI Template (Volumes 0–4)

    This index is grounded in: - EVA Agile Crew Charter (mission, epics, roles) - Pods Map (POD-F, POD-X, POD-O, POD-S-) - *EVA Agile Agentic Lifecycle Map** (Phases 0–6) - DevMaster Directives (orchestration rules)

    EVA Pods Structure¶

    EVA is organized into 4 pod families aligned with platform capabilities and solution domains:

    Pod Full Name Responsibilities Key Repos
    POD-F Foundation & Platform RAG, APIs, infrastructure, core platform services eva-meta, eva-orchestrator, eva-apim-integration, eva-rag
    POD-X Experience & UI EVA Digital Assistant, Chat UI, user experience eva-ui, eva-da, eva-chat
    POD-O Operations Safety, metering, logging, SLOs, governance, monitoring eva-orchestrator, eva-telemetry, eva-audit
    POD-S-* Solution Pods Domain-specific solutions (Jurisprudence, AssistMe, Knowledge Management, CPP-D) eva-juris, eva-assist, eva-km, eva-cpp-d

    Pod Coordination: - Pods are autonomous but coordinated via: - Sprint configs: eva-meta/sprints/sprint-XXX.yaml - Pods map: eva-meta/config/pods-map.yaml - Repo manifests: .eva-pod.yaml in each repo - P15-DVM (Dev Master Orchestrator) coordinates cross-pod work - P03-SCR (Scrum Master) manages backlog and workflow health

    EVA Agile Agentic Lifecycle (Phases 0–6)¶

    All EVA work follows a 7-phase lifecycle that maps to agile ceremonies and gates:

    Phase Name Description Key Activities Exit Criteria
    0 Idea / Intake / Sandbox Exploration, research, proof-of-concept Problem definition, research spikes, sandbox experiments Go/No-Go decision, initial architecture
    1 Design & Architecture Detailed design, ADRs, architecture diagrams User stories, acceptance criteria, ADRs, API contracts Design review approved, stories ready
    2 Build / Code / Test Implementation, unit tests, integration tests Coding, test generation, PR reviews Tests passing, code reviewed, docs updated
    3 Internal Demo / Alpha Internal stakeholder demo, feedback collection Demo preparation, feedback analysis, iteration Stakeholder approval, alpha release
    4 Pilot / Beta Limited production deployment, beta testing Pilot deployment, monitoring, incident response Pilot success metrics met, beta feedback positive
    5 GA / Scale General availability, full production rollout Production deployment, scaling, documentation finalization Production stable, SLOs met, docs complete
    6 Day-2 Ops & Continuous Improvement Monitoring, maintenance, optimization, retrospectives Incident management, performance tuning, pattern updates Continuous improvement cycle established

    Lifecycle Governance: - Phase 0–1: Exploration and design (low risk, high flexibility) - Phase 2–3: Implementation and validation (medium risk, code/config changes) - Phase 4–5: Production deployment (high risk, requires governance approval) - Phase 6: Operations and improvement (ongoing, monitored)

    Human Roles¶

    While EVA agents automate much of the work, human roles provide oversight, direction, and critical decision-making:

    Role Name Responsibilities Authority Level
    Product Owner (PO) Marco Presta Vision, roadmap, prioritization, go/no-go decisions Final authority on product direction
    Local Scrum Master GitHub Copilot (P03-SCR) Backlog management, workflow orchestration, sprint ceremonies Day-to-day workflow management
    Platform Service Owner Designated human Platform health, infrastructure, SLOs Platform stability and performance
    Tech Leads Domain experts Architecture, design reviews, technical guidance Technical decisions within domain
    Developers Human + Agent crew Implementation, code reviews, testing Code quality and implementation
    DevOps / SRE Human + P09-RUN, P14-LVO agents Deployment, incident response, reliability Production stability and operations
    AI Steward Designated human AI governance, compliance, ethical oversight AI usage compliance and governance
    Senior Advisor External LLM (GPT-4, Claude) Strategic guidance, complex problem-solving Advisory (no direct authority)

    Human-AI Teaming Principle: Humans set direction, agents do the heavy lifting. Critical decisions (C2–C3 actions) require human approval.

    Dev Master Orchestrator (P15/DVM) & Scrum Master¶

    Dev Master Orchestrator (P15/DVM)¶

    Primary Location: eva-meta (POD-F)

    Core Responsibilities: - Cross-repo coordination: Orchestrates agents across all EVA repos and pods - Sprint execution: Interprets sprint configs (eva-meta/sprints/sprint-XXX.yaml) - Work routing: Routes work to specialized agents (REQ, SCR, LIB, SCA, REV, TST, CICD, etc.) - Compliance enforcement: Ensures alignment with pods map, repo manifests, DevMaster Directives

    Key Constraints (from DevMaster Directives): 1. Does not merge PRs or push to main (humans approve merges) 2. Creates issues, PR drafts, reports, logs (agents propose, humans dispose) 3. Uses local knowledge before external advisors (RAG, CDD, ADRs first) 4. Logs everything and stays explainable (audit trail, provenance) 5. Fails safe; uses runbooks for self-healing (safety over speed)

    Configuration Sources: - Sprint configs: eva-meta/sprints/sprint-XXX.yaml - Pods map: eva-meta/config/pods-map.yaml - Repo manifests: .eva-pod.yaml (in each repo)

    Scrum Master (P03-SCR)¶

    Primary Role: Local workflow orchestrator and backlog manager

    Core Responsibilities: - Backlog grooming: Maintains Epics A–D with stories/tasks - Workflow health checks: Monitors agent workflows, detects issues - Risk monitoring: Watches for low-confidence trends, failures, violations - Sprint reporting: Generates human-readable summaries and retrospectives

    Operating Principles (from EVA Crew Charter): 1. Safety before speed (halt and escalate if uncertain) 2. Small, observable steps (prefer small PRs over big-bang changes) 3. Evidence-based progress (track metrics, show results) 4. "Humans set direction, agents do the heavy lifting"

    New Agentic Capability Pillars¶

    Using the GC Agentic AI Template Vol.0 and Volumes 1–4, we group new capabilities into 5 pillars:

    1. Awareness & Metacognition
      Agents reason about their own outputs, assess confidence, and identify risks.

    2. Multi-Agent / Swarm Software Engineering
      Specialized agents collaborate like a team, with defined roles and handoffs.

    3. Swarm PR & Tool Onboarding
      Parallel micro-agents for checks (A11y, i18n, security) and automated tool integration.

    4. Provenance, Explainability & Governance
      Auditability, data lineage, traceability, compliance with TBS/PADI/EVA standards.

    5. Continuous Self-Improvement
      Retrospectives, pattern evolution, learning from logs and metrics.

    These pillars become Patterns P16–P21.

    New Patterns P16–P21¶

    P16–P21 extend the DevTools catalog per DevMaster "P16+" clause. They implement the 5 agentic capability pillars.

    P16 – Awareness Protocol Pattern¶

    Goal: Every agent run emits awareness metadata so DVM and humans can gauge reliability and risk.

    Scope: All Pods & all agents (P01–P21)

    Output Contract (per action):

    {
  "awareness": {
    "confidence": 0.85,
    "confidenceLevel": "High",
    "reflection": "Code follows established patterns. No edge cases detected. Tests cover main scenarios.",
    "risks": [
      "Untested edge case: null input handling",
      "Assumption: API endpoint returns valid JSON"
    ],
    "missingContext": [
      "Production environment variables",
      "Rate limiting policies for APIM"
    ],
    "suggestedNextAction": "proceed"
  }
}

    Fields: - confidence: Numeric score (0.0–1.0) - confidenceLevel: Qualitative (Low / Medium / High) - reflection: Self-assessment ("What might be wrong?") - risks: Identified risks or edge cases - missingContext: Information gaps - suggestedNextAction: proceed | retry | ask_human | call_another_agent | stop

    Lifecycle Integration: - Phase 2 (Build): Test flakiness & code suggestions are explained - Phase 4–6 (Pilot/GA/Ops): Governance and LiveOps agents explain alerts

    Reinforces: DevMaster Directive 6 ("Keep everything explainable"), now at per-agent level

    P17 – Swarm PR Review Pattern¶

    Goal: PRs are reviewed by a swarm of specialized micro-agents, with a single aggregated summary.

    Typical Swarm per PR: - A11y (UXA – P12): WCAG compliance, accessibility checks - i18n (UXA – P12): Bilingual requirements, language externalization - GC Design and Content (UXA / SEC – P11/P12): Government design system compliance - FinOps / APIM Cost Headers (MET/LVO – P10/P14): Cost tracking, resource usage - Security (SEC – P11): Secrets detection, vulnerability scanning - Testing & QA (TST / CICD – P07/P08): Test coverage, CI pipeline checks

    SM / DVM Responsibilities: 1. Trigger swarm workflows on each PR (Phases 2–6) 2. Enforce single aggregated comment: - Blocking issues (must fix) - Warnings (should fix) - Informational (nice to have) 3. Link each finding to: - Files, tests, policies, design decisions

    Built on Top Of: P06 (REV), P07 (TST), P08 (CICD), P11 (SEC), P12 (UXA), P14 (LVO)

    Example Aggregated Comment:

    ## 🤖 Swarm Review Summary

**Blocking Issues (2)**:
- [A11y] Missing alt text on 3 images (WCAG 2.1 AA violation) – `Dashboard.tsx:15,23,31`
- [Security] Hardcoded API key detected in `config.ts:42`

**Warnings (1)**:
- [FinOps] Missing cost header on new APIM call (recommended: add X-Cost-Center header)

**Info (3)**:
- [i18n] All strings externalized correctly ✅
- [API Governance] APIM policy compliant ✅
- [Tests] Coverage increased from 78% → 82% ✅

**Suggested Fixes**:
- For A11y: Add `alt` attributes to `<img>` tags
- For Security: Move API key to Azure Key Vault, reference via environment variable

    P18 – Tool / API Auto-Onboarding Pattern¶

    Goal: Automate the "boring" parts of integrating a new endpoint / API / tool.

    Trigger Conditions: - New OpenAPI spec appears in contracts/ or apis/ - New APIM endpoint added to eva-apim-integration

    Agents Involved: - P05-SCA (Scaffolder): Generate typed client/wrapper - P07-TST (Test): Generate test stubs and example usage - P08-CICD (CI/CD): Add pipeline hooks - P09-RUN (Runbook): Create runbook stubs - P10-MET (Metrics): Define metrics plan - P11-SEC (Security): Basic security checklist

    Outputs: - Typed client/wrapper (TypeScript, Python, etc.) - Tests and example usage (unit tests, integration tests) - Minimal docs (README fragment + usage snippet) - Governance notes (classification, risk hints) - Optional: APIM policy and cost header stubs

    Lifecycle Usage: Heavily used by POD-F and POD-S in Phases 1–3 (Design → Build → Demo)

    Example Workflow:

    name: Tool Onboarding
on:
  push:
    paths:
      - 'contracts/**/*.yaml'
      - 'apis/**/*.yaml'
jobs:
  onboard-tool:
    runs-on: ubuntu-latest
    steps:
      - name: P05-SCA - Generate Client
        run: copilot generate-client --spec=${{ github.event.path }}
      - name: P07-TST - Generate Tests
        run: copilot generate-tests --client=generated/client.ts
      - name: P09-RUN - Generate Runbook
        run: copilot generate-runbook --service=${{ matrix.service }}
      - name: P10-MET - Define Metrics
        run: copilot define-metrics --service=${{ matrix.service }}
      - name: P11-SEC - Security Checklist
        run: copilot security-checklist --spec=${{ github.event.path }}
      - name: Create PR for Human Approval
        run: gh pr create --title "Auto-onboard: ${{ matrix.service }}" --body "Generated by P18 Tool Onboarding"

    P19 – Action Classification & Safety Pattern¶

    Goal: Every agent action gets a risk class and is constrained by what it is allowed to do.

    Action Classes:

    Class Description Autonomy Level Examples
    C0 Read/Analyze only Always allowed Code analysis, documentation review, metric reporting
    C1 Dev-branch code/config changes Autonomous (via PR drafts) Code formatting, test generation, README updates
    C2 Governance / policy touching changes Requires human approval API contract changes, security configs, governance docs
    C3 Production / citizen-facing / infra changes Requires human + governance approval Production deployments, infrastructure changes, citizen-facing UI

    Rules: - C0–C1: May be performed autonomously via PR drafts (never direct merges) - C2–C3: Require explicit human approval and often governance sign-off

    Integration: - Ties directly into GC Template Vol.3 (LiveOps & Governance) - Extends DevMaster Directive 3/6 (no merging; log everything)

    Enforcement: - GitHub Actions workflows check action class before execution - SM/DVM enforces classification rules, halts on violations

    Example Classification:

    {
  "action": {
    "actionClass": "C2",
    "operation": "config_change",
    "description": "Update APIM policy to require authentication",
    "requiresApproval": true,
    "approvalRequired": ["human", "governance"],
    "blockingUntilApproved": true
  }
}

    P20 – Continuous Self-Improvement / Retrospective Pattern¶

    Goal: Make the crew learn from its own logs.

    Inputs: - Orchestration logs: eva-meta/orchestration/ - Metrics from P10-MET (Metrics agent) - Incidents from P09-RUN (Runbook) & P14-LVO (LiveOps) - PR review comments and swarm outcomes

    Outputs: - Proposals for: - New checks or micro-agents - Prompt updates for P02–P15 - Improvements to sprint configs and pods map - Retrospective summaries per sprint/quarter

    Lifecycle Integration: Sits on top of Phase 6 (Day-2 Ops & Continuous Improvement)

    Example Retrospective:

    ## Sprint 12 Retrospective (P20 Self-Improvement)

### What Went Well ✅
- Swarm PR reviews caught 23 A11y issues before merge
- Tool onboarding (P18) reduced integration time from 8h → 2h
- Awareness protocol (P16) escalated 5 low-confidence cases correctly

### What Needs Improvement 🔧
- 12 PRs had repeated test failures (flaky tests in `eva-da`)
- 3 governance violations slipped through (missing bilingual content)
- Agent confidence scores trending lower in complex refactors

### Proposed Actions 📋
- **New micro-agent**: Flaky test detector (P07-TST extension)
- **Prompt update**: Enhance P11-SEC to check bilingual requirements earlier
- **Threshold adjustment**: Lower confidence threshold for refactors (0.8 → 0.9)

### Metrics 📊
- PRs reviewed: 87
- Agent-generated PRs: 34 (39%)
- Human intervention rate: 18% (down from 25%)
- Average PR cycle time: 1.2 days (target: < 2 days)

    P21 – Provenance & Fingerprinting Pattern¶

    Goal: Make every agent action reconstructible: who, what, where, when, why, and with what knowledge.

    Fingerprint Fields (per action):

    {
  "agentFingerprint": {
    "agentId": "P07-TST",
    "agentVersion": "2.1.0",
    "pod": "POD-F",
    "service": "eva-apim-integration",
    "repo": "eva-apim-integration",
    "environment": "dev"
  },
  "taskContext": {
    "projectId": "EVA Suite",
    "sprintId": "Sprint-12",
    "workItemId": "GH-123",
    "workItemTitle": "Add authentication to APIM policy"
  },
  "awareness": {
    "confidence": 0.85,
    "confidenceLevel": "High",
    "reflection": "Implementation follows APIM best practices. Tests cover main scenarios.",
    "risks": ["Untested: concurrent auth token refresh"],
    "missingContext": ["Production rate limits"],
    "suggestedNextAction": "proceed"
  },
  "provenance": {
    "promptRef": "eva-meta/prompts/p07-tst-v2.1.md",
    "contextSources": [
      "docs/agentic/pattern-index.md",
      "docs/agentic/eva-crew-charter.md",
      "eva-apim-integration/README.md"
    ],
    "model": {
      "deploymentName": "gpt-4-turbo",
      "temperature": 0.2,
      "maxTokens": 4096
    }
  },
  "changeSummary": {
    "actionClass": "C1",
    "operation": "code_change",
    "filesTouched": [
      "src/policies/auth-policy.ts",
      "tests/auth-policy.test.ts"
    ],
    "git": {
      "before": "abc123",
      "after": "def456",
      "branch": "feature/apim-auth-policy"
    }
  },
  "humanApproval": {
    "required": false,
    "approvedBy": null,
    "approvedAt": null,
    "prUrl": "https://github.com/org/eva-apim-integration/pull/456"
  },
  "timestamp": "2025-12-11T14:23:45Z"
}

    Storage: - Single, append-only JSONL or database table: - eva-meta/logs/agents/audit-log.jsonl - Indexed by: agentId, sprintId, workItemId, timestamp

    This Pattern Concretizes: - DevMaster Directive 6 ("Keep everything explainable") - EVA Crew Charter requirement for traceable, auditable AI-assisted work - TBS GenAI Guidelines for AI governance and transparency

    How to Use This Index¶

    For Product Owners & Tech Leads¶

    • Reference this index when planning new features or capabilities
    • Use lifecycle phases to understand where work happens
    • Check pod mapping to see which agents are responsible

    For Developers & DevOps¶

    • Use patterns P01–P15 for day-to-day development tasks
    • Use patterns P16–P21 for governance, safety, and continuous improvement
    • Follow action classification (P19) to understand approval requirements

    For Agents & Automation¶

    • Pattern Index is the source of truth for agent capabilities and responsibilities
    • All agent actions must include awareness metadata (P16) and provenance (P21)
    • Swarm reviews (P17) are mandatory for all PRs in Phases 2–6
    • Tool onboarding (P18) automates integration of new services/APIs

    Next Steps for Implementation¶

    This document is the master catalog (P00). For each pattern P01–P21:

    Generate: 1. Copilot system prompt (who the agent is, what it must obey) 2. Implementation prompts (what files to create/update, where, with what contracts) 3. Safety & logging requirements (P16, P19, P21 baked in)

    To request implementation-level prompts, say: - "Generate Copilot prompts for P16–P21 based on this index" - "Create SM orchestration workflow prompt for Epic B" - "Define swarm PR review workflow prompt for Epic C"

    And the SM will go pattern-by-pattern, wired to pods, lifecycle phases, and repos exactly as described here.
