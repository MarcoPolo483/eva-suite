    EVA DevTools Agent Pattern – Dev Master Orchestrator Agent¶

    Pattern ID: P15 Subtitle: Conductor of the EVA AI Agile Dev Crew (Jarvis / DevMaster) Version: Draft 0.2 Date: YYYY-MM-DD

    1. Pattern Overview¶

    Agent Name: EVA-DevMaster-Agent (Dev Master Orchestrator / “Jarvis”) Category: DevTools / Coordination / Meta-Agent Audience: Tech Leads, Product Owners, Dev/SRE Leads, Platform Owners

    The EVA-DevMaster-Agent is the “conductor” of the EVA AI Agile Dev Crew.

    It:

    • Coordinates the work of all DevTools agents (P1–P14).
    • Orchestrates multi-step development workflows (idea → requirements → design → code → tests → CI/CD → runbooks → LiveOps).
    • Helps humans see, steer, and tune the overall AI-assisted development process.

    It is not a boss or an auto-pilot. It is a meta-level assistant that:

    • Knows which specialist agent to call, in which order.
    • Keeps track of artefacts and decisions across steps and repos.
    • Surfaces what matters to humans so they stay in control.

    2. Intended Use & Non-Use¶

    2.1 Intended Use¶

    Use EVA-DevMaster-Agent to:

    • Run structured development workflows, such as:

    • “Refine requirements → propose design → scaffold code → review → test → CI/CD → runbook update.”

    • Coordinate specialist agents, e.g.:

    • P1 Requirements Agent

    • P2 Design & Architecture Agent
    • P3/P4 Coding Agents
    • P7 Testing & Failure Explainer
    • P8 CI/CD Guardian
    • P9 Runbook & SRE Agent
    • P10 Metrics & Insight Agent
    • P11 Security & Compliance Helper
    • P12 UX, Copy & Accessibility Agent
    • P13 Onboarding & Sandbox Coach Agent
    • P14 DevOps LiveOps Companion Agent

    • Maintain lightweight process memory:

    • Which agent ran when.

    • Which artefacts were produced.
    • Which recommendations were accepted or rejected.

    • Provide human-facing views, such as:

    • “Where are we in the lifecycle for Feature X?”

    • “Which DevTools agents have already helped and what did they say?”

    Typical scenarios:

    • New feature: DevMaster runs a guided “from idea to PR” workflow.
    • Infra/platform change: DevMaster coordinates design → pipelines → runbooks → LiveOps checks.
    • Cross-repo work: DevMaster keeps a meta-thread across repos, pods, and agents.

    2.2 Non-Use¶

    Do not use EVA-DevMaster-Agent to:

    • Replace Scrum Masters, Tech Leads, Product Owners, or Release Managers.
    • Make unilateral decisions on scope, priority, or release readiness.
    • Execute infrastructure changes, approvals, or HR or financial actions.

    DevMaster supports and structures human-led work; it never becomes an autonomous project manager or change approver.

    3. Role in the GC Agentic Template¶

    • Vol. 1 – Platform & Agentic Requirements

    • Implements the “agent-of-agents” orchestration layer for engineering work.

    • Ensures each workflow respects autonomy levels, security boundaries, and audit expectations.

    • Vol. 2 – Engineering & Delivery Guide

    • Encodes standard SDLC flows as orchestrated sequences of DevTools agents.

    • Makes these flows reusable across projects, pods, and departments.

    • Vol. 3 – LiveOps & Governance Model

    • Tracks which agents contributed to which artefacts and decisions.

    • Provides an audit trail of how we got from requirement → deploy → runbook → LiveOps signal.

    • Vol. 4 – Use Cases, Patterns & Templates

    • Exposes recipe-level patterns (“Feature Dev Recipe”, “Infra Change Recipe”, “Bug Fix Recipe”) that call P1–P14 agents under DevMaster’s coordination.

    4. High-Level Behaviour¶

    EVA-DevMaster-Agent:

    1. Understands workflows

    2. “For this type of work (feature, bug, infra, spike), which agents should be involved, in what order, with what artefacts?”

    3. Plans and sequences calls

    4. Chooses which specialist agent(s) to call next.

    5. Passes the right context between them (e.g., refined requirements → design sketch → scaffolding → tests → pipeline suggestions → runbooks).

    6. Maintains a meta-level record

    7. Tracks steps, outputs, decisions, and open questions.

    8. Summarizes state for humans (“You’re here, next suggested step is…”).

    9. Respects humans and governance

    10. Never bypasses approvals or mandatory gates.

    11. Always offers clear Stop / Pause / Continue / Restart options.

    DevMaster is a coordination layer, not an additional domain expert. It orchestrates P1–P14 rather than duplicating their capabilities.

    5. Inputs & Outputs¶

    5.1 Inputs¶

    • Work item / request context

    • Type (feature, bug, tech debt, infra change, spike).

    • Project/repo/service and pod (from .eva-pod.yaml and pods-map).

    • Priority / target sprint (if known).

    • Human goals & constraints

    • “I want a minimal MVP spec.”

    • “Do not touch production infra yet.”

    • “Protected B only; no external data; follow EVA governance.”

    • Current artefacts (optional)

    • Existing requirements, design docs, CDD snippets.

    • Existing code, tests, pipelines, runbooks, metrics.

    • Org templates and policies

    • Canonical recipes for certain work types.

    • Mandatory checkpoints (security, privacy, a11y, RAIDE/PMD evidence).

    5.2 Outputs¶

    • Workflow plans

    • “For this item, the recommended sequence is: P1 → P2 → P3/P4 → P7 → P8 → P9 → P12 → P11 (if needed).”

    • Orchestration logs

    • Structured record of which agent ran, when, with which key results.

    • Stored as orchestration/FEATURE-123-devmaster-log.md or similar.

    • Human summaries

    • “Here is what Requirements proposed, what Design agreed, what Code implemented, and what Tests and Security found.”

    • Action prompts

    • “Next suggested step: run the Testing Agent on PR #123.”

    • “Security & Compliance review not yet done; consider calling P11 before merging.”

    6. Triggers¶

    • New work item created

    • Item labelled orchestrated-by-devmaster or created from an EVA template.

    • Human request

    • /eva-devmaster plan FEATURE-123

    • /eva-devmaster continue FEATURE-123

    • /eva-devmaster status FEATURE-123

    • Lifecycle transitions (via GitHub / AzDO hooks)

    • PR opened/updated.

    • Pipeline success/failure.
    • Release tagged or deployment completed.

    DevMaster runs only when explicitly invoked or when configured hooks fire; no hidden background orchestration.

    7. Interaction Flows¶

    7.1 Feature Development Flow (End-to-End)¶

    1. Kickoff

    2. PO/Tech Lead calls: /eva-devmaster plan FEATURE-123 type=feature project=eva-da-ms-pubsec

    3. Planning

    4. DevMaster reads work item and repo/pod context.

    5. Selects the standard Feature Flow recipe (from P01 toolbox roadmap):

      • P1 Requirements
      • P2 Design
      • P3/P4 Coding
      • P7 Testing & Failure Explainer
      • P8 CI/CD Guardian
      • P9 Runbook & SRE
      • P10 Metrics (if applicable)
      • P12 UX/A11Y
      • P11 Security & Compliance (where needed)

    6. Produces a plan with steps, each mapped to a specialist agent and repo.

    7. Execution (Agent Orchestration) For each step, DevMaster:

    8. Calls the appropriate agent with scoped context (files, issues, PRs, CDD snippets).

    9. Receives outputs (requirements, design notes, scaffolds, tests, pipeline recommendations, runbook drafts, etc.).

    10. Logs outputs and highlights key decisions and TODOs.

    11. Human Control

    12. At each step, humans:

      • Accept, adjust, or reject outputs.

      • DevMaster updates its state:

      • “Design accepted; code scaffolding partially applied; tests pending; security review outstanding.”

    13. Completion Summary

    14. DevMaster produces an end-to-end summary:

      • What changed across repos.
      • Which agents were involved.
      • Outstanding risks and follow-up items.

    15. Summary is stored and linked from the work item and relevant PRs.

    7.2 Infra / Platform Change Flow¶

    1. Infra lead calls: /eva-devmaster plan TASK-456 type=infra project=eva-infra

    2. DevMaster builds an infra-oriented recipe, emphasising:

    3. P2 Design & Architecture Agent

    4. P7 Testing & Failure Explainer
    5. P8 CI/CD Guardian
    6. P9 Runbook & SRE
    7. P10 Metrics & Insight
    8. P11 Security & Compliance

    9. P14 DevOps LiveOps Companion

    10. DevMaster coordinates steps, gathers outputs, and surfaces a clear pre-deploy checklist.

    7.3 “Catch Me Up / Continue” Flow¶

    1. A human returns to a partially completed item and calls: /eva-devmaster status FEATURE-123

    2. DevMaster responds with:

    3. Which steps are complete.

    4. Which agents have contributed so far.

    5. What the next recommended step is.

    6. Human can:

    7. Continue the workflow.

    8. Skip or reorder steps.
    9. Add or remove agents.

    8. Profile Configuration (EVA Runtime)¶

    Conceptual profile (YAML-style, for EVA DevTools runtime / Jarvis):

    agent_id: eva-devmaster-agent display_name: "EVA Dev Master Orchestrator Agent" description: "Coordinates EVA DevTools agents, orchestrating end-to-end AI-assisted development workflows." autonomy_level: assistive

    triggers:

    • manual commands (/eva-devmaster ...)
    • work item label events (label:orchestrated-by-devmaster)
    • lifecycle hooks (PRs, builds, releases) where configured

    context_requirements:

    • work item metadata
    • project and repo metadata (including pod and service info)
    • DevTools agent catalog (P1–P14 capabilities)
    • workflow recipes catalog
    • orchestration history for the item

    outputs:

    • workflow plans
    • orchestration logs
    • human-readable summaries
    • next-step recommendations

    policies:

    • require human review flag: true
    • no direct code or infra changes: true
    • mark items with tag: generated-by:eva-devmaster-agent

    9. Prompting Model (High Level)¶

    9.1 System Prompt (DevTools Global – reused)¶

    Ensures DevMaster:

    • Treats all specialist agents as tools to be orchestrated.
    • Respects each agent’s autonomy level and guardrails.
    • Keeps humans in charge, with full visibility into steps and outputs.

    9.2 Role Prompt (Dev Master Orchestrator Agent)¶

    Essence of the devmaster_agent role:

    • “You coordinate specialist DevTools agents across the software lifecycle. You choose appropriate agents and order based on work type, governance rules, and available context. You maintain a clear orchestration log and provide concise status and next steps. You do not directly modify code or infrastructure or override governance.”

    Example orchestration state (conceptual JSON):

    • work_item_id (e.g., FEATURE-123)
    • type (feature/bug/infra/spike)
    • project (repo/service)
    • current_step
    • completed_steps (with agent, outcome, summary)
    • pending_steps (with agent and status)
    • notes_for_humans (risks, reminders, governance flags)

    10. Guardrails & Governance¶

    • No hidden automation

    • DevMaster orchestrates agents, not infrastructure.

    • Any code or infra changes remain in PRs and pipelines controlled by humans.

    • Transparent orchestration

    • Every workflow and step is logged and attributable to specific agents and human decisions.

    • Respect for domain boundaries

    • Security, privacy, and compliance agents retain their advisory authority.

    • DevMaster never downgrades or suppresses warnings; it can only highlight, group, and prioritise them.

    • Human override

    • At any time, humans can stop a workflow, change step order, or add/remove agents.

    11. Implementation Notes for Jarvis + GitHub¶

    • DevMaster is implemented as “Jarvis” inside eva-meta and service repos:

    • jarvis-* workflows (jarvis-refine, jarvis-plan, jarvis-execute, jarvis-daily, jarvis-review, jarvis-release, jarvis-uat, jarvis-retro) are the concrete hooks.

    • Each workflow calls DevTools agents via EVA Foundation APIs, with DevMaster coordinating which one to invoke and when.

    • Integration points:

    • GitHub Issues / Projects for work items.

    • .eva-pod.yaml and pods-map.yaml to understand pods and services.
    • GitHub Actions for CI/CD and lifecycle hooks.

    • EVA DevTools Agent Catalog for P1–P14 capabilities and endpoints.

    • Storage and traceability:

    • Per-work-item orchestration files under orchestration/.

    • Links from issues, PRs, and sprint reports back to DevMaster logs.

    • Recipes:

    • Feature, bug fix, infra change, spike/experiment, pilot support, etc.

    • Versioned and tunable to match EVA’s maturity and GC governance expectations.

    12. Adoption & Maturity¶

    • Phase 1 – Orchestration Logger

    • DevMaster simply logs which agents humans call and when.

    • Phase 2 – Guided Recipes

    • DevMaster suggests “next best agent to call” per work item.

    • Phase 3 – Integrated Orchestrator

    • Hooks into issues, PRs, CI events; automatically updates status and suggests next steps.

    • Phase 4 – GC-Wide Template

    • Recipes generalised and shared as a Canada-grade AI Dev Orchestration template; other departments can plug in their own agents, repos, and policies.

    13. Benefits Summary¶

    • Provides a single mental model of how AI agents support development across EVA Suite.

    • Reduces cognitive load on humans:

    • “Which agent should I use now? What did they say last time?”

    • Creates a living audit trail of AI involvement in design, coding, testing, and ops.
    • Makes EVA’s AI Agile Dev Crew exportable, repeatable, and explainable across GC teams in a controlled, transparent way.
