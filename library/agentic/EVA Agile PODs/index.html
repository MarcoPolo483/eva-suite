    EVA Agile PODs

    EVA REPOSITORIES AND PODS MAP – CURRENT REALITY → TARGET STRUCTURE Version: 0.2 Context date: Nov 2025 Audience: Jarvis (EVA DevMaster), Copilot Pro+, Marco

    Goal: Connect the current single-workspace layout to the target EVA repo + pods model, so Jarvis and Copilot know:

    • which folders become which GitHub repos
    • which pod owns which repo
    • how to populate pods-map.yaml and .eva-pod.yaml with today’s reality

    1. Current State (Local Workspace)¶

    Main workspace

    c:\Users\marco.presta\dev\EVA Suite\info-assistant-reference

    Relevant structure inside it

    eva-ecosystem/

    • EVA Automation Scripts (login.ps1, logoff.ps1, etc.)
    • EVA PowerShell Modules (EVATimeTracking.psm1, etc.)
    • project-apim-integration/ (API Management implementation)
    • enhanced-cosmos/ (Cosmos DB optimizations)
    • PubSec-Info-Assistant/ (Microsoft platform + EVA enhancements)
    • Documentation and planning files

    Additional notes

    • The whole workspace is currently treated as one local repo: info-assistant-reference.
    • PubSec-Info-Assistant is a Git submodule pointing to https://github.com/microsoft/PubSec-Info-Assistant.git plus EVA customizations.

    So everything EVA-related lives in one big tree, not yet split into the target EVA Suite repos.

    2. Target Repositories (Short-Term, Realistic)¶

    We want to evolve from “one big repo” into a set of focused GitHub repos that still match your EVA naming scheme, but also reflect what actually exists today.

    Short-term target repos:

    1. eva-meta

    Status: new Purpose: EVA lifecycle docs, CDDs, Jarvis workflows, sprint YAML files, pods map and other meta-assets. Location (initial): can start as a folder under EVA Suite, later its own GitHub repo.

    1. eva-core (optional for now)

    Purpose: shared building blocks not specific to APIM, Cosmos, automation, or UI. Current mapping: could include planning files and generic helpers. Note: creation can be delayed until a clear core emerges.

    1. eva-apim-integration

    Source: eva-ecosystem/project-apim-integration/ Purpose: all APIM implementation for EVA, including backend contracts, policies and APIM scripts.

    1. eva-cosmos-enhancements

    Source: eva-ecosystem/enhanced-cosmos/ Purpose: Cosmos DB optimizations, schemas, indexing strategies and helper scripts.

    1. eva-automation-scripts

    Source:

    • eva-ecosystem/EVA Automation Scripts
    • eva-ecosystem/EVA PowerShell Modules

    Purpose: login and logoff scripts, time-tracking modules, developer productivity helpers and environment automation.

    1. eva-da-ms-pubsec

    Source: eva-ecosystem/PubSec-Info-Assistant/ (forked or wrapped) Purpose: EVA’s current Domain Assistant UI and backend based on Microsoft PubSec Info Assistant, including EVA-specific enhancements.

    1. eva-documentation

    Source:

    • documentation and planning files
    • EVA_REPOSITORY_LIST.md
    • other meta documents

    Shape options:

    • a separate repo, or
    • folded into eva-meta under a docs/ folder.

    Planned future repos (not yet implemented but part of the EVA vision) include:

    • eva-qa-agent
    • eva-orchestrator
    • eva-ui
    • eva-ops
    • eva-infra
    • eva-seed

    For now, Jarvis should treat the concrete repos above as the primary working set.

    3. Pod Assignments for Current Repos¶

    Pod definitions

    POD-F = Foundation and Platform

    • backend, infrastructure, APIM, Cosmos, DevTools runtime

    POD-X = Experience and UI

    • frontends, UX, accessibility

    POD-O = Operations and LiveOps

    • monitoring, cost, observability, automation

    POD-S = Solution pods (Juris, Assist, KM and other solution spaces, later)

    Mapping of current repos to pods:

    eva-apim-integration

    • Pod: POD-F
    • Role: platform backbone, APIM routing and contracts

    eva-cosmos-enhancements

    • Pod: POD-F
    • Role: data layer optimization, queries and indexes

    eva-automation-scripts

    • Pod: POD-O (primary) with POD-F support
    • Role: developer and operations scripting, local tooling, environment automation

    eva-da-ms-pubsec

    • Pod: POD-X (primary) with POD-F support
    • Role: EVA Domain Assistant UI, RAG and backend based on PubSec Info Assistant

    eva-meta

    • Pod: POD-F for ownership, but cross-pod by nature
    • Role: documentation, Jarvis workflows, pods map, sprint configuration and other orchestration assets

    eva-documentation (if separate)

    • Pod: POD-F (custodian)
    • Role: CDDs, playbooks, business documents, AI governance documentation

    At this moment there are no explicit POD-S-Juris or POD-S-Assist repos; these will be added as EVA Suite grows.

    6. Housekeeping: Migrating from Current Workspace to Target Repos¶

    Tasks Jarvis or Copilot can help with, based on this map:

    1. Create target directories and Git repos locally

    Target repo names:

    • eva-apim-integration
    • eva-cosmos-enhancements
    • eva-automation-scripts
    • eva-da-ms-pubsec
    • eva-meta
    • eva-documentation (optional)

    Ask Copilot: “Using EVA_REPOSITORY_AND_PODS_MAP_v0_2, generate a PowerShell script that:

    1. creates these folders under c:\Users\marco.presta\dev\EVA Suite,
    2. initializes each folder as a Git repo,

    3. seeds each repo with a README and .eva-pod.yaml using the spec in this document.”

    4. Move code from the monorepo into the new repos

    Current to target mapping:

    • eva-ecosystem/project-apim-integration → eva-apim-integration
    • eva-ecosystem/enhanced-cosmos → eva-cosmos-enhancements
    • eva-ecosystem/EVA Automation Scripts plus EVA PowerShell Modules → eva-automation-scripts
    • eva-ecosystem/PubSec-Info-Assistant → eva-da-ms-pubsec (as a fork or vendor subfolder)

    Copilot can draft a migration script, or you can move and commit manually.

    1. Add pods-map.yaml into eva-meta

    2. Create the file eva-meta/config/pods-map.yaml using the content from section 4.

    3. Future Jarvis workflows (jarvis-plan, jarvis-refine) will read this file.

    4. Add .github/copilot-instructions.txt into each repo

    Each file should describe:

    • the repo’s purpose
    • languages and tech stack
    • where tests live
    • branch and PR norms

    7. How to Use This Document With Copilot¶

    Once this file is saved (for example as EVA_REPOSITORY_AND_PODS_MAP_v0_2.md), you can instruct Copilot or Jarvis:

    Example prompts:

    “Read EVA_REPOSITORY_AND_PODS_MAP_v0_2.md and:

    1. create config/pods-map.yaml in eva-meta,
    2. create .eva-pod.yaml files in eva-apim-integration, eva-cosmos-enhancements, eva-automation-scripts, eva-da-ms-pubsec, eva-meta and eva-documentation using the definitions in this document.”

    “Using the mapping in EVA_REPOSITORY_AND_PODS_MAP_v0_2.md, generate a PowerShell script that:

    • creates the target repo folders under EVA Suite,
    • copies content from eva-ecosystem subfolders into the correct repos,
    • initializes Git in each new repo with an initial commit.”

    “Propose .github/copilot-instructions.txt content for each repo using the pod assignments and roles described in EVA_REPOSITORY_AND_PODS_MAP_v0_2.md.”

    This lets your current reality (one EVA ecosystem repo with subfolders) evolve into a clean EVA Suite repo layout aligned with your pod model and the Jarvis orchestration story.
