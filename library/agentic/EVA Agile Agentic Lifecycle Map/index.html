    Here’s an updated, GitHub/Jarvis-aligned version of the EVA Suite – Agentic Lifecycle Map in plain markdown (no nested bullets).

    EVA Suite – Agentic Lifecycle Map¶

    Subtitle: Where Pods, Agents and Jarvis Act Across Design, Build, Pilot, GA and Operations Version: Draft 0.2 Date: YYYY-MM-DD

    1. Legend and Context¶

    1.1 Pods (domains)¶

    • POD-F – Foundation & Platform RAG backbone, APIs, infra, DevTools runtime, APIM, Cosmos, core services.

    • POD-X – Experience & UI EVA Domain Assistant, EVA Chat, component library, UX, i18n, a11y.

    • POD-O – Operations & LiveOps Safety, metering, logging, observability, MTI / trust posture, cost.

    • POD-S – Solution & Seed Templates, sandboxes, and solution pods such as Jurisprudence and AssistMe. Example solution pods that follow the same pattern: POD-S-Juris (Jurisprudence), POD-S-Assist (AssistMe / BDM), POD-S-KM (Knowledge Management).

    1.2 DevTools Agents (short names for P1–P15)¶

    • REQ – Requirements Agent (CDD → epics → stories).
    • DES – Design & Architecture Agent (ADRs, diagrams, contracts).
    • COD – Coding Agent (code changes, refactors, scaffolding).
    • TST – Testing & Failure Explainer (tests, coverage, failure analysis).
    • CICD – CI/CD Guardian (pipelines, quality gates).
    • RUN – Runbook & SRE Agent (operations guidance).
    • MET – Metrics & Insight Agent (what to measure, how to read it).
    • SEC – Security & Compliance Helper (privacy, security hints, controls).
    • UXA – UX, Copy & Accessibility Agent (flows, wording, EN/FR, WCAG).
    • ONB – Onboarding & Sandbox Coach (templates, getting started, guardrails).
    • LVO – LiveOps Companion Agent (status, incidents, environment checks).
    • DVM – DevMaster Orchestrator Agent (Jarvis brain; coordinates other agents).

    Agents work through GitHub and EVA DevTools. They suggest and orchestrate. They do not silently change production; humans approve and merge.

    1.3 Jarvis and GitHub¶

    • Jarvis DevMaster (DVM) lives primarily in the eva-meta repo and is implemented through:

    • Jarvis workflows such as jarvis-refine, jarvis-plan, jarvis-execute, jarvis-daily, jarvis-review, jarvis-release, jarvis-uat, jarvis-retro.

    • Helper scripts under eva-meta/scripts that call EVA DevTools and EVA Foundation APIs.
    • Repos are pod-aware via .eva-pod.yaml and Copilot-aware via .github/copilot-instructions.
    • Sprint configuration lives in eva-meta/sprints/sprint-XXX.yaml, which Jarvis uses as the contract between planning and execution.

    2. Lifecycle Phases Overview¶

    Phase Goal Pods Mainly Involved Most Active Agents Primary Artefacts
    0. Idea / Intake / Sandbox Turn “we should do X with EVA” into a safe, concrete experiment. POD-S, POD-F REQ, DES, ONB, SEC, DVM Intake notes, sandbox plan, minimal design sketch, risk notes
    1. Design & Architecture Define requirements, architecture, and RAG patterns aligned to EVA spine. POD-F, POD-X, POD-O REQ, DES, SEC, UXA, MET, DVM PRD/requirements, ADRs, UX flows, metrics plan
    2. Build / Code / Test Implement features across EVA repos with tests and pipelines. POD-F, POD-X, POD-S REQ, DES, COD, TST, CICD, SEC, UXA, RUN, DVM Code, tests, pipelines, early runbooks, orchestration logs
    3. Internal Demo / Alpha Have a clickable, explainable demo for internal stakeholders. POD-X, POD-F, POD-S UXA, REQ, DES, MET, ONB, DVM Demo flows, demo data, demo guide, alpha feedback log
    4. Pilot / Beta Safely expose EVA features to limited real users and observe. All pods LVO, MET, SEC, RUN, REQ, DES, DVM Pilot config, usage metrics, pilot runbooks, pilot issues backlog
    5. GA / Scale Run EVA as a governed, multi-client service. POD-F, POD-X, POD-O LVO, MET, RUN, SEC, CICD, DVM SLAs/SLOs, GA runbooks, LiveOps dashboard, cost and usage reports
    6. Day-2 Ops & Improvement Keep EVA reliable, safe and compliant while improving it. POD-O, POD-F, POD-X LVO, MET, RUN, SEC, ONB, DVM Incident reports, MTI-style reports, improvement epics, updated docs

    For each phase, Jarvis selects which agents to use, which repos and pods to touch, and which artefacts must be created or updated.

    3. Phase Details and Jarvis Touchpoints¶

    3.1 Phase 0 – Idea / Intake / Sandbox¶

    Goal: Turn high-level ideas into sandboxed, Canada-grade experiments backed by EVA templates and guardrails.

    Pods: POD-S (Seed & Solution pods), POD-F (Platform and contracts).

    Agents in action: REQ clarifies the problem, users and data classification. DES sketches minimal EVA-aligned architecture (RAG, APIs, UI shell). ONB guides the team into the right sandbox template and explains boundaries. SEC flags obvious security and privacy issues early. DVM records which agents and repos will be involved later.

    Typical artefacts: docs/intake/IDEA-xxx.md – business problem, context and data classification. docs/design/IDEA-xxx-sketch.md – minimal architecture and RAG topology. onboarding/IDEA-xxx-sandbox-setup.md – how to spin up and safely use the sandbox.

    Jarvis workflows: At this phase Jarvis mostly uses jarvis-refine to convert intake notes into early epics and capabilities linked to the CDDs that will be created in Phase 1.

    3.2 Phase 1 – Design & Architecture¶

    Goal: Turn ideas and CDDs into clear requirements, patterns and architecture that fit the EVA spine.

    Pods: POD-F for EVA foundation, agent runtime, RAG and APIM. POD-X for EVA DA / EVA Chat UX, i18n and accessibility. POD-O for early thinking on logging, metrics and trust posture. Solution pods under POD-S are looped in whenever the work is client or domain specific.

    Agents in action: REQ formalizes user stories, acceptance criteria and non-functional needs. DES produces ADRs, contracts and sequence diagrams. SEC reviews data flows and boundaries for classification and privacy. UXA reviews UX flows, copy, EN/FR and a11y constraints. MET proposes what to measure and where (usage, ROI proxies, error rates). DVM coordinates requirements and design across repos and pods.

    Typical artefacts: docs/cdd/cdd-.md – Conceptual Design Document, one per major project. docs/requirements/FEATURE-xxx.md – requirements derived from CDDs. docs/architecture/ADR-xxx.md – decisions with context and consequences. docs/ux/flows-xxx.md – key UI flows and interaction notes. docs/metrics/plan-xxx.md – what will be measured and why.

    Jarvis workflows: jarvis-refine uses CDDs and high-level issues to generate epics, features and first-cut stories. jarvis-plan prepares candidate sprint configurations based on refined, ready items.

    3.3 Phase 2 – Build / Code / Test¶

    Goal: Implement EVA Suite features as working, testable software with pipelines and basic runbooks.

    Pods: POD-F for core services such as eva-api, eva-core, eva-rag, eva-infra. POD-X for EVA UI, component library, i18n/i11y. POD-S for seed and solution repos using the latest capabilities.

    Agents in action: REQ and DES keep requirements and design in sync as reality evolves. COD suggests code changes, skeletal implementations and refactors. TST generates tests, explains failures and proposes coverage improvements. CICD strengthens build and test pipelines. SEC flags security and privacy concerns in PRs. UXA improves UI markup and copy for clarity and accessibility. RUN drafts initial runbooks where behaviour is non-trivial. DVM orchestrates the flow from story to merged PR and keeps orchestration logs.

    Typical artefacts: Source code and tests updated in all EVA repos. CI/CD workflows under .github/workflows or pipelines/ directories. runbooks/-basic.md created or updated for key services. orchestration/FEATURE-xxx-devmaster-log.md capturing which agents helped.

    Jarvis workflows: jarvis-execute reads sprint YAML, assigns work to Copilot coding agent and humans, and updates orchestration logs. jarvis-daily produces sprint status summaries for stand-ups.

    3.4 Phase 3 – Internal Demo / Alpha¶

    Goal: Produce a coherent EVA story and clickable demo for internal stakeholders, across EVA DA, EVA Chat and the platform.

    Pods: POD-X for the visible flows and demo experience. POD-F for stable enough APIs and RAG behaviour. POD-S for safe demo data and configurations.

    Agents in action: UXA polishes flows, screens and copy to tell a clear story in both languages. REQ and DES translate demo feedback into refined stories. MET captures early value proxies such as time saved or usability signals. ONB creates “how to try the alpha” guides. DVM summarises what is in the alpha, what is experimental and what is next.

    Typical artefacts: docs/demo/EVA-alpha-scenarios.md – demo scenarios and scripts. onboarding/demo-how-to.md – instructions for internal testers. metrics/alpha-summary.md – early metrics and qualitative feedback.

    Jarvis workflows: jarvis-review can be used in mini form to summarise each alpha milestone, even before formal pilots.

    3.5 Phase 4 – Pilot / Beta¶

    Goal: Run EVA features with real users in controlled pilots, observe behaviour and improve.

    Pods: All pods are active: POD-F and POD-X deliver the capability. POD-O watches safety, metrics and incidents. POD-S runs domain-specific pilots such as Jurisprudence or AssistMe.

    Agents in action: LVO provides live status, post-deploy checks and quick environment sanity checks. MET builds pilot usage and outcome reports. SEC confirms alignment with AI policies and data classification rules. RUN updates pilot runbooks based on real incidents and edge cases. REQ and DES feed pilot learnings back into new stories and design changes. DVM coordinates “pilot improvement” epics across pods and repos.

    Typical artefacts: config/pilots/.yaml – pilot-specific flags and settings. metrics/pilot--report.md – usage, incidents and value observations. runbooks/pilot-.md – operational playbooks for the pilot. docs/governance/pilot--summary.md – short summaries for RAIDE, PMD and governance bodies.

    Jarvis workflows: jarvis-review produces per-sprint summaries focused on pilot outcomes. jarvis-release and jarvis-uat build and validate UAT candidates for each pilot iteration.

    3.6 Phase 5 – GA / Scale¶

    Goal: Operate EVA as a multi-client, enterprise-grade service with clear SLAs, SLOs and monitoring.

    Pods: POD-F maintains and evolves the platform and agent runtime. POD-X evolves EVA DA and EVA Chat for broad internal use. POD-O owns LiveOps, cost management, safety and trust posture.

    Agents in action: LVO provides daily and weekly health snapshots and incident context. MET maintains EVA Suite metrics by client, app or project. RUN keeps runbooks current and complete. SEC supports ongoing security and compliance evidence. CICD ensures that deployments remain safe, repeatable and traceable. DVM orchestrates large GA epics and cross-pod platform changes.

    Typical artefacts: liveops/weekly-status.md – short weekly status snapshots. metrics/clients/-dashboard.md – per-client or per-domain metrics in markdown form or as links to dashboards. docs/governance/MTI-or-equivalent.md – trust posture and AI risk documentation. docs/SLAs/ and docs/SLOs/ – service expectations per app or client.

    Jarvis workflows: jarvis-daily focuses on production and pilot status as well as sprint status. jarvis-review and jarvis-release integrate operational results into planning.

    3.7 Phase 6 – Day-2 Operations and Continuous Improvement¶

    Goal: Keep EVA reliable, safe, explainable and evolving as new requirements and guidance appear.

    Pods: POD-O at the centre, with POD-F and POD-X executing improvement work. Solution pods (POD-S-*) close the loop from operational findings back to project owners.

    Agents in action: LVO gives first-pass situational awareness across environments. MET tracks long-term trends such as incidents, error rates and adoption. RUN evolves runbooks after each incident and major change. SEC helps assemble evidence for audits and AI governance reviews. ONB keeps new projects and departments aligned with “how EVA works”. DVM coordinates improvement epics and refactoring programs.

    Typical artefacts: incidents/INC-xxxx.md – incident timelines, decisions and follow-ups. runbooks/ – updated operational guidance. metrics/quarterly-eva-summary.md – trend view over quarters. docs/improvement/EPIC-xxxx.md – continuous improvement and refactoring initiatives.

    Jarvis workflows: jarvis-retro uses metrics, incidents and sprint data to propose “what to keep”, “what to change” and “experiments to try” for the next cycles.

    4. How to Use This Map¶

    Use this lifecycle map in three ways.

    As a design scaffold: Identify the current phase, pods, agents and required artefacts when starting or evolving any EVA feature, client or pilot. Link work items in GitHub to the relevant phase and artefacts.

    As a demo narrative: Walk stakeholders through “how EVA is designed, built, piloted and operated” by pointing to phases, pods and agents, backed by real files in eva-meta and the service repos.

    As an implementation checklist: For each phase, confirm that the right repos, pods, Jarvis workflows and DevTools agents are in place, and that expected artefacts exist in Git and are kept up to date.

    This file can live as docs/eva-suite-lifecycle-map.md in eva-meta and act as the reference map for Jarvis DevMaster, the EVA AI Agile Crew and all new solution pods joining the EVA ecosystem.
