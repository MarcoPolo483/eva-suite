    EVA Agile Folders n Maintenance

    Here’s a cleaned-up, aligned version of your folders + housekeeping spec, consistent with the other EVA docs (Charter, Lifecycle, GitHub Blueprint). Plain markdown, no code fences, ready to drop into a .txt or .md file (for example: eva-suite-folder-structure-and-housekeeping.txt).

    EVA SUITE – FOLDER STRUCTURE AND HOUSEKEEPING SCRIPTS

    Version: 0.2 Audience: Jarvis (EVA DevMaster), Copilot Pro+, EVA Agile Crew, AICOE devs

    Goal: Define standard folder structures for EVA repos and the housekeeping scripts Jarvis should use to keep everything clean, consistent, and ready for DevTools and GitHub workflows.

    1. TOP-LEVEL REPO TYPES

    There are two main classes of repos in EVA Suite:

    1. eva-meta Purpose: Cross-project coordination, sprint configs, Jarvis workflows, reports, and global settings for EVA Suite.

    2. Service repos Examples: eva-api, eva-core, eva-ui, eva-ops, eva-infra, eva-seed, plus solution repos (e.g., juris-api, assist-ui). Purpose: Actual application, infra and solution code and tests.

    Each class has its own folder structure and housekeeping scripts, but follows common conventions so Jarvis and Copilot can navigate them in a predictable way.

    1. EVA-META REPO – FOLDER STRUCTURE

    Root layout (all directories are relative to the eva-meta repo):

    docs/ – docs/cdd/ – cdd-eva-suite.txt – cdd-jurisprudence.txt – cdd-assistme.txt – docs/lifecycle/ – eva-suite-lifecycle-map.txt – eva-agentic-pods-map.txt – eva-devmaster-recipes.txt – docs/playbooks/ – eva-github-implementation-blueprint.txt – jarvis-story-from-cdds-to-tested-code.txt – jarvis-devmaster-orchestration-guide.txt

    sprints/ – sprint-001.yaml – sprint-002.yaml – sprint-003.yaml (and so on)

    config/ – pods-map.yaml (which repos belong to which pod) – devmaster-rules.yaml (which agents to apply when) – labels-taxonomy.yaml (canonical label set for all EVA repos)

    reports/ – sprint-001/ – daily-YYYYMMDD.txt – review.txt – retro-input.txt – sprint-002/ – daily-YYYYMMDD.txt – review.txt – retro-input.txt – housekeeping-YYYYMMDD.txt (nightly housekeeping summary, optional)

    uat/ – uat-plan-sprint-001.txt – uat-report-sprint-001.txt – uat-plan-sprint-002.txt – uat-report-sprint-002.txt

    orchestration/ – sprint-001-devmaster-log.txt – sprint-002-devmaster-log.txt

    scripts/ – jarvis_refine.py – jarvis_plan.py – jarvis_execute.py – jarvis_daily.py – jarvis_review.py – jarvis_release.py – jarvis_uat.py – jarvis_retro.py – check_sprint_yaml.py – sync_labels_all_repos.py – check_repo_health.py – ensure_standard_structure.py

    .github/ – workflows/ – jarvis-refine.yml – jarvis-plan.yml – jarvis-execute.yml – jarvis-daily.yml – jarvis-review.yml – jarvis-release.yml – jarvis-uat.yml – jarvis-retro.yml – housekeeping-nightly.yml

    .eva-meta-settings/ – repositories-list.yaml (canonical list of EVA repos and their Git URLs) – security-policy.txt (meta-level security and usage notes for Jarvis and DevTools)

    1. SERVICE REPOS – COMMON FOLDER STRUCTURE

    Each service repo (eva-api, eva-core, eva-ui, eva-ops, eva-infra, eva-seed, plus solution repos like juris-api or assist-ui) should share a common shape so Jarvis and Copilot can reuse patterns.

    Typical layout for an application or backend repo (e.g., eva-api, eva-ui):

    src/ or app/ – Primary source code, organized by language or framework conventions.

    tests/ or test/ – Automated tests (unit, integration and possibly end-to-end).

    docs/ – docs/requirements/ – REQ-XXXX-feature-name.txt – docs/architecture/ – ARCH-overview.txt – ARCH-component-X.txt – docs/ux/ – UX-flows.txt (if relevant) – docs/api/ – openapi.yaml or equivalent API contract

    runbooks/ – RUN-deployment.txt – RUN-troubleshooting.txt – RUN-rollback.txt

    orchestration/ – sprint-XXX-log.txt (optional per-sprint log generated by Jarvis) – devtools-experiments.txt (notes when DevTools agents are trialed here)

    config/ – config/appsettings/ (non-secret config templates only) – config/pilots/ (pilot-specific config files, if any)

    scripts/ – local-dev-setup.ps1 or .sh – run-tests-local.ps1 or .sh – format-and-lint.ps1 or .sh

    .github/ – workflows/ – ci-build-and-test.yml – ci-lint.yml – ci-security-scan.yml – cd-deploy-dev.yml (where applicable)

    .eva-pod.yaml (pod and agent mapping for this repo)

    .github/copilot-instructions.txt – Guidance for Copilot (stack, layout, testing command, coding standards, do/do-not rules).

    Infra-specific repos (e.g., eva-infra) follow the same spirit with infra-oriented folders:

    terraform/ or infra/ modules/ environments/ docs/infra/ scripts/ (plan/apply wrappers, validation scripts) .github/workflows/ci-infra.yml

    1. HOUSEKEEPING SCRIPTS – EVA-META

    The following scripts live under eva-meta/scripts and are invoked by scheduled workflows or on demand.

    4.1 check_sprint_yaml.py

    Purpose: Validate all sprint files under sprints/ so Jarvis and humans can trust them.

    Responsibilities: – Parse each sprint-XXX.yaml. – Validate required fields: sprint, team, start_date, end_date, goal, items. – For each item: – Verify the issue URL points to a real GitHub issue (via API or dry-run). – Optionally verify labels such as level:story, level:task, ready-for-sprint. – Produce a summary report indicating: – Sprints valid vs invalid. – Missing issues or mislabelled items.

    Output: – reports/sprint-validation-YYYYMMDD.txt, or appended to reports/housekeeping-YYYYMMDD.txt.

    4.2 sync_labels_all_repos.py

    Purpose: Ensure all EVA repos share a consistent label taxonomy.

    Inputs: – config/labels-taxonomy.yaml (canonical labels, colours, descriptions). – .eva-meta-settings/repositories-list.yaml (list of repos Jarvis manages).

    Responsibilities: – For each repo in repositories-list.yaml: – Create labels that are missing. – Update existing labels that differ (name, colour, description). – Optionally mark labels not in the taxonomy as deprecated (e.g., prefix “deprecated/”).

    Trigger: – housekeeping-nightly.yml on a schedule (for example, once per day).

    4.3 check_repo_health.py

    Purpose: Identify repos with missing mandatory files or obvious health issues.

    Checks per repo: – Presence of .eva-pod.yaml. – Presence of .github/copilot-instructions.txt. – Presence of at least one CI workflow in .github/workflows. – Presence of docs/, tests/ and runbooks/ folders (or record if intentionally absent). – Optionally, PRs older than a threshold (e.g., 30 days) without activity.

    Output: – reports/health-report-YYYYMMDD.txt summarizing: – Missing files per repo. – Repos with no CI. – Repos with stale PRs.

    4.4 ensure_standard_structure.py

    Purpose: Optionally auto-create missing standard folders and stub files for repos being onboarded into EVA Suite.

    Behaviour: – For each repo in repositories-list.yaml: – Check for docs/, docs/requirements/, docs/architecture/, runbooks/, orchestration/. – If missing, create the folder and add a simple placeholder file such as: – docs/requirements/README.txt – runbooks/README.txt – Never overwrite existing files; only create when absent.

    Use: – Run manually when a new repo is registered in repositories-list.yaml. – Can also be called by a GitHub workflow in “report-only” mode to show what would be created.

    1. HOUSEKEEPING – SERVICE REPOS

    Each service repo should include a minimal set of local helper scripts in scripts/.

    5.1 local-dev-setup (shell or PowerShell)

    Purpose: Help developers quickly bootstrap a working dev environment for that repo.

    Typical actions: – Install local dependencies or tools (within allowed boundaries). – Create or update local config files from templates in config/appsettings. – Print clear instructions for running the app and tests.

    5.2 run-tests-local

    Purpose: Standard, single entry point for running tests locally.

    Behaviour: – Run unit tests. – Optionally run fast integration tests. – Exit with non-zero code on failure so CI and Jarvis workflows can depend on it.

    Mapping to CI: – ci-build-and-test.yml should call the same command used here.

    5.3 format-and-lint

    Purpose: Apply formatting and linting locally before committing, to keep repos clean and Copilot-friendly.

    Behaviour: – Run language-specific formatters (prettier, black, gofmt, etc.). – Run linters (eslint, flake8, etc.). – Optionally fix simple issues automatically when safe.

    These scripts are highly recommended, but Jarvis only depends on them indirectly through CI workflows.

    1. HOUSEKEEPING WORKFLOWS IN EVA-META

    The housekeeping-nightly.yml workflow (under eva-meta/.github/workflows) orchestrates nightly maintenance.

    Suggested steps:

    1. Checkout eva-meta.
    2. Run check_sprint_yaml.py.
    3. Run sync_labels_all_repos.py.
    4. Run check_repo_health.py.
    5. Optionally run ensure_standard_structure.py in report-only mode.

    Results: – Write or update reports/housekeeping-YYYYMMDD.txt with a concise summary. – Optionally append a note to a dedicated “EVA Housekeeping” issue in eva-meta or a central tracking repo.

    1. HOW TO ASK COPILOT TO IMPLEMENT THIS

    Once this spec is present in eva-meta (under docs/ or at the root), you can use prompts such as:

    – “Read the EVA Suite – Folder Structure and Housekeeping Scripts spec and:

    1. create the folder skeleton in eva-meta (docs, sprints, config, reports, uat, orchestration, scripts, .github/workflows),
    2. generate placeholder files and a basic housekeeping-nightly.yml workflow that calls stub Python scripts in scripts/.”

    – “In eva-api, apply the service repo structure from the spec: create docs/requirements, docs/architecture, runbooks, orchestration, scripts, and .github/workflows/ci-build-and-test.yml without overwriting existing files. Add a minimal .eva-pod.yaml and .github/copilot-instructions.txt.”

    – “Generate Python script stubs for check_sprint_yaml.py, sync_labels_all_repos.py, check_repo_health.py and ensure_standard_structure.py according to the responsibilities described in the spec, wired to use GITHUB_TOKEN and log results under reports/.”

    This spec gives Copilot and Jarvis a concrete folder layout and housekeeping task list so they can materialize the EVA + GitHub blueprint as real files, scripts and nightly workflows.
