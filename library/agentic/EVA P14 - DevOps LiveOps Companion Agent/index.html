    EVA DevTools Agent Pattern – DevOps LiveOps Companion Agent¶

    Subtitle: Eyes on EVA in Real Time – Without Taking the Wheel
    Version: Draft 0.1
    Date: YYYY-MM-DD

    1. Pattern Overview¶

    Agent Name: EVA-LiveOps-Agent (DevOps LiveOps Companion Agent)
    Category: DevTools / DevOps / SRE / LiveOps
    Audience: DevOps, SRE, Platform Engineers, Product Owners, Incident Commanders

    The EVA-LiveOps-Agent helps the EVA Agile Crew:

    • Maintain situational awareness of deployments, health, and incidents in near real time.
    • Connect changes (deploys, config, feature flags) with observed behaviour (errors, latency, incidents).
    • Provide human-ready summaries and guardrail suggestions during normal operations and early incident stages.

    It is a co-pilot for LiveOps: it observes, summarizes, and suggests – it does not execute operational actions on its own.

    2. Intended Use & Non-Use¶

    2.1 Intended Use¶

    Use EVA-LiveOps-Agent to:

    • Provide status snapshots on demand:
    • “What changed in the last 24h on EVA Domain Assistant?”

    • “What is the current health of the Jurisprudence stack?”

    • Support incident triage:

    • Summarize alerts, recent changes, and existing runbooks.

    • Suggest relevant runbook sections or mitigation options (for humans to execute).

    • Assist with operational reviews:

    • Provide change + incident correlation over a period.
    • Highlight recurring hotspots and candidate areas for hardening.

    Typical scenarios:

    • 09:00 standup: “Give us a LiveOps snapshot for EVA services.”
    • Early incident: “Summarize what you see for service X and which runbooks apply.”
    • Post-release: “Did error rates or latency change after deploy Y?”

    2.2 Non-Use¶

    Do not use EVA-LiveOps-Agent to:

    • Restart services, scale infrastructure, or flip feature flags directly.
    • Bypass or replace formal incident management processes or on-call rotation.
    • Act as a monitoring platform by itself; it relies on existing tools for data.

    It is an advisor, not an SRE replacement or orchestrator.

    3. Role in the GC Agentic Template¶

    • Vol. 1 – Platform & Agentic Requirements

    • Surfaces real-time and historical signals needed to validate trust posture, autonomy levels, and safety (e.g., kill switch usage, error spikes).

    • Vol. 2 – Engineering & Delivery Guide

    • Connects deployments and configuration changes to observed behaviour in production and pre-production.

    • Vol. 3 – LiveOps & Governance Model

    • Works closely with:
      • Runbook & SRE Agent (P9) for procedures.
      • Metrics & Insight Agent (P10) for trends.

    • Helps operational leaders answer “What is happening right now and what changed recently?”

    • Vol. 4 – Use Cases, Patterns & Templates

    • Offers re-usable LiveOps views for GC services using EVA patterns.

    4. High-Level Behaviour¶

    EVA-LiveOps-Agent:

    1. Pulls and correlates data from:
    2. Monitoring (metrics, alerts, SLOs).
    3. Logging / tracing.
    4. Deployment tools (who deployed what, where, when).

    5. Feature flag/config platforms.

    6. Builds human-readable snapshots of the current state of a service, project, or environment.

    7. Highlights anomalies and possible causes, in collaboration with other agents:

    8. For example, “Error rate increased after deploy R12; refer to Runbook section X.”

    9. Suggests guardrails and next steps:

    10. Add alerts, tighten gates, create or update runbooks, investigate specific components.

    All suggestions are advisory and require human action.

    5. Inputs & Outputs¶

    5.1 Inputs¶

    • Service and environment identifiers (e.g., eva-domain-assistant / prod).
    • Monitoring feeds (metrics, SLOs, alert states).
    • Deployment metadata:
    • Releases, versions, build IDs, deployment times, initiators (anonymized if needed).
    • Feature flag/config change logs.
    • Incident records (if any currently open or recently resolved).
    • Runbook index and metrics summaries from P9 and P10.

    5.2 Outputs¶

    • LiveOps snapshots:
    • Current health (OK/degraded/incident) with supporting metrics.

    • List of recent changes (deploys, configs, flags).

    • Anomaly summaries:

    • Short explanations of current alerts or metric anomalies.

    • Hypotheses linking them to recent changes.

    • Operational recommendations:

    • “Consult runbook section X.”
    • “Consider temporarily disabling feature flag Y if permitted.”

    • “Create a follow-up task to add alert Z.”

    • Operational review reports:

    • For a given period, correlate changes, incidents, and performance shifts.

    Outputs are typically rendered as Markdown and in tool-specific comments/notes, tagged generated-by:eva-liveops-agent.

    6. Triggers¶

    • On-demand status checks:

    • /eva-liveops status eva-domain-assistant prod

    • Alert-driven triggers:

    • When an alert reaches a specific severity, the incident bridge or channel can call:

      • /eva-liveops summarize service=eva-foundation environment=prod.

    • Post-deploy checks:

    • After a deployment completes:

      • /eva-liveops post-deploy-check RELEASE-ID.

    • Operational review:

    • Before weekly LiveOps meetings:
      • /eva-liveops weekly-review SERVICE-X 7d.

    All triggers are explicit and logged.

    7. Interaction Flows¶

    7.1 Live Status Snapshot Flow¶

    1. On-call or DevOps requests:

    2. /eva-liveops status eva-domain-assistant prod.

    3. Runtime gathers for that service/environment:

    4. Current alerts and their severities.
    5. Key metrics (error rate, latency, traffic, resource usage).
    6. Recent deployments and config changes (e.g., last 24h).

    7. Any active incidents.

    8. Agent composes a status snapshot:

    9. “Overall state: Degraded / Healthy.”

    10. Top alerts and their short explanation.
    11. Last N changes with timestamps.

    12. Quick pointers:

      • “Runbook sections relevant: A, B.”
      • “No recent releases; likely environmental or external dependency.”

    13. Output appears in incident/ops channel and optionally written as liveops/status-YYYYMMDD-HHMM.md.

    14. Humans decide what actions to take.

    7.2 Early Incident Support Flow¶

    1. An incident starts (alerts fire and/or humans declare an incident).
    2. Someone calls:

    3. /eva-liveops incident-view SERVICE-X INCIDENT-ID.

    4. Runtime gathers:

    5. Recent metrics, logs, alerts, deploys, and runbook references.

    6. Any related incidents in the last N days.

    7. Agent provides:

    8. Timeline of recent changes and alerts.

    9. Suggestions:

      • “This looks similar to incident INC-1234 on DATE; see runbook section Z.”
      • “Last deploy R10 touched component ‘ingestion’ which shows increased latency.”

    10. Output is used in the incident bridge to inform human decisions and check possible mitigations.

    7.3 Post-Deploy Check Flow¶

    1. A deployment completes successfully for SERVICE-Y.
    2. CI/CD Guardian or human triggers:

    3. /eva-liveops post-deploy-check RELEASE-ID.

    4. Runtime collects for a short “burn-in” window:

    5. Error rates, latency, key business metrics, alerts.

    6. Previous baseline metrics (e.g., previous version).

    7. Agent compares:

    8. “Error rate: stable/increased/decreased.”
    9. “Latency: stable/increased/decreased.”

    10. “Any new alerts triggered?”

    11. Output includes:

    12. A short release health report.

    13. Recommendations:

      • “Monitor for X more minutes.”
      • “If error rate continues to increase, consult runbook Y and consider rollback.”

    14. Humans decide if further action is needed.

    7.4 Weekly LiveOps Review Flow¶

    1. Before a weekly LiveOps meeting, someone requests:

    2. /eva-liveops weekly-review EVA-SUITE 7d.

    3. Runtime gathers:

    4. Incidents, alerts, major deploys, and key metrics across services.

    5. Agent summarizes:

    6. “Number of incidents, MTTR, notable degradations.”
    7. “High-change areas vs low-change areas.”

    8. “Recurring hotspots and candidate improvements.”

    9. Output:

    10. A Markdown report used as the agenda backbone for the LiveOps meeting.

    11. Follow-up actions:

    12. Feed into Runbook updates, CI/CD improvements, or backlog items.

    8. Profile Configuration (EVA Runtime)¶

    Conceptual profile (YAML-style):

    agent_id: eva-liveops-agent
display_name: "EVA DevOps LiveOps Companion Agent"
description: "Provides real-time and historical operational context, correlating changes with health and incidents."
autonomy_level: advisory   # Observes and suggests; no direct actions.

triggers:
  - type: manual_command
    command: "/eva-liveops status"
  - type: manual_command
    command: "/eva-liveops incident-view"
  - type: manual_command
    command: "/eva-liveops post-deploy-check"
  - type: manual_command
    command: "/eva-liveops weekly-review"
  - type: alert_webhook
    event: "critical_alert"

context_requirements:
  - service_and_environment_metadata
  - monitoring_and_alert_feeds
  - deployment_and_release_history
  - feature_flag_and_config_change_logs
  - incident_records_optional
  - runbook_index_optional
  - metrics_summaries_optional

outputs:
  - type: status_snapshots
  - type: incident_context_summaries
  - type: post_deploy_reports
  - type: weekly_liveops_reports

llm_profile:
  system_prompt_ref: devtools_global
  role_prompt_ref: liveops_agent
  output_schema_ref: liveops_companion_v1

policies:
  require_human_review_flag: true
  no_direct_control_of_infra: true
  anonymize_individual_operators: true
  mark_items_with_tag: "generated-by:eva-liveops-agent"

    9. Prompting Model (High Level)¶

    9.1 System Prompt (DevTools Global – reused)¶

    Ensures the agent:

    • Works strictly as a context and explanation layer atop existing monitoring/ops data.
    • Avoids making promises or guarantees (“system is fine forever”).
    • Respects privacy of operators (no call-out of individuals for blame).

    9.2 Role Prompt (DevOps LiveOps Companion)¶

    Essence of the liveops_agent role:

    • You are the EVA DevOps LiveOps Companion Agent.
    • You synthesize monitoring, logs, deployments, and incidents into clear situational awareness for humans.

    You MUST:

    • Present information clearly and concisely, highlighting:
    • Current state.
    • Recent changes.

    • Relevant historical context (similar incidents, prior patterns).

    • Distinguish between:

    • Facts (metrics, events).
    • Hypotheses (possible causes, correlations).
    • Recommendations (next steps, runbooks, checks).

    You MUST NOT:

    • Say you have changed any system; always clarify that actions are suggestions.
    • Directly blame individuals; focus on systems and events.
    • Overstate certainty – when unsure, say so and suggest where to look.

    Example structured output:

    {
  "scope": {
    "service": "eva-domain-assistant",
    "environment": "prod",
    "time_window": "last_60m"
  },
  "current_state": {
    "status": "degraded",
    "key_metrics": {
      "error_rate_percent": 3.2,
      "latency_p95_ms": 1800,
      "request_volume": 12000
    },
    "active_alerts": [
      {
        "id": "ALERT-500-RATE",
        "severity": "high",
        "summary": "HTTP 500 rate above 2% for 15 minutes"
      }
    ]
  },
  "recent_changes": [
    {
      "type": "deployment",
      "id": "release-2025-11-20-01",
      "timestamp": "2025-11-20T13:05Z",
      "notes": "Backend API update for search filters"
    }
  ],
  "historical_context": {
    "similar_incidents": [
      {
        "incident_id": "INC-1234",
        "date": "2025-10-02",
        "root_cause": "Unoptimized DB query after feature rollout"
      }
    ]
  },
  "insights": [
    "Error rate increase started shortly after release-2025-11-20-01.",
    "Pattern resembles INC-1234, where search queries caused DB saturation."
  ],
  "recommendations": [
    "Monitor DB and search-related metrics; if they remain elevated, consider rollback of release-2025-11-20-01.",
    "Consult runbook section 'Search performance issues' for detailed triage steps.",
    "Create a follow-up task to add more targeted alerting on search latency."
  ]
}

    10. Guardrails & Governance¶

    • No direct control over infrastructure

    • The agent does not:

      • Restart services, change configurations, or flip flags.
      • Execute scripts, Terraform, or cloud operations.

    • Operator privacy

    • When referencing changes, it focuses on:

      • Releases, pipelines, systems – not naming individuals, unless explicitly requested and policy permits.

    • Incident process alignment

    • It supplements, not replaces:

      • Incident commander role.
      • Incident timelines.
      • Formal communication and reporting.

    • Transparency

    • Every report:

      • Indicates data sources and time windows.
      • Is tagged generated-by:eva-liveops-agent.

    11. Implementation Notes for EVA¶

    • Runs on DevTools / LiveOps Runtime:

    • Trigger → Fetch metrics/logs/deploy/incident context → Policy filters → LLM → Snapshot/report → Logging.

    • Integrations (conceptual):

    • Monitoring/metrics:

      • Azure Monitor / Application Insights / Prometheus, etc.

    • Logging/tracing:

      • Central log platform, correlation IDs from APIM and services.

    • CI/CD:

      • GitHub Actions, Azure DevOps, or equivalent for deployment metadata.

    • Feature flags/config:

      • Flag platform or config repo.

    • Incident & runbook:

      • Incident tracker and runbook repository.

    • Storage:

    • LiveOps snapshots and reports can be:

      • Stored in a liveops/ folder in eva-ops or eva-liveops repos.
      • Linked from incident records or dashboards.

    • Feedback loop:

    • Works hand-in-hand with:

      • Runbook Agent (P9) to create/update runbooks based on incidents.
      • Metrics Agent (P10) for trends.
      • CI/CD Guardian (P8) when repeated issues link to deployments or pipelines.

    12. Adoption & Maturity¶

    • Phase 1 – Manual Snapshot Companion

    • On-demand /status and /weekly-review reports only.

    • Phase 2 – Incident Sidekick

    • Integrated into incident channels for early context and summaries.

    • Phase 3 – Post-Deploy Health Partner

    • Automatically called after key deployments to provide early health checks.

    • Phase 4 – Full LiveOps Hub Companion

    • Integrated into an EVA LiveOps console:

      • Single-pane-of-glass view.
      • Deep links to dashboards, runbooks, and incidents.

    13. Benefits Summary¶

    • Improves situational awareness without adding more dashboards for humans to manually stitch together.
    • Reduces time to understand what changed when something goes wrong.
    • Strengthens the connection between deployments, configs, and runtime behaviour.
    • Provides a GC-ready LiveOps pattern that any department can reuse on top of its existing monitoring/devops stack.

    14. Alignment with EVA Pods, Jarvis and the Lifecycle¶

    The DevOps LiveOps Companion Agent sits primarily in:

    • POD-O – Operations & LiveOps

    • Owns day-2 reliability, safety, trust posture and cost.

    • With strong ties to:

    • POD-F for deployment events, infra changes, APIM, RAG/runtime health.

    • POD-X for UX outages, DA/Chat incidents and user-visible issues.
    • POD-S for pilots and sandboxes that need “lightweight LiveOps”.

    Lifecycle alignment:

    • Phase 2 – Build / Test:

    • Starts wiring basic signals (builds, deployments, smoke checks) so they are visible to LiveOps dashboards later.

    • Phase 3 – Internal Demo / Alpha:

    • Provides “what changed recently?” context when demo environments misbehave.

    • Phase 4 – Pilot / Beta:

    • Becomes the primary way to see pilot health, incidents and post-deploy checks.

    • Phase 5 – GA / Scale:

    • Is the standard pattern for GA LiveOps across EVA clients/spaces.

    • Phase 6 – Day-2 Ops:

    • Feeds trend reports and improvement epics (refactoring, resilience, UX issues).

    Relationship to other DevTools patterns (P01–P15):

    • Consumes metrics and signals curated by P10 – Metrics & Insight Agent.
    • Relies on P09 – Runbook & SRE Agent to provide actionable procedures.
    • Surfaces security/compliance signals from P11 – Security & Compliance Helper.
    • Coordinates with P08 – CICD Guardian Agent so deployment events and health checks line up.
    • Is orchestrated at the top level by P15 – DevMaster (DVM), especially during incidents or post-deploy checks.

    15. How Jarvis and Copilot Should Use This Pattern¶

    When this document (P14) is present in a repo or in eva-meta/docs/playbooks:

    • Jarvis DevMaster (DVM) should:

    • Use P14 as the contract for anything that calls itself “LiveOps companion”, “status bot”, or “Ops agent”.

    • Ensure that any new EVA solution that reaches Pilot or GA has:

      • At least one LiveOps view or report following this pattern.
      • A minimal set of signals: recent deployments, key SLOs, open incidents, links to relevant runbooks.
      • When running jarvis-daily, jarvis-review, or jarvis-uat, pull in LiveOps Companion insights (for example: “last 24h incidents”, “recent deploys affecting this sprint/scope”).

    • Copilot / coding agents should:

    • When asked to “implement a LiveOps companion” for a repo:

      • Read P14 and generate:

      • A small LiveOps status endpoint or script (for example /liveops/status or scripts/liveops-status.ps1).

      • Minimal wiring into monitoring/metrics (logs, traces, key health checks).
      • Markdown snippets for dashboards or comments that match the examples in this pattern.

      • When asked to “add LiveOps to a new pilot/space”:

      • Propose:

      • Where to log pilot-specific metrics.

      • How to tag events with project, space, sprint, release headers so EVA LiveOps can aggregate them.
      • Which runbooks (P09) should be created or extended.

    • In GitHub / EVA-meta context:

    • jarvis-daily.yml and jarvis-review.yml can:

      • Call a small script (for example eva-meta/scripts/liveops_snapshot.py) that:

      • Queries LiveOps endpoints or metrics backends.

      • Embeds a short “LiveOps Companion” section in:

        • reports/sprint-XXX/daily-YYYYMMDD.txt
        • reports/sprint-XXX/review.txt
          • Use P14 as the reference for what belongs in that section (deploys, incidents, SLOs, runbooks, “notable changes”).

    By treating this document as the single source of truth for the LiveOps Companion pattern, Jarvis, Copilot and human SREs all share the same expectations about:

    • What “good LiveOps” looks like in EVA.
    • Which signals matter.
    • How LiveOps integrates with sprints, pilots, GA and day-2 operations.
