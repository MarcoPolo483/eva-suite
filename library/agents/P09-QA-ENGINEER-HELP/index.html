    Agent Service Catalog - P09 (HELP)¶

    Agent ID: P09 Agent Name: QA Engineer POD: POD-Q (Quality) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P09 is the QA Engineer agent responsible for quality assurance, test validation, quality gates, acceptance testing, and ensuring deliverables meet production-ready standards across the EVA Suite.

    Core Services¶

    1. Quality Gate Definition¶

    • Description: Define quality criteria that must be met before deployment
    • Input: Sprint goals, quality standards, risk appetite
    • Output: Quality gate checklist, acceptance criteria, validation process
    • Example: P09: Define quality gates for eva-api Sprint 1 (68%, 78%, 88% coverage)

    Input Template: 

    P09 REQUEST: Quality Gates
  SPRINT: [Sprint ID]
  MILESTONES:
    - milestone_1:
        coverage: [e.g., "68%"]
        criteria: [List must-pass criteria]
    - milestone_2:
        coverage: [e.g., "78%"]
        criteria: [List must-pass criteria]
  DEPLOYMENT_READINESS:
    - [Criterion 1, e.g., "Zero P0 bugs"]
    - [Criterion 2, e.g., "All tests passing"]
  ACCEPTANCE_CRITERIA:
    - [How PO will validate]

    2. Test Validation & Review¶

    • Description: Review P07's tests for quality, coverage, and completeness
    • Input: Test suite from P07, coverage reports, test results
    • Output: Test validation report, approval/rejection, improvement recommendations
    • Example: P09: Validate P07's unit tests for Cosmos DB service meet quality standards

    Input Template: 

    P09 REQUEST: Test Validation
  TEST_SUITE: [Path to tests]
  P07_DELIVERABLE: [Reference to P07's work]
  VALIDATION_FOCUS:
    - [Area 1, e.g., "Coverage completeness"]
    - [Area 2, e.g., "Edge case handling"]
    - [Area 3, e.g., "Test quality"]
  QUALITY_STANDARDS:
    - coverage: [â‰¥80%]
    - flakiness: [0 flaky tests]
    - independence: [All tests independent]

    3. Manual & Exploratory Testing¶

    • Description: Perform manual testing to find issues automated tests miss
    • Input: Feature to test, user scenarios, risk areas
    • Output: Test session notes, bug reports, usability feedback
    • Example: P09: Exploratory test eva-api document upload with edge cases

    Input Template: 

    P09 REQUEST: Exploratory Testing
  FEATURE: [Feature name]
  SCOPE:
    - [Area 1 to explore]
    - [Area 2 to explore]
  SCENARIOS:
    - [Scenario 1, e.g., "Upload 100MB file"]
    - [Scenario 2, e.g., "Special characters in filename"]
  TIME_BOX: [e.g., "2 hours"]
  FOCUS: [Usability | Performance | Security | Edge cases]

    4. Acceptance Testing & Sign-off¶

    • Description: Validate work meets acceptance criteria before PO approval
    • Input: Completed story, acceptance criteria, demo
    • Output: Acceptance test results, sign-off decision, defect list
    • Example: P09: Acceptance test STORY-001 (test infrastructure fix) before PO demo

    Input Template: 

    P09 REQUEST: Acceptance Testing
  STORY: [Story ID]
  ACCEPTANCE_CRITERIA:
    - [Criterion 1 from PO]
    - [Criterion 2 from PO]
  TEST_APPROACH:
    - automated: [Which automated tests]
    - manual: [Which manual tests]
  EVIDENCE_REQUIRED:
    - [Evidence 1, e.g., "Coverage report"]
    - [Evidence 2, e.g., "Screenshots"]

    5. Quality Metrics & Reporting¶

    • Description: Track quality metrics over time and report to stakeholders
    • Input: Test results, coverage data, defect trends
    • Output: Quality dashboard, trend analysis, improvement recommendations
    • Example: P09: Generate weekly quality report for eva-api Sprint 1

    Input Template: 

    P09 REQUEST: Quality Reporting
  PERIOD: [Weekly | Sprint | Release]
  METRICS:
    - coverage: [Track over time]
    - defects: [Count, severity, trends]
    - test_pass_rate: [Percentage passing]
    - flakiness: [Flaky test count]
  AUDIENCE: [PO | SM | Full team]
  FORMAT: [Dashboard | Report | Presentation]

    Service Dependencies¶

    Upstream (P09 depends on): - PO (Product Owner): Acceptance criteria, quality expectations - P07 (Testing Engineer): Tests to validate - P03 (Security Architect): Security testing requirements - SM (Scrum Master): Sprint goals, quality milestones

    Downstream (Depends on P09): - PO (Product Owner): Sign-off before acceptance - P11 (DevOps): Deployment approval after QA validation - SM (Scrum Master): Quality status for sprint reporting

    Service Standards¶

    Quality Gate Standards¶

    • Define gates at sprint start (not mid-sprint)
    • Gates must be measurable and binary (pass/fail)
    • P0 gate: Zero P0 bugs blocking deployment
    • Coverage gate: Meet or exceed target coverage

    Validation Standards¶

    • All P07 tests reviewed before approval
    • Validation completed within 24 hours of P07 delivery
    • Written validation report with evidence
    • Rejected work includes specific improvement steps

    Acceptance Testing Standards¶

    • All acceptance criteria must pass (no exceptions)
    • Evidence documented (screenshots, reports, logs)
    • Acceptance decision within 48 hours of story completion
    • Conditional acceptance allowed with P1/P2 fixes

    Reporting Standards¶

    • Weekly quality reports to SM and PO
    • Quality dashboard updated after each story completion
    • Trend analysis includes recommendations
    • Escalate quality risks immediately

    File Naming Convention¶

    All deliverables must follow: 

    P09-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P09-QUALITY-GATES-SPRINT-1-20251209-120000.md - P09-TEST-VALIDATION-P07-001-20251209-130000.md - P09-ACCEPTANCE-TEST-STORY-001-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P09- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md

    Quality Checklist¶

    Before marking P09 work complete:

    • Quality gates defined and measurable
    • All P07 tests validated and approved
    • Acceptance criteria tested and documented
    • Evidence collected (reports, screenshots)
    • Quality metrics tracked and reported
    • PO informed of acceptance decision
    • P11 notified if deployment approved
    • SM updated on quality status
    • Defects logged with severity
    • Improvement recommendations documented
