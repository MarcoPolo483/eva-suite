    Chapter P05: Scaffolder Agent¶

    Pattern Code: P05 Short Code: SCA Name: Scaffolder Agent Type: DevTools Pattern Scope: Repository Initialization & Standardization Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Scaffolder Agent (P05-SCA) accelerates EVA Suite development by bootstrapping complete repository structures from battle-tested templates. Think of it as the "project generator" that transforms a simple command into a production-ready repository with documentation, CI/CD pipelines, testing infrastructure, and bilingual support‚Äîall in under 60 seconds.

    In modern multi-repository architectures, consistency is critical. Every new service, library, or UI component should follow the same conventions: same folder structure, same tooling, same quality gates, same documentation patterns. Without standardization, each repo becomes a unique snowflake, increasing cognitive load and slowing down development.

    The Problem P05 Solves: - Setup Time: Creating a new repo from scratch takes hours (boilerplate code, config files, CI/CD, docs) - Inconsistency: Each developer invents their own project structure, tooling, and conventions - Missing Best Practices: Forgot to add test coverage reporting? Security scanning? Bilingual docs? Each omission creates technical debt - Documentation Debt: New repos often skip README, ADRs, and contribution guidelines - CI/CD Gaps: Manual deployment scripts instead of automated pipelines

    The Value P05 Delivers: - Speed: Repository creation reduced from 4+ hours to <5 minutes (including smoke test) - Consistency: All repos follow identical structure, tooling, and conventions - Quality: Built-in test coverage, linting, security scanning, accessibility checks - Compliance: Bilingual documentation (EN-CA/FR-CA) and GC Design System integration from day one - Maintainability: Standardized structure makes cross-repo work predictable and fast

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P05-SCA serves as the Repository Initialization Specialist for the EVA Autonomous Agile Command. It ensures:

    1. Rapid Bootstrapping: New repos created in minutes, not hours or days
    2. Standardization: All EVA repos follow identical structure and conventions
    3. Best Practices: Test coverage, linting, security scanning baked in from start
    4. Documentation First: README, ADRs, and contribution guidelines auto-generated
    5. CI/CD Ready: GitHub Actions workflows included for build, test, deploy

    Mission Statement¶

    "Enable rapid, consistent, and quality-assured repository creation through standardized templates, automated scaffolding, and validated smoke tests."

    Value Proposition¶

    For Product Owners: - Accelerate feature delivery (new services/libraries ready in minutes) - Reduce onboarding time (predictable structure across all repos) - Built-in compliance (bilingual docs, GC Design System, accessibility)

    For Developers: - Skip boilerplate setup (focus on business logic immediately) - Familiar structure in every repo (no learning curve) - Proven tooling (linters, test frameworks, CI/CD already configured)

    For Operations: - Consistent deployment patterns (all repos use same CI/CD workflows) - Predictable dependency management (lock files, pinned versions) - Security by default (automated scans in every repo)

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Repository Structure Generation
    2. Create standardized folder layout (src/, tests/, docs/, .github/)
    3. Initialize package management (package.json, pyproject.toml, etc.)
    4. Configure editor settings (.editorconfig, .vscode/)

    5. Set up version control (.gitignore, .gitattributes)

    6. Documentation Scaffolding

    7. Generate README with all required sections (Quick Start, Architecture, Development)
    8. Create initial ADR (0001-initial-architecture.md)
    9. Populate CONTRIBUTING.md with team conventions
    10. Create LICENSE file (MIT, Apache 2.0, etc.)

    11. Generate README.fr.md for bilingual compliance (if --bilingual flag)

    12. Tooling Configuration

    13. Set up linting (ESLint for JS/TS, Pylint/Ruff for Python)
    14. Configure testing (Vitest for JS/TS, pytest for Python)
    15. Initialize formatting (Prettier for JS/TS, Black for Python)

    16. Add security scanning (npm audit, bandit)

    17. CI/CD Pipeline Creation

    18. Generate GitHub Actions workflows (ci.yml, cd.yml, security-scan.yml)
    19. Configure automated testing on PR
    20. Set up coverage reporting (Codecov integration)
    21. Add deployment workflows (staging, production)

    Supporting Responsibilities¶

    1. Template Maintenance
    2. Keep templates up-to-date with latest tooling versions
    3. Incorporate new best practices as they emerge

    4. Maintain template variants (api, ui, lib, infra, pipeline)

    5. Smoke Testing

    6. Validate generated repos build successfully
    7. Ensure tests pass out-of-the-box

    8. Verify CI/CD workflows execute without errors

    9. Integration Points

    10. Coordinate with P04 (Repo Librarian) to ensure documentation standards met
    11. Provide metadata to P01 (Agent Patterns Toolbox) for pattern registry
    12. Support P15 (Dev Master Orchestrator) for automated repo creation

    üöÄ When to Use P05¶

    Primary Use Cases¶

    1. Creating a New Microservice¶

    Scenario: Product Owner approves new feature requiring dedicated API service.

    Action: Use P05 to scaffold API service repo with Express/Fastify, TypeScript, and CI/CD.

    Example: 

    # Create new API service for notifications
eva-scaffold create eva-notifications \
    --type api \
    --pod POD-F \
    --owner P15-DVM \
    --bilingual \
    --gc-design

# Scaffolder generates:
# - eva-notifications/ (repo folder)
# - src/index.ts (Express server boilerplate)
# - tests/index.test.ts (sample test)
# - .github/workflows/ci.yml (automated testing)
# - README.md + README.fr.md (bilingual docs)
# - docs/adr/0001-initial-architecture.md (ADR)

# Smoke test runs automatically
# ‚úÖ npm install: success
# ‚úÖ npm run lint: success
# ‚úÖ npm test: success (1 passing)
# ‚úÖ npm run build: success

    Expected Outcome: Production-ready repo in <5 minutes, ready for first feature implementation.

    2. Creating a UI Component Library¶

    Scenario: Need to extract reusable UI components into shared library.

    Action: Use P05 to scaffold library repo with Vite, React, and Storybook.

    Example: 

    # Create shared UI library
eva-scaffold create eva-ui-components \
    --type lib \
    --pod POD-X \
    --owner P12-UXA \
    --gc-design

# Scaffolder generates:
# - src/components/ (component source)
# - tests/components/ (component tests)
# - .storybook/ (Storybook config)
# - package.json (configured for library publishing)
# - vite.config.ts (library build mode)

    Expected Outcome: Library ready for component development and npm publishing.

    3. Bootstrapping Infrastructure Repo¶

    Scenario: Setting up infrastructure-as-code for new Azure environment.

    Action: Use P05 to scaffold infrastructure repo with Terraform and Azure pipelines.

    Example: 

    # Create infrastructure repo
eva-scaffold create eva-infra-pilot \
    --type infra \
    --pod POD-F \
    --owner DevOps

# Scaffolder generates:
# - modules/ (Terraform modules)
# - environments/ (dev, staging, prod configs)
# - .github/workflows/terraform-plan.yml (PR preview)
# - .github/workflows/terraform-apply.yml (deployment)
# - docs/runbooks/ (operational guides)

    Expected Outcome: Infrastructure repo ready for Terraform module development.

    4. Quick Prototyping¶

    Scenario: Hackathon or proof-of-concept needs fast setup.

    Action: Use P05 to scaffold minimal viable repo, skip optional features.

    Example: 

    # Minimal API scaffold for prototype
eva-scaffold create eva-prototype-api \
    --type api \
    --pod POD-F \
    --minimal  # Skip bilingual, GC Design, extensive docs

# Scaffolder generates:
# - src/index.ts (minimal server)
# - tests/ (basic test setup)
# - README.md (minimal Quick Start)
# - .github/workflows/ci.yml (basic CI only)

    Expected Outcome: Lightweight repo in <2 minutes, suitable for rapid experimentation.

    When NOT to Use P05¶

    • Don't use P05 to modify existing repos - P05 is for new repo creation only (use manual edits or P04 for updates)
    • Don't use P05 for non-EVA projects - Templates assume EVA Suite conventions and tooling
    • Don't use P05 to bypass standards - Templates enforce best practices; don't fight them
    • Don't use P05 for one-off scripts - Simple automation doesn't need full repo structure

    üé≠ Sample Prompts and Workflows¶

    System Prompt (For AI Agents)¶

    You are P05-SCA (Scaffolder Agent), the Repository Initialization Specialist for EVA Suite.

**Your Capabilities**:
1. Generate complete repository structures from templates (api, ui, lib, infra, pipeline)
2. Configure tooling (linters, formatters, test frameworks, CI/CD)
3. Create documentation scaffolds (README, ADRs, CONTRIBUTING)
4. Run smoke tests to validate generated repos build and test successfully
5. Apply customizations (bilingual docs, GC Design System, POD/owner metadata)

**Your Limitations**:
- You do NOT implement business logic - only boilerplate structure
- You do NOT modify existing repos - only create new ones
- You do NOT choose architecture - templates embody pre-approved patterns

**Autonomy Level**: C1 (Human-on-the-loop)
- Repository generation is automated
- Smoke tests run automatically
- Human reviews generated code before first commit
- Critical repos (prod-bound) require manual validation

**Output Format**:
- Repository structure tree with explanations
- Smoke test results (‚úÖ/‚ùå for build, test, lint)
- Next steps for post-scaffolding customization

**Safety Rules**:
- Never overwrite existing repos without explicit confirmation
- Always run smoke tests before declaring success
- Flag template inconsistencies for P04 (Repo Librarian) review
- Escalate if smoke tests fail (don't deliver broken scaffolds)

    üîê Security and Compliance Considerations¶

    Template Dependency Management¶

    Risk: Scaffolded repos inherit vulnerable dependencies from templates

    Mitigation: - Weekly Dependency Scans: Automated workflow runs npm audit on all templates - Automated Updates: Dependabot PRs for template dependencies - Pre-Scaffold Validation: Smoke test includes security scan before generation

    Example Workflow:

    # .github/workflows/template-security-scan.yml
name: Template Security Scan

on:
  schedule:
    - cron: '0 2 * * 1'  # Every Monday at 2 AM
  workflow_dispatch:

jobs:
  scan-templates:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        template: [api-template, ui-template, lib-template]

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Audit Dependencies
        working-directory: tools/scaffolder/templates/${{ matrix.template }}
        run: |
          npm audit --audit-level=moderate

      - name: Create Issue if Vulnerabilities Found
        if: failure()
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: '[P05] Security vulnerability in ${{ matrix.template }}',
              body: 'Automated scan detected vulnerabilities. Run `npm audit` for details.',
              labels: ['security', 'p05-scaffolder']
            })

    Secrets Management¶

    Risk: Scaffolded repos may include hardcoded secrets or credentials

    Mitigation: - No Secrets in Templates: Templates use .env.example with placeholders - Secret Scanning: GitHub secret scanning enabled for all generated repos - Best Practices Docs: CONTRIBUTING.md includes secrets management guidance

    üîß Implementation Guidance¶

    Initial Setup (One-Time)¶

    Step 1: Create P05 tooling directory and templates

    # In eva-orchestrator repo
New-Item -Path "tools/scaffolder" -ItemType Directory
New-Item -Path "tools/scaffolder/templates" -ItemType Directory

# Create template subdirectories
@('api-template', 'ui-template', 'lib-template', 'infra-template') | ForEach-Object {
    New-Item -Path "tools/scaffolder/templates/$_" -ItemType Directory
}

    Step 2: Populate templates with reference implementations

    # Example: Copy existing eva-api as reference for api-template
Copy-Item "../eva-api/*" "tools/scaffolder/templates/api-template/" -Recurse

# Parameterize template variables
# Replace hardcoded values with {{repo-name}}, {{pod}}, {{owner}} placeholders

    Step 3: Create scaffold CLI script

    # tools/scaffolder/scaffold.ps1
param(
    [Parameter(Mandatory=$true)]
    [string]$RepoName,

    [Parameter(Mandatory=$true)]
    [ValidateSet('api', 'ui', 'lib', 'infra', 'pipeline')]
    [string]$Type,

    [string]$Pod = "POD-F",
    [string]$Owner = "Team",
    [switch]$Bilingual,
    [switch]$GcDesign
)

Write-Host "üèóÔ∏è Scaffolding new $Type repository: $RepoName" -ForegroundColor Cyan

# Copy template
$templatePath = "tools/scaffolder/templates/$Type-template"
$targetPath = "../$RepoName"

Copy-Item $templatePath $targetPath -Recurse

# Replace template variables
Get-ChildItem $targetPath -Recurse -File | ForEach-Object {
    $content = Get-Content $_.FullName -Raw
    $content = $content -replace '\{\{repo-name\}\}', $RepoName
    $content = $content -replace '\{\{pod\}\}', $Pod
    $content = $content -replace '\{\{owner\}\}', $Owner
    Set-Content $_.FullName $content
}

# Add bilingual docs if requested
if ($Bilingual) {
    Copy-Item "$targetPath/README.md" "$targetPath/README.fr.md"
    Write-Host "‚úÖ Created README.fr.md (bilingual)" -ForegroundColor Green
}

# Run smoke test
Write-Host "`nüß™ Running smoke tests..." -ForegroundColor Cyan
& "tools/scaffolder/smoke-test.ps1" -RepoPath $targetPath

Write-Host "`n‚úÖ Scaffold complete! Repository created at: $targetPath" -ForegroundColor Green
