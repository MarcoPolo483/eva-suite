    Chapter P06: Review Agent¶

    Pattern Code: P06 Short Code: REV Name: Review Agent Type: DevTools Pattern Scope: Code Review & Quality Assurance Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Review Agent (P06-REV) serves as the first line of defense in code quality assurance, automating PR review checklist validation and providing intelligent first-pass analysis before human review. In modern software teams, code review is a critical quality gate‚Äîbut it's also a bottleneck. Human reviewers spend significant time on mechanical checks (lint errors, test failures, coverage gaps) that machines can detect instantly.

    P06-REV addresses this by automating the "easy but tedious" parts of code review, freeing human reviewers to focus on architectural decisions, business logic correctness, and design patterns. Think of it as a tireless junior reviewer who never sleeps, never gets fatigued, and always applies the same standards consistently.

    The Problem P06 Solves: - Reviewer Cognitive Load: Humans waste time on mechanical issues (lint errors, missing tests) - Inconsistent Standards: Review quality varies by reviewer fatigue, experience, bias - Slow Feedback Loops: Developers wait hours/days for basic "did it build?" feedback - Missed Issues: Human reviewers overlook subtle bugs or style violations - Review Bottlenecks: Limited reviewer bandwidth delays merges

    The Value P06 Delivers: - Fast Feedback: Automated checks complete in <5 minutes (vs hours for human review) - Consistent Standards: Same quality bar applied to every PR, every time - Reduced Reviewer Burden: Human reviewers focus on meaningful feedback, not mechanical issues - Higher Quality: Automated detection of coverage gaps, security vulnerabilities, broken tests - Accelerated Velocity: PRs merge faster when mechanical issues caught early

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P06-REV serves as the Automated PR Review Agent for the EVA Autonomous Agile Command. It ensures:

    1. Automated Quality Gates: Lint, tests, coverage, security scans run on every PR
    2. First-Pass Analysis: Intelligent parsing of failures with actionable suggestions
    3. Review Comment Generation: Auto-posted comments highlighting issues and fixes
    4. Human-Ready Context: Failures explained in developer-friendly language
    5. Continuous Quality: Every PR meets minimum standards before human review

    Mission Statement¶

    "Accelerate PR review cycles and improve code quality by automating mechanical checks, explaining failures intelligently, and freeing human reviewers to focus on high-value architectural and business logic feedback."

    Value Proposition¶

    For Product Owners: - Faster feature delivery (PRs merge sooner with automated pre-checks) - Predictable quality (every PR meets same standards) - Reduced review costs (less human reviewer time per PR)

    For Developers: - Immediate feedback (no waiting for human reviewer on lint errors) - Clear failure explanations (not just "tests failed" but "why and how to fix") - Self-service quality checks (run same checks locally before pushing)

    For Reviewers: - Reduced cognitive load (focus on logic, not formatting) - Consistent baseline (all PRs pre-screened for basic quality) - Better context (automated checks surface issues humans might miss)

    üìã Responsibilities¶

    Core Responsibilities¶

    1. PR Review Checklist Validation
    2. Lint passes (ESLint for JS/TS, Pylint/Ruff for Python)
    3. Type checking passes (TypeScript, mypy)
    4. Formatting consistent (Prettier, Black)

    5. Security vulnerabilities absent (npm audit, bandit, Snyk)

    6. Test Quality Enforcement

    7. All tests pass (unit, integration, e2e)
    8. Coverage ‚â•80% (configurable threshold)
    9. No flaky tests (identify tests with inconsistent results)

    10. E2E tests green (if applicable for UI/API changes)

    11. Documentation Validation

    12. README updated if public API changes
    13. API docs generated (OpenAPI/Swagger for APIs)
    14. ADR created for architectural changes

    15. CHANGELOG updated for user-facing changes

    16. Dependency Auditing

    17. No new vulnerable dependencies (security scan)
    18. Lock files updated (package-lock.json, poetry.lock)
    19. Dependency versions pinned (no wildcards in production)

    Supporting Responsibilities¶

    1. Failure Explanation
    2. Parse test failures and extract root cause
    3. Generate suggested fixes for common errors

    4. Link to relevant documentation for complex issues

    5. Review Comment Automation

    6. Post GitHub PR comments summarizing checks
    7. Highlight failures with severity levels

    8. Provide code snippets showing before/after for fixes

    9. Integration Points

    10. Coordinate with P07 (Testing & Failure Explainer) for detailed test analysis
    11. Provide data to P10 (Metrics & Insights) for review cycle metrics
    12. Support P08 (CI/CD Guardian) for pipeline integration

    üöÄ When to Use P06¶

    Primary Use Cases¶

    1. Every Pull Request (Automated)¶

    Scenario: Developer opens PR ‚Üí P06 runs automatically via GitHub Actions.

    Action: Execute full PR review checklist and post results as comment.

    Example: 

    # .github/workflows/pr-review.yml
name: PR Review Automation

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  automated-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run Lint
        run: npm run lint
        continue-on-error: true
        id: lint

      - name: Run Tests
        run: npm test
        continue-on-error: true
        id: tests

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const lintPassed = '${{ steps.lint.outcome }}' === 'success';
            const testsPassed = '${{ steps.tests.outcome }}' === 'success';

            const body = `## ü§ñ Automated Review

            | Check | Status |
            |-------|--------|
            | Lint | ${lintPassed ? '‚úÖ' : '‚ùå'} |
            | Tests | ${testsPassed ? '‚úÖ' : '‚ùå'} |

            ${!lintPassed || !testsPassed ?
              '‚ö†Ô∏è Please fix failing checks before requesting human review.' :
              '‚úÖ All automated checks passed! Ready for human review.'}
            `;

            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: body
            });

    Expected Outcome: PR comment posted within 5 minutes summarizing check results.

    2. Pre-Commit Local Validation¶

    Scenario: Developer wants to run same checks locally before pushing.

    Action: Run PR review checks via CLI script.

    Example: 

    # tools/review/local-check.sh
#!/bin/bash

echo "üîç Running local PR review checks..."

# Lint
echo "\nüìã Linting..."
npm run lint
if [ $? -ne 0 ]; then
  echo "‚ùå Lint failed"
  exit 1
fi

# Tests
echo "\nüß™ Testing..."
npm test
if [ $? -ne 0 ]; then
  echo "‚ùå Tests failed"
  exit 1
fi

# Coverage
echo "\nüìä Coverage..."
npm run test:coverage
coverage=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
if [ $(echo "$coverage < 80" | bc) -eq 1 ]; then
  echo "‚ùå Coverage below 80%: $coverage%"
  exit 1
fi

echo "\n‚úÖ All checks passed! Safe to push."

    Expected Outcome: Developer knows PR will pass automated checks before pushing.

    3. Review Comment Analysis (Manual Trigger)¶

    Scenario: Human reviewer requests detailed failure analysis for complex test failure.

    Action: Trigger P06 failure explanation for specific test.

    Example: 

    # Analyze specific test failure
tools/review/explain-failure.js \
  --test "tests/api/users.test.ts" \
  --error "TypeError: Cannot read property 'name' of undefined"

# Output:

**Test**: tests/api/users.test.ts - should return user profile
**Error**: TypeError: Cannot read property 'name' of undefined

### Likely Cause
Missing null check or async data not loaded before accessing user.name property.

### Suggested Fix
```typescript
// Before (failing):
export function getUserProfile(userId: string) {
  const user = db.users.find(userId);
  return { name: user.name }; // ‚ùå user might be undefined
}

// After (fixed):
export function getUserProfile(userId: string) {
  const user = db.users.find(userId);
  if (!user) throw new Error(`User not found: ${userId}`);
  return { name: user.name }; // ‚úÖ user guaranteed to exist
}

    **Expected Outcome**: Detailed explanation with code fix suggestion.

#### 4. Security Scan Enforcement
**Scenario**: Ensuring no vulnerable dependencies introduced in PR.

**Action**: Run security audit and block PR if high/critical vulnerabilities found.

**Example**:
```yaml
# .github/workflows/security-scan.yml
- name: Security Scan
  run: npm audit --audit-level=high

- name: Block PR if Vulnerabilities
  if: failure()
  run: |
    echo "‚ùå Security vulnerabilities detected. Update dependencies before merging."
    exit 1

    Expected Outcome: PR blocked until vulnerabilities patched.

    When NOT to Use P06¶

    • Don't use P06 for architectural review - P06 checks mechanics, not design patterns (human reviewers assess architecture)
    • Don't use P06 to bypass testing - P06 validates tests pass, not that tests are correct
    • Don't use P06 for content review - P06 checks structure, not prose quality (use P12 for UX/copy)
    • Don't use P06 to replace human judgment - P06 accelerates review, doesn't eliminate need for human insight

    üß™ Failure Detection and Explanation¶

    Test Failure Parsing¶

    Script: tools/review/parse-failures.ts

    interface TestFailure {
  file: string;
  test: string;
  error: string;
  stackTrace: string;
  suggestion: string;
}

export function parseTestFailures(log: string): TestFailure[] {
  const failures: TestFailure[] = [];

  // Parse Vitest/Jest output
  const failureBlocks = log.match(/FAIL\s+(.+?)\n([\s\S]+?)(?=FAIL|$)/g);

  failureBlocks?.forEach(block => {
    const fileMatch = block.match(/FAIL\s+(.+)/);
    const testMatch = block.match(/‚úï\s+(.+)/);
    const errorMatch = block.match(/Error:\s+(.+)/);
    const stackMatch = block.match(/(at\s+.+\n?)+/);

    const file = fileMatch?.[1] || 'unknown';
    const test = testMatch?.[1] || 'unknown';
    const error = errorMatch?.[1] || 'unknown';
    const stackTrace = stackMatch?.[0] || '';

    failures.push({
      file,
      test,
      error,
      stackTrace,
      suggestion: generateSuggestion(error)
    });
  });

  return failures;
}

function generateSuggestion(error: string): string {
  // Pattern matching for common errors

  if (error.includes('Cannot read property') || error.includes('undefined')) {
    return 'Add optional chaining (?.) or null guard (if/else check)';
  }

  if (error.includes('timeout')) {
    return 'Increase test timeout or optimize code to complete faster';
  }

  if (error.includes('404') || error.includes('ECONNREFUSED')) {
    return 'Mock external dependencies or ensure test environment is configured';
  }

  if (error.includes('Expected') && error.includes('Received')) {
    return 'Review assertion - expected value may be incorrect';
  }

  if (error.includes('Async callback was not invoked')) {
    return 'Add await or return promise from test';
  }

  return 'Review error message and stack trace for root cause';
}

    Lint Error Extraction¶

    Script: tools/review/parse-lint-errors.ts

    interface LintError {
  file: string;
  line: number;
  column: number;
  rule: string;
  severity: 'error' | 'warning';
  message: string;
  suggestedFix?: string;
}

export function parseLintErrors(eslintOutput: string): LintError[] {
  const errors: LintError[] = [];

  // Parse ESLint JSON output
  const results = JSON.parse(eslintOutput);

  results.forEach(result => {
    result.messages.forEach(message => {
      errors.push({
        file: result.filePath,
        line: message.line,
        column: message.column,
        rule: message.ruleId,
        severity: message.severity === 2 ? 'error' : 'warning',
        message: message.message,
        suggestedFix: message.fix ? 'Auto-fixable with --fix' : undefined
      });
    });
  });

  return errors;
}

export function formatLintErrors(errors: LintError[]): string {
  const grouped = errors.reduce((acc, err) => {
    if (!acc[err.file]) acc[err.file] = [];
    acc[err.file].push(err);
    return acc;
  }, {} as Record<string, LintError[]>);

  let output = '## üîç Lint Errors\n\n';

  Object.entries(grouped).forEach(([file, fileErrors]) => {
    output += `### ${file}\n\n`;

    fileErrors.forEach(err => {
      const icon = err.severity === 'error' ? '‚ùå' : '‚ö†Ô∏è';
      output += `**Line ${err.line}:${err.column}** ${icon} ${err.message}\n`;
      output += `- Rule: \`${err.rule}\`\n`;
      if (err.suggestedFix) {
        output += `- Fix: ${err.suggestedFix}\n`;
      }
      output += '\n';
    });
  });

  return output;
}

    ‚úÖ Definition of Done¶

    For Automated Review Execution¶

    • All checklist items executed (lint, tests, coverage, security, build)
    • PR comment posted summarizing check results
    • Failures include actionable suggestions (not just "failed")
    • Check results visible in PR status (GitHub Checks API)
    • Execution time <5 minutes (performance SLA)
    • No false positives (manual audit of random sample)

    For Review Comment Quality¶

    • Comment includes table with ‚úÖ/‚ùå status per check
    • Failing checks have "How to Fix" instructions
    • Coverage percentage displayed (if applicable)
    • Links to full logs for complex failures
    • Tone is helpful, not accusatory ("Please fix" not "You broke")
    • Comment auto-updates on subsequent pushes (replace, not duplicate)

    For Quality Gate Enforcement¶

    • PR cannot merge if lint fails
    • PR cannot merge if tests fail
    • PR cannot merge if coverage <80%
    • PR cannot merge if high/critical security vulnerabilities
    • PR cannot merge if build fails
    • Override mechanism exists for Tech Lead exceptions (documented)

    üö® Common Issues and Troubleshooting¶

    Issue 1: Flaky Test Detected¶

    Symptom: Test passes locally but fails in CI intermittently

    Diagnosis: Run test 10 times, check for >1 failure

    # Detect flaky test
for i in {1..10}; do
  npm test -- tests/api/users.test.ts
done

# If >1 failure: test is flaky

    Solution: 1. Stabilize test (fix race condition, add proper waits) 2. If urgent, mark test as skip temporarily 3. Create issue to fix flaky test

    Prevention: Add test:retry script to catch flaky tests before merge

    Issue 2: Coverage Drop on Valid Refactor¶

    Symptom: Coverage drops from 85% to 75% after refactoring, even though logic unchanged

    Cause: Lines moved to different files not covered by existing tests

    Solution: 

    # Identify uncovered lines
npm run test:coverage -- --reporter=html
open coverage/lcov-report/index.html

# Add tests for newly uncovered lines

    Prevention: Review coverage diff in PR, explain intentional coverage drops in PR description

    Issue 3: Security Vulnerability in Transitive Dependency¶

    Symptom: npm audit fails due to vulnerability in dependency's dependency

    Cause: Direct dependency hasn't updated vulnerable sub-dependency

    Solution: 

    # Option 1: Update direct dependency
npm update <dependency>

# Option 2: Override vulnerable sub-dependency
# package.json
{
  "overrides": {
    "vulnerable-package": "^1.2.3"  # Safe version
  }
}

# Option 3: Accept risk (document in PR)
# Only if vulnerability doesn't apply to our use case

    Prevention: Weekly dependency updates via Dependabot

    üéì Key Takeaways¶

    1. Automate the easy, focus humans on the hard - Machines check lint, humans review architecture
    2. Fast feedback is critical - 5-minute automated checks >> 4-hour human review wait
    3. Consistency beats perfection - Same standards every PR > varying quality based on reviewer
    4. Suggest, don't dictate - P06 recommends fixes, humans decide implementation
    5. Quality gates protect quality - Blocking <80% coverage prevents technical debt accumulation
    6. Failures need explanations - "Tests failed" is useless; "Null reference in users.ts:45" is actionable
    7. Review cycle time compounds - Faster PRs = faster features = faster value delivery
