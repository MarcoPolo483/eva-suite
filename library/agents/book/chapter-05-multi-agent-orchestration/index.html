    Chapter 05: Multi-Agent Orchestration¶

    Focus: P15 Orchestration Workflows in Practice Audience: Technical Leads, DevOps Engineers, SREs Prerequisites: Chapters P01-P15, GC Deployment Patterns

    üìñ Chapter Overview¶

    Multi-agent orchestration is where the EVA DevTools ecosystem transitions from individual patterns to coordinated workflows that deliver end-to-end value. While P01-P14 provide discrete capabilities (requirements, scaffolding, testing, security, etc.), P15 (Dev Master Orchestrator) composes them into production-grade pipelines.

    This chapter provides real, battle-tested orchestration workflows used in EVA 2.0 development, with actual code samples, execution traces, and GC-specific adaptations (Protected B, bilingual UI, WCAG 2.1 AA compliance).

    What You'll Learn: - Real P15 workflow definitions (new-feature, sprint-retro, production-deploy) - Cross-repo coordination patterns (eva-foundation ‚Üí eva-ui ‚Üí eva-api) - Dependency resolution and impact analysis - Error handling, rollback, and observability - GC compliance checkpoints (security, accessibility, i18n)

    üéØ The Orchestration Challenge¶

    Why Manual Multi-Pattern Invocation Fails¶

    Scenario: You need to add bilingual Protected B data ingestion to EVA Foundation.

    Manual Process (error-prone, 3-4 hours): 1. Write requirements in Word doc ‚Üí email to team 2. Manually create epic in Azure DevOps ‚Üí forget to link design doc 3. Scaffold repository structure ‚Üí inconsistent folder naming 4. Write tests ‚Üí miss edge cases for French locale 5. Security scan ‚Üí forget to check for credential leaks 6. Accessibility review ‚Üí discover violations late (rework needed) 7. Create PR ‚Üí reviewers ask for missing test coverage

    Pain Points: - Context loss: Requirements doc disconnected from code - Forgotten steps: Security scan only run when reminded - Inconsistency: Each developer follows different order - Rework: Accessibility issues found after code complete - Slow feedback: Wait hours/days between steps

    The P15 Orchestration Solution¶

    Automated Workflow (1 workflow invocation, 30 min): 

    # Single command orchestrates all 7 steps
tools/jarvis/orchestrator.ps1 -WorkflowName new-feature -Context @{
    brief_path = "docs/features/bilingual-data-ingestion.md"
    project = "eva-foundation"
    template = "azure-service"
    compliance = @("protected-b", "wcag-aa", "bilingual")
}

    What Happens (automated, consistent, audited): 1. P02: Extract requirements from brief ‚Üí generate CDD 2. P03: Create epic + stories in Azure DevOps with traceability 3. P05: Scaffold boilerplate (Azure Functions + i18n structure) 4. P07: Generate unit tests (including bilingual edge cases) 5. P11: Run security scan (credential check, dependency vulnerabilities) 6. P12: Validate accessibility (WCAG checklist, i18n completeness) 7. P06: Create PR with all findings, test results, compliance evidence

    Benefits: - Speed: 30 min vs 3-4 hours - Consistency: Every feature follows same process - Quality: No forgotten steps (security, accessibility always run) - Traceability: Full audit trail (P21 provenance) - Early feedback: Issues found in minutes, not days

    üöÄ Workflow 1: New Feature Development¶

    Workflow Definition¶

    File: tools/jarvis/workflows/new-feature.yaml

    workflow:
  name: new-feature
  description: "End-to-end feature development from brief to PR"
  autonomy: C1  # Human can pause/stop at any time
  timeout: 3600  # 1 hour max

  parameters:
    brief_path:
      type: string
      required: true
      description: "Path to feature brief (markdown)"
    project:
      type: string
      required: true
      enum: ["eva-foundation", "eva-ui", "eva-api", "eva-rag"]
    template:
      type: string
      required: false
      default: "azure-service"
      enum: ["azure-service", "react-component", "api-endpoint", "rag-pipeline"]
    compliance:
      type: array
      required: false
      default: ["protected-b", "wcag-aa", "bilingual"]
      items:
        enum: ["protected-b", "wcag-aa", "bilingual", "finops"]

  steps:
    - id: requirements
      agent: P02-REQ
      description: "Extract structured requirements from brief"
      inputs:
        brief_path: "{{context.brief_path}}"
        output_format: "cdd"  # Context-Driven Development Document
      outputs:
        cdd_path: "{{workspace}}/docs/requirements/{{feature_id}}-cdd.md"
        epic_summary: "{{cdd.epic.summary}}"
      error_handling: fail  # Cannot proceed without requirements

    - id: backlog
      agent: P03-SCR
      description: "Create epic and stories in Azure DevOps"
      inputs:
        cdd_path: "{{steps.requirements.outputs.cdd_path}}"
        project: "{{context.project}}"
        sprint: "{{current_sprint}}"
      outputs:
        epic_id: "{{azure_devops.epic.id}}"
        story_ids: "{{azure_devops.stories[*].id}}"
      error_handling: continue  # Can manually create epic if this fails

    - id: scaffold
      agent: P05-SCA
      description: "Generate boilerplate code from template"
      inputs:
        template: "{{context.template}}"
        feature_name: "{{steps.requirements.outputs.cdd.feature_name}}"
        target_repo: "{{context.project}}"
        compliance: "{{context.compliance}}"
      outputs:
        scaffolded_files: "{{files.created}}"
        entry_point: "{{files.entry_point}}"
      error_handling: fail  # Cannot proceed without code structure

    - id: testing
      agent: P07-TST
      description: "Generate unit tests with edge cases"
      inputs:
        source_files: "{{steps.scaffold.outputs.scaffolded_files}}"
        compliance: "{{context.compliance}}"
        coverage_target: 80
      outputs:
        test_files: "{{tests.created}}"
        coverage_estimate: "{{tests.coverage_percentage}}"
      error_handling: continue  # Can manually write tests if generation fails

    - id: security_scan
      agent: P11-SEC
      description: "Run security and compliance checks"
      inputs:
        changed_files: "{{steps.scaffold.outputs.scaffolded_files}}"
        compliance: "{{context.compliance}}"
      outputs:
        security_findings: "{{scan.findings}}"
        compliance_status: "{{scan.compliance_status}}"
      error_handling: fail  # Block PR if critical security issues found

    - id: accessibility_check
      agent: P12-UXA
      description: "Validate WCAG 2.1 AA and bilingual requirements"
      condition: "'wcag-aa' in context.compliance or 'bilingual' in context.compliance"
      inputs:
        ui_files: "{{steps.scaffold.outputs.scaffolded_files | filter(*.tsx, *.vue)}}"
        i18n_files: "{{steps.scaffold.outputs.scaffolded_files | filter(**/locales/**)}}"
      outputs:
        wcag_findings: "{{accessibility.wcag_violations}}"
        i18n_completeness: "{{accessibility.i18n_coverage_percentage}}"
      error_handling: continue  # Flag issues but don't block PR

    - id: code_review
      agent: P06-REV
      description: "Perform automated code review and create PR"
      inputs:
        changed_files: "{{steps.scaffold.outputs.scaffolded_files}}"
        test_files: "{{steps.testing.outputs.test_files}}"
        security_findings: "{{steps.security_scan.outputs.security_findings}}"
        accessibility_findings: "{{steps.accessibility_check.outputs.wcag_findings}}"
        epic_id: "{{steps.backlog.outputs.epic_id}}"
      outputs:
        pr_url: "{{github.pull_request.url}}"
        review_summary: "{{review.summary}}"
      error_handling: continue  # Manually create PR if automation fails

  notifications:
    on_start:
      - type: slack
        channel: "#eva-dev"
        message: "üöÄ New feature workflow started: {{context.brief_path}}"
    on_complete:
      - type: slack
        channel: "#eva-dev"
        message: "‚úÖ Feature ready for review: {{steps.code_review.outputs.pr_url}}"
    on_error:
      - type: slack
        channel: "#eva-alerts"
        message: "‚ùå Feature workflow failed at step {{failed_step}}: {{error_message}}"

    ### Cross-Repo Coordination Example

**Scenario**: Feature requires changes in **3 repositories** (eva-foundation + eva-ui + eva-api).

**Challenge**: Ensure changes deploy in correct order (foundation first, then API, then UI).

**Solution**: Multi-repo workflow with dependency resolution.

**Workflow Definition**: `tools/jarvis/workflows/new-feature-multi-repo.yaml`

```yaml
workflow:
  name: new-feature-multi-repo
  description: "Feature spanning multiple repositories"

  parameters:
    brief_path: string
    affected_repos:
      type: array
      items:
        type: object
        properties:
          repo: string  # eva-foundation, eva-ui, eva-api
          changes: string  # Description of changes
          depends_on: array  # Other repos this depends on

  steps:
    - id: dependency_graph
      agent: P15-DVM
      description: "Build deployment dependency graph"
      inputs:
        affected_repos: "{{context.affected_repos}}"
      outputs:
        deployment_order: ["eva-foundation", "eva-api", "eva-ui"]

    - id: feature_foundation
      agent: P15-DVM
      description: "Run new-feature workflow in eva-foundation"
      inputs:
        workflow: new-feature
        context:
          brief_path: "{{context.brief_path}}"
          project: "eva-foundation"
      outputs:
        pr_url: "{{steps.code_review.outputs.pr_url}}"

    - id: feature_api
      agent: P15-DVM
      description: "Run new-feature workflow in eva-api"
      condition: "steps.feature_foundation.status == 'complete'"
      inputs:
        workflow: new-feature
        context:
          brief_path: "{{context.brief_path}}"
          project: "eva-api"
          dependencies:
            - repo: eva-foundation
              pr: "{{steps.feature_foundation.outputs.pr_url}}"

    - id: feature_ui
      agent: P15-DVM
      description: "Run new-feature workflow in eva-ui"
      condition: "steps.feature_api.status == 'complete'"
      inputs:
        workflow: new-feature
        context:
          brief_path: "{{context.brief_path}}"
          project: "eva-ui"
          dependencies:
            - repo: eva-foundation
              pr: "{{steps.feature_foundation.outputs.pr_url}}"
            - repo: eva-api
              pr: "{{steps.feature_api.outputs.pr_url}}"

    Execution: 

    tools/jarvis/orchestrator.ps1 -WorkflowName new-feature-multi-repo -Context @{
    brief_path = "docs/features/user-profile-bilingual.md"
    affected_repos = @(
        @{ repo = "eva-foundation"; changes = "User profile data model + Cosmos DB schema" },
        @{ repo = "eva-api"; changes = "User profile CRUD endpoints"; depends_on = @("eva-foundation") },
        @{ repo = "eva-ui"; changes = "User profile UI components"; depends_on = @("eva-foundation", "eva-api") }
    )
}

    üîÑ Workflow 2: Sprint Retrospective¶

    Purpose¶

    After each sprint (weekly cadence), gather insights for retrospective meeting: - Delivery metrics (velocity, burndown) - Incidents and alerts (LiveOps) - Security findings (vulnerabilities, compliance) - Runbook updates (new procedures discovered)

    Workflow Definition¶

    File: tools/jarvis/workflows/sprint-retro.yaml

    workflow:
  name: sprint-retro
  description: "Automated sprint retrospective data collection"
  autonomy: C0  # Human reviews all outputs

  parameters:
    sprint_id:
      type: string
      required: true
      pattern: "^sprint-\\d{3}$"  # e.g., sprint-007

  steps:
    - id: metrics
      agent: P10-MET
      description: "Collect sprint delivery metrics"
      inputs:
        sprint_id: "{{context.sprint_id}}"
        metrics:
          - velocity
          - burndown
          - cycle_time
          - lead_time
          - defect_rate
      outputs:
        velocity: "{{metrics.velocity}}"
        burndown_chart: "{{metrics.burndown_chart_url}}"
        cycle_time_avg: "{{metrics.cycle_time.average}}"

    - id: incidents
      agent: P14-LVO
      description: "Analyze incidents and alerts during sprint"
      inputs:
        sprint_id: "{{context.sprint_id}}"

    üéì Best Practices¶

    1. Design Workflows for Failure¶

    Anti-Pattern: Assume all steps will succeed.

    Best Practice: Define error_handling for every step (fail, continue, rollback).

    Example: 

    steps:
  - id: security_scan
    error_handling: fail  # Critical - block deployment
  - id: accessibility_check
    error_handling: continue  # Non-critical - flag issues but proceed
  - id: deploy
    error_handling: rollback  # Revert on failure

    2. Use Approval Gates for High-Risk Workflows¶

    Anti-Pattern: Fully automate production deployments without human oversight.

    Best Practice: Require approval from Product Owner + Security Lead for production changes.

    Example: 

    approval_gates:
  - stage: pre_deployment
    approvers: ["product-owner", "security-lead"]
    timeout: 24h

    3. Maintain Full Audit Trail¶

    Anti-Pattern: No record of who approved what.

    Best Practice: Use P21 Provenance to record all workflow executions, approvals, evidence.

    Example: 

    steps:
  - id: provenance_record
    agent: P21-PRV
    inputs:
      approvers: "{{approval_gate_1.approvers}}"
      evidence: "{{all_step_outputs}}"

    4. Gradual Rollout for Production Changes¶

    Anti-Pattern: Shift 100% traffic immediately (big-bang deployment).

    Best Practice: Gradual rollout (10% ‚Üí 25% ‚Üí 50% ‚Üí 100%) with monitoring.

    Example: 

    steps:
  - id: traffic_shift
    inputs:
      rollout_strategy:
        - percent: 10
          duration: 15m
        - percent: 100
          duration: permanent
