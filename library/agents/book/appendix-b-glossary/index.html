    Appendix B: Glossary¶

    EVA Agentic Book Terms & Definitions

    Comprehensive glossary of EVA Suite terminology, Government of Canada terms, pattern codes, and technical concepts.

    EVA Core Terms¶

    Agent¶

    A software entity with autonomous capabilities operating within defined autonomy levels (C0-C3). Agents can be service personas (SPxx) facing users or devtools patterns (Pxx) supporting development.

    Agile Command¶

    The leadership and coordination layer of EVA development, consisting of Product Owner (Marco), Scrum Master (GitHub Copilot), and Senior Advisor (ChatGPT).

    Agile Crew¶

    The execution layer of autonomous agents coordinating development work using patterns P01-P21. Includes Dev Masters (ai2, windowsGuy) and specialized agents.

    Awareness Block¶

    Structured metadata indicating AI-generated content, autonomy level, provenance, and human review status. Required for all AI-generated outputs.

    Awareness-First Design¶

    Design principle requiring all AI interactions to surface autonomy level, confidence, provenance, and human oversight status to users.

    CDD (Concept Definition Document)¶

    Foundational product document describing feature vision, value hypothesis, success criteria, user stories, and constraints. Input to P02-REQ pattern.

    DUA (Developer-User Agreement)¶

    Standardized archive format (.dua.zip) containing conversation history, metadata, provenance, and artifacts. Ensures portability and auditability.

    DVM (Dev Master)¶

    Development agent coordinating code implementation, reviews, and technical decisions. Examples: ai2 (Linux/Python), windowsGuy (Windows/.NET).

    Epic¶

    Large body of work decomposed into 3-8 user stories. Typically represents a CDD or major feature. Tracked as GitHub issues with epic label.

    EVA 2.0¶

    Enterprise Virtual Assistant platform version 2.0. Cloud-agnostic AI assistant suite with 40+ products built for Government of Canada deployment.

    EVA Foundation¶

    Cloud services layer providing Azure OpenAI, Cosmos DB, Azure AI Search, and other infrastructure. Vendor-agnostic design for future multi-cloud support.

    EVA Matrix¶

    Public documentation hub (MkDocs/Jekyll site) containing architecture narratives, patterns, and EVA Suite stories.

    EVA Meta¶

    Governance registry repository containing agent definitions, ATO controls, compliance documentation, and canonical registries.

    EVA Sovereign UI¶

    UI component library implementing GC Design System, WCAG 2.1 AA compliance, and bilingual support (EN-CA/FR-CA). Foundation for all EVA Suite products.

    EVA Suite¶

    Collection of 40+ AI-powered products built on EVA Foundation. Includes EVA DA 2.0, EVA Chat 2.0, domain-specific assistants, and admin tools.

    FinOps¶

    Financial operations practice tracking cloud costs, attributing expenses to products/teams, forecasting budgets, and optimizing spend.

    Maturity Level (L0-L3)¶

    Agent capability classification: L0 (Task Agent), L1 (Multi-Step Agent), L2 (Workflow Agent), L3 (Strategic Agent). See Autonomy Levels section.

    Orchestrator (eva-orchestrator)¶

    SDLC brain and control plane for EVA Suite. Manages workflows, sprint automation, multi-repo coordination, and Agile Crew patterns.

    Pattern¶

    Reusable automation workflow (P01-P21) or service persona (SP00-SPxx). Patterns have defined triggers, workflows, quality gates, and observability.

    Persona (Service Persona)¶

    Live agent serving end-users (e.g., EVA Chat Assistant, EVA Data Ingestion). Defined by SPxx codes. Subject to governance (PIA, AIIA, ATO).

    POD (Product Operational Domain)¶

    Organizational unit owning specific repositories and capabilities. Examples: POD-F (Foundation), POD-X (UI/UX), POD-O (Operations).

    Provenance Block¶

    Metadata documenting content origin, generation process, human review, data sources, and modification history. Embedded in code, docs, and API responses.

    QA Gate¶

    Quality checkpoint requiring passing criteria before work advances (e.g., Ready In Progress QA Gate Done). Enforced by QA Master.

    QA Master¶

    Role ensuring quality standards, running quality gates, blocking non-compliant releases. Typically GitHub Copilot in Scrum Master capacity.

    Scrum Master¶

    Role facilitating sprint ceremonies, managing backlog, coordinating agents, removing blockers. GitHub Copilot in EVA Agile Command.

    Sprint¶

    Time-boxed development cycle (typically 1 week). Contains planning, stand-ups, QA gates, review, and retrospective ceremonies.

    Story (User Story)¶

    Single unit of deliverable work written as "As a [persona], I want [feature] so that [value]." Contains acceptance criteria and estimates.

    Tier¶

    Deployment environment: Sandbox (experimental), Pilot (limited users), Production (full scale). Each tier has different quality/compliance requirements.

    Government of Canada Terms¶

    AIIA (Algorithmic Impact Assessment)¶

    Government of Canada assessment evaluating AI system risks, impacts, and mitigation strategies. Required for AI systems affecting citizens.

    ATO (Authority to Operate)¶

    Formal authorization allowing a system to operate in production. Based on security controls, compliance verification, and risk acceptance.

    Canada.ca Theme¶

    Official GC Design System theme providing consistent branding, typography, colors, and components for Government of Canada websites.

    ESDC (Employment and Social Development Canada)¶

    Federal department delivering social programs (EI, CPP, OAS). Primary stakeholder for EVA deployment.

    GC Design System¶

    Official design system for Government of Canada digital services. Includes components, patterns, styles, and accessibility standards.

    GCNet¶

    Government of Canada secure network connecting federal departments. All production traffic routed via GCNet.

    IRCC (Immigration, Refugees and Citizenship Canada)¶

    Federal department managing immigration and citizenship programs. Potential EVA Suite stakeholder.

    Official Languages Act¶

    Canadian legislation requiring federal services in English and French. EVA Suite must support EN-CA and FR-CA.

    PIA (Privacy Impact Assessment)¶

    Government of Canada assessment evaluating privacy risks of systems handling personal information. Required for Protected B data.

    PoLP (Principle of Least Privilege)¶

    Security principle granting minimum access rights necessary for users/services to perform functions. Enforced via RBAC.

    Protected B¶

    Canadian government information classification for sensitive data (SSN, medical records, financial data). Requires encryption, audit logs, data residency controls.

    RBAC (Role-Based Access Control)¶

    Access control model assigning permissions based on user roles (Admin, Editor, Viewer, etc.). Enforced at API/DB/Search layers.

    SCED (Secure Channel to External Destinations)¶

    SSC-managed network route for accessing external services (Azure, AWS) from GCNet. All EVA production traffic uses SCED.

    SSC (Shared Services Canada)¶

    Federal organization providing IT infrastructure, security, and telecommunications for Government of Canada.

    TBS (Treasury Board Secretariat)¶

    Central agency setting policies, standards, and directives for federal government operations, including IT security and accessibility.

    Unclassified¶

    Canadian government information classification for non-sensitive data. Public-facing content, general documentation.

    WCAG 2.1 AA¶

    Web Content Accessibility Guidelines Level AA. Mandatory standard for Government of Canada websites. Covers keyboard navigation, screen readers, contrast, captions.

    Pattern Codes (P01-P21)¶

    P01: Agent Patterns (Reserved)¶

    Meta-pattern documenting all patterns. Pattern registry and taxonomy.

    P02: REQ (Requirements Refinement)¶

    Decomposes CDDs into epics and user stories. Writes acceptance criteria, estimates, and prioritizes backlog.

    P03: SCR (Scrum Support)¶

    Facilitates sprint ceremonies (planning, stand-ups, reviews, retros). Maintains sprint artifacts and burn charts.

    P04: LIB (Librarian)¶

    Maintains repository documentation, READMEs, ADRs, and consistency across codebases.

    P05: SCA (Scaffolding)¶

    Bootstraps new repositories, modules, and services with templates, CI/CD pipelines, and configuration.

    P06: REV (Code Review)¶

    First-pass PR analysis checking code quality, security, testing, and documentation before human review.

    P07: TST (Testing)¶

    Generates test plans, unit/integration tests, and explains test failures. Tracks coverage and quality gates.

    P08: CICD (CI/CD Pipeline)¶

    Configures and maintains GitHub Actions workflows, quality gates, and deployment automation.

    P09: RUN (Runbook & SRE)¶

    Creates operational runbooks, incident response playbooks, and Site Reliability Engineering documentation.

    P10: MET (Metrics & Insights)¶

    Tracks sprint velocity, burn charts, quality metrics, and generates performance insights.

    P11: SEC (Security & Compliance)¶

    Scans for vulnerabilities, enforces security policies, manages secrets, and ensures compliance.

    P12: UXA (UX & Accessibility)¶

    Designs user experiences, writes copy, ensures WCAG 2.1 AA compliance, and implements GC Design System.

    P13: ONB (Onboarding)¶

    Coaches new developers, creates sandbox environments, and provides learning paths.

    P14: LVO (LiveOps Companion)¶

    Supports production operations, monitors health, and assists with incident triage.

    P15: DVM (Dev Master Orchestrator)¶

    High-level coordination agent (Jarvis) orchestrating other patterns and managing complex workflows.

    P16-P21: Reserved¶

    Future patterns for specialized capabilities (performance tuning, cost optimization, ML ops, etc.).

    Autonomy Levels (C0-C3)¶

    C0: Full Autonomy¶

    Agent operates independently without human approval. Used for routine, low-risk tasks (linting, formatting, documentation updates). Human review optional.

    Use Cases: Automated code formatting, dependency updates, routine backups Risk Level: Low Human Checkpoint: None required Examples: Auto-merge Dependabot PRs, scheduled maintenance tasks

    C1: Human Approval Required¶

    Agent generates output but requires explicit human approval before execution. Used for medium-risk operations (code changes, configuration updates).

    Use Cases: Generated code, architecture decisions, security patches Risk Level: Medium Human Checkpoint: Approve/reject before execution Examples: Generated test suites, API endpoint additions, database migrations

    C2: Collaborative¶

    Agent and human work iteratively, refining outputs through multiple rounds. Used for complex, creative tasks requiring human judgment.

    Use Cases: Design decisions, complex problem-solving, strategic planning Risk Level: Medium-High Human Checkpoint: Continuous collaboration Examples: Architecture design, UX workflows, incident response planning

    C3: Fully Manual (Human-Led)¶

    Human performs work with optional AI assistance for suggestions, spell-check, or research. AI does not generate primary content.

    Use Cases: Final approval decisions, legal/compliance review, strategic direction Risk Level: High (requires human expertise) Human Checkpoint: Human is primary actor Examples: Contract review, ATO approval, go/no-go decisions

    Architecture Terms¶

    API Gateway¶

    Entry point for all API requests. Handles routing, authentication, rate limiting, and logging. EVA uses Azure API Management (APIM).

    Azure AI Search¶

    Managed search service providing vector search, keyword search, and hybrid retrieval. Core of EVA RAG capabilities.

    Azure OpenAI Service¶

    Microsoft's hosted OpenAI models (GPT-4, GPT-3.5) with enterprise security, compliance, and data residency guarantees.

    CMK (Customer-Managed Key)¶

    Encryption key managed by customer (not cloud provider). Stored in Azure Key Vault. Required for Protected B data.

    Cosmos DB¶

    Multi-model NoSQL database providing global distribution, low latency, and guaranteed uptime. EVA's primary data store.

    Hybrid Search¶

    Search approach combining vector similarity and keyword matching. Improves retrieval accuracy by leveraging both semantic and lexical relevance.

    MCP (Model Context Protocol)¶

    Protocol for structured communication between AI models and external systems. Used in EVA for tool invocation and context passing.

    Multi-Region Deployment¶

    Deploying services across multiple Azure regions (Canada Central + Canada East) for high availability and disaster recovery.

    Private Endpoint¶

    Network interface providing private connectivity to Azure services. No public IP exposure. Required for Protected B compliance.

    RAG (Retrieval-Augmented Generation)¶

    AI pattern combining information retrieval with generative models. Retrieves relevant documents, then generates answers grounded in retrieved context.

    TLS 1.2+¶

    Transport Layer Security protocol version 1.2 or higher. Required for encrypted communication. TLS 1.0/1.1 prohibited.

    Vector Database¶

    Database optimized for storing and searching high-dimensional vectors (embeddings). Enables semantic search and similarity matching.

    Agile & Process Terms¶

    Acceptance Criteria¶

    Testable conditions that must be met for a story to be considered complete. Written in "Given-When-Then" format.

    Backlog¶

    Prioritized list of work items (epics, stories, tasks) awaiting development. Managed by Product Owner and Scrum Master.

    Blocker¶

    Issue preventing progress on a story or sprint. Requires immediate escalation and resolution.

    Burn Chart¶

    Visual representation of work completed vs. remaining over sprint duration. Used to track progress and forecast completion.

    Capacity¶

    Team's available working hours/points for a sprint. Accounts for vacations, training, support work, etc.

    Ceremony¶

    Structured meeting in Scrum framework (Planning, Stand-up, Review, Retrospective). Time-boxed and agenda-driven.

    Definition of Done (DoD)¶

    Checklist of quality standards required for work to be considered complete (tests passing, docs updated, QA gate passed, deployed).

    Increment¶

    Sum of all completed work at end of sprint. Must be potentially shippable and meet Definition of Done.

    Planning Poker¶

    Estimation technique where team members assign story points using Fibonacci sequence (1, 2, 3, 5, 8, 13, 21).

    Product Owner (PO)¶

    Role owning product vision, roadmap, backlog prioritization, and acceptance criteria. Marco Presta in EVA context.

    Sprint Backlog¶

    Subset of product backlog committed for current sprint. Contains stories, tasks, and sprint goal.

    Sprint Goal¶

    One-sentence objective for sprint. Guides team decisions and provides focus.

    Sprint Retrospective¶

    Ceremony for team reflection on what went well, what needs improvement, and action items for next sprint.

    Sprint Review¶

    Ceremony demonstrating completed work to stakeholders, gathering feedback, and adjusting backlog.

    Stand-up (Daily Scrum)¶

    Brief daily meeting (15 min) where team shares: yesterday's work, today's plan, and blockers.

    Story Points¶

    Abstract unit of effort estimation representing complexity, uncertainty, and work volume (not time).

    T-Shirt Sizing¶

    Estimation technique using sizes (XS, S, M, L, XL) instead of numeric points. Useful for high-level epic estimation.

    Velocity¶

    Average story points completed per sprint. Used for forecasting and capacity planning.

    Security & Compliance Terms¶

    Audit Log¶

    Immutable record of security-relevant events (authentication, authorization, data access, configuration changes). Retained 7 years for Protected B.

    Compliance¶

    Adherence to laws, regulations, standards, and policies (WCAG, Official Languages Act, TBS directives, etc.).

    Data Residency¶

    Requirement that data be stored and processed within specific geographic boundaries (Canada for Protected B).

    Encryption at Rest¶

    Encrypting data stored on disk. EVA uses AES-256 with customer-managed keys in Azure Key Vault.

    Encryption in Transit¶

    Encrypting data during transmission. EVA enforces TLS 1.2+ for all connections.

    Key Rotation¶

    Periodic replacement of encryption keys. EVA rotates customer-managed keys every 90 days.

    MFA (Multi-Factor Authentication)¶

    Authentication requiring multiple verification methods (password + OTP, password + biometric). Required for all EVA admin access.

    PII (Personally Identifiable Information)¶

    Data that can identify individuals (SSN, address, phone, email). Requires Protected B classification and special handling.

    Secret Scanning¶

    Automated detection of credentials, API keys, and tokens in code. Prevents accidental exposure of secrets.

    SSO (Single Sign-On)¶

    Authentication mechanism allowing users to access multiple services with one set of credentials. EVA uses Azure AD SSO.

    Vulnerability Scanning¶

    Automated scanning for known security vulnerabilities in code and dependencies. EVA uses Snyk and Dependabot.

    Zero Trust¶

    Security model assuming no implicit trust. Every access request verified regardless of location or network.

    Accessibility Terms¶

    ARIA (Accessible Rich Internet Applications)¶

    W3C specification for making dynamic web content accessible to screen readers. Provides roles, states, and properties.

    Alt Text¶

    Text alternative for images, charts, and graphics. Enables screen reader users to understand visual content.

    Color Contrast Ratio¶

    Ratio of luminance between foreground and background colors. WCAG AA requires 4.5:1 for normal text, 3:1 for large text.

    Focus Indicator¶

    Visual indicator showing which element has keyboard focus. Required for keyboard navigation. Must meet contrast requirements.

    Keyboard Navigation¶

    Ability to use website/application using keyboard only (no mouse). Required for users with motor disabilities.

    Screen Reader¶

    Assistive technology reading on-screen content aloud. Popular screen readers: JAWS, NVDA, VoiceOver, TalkBack.

    Semantic HTML¶

    Using HTML elements for their intended purpose (e.g., <button> for buttons, <nav> for navigation). Improves accessibility and SEO.

    Skip Link¶

    Hidden link allowing keyboard users to skip repetitive content (navigation menus) and jump to main content.

    Tab Order¶

    Sequence in which elements receive focus when user presses Tab key. Should follow logical reading order.

    WCAG (Web Content Accessibility Guidelines)¶

    W3C standard defining how to make web content accessible. Three levels: A (minimum), AA (mid-range), AAA (highest).

    WCAG 2.1 AA¶

    Specific version and conformance level required for Government of Canada websites. Includes 50+ success criteria across 4 principles: Perceivable, Operable, Understandable, Robust.

    Additional Terms¶

    Bilingual¶

    Supporting two languages (English and French for Canada). EVA Suite must provide equal functionality and content quality in EN-CA and FR-CA.

    i18n (Internationalization)¶

    Designing software to support multiple languages and locales without code changes. Foundation for translation.

    l10n (Localization)¶

    Adapting software for specific languages/regions (translation, date formats, currency, cultural norms).

    P95 Latency¶

    95th percentile response time. 95% of requests complete faster than this threshold. EVA target: <300ms P95.

    SLA (Service Level Agreement)¶

    Commitment defining expected service availability, performance, and support. EVA production tier: 99.9% uptime.

    SLI (Service Level Indicator)¶

    Measurable metric used to track SLA compliance (e.g., uptime percentage, P95 latency, error rate).

    SLO (Service Level Objective)¶

    Target value for SLI. Example: "P95 latency < 300ms" or "Error rate < 0.1%".

    Technical Debt¶

    Cost of additional rework caused by choosing quick solutions over better long-term approaches. Tracked and managed in backlog.
