    Chapter P14: DevOps LiveOps Companion¶

    Pattern Code: P14 Short Code: LVO Name: DevOps LiveOps Companion Type: DevTools Pattern Scope: Operations, Incident Response, Observability, SRE Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The DevOps LiveOps Companion (P14-LVO) is the real-time operational support system for the EVA Suite. It helps interpret logs, triage incidents, analyze anomalies, and guide on-call engineers through incident response. Think of it as your 24/7 SRE partner‚Äîalways watching production metrics, ready to explain what's happening, and suggesting solutions when things go wrong.

    Without P14, on-call engineers drown in Application Insights logs searching for needles in haystacks, incidents take hours to triage (not minutes), and root causes hide in correlation noise. With P14, log analysis is automated, anomalies are surfaced immediately, and incident summaries write themselves‚Äîturning reactive firefighting into proactive problem-solving.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P14-LVO serves as the Incident Response Assistant & Observability Interpreter for the EVA Autonomous Agile Command. It ensures:

    1. Rapid Incident Detection: Anomalies surfaced within minutes (not hours)
    2. Clear Root Cause Hypotheses: Correlation analysis points to likely causes
    3. Automated Incident Summaries: Natural language reports for handoffs
    4. Runbook Integration: Context-aware links to relevant runbooks (P09)
    5. Proactive Alerting: Threshold breaches flagged before they become incidents
    6. Post-Incident Learning: Automated postmortem generation and root cause analysis

    Mission Statement¶

    "Accelerate incident detection, triage, and resolution through automated log analysis, anomaly detection, and context-aware operational guidance‚Äîempowering on-call engineers to focus on solutions, not search."

    Value Proposition¶

    For On-Call Engineers: - Faster incident triage (anomaly detection highlights what changed) - Clear incident summaries (no more manual log parsing) - Context-aware runbooks (right guidance at the right time) - Automated postmortems (less manual documentation)

    For Product Owners: - Reduced MTTR (mean time to resolution) - Improved SLA/SLO compliance (proactive alerts) - Transparent incident history (audit trail) - Data-driven reliability improvements

    For Operations Team: - Reduced toil (automated log analysis) - Consistent incident response (runbook integration) - Proactive capacity planning (trend analysis) - Knowledge retention (automated postmortems)

    For Customers: - Faster incident resolution (less downtime) - Proactive issue prevention (anomalies caught early) - Transparent incident communication (clear status updates)

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Automated Log Analysis
    2. Query Application Insights for errors, warnings, exceptions
    3. Parse and correlate log entries across services
    4. Identify error patterns and trends

    5. Surface top error messages and stack traces

    6. Real-Time Anomaly Detection

    7. Detect error rate spikes (Z-score analysis)
    8. Detect latency anomalies (percentile threshold breaches)
    9. Detect throughput drops (request volume changes)

    10. Detect availability issues (health check failures)

    11. Incident Response Automation

    12. Auto-create GitHub issues for detected incidents
    13. Notify on-call team (Teams/Slack/PagerDuty)
    14. Attach diagnostic logs and anomaly reports

    15. Link to relevant runbooks (P09)

    16. LiveOps Dashboard Interpretation

    17. Explain metric trends in natural language
    18. Flag threshold breaches (SLO violations)
    19. Provide context-aware recommendations
    20. Highlight correlated events (deployments, config changes)

    Supporting Responsibilities¶

    1. Postmortem Generation
    2. Automated timeline reconstruction (from logs)
    3. Root cause hypothesis generation
    4. Impact analysis (affected users, requests)

    5. Action items tracking (remediation tasks)

    6. Capacity Planning Support

    7. Forecast resource needs (trend extrapolation)
    8. Identify saturation points (CPU, memory, IOPS)
    9. Recommend scaling actions (horizontal vs vertical)

    10. Estimate cost impacts

    11. Knowledge Management

    12. Update runbooks with incident learnings (P09)
    13. Track common failure patterns
    14. Build incident knowledge base (searchable)
    15. Suggest reliability improvements

    üöÄ When to Use P14¶

    Primary Use Cases¶

    1. Real-Time Incident Detection¶

    Scenario: Production experiencing elevated error rate‚Äîneed immediate triage.

    Action: P14 detects anomaly via Application Insights query and alerts on-call.

    Example:

    When NOT to Use P14¶

    • Don't use P14 for security incidents - Escalate to Security Incident Response Team (SIRT)
    • Don't use P14 for data breaches - Follow Data Breach Protocol (Privacy Officer)
    • Don't use P14 for infrastructure outages - Azure Support (P0 incidents)
    • Don't use P14 for business decisions - Operations data informs, doesn't decide

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P14-LVO (DevOps LiveOps Companion), the incident response assistant for EVA DevTools.

**Your Capabilities**:
1. Analyze Application Insights logs for errors, warnings, exceptions
2. Detect anomalies using statistical methods (Z-scores, thresholds)
3. Correlate incidents with deployments and config changes
4. Generate natural language incident summaries
5. Provide runbook recommendations (context-aware)
6. Create automated postmortems

**Your Limitations**:
- You do NOT fix incidents (that's on-call engineers)
- You do NOT make deployment decisions (rollback requires human approval)
- You do NOT access production data (logs only, no PII)
- You do NOT escalate to customers (that's customer support)

**Autonomy Level**: C1 (Human-on-the-loop)
- Automatically detect anomalies and create incident tickets
- Humans triage and resolve incidents
- Automated postmortems require human review before publishing

**Output Format**:
- Use clear, concise language (not technical jargon)
- Prioritize severity (üö® CRITICAL, ‚ö†Ô∏è WARNING, ‚ÑπÔ∏è INFO)
- Include timestamps (UTC) and durations
- Provide actionable recommendations (specific, not vague)
- Link to runbooks and documentation

**Safety Rules**:
- NEVER expose PII in logs or incident reports
- NEVER auto-rollback deployments without human approval
- NEVER assume correlation = causation (flag as hypothesis)
- ALWAYS validate incident severity with on-call engineer

    üé™ Awareness Examples (P16 Format)¶

    Example 1: Anomaly Detection¶

    When P14 detects production anomaly:

    --- Awareness (P16 ‚Äì AWR) ---
pod(s): POD-O (Operations), POD-F (Foundation)
phase: 5 (Production / Day-1)
confidence: High
assumptions:
  - "Baseline metrics accurately reflect normal behavior (30-day rolling average)"
  - "Z-score threshold (3.0) balances sensitivity vs false positives"
  - "Application Insights data is complete (no sampling gaps)"
  - "Anomaly timestamp accuracy ¬±5 minutes (log ingestion delay)"
risks:
  - "False positive (e.g., planned load test not communicated)"
  - "Root cause may not be detectable from metrics alone (deeper investigation needed)"
  - "Correlated events may be coincidental (deployment timing doesn't prove causation)"
  - "Anomaly may resolve before human triage (transient issue)"
actionClass: C1 (human-on-loop: anomaly detected, humans triage and resolve)
suggestedNextAction:
  - "Review Application Insights logs for error patterns"
  - "Correlate with recent deployments/config changes (last 2 hours)"
  - "Validate with on-call engineer (is this expected behavior?)"
  - "If confirmed incident: Create ticket, page on-call, follow runbook"
  - "If false positive: Adjust anomaly detection threshold (reduce sensitivity)"
evidenceRefs:
  - "Application Insights: /query API results"
  - "docs/runbooks/performance-degradation.md"
  - "GitHub Actions: Recent deployment logs"
