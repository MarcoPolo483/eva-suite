    Chapter P10: Metrics & Insight Agent¶

    Pattern Code: P10 Short Code: MET Name: Metrics & Insight Agent Type: DevTools Pattern Scope: Observability & Analytics Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Metrics & Insight Agent (P10-MET) is the observability backbone of the EVA DevTools ecosystem. It automatically collects, analyzes, and interprets delivery and operational metrics to provide actionable insights for continuous improvement. Think of it as your team's data analyst‚Äîalways watching, always learning, always suggesting improvements.

    Without P10, teams fly blind, relying on intuition and anecdotes to make critical decisions about performance, quality, and process. With P10, every decision is backed by data, trends are spotted early, and improvement opportunities are surfaced automatically.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P10-MET serves as the Metrics Collector & Insight Generator for the EVA Autonomous Agile Command. It ensures:

    1. Transparency: All metrics are visible and accessible to stakeholders
    2. Accountability: Performance tracked against targets and SLOs
    3. Continuous Improvement: Data-driven insights guide optimization
    4. Early Warning: Anomalies detected before they become incidents

    Mission Statement¶

    "Enable data-driven decision making through automated metrics collection, trend analysis, and actionable insights that drive continuous improvement across the EVA Suite."

    Value Proposition¶

    For Product Owners: - Understand delivery velocity and sprint predictability - Track product health metrics (uptime, latency, errors) - Justify investment decisions with quantitative data - Monitor cost efficiency and ROI

    For Scrum Masters: - Measure sprint velocity and cycle time trends - Identify process bottlenecks and improvement opportunities - Track team capacity and utilization - Forecast delivery timelines with confidence

    For Developers: - Monitor code quality metrics (test coverage, complexity) - Track build/deployment success rates - Understand performance baselines and regressions - Get early warning on anomalies

    For Operations: - Real-time operational health dashboards - SLA/SLO compliance tracking - Cost attribution and optimization insights - Capacity planning data

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Sprint Delivery Metrics Collection
    2. Velocity (story points completed per sprint)
    3. Burn-down/burn-up charts
    4. Cycle time (creation ‚Üí completion)
    5. Issue throughput and completion rate

    6. Per-assignee productivity metrics

    7. Operational Metrics Collection

    8. Availability (uptime percentage)
    9. Latency (P50, P95, P99 response times)
    10. Error rate and success rate
    11. Request throughput (requests/minute)

    12. Active user counts

    13. Trend Analysis

    14. Detect increasing/decreasing/stable trends
    15. Identify anomalies using statistical methods
    16. Forecast future metrics (linear regression)

    17. Compare against historical baselines

    18. Insight Generation

    19. Surface performance issues (latency spikes)
    20. Highlight quality concerns (elevated error rates)
    21. Flag process problems (declining velocity)
    22. Recommend cost optimizations

    Supporting Responsibilities¶

    1. Cost Tracking
    2. Monthly cloud spend (Azure Cost Management)
    3. Cost per user, cost per request
    4. Budget variance alerts

    5. Cost optimization recommendations

    6. Quality Metrics

    7. Test coverage percentage
    8. Build success/failure rates
    9. Deployment frequency and lead time

    10. Mean time to recovery (MTTR)

    11. Reporting & Visualization

    12. Generate markdown/HTML reports
    13. Export data for dashboards (Power BI, Grafana)
    14. Automated weekly/monthly summaries
    15. Executive-level KPI snapshots

    üöÄ When to Use P10¶

    Primary Use Cases¶

    1. Sprint Retrospectives¶

    Scenario: Team wants to review sprint performance and identify improvement areas.

    Action: Run P10 to collect velocity, cycle time, and completion rate data for the sprint.

    Example: 

    .\tools\metrics\collect-sprint-metrics.ps1 -SprintId "Sprint-5"

    Output: 

    üìä Sprint Summary:
  Total Issues: 24
  Completed: 20 (83.3%)
  In Progress: 3
  Not Started: 1

üìä Story Points:
  Velocity: 32 / 40 (80%)

‚è±Ô∏è Cycle Time:
  Average: 4.2 days

    2. Operational Health Checks¶

    Scenario: Operations team needs to validate SLO compliance for the past week.

    Action: Query P10 for availability, latency, and error rate metrics from Application Insights.

    Example: 

    const kpis = await collector.collectKPIs(appInsightsId, '7d');

console.log(`Availability: ${kpis.availability}%`);  // Target: 99.9%
console.log(`P95 Latency: ${kpis.latency_p95}ms`);   // Target: <300ms
console.log(`Error Rate: ${kpis.error_rate}%`);      // Target: <1%

    Output: 

    Availability: 99.94%  ‚úÖ
P95 Latency: 245ms    ‚úÖ
Error Rate: 0.7%      ‚úÖ

    3. Trend Analysis for Capacity Planning¶

    Scenario: Team needs to forecast infrastructure capacity for next quarter.

    Action: Use P10 trend analysis to project request volume and user growth.

    Example: 

    analyzer = TrendAnalyzer(metrics_df)
forecast, confidence = analyzer.forecast_next_week('requests_per_minute')

print(f"Forecast RPM: {forecast:.0f} ¬± {confidence:.0f}")

    Output: 

    Forecast RPM: 1250 ¬± 150
Recommendation: Scale to support 1500 RPM (20% headroom)

    4. Anomaly Detection During Incidents¶

    Scenario: Production incident detected‚Äîneed to identify root cause metrics.

    Action: Run P10 anomaly detector to find unusual patterns in last 2 hours.

    Example: 

    detector = AnomalyDetector(app_insights_id)
anomalies = detector.detect_error_rate_anomalies(hours=2)

print(f"Anomalies detected: {len(anomalies)}")

    Output: 

    ‚ö†Ô∏è Error Rate Spike Detected:
  - 2025-12-12 14:35:00 : 87 errors (12.3%)
  - Z-Score: 5.2 (Critical)

    When NOT to Use P10¶

    • Don't use P10 for real-time alerting - Use Azure Monitor alerts for immediate notifications
    • Don't use P10 for exploratory data analysis - Use Application Insights or Kusto for ad-hoc queries
    • Don't use P10 as a data warehouse - Store raw data in Azure Storage/Data Lake

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P10-MET (Metrics & Insight Agent), the Observability & Analytics expert for EVA DevTools.

**Your Capabilities**:
1. Collect sprint delivery metrics from GitHub (velocity, cycle time, throughput)
2. Query operational metrics from Azure Application Insights (latency, errors, availability)
3. Analyze trends using statistical methods (Z-scores, rolling averages)
4. Generate actionable insights from metric anomalies
5. Forecast future metrics using linear regression

**Your Limitations**:
- You do NOT configure Application Insights or Azure Monitor (that's infrastructure)
- You do NOT set SLOs/SLAs (that's product/business decision)
- You do NOT modify source data (read-only access to telemetry)

**Autonomy Level**: C1 (Human-on-the-loop)
- Automatically collect metrics and detect anomalies
- Generate insights and recommendations
- Humans review insights before taking action

**Output Format**:
- Use structured markdown with tables and charts (when applicable)
- Include metric values, targets, and deltas
- Provide specific recommendations, not vague suggestions
- Link to relevant runbooks or documentation

**Safety Rules**:
- Always compare metrics against established baselines/targets
- Flag anomalies with severity levels (info/warning/critical)
- Include confidence intervals for forecasts
- Never assume causation from correlation alone

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C1 (Human-on-the-Loop)¶

    P10 Itself: C1 - Automatically collects metrics and detects anomalies - Generates insights without human approval - Humans review insights before taking action

    Why C1? - Metrics collection is low-risk (read-only operations) - Anomaly detection benefits from immediate flagging - Insights are advisory‚Äîhumans decide on actions

    Safety Rules¶

    Rule 1: Data Integrity¶

    Risk: Corrupt/missing data leads to false insights

    Mitigation: - ‚úÖ Validate data completeness before analysis (check for gaps) - ‚úÖ Cross-reference multiple data sources (GitHub + App Insights) - ‚úÖ Flag low-confidence insights (insufficient data) - ‚úÖ Archive raw data for audit trail

    Rule 2: Baseline Accuracy¶

    Risk: Incorrect baselines cause false anomalies

    Mitigation: - ‚úÖ Use rolling 30-day average for baselines (not all-time) - ‚úÖ Exclude known incidents from baseline calculation - ‚úÖ Recalibrate baselines quarterly - ‚úÖ Allow manual baseline overrides (approved by team)

    Rule 3: Correlation vs Causation¶

    Risk: Recommend actions based on spurious correlations

    Mitigation: - ‚úÖ Always flag insights as "correlation, not causation" - ‚úÖ Require human validation before attributing root cause - ‚úÖ Provide multiple hypotheses, not single conclusion - ‚úÖ Link to runbooks for systematic troubleshooting

    Rule 4: Privacy & Security¶

    Risk: Expose sensitive data in reports (PII, credentials)

    Mitigation: - ‚úÖ Never include PII in metric reports (anonymize user IDs) - ‚úÖ Redact sensitive error messages (credentials, tokens) - ‚úÖ Restrict access to raw telemetry (RBAC) - ‚úÖ Encrypt metric data at rest and in transit

    Autonomy Level Definitions (Reference)¶

    Level Name Description P10 Usage
    C0 Human-in-the-loop AI suggests, human approves every action Insight prioritization
    C1 Human-on-the-loop AI acts, human monitors Metrics collection, anomaly detection
    C2 Human-over-the-loop AI acts with constraints, periodic review Not applicable (P10 is read-only)
    C3 Full autonomy AI acts autonomously within boundaries Not applicable (P10 is read-only)

    ‚úÖ End-of-Chapter Checklist¶

    For Metrics Consumers (Reviewing Insights)¶

    • I understand what metrics are being collected and how
    • I have validated metric baselines/targets are appropriate
    • I have reviewed anomalies flagged by P10
    • I have cross-referenced insights with other data sources
    • I have prioritized recommended actions by severity/impact
    • I have added high-priority actions to sprint backlog
    • I understand the confidence level of insights (high/medium/low)

    For Metrics Producers (Configuring P10)¶

    • Application Insights instrumentation is complete
    • GitHub issue labels follow consistent naming convention
    • Sprint milestones accurately reflect planned work
    • Cost allocation tags are applied to Azure resources
    • Telemetry data retention meets compliance requirements (7 years)
    • RBAC configured for metric access (principle of least privilege)
    • Alerting thresholds reviewed and approved by team

    For Governance (Periodic Audit)¶

    • Metric collection scripts are up-to-date
    • No PII/sensitive data exposed in reports
    • Baselines recalibrated within last 90 days
    • Anomaly detection false positive rate <10%
    • All insights tracked to resolution (closed-loop)
    • Cost metrics align with Azure Cost Management
    • SLO compliance tracked and reported

    üéì Key Takeaways¶

    1. Metrics enable improvement - You can't improve what you don't measure
    2. Trends reveal patterns - A single data point is noise; trends are signal
    3. Baselines matter - Without baselines, anomalies are undetectable
    4. Insights require action - Collecting data without action is waste
    5. Correlation ‚â† causation - Always validate hypotheses before acting
    6. Privacy is paramount - Never expose PII or sensitive data in metrics
    7. Automation scales - Manual metric collection doesn't scale to 40+ products
    8. Confidence matters - Flag low-confidence insights to avoid bad decisions
