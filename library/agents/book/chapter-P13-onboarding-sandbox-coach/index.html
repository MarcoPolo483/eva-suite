    Chapter P13: Onboarding & Sandbox Coach¶

    Pattern Code: P13 Short Code: ONB Name: Onboarding & Sandbox Coach Type: DevTools Pattern Scope: Onboarding, Sandbox, Self-Service, Education Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Onboarding & Sandbox Coach (P13-ONB) is the self-service enablement engine for the EVA Suite. It guides new teams into EVA Sandbox environments, provides templates and scaffolding for rapid experimentation, enforces guardrails (cost, security, compliance), and coaches teams on EVA Suite best practices. Think of it as your friendly onboarding concierge‚Äîautomated provisioning, guided tutorials, and safety nets to prevent costly mistakes.

    Without P13, onboarding is a bottleneck (manual provisioning takes days, not minutes), teams flounder without guidance (wasting time on solved problems), and sandbox environments spiral out of control (cost overruns, security gaps). With P13, new teams are productive in hours, experimentation is safe and fast, and best practices are baked in from day one.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P13-ONB serves as the Self-Service Onboarding Platform for the EVA Autonomous Agile Command. It ensures:

    1. Rapid Onboarding: New teams provisioned in <5 minutes (not days)
    2. Guided Learning: Templates and tutorials accelerate time-to-productivity
    3. Safe Experimentation: Guardrails prevent cost/security/compliance disasters
    4. Best Practices: Bake in EVA Suite patterns from day one
    5. Self-Service: Teams unblock themselves without waiting for platform team
    6. Graduation Path: Clear criteria for Sandbox ‚Üí Pilot ‚Üí Production promotion

    Mission Statement¶

    "Accelerate team productivity and reduce onboarding friction through automated sandbox provisioning, guided templates, and proactive guardrails‚Äîenabling safe, fast, self-service experimentation."

    Value Proposition¶

    For New Teams: - Get started in minutes (not waiting for manual provisioning) - Pre-validated templates (not starting from scratch) - Clear guardrails (avoid costly mistakes) - Coaching on demand (not searching scattered docs)

    For Platform Team: - Reduce manual provisioning toil (automate repetitive work) - Scale onboarding without scaling headcount - Proactive cost/security enforcement (prevent vs remediate) - Self-service reduces support burden

    For Finance (FinOps): - Budget enforcement prevents cost overruns - Automatic resource cleanup (no zombie sandboxes) - Cost attribution by team (chargebacks) - Forecasting based on usage patterns

    For Security: - Security guardrails enforced from day one - No public IPs or non-compliant resources - Automatic compliance validation - Audit trail of all sandbox activities

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Automated Sandbox Provisioning
    2. Create Azure resource group with team-specific naming
    3. Deploy baseline infrastructure (App Service, AI Search, Cosmos DB, Storage)
    4. Apply cost budget alerts (CAD 500/month default)
    5. Configure RBAC (team lead = Contributor)

    6. Generate onboarding guide (team-specific)

    7. Template & Sample Library

    8. Minimal API template (FastAPI starter)
    9. RAG chatbot template (Azure OpenAI + AI Search)
    10. Document processing template (Azure Functions)
    11. Full-stack template (React + FastAPI + Cosmos DB)

    12. Infrastructure templates (Bicep/ARM)

    13. Guardrail Enforcement

    14. Cost budget alerts (80%, 100%, 110% thresholds)
    15. Security policies (private endpoints only, encryption required)
    16. Data policies (Canada residency, no PII without approval)

    17. Auto-shutdown (non-critical resources at 110% budget)

    18. Coaching & Tutorials

    19. Onboarding playbook (week-by-week milestones)
    20. Video tutorials (Azure basics, EVA Suite patterns)
    21. Office hours (weekly Q&A sessions)
    22. Community support (#eva-sandbox-support channel)

    Supporting Responsibilities¶

    1. Sandbox Lifecycle Management
    2. Auto-expiry (3-month default, extension on request)
    3. Cleanup of idle resources (>30 days unused)
    4. Graduation criteria (Sandbox ‚Üí Pilot checklist)

    5. Decommissioning workflow (archive data, delete resources)

    6. Cost Monitoring & Optimization

    7. Daily cost reports (per sandbox)
    8. Cost optimization recommendations (right-size resources)
    9. Budget variance alerts (proactive warnings)

    10. Cost attribution (team chargebacks)

    11. Compliance Validation

    12. Security scan (P11) on all sandbox resources
    13. Accessibility check (P12) on UI components
    14. GC Design System validation (for UI apps)
    15. Audit trail for ATO evidence

    When NOT to Use P13¶

    • Don't use P13 for Production provisioning - That's infrastructure team (IaC pipelines)
    • Don't use P13 for long-term development - Sandbox is time-limited (3 months default)
    • Don't use P13 for production data - Sandbox is for experimentation only
    • Don't use P13 to bypass governance - Graduation criteria are mandatory

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P13-ONB (Onboarding & Sandbox Coach), the self-service onboarding platform for EVA DevTools.

**Your Capabilities**:
1. Provision sandbox environments (Azure resource groups, baseline infrastructure)
2. Deploy sample applications (RAG chatbot, minimal API, document processing)
3. Enforce guardrails (cost budgets, security policies, compliance checks)
4. Provide onboarding guidance (playbooks, tutorials, best practices)
5. Monitor sandbox health (cost, security, compliance)
6. Validate graduation criteria (Sandbox ‚Üí Pilot)

**Your Limitations**:
- You do NOT provision Production environments (that's infrastructure team)
- You do NOT bypass security policies (guardrails are mandatory)
- You do NOT extend budgets without approval (escalate to FinOps)
- You do NOT store production data in sandboxes (compliance violation)

**Autonomy Level**: C1 (Human-on-the-loop)
- Automatically provision sandboxes and deploy templates
- Automatically enforce cost/security guardrails
- Humans approve budget extensions and Pilot graduations

**Output Format**:
- Use clear step-by-step instructions
- Provide resource group names, URLs, credentials (securely)
- Include cost estimates and budget impacts
- Link to relevant documentation and tutorials

**Safety Rules**:
- NEVER create resources outside Canada regions (data residency)
- NEVER disable security policies (private endpoints, encryption)
- NEVER allow PII in sandboxes without Data Steward approval
- ALWAYS enforce budget limits (auto-shutdown at 110%)

pod(s): POD-S (Sandbox), requesting team

    Example 2: Budget Alert¶

    When P13 detects budget threshold exceeded:

    --- Awareness (P16 ‚Äì AWR) ---
pod(s): POD-S (Sandbox), team exceeding budget
phase: 0 (Intake / Sandbox) or 1 (Pilot)
confidence: High
assumptions:
  - "Cost data from Azure Cost Management is accurate (may be 24hr delayed)"
  - "Team is actively using sandbox (not abandoned)"
  - "Budget alerts configured correctly (80%, 100%, 110%)"
risks:
  - "Team may ignore alerts (escalation to lead required)"
  - "Cost optimization may require downtime (coordinate with team)"
  - "Auto-shutdown may disrupt active work (warn team in advance)"
actionClass: C1 (human-on-loop: alert sent, humans decide on action)
suggestedNextAction:
  - "Review cost breakdown in Azure Cost Management"
  - "Identify top cost drivers (AI Search, App Service, OpenAI)"
  - "Scale down or delete unused resources"
  - "Request budget extension (if justified) via #eva-sandbox-support"
  - "At 110%: Non-critical resources auto-shutdown (critical preserved)"
evidenceRefs:
  - "Azure Cost Management: /costAnalysis"
  - "tools/sandbox/cost-optimization-guide.md"

## Security (P11)

### Secrets Management
- [ ] No hardcoded secrets (checked via TruffleHog, git grep)
- [ ] All secrets stored in Azure Key Vault

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C1 (Human-on-the-Loop)¶

    P13 Itself: C1 - Automatically provisions sandboxes and enforces guardrails - Humans monitor costs and approve budget extensions - Humans approve Pilot graduations

    Why C1? - Sandbox provisioning is low-risk (time-limited, budget-capped) - Guardrails prevent catastrophic failures (auto-shutdown at 110%) - Graduation requires human judgment (not fully automatable)

    Safety Rules¶

    Rule 1: Budget Enforcement¶

    Risk: Runaway costs from unmonitored resources

    Mitigation: - ‚úÖ Hard budget caps enforced (CAD 500/month default) - ‚úÖ Automated alerts at 80%, 100%, 110% - ‚úÖ Auto-shutdown non-critical resources at 110% - ‚úÖ Budget extensions require FinOps approval

    Rule 2: Data Residency¶

    Risk: Data stored outside Canada (compliance violation)

    Mitigation: - ‚úÖ Azure Policy enforces Canada Central/East only - ‚úÖ Cross-region replication disabled - ‚úÖ Provisioning script validates region parameter - ‚úÖ Audit trail of all resource creations

    Rule 3: Security Policies¶

    Risk: Insecure resources (public IPs, weak encryption)

    Mitigation: - ‚úÖ Azure Policy blocks public IPs - ‚úÖ Encryption at rest mandatory (AES-256) - ‚úÖ Private endpoints required for all services - ‚úÖ Security scan (P11) before Pilot graduation

    Rule 4: Sandbox Lifecycle¶

    Risk: Zombie sandboxes consuming resources indefinitely

    Mitigation: - ‚úÖ Auto-expiry after 3 months (configurable) - ‚úÖ Reminder sent 2 weeks before expiry - ‚úÖ Extension requires justification - ‚úÖ Unused resources (>30 days idle) flagged for cleanup

    Autonomy Level Definitions (Reference)¶

    Level Name Description P13 Usage
    C0 Human-in-the-loop AI suggests, human approves every action Pilot graduation approval
    C1 Human-on-the-loop AI acts, human monitors Sandbox provisioning, guardrail enforcement
    C2 Human-over-the-loop AI acts with constraints, periodic review Not applicable (sandboxes too dynamic)
    C3 Full autonomy AI acts autonomously within boundaries Not applicable (cost/security risks)
