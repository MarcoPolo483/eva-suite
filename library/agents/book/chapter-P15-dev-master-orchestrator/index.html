    Chapter P15: Dev Master Orchestrator (Jarvis)¶

    Pattern Code: P15 Short Code: DVM Name: Dev Master Orchestrator (Jarvis) Type: DevTools Pattern Scope: Multi-Agent Coordination & Workflow Orchestration Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Dev Master Orchestrator, affectionately known as "Jarvis," is the conductor of the EVA DevTools symphony. It coordinates all other DevTools agents (P01-P14, P16-P21) across multiple repositories, orchestrating complex multi-step workflows that would be tedious and error-prone if executed manually.

    Without P15, teams would manually chain together pattern invocations, manage dependencies, handle errors, and coordinate cross-repo impacts. With P15, workflows become declarative, repeatable, and automated‚Äîwhile keeping humans in control through monitoring and intervention capabilities.

    P15 is the pattern that makes the EVA Autonomous Agile Command truly autonomous (within C0-C2 guardrails), enabling patterns to compose into sophisticated pipelines that accelerate feature delivery, improve quality, and reduce toil.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P15-DVM serves as the Multi-Agent Workflow Orchestrator and Cross-Repo Coordinator for the EVA Autonomous Agile Command. It ensures:

    1. Workflow Automation: Multi-step processes execute end-to-end without manual handoffs
    2. Pattern Composition: DevTools patterns (P01-P14, P16-P21) work together seamlessly
    3. Dependency Management: Cross-repo dependencies are validated before execution
    4. Error Handling: Failures are handled gracefully (fail, continue, rollback)
    5. Human Oversight: Humans can monitor, pause, intervene at any time (C1 autonomy)

    Mission Statement¶

    "Enable rapid, reliable, and composable multi-agent workflows that accelerate software delivery while maintaining human control and visibility."

    Value Proposition¶

    For Product Owners: - Accelerate feature delivery through automated end-to-end workflows - Reduce risk through consistent, tested workflow execution - Improve predictability with workflow templates and metrics

    For Developers: - Eliminate manual handoffs between tools and processes - Focus on coding instead of orchestration toil - Benefit from proven workflow patterns (new feature, deployment, retro)

    For Operations: - Standardized deployment and operational workflows - Clear error handling and rollback procedures - Audit trail for all workflow executions (via P21-PRV)

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Maintain Agent Registry (tools/jarvis/agent-registry.yaml)
    2. Map pattern codes (P01-P21) to execution endpoints
    3. Track agent capabilities and dependencies

    4. Document agent types (meta, requirements, testing, security, etc.)

    5. Execute Multi-Step Workflows

    6. Parse workflow definitions (YAML)
    7. Resolve agent endpoints from registry
    8. Build parameters from context and previous step outputs
    9. Handle errors per step configuration (fail, continue, rollback)

    10. Store intermediate results for downstream steps

    11. Coordinate Cross-Repo Operations

    12. Validate dependencies before cross-repo workflows
    13. Analyze impact of changes across repository graph
    14. Generate coordination plans for breaking changes

    15. Track repository ownership and relationships

    16. Provide Pre-Defined Workflows

    17. New feature development (P02 ‚Üí P03 ‚Üí P05 ‚Üí P07 ‚Üí P11 ‚Üí P12 ‚Üí P06)
    18. Sprint retrospective (P10 ‚Üí P14 ‚Üí P11 ‚Üí P09 ‚Üí P04)
    19. Production deployment (P08 ‚Üí P11 ‚Üí P12 ‚Üí P09 ‚Üí P08 ‚Üí P14 ‚Üí P04)
    20. Code review swarm (P17 ‚Üí P06 ‚Üí P07 ‚Üí P11 ‚Üí P16)

    Supporting Responsibilities¶

    1. Workflow Monitoring
    2. Real-time status dashboard (which steps executing, completed, failed)
    3. Human pause/stop mechanisms

    4. Alert on anomalous behavior (long execution times, high error rates)

    5. Workflow Metrics

    6. Track workflow execution times and success rates
    7. Identify bottleneck steps

    8. Measure impact of workflow optimizations

    9. Workflow Documentation

    10. Auto-generate workflow diagrams
    11. Document workflow dependencies and prerequisites
    12. Provide workflow templates for common scenarios

    üöÄ When to Use P15¶

    Primary Use Cases¶

    1. New Feature Development Workflow¶

    Scenario: You're starting a new feature and want to automate the entire process from requirements to PR.

    Action: Invoke new-feature workflow with feature brief.

    Example: 

    tools/jarvis/orchestrator.ps1 -WorkflowName new-feature -Context @{
    brief_path = "docs/features/multi-region-cosmos.md"
    project = "eva-foundation"
    template = "azure-service"
}

    Steps Executed: 1. P02 Requirements Engine: Refine requirements from brief 2. P03 Scrum Agent: Create epic and stories 3. P05 Scaffolder: Generate boilerplate code 4. P07 Testing Agent: Generate unit tests 5. P11 Security: Run security scan 6. P12 UX/Accessibility: Check UI compliance (if applicable) 7. P06 Review Agent: Create PR with findings

    Output: PR URL, epic ID, all artifacts ready for review

    2. Sprint Retrospective Workflow¶

    Scenario: Sprint ended, need to gather insights for retrospective meeting.

    Action: Invoke sprint-retro workflow with sprint ID.

    Example: 

    tools/jarvis/orchestrator.ps1 -WorkflowName sprint-retro -Context @{
    sprint_id = "sprint-007"
}

    Steps Executed: 1. P10 Metrics Agent: Collect sprint delivery metrics 2. P14 LiveOps: Analyze incidents and alerts 3. P11 Security: Review security findings 4. P09 Runbook Agent: Update runbooks based on incidents 5. P04 Librarian: Generate retrospective report

    Output: Retrospective report with quantitative data and actionable insights

    3. Production Deployment Workflow¶

    Scenario: Deploying a new version to Production with full quality gates.

    Action: Invoke production-deploy workflow with app details.

    Example: 

    tools/jarvis/orchestrator.ps1 -WorkflowName production-deploy -Context @{
    app_name = "eva-api"
    version = "v1.2.3"
    app_path = "../eva-api"
    staging_url = "https://staging-eva-api.azurewebsites.net"
}

    Steps Executed: 1. P08 CI/CD Guardian: Validate pipeline configuration 2. P11 Security: Run security scan (fail on medium+ severity) 3. P12 UX/Accessibility: Audit WCAG compliance 4. P09 Runbook Agent: Generate deployment runbook 5. P08 CI/CD Guardian: Execute deployment 6. P14 LiveOps: Enable monitoring and alerting 7. P04 Librarian: Update documentation with deployment details

    Output: Deployment timestamp, runbook path, monitoring confirmation

    4. Cross-Repo Impact Analysis¶

    Scenario: You're making a breaking change to eva-foundation and need to coordinate updates across all dependent repos.

    Action: Invoke impact-analysis with target repo and change type.

    Example: 

    tools/jarvis/impact-analysis.ps1 -TargetRepo eva-foundation -ChangeType breaking

    Steps Executed: 1. Load repository dependency graph 2. Identify all dependent repos (eva-api, eva-rag, eva-auth) 3. Check transitive dependents 4. Generate impact report with coordination checklist 5. Create workflow for coordinated release

    Output: Impact report with all affected repos and coordination plan

    When NOT to Use P15¶

    • Don't use P15 for single-pattern invocations - Call patterns directly if no coordination needed
    • Don't use P15 for one-off workflows - Reserve for reusable, multi-agent processes
    • Don't use P15 to bypass governance - Workflows must respect C0-C3 autonomy levels

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C1 (Human-on-the-Loop)¶

    P15 Itself: C1 - Workflows execute automatically once triggered - Humans monitor workflow progress in real-time - Humans can pause/stop workflows at any time - Humans review workflow outcomes within 24 hours - Critical errors trigger immediate human notification

    Why C1? - Workflows automate tedious multi-step processes (saves hours per workflow) - Human monitoring prevents cascading failures - Pause/stop capability provides safety net - Audit trail (P21-PRV) enables retrospective analysis

    Safety Rules¶

    Rule 1: Validate Before Execute¶

    Risk: Workflow proceeds with invalid inputs, causing failures downstream

    Mitigation: - ‚úÖ Validate all context parameters before workflow starts - ‚úÖ Check prerequisites (agent availability, repo access) - ‚úÖ Dry-run mode for testing workflows without side effects - ‚úÖ Human approval required for destructive workflows (deployment, migration)

    Rule 2: Fail Fast on Critical Errors¶

    Risk: Workflow continues despite critical failure, compounding problems

    Mitigation: - ‚úÖ Mark steps as on_error: fail for critical operations (security scans, backups) - ‚úÖ Surface errors immediately to monitoring dashboard - ‚úÖ Provide clear error messages with recovery guidance - ‚úÖ Rollback capability for destructive operations

    Rule 3: Respect Autonomy Levels¶

    Risk: Workflow bypasses C0 (human-in-loop) patterns, executing without approval

    Mitigation: - ‚úÖ Each workflow step respects pattern's autonomy level - ‚úÖ C0 patterns (e.g., P02-REQ for new features) pause for human approval - ‚úÖ C2 patterns execute with constraints, no human intervention needed - ‚úÖ Workflow documentation specifies which steps require human interaction

    Rule 4: Audit All Executions¶

    Risk: No record of workflow execution for troubleshooting or compliance

    Mitigation: - ‚úÖ Every workflow execution logged to P21-PRV (provenance fingerprint) - ‚úÖ Capture: workflow name, context, steps executed, outputs, errors - ‚úÖ 7-year retention per GC requirements - ‚úÖ Queryable for debugging and audit purposes

    Autonomy Level Definitions (Reference)¶

    Level Name Description P15 Usage
    C0 Human-in-the-loop AI suggests, human approves every action Workflows with C0 steps pause for approval
    C1 Human-on-the-loop AI acts, human monitors P15 workflows (human can pause/stop)
    C2 Human-over-the-loop AI acts with constraints, periodic review Individual workflow steps (e.g., P07 test generation)
    C3 Full autonomy AI acts autonomously within boundaries Not used in P15 (reserved for future)

    Workflow Execution Engine¶

    File: tools/jarvis/orchestrator.ps1

    # Jarvis Orchestrator - Multi-agent workflow coordination
param(
    [Parameter(Mandatory)]
    [string]$WorkflowName,

    [hashtable]$Context = @{},

    [switch]$DryRun
)

Write-Host "ü§ñ Jarvis Orchestrator Starting..." -ForegroundColor Cyan
Write-Host "Workflow: $WorkflowName" -ForegroundColor Yellow

if ($DryRun) {
    Write-Host "üß™ DRY RUN MODE - No actions will be executed" -ForegroundColor Magenta
}

# Load workflow definition
$workflowPath = "tools/jarvis/workflows/$WorkflowName.yaml"
if (!(Test-Path $workflowPath)) {
    Write-Error "Workflow not found: $WorkflowName"
    exit 1
}

$workflow = Get-Content $workflowPath | ConvertFrom-Yaml

Write-Host "`nWorkflow Steps: $($workflow.steps.Count)" -ForegroundColor Cyan

# Validate prerequisites
if ($workflow.prerequisites) {
    Write-Host "`nüîç Validating Prerequisites..." -ForegroundColor Yellow
    foreach ($prereq in $workflow.prerequisites) {
        Write-Host "  - $($prereq.description)" -ForegroundColor Gray
        # Validation logic here
    }
}

# Execute workflow steps
$results = @{}
$errors = @()

foreach ($step in $workflow.steps) {
    Write-Host "`n‚ñ∂Ô∏è Step $($step.id): $($step.name)" -ForegroundColor Green

    # Resolve agent from registry
    $agent = Get-Agent -PatternCode $step.agent

    if (!$agent) {
        Write-Error "Agent not found: $($step.agent)"
        $errors += "Step $($step.id): Agent not found"

        if ($step.on_error -eq 'fail') {
            exit 1
        } else {
            continue
        }
    }

    # Build parameters from context and previous outputs
    $params = @{}
    foreach ($key in $step.params.Keys) {
        $value = $step.params[$key]

        # Resolve references to previous step outputs
        if ($value -match '^\$\{(.+)\}$') {
            $reference = $Matches[1]
            $parts = $reference -split '\.'

            if ($parts[0] -eq 'context') {
                $value = $Context[$parts[1]]
            } else {
                $value = $results[$parts[0]][$parts[1]]
            }
        }

        $params[$key] = $value
    }

    # Execute agent (or skip if dry run)
    Write-Host "  Invoking: $($agent.name)" -ForegroundColor Yellow

    if ($DryRun) {
        Write-Host "  ‚è≠Ô∏è SKIPPED (dry run)" -ForegroundColor Magenta
        $output = @{ status = "dry_run_skipped" }
    } else {
        try {
            $output = & $agent.endpoint @params
            Write-Host "  ‚úÖ Step $($step.id) completed" -ForegroundColor Green
        } catch {
            Write-Error "Step $($step.id) failed: $_"
            $errors += "Step $($step.id): $_"

            switch ($step.on_error) {
                'fail' {
                    Write-Error "Workflow failed at step $($step.id)"
                    exit 1
                }
                'continue' {
                    Write-Warning "Continuing despite error (on_error=continue)"
                }
                'rollback' {
                    Write-Warning "Rolling back workflow (on_error=rollback)"
                    # Rollback logic here
                    exit 1
                }
                'alert' {
                    Write-Warning "Alerting SRE (on_error=alert)"
                    # Alert logic here
                }
            }
        }
    }

    # Store result for downstream steps
    $results[$step.id] = $output
}

# Generate final outputs
$finalOutputs = @{}
foreach ($key in $workflow.outputs.Keys) {
    $reference = $workflow.outputs[$key]

    if ($reference -match '^\$\{(.+)\}$') {
        $refParts = $Matches[1] -split '\.'
        $finalOutputs[$key] = $results[$refParts[0]][$refParts[1]]
    } else {
        $finalOutputs[$key] = $reference
    }
}

# Generate provenance fingerprint (P21)
$fingerprint = @{
    workflow_name = $WorkflowName
    context = $Context
    steps_executed = $results.Keys
    outputs = $finalOutputs
    errors = $errors
    timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
}

# Save fingerprint for audit trail
$fingerprint | ConvertTo-Json -Depth 10 | Set-Content "provenance/workflow-$WorkflowName-$(Get-Date -Format 'yyyyMMdd-HHmmss').json"

# Print summary
Write-Host "`n‚úÖ Workflow '$WorkflowName' completed!" -ForegroundColor Green
Write-Host "`nOutputs:" -ForegroundColor Cyan
$finalOutputs | ConvertTo-Json -Depth 5

if ($errors.Count -gt 0) {
    Write-Host "`n‚ö†Ô∏è Errors encountered:" -ForegroundColor Yellow
    $errors | ForEach-Object { Write-Host "  - $_" -ForegroundColor Red }
}

return $finalOutputs

    timeout: 2m

  epic_id: ${create_epic.epic_id}
  branch: ${scaffold_code.branch}

    For Workflow Users¶

    • I have reviewed available pre-defined workflows
    • I understand workflow prerequisites and context parameters
    • I know how to monitor workflow execution in real-time
    • I know how to pause/stop a workflow if needed
    • I have tested workflows in dry-run mode first
    • I understand which steps require human approval (C0 patterns)
    • I have a plan for handling workflow failures

    For Workflow Creators¶

    • Workflow solves a real, reusable multi-step problem
    • All workflow steps use registered agents from registry
    • Error handling strategy defined for each step (fail/continue/rollback/alert)
    • Prerequisites documented and validated before execution
    • Outputs clearly defined for downstream consumption
    • Workflow documented with usage examples
    • Workflow tested end-to-end in non-production environment
    • Autonomy levels for each step reviewed and appropriate
    • Provenance fingerprint generated for audit trail (P21)

    For Operations¶

    • Monitoring dashboard shows active workflows
    • Pause/stop mechanisms work and are accessible
    • Error notifications reach correct teams
    • Rollback procedures tested for destructive workflows
    • Workflow execution metrics tracked (success rate, duration)
    • Provenance audit trail queryable for troubleshooting
    • Cross-repo coordination plans exist for breaking changes

    üéì Key Takeaways¶

    1. P15 is the conductor - It doesn't do the work, it coordinates others
    2. Workflows are declarative - YAML definitions are readable, testable, versionable
    3. Error handling is explicit - Every step declares fail/continue/rollback/alert
    4. Humans stay in control - C1 autonomy means monitoring and intervention always possible
    5. Cross-repo coordination - Impact analysis prevents surprises in distributed systems
    6. Provenance is automatic - Every workflow execution logged for audit (P21)
    7. Templates accelerate - Pre-defined workflows (new-feature, deploy, retro) save hours
    8. Composition scales - Complex processes emerge from simple pattern combinations
