    Chapter P12: UX, Copy & Accessibility Agent¶

    Pattern Code: P12 Short Code: UXA Name: UX, Copy & Accessibility Agent Type: DevTools Pattern Scope: User Experience, Accessibility, Bilingualism, GC Design System Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The UX, Copy & Accessibility Agent (P12-UXA) is the accessibility champion and bilingual guardian for the EVA Suite. It validates UX flows, enforces WCAG 2.2 AA compliance, reviews bilingual copy (EN-CA/FR-CA), and ensures alignment with the Government of Canada Design System. Think of it as your accessibility advocate, bilingual editor, and design system enforcer‚Äîall rolled into one.

    Without P12, accessibility bugs slip into production (excluding users with disabilities), French translations go missing (violating Official Languages Act), and UI components drift from GC Design System standards (inconsistent user experience). With P12, every UI component is accessible, every text string is bilingual, and every design token follows GC standards‚Äîmaking EVA Suite inclusive, compliant, and user-friendly for all Canadians.

    This chapter is critical for Government of Canada deployments where accessibility (WCAG 2.2 AA) and bilingualism (Official Languages Act) are legal requirements, not optional features.

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P12-UXA serves as the Accessibility Validator & Bilingual Compliance Enforcer for the EVA Autonomous Agile Command. It ensures:

    1. WCAG 2.2 AA Compliance: All UI components meet accessibility standards (keyboard nav, screen readers, contrast)
    2. Bilingual Completeness: Every user-facing string has EN-CA and FR-CA translations
    3. GC Design System Alignment: All UI uses GC design tokens, components, and patterns
    4. Inclusive UX: User flows work for all users (vision/hearing/motor/cognitive disabilities)
    5. Copy Quality: Clear, concise, plain language (not government jargon)
    6. Official Languages Act Compliance: Equal quality of French and English (not auto-translation)

    Mission Statement¶

    "Ensure EVA Suite is accessible to all Canadians‚Äîregardless of disability, language preference, or assistive technology‚Äîthrough automated validation, bilingual review, and GC Design System compliance enforcement."

    Value Proposition¶

    For Users with Disabilities: - Keyboard navigation works everywhere (no mouse required) - Screen readers announce content correctly (ARIA labels) - Color contrast meets WCAG standards (readable for low vision) - Focus indicators visible (know where you are)

    For French-Speaking Canadians: - Full French translation (not English-only) - Equal quality (not auto-translated gibberish) - GC-approved terminology (from TERMIUM Plus) - Language switcher on every page

    For Developers: - Automated accessibility testing (catch issues early) - Bilingual i18n helpers (no manual string extraction) - GC Design System components (pre-validated) - Clear remediation guidance (how to fix, not just "broken")

    For Auditors: - Automated WCAG compliance reports - Bilingual completeness metrics (% translated) - GC Design System compliance evidence - Audit trail of accessibility fixes

    üìã Responsibilities¶

    Core Responsibilities¶

    1. WCAG 2.2 AA Validation
    2. Run axe-core automated accessibility scans
    3. Test keyboard navigation (tab order, focus management)
    4. Validate screen reader compatibility (NVDA, JAWS, VoiceOver)
    5. Check color contrast (4.5:1 for normal text, 3:1 for large)
    6. Verify focus indicators (visible on all interactive elements)

    7. Test with zoom (200% without horizontal scroll)

    8. Bilingual Copy Review

    9. Extract all user-facing strings from UI
    10. Verify EN-CA translation exists
    11. Verify FR-CA translation exists
    12. Check consistency with GC terminology (TERMIUM Plus)
    13. Flag missing translations (incomplete i18n)

    14. Validate translation quality (not auto-translated)

    15. GC Design System Compliance

    16. Validate use of GC design tokens (CSS custom properties)
    17. Check for canada-gc theme class
    18. Verify GC header/footer components
    19. Validate bilingual language switcher
    20. Check AI content indicators (eva-gc-ai-content-indicator)

    21. Validate RAG citation cards (eva-gc-citation-card)

    22. UX Flow Validation

    23. Test user journeys end-to-end
    24. Validate error states (clear, actionable)
    25. Check form validation (inline, accessible)
    26. Test responsive behavior (mobile, tablet, desktop)
    27. Verify loading states (not just blank screens)

    Supporting Responsibilities¶

    1. Plain Language Review
    2. Flag government jargon (replace with plain language)
    3. Check reading level (Grade 8 or lower)
    4. Verify clear calls-to-action (not vague "click here")

    5. Test sentence length (15-20 words max)

    6. Inclusive Design Patterns

    7. Validate alternative text for images (meaningful, not "image")
    8. Check video captions (for deaf/hard-of-hearing)
    9. Test audio transcripts (for accessibility)

    10. Verify time-based content (pausing, extending)

    11. Compliance Reporting

    12. Generate WCAG compliance reports (automated + manual)
    13. Track bilingual completeness (% translated)
    14. Export GC Design System compliance metrics
    15. Produce accessibility audit evidence (for ATO)

    When NOT to Use P12¶

    • Don't use P12 for content writing - That's content strategists (humans)
    • Don't use P12 for design mockups - That's UX designers (Figma/Sketch)
    • Don't use P12 for user research - That's UX researchers (interviews/surveys)
    • Don't use P12 for A/B testing - That's analytics/experimentation platforms

    ‚ôø WCAG 2.2 AA Compliance (Mandatory for GC)¶

    What is WCAG 2.2 AA?¶

    WCAG = Web Content Accessibility Guidelines 2.2 = Latest version (published October 2023) AA = Conformance level (A = minimum, AA = standard, AAA = enhanced)

    Government of Canada Standard: WCAG 2.2 Level AA is MANDATORY for all public-facing websites and applications per Standard on Web Accessibility.

    WCAG 2.2 AA Checklist (All Mandatory for EVA Suite)¶

    Principle 1: Perceivable (Information and user interface components must be presentable to users in ways they can perceive)¶

    1.1 Text Alternatives¶
    • 1.1.1 Non-text Content (A): All images have alt text
    • ‚úÖ Meaningful images: Descriptive alt text ("Submit button", not "image")
    • ‚úÖ Decorative images: Empty alt (alt="") to skip in screen readers
    • ‚úÖ Complex images: Long descriptions or data tables
    • Test: Automated (axe-core), Manual (screen reader)
    1.2 Time-based Media¶
    • 1.2.1 Audio-only and Video-only (A): Transcripts for audio, descriptions for video
    • 1.2.2 Captions (A): Captions for all videos (not auto-generated, accurate)
    • 1.2.3 Audio Description or Media Alternative (A): Audio descriptions for videos
    • 1.2.4 Captions (Live) (AA): Live captions for webinars/live streams
    • 1.2.5 Audio Description (AA): Audio descriptions for all videos
    • Test: Manual review (video content)
    1.3 Adaptable¶
    • 1.3.1 Info and Relationships (A): Structure conveyed programmatically (headings, lists, tables)
    • ‚úÖ Headings: h1 ‚Üí h2 ‚Üí h3 (hierarchical, not visual)
    • ‚úÖ Lists: <ul>, <ol>, not DIVs styled as lists
    • ‚úÖ Tables: <th>, <caption>, scope attributes

    • Test: Automated (axe-core), Manual (DOM inspection)

    • 1.3.2 Meaningful Sequence (A): Reading order matches visual order

    • ‚úÖ Tab order logical (top‚Üíbottom, left‚Üíright)

    • Test: Manual (keyboard navigation)

    • 1.3.3 Sensory Characteristics (A): Instructions don't rely on shape/color/location alone

    • ‚ùå BAD: "Click the green button on the right"
    • ‚úÖ GOOD: "Click the Submit button (green, on the right)"

    • Test: Manual (content review)

    • 1.3.4 Orientation (AA): Content works in portrait and landscape

    • Test: Manual (rotate device/browser)

    • 1.3.5 Identify Input Purpose (AA): Autocomplete attributes on form fields

    • ‚úÖ autocomplete="email", autocomplete="given-name"
    • Test: Automated (axe-core)
    1.4 Distinguishable¶
    • 1.4.1 Use of Color (A): Color not the only way to convey info
    • ‚ùå BAD: Red text for errors (invisible to colorblind users)
    • ‚úÖ GOOD: Red text + icon + "Error:" prefix

    • Test: Manual (grayscale test)

    • 1.4.2 Audio Control (A): Auto-playing audio can be paused

    • Test: Manual (audio content)

    • 1.4.3 Contrast (Minimum) (AA): 4.5:1 for normal text, 3:1 for large text (18pt+)

    • ‚úÖ Body text: #26374A on #FFFFFF = 10.7:1 ‚úÖ
    • ‚úÖ Large headings: #26374A on #F5F5F5 = 9.2:1 ‚úÖ
    • Tool: WebAIM Contrast Checker

    • Test: Automated (axe-core)

    • 1.4.4 Resize Text (AA): Text can be zoomed to 200% without loss of content

    • ‚úÖ No horizontal scroll at 200% zoom
    • ‚úÖ Text not clipped or overlapping

    • Test: Manual (browser zoom)

    • 1.4.5 Images of Text (AA): Use real text, not images of text (unless logo)

    • Test: Manual (content review)

    • 1.4.10 Reflow (AA): Content reflows at 320px width (no 2D scrolling)

    • Test: Manual (responsive design)

    • 1.4.11 Non-text Contrast (AA): 3:1 contrast for UI components and graphics

    • ‚úÖ Buttons, form borders, icons: 3:1 minimum

    • Test: Automated (axe-core)

    • 1.4.12 Text Spacing (AA): Content doesn't break when text spacing increased

    • Test: Manual (CSS override: line-height 1.5, paragraph spacing 2x)

    • 1.4.13 Content on Hover or Focus (AA): Hover/focus content dismissible, hoverable, persistent

    • ‚úÖ Tooltips can be dismissed (Esc key)
    • ‚úÖ Tooltips hoverable (mouse can move to them)
    • ‚úÖ Tooltips persist until dismissed (not auto-hide on mouseover)
    • Test: Manual (interactive components)

    Principle 2: Operable (User interface components and navigation must be operable)¶

    2.1 Keyboard Accessible¶
    • 2.1.1 Keyboard (A): All functionality available via keyboard
    • ‚úÖ No mouse-only interactions
    • ‚úÖ Tab reaches all interactive elements
    • ‚úÖ Enter/Space activates buttons/links

    • Test: Manual (keyboard-only navigation)

    • 2.1.2 No Keyboard Trap (A): Keyboard focus never trapped

    • ‚úÖ Modal dialogs can be closed with Esc
    • ‚úÖ Tab cycles through modal, doesn't escape

    • Test: Manual (keyboard navigation)

    • 2.1.4 Character Key Shortcuts (A): Single-character shortcuts can be disabled/remapped

    • Test: Manual (if shortcuts implemented)
    2.2 Enough Time¶
    • 2.2.1 Timing Adjustable (A): Time limits can be extended (or turned off)
    • ‚úÖ Session timeout warnings (extend or log out)

    • Test: Manual (timed content)

    • 2.2.2 Pause, Stop, Hide (A): Auto-updating content can be paused

    • ‚úÖ Carousels have pause button
    • Test: Manual (animated content)
    2.3 Seizures and Physical Reactions¶
    • 2.3.1 Three Flashes or Below Threshold (A): No flashing >3 times/second
    • Test: Manual (video/animation review)
    2.4 Navigable¶
    • 2.4.1 Bypass Blocks (A): Skip navigation links
    • ‚úÖ "Skip to main content" link at top

    • Test: Manual (keyboard navigation)

    • 2.4.2 Page Titled (A): Unique, descriptive page titles

    • ‚úÖ <title>Dashboard - EVA Admin</title>

    • Test: Automated (axe-core)

    • 2.4.3 Focus Order (A): Focus order is logical

    • ‚úÖ Tab order matches reading order

    • Test: Manual (keyboard navigation)

    • 2.4.4 Link Purpose (In Context) (A): Link text describes destination

    • ‚ùå BAD: "Click here"
    • ‚úÖ GOOD: "View full report"

    • Test: Manual (content review)

    • 2.4.5 Multiple Ways (AA): Multiple ways to find pages (search, sitemap, navigation)

    • ‚úÖ Search bar on every page
    • ‚úÖ Main navigation menu

    • Test: Manual (site structure)

    • 2.4.6 Headings and Labels (AA): Descriptive headings and labels

    • ‚úÖ Form labels: "Email address" (not "Email")

    • Test: Manual (content review)

    • 2.4.7 Focus Visible (AA): Keyboard focus indicator visible

    • ‚úÖ Blue outline on focused elements (2px solid)

    • Test: Manual (keyboard navigation)

    • 2.4.11 Focus Not Obscured (Minimum) (AA - NEW in 2.2): Focused element not completely hidden

    • ‚úÖ Sticky headers don't cover focused elements
    • Test: Manual (keyboard navigation with sticky elements)
    2.5 Input Modalities¶
    • 2.5.1 Pointer Gestures (A): No multipoint or path-based gestures required
    • ‚úÖ Pinch-to-zoom has + / - buttons

    • Test: Manual (touch interactions)

    • 2.5.2 Pointer Cancellation (A): Click cancellable (mouseup, not mousedown)

    • Test: Manual (button interactions)

    • 2.5.3 Label in Name (A): Visible label matches accessible name

    • ‚úÖ Button text "Submit" matches aria-label="Submit"

    • Test: Automated (axe-core)

    • 2.5.4 Motion Actuation (A): Motion-triggered actions have alternative

    • ‚úÖ Shake-to-undo has undo button

    • Test: Manual (motion-based features)

    • 2.5.7 Dragging Movements (AA - NEW in 2.2): Drag-and-drop has single-pointer alternative

    • ‚úÖ File upload drag-drop has "Browse" button

    • Test: Manual (drag-drop features)

    • 2.5.8 Target Size (Minimum) (AA - NEW in 2.2): Interactive targets ‚â•24x24 CSS pixels

    • ‚úÖ Buttons, links: 44x44px (exceeds minimum)
    • Test: Manual (measure targets)

    Principle 3: Understandable (Information and operation of user interface must be understandable)¶

    3.1 Readable¶
    • 3.1.1 Language of Page (A): Page language identified (<html lang="en-CA">)
    • ‚úÖ English pages: lang="en-CA"
    • ‚úÖ French pages: lang="fr-CA"

    • Test: Automated (axe-core)

    • 3.1.2 Language of Parts (AA): Language changes identified

    • ‚úÖ French quote in English page: <span lang="fr-CA">Bonjour</span>
    • Test: Manual (multilingual content)
    3.2 Predictable¶
    • 3.2.1 On Focus (A): Focus doesn't trigger unexpected actions
    • ‚ùå BAD: Form submits on focus
    • ‚úÖ GOOD: Form submits on button click

    • Test: Manual (keyboard navigation)

    • 3.2.2 On Input (A): Input doesn't trigger unexpected actions

    • ‚ùå BAD: Selecting dropdown auto-submits
    • ‚úÖ GOOD: Selecting dropdown updates form, submit button explicit

    • Test: Manual (form interactions)

    • 3.2.3 Consistent Navigation (AA): Navigation order consistent across pages

    • ‚úÖ Main nav same order on all pages

    • Test: Manual (multi-page review)

    • 3.2.4 Consistent Identification (AA): Components with same function labeled consistently

    • ‚úÖ Search icon always labeled "Search" (not "Find" on some pages)

    • Test: Manual (component review)

    • 3.2.6 Consistent Help (A - NEW in 2.2): Help mechanism in consistent location

    • ‚úÖ Help link in top-right on all pages
    • Test: Manual (multi-page review)
    3.3 Input Assistance¶
    • 3.3.1 Error Identification (A): Errors identified and described
    • ‚úÖ "Email address is required" (not just red border)

    • Test: Manual (form validation)

    • 3.3.2 Labels or Instructions (A): Labels/instructions for user input

    • ‚úÖ Form fields have labels
    • ‚úÖ Required fields marked

    • Test: Automated (axe-core)

    • 3.3.3 Error Suggestion (AA): Suggestions provided for errors

    • ‚úÖ "Email format invalid. Example: name@example.com"

    • Test: Manual (form validation)

    • 3.3.4 Error Prevention (Legal, Financial, Data) (AA): Reversible, checked, or confirmed

    • ‚úÖ Delete confirmation: "Are you sure? This cannot be undone."

    • Test: Manual (critical actions)

    • 3.3.7 Redundant Entry (A - NEW in 2.2): Don't re-ask for same info

    • ‚úÖ Autocomplete pre-fills previously entered data

    • Test: Manual (multi-step forms)

    • 3.3.8 Accessible Authentication (Minimum) (AA - NEW in 2.2): No cognitive function test required

    • ‚ùå BAD: CAPTCHA ("type these distorted letters")
    • ‚úÖ GOOD: reCAPTCHA (invisible) or passkey/biometric
    • Test: Manual (authentication flows)

    Principle 4: Robust (Content must be robust enough to be interpreted by a wide variety of user agents, including assistive technologies)¶

    4.1 Compatible¶
    • 4.1.2 Name, Role, Value (A): Name, role, value for UI components
    • ‚úÖ Custom components have ARIA roles
    • ‚úÖ State changes announced (aria-pressed, aria-expanded)

    • Test: Automated (axe-core), Manual (screen reader)

    • 4.1.3 Status Messages (AA): Status messages announced without focus change

    • ‚úÖ Success toast: <div role="status" aria-live="polite">Saved!</div>
    • Test: Manual (screen reader)

    üåê Bilingualism (Official Languages Act Compliance)¶

    Legal Requirement¶

    Official Languages Act (1988): Federal institutions must provide services in English AND French of equal quality.

    Treasury Board Standard on Web Accessibility: All public-facing web content must be available in both official languages.

    EVA Suite Requirement: 100% bilingual (EN-CA/FR-CA) for all user-facing content‚Äîno exceptions.

    GC-Approved Terminology (TERMIUM Plus)¶

    TERMIUM Plus is the Government of Canada's official terminology database. Use it to validate translations.

    Common GC Terms¶

    What is the GC Design System?¶

    The Canada.ca Design System provides templates, patterns, and design principles for GC digital services. It ensures consistency across all Government of Canada websites and applications.

    EVA Suite Requirement: All UI components must use GC Design System tokens, components, and patterns.

    ‚úÖ End-of-Chapter Checklist¶

    For Developers (Building Accessible UIs)¶

    • I use GC Design System tokens (not hardcoded values)
    • All images have alt text (meaningful, not "image")
    • All form fields have labels
    • All interactive elements keyboard accessible
    • Color contrast meets WCAG 2.2 AA (4.5:1 minimum)
    • All text strings i18n-ready (EN-CA and FR-CA)
    • I test with keyboard only (no mouse)
    • I test with screen reader (NVDA or VoiceOver)

    For UX Designers (Designing Accessible Experiences)¶

    • All mockups use GC Design System colors/fonts
    • All copy provided in EN-CA and FR-CA
    • All user flows keyboard-navigable
    • Focus states visible in mockups
    • Error states clear and actionable
    • Alternative text provided for images
    • Reading order logical (not just visual)

    For Translators (Bilingual Content)¶

    • All translations professional (not auto-translate)
    • GC-approved terminology used (TERMIUM Plus)
    • Equal quality (FR not inferior to EN)
    • Cultural appropriateness (Canadian French, not European)
    • Consistent terminology across all files
    • No missing translation keys

    For QA Testers (Accessibility Validation)¶

    • axe-core scan passes (0 critical/high violations)
    • Keyboard navigation tested (Tab, Enter, Esc, Arrow keys)
    • Screen reader tested (NVDA, JAWS, or VoiceOver)
    • Zoom tested (200% without horizontal scroll)
    • Color contrast validated (WebAIM tool)
    • Bilingual completeness verified (100%)
    • GC Design System compliance checked
