    Chapter P11: Security & Compliance Helper¶

    Pattern Code: P11 Short Code: SEC Name: Security & Compliance Helper Type: DevTools Pattern Scope: Security, Compliance, GC Standards Maturity: Stable Autonomy Level: C1 (Human-on-the-loop)

    üìñ Chapter Overview¶

    The Security & Compliance Helper (P11-SEC) is the security guardian and compliance enforcer for the EVA Suite. It automates security scanning, vulnerability detection, secret detection, and compliance validation to ensure all products meet Government of Canada security standards, Protected B requirements, and industry best practices.

    Without P11, security vulnerabilities slip through code review, secrets leak into repositories, compliance gaps go undetected, and audit readiness becomes a manual nightmare. With P11, every commit is scanned, every deployment is validated, and compliance evidence is generated automatically‚Äîmaking security everyone's job, not just the security team's burden.

    This chapter is critical for Government of Canada deployments where security compliance is not optional‚Äîit's a precondition for ATO (Authority to Operate).

    üéØ Role Definition and Purpose¶

    Primary Role¶

    P11-SEC serves as the Security Scanner & Compliance Validator for the EVA Autonomous Agile Command. It ensures:

    1. Zero Tolerance for Secrets: No API keys, passwords, tokens, or credentials in source code
    2. Vulnerability-Free Dependencies: All npm/pip packages scanned for known CVEs
    3. Secure Code Practices: Static analysis detects insecure patterns (SQL injection, XSS)
    4. GC Security Compliance: All infrastructure follows GC Cloud Security Controls
    5. Protected B Readiness: Encryption, access controls, audit logging validated
    6. Audit Trail: Compliance evidence generated for ATO reviews

    Mission Statement¶

    "Enforce security best practices and Government of Canada compliance requirements through automated scanning, validation, and evidence generation‚Äîensuring EVA Suite is always audit-ready and breach-resistant."

    Value Proposition¶

    For Security Teams: - Automated security scans reduce manual review burden - Compliance evidence pre-generated for ATO submissions - Early detection of vulnerabilities (shift-left security) - Consistent enforcement across 40+ EVA Suite products

    For Developers: - Immediate feedback on security issues (pre-commit hooks) - Clear remediation guidance (not just "fix this") - No more "security team blocked my PR" surprises - Learn secure coding through automated feedback

    For Product Owners: - Accelerate ATO approval with compliance evidence - Reduce security incident risk (prevent vs react) - Demonstrate due diligence to auditors - Avoid costly post-deployment security retrofits

    For Auditors: - Automated compliance reports (weekly/monthly) - Immutable audit trail of security scans - Evidence of continuous compliance (not point-in-time) - Clear mapping to GC security controls

    üìã Responsibilities¶

    Core Responsibilities¶

    1. Secret Detection & Prevention
    2. Scan commits for hardcoded API keys, passwords, tokens
    3. Use TruffleHog, git-secrets, or custom regex patterns
    4. Block commits containing secrets (pre-commit hooks)

    5. Integrate with GitHub secret scanning alerts

    6. Dependency Vulnerability Scanning

    7. Run npm audit for Node.js dependencies
    8. Run pip-audit or Bandit for Python dependencies
    9. Integrate Snyk or Dependabot for continuous monitoring
    10. Track CVE severity (critical/high/moderate/low)

    11. Auto-generate dependency update PRs (Dependabot)

    12. Static Code Analysis

    13. Run Bandit (Python security linter)
    14. Run ESLint with security rules (Node.js)
    15. Detect SQL injection, XSS, insecure deserialization

    16. Flag use of deprecated/insecure APIs

    17. Infrastructure Security Validation

    18. Validate Azure Key Vault usage (no hardcoded secrets)
    19. Verify TLS 1.2+ enforcement (no TLS 1.0/1.1)
    20. Confirm private endpoints only (no public IPs)

    21. Check RBAC configuration (principle of least privilege)

    22. GC Security Controls Compliance

    23. Map EVA infrastructure to GC Cloud Security Controls
    24. Validate Protected B encryption requirements (AES-256)
    25. Verify data residency (Canada Central + East only)
    26. Confirm audit logging (Application Insights, Cosmos DB)
    27. Track key rotation (90-day CMK rotation)

    Supporting Responsibilities¶

    1. Compliance Reporting
    2. Generate weekly security scan summaries
    3. Produce ATO evidence packages (PDF/markdown)
    4. Track compliance gaps and remediation status

    5. Export compliance data for GRC tools

    6. Security Training & Education

    7. Flag insecure patterns with educational links
    8. Maintain security best practices documentation
    9. Provide secure coding examples (do's and don'ts)

    10. Track common vulnerability patterns (trend analysis)

    11. Incident Response Support

    12. Rapid vulnerability assessment during incidents
    13. Secret rotation guidance (compromised credentials)
    14. Breach impact analysis (what data was exposed?)
    15. Post-incident compliance validation

    When NOT to Use P11¶

    • Don't use P11 for penetration testing - That's manual security testing (external specialists)
    • Don't use P11 for runtime security - Use Azure Defender/Sentinel for runtime threats
    • Don't use P11 for security architecture review - That's P02 (Requirements) + Security SME
    • Don't use P11 for credential management - Use Azure Key Vault directly

    üèõÔ∏è Government of Canada Security Controls¶

    Protected B Compliance Requirements¶

    Protected B is a security classification for Government of Canada information that requires: - Confidentiality (unauthorized access could cause serious injury) - Integrity (unauthorized modification prevented) - Availability (authorized users can access when needed)

    Protected B Controls (Mandatory for EVA Suite)¶

    Control ID Name Requirement EVA Implementation P11 Validation
    SC-8 Transmission Confidentiality TLS 1.2+ for all network traffic Azure APIM enforces TLS 1.2 min ‚úÖ Scan APIM policies
    SC-13 Cryptographic Protection AES-256 encryption at rest Azure Storage SSE + CMK ‚úÖ Verify Key Vault CMKs
    AC-2 Account Management Centralized authentication Azure AD SSO + MFA ‚úÖ Check RBAC policies
    AC-3 Access Enforcement Least privilege (RBAC) Cosmos DB + API RBAC ‚úÖ Audit RBAC assignments

    GC Cloud Security Controls (CCCS ITSG-33)¶

    EVA Suite follows ITSG-33 Medium Confidentiality controls (aligned with Protected B):

    Access Control (AC)¶

    • AC-2: Account Management ‚Üí Azure AD managed identities
    • AC-3: Access Enforcement ‚Üí RBAC at API/DB/Search layers
    • AC-4: Information Flow Enforcement ‚Üí Private Link, NSGs

    Audit and Accountability (AU)¶

    • AU-2: Audit Events ‚Üí Application Insights (all API calls)

    • AU-6: Audit Review ‚Üí P14 (LiveOps) automated anomaly detection

    • SC-12: Cryptographic Key Management ‚Üí Azure Key Vault
    • SC-13: Cryptographic Protection ‚Üí AES-256 at rest

    üí° Sample Prompts¶

    System Prompt (For AI Agents)¶

    You are P11-SEC (Security & Compliance Helper), the security guardian for EVA DevTools.

**Your Capabilities**:
1. Scan code for hardcoded secrets (API keys, passwords, tokens)
2. Run dependency vulnerability scans (npm audit, Bandit, Snyk)
3. Perform static code analysis (ESLint security rules, Bandit)
4. Validate GC security controls compliance (ITSG-33, Protected B)
5. Generate ATO compliance evidence reports
6. Provide remediation guidance for detected issues

**Your Limitations**:
- You do NOT perform penetration testing (that's manual security experts)
- You do NOT manage Azure Key Vault (that's infrastructure team)
- You do NOT set security policies (that's Security SME + compliance team)
- You do NOT rotate credentials on your own (humans approve rotation)

**Autonomy Level**: C1 (Human-on-the-loop)
- Automatically scan code and flag issues
- Block deployments with critical vulnerabilities
- Humans review and remediate flagged issues

**Output Format**:
- Use severity levels: üö® CRITICAL, ‚ö†Ô∏è WARNING, ‚ÑπÔ∏è INFO
- Include remediation steps, not just "fix this"
- Link to GC security control IDs (e.g., SC-8, AC-2)
- Provide secure code examples (before/after)

**Safety Rules**:
- NEVER output secrets/credentials in logs or reports
- NEVER auto-rotate credentials without human approval
- NEVER suppress critical vulnerabilities without Security SME sign-off
- ALWAYS validate compliance against GC standards, not just industry best practices

  schedule:
    - cron: '0 2 * * *'  # 2 AM UTC

    Example 1: Secret Detection¶

    When P11 detects hardcoded secret:

    --- Awareness (P16 ‚Äì AWR) ---
pod(s): All PODs
phase: 2 (Development)
confidence: High
assumptions:
  - "Pattern match indicates actual secret (not example/placeholder)"
  - "Secret is active and not already rotated"
  - "Developer has access to Azure Key Vault for secure storage"
risks:
  - "Secret may have already been committed to remote (rotate immediately)"
  - "Secret may be in use by other services (coordinate rotation)"
  - "False positive (e.g., test data) wastes remediation effort"
actionClass: C1 (human-on-loop: secret detected, developer must rotate)
suggestedNextAction:
  - "DO NOT COMMIT - Remove secret from code immediately"
  - "Rotate secret in source system (invalidate old key)"
  - "Store secret in Azure Key Vault"
  - "Update code to fetch from Key Vault via managed identity"
  - "Run git history scan to check if already committed (BFG Repo-Cleaner)"
evidenceRefs:
  - "tools/security/secret-patterns.yaml"
  - "docs/runbooks/secret-rotation.md"

    Example 2: Dependency Vulnerability¶

    When P11 detects vulnerable dependency:

    --- Awareness (P16 ‚Äì AWR) ---
pod(s): POD-F (Foundation), affected service POD
phase: 2 (Development) or 6 (Day-2 Operations)
confidence: High
assumptions:
  - "CVE severity accurately reflects actual risk (context-dependent)"
  - "Update to patched version does not introduce breaking changes"
  - "Vulnerability is exploitable in EVA Suite context (not all CVEs are)"
risks:
  - "Update may break existing functionality (regression risk)"
  - "No patch available yet (zero-day or unpatched)"
  - "Dependency is transitive (not directly controlled by EVA)"
actionClass: C1 (human-on-loop: vulnerability detected, developer reviews and patches)
suggestedNextAction:
  - "Review CVE details (exploitability, impact, availability of patch)"
  - "If critical: Update immediately, test, deploy"
  - "If high: Update within 7 days (next sprint)"
  - "If moderate/low: Schedule in backlog, update opportunistically"
  - "Test updated dependency in sandbox before production"
  - "If no patch: Implement workaround (e.g., input validation, rate limiting)"
evidenceRefs:
  - "npm-audit-results.json or bandit-report.json"
  - "https://cve.mitre.org/cgi-bin/cvename.cgi?name={CVE-ID}"
  - "docs/runbooks/dependency-updates.md"

    # Runbook: Secret Rotation

**Purpose**: Rotate compromised or expired secrets securely
**Triggered By**: P11 secret detection, scheduled rotation (90 days), suspected breach
**Owner**: POD-F + Security Team

---

## Pre-Requisites

- [ ] Access to Azure Key Vault (Contributor role)
- [ ] Access to source system (API provider, database)
- [ ] List of services consuming the secret

---

## Steps

### 1. Generate New Secret

```powershell
# Example: Rotate Azure Storage account key
$storageAccountName = "evastorageaccount"
$resourceGroup = "rg-eva-prod"

# Rotate key2 first (key1 still active)
az storage account keys renew \
  --account-name $storageAccountName \
  --resource-group $resourceGroup \
  --key key2

    2. Update Key Vault¶

    # Store new key in Key Vault
$newKey = az storage account keys list \
  --account-name $storageAccountName \
  --resource-group $resourceGroup \
  --query "[1].value" -o tsv

az keyvault secret set \
  --vault-name "eva-key-vault" \
  --name "storage-account-key" \
  --value $newKey

    3. Verify Services Pick Up New Key¶

    # Services using managed identity auto-fetch from Key Vault
# Verify no errors in Application Insights

az monitor app-insights query \
  --app "eva-app-insights" \
  --analytics-query "exceptions | where timestamp > ago(10m)" \
  --output table

    4. Invalidate Old Secret¶

    # After verifying new key works, rotate key1
az storage account keys renew \
  --account-name $storageAccountName \
  --resource-group $resourceGroup \
  --key key1

    5. Update Documentation¶

    • Update secret rotation log: security/rotation-log.md
    • Record rotation date in Key Vault tags
    • Schedule next rotation (90 days)

    Rollback Plan¶

    If new secret causes service failures:

    1. Restore old secret to Key Vault (from backup)
    2. Wait 5 minutes for services to pick up old secret
    3. Investigate root cause before re-attempting rotation

    Post-Rotation Validation¶

    • No errors in Application Insights (last 1 hour)
    • All services healthy (health check endpoints)
    • Old secret invalidated in source system
    • New secret stored in Key Vault (verify with az keyvault secret show)

    Last Updated: 2025-12-12 Related: P11 (Security & Compliance Helper) ```

    ‚ö†Ô∏è Safety Considerations (C0-C3)¶

    Autonomy Level: C1 (Human-on-the-Loop)¶

    P11 Itself: C1 - Automatically scans code and flags issues - Blocks deployments with critical vulnerabilities - Humans review flagged issues and approve remediations

    Why C1? - Security scanning is low-risk (read-only detection) - Blocking deployments prevents bad code from reaching production - Remediation requires human judgment (context-dependent)

    Safety Rules¶

    Rule 1: Never Auto-Rotate Secrets¶

    Risk: Automated rotation breaks services if not coordinated

    Mitigation: - ‚úÖ P11 DETECTS secrets, but NEVER rotates them automatically - ‚úÖ Humans must approve and coordinate secret rotation - ‚úÖ Follow runbook: generate new ‚Üí update services ‚Üí invalidate old - ‚úÖ Test rotation in Sandbox before Production

    Rule 2: Never Suppress Critical Vulnerabilities¶

    Risk: Ignoring critical CVEs leads to exploits

    Mitigation: - ‚úÖ Critical vulnerabilities MUST be remediated before Production - ‚úÖ Only Security SME can approve exceptions (with documented risk acceptance) - ‚úÖ Track all exceptions in compliance register - ‚úÖ Re-assess exceptions quarterly (risk may change)

    Rule 3: Protect Secrets in Logs/Reports¶

    Risk: Logging secrets defeats the purpose of detection

    Mitigation: - ‚úÖ P11 NEVER outputs detected secrets in logs (redact with ***) - ‚úÖ Secret scan results stored securely (encrypted at rest) - ‚úÖ Access to scan results restricted (RBAC) - ‚úÖ Scan artifacts deleted after 90 days (except compliance evidence)

    Rule 4: Validate Compliance Evidence¶

    Risk: Stale/inaccurate evidence fails ATO audit

    Mitigation: - ‚úÖ Compliance evidence re-generated quarterly (not copy/paste) - ‚úÖ Screenshots include timestamps (verifiable freshness) - ‚úÖ Azure configuration validated against live state (no drift) - ‚úÖ Security SME signs off on all compliance reports

    Autonomy Level Definitions (Reference)¶

    Level Name Description P11 Usage
    C0 Human-in-the-loop AI suggests, human approves every action Secret rotation, vulnerability exceptions
    C1 Human-on-the-loop AI acts, human monitors Security scanning, deployment blocking
    C2 Human-over-the-loop AI acts with constraints, periodic review Not applicable (P11 requires tight oversight)
    C3 Full autonomy AI acts autonomously within boundaries Not applicable (security is too critical)

    ‚úÖ End-of-Chapter Checklist¶

    For Developers (Using P11)¶

    • I run P11 pre-commit scan before every commit
    • I store all secrets in Azure Key Vault (never hardcode)
    • I update dependencies when P11 flags vulnerabilities
    • I understand severity levels (critical vs moderate vs low)
    • I follow remediation guidance (not just "fix it")
    • I test fixes in Sandbox before Production
    • I document any security exceptions (with Security SME approval)

    For Security Teams (Managing P11)¶

    • P11 scans run automatically (CI/CD + scheduled)
    • Secret detection patterns are up-to-date (TruffleHog rules)
    • Vulnerability thresholds configured (critical/high blocked)
    • GC security controls checklist validated quarterly
    • Compliance evidence archived for 7 years (audit trail)
    • RBAC configured for scan result access (least privilege)
    • Security SME reviews all P11 findings weekly

    For Governance (Periodic Audit)¶

    • All critical vulnerabilities remediated (no exceptions without approval)
    • No secrets detected in any repository (last 30 days)
    • GC security controls compliance validated (quarterly)
    • Protected B encryption requirements met (AES-256, CMK rotation)
    • Audit logging enabled and retained (7 years)
    • ATO compliance evidence current (<90 days old)
    • Security training completed by all developers (annual)

    üéì Key Takeaways¶

    1. Security is everyone's job - P11 democratizes security through automation
    2. Shift-left security - Catch issues before they reach production
    3. Compliance is continuous - Not a one-time ATO checkbox
    4. Secrets in code = breach - Always use Azure Key Vault
    5. Critical CVEs = immediate action - No exceptions without Security SME
    6. GC standards are non-negotiable - Protected B is table stakes for GoC
    7. Evidence is perishable - Quarterly refresh for ATO compliance
    8. Automation scales - Manual security review doesn't scale to 40+ products
