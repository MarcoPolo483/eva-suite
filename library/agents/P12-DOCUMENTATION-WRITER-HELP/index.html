    Agent Service Catalog - P12 (HELP)¶

    Agent ID: P12 Agent Name: Documentation Writer POD: POD-D (Delivery) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P12 is the Documentation Writer agent responsible for technical documentation, user guides, API documentation, runbooks, and ensuring comprehensive, up-to-date documentation across the EVA Suite.

    Core Services¶

    1. Technical Documentation¶

    • Description: Write comprehensive technical documentation for developers and architects
    • Input: Code, architecture, technical decisions, target audience
    • Output: Technical docs (README, design docs, ADRs), developer guides
    • Example: P12: Document eva-api architecture (components, data flow, Azure services)

    Input Template: 

    P12 REQUEST: Technical Documentation
  TOPIC: [Documentation subject]
  SCOPE:
    - [Area 1, e.g., "Architecture overview"]
    - [Area 2, e.g., "Component details"]
    - [Area 3, e.g., "Integration patterns"]
  AUDIENCE: [Developers | Architects | Operators]
  SOURCES:
    - [Source 1, e.g., "P01 architecture docs"]
    - [Source 2, e.g., "Codebase"]
  FORMAT: [Markdown | Wiki | MkDocs]
  DELIVERABLES: [Docs location and structure]

    2. User Guides & Tutorials¶

    • Description: Create user-facing documentation and step-by-step tutorials
    • Input: Feature, user personas, common tasks, screenshots
    • Output: User guides, tutorials, how-to articles
    • Example: P12: Write user guide for eva-ui document upload (with screenshots, troubleshooting)

    Input Template: 

    P12 REQUEST: User Guide
  FEATURE: [Feature name]
  USER_PERSONAS:
    - [Persona 1, e.g., "Government analyst"]
    - [Persona 2, e.g., "System administrator"]
  TASKS:
    - [Task 1, e.g., "Upload a document"]
    - [Task 2, e.g., "Query documents"]
  SCREENSHOTS: [Required? Yes/No]
  TROUBLESHOOTING: [Common issues to document]
  FORMAT: [User manual | Video tutorial | Interactive guide]

    3. API Documentation¶

    • Description: Generate and maintain API documentation from OpenAPI/GraphQL specs
    • Input: OpenAPI spec, GraphQL schema, code examples
    • Output: API reference docs, interactive API explorer
    • Example: P12: Generate API docs for eva-api v1 (OpenAPI â†’ Swagger UI + Markdown)

    Input Template: 

    P12 REQUEST: API Documentation
  API_SPEC: [Path to OpenAPI/GraphQL spec]
  SCOPE:
    - [Scope 1, e.g., "All endpoints"]
    - [Scope 2, e.g., "Authentication guide"]
  CODE_EXAMPLES:
    - [Language 1, e.g., "Python"]
    - [Language 2, e.g., "JavaScript"]
  INTERACTIVE: [Swagger UI | GraphQL Playground]
  HOSTING: [docs.eva-suite.com | GitHub Pages]

    4. Runbooks & Operational Docs¶

    • Description: Create operational documentation for incidents, deployments, troubleshooting
    • Input: Operational procedures, common incidents, escalation paths
    • Output: Runbooks, troubleshooting guides, escalation docs
    • Example: P12: Create runbook for eva-api deployment rollback (steps, validation, contacts)

    Input Template: 

    P12 REQUEST: Runbook
  PROCEDURE: [Procedure name]
  TRIGGER: [When this is needed]
  STEPS:
    - [Step 1, e.g., "Check health endpoint"]
    - [Step 2, e.g., "Swap deployment slot"]
  VALIDATION:
    - [Validation 1, e.g., "Health check returns 200"]
  ROLLBACK:
    - [Rollback steps if procedure fails]
  CONTACTS: [Escalation contacts]
  FREQUENCY: [How often used, e.g., "Every deployment"]

    5. Documentation Maintenance & Updates¶

    • Description: Keep documentation up-to-date with code/feature changes
    • Input: Code changes, feature releases, deprecated features
    • Output: Updated documentation, changelog, deprecation notices
    • Example: P12: Update eva-api docs for v2 release (breaking changes, migration guide)

    Input Template: 

    P12 REQUEST: Documentation Update
  CHANGES:
    - [Change 1, e.g., "API v2 released"]
    - [Change 2, e.g., "New query endpoint"]
    - [Change 3, e.g., "Deprecated /v1/old-endpoint"]
  AFFECTED_DOCS:
    - [Doc 1, e.g., "API reference"]
    - [Doc 2, e.g., "User guide"]
  MIGRATION_GUIDE: [Required? Yes/No]
  CHANGELOG: [Update CHANGELOG.md]
  DEPRECATION_NOTICES: [Where to add notices]

    Example Requests¶

    Example 1: Technical Documentation¶

    P12: Document eva-api architecture
- Scope: Components, data flow, Azure services, security
- Audience: Developers and architects
- Sources: P01 architecture docs, codebase, deployment configs
- Format: Markdown in docs/architecture/
- Deliverables: README.md, architecture.md, ADRs

    Example 2: User Guide¶

    P12 REQUEST:
  TASK: Write user guide for document upload
  FEATURE: eva-ui document upload interface
  PERSONAS: Government analysts, researchers
  TASKS: Upload single file, bulk upload, check status
  SCREENSHOTS: Required (upload flow, progress bar)
  TROUBLESHOOTING: File format errors, size limits
  FORMAT: User manual with images

    Example 3: API Documentation¶

    P12: Generate API docs for eva-api v1
- Source: P08-OPENAPI-SPEC-EVA-API-V1.yaml
- Scope: All endpoints, authentication, rate limiting
- Examples: Python and JavaScript code samples
- Interactive: Swagger UI at docs.eva-suite.com/api
- Hosting: GitHub Pages with MkDocs

    Service Dependencies¶

    Upstream (P12 depends on): - P01 (System Architect): Architecture decisions, ADRs - P08 (API Designer): OpenAPI/GraphQL specs - P05 (UX Designer): User flows, UI designs - All agents: Subject matter expertise

    Downstream (Depends on P12): - All users: Developers, operators, end-users - P09 (QA Engineer): Documentation validation - PO (Product Owner): Documentation approval

    Service Standards¶

    Documentation Standards¶

    • Markdown format (.md) for version control
    • Clear structure (headings, bullets, code blocks)
    • Code examples tested and working
    • Screenshots current (update with UI changes)
    • Links valid (no broken links)
    • Grammar/spelling checked (Grammarly, LanguageTool)

    Technical Documentation Standards¶

    • README.md in every repo (purpose, setup, usage)
    • Architecture docs in docs/architecture/
    • ADRs for major decisions (P01 collaboration)
    • Code comments for complex logic
    • Inline documentation (docstrings, JSDoc)

    User Documentation Standards¶

    • Task-oriented (how to accomplish X)
    • Step-by-step instructions with visuals
    • Troubleshooting section for common issues
    • Glossary for technical terms
    • Contact info for support

    API Documentation Standards¶

    • Generated from OpenAPI/GraphQL specs
    • Code examples in Python and JavaScript
    • Authentication guide clear
    • Error codes documented
    • Rate limits and quotas explained
    • Interactive explorer (Swagger UI, GraphQL Playground)

    ### Pattern 2: Runbook Template
```markdown
# Runbook: [Procedure Name]

**Owner**: P11 (DevOps)
**Last Updated**: 2025-12-09
**Frequency**: [How often this is used]

## Purpose
[Why this procedure exists]

## Trigger
[When to execute this procedure]

## Prerequisites
- [ ] Access to Azure Portal
- [ ] P09 sign-off received
- [ ] Backup of current version

## Procedure

### Step 1: Pre-Deployment Validation
1. Check health endpoint: `curl https://api.eva-suite.com/health`
2. Expected: `200 OK`
3. If not healthy, escalate to P11

### Step 2: Deploy to Staging Slot
```bash
az webapp deployment slot swap \
  --resource-group eva-rg \
  --name eva-api \
  --slot staging \
  --target-slot production

    Step 3: Post-Deployment Validation¶

    1. Check health endpoint (production)
    2. Run smoke tests: pytest tests/smoke/
    3. Check Application Insights for errors

    Validation Criteria¶

    • Health check returns 200
    • Smoke tests pass
    • Error rate <1% (first 15 minutes)

    Rollback Procedure¶

    If validation fails: 1. Swap slots back: az webapp deployment slot swap ... 2. Verify previous version healthy 3. Create incident: [Template link]
