    Agent Service Catalog - P08 (HELP)¶

    Agent ID: P08 Agent Name: API Designer POD: POD-E (Experience) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P08 is the API Designer agent responsible for REST/GraphQL API design, OpenAPI specifications, API versioning, developer experience, and ensuring consistent, intuitive APIs across the EVA Suite.

    Core Services¶

    1. REST API Design¶

    • Description: Design RESTful APIs with clear resources, methods, and responses
    • Input: Feature requirements, data models, client needs
    • Output: API specification (OpenAPI 3.1), endpoint documentation, examples
    • Example: P08: Design REST API for document management (upload, query, delete)

    Input Template: 

    P08 REQUEST: REST API Design
  FEATURE: [Feature name]
  RESOURCES:
    - [Resource 1, e.g., "Documents"]
    - [Resource 2, e.g., "Queries"]
  OPERATIONS:
    - resource: [Resource name]
      methods: [GET, POST, PUT, DELETE]
      use_cases:
        - [Use case 1]
        - [Use case 2]
  AUTHENTICATION: [Bearer token | API key | OAuth2]
  VERSIONING: [v1 | v2]
  DELIVERABLES: [OpenAPI spec | Postman collection | Both]

    2. GraphQL Schema Design¶

    • Description: Design GraphQL schemas with types, queries, mutations
    • Input: Data model, query patterns, performance requirements
    • Output: GraphQL schema (SDL), resolver design, documentation
    • Example: P08: Design GraphQL schema for document analytics (queries for insights, aggregations)

    Input Template: 

    P08 REQUEST: GraphQL Schema
  FEATURE: [Feature name]
  TYPES:
    - [Type 1, e.g., "Document { id, title, content, ... }"]
    - [Type 2, e.g., "Query { document(id), documents(filter) }"]
  QUERIES:
    - [Query 1, e.g., "document(id: ID!): Document"]
    - [Query 2, e.g., "documents(filter: Filter): [Document]"]
  MUTATIONS:
    - [Mutation 1, e.g., "uploadDocument(file: Upload!): Document"]
  SUBSCRIPTIONS: [Real-time updates needed? Yes/No]
  PAGINATION: [Cursor-based | Offset-based]

    3. API Versioning Strategy¶

    • Description: Design versioning strategy to evolve APIs without breaking clients
    • Input: Current API, breaking changes needed, client usage
    • Output: Versioning strategy, migration plan, deprecation policy
    • Example: P08: Design v2 API for eva-api with breaking changes to query response format

    Input Template: 

    P08 REQUEST: API Versioning
  CURRENT_VERSION: [e.g., "v1"]
  BREAKING_CHANGES:
    - [Change 1, e.g., "Query response now includes metadata"]
    - [Change 2, e.g., "Renamed field 'doc_id' to 'document_id'"]
  VERSIONING_STRATEGY: [URL path /v2 | Header | Query param]
  MIGRATION_PLAN:
    - [Step 1, e.g., "Release v2 alongside v1"]
    - [Step 2, e.g., "Deprecate v1 in 6 months"]
  DEPRECATION_POLICY: [How long to support old versions]

    4. Developer Experience (DX) Optimization¶

    • Description: Improve API usability for developers (docs, SDKs, error messages)
    • Input: API usage patterns, developer feedback, pain points
    • Output: Improved docs, SDK design, error handling guide
    • Example: P08: Improve eva-api error messages (add error codes, troubleshooting tips)

    Input Template: 

    P08 REQUEST: DX Optimization
  PAIN_POINTS:
    - [Pain 1, e.g., "Error messages too vague"]
    - [Pain 2, e.g., "No code examples in docs"]
  IMPROVEMENTS:
    - [Improvement 1, e.g., "Structured error codes"]
    - [Improvement 2, e.g., "SDK for Python/JavaScript"]
  TARGET_AUDIENCE: [Frontend devs | Backend devs | Both]
  DELIVERABLES:
    - [Deliverable 1, e.g., "Enhanced OpenAPI spec"]
    - [Deliverable 2, e.g., "SDK design doc"]

    5. API Security & Rate Limiting¶

    • Description: Design authentication, authorization, and rate limiting
    • Input: Security requirements, expected traffic, abuse scenarios
    • Output: Auth strategy, rate limit policy, security best practices
    • Example: P08: Design rate limiting for eva-api (1000 req/hour per user, burst to 50 req/min)

    Input Template: 

    P08 REQUEST: API Security
  AUTHENTICATION: [Bearer token | API key | OAuth2]
  AUTHORIZATION:
    - [Policy 1, e.g., "Users can only access own documents"]
    - [Policy 2, e.g., "Admins can access all documents"]
  RATE_LIMITING:
    - tier: [Free | Premium | Enterprise]
      limits: [e.g., "1000 req/hour, 50 req/min burst"]
  ABUSE_SCENARIOS:
    - [Scenario 1, e.g., "Brute force query attempts"]
  SECURITY_STANDARDS: [OWASP API Security Top 10]

    Example Requests¶

    Example 1: REST API¶

    P08: Design REST API for document management
- Resources: Documents, Queries, Users
- Endpoints: POST /documents (upload), GET /documents/{id}, POST /queries
- Auth: Bearer token (Azure AD)
- Versioning: /api/v1
- Deliverables: OpenAPI 3.1 spec, Postman collection

    Example 2: GraphQL Schema¶

    P08 REQUEST:
  TASK: Design GraphQL schema for analytics
  TYPES: Document, Query, Insight, Metric
  QUERIES: documents(filter), insights(docId), metrics(period)
  MUTATIONS: uploadDocument(file), deleteDocument(id)
  PAGINATION: Cursor-based (Relay spec)
  DELIVERABLES: SDL schema, resolver design, docs

    Example 3: API Versioning¶

    P08: Plan v2 API migration for eva-api
- Breaking changes: Query response format, field renames
- Strategy: URL path versioning (/v1, /v2)
- Migration: v2 release Dec 2025, v1 deprecated June 2026
- Support: v1 supported until Dec 2026 (12 months)

    Service Dependencies¶

    Upstream (P08 depends on): - P02 (Data Architect): Data models, database schema - P01 (System Architect): Integration patterns, architecture - P03 (Security Architect): Auth/authz requirements - PO (Product Owner): Feature requirements, client needs

    Downstream (Depends on P08): - P04 (Library Manager): SDK implementation - P07 (Testing Engineer): API testing (contract tests) - P12 (Documentation Writer): API documentation - All client applications: Frontend, integrations

    Service Standards¶

    REST API Standards¶

    • RESTful design (resources, HTTP methods, status codes)
    • OpenAPI 3.1 specification (machine-readable)
    • Consistent naming: snake_case for JSON fields
    • Versioning: URL path (/api/v1)
    • Pagination: Cursor-based for large datasets
    • Error responses: Structured (code, message, details)

    GraphQL Standards¶

    • Schema-first design (SDL before resolvers)
    • Relay-compliant pagination (cursor-based)
    • Naming: camelCase for fields
    • Error handling: Use GraphQL errors array
    • Deprecation: @deprecated directive for fields
    • N+1 query prevention: DataLoader pattern

    Developer Experience Standards¶

    • Complete OpenAPI spec (all endpoints, examples)
    • Code examples for common use cases
    • Error codes documented with troubleshooting
    • SDKs for Python and JavaScript
    • Interactive API explorer (Swagger UI, GraphQL Playground)
    • Changelog for API versions

    Security Standards¶

    • HTTPS only (TLS 1.3)
    • Authentication on all endpoints (except health checks)
    • Authorization checks before data access
    • Rate limiting (per user, per endpoint)
    • Input validation (reject malformed requests)
    • CORS policy (whitelist allowed origins)

    Common Patterns¶

    Pattern 1: OpenAPI Specification¶

    openapi: 3.1.0
info:
  title: EVA Document API
  version: 1.0.0
  description: API for document upload, query, and management

servers:
  - url: https://api.eva-suite.com/v1
    description: Production

paths:
  /documents:
    post:
      summary: Upload a document
      operationId: uploadDocument
      security:
        - bearerAuth: []
      requestBody:
        required: true
        content:
          multipart/form-data:
            schema:
              type: object
              properties:
                file:
                  type: string
                  format: binary
      responses:
        '201':
          description: Document uploaded successfully
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Document'
        '400':
          $ref: '#/components/responses/BadRequest'
        '401':
          $ref: '#/components/responses/Unauthorized'

components:
  schemas:
    Document:
      type: object
      properties:
        id:
          type: string
          format: uuid
        title:
          type: string
        uploaded_at:
          type: string
          format: date-time

  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT

    Pattern 2: GraphQL Schema (SDL)¶

    type Query {
  document(id: ID!): Document
  documents(
    filter: DocumentFilter
    first: Int
    after: String
  ): DocumentConnection!
}

type Mutation {
  uploadDocument(input: UploadDocumentInput!): UploadDocumentPayload!
  deleteDocument(id: ID!): DeleteDocumentPayload!
}

type Document {
  id: ID!
  title: String!
  content: String!
  uploadedAt: DateTime!
  createdBy: User!
}

type DocumentConnection {
  edges: [DocumentEdge!]!
  pageInfo: PageInfo!
  totalCount: Int!
}

type DocumentEdge {
  node: Document!
  cursor: String!
}

input DocumentFilter {
  title: String
  uploadedAfter: DateTime
}

    Pattern 3: Error Response Format¶

    {
  "error": {
    "code": "INVALID_DOCUMENT_FORMAT",
    "message": "Document format not supported. Supported formats: PDF, DOCX, TXT",
    "details": {
      "field": "file",
      "uploaded_format": "exe",
      "supported_formats": ["pdf", "docx", "txt"]
    },
    "timestamp": "2025-12-09T12:34:56Z",
    "trace_id": "abc123-def456",
    "help_url": "https://docs.eva-suite.com/errors/INVALID_DOCUMENT_FORMAT"
  }
}

    Versioning & Migration Plan¶

    File Naming: 

    P08-API-MIGRATION-[VERSION]-[YYYYMMDD-HHMMSS].md

    File Naming Convention¶

    All deliverables must follow: 

    P08-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].[ext]

    Examples: - P08-OPENAPI-SPEC-EVA-API-V1-20251209-120000.yaml - P08-GRAPHQL-SCHEMA-ANALYTICS-V1-20251209-130000.graphql - P08-API-DESIGN-DOCUMENT-MGMT-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P08- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Extension: .yaml, .graphql, .md as appropriate 5. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md

    Quality Checklist¶

    Before marking P08 work complete:

    • API follows RESTful or GraphQL best practices
    • OpenAPI/GraphQL spec complete and valid
    • All endpoints documented with examples
    • Error responses structured and documented
    • Authentication/authorization designed
    • Rate limiting strategy defined
    • Versioning strategy clear
    • Breaking changes migration plan ready
    • P04 reviewed for SDK feasibility
    • P03 reviewed for security compliance
