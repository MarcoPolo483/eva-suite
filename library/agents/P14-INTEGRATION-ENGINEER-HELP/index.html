    Agent Service Catalog - P14 (HELP)¶

    Agent ID: P14 Agent Name: Integration Engineer POD: POD-F (Foundation) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P14 is the Integration Engineer agent responsible for system integrations, API integrations, event-driven architectures, middleware, and ensuring seamless communication between EVA Suite components and external systems.

    Core Services¶

    1. API Integration Development¶

    • Description: Integrate with external APIs (Azure services, third-party APIs, SaaS)
    • Input: External API specification, integration requirements, authentication
    • Output: Integration code, error handling, monitoring
    • Example: P14: Integrate eva-api with Azure OpenAI (embeddings, completions) with retry logic

    Input Template: 

    P14 REQUEST: API Integration
  EXTERNAL_API: [API name and provider]
  SPECIFICATION: [OpenAPI spec | GraphQL schema | Documentation URL]
  AUTHENTICATION: [API key | OAuth2 | Managed identity]
  USE_CASES:
    - [Use case 1, e.g., "Generate embeddings"]
    - [Use case 2, e.g., "Chat completions"]
  ERROR_HANDLING:
    - [Scenario 1, e.g., "Rate limit exceeded"]
    - [Scenario 2, e.g., "Service unavailable"]
  DELIVERABLES: [Integration module, tests, docs]

    2. Event-Driven Architecture¶

    • Description: Design and implement event-driven communication between services
    • Input: Event sources, event consumers, event schema
    • Output: Event broker configuration, producers, consumers, schema registry
    • Example: P14: Implement event-driven document processing (upload event → processing → completion event)

    Input Template: 

    P14 REQUEST: Event Architecture
  EVENT_BROKER: [Azure Event Grid | Azure Service Bus | Kafka]
  EVENTS:
    - name: [Event name, e.g., "DocumentUploaded"]
      schema:
        - [Field 1, e.g., "document_id"]
        - [Field 2, e.g., "user_id"]
      producers: [Which services emit this event]
      consumers: [Which services consume this event]
  DELIVERY_GUARANTEE: [At-least-once | Exactly-once]
  ORDERING: [Required? Yes/No]
  DELIVERABLES: [Event schemas, producers, consumers, docs]

    3. Middleware & Service Mesh¶

    • Description: Implement middleware for cross-cutting concerns (logging, auth, rate limiting)
    • Input: Services to integrate, cross-cutting concerns, performance requirements
    • Output: Middleware implementation, service mesh configuration
    • Example: P14: Implement API gateway middleware (auth, rate limiting, logging) for eva-api

    Input Template: 

    P14 REQUEST: Middleware Implementation
  SERVICES: [Services to integrate]
  MIDDLEWARE_CONCERNS:
    - [Concern 1, e.g., "Authentication (JWT validation)"]
    - [Concern 2, e.g., "Rate limiting (100 req/min)"]
    - [Concern 3, e.g., "Request logging"]
  PLATFORM: [Azure API Management | Kong | Custom middleware]
  PERFORMANCE: [Latency overhead target, e.g., "<10ms"]
  DELIVERABLES: [Middleware code, config, performance report]

    4. Data Synchronization & ETL¶

    • Description: Synchronize data between systems or perform ETL operations
    • Input: Source system, target system, data mappings, sync frequency
    • Output: ETL pipeline, data validation, error handling
    • Example: P14: Sync user data from Azure AD to Cosmos DB (nightly, incremental updates)

    Input Template: 

    P14 REQUEST: Data Synchronization
  SOURCE:
    - system: [Source system, e.g., "Azure AD"]
    - data: [What data, e.g., "Users, groups"]
  TARGET:
    - system: [Target system, e.g., "Cosmos DB"]
    - container: [Container/table name]
  SYNC_STRATEGY: [Full sync | Incremental | CDC]
  FREQUENCY: [Real-time | Hourly | Daily]
  TRANSFORMATIONS:
    - [Transform 1, e.g., "Map Azure AD id to user_id"]
    - [Transform 2, e.g., "Flatten nested groups"]
  DELIVERABLES: [ETL pipeline, validation, monitoring]

    5. Integration Testing & Monitoring¶

    • Description: Test integrations and set up monitoring for integration health
    • Input: Integration points, test scenarios, monitoring requirements
    • Output: Integration tests, health checks, monitoring dashboards
    • Example: P14: Set up integration tests for Azure OpenAI API (embeddings, completions, error cases)

    Input Template: 

    P14 REQUEST: Integration Testing
  INTEGRATIONS:
    - [Integration 1, e.g., "Azure OpenAI"]
    - [Integration 2, e.g., "Azure Blob Storage"]
  TEST_SCENARIOS:
    - happy_path: [Normal operation]
    - error_cases:
        - [Error 1, e.g., "API timeout"]
        - [Error 2, e.g., "Invalid credentials"]
  MONITORING:
    - [Metric 1, e.g., "API call success rate"]
    - [Metric 2, e.g., "Latency (p95)"]
  DELIVERABLES: [Integration tests, health checks, dashboards]

    Service Standards¶

    API Integration Standards¶

    • Use official SDKs when available (Azure SDK, OpenAI SDK)
    • Implement retry logic with exponential backoff
    • Respect rate limits (implement client-side throttling)
    • Timeout configuration (reasonable defaults, configurable)
    • Error logging with correlation IDs

    Event-Driven Standards¶

    • Event schema versioning (backward-compatible changes)
    • Dead-letter queues for failed events
    • Event idempotency (consumers handle duplicates)
    • Event metadata (timestamp, correlation ID, source)
    • Schema registry for event contracts

    Middleware Standards¶

    • Minimal latency overhead (<10ms per middleware)
    • Fail-open vs fail-closed documented
    • Circuit breaker for downstream dependencies
    • Health checks exposed (/health endpoint)
    • Metrics exposed (request count, latency, errors)

    Data Sync Standards¶

    • Validation before and after sync
    • Error handling (log, alert, retry)
    • Idempotent operations (safe to retry)
    • Audit trail (what changed, when, by whom)
    • Rollback capability for failed syncs

    Common Patterns¶

    Pattern 1: API Integration with Retry¶

    from openai import AzureOpenAI
from azure.identity import DefaultAzureCredential
import time

class OpenAIIntegration:
    def __init__(self):
        self.client = AzureOpenAI(
            azure_endpoint="https://eva-openai.openai.azure.com",
            azure_ad_token_provider=DefaultAzureCredential(),
            api_version="2024-02-01"
        )
        self.max_retries = 3
        self.base_delay = 1  # seconds

    def generate_embeddings(self, texts: list[str]) -> list[list[float]]:
        """Generate embeddings with retry logic."""
        for attempt in range(self.max_retries):
            try:
                response = self.client.embeddings.create(
                    model="text-embedding-3-small",
                    input=texts
                )
                return [item.embedding for item in response.data]

            except Exception as e:
                if attempt == self.max_retries - 1:
                    raise  # Re-raise on final attempt

                # Exponential backoff
                delay = self.base_delay * (2 ** attempt)
                print(f"Retry {attempt + 1}/{self.max_retries} after {delay}s")
                time.sleep(delay)

    Pattern 3: Middleware Example¶

    from fastapi import Request, HTTPException
from functools import wraps
import time

# Rate limiting middleware
class RateLimiter:
    def __init__(self, max_requests: int, window_seconds: int):
        self.max_requests = max_requests
        self.window_seconds = window_seconds
        self.requests = {}  # user_id -> [(timestamp, count)]

    async def __call__(self, request: Request, call_next):
        user_id = request.state.user_id  # From auth middleware
        now = time.time()

        # Clean old requests
        if user_id in self.requests:
            self.requests[user_id] = [
                (ts, count) for ts, count in self.requests[user_id]
                if now - ts < self.window_seconds
            ]

        # Check rate limit
        current_count = sum(count for _, count in self.requests.get(user_id, []))
        if current_count >= self.max_requests:
            raise HTTPException(429, "Rate limit exceeded")

        # Record request
        if user_id not in self.requests:
            self.requests[user_id] = []
        self.requests[user_id].append((now, 1))

        return await call_next(request)

    File Naming Convention¶

    All deliverables must follow: 

    P14-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].[ext]

    Examples: - P14-API-INTEGRATION-OPENAI-20251209-120000.py - P14-EVENT-SCHEMA-DOCUMENT-UPLOADED-20251209-130000.json - P14-MIDDLEWARE-RATE-LIMITER-20251209-140000.py

    Naming Rules: 1. Prefix: Always start with P14- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Extension: .py, .json, .md as appropriate 5. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md

    Quality Checklist¶

    Before marking P14 work complete:

    • Integration tested with external system
    • Error handling implemented (retry, timeout, fallback)
    • Rate limiting respected (no API abuse)
    • Monitoring and health checks configured
    • Event schemas versioned and documented
    • Idempotency ensured for critical operations
    • Security reviewed (credentials, encryption)
    • Performance tested (latency, throughput)
    • P07 integration tests passing
    • Documentation complete for developers
