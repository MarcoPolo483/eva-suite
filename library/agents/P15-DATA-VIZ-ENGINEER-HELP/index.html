    Agent Service Catalog - P15 (HELP)¶

    Agent ID: P15 Agent Name: Data Visualization Engineer POD: POD-E (Experience) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P15 is the Data Visualization Engineer agent responsible for creating dashboards, data visualizations, analytics interfaces, reporting systems, and ensuring data is presented clearly and actionably across the EVA Suite.

    Core Services¶

    1. Dashboard Development¶

    • Description: Build interactive dashboards for analytics and operations
    • Input: Data sources, metrics to visualize, user requirements
    • Output: Dashboard implementation, visualizations, interactivity
    • Example: P15: Create operations dashboard for eva-api (requests/sec, latency, errors, user activity)

    Input Template: 

    P15 REQUEST: Dashboard Development
  DASHBOARD_NAME: [Dashboard name]
  DATA_SOURCES:
    - [Source 1, e.g., "Application Insights"]
    - [Source 2, e.g., "Cosmos DB"]
  METRICS:
    - [Metric 1, e.g., "API requests per second"]
    - [Metric 2, e.g., "Latency (p50, p95, p99)"]
    - [Metric 3, e.g., "Error rate"]
  VISUALIZATIONS:
    - [Viz 1, e.g., "Time series chart (requests)"]
    - [Viz 2, e.g., "Heatmap (latency by endpoint)"]
  INTERACTIVITY:
    - [Feature 1, e.g., "Date range filter"]
    - [Feature 2, e.g., "Drill-down by user"]
  PLATFORM: [Power BI | Grafana | Custom (React + D3)]

    2. Data Visualization Design¶

    • Description: Design effective visualizations for complex data
    • Input: Data to visualize, audience, insights to communicate
    • Output: Visualization designs, chart recommendations, best practices
    • Example: P15: Design visualization for document query patterns (heatmap of query topics over time)

    Input Template: 

    P15 REQUEST: Visualization Design
  DATA_TYPE: [Time series | Hierarchical | Relational | Geospatial]
  DATA_CHARACTERISTICS:
    - [Char 1, e.g., "High cardinality (1000s of categories)"]
    - [Char 2, e.g., "Sparse data (many nulls)"]
  AUDIENCE: [Executives | Analysts | Developers | End users]
  INSIGHTS_TO_COMMUNICATE:
    - [Insight 1, e.g., "Identify usage trends"]
    - [Insight 2, e.g., "Detect anomalies"]
  CONSTRAINTS:
    - [Constraint 1, e.g., "Mobile-friendly"]
    - [Constraint 2, e.g., "Colorblind-accessible"]
  DELIVERABLES: [Design mockups, chart recommendations]

    3. Reporting System Implementation¶

    • Description: Build automated reporting systems (scheduled reports, alerts)
    • Input: Report requirements, data sources, distribution list
    • Output: Report templates, automation scripts, distribution setup
    • Example: P15: Create weekly executive report for EVA Suite (usage stats, performance, incidents, PDF email)

    Input Template: 

    P15 REQUEST: Reporting System
  REPORT_TYPE: [Operational | Executive | Compliance | Custom]
  FREQUENCY: [Real-time | Daily | Weekly | Monthly]
  DATA_SOURCES:

  FORMAT: [PDF | Excel | Interactive web report]

    Service Standards¶

    Visualization Standards¶

    • Choose chart type appropriate for data (bar, line, scatter, heatmap, etc.)
    • Use color intentionally (not decorative)
    • Colorblind-safe palettes (avoid red/green only)
    • Accessible (WCAG 2.1 AA for web visualizations)
    • Clearly labeled axes and legends
    • Show data source and last updated time

    Dashboard Standards¶

    • Mobile-responsive (work on tablets, phones)
    • Loading time <3 seconds for initial view
    • Auto-refresh for real-time dashboards (30s-5min)
    • Filters persist across page refresh
    • Export capability (PNG, CSV, PDF)
    • Share links work (deep linking)

    Performance Standards¶

    • Render <2 seconds for 1M data points (server-side aggregation)
    • Smooth interactions (60 FPS for animations)
    • Data caching (avoid redundant queries)
    • Progressive loading (show data as it arrives)
    • Lazy loading for off-screen visualizations

    Reporting Standards¶

    • Automated generation (no manual steps)
    • Reliable delivery (retries on failure)
    • Version control (track report definitions)
    • Audit trail (who generated, when, what data)
    • Consistent branding (logos, colors, fonts)

    Common Patterns¶

    Pattern 1: Dashboard Component (React + Recharts)¶

    import { LineChart, Line, XAxis, YAxis, CartesianGrid, Tooltip, Legend } from 'recharts';
import { useQuery } from 'react-query';

interface MetricData {
  timestamp: string;
  requests_per_sec: number;
  latency_p95: number;
}

export function ApiMetricsDashboard() {
  const { data, isLoading } = useQuery<MetricData[]>(
    'api-metrics',
    async () => {
      const response = await fetch('/api/metrics?window=1h');
      return response.json();
    },
    { refetchInterval: 30000 } // Auto-refresh every 30s
  );

  if (isLoading) return <div>Loading...</div>;

  return (
    <div className="dashboard">
      <h2>API Performance (Last Hour)</h2>

      <LineChart width={800} height={400} data={data}>
        <CartesianGrid strokeDasharray="3 3" />
        <XAxis dataKey="timestamp" />
        <YAxis yAxisId="left" label={{ value: 'Requests/sec', angle: -90 }} />
        <YAxis yAxisId="right" orientation="right" label={{ value: 'Latency (ms)', angle: 90 }} />
        <Tooltip />
        <Legend />
        <Line yAxisId="left" type="monotone" dataKey="requests_per_sec" stroke="#8884d8" name="Requests/sec" />
        <Line yAxisId="right" type="monotone" dataKey="latency_p95" stroke="#82ca9d" name="Latency p95" />
      </LineChart>
    </div>
  );
}

    Pattern 2: Visualization Best Practices¶

    ## Chart Type Selection Guide

### Time Series Data
- **Use**: Line chart or area chart
- **When**: Showing trends over time (requests/sec, latency)
- **Avoid**: Bar chart (too cluttered for many time points)

### Comparisons (Categorical)
- **Use**: Bar chart (horizontal for long labels)
- **When**: Comparing values across categories (errors by endpoint)
- **Avoid**: Pie chart (hard to compare similar values)

### Distributions
- **Use**: Histogram or box plot
- **When**: Showing data distribution (latency distribution)
- **Avoid**: Bar chart with too many bins

### Correlations
- **Use**: Scatter plot
- **When**: Showing relationship between two variables
- **Add**: Trend line for clarity

### Hierarchical Data
- **Use**: Treemap or sunburst chart
- **When**: Showing nested categories (folder structures, cost breakdown)

### Geographic Data
- **Use**: Choropleth map or bubble map
- **When**: Showing regional differences
- **Avoid**: Table (map is more intuitive)

## Color Palette (Colorblind-Safe)
- Primary: #0078D4 (blue)
- Secondary: #107C10 (green)
- Alert: #D83B01 (orange)
- Error: #A80000 (red)
- Neutral: #605E5C (gray)

## Accessibility Checklist
- [ ] Color is not the only indicator (use icons, patterns)
- [ ] Contrast ratio 4.5:1 (text on background)
- [ ] Interactive elements keyboard-accessible
- [ ] Screen reader support (ARIA labels)

    Pattern 3: Automated Report Generation¶

    from jinja2 import Template
from weasyprint import HTML
import pandas as pd
from datetime import datetime, timedelta

def generate_weekly_report():
    """Generate weekly executive report as PDF."""

    # Query data for last week
    end_date = datetime.now()
    start_date = end_date - timedelta(days=7)

    # Get metrics
    usage_data = query_usage_metrics(start_date, end_date)
    performance_data = query_performance_metrics(start_date, end_date)
    incidents = query_incidents(start_date, end_date)

    # Prepare report data
    report_data = {
        'report_date': end_date.strftime('%Y-%m-%d'),
        'period': f"{start_date.strftime('%Y-%m-%d')} to {end_date.strftime('%Y-%m-%d')}",
        'total_users': usage_data['total_users'],
        'total_documents': usage_data['total_documents'],
        'total_queries': usage_data['total_queries'],
        'avg_latency_p95': performance_data['avg_latency_p95'],
        'uptime_percentage': performance_data['uptime_percentage'],
        'incidents': incidents,
        'chart_url': generate_chart(usage_data)  # Generate chart image
    }

    # Render HTML template
    template = Template(open('report_template.html').read())
    html_content = template.render(**report_data)

    # Convert to PDF
    pdf_filename = f"eva_weekly_report_{end_date.strftime('%Y%m%d')}.pdf"
    HTML(string=html_content).write_pdf(pdf_filename)

    # Email report
    send_email(
        to=['executives@eva-suite.com'],
        subject=f"EVA Suite Weekly Report - {end_date.strftime('%Y-%m-%d')}",
        body="Weekly report attached.",
        attachments=[pdf_filename]
    )

    return pdf_filename

    File Naming Convention¶

    All deliverables must follow: 

    P15-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].[ext]

    Examples: - P15-DASHBOARD-API-METRICS-20251209-120000.tsx - P15-VIZ-DESIGN-QUERY-PATTERNS-20251209-130000.md - P15-REPORT-TEMPLATE-WEEKLY-EXECUTIVE-20251209-140000.html

    Naming Rules: 1. Prefix: Always start with P15- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Extension: .tsx, .md, .html, .py as appropriate 5. Purpose: Traceability
