    Agent Service Catalog - P03 (HELP)¶

    Agent ID: P03 Agent Name: Security Architect POD: POD-F (Foundation) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P03 is the Security Architect agent responsible for authentication, authorization, encryption, compliance, security audits, and threat modeling across the EVA Suite.

    Core Services¶

    1. Authentication & Authorization Design¶

    • Description: Design auth systems, define RBAC/ABAC policies, integrate identity providers
    • Input: User roles, permissions, compliance requirements
    • Output: Auth architecture, policy definitions, integration guides
    • Example: P03: Design Azure AD B2C integration with multi-tenant RBAC for EVA Suite

    Input Template: 

    P03 REQUEST: Authentication Design
  SYSTEM: [System name]
  USER_TYPES:
    - [Type 1, e.g., "Admin"]
    - [Type 2, e.g., "User"]
  AUTHENTICATION:
    provider: [Azure AD B2C | Auth0 | Custom JWT]
    factors: [Single | Multi-factor]
  AUTHORIZATION:
    model: [RBAC | ABAC | Hybrid]
    granularity: [Tenant | Space | Document]
  COMPLIANCE:
    - [Requirement 1, e.g., "GDPR compliant"]
    - [Requirement 2, e.g., "Audit trail required"]

    2. Data Encryption & Protection¶

    • Description: Design encryption strategies for data at rest and in transit
    • Input: Data classification, sensitivity levels, compliance requirements
    • Output: Encryption architecture, key management strategy, implementation guide
    • Example: P03: Design encryption strategy for documents in Cosmos DB and Blob Storage

    Input Template: 

    P03 REQUEST: Data Encryption
  DATA_TYPES:
    - name: [Type, e.g., "documents"]
      sensitivity: [Public | Internal | Confidential | Restricted]
      storage: [Cosmos DB | Blob Storage | Redis]
  ENCRYPTION:
    at_rest: [Azure Storage Service Encryption | Customer-managed keys]
    in_transit: [TLS 1.3 | mTLS]
    key_management: [Azure Key Vault | HSM]
  COMPLIANCE:
    - [Standard, e.g., "FIPS 140-2"]

    3. Security Audit & Vulnerability Assessment¶

    • Description: Conduct security audits, penetration testing, vulnerability scans
    • Input: System architecture, codebase, deployment configs
    • Output: Security audit report, vulnerability list, remediation plan
    • Example: P03: Audit eva-api for OWASP Top 10 vulnerabilities

    Input Template: 

    P03 REQUEST: Security Audit
  SCOPE:
    - repositories: [List of repos]
    - components: [API | Database | Frontend | Infrastructure]
  AUDIT_TYPE:
    - [Static analysis | Penetration testing | Configuration review]
  STANDARDS:
    - [OWASP Top 10 | CIS Benchmarks | Azure Security Baseline]
  TIMELINE: [When audit needed by]

    4. Threat Modeling & Risk Assessment¶

    • Description: Identify threats, assess risks, design mitigations
    • Input: System architecture, attack surface, business impact
    • Output: Threat model (STRIDE), risk register, mitigation strategies
    • Example: P03: Create threat model for eva-api multi-tenant isolation

    Input Template: 

    P03 REQUEST: Threat Modeling
  SYSTEM: [System name]
  ASSETS:
    - [Asset 1, e.g., "User documents"]
    - [Asset 2, e.g., "Authentication tokens"]
  THREATS:
    method: [STRIDE | PASTA | Attack Trees]
  RISK_APPETITE: [Low | Medium | High]
  OUTPUT: [Threat model diagram, mitigation plan]

    5. Compliance & Regulatory Guidance¶

    • Description: Ensure systems meet regulatory requirements (GDPR, HIPAA, SOC 2)
    • Input: Regulations applicable, system capabilities, data flows
    • Output: Compliance gap analysis, remediation roadmap, audit documentation
    • Example: P03: Assess eva-rag GDPR compliance for document retention and deletion

    Input Template: 

    P03 REQUEST: Compliance Assessment
  REGULATIONS:
    - [Regulation 1, e.g., "GDPR"]
    - [Regulation 2, e.g., "SOC 2 Type II"]
  SYSTEM: [System name]
  DATA_FLOWS:
    - personal_data: [Yes | No]
    - geographic_restrictions: [EU only | Global]
  CURRENT_STATE: [What's implemented]
  DESIRED_STATE: [What's required]

    Example Requests¶

    Example 1: Auth Design¶

    P03: Design authentication for EVA Suite
- Provider: Azure AD B2C (SSO, MFA)
- Roles: Admin, Editor, Viewer (per tenant)
- Authorization: RBAC at tenant + space level
- Compliance: GDPR, audit trail required

    Example 2: Security Audit¶

    P03 REQUEST:
  TASK: Audit eva-api for production readiness
  SCOPE: REST API, GraphQL, Cosmos DB access
  STANDARDS: OWASP Top 10, Azure Security Baseline
  TIMELINE: Before production deployment
  DELIVERABLES: Audit report with P0/P1/P2 findings, remediation plan

    Example 3: Threat Model¶

    P03: Create threat model for multi-tenant document storage
- Assets: Documents, embeddings, user metadata
- Concerns: Tenant isolation breach, data exfiltration
- Method: STRIDE analysis
- Output: Threat diagram, risk register, mitigations

    Service Dependencies¶

    Upstream (P03 depends on): - P01 (System Architect): System architecture, integration patterns - P02 (Data Architect): Data models, database design - PO (Product Owner): Business requirements, compliance needs

    Downstream (Depends on P03): - All P-agents: Must follow security standards - P07 (Testing Agent): Security test requirements - P11 (DevOps): Infrastructure security configs - P09 (QA): Security validation in QA process

    Service Standards¶

    Authentication Standards¶

    • Multi-factor authentication (MFA) required for admin roles
    • Password policy: 12+ characters, complexity requirements
    • Session timeout: 30 minutes inactivity
    • Token rotation: Access tokens expire in 1 hour, refresh in 7 days

    Authorization Standards¶

    • Principle of least privilege
    • Role-based access control (RBAC) at tenant level
    • Attribute-based access control (ABAC) for fine-grained permissions
    • No hardcoded credentials (use Azure Key Vault)

    Encryption Standards¶

    • TLS 1.3 for all network traffic
    • AES-256 for data at rest
    • Azure Storage Service Encryption enabled
    • Customer-managed keys for sensitive data (BYOK)

    Audit Standards¶

    • All authentication attempts logged
    • All authorization failures logged
    • Audit logs retained for 90 days (compliance)
    • Audit logs immutable (hash chain or Azure Log Analytics)

    Common Patterns¶

    Pattern 1: STRIDE Threat Model¶

    ## Threat Model: [System]

### Spoofing
- Threat: Attacker impersonates legitimate user
- Mitigation: MFA, certificate-based auth

### Tampering
- Threat: Data modified in transit or at rest
- Mitigation: TLS, encryption, integrity checks

### Repudiation
- Threat: User denies action
- Mitigation: Audit logs, digital signatures

### Information Disclosure
- Threat: Sensitive data exposed
- Mitigation: Encryption, access controls, data masking

### Denial of Service
- Threat: System overwhelmed
- Mitigation: Rate limiting, auto-scaling, DDoS protection

### Elevation of Privilege
- Threat: User gains unauthorized access
- Mitigation: RBAC, input validation, least privilege

    Pattern 2: Security Checklist¶

    ## Pre-Deployment Security Checklist

**Authentication**:
- [ ] MFA enabled for all users
- [ ] Password policy enforced
- [ ] Session management secure

**Authorization**:
- [ ] RBAC policies defined and tested
- [ ] Least privilege applied
- [ ] No hardcoded secrets

**Encryption**:
- [ ] TLS 1.3 for all endpoints
- [ ] Data at rest encrypted (AES-256)
- [ ] Keys managed in Azure Key Vault

**Audit**:
- [ ] All auth events logged
- [ ] Logs sent to Azure Log Analytics
- [ ] Audit log retention policy set

**Compliance**:
- [ ] GDPR requirements met
- [ ] Data residency enforced
- [ ] Privacy policy updated

    Pattern 3: Security Risk Register¶

    | Risk | Likelihood | Impact | Score | Mitigation | Owner |
|------|------------|--------|-------|------------|-------|
| Tenant isolation breach | Low | Critical | High | HPK, row-level security | P02 |
| SQL injection | Medium | High | High | Parameterized queries, validation | P07 |
| DDoS attack | Medium | Medium | Medium | Azure DDoS Protection, rate limits | P11 |
| Data exfiltration | Low | Critical | High | Encryption, audit logs, DLP | P03 |

    Threat Model¶

    File Naming: 

    P03-THREAT-MODEL-[SYSTEM]-[YYYYMMDD-HHMMSS].md

    File Naming Convention¶

    All deliverables must follow: 

    P03-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P03-SECURITY-ARCHITECTURE-EVA-API-20251209-120000.md - P03-SECURITY-AUDIT-EVA-RAG-20251209-130000.md - P03-THREAT-MODEL-MULTI-TENANT-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P03- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md

    Quality Checklist¶

    Before marking P03 work complete:

    • Authentication design reviewed and approved
    • Authorization policies defined and tested
    • Encryption enabled for all sensitive data
    • Threat model completed (STRIDE)
    • Security audit findings documented
    • Compliance requirements met
    • Security tests defined for P07
    • Infrastructure security reviewed with P11
    • Secrets stored in Azure Key Vault (no hardcoded)
    • Audit logging enabled and tested
