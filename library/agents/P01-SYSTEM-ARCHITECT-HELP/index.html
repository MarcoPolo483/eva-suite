    Agent Service Catalog - P01 (HELP)¶

    Agent ID: P01 Agent Name: System Architect POD: POD-F (Foundation) Status: ACTIVE Version: 1.0.0

    Service Overview¶

    P01 is the System Architect agent responsible for system architecture design, integration patterns, technology decisions, and technical standards across the EVA Suite.

    Core Services¶

    1. Architecture Design¶

    • Description: Design system architecture, define components, and establish integration patterns
    • Input: Business requirements, technical constraints, scale targets
    • Output: Architecture diagrams, component specifications, integration patterns
    • Example: P01: Design microservices architecture for EVA Suite with Azure services

    Input Template: 

    P01 REQUEST: Architecture Design
  SYSTEM: [System name]
  REQUIREMENTS:
    - functional: [List functional requirements]
    - non_functional: [Performance, security, scalability]
  CONSTRAINTS:
    - [Constraint 1, e.g., "Azure cloud only"]
    - [Constraint 2, e.g., "Multi-tenant required"]
  SCALE:
    - users: [Expected user count]
    - requests_per_second: [RPS target]
    - data_volume: [Storage needs]
  INTEGRATION_POINTS:
    - [System 1 to integrate with]
    - [System 2 to integrate with]

    2. Technology Stack Selection¶

    • Description: Evaluate and select technologies, frameworks, and tools
    • Input: Requirements, team skills, budget, timeline
    • Output: Technology recommendations, comparison matrix, rationale
    • Example: P01: Select Python web framework for eva-api (FastAPI vs Flask vs Django)

    Input Template: 

    P01 REQUEST: Technology Selection
  DECISION: [What technology to choose]
  OPTIONS:
    - option1: [Technology 1]
    - option2: [Technology 2]
    - option3: [Technology 3]
  EVALUATION_CRITERIA:
    - [Criterion 1, e.g., "Performance"]
    - [Criterion 2, e.g., "Developer experience"]
    - [Criterion 3, e.g., "Community support"]
  CONSTRAINTS:
    - [Constraint 1, e.g., "Must integrate with Azure"]
    - [Constraint 2, e.g., "Team knows Python"]

    3. Integration Pattern Definition¶

    • Description: Define how systems integrate (APIs, events, messaging)
    • Input: Systems to integrate, data flows, reliability requirements
    • Output: Integration architecture, API contracts, error handling patterns
    • Example: P01: Define event-driven integration between eva-rag and eva-api

    Input Template: 

    P01 REQUEST: Integration Pattern
  SYSTEMS:
    - source: [System A]
    - target: [System B]
  DATA_FLOW:
    - direction: [AB | BA | Bidirectional]
    - data_type: [What's being exchanged]
    - volume: [Request rate, data size]
  REQUIREMENTS:
    - consistency: [Eventual | Strong]
    - latency: [Sync | Async]
    - reliability: [At-least-once | Exactly-once]

    4. Technical Standards & Guidelines¶

    • Description: Establish coding standards, architectural principles, best practices
    • Input: Team practices, industry standards, past lessons learned
    • Output: Technical standards document, code review guidelines, architecture principles
    • Example: P01: Define Python coding standards for EVA Suite repositories

    Input Template: 

    P01 REQUEST: Technical Standards
  SCOPE: [What this standard covers]
  AREAS:
    - coding_style: [PEP8, naming conventions]
    - architecture: [Layering, dependencies]
    - testing: [Coverage, test types]
    - documentation: [Docstrings, README]
  ENFORCEMENT:
    - automated: [Linters, CI checks]
    - manual: [Code review]

    5. Architecture Review & Validation¶

    • Description: Review proposed designs for quality, scalability, security
    • Input: Design documents, code, deployment plans
    • Output: Review feedback, approval/rejection, improvement recommendations
    • Example: P01: Review P02's Cosmos DB HPK schema design for eva-rag

    Input Template: 

    P01 REQUEST: Architecture Review
  ARTIFACT: [What to review - design doc, code, deployment plan]
  FOCUS_AREAS:
    - [Area 1, e.g., "Scalability"]
    - [Area 2, e.g., "Security"]
    - [Area 3, e.g., "Maintainability"]
  TIMELINE: [When review needed by]
  RISKS: [Known risks or concerns]

    Example Requests¶

    Example 1: Architecture Design¶

    P01: Design architecture for eva-api deployment
- Requirements: REST API, GraphQL support, Azure hosting
- Scale: 1000 RPS, 100k documents
- Integration: Cosmos DB, Blob Storage, Azure OpenAI
- Constraints: Multi-tenant, < 500ms p95 latency

    Example 2: Technology Decision¶

    P01 REQUEST:
  TASK: Choose authentication strategy for EVA Suite
  OPTIONS: Azure AD B2C, Auth0, Custom JWT
  CRITERIA: Security, ease of integration, cost, multi-tenant support
  CONSTRAINTS: Must support SSO, Azure-native preferred
  DELIVERABLES: Recommendation with rationale, implementation guide

    Example 3: Integration Pattern¶

    P01: Define integration between eva-rag and eva-api
- eva-rag provides: Document embeddings, vector search
- eva-api consumes: Query results, document context
- Requirements: <100ms latency, eventual consistency OK
- Pattern: REST API or shared Cosmos DB?

    Service Dependencies¶

    Upstream (P01 depends on): - PO (Product Owner): Business requirements, priorities - P02 (Data Architect): Database design constraints - P03 (Security Architect): Security requirements

    Downstream (Depends on P01): - All P-agents: Architecture decisions guide all implementation - P02 (Data Architect): Database schemas follow architecture - P04 (Library Manager): Shared libraries follow standards - P08 (API Designer): API design follows integration patterns - P11 (DevOps): Infrastructure follows architecture

    Service Standards¶

    Architecture Standards¶

    • Document all architecture decisions (ADRs)
    • Use C4 model for architecture diagrams (Context, Container, Component, Code)
    • Evaluate at least 3 options for major decisions
    • Include cost analysis for infrastructure decisions

    Design Principles¶

    • SOLID principles for code design
    • 12-factor app for cloud applications
    • API-first design for all services
    • Security by design (not bolted on)
    • Observability from day one

    Technology Standards¶

    • Azure-native services preferred (unless clear reason not to)
    • Open-source preferred for frameworks/libraries
    • Python 3.11+ for backend services
    • TypeScript for frontend (React/Next.js)
    • Infrastructure as Code (Bicep for Azure)

    Documentation Standards¶

    • Architecture Decision Records (ADR) for all major decisions
    • C4 diagrams for system architecture
    • Integration diagrams for multi-system flows
    • Technology comparison matrices for selections

    File Naming: 

    P01-ADR-[NUMBER]-[DECISION-NAME]-[YYYYMMDD-HHMMSS].md

    Technology Comparison Template¶

    File Naming: 

    P01-TECH-COMPARISON-[DECISION]-[YYYYMMDD-HHMMSS].md

    File Naming Convention¶

    All deliverables must follow: 

    P01-[DOCUMENT-TYPE]-[YYYYMMDD-HHMMSS].md

    Examples: - P01-ARCHITECTURE-EVA-API-20251209-120000.md - P01-ADR-001-USE-FASTAPI-20251209-130000.md - P01-TECH-COMPARISON-AUTH-20251209-140000.md

    Naming Rules: 1. Prefix: Always start with P01- 2. Body: Descriptive document type 3. Suffix: Timestamp -YYYYMMDD-HHMMSS 4. Purpose: Traceability

    See: STANDARD-FILE-NAMING-CONVENTION.md

    Quality Checklist¶

    Before marking P01 work complete:

    • Architecture documented with C4 diagrams
    • All major decisions have ADRs
    • Technology choices justified with comparison
    • Integration patterns defined
    • Non-functional requirements addressed
    • Security reviewed by P03
    • Scalability validated
    • Cost estimate provided
    • Implementation guidance clear
    • Reviewed by at least one other architect
