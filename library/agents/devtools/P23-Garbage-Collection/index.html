    P23 - EVA Garbage Collection Pattern¶

    Version: 2.0.0 Status: Beta Owner: P15-DVM (Dev Master Orchestrator) Category: Operations & Maintenance Related Patterns: P21-Provenance-Fingerprinting (ATO compliance integration)

    Overview¶

    Automated garbage cleanup pattern for EVA Suite repositories with P21 provenance integration for full audit trail and ATO compliance. Identifies and removes experimental code, build artifacts, orphaned files, and misplaced features that accumulate during rapid development ("building an airplane while flying it").

    NEW in v2.0: Formal archival with P21 provenance metadata (fingerprinting, execution logs, audit trail for government systems).

    Problem Statement¶

    During rapid AI-assisted development: - Experimental code gets committed ("let's try this...") - Build artifacts accumulate (.pytest_cache, htmlcov, .ruff_cache) - Features land in wrong repos (Chat demo in eva-api) - Test data/scripts proliferate at repo root - Logs and archives pile up (20-30 MB per repo)

    Impact: Repos become cluttered, confusing, and harder to navigate. Technical debt accumulates invisibly.

    Government Context: Cleanup operations require audit trail for ATO compliance. Simple "delete and forget" violates governance requirements.

    Solution: 4-Phase Garbage Collection with P21 Integration¶

    Phase 1: Identify (Automated Scanning)¶

    # Scan repo for garbage patterns
.\tools\eva-library\identify-garbage.ps1 -RepoPath "C:\path\to\eva-api"

    Identifies: - Root-level artifacts (node_modules/, .DS_Store, Thumbs.db, *.log, *.tmp at repo root - should be in subdirectories or gitignored) - Build artifacts (.pytest_cache, htmlcov, .ruff_cache, __pycache__) - Logs and archives (.zip, .log, azure-logs/) - Root-level test scripts (test_*.py, test-*.ps1 at repo root) - Experimental scripts (naming patterns: check_*.py, analyze_*.py, ingest_*.py) - Misplaced features (e.g., chat-ui in eva-api) - Orphaned data files (.csv, .json at repo root)

    Output: _garbage-inventory.json with categorized items

    Phase 2: Categorize (Risk Assessment)¶

    categories:
  safe_to_delete:        # Build artifacts, temp files (LOW risk)
  move_to_correct_repo:  # Misplaced features (MEDIUM risk)
  archive_first:         # Potentially useful (HIGH risk)
  keep:                  # False positives (NO risk)

    Risk Levels: - LOW: Can delete immediately (build artifacts, logs) - MEDIUM: Move to correct location first - HIGH: Archive before deletion (data, experimental scripts)

    Phase 3: Formal Archive with P21 Provenance (NEW in v2.0)¶

    Archive Location: 

    .eva-cache/garbage-archive/{timestamp}/
├── eva-api/                  # Preserved folder structure
│   ├── chat-ui/             # (15 files as-is)
│   └── load-tests/
├── eva-rag/
│   ├── data/
│   └── data-sources/
├── _provenance.json          # P21 metadata (who, what, when, why, how)
├── _fingerprints.json        # SHA256 hashes for integrity verification
├── _execution-log.json       # Machine-readable execution record
├── _execution-log.md         # Human-readable report
└── _manifest.json            # Complete inventory of archived items

    P21 Integration: Every garbage cleanup generates: 1. Provenance metadata (_provenance.json): 

    {
  "operation": "garbage-collection",
  "pattern": "P23",
  "timestamp": "2025-12-14T13:17:56Z",
  "operator": "GitHub Copilot (SM)",
  "approver": "Marco Presta",
  "reason": "Cleanup eva-api and eva-rag - 35 MB garbage",
  "riskLevel": "LOW-MEDIUM",
  "reversible": true,
  "compliance": {
    "atoRelevant": true,
    "auditTrail": true,
    "p21Integration": true
  }
}

    1. Fingerprints (_fingerprints.json): 

      {
  "algorithm": "SHA256",
  "files": [
    {
      "path": "eva-api/chat-ui/index.html",
      "sha256": "abc123...",
      "size": 72840,
      "lastModified": "2025-12-10T15:30:00Z"
    }
  ]
}

    2. Execution logs (JSON + Markdown for machine + human readers)

    3. Manifest (complete inventory with source/destination paths)

    Benefits: - ✅ Full reversibility (can restore exact state with integrity verification) - ✅ ATO compliance (audit trail for government systems) - ✅ Traceability (who approved, why, when, what was deleted) - ✅ Automation-friendly (JSON for tooling integration)

    Phase 4: Execute (Automated Cleanup)¶

    # Dry run first (ALWAYS)
.\scripts\cleanup-garbage.ps1 -DryRun

# Execute cleanup
.\scripts\cleanup-garbage.ps1

# Verify results
git status

    Operations: - Delete safe items (build artifacts) - Move misplaced items to correct repos - Archive risky items - Commit with detailed cleanup message

    Usage¶

    Quick Start¶

    # 1. Test (dry run)
cd "C:\Users\marco\Documents\_AI Dev\EVA Suite\eva-orchestrator"
.\scripts\cleanup-garbage.ps1 -DryRun

# 2. Execute
.\scripts\cleanup-garbage.ps1

# 3. Verify
cd ..\eva-api
git status  # Should show deletions
cd ..\eva-rag

    Monthly Maintenance¶

    # Run on all repos monthly
$repos = @('eva-api', 'eva-rag', 'eva-core', 'eva-auth', 'eva-ui', 'eva-suite')
foreach ($repo in $repos) {
    cd "C:\Users\marco\Documents\_AI Dev\EVA Suite\$repo"
    pwsh ..\eva-orchestrator\scripts\cleanup-garbage.ps1 -DryRun
}

    Guardrails¶

    Safety Checks¶

    1. Always dry-run first (-DryRun flag required)
    2. Automatic backup (Git bundle + archive folder)
    3. Risk categorization (safe vs. risky deletions)
    4. User confirmation for risky operations
    5. Rollback plan (git bundle restore in 5 min)

    Quality Gates¶

    • Pre-cleanup: Commit all changes ("BEFORE CLEANUP")
    • Post-cleanup: Run tests to verify nothing broke
    • Validation: Check git status, repo size reduction
    • Documentation: Update cleanup log

    Success Metrics¶

    Per Cleanup¶

    • Repo Size Reduction: 15-30 MB per repo
    • File Count Reduction: 40-50 files per repo
    • Execution Time: <5 minutes per repo
    • Accuracy: 0 false deletions (safety checks)

    Ecosystem-Wide¶

    • Monthly Cleanups: 15-20 repos
    • Total Garbage Removed: 200-300 MB/month
    • Technical Debt Prevented: High (keeps repos navigable)

    Related Patterns¶

    • P04-LIB (Repo Librarian): Keeps docs in sync (complements garbage collection)
    • P06-REV (Review Agent): Catches garbage during PR review
    • P15-DVM (Dev Master): Orchestrates cleanup across repos
    • P22-DPB (Documentation Bot): Generates clean docs (depends on clean repos)

    Lessons Learned¶

    LESSON-032: Garbage Accumulates Invisibly¶

    Discovery: eva-api and eva-rag had 35 MB combined garbage (Dec 14, 2025) Impact: Repos harder to navigate, slower clones, confused contributors Resolution: P23 pattern automates detection and cleanup

    LESSON-033: "Airplane While Flying" Requires Cleanup Discipline¶

    Insight: Rapid AI-assisted development creates more garbage than traditional development Pattern: Monthly cleanups prevent technical debt accumulation Motto: "Clean as you fly" (weekly) vs. "Clean up after landing" (monthly)
