    Three Concepts Script Header Standard¶

    Version: 1.0 Date: 2025-12-07 Status: Active Applies To: All PowerShell scripts in eva-orchestrator

    Purpose¶

    This document defines the standardized header format for PowerShell scripts implementing the Three Concepts Pattern: Context Engineering (CE), Housekeeping (HK), and Workspace Management (WM).

    Why Three Concepts Headers?¶

    Problem: "Vibe Coding" and Context Loss¶

    Without explicit documentation, AI assistants and developers: - Guess at script purpose and constraints - Miss critical dependencies and side effects - Don't understand what the script validates or monitors - Can't track workspace state changes

    Solution: Structured Headers¶

    Three Concepts headers provide instant context loading for: - Humans: Quick understanding of script behavior - AI Assistants: Accurate code suggestions and safe refactoring - Automation: Dependency tracking and impact analysis

    Standard Header Template¶

    <#
ðŸ§© CONTEXT:
[One paragraph explaining WHAT this script does and WHY it exists.
Focus on the business/operational value, not implementation details.]

ðŸ§© SYNOPSIS:
    [One-line description for Get-Help]

ðŸ§© DESCRIPTION:
    [Detailed explanation of functionality, use cases, and workflow.
    Include bullet points for key features.]

ðŸ§© CONTEXT_ENGINEERING:
    Mission: [Primary goal this script achieves]
    Constraints: [Critical limitations, requirements, or assumptions]
    Reuses: [Dependencies on other scripts, modules, or services]
    Validates: [What this script checks/verifies before executing]

ðŸ§© HOUSEKEEPING:
    Creates: [Files, folders, or resources this script generates]
    Modifies: [Files, configs, or state this script changes]
    Validates: [Health checks or quality gates enforced]
    Cleans: [Temporary files, old artifacts, or garbage collection]
    Monitors: [Metrics, logs, or observability this script tracks]

ðŸ§© WORKSPACE_MANAGEMENT:
    TreeUpdates: [Does this script modify repo structure? Yes/No]
    Navigation: [Does this script change working directory? Yes/No]
    Caching: [Does this script use .eva-cache/? What files?]
    SessionState: [Does this script modify session environment? How?]

ðŸ§© COMPLIANCE:
    WCAG: [Accessibility considerations: N/A for backend scripts]
    Bilingual: [EN/FR support: Output messages, error handling]
    RBAC: [Role-Based Access Control: Who can run this?]
    ProtectedB: [Handles Protected B data? Yes/No - document sensitivity]
    Audit: [Logging/audit trail requirements]

.PARAMETER ParameterName
    [Description]

.EXAMPLE
    .\Script-Name.ps1
    [Description of example]

.NOTES
    POD: [POD-O, POD-F, POD-X, etc.]
    Owner: [Agent/Team responsible]
    Last Modified: [YYYY-MM-DD]
#>

    Field Definitions¶

    ðŸ§© CONTEXT¶

    Purpose: One-paragraph elevator pitch for the script. Format: Plain English, business-focused. Length: 2-4 sentences maximum. Example: "This script verifies the Three Concepts Pattern at session start: Context Engineering, Housekeeping, and Directory Mapping. Auto-remediates missing/stale artifacts to ensure session starts clean."

    ðŸ§© CONTEXT_ENGINEERING¶

    Purpose: Prevent "vibe coding" by documenting constraints and validation.

    • Mission: Primary goal (e.g., "Bootstrap EVA Memory System for new repos")
    • Constraints: Critical limitations (e.g., "Requires git CLI authenticated", "Minimum PowerShell 7.0")
    • Reuses: Dependencies (e.g., "Calls bootstrap-repo.ps1 for each POD")
    • Validates: Pre-execution checks (e.g., "Confirms .eva-memory.json schema v2.0")

    ðŸ§© HOUSEKEEPING¶

    Purpose: Document all file operations and side effects.

    • Creates: New files/folders (e.g., ".eva-cache/repo-tree.txt")
    • Modifies: Changed files (e.g., "Updates .eva-memory.json LastBootstrap timestamp")
    • Validates: Health checks (e.g., "Confirms all 5 repos have clean git status")
    • Cleans: Garbage collection (e.g., "Removes .eva-cache/repo-tree.txt if > 7 days old")
    • Monitors: Observability (e.g., "Tracks script execution time in reports/housekeeping-*.md")

    ðŸ§© WORKSPACE_MANAGEMENT¶

    Purpose: Track workspace state changes for safe navigation.

    • TreeUpdates: Does script modify repo structure? (Yes/No)
    • Navigation: Does script change $PWD or push/pop location? (Yes/No)
    • Caching: Uses .eva-cache/? Which files? (e.g., "Reads .eva-cache/repo-tree.txt")
    • SessionState: Modifies environment variables or session config? (e.g., "Sets $env:EVA_SESSION_ID")

    ðŸ§© COMPLIANCE¶

    Purpose: Document regulatory and governance requirements.

    • WCAG: Web accessibility (N/A for backend scripts, mandatory for UI)
    • Bilingual: EN/FR output messages (mandatory for user-facing scripts)
    • RBAC: Who can run this? (e.g., "Requires POD-O Scrum Master role")
    • ProtectedB: Handles sensitive data? (Yes/No + data classification)
    • Audit: Logging requirements (e.g., "Logs all file modifications to .eva-audit.log")

    Examples by Script Category¶

    Example 1: Session Management Script¶

    Script: Start-AISession.ps1

    <#
ðŸ§© CONTEXT:
This script verifies the Three Concepts Pattern at session start:
1. Context Engineering (copilot-instructions.md exists)
2. Housekeeping (quality gates configured, automation health)
3. Directory Mapping (repo tree fresh < 24 hours)
Auto-remediates missing/stale artifacts to ensure session starts clean.

ðŸ§© CONTEXT_ENGINEERING:
    Mission: Validate Three Concepts compliance before AI session begins
    Constraints: Requires .eva-memory.json, minimum PowerShell 7.0
    Reuses: bootstrap-repo.ps1 (if remediation needed)
    Validates: .eva-cache/repo-tree.txt < 24 hours, .github/copilot-instructions.md exists

ðŸ§© HOUSEKEEPING:
    Creates: .eva-cache/session-start-*.md (session log)
    Modifies: .eva-cache/repo-tree.txt (if stale)
    Validates: .eva-memory.json schema, automation health report < 24 hours
    Cleans: .eva-cache/session-start-*.md files > 7 days old
    Monitors: Session start time, validation failures, remediation actions

ðŸ§© WORKSPACE_MANAGEMENT:
    TreeUpdates: No
    Navigation: No (stays in repo root)
    Caching: Reads and updates .eva-cache/repo-tree.txt
    SessionState: Sets $env:EVA_SESSION_ID, $env:EVA_SESSION_START

ðŸ§© COMPLIANCE:
    WCAG: N/A (backend script)
    Bilingual: Console output messages in EN/FR
    RBAC: Any POD member can run
    ProtectedB: No (reads repo metadata only)
    Audit: Logs session start to .eva-audit.log
#>

    Example 2: Infrastructure Script¶

    Script: bootstrap-repo.ps1

    <#
ðŸ§© CONTEXT:
This script bootstraps the EVA Memory System for any repository, deploying
the Three Concepts Pattern (Context Engineering, Housekeeping, Directory Mapping).
It generates .eva-memory.json, .github/copilot-instructions.md, .eva-housekeeping.json,
and .eva-cache/repo-tree.txt to enable instant AI context and quality gates.

ðŸ§© CONTEXT_ENGINEERING:
    Mission: Deploy Three Concepts Pattern to new EVA repositories
    Constraints: Requires git CLI authenticated, GitHub repo must exist
    Reuses: None (standalone bootstrap)
    Validates: GitHub CLI authenticated (gh auth status), repo not already bootstrapped

ðŸ§© HOUSEKEEPING:
    Creates: .eva-memory.json, .github/copilot-instructions.md, .eva-housekeeping.json, .eva-cache/repo-tree.txt
    Modifies: None (only creates new files)
    Validates: JSON schema compliance, file permissions correct
    Cleans: None (bootstrap is idempotent)
    Monitors: Bootstrap execution time, file creation success rate

ðŸ§© WORKSPACE_MANAGEMENT:
    TreeUpdates: Yes (creates .eva-cache/, .github/ folders)
    Navigation: No (operates in target repo path)
    Caching: Creates .eva-cache/repo-tree.txt (176 KB average)
    SessionState: No environment changes

ðŸ§© COMPLIANCE:
    WCAG: N/A (backend script)
    Bilingual: Console output and .github/copilot-instructions.md in EN/FR
    RBAC: POD-O Scrum Master only
    ProtectedB: No (creates public repo structure)
    Audit: Logs bootstrap actions to .eva-audit.log
#>

    Migration Checklist¶

    When adding Three Concepts headers to existing scripts:

    • Add ðŸ§© CONTEXT: paragraph at top
    • Document CONTEXT_ENGINEERING section (Mission, Constraints, Reuses, Validates)
    • Document HOUSEKEEPING section (Creates, Modifies, Validates, Cleans, Monitors)
    • Document WORKSPACE_MANAGEMENT section (TreeUpdates, Navigation, Caching, SessionState)
    • Document COMPLIANCE section (WCAG, Bilingual, RBAC, ProtectedB, Audit)
    • Preserve existing .PARAMETER, .EXAMPLE, .NOTES sections
    • Test script still runs correctly with new header
    • Update .NOTES Last Modified date

    Validation¶

    All scripts with Three Concepts headers must: 1. Have all 5 sections present (CONTEXT, CE, HK, WM, COMPLIANCE) 2. Use emoji markers (ðŸ§©) for visual scanning 3. Include concrete details (not "TODO" or "N/A" without explanation) 4. Maintain accurate information (headers must match actual behavior) 5. Update Last Modified date when script logic changes

    Benefits¶

    For AI Assistants¶

    • Instant Context: No guessing at script purpose or constraints
    • Safe Refactoring: Understand all side effects before suggesting changes
    • Accurate Suggestions: Know what files are created/modified
    • Compliance Awareness: Respect WCAG, bilingual, RBAC, ProtectedB requirements

    For Humans¶

    • Quick Understanding: Scan headers to understand script in 30 seconds
    • Impact Analysis: See all files modified before running
    • Dependency Tracking: Know what other scripts this depends on
    • Onboarding: New team members understand scripts faster

    For Automation¶

    • Dependency Graphs: Auto-generate script call chains
    • Health Monitoring: Track which scripts create/clean which files
    • Compliance Audits: Verify all scripts document RBAC and ProtectedB handling
    • Change Impact: Detect which scripts affected by file/folder renames
