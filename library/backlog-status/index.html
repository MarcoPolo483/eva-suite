    üîÆ DEFERRED INITIATIVES (Parked Until Feature Completeness)¶

    1. EVA Agent Communication Protocol (EACP) (Sprint 20-30+)¶

    Status: üîµ PARKED (platform layer, not next feature) Priority: FOUNDATIONAL (activates after all 40+ products functionally complete) Effort: ~120 story points (6 months) Source: to upload/to upload/Comms.md (3,001 lines - ChatGPT conversation)

    Strategic Context: - Communication is the OS/spine of EVA Suite, not a standalone feature - Build all agents/features first (EVA Chat, DA, Sovereign-UI, ingestion, ATO, CPP-D, Jurisprudence), THEN unify under EACP - Avoids premature orchestration anti-pattern (building the bus before knowing what rides on it) - When features are stable/modular/testable, transformation to "agents" becomes mechanical

    Core Components:

    1.1 EVA Router (Agent-as-a-Service Dispatcher)¶

    • Central classifier routing requests to EVA microservices (Sovereign-UI, Ingestion, Translation, CPP-D, etc.)
    • Pattern: Level-2 Agentic Dispatcher inside APIM
    • Business Value: Single entry point for all EVA capabilities

    1.2 Intent Analyzer¶

    • Analyzes user/agent messages to extract:
    • Intent labels (e.g., "summarize_doc", "search_jurisprudence")
    • Entities (topic, time_range, jurisdiction, language)
    • Confidence scores (0.0-1.0)
    • Missing slots (required parameters not provided)
    • Implemented via: Base LLM + Pydantic schema + EVA Intent Catalog (YAML/JSON)
    • Business Value: Structured request understanding eliminates routing ambiguity

    1.3 Ambiguity Resolver¶

    • Detects underspecified/conflicting/incomplete messages
    • Generates clarifying questions (works for user‚Üíagent AND agent‚Üíagent)
    • Maintains clarification memory (conversation history + user/agent profile)
    • Critical Insight: Agent-to-agent communication also has ambiguity (e.g., "get me similar cases" - what similarity metric?)
    • Business Value: Dramatically increases accuracy in enterprise settings by eliminating guesswork

    1.5 Kill Switch / Command Center (Star Trek Bridge Model)¶

    • Global alert levels: NORMAL / YELLOW / RED / HARD_STOP
    • Command Center sets alert level ‚Üí Router injects into all messages ‚Üí Agents obey
    • Heartbeats: Each agent publishes to eva.heartbeats every N seconds (status: OK/BUSY/DEGRADED, current task)
    • Graceful Shutdown: Agents check control.alert_level on message receive and in internal loops
    • Rule: "If alert >= RED, you MUST stop X and NEVER do Y" (encoded in persona contracts + pattern rules)
    • Business Value: Circuit breaker for collective - instant kill switch across 40+ products

    2. EVA Awareness + Memory Protocol (P16 + Arxiv 2504.20084) (Sprint 20-30+)¶

    Status: üîµ PARKED (activates with EACP) Priority: FOUNDATIONAL (safety/governance layer for agent autonomy) Effort: ~80 story points (4 months) Source: - to upload/to upload/EVA Awareness Protocol v1-arxiv-2504-20084-model.md (comprehensive spec) - to upload/to upload/Memmory an Awareness plan.md (empty - placeholder for memory architecture)

    Strategic Context: - Awareness Protocol defines how every EVA agent knows who it is, where it is, what it's allowed to do - Monitors reasoning quality, confidence, performs pre-action safety checks - Emits structured awareness events for audit/ATO evidence - Must be implemented as orchestrator-level behavior so all EVA apps get it "for free" - Memory Protocol (TBD) defines short-term (session), long-term (cross-project), entity memory (stakeholders, definitions)

    Core Concepts:

    2.1 Four Awareness Dimensions¶

    1. Meta-cognition: "How sure am I? Did I check my work?"
    2. confidenceScore (0.0-1.0), perceivedRisk (low/medium/high), riskNotes

    3. Reflection trigger: if confidenceScore < 0.6 or perceivedRisk === "high" ‚Üí internal reflection step (second model call)

    4. Self-awareness: "Who am I? Which app/model/version? What are my limits?"

    5. appId, agentId, modelId, version, capabilities, constraints

    6. Situational awareness: "Which environment? Which tenant? What autonomy level?"

    7. environment (sandbox/pilot/prod), tenantId, autonomyLevel (C0/C1/C2/C3), dataSensitivity (Public/ProtectedA/B/C+)

    8. Social awareness: "Who is the user? What role? What expectations and policies apply?"

    9. userId, userRole, userDisplayName, RBAC scope, policies (e.g., ["no_personal_data", "no_prod_writes"])

    2.2 Autonomy Levels (C0-C3) - First-Class Citizens¶

    • C0 (Read-Only Assistant): No tool calls with side effects, only reads internal data
    • C1 (Guided Tool User): Mild side effects (search, retrieval, calculations), no state mutation
    • C2 (Controlled Actor): State-changing actions under policy + confirmation (draft configs, create tickets, propose SQL but not execute)
    • C3 (High-Impact Actor): Rare, heavy guardrails, explicit human approval, watchdog monitoring (disabled/constrained for EVA v1)

    2.3 Pre-Action Boundary Check¶

    Before ANY tool call/external action: 

    export interface BoundaryCheckResult {
  allowed: boolean;
  reason?: string;
  requireConfirmation?: boolean;  // e.g. show to user first
}

    Rules (examples): - If autonomyLevel === "C0" ‚Üí no tools allowed - If autonomyLevel === "C1" ‚Üí only tools marked readOnly: true - If autonomyLevel === "C2" ‚Üí state-changing tools allowed but requireConfirmation = true - If environment === "prod" and tool flagged as dangerous ‚Üí allowed = false unless explicit override - If data sensitivity & tool scope conflict (Protected B to external system not approved) ‚Üí block

    2.4 Awareness Middleware Architecture¶

    1. Request In (user message + tenant + app + environment + autonomy + tools)
   ‚Üì
2. Awareness Middleware (Pre-Model)
   - Build AwarenessContext
   - Run evaluateAwareness() (lightweight check)
   - Construct Awareness System Prompt Header
   ‚Üì
3. Model Call
   - System prompt = [Awareness Header] + [App/System Prompt]
   - Model returns answer + AWARENESS_META JSON block
   ‚Üì
4. Awareness Middleware (Post-Model)
   - Parse AWARENESS_META (confidence, risk, escalation)
   - Emit AwarenessEvent log entry
   - If model proposes tool calls ‚Üí Pre-Action Boundary Check
   ‚Üì
5. Action Execution or Escalation
   - Enforce C-level limits and policies
   - If blocked/escalated ‚Üí return explanation to user
