    CDD: Bilingual Quality Evaluation Framework¶

    Last updated: 2025-11-15
    Source: docs/EVA-2.0/ESDC/EVA Backlog/EVA Backlog/CDD Bilingual Quality Evaluation Framework.docx
    Status: Backlog
    Priority: HIGH

    Vision¶

    Build comprehensive evaluation framework for bilingual (en-US/fr-FR) AI assistant quality, ensuring:

    • Linguistic accuracy and cultural appropriateness
    • Consistent quality across both languages
    • Official Languages Act compliance
    • Automated quality gates in CI/CD

    Key Evaluation Dimensions¶

    1. Linguistic Quality¶

    • Grammar & Syntax: Correct sentence structure, verb conjugation, agreement
    • Vocabulary Precision: Domain-appropriate terminology, no anglicisms in French
    • Tone & Register: Formal government communication style
    • Idiomatic Expressions: Natural phrasing, avoid literal translations

    2. Content Quality¶

    • Factual Accuracy: Responses grounded in authoritative sources
    • Completeness: All relevant information provided
    • Relevance: Answers directly address user's question
    • Citations: Proper attribution to source documents

    3. Bilingual Parity¶

    • Semantic Equivalence: Same meaning in both languages
    • Response Length: Similar detail level (not word-for-word)
    • Cultural Adaptation: Context-appropriate examples and references
    • Availability: Feature parity (no English-only features)

    4. Safety & Compliance¶

    • Content Moderation: No harmful, biased, or inappropriate content
    • Privacy: No PII leakage or unauthorized data access
    • Accessibility: WCAG 2.1 AA compliance for all outputs
    • Legal: Aligned with ESDC policies and OLA requirements

    User Stories¶

    US-1: As a Dev Master, I want automated quality checks in CI/CD so that I catch bilingual issues before deployment¶

    Acceptance Criteria: - LLM-as-judge runs on every PR with sample test cases - Grammar checks validate both en-US and fr-FR responses - Bilingual parity score computed (semantic similarity >0.85) - PR comment shows quality report with pass/fail status

    US-2: As a Translation Bureau Reviewer, I want to validate EVA's French quality against TERMIUM Plus so that we maintain government standards¶

    Acceptance Criteria: - Terminology validator flags non-TERMIUM Plus terms - Review UI shows side-by-side en-US/fr-FR responses - Reviewer can approve/reject responses with feedback - Feedback loop updates prompts and grounding documents

    US-3: As a Product Owner, I want quality trend dashboards so that I can monitor bilingual quality over time¶

    Acceptance Criteria: - Application Insights dashboard shows quality metrics (accuracy, parity, CSAT) - Trends visible by sprint, repo, and language - Alerts trigger if quality drops below 4.0/5.0 threshold - Drill-down to individual responses for investigation

    US-4: As an End User, I want consistent quality in both languages so that my experience is seamless regardless of language choice¶

    Acceptance Criteria: - CSAT >4.0/5.0 in both en-US and fr-FR - Response times within 10% variance between languages - Feature parity (no English-only features) - Cultural appropriateness (no literal translations or anglicisms)

    US-5: As a Security Sentinel, I want to ensure no OLA violations reach production so that ESDC stays compliant¶

    Acceptance Criteria: - Zero OLA violations detected in production (monthly audit) - Automated checks block deployments with critical violations - Audit logs track all bilingual quality checks - Compliance reports generated quarterly for leadership

    Functional Requirements¶

    FR-1: LLM-as-Judge Evaluation¶

    System SHALL use GPT-4 or Claude to rate responses on 1-5 scale for accuracy, relevance, tone, completeness, and bilingual parity.

    FR-2: Grammar & Spell Checking¶

    System SHALL integrate LanguageTool to validate grammar, spelling, and syntax for both en-US and fr-FR responses.

    FR-3: Terminology Validation¶

    System SHALL validate terminology against TERMIUM Plus database and flag non-approved terms for review.

    FR-4: Semantic Similarity¶

    System SHALL compute semantic similarity between en-US and fr-FR responses using embeddings (target: >0.85).

    FR-5: Readability Scoring¶

    System SHALL compute readability scores (Flesch-Kincaid, Coleman-Liau) to ensure accessible language (target: Grade 8-10).

    FR-6: Response Length Parity¶

    System SHALL track response length ratio between languages (French typically 15-25% longer) and flag anomalies.

    FR-7: Feature Parity Checks¶

    System SHALL validate that all API endpoints, UI components, and features are available in both languages.

    FR-8: Quality Gates¶

    System SHALL block PR merges if automated quality checks fall below defined thresholds (e.g., 4.0/5.0 for accuracy).

    FR-9: Quality Reporting¶

    System SHALL generate quality reports in PR comments showing pass/fail status, scores, and flagged issues.

    FR-10: Quality Dashboards¶

    System SHALL publish quality metrics to Application Insights for trend analysis and alerting.

    FR-11: Review UI¶

    System SHALL provide evaluation UI for expert reviewers to assess responses side-by-side (en-US/fr-FR).

    FR-12: Feedback Loop¶

    System SHALL capture reviewer feedback and use it to improve prompts, grounding documents, and evaluation criteria.

    FR-13: Test Dataset Management¶

    System SHALL maintain bilingual test dataset with ground truth answers and quality annotations (100+ question pairs).

    FR-14: Cultural Appropriateness¶

    System SHALL flag literal translations, anglicisms, and culturally inappropriate content for review.

    FR-15: Compliance Audits¶

    System SHALL generate quarterly compliance reports tracking OLA adherence, WCAG violations, and PII leakage.

    Non-Functional Requirements¶

    Performance¶

    • Automated quality checks complete in <5 minutes per PR
    • LLM-as-judge responses return in <10 seconds per evaluation
    • Semantic similarity computation <2 seconds per response pair
    • Quality dashboard queries return in <3 seconds

    Reliability¶

    • 99.9% uptime for evaluation pipeline (CI/CD integration)
    • Automated retries for transient LLM API failures
    • Graceful degradation if TERMIUM Plus API unavailable (log and continue)

    Security¶

    • Evaluation data encrypted at rest (AES-256) and in transit (TLS 1.3)
    • PII detection and masking in test datasets
    • Role-based access control (RBAC) for review UI
    • Audit logs for all quality checks and reviewer actions

    Accessibility¶

    • Review UI compliant with WCAG 2.1 AA
    • Keyboard navigation for all evaluation workflows
    • Screen reader support for quality reports
    • High-contrast mode for dashboards

    i18n¶

    • All UI text in evaluation tools bilingual (en-US/fr-FR)
    • ICU formatting for dates, numbers, currencies in reports
    • Right-to-left (RTL) support not required (French is LTR)

    Observability¶

    • 100% of quality checks logged to Application Insights
    • Distributed tracing for evaluation pipeline (OpenTelemetry)
    • Alerts for quality threshold violations
    • Custom metrics: quality scores, parity scores, review feedback

    Scalability¶

    • Support 1000+ evaluations per day (all repos PRs)
    • Parallel execution of quality checks (10+ concurrent)
    • Incremental evaluation (only test cases affected by PR changes)

    Evaluation Approach¶

    Automated Metrics (CI/CD)¶

    LLM-as-Judge:

    • Use GPT-4 or Claude to evaluate responses on 1-5 scale
    • Prompt: "Rate this response for accuracy, relevance, tone, and completeness"
    • Run on every commit with sample test cases

    Linguistic Checks:

    • Grammar checkers: LanguageTool (open-source)
    • Terminology validation: Approved term database (TERMIUM Plus)
    • Readability scores: Flesch-Kincaid, Coleman-Liau

    Bilingual Parity:

    • Semantic similarity: Embeddings distance between en-US and fr-FR responses
    • Response length ratio: French should be 15-25% longer (typical)
    • Feature coverage: API endpoints available in both languages

    Human Evaluation (Periodic)¶

    Expert Review:

    • ESDC translation bureau review (quarterly)
    • Subject matter experts (SMEs) for domain accuracy
    • Accessibility specialists for WCAG validation

    User Testing:

    • A/B testing with bilingual users
    • CSAT surveys in both languages
    • Session recordings and feedback analysis

    Implementation Phases¶

    Phase 1: Test Dataset Creation¶

    • Build bilingual test dataset (100+ question pairs)
    • Cover all EVA use cases (benefits finder, HR assistant, etc.)
    • Include edge cases (slang, acronyms, ambiguous queries)
    • Annotate with ground truth answers and quality scores

    Phase 2: Automated Evaluation Pipeline¶

    • Integrate LLM-as-judge in CI/CD (Azure DevOps)
    • Configure LanguageTool for grammar checks
    • Build terminology validator (TERMIUM Plus API)
    • Compute semantic similarity for bilingual parity

    Phase 3: Quality Gates¶

    • Set minimum thresholds (e.g., 4.0/5.0 for accuracy)
    • Block deployments if thresholds not met
    • Generate quality reports in PR comments
    • Track quality trends over time (Application Insights)

    Phase 4: Human-in-the-Loop¶

    • Build evaluation UI for expert reviewers
    • Schedule quarterly reviews with translation bureau
    • Collect user feedback (CSAT, thumbs up/down)
    • Analyze feedback to improve prompts and grounding

    Success Criteria¶

    • Automated quality checks run in <5 minutes per PR
    • 90% of responses pass automated thresholds (4.0/5.0)
    • Bilingual parity score >0.85 (semantic similarity)
    • Zero Official Languages Act violations in production
    • CSAT >4.0/5.0 in both languages
    • 100% terminology aligned with TERMIUM Plus

    Dependencies¶

    • eva-i18n: Catalog management, language routing
    • eva-openai: Prompt templates for bilingual generation
    • eva-rag: Bilingual document ingestion and retrieval
    • eva-seed: Bilingual test datasets and ground truth
    • AI Toolkit: Evaluation planner and code gen tools
    • Azure DevOps: CI/CD pipeline integration
    • TERMIUM Plus API: Approved terminology database

    Tools & Technologies¶

    • Azure AI Evaluation SDK: LLM-as-judge, semantic similarity
    • LanguageTool: Open-source grammar/spell checker
    • TERMIUM Plus: Government of Canada terminology database
    • Flesch-Kincaid: Readability scoring
    • pytest: Test framework with bilingual fixtures
    • Application Insights: Quality metrics dashboarding

    Open Questions¶

    1. Which LLM for judging? GPT-4, Claude, or fine-tuned model?
    2. How to handle Quebec French vs. European French nuances?
    3. Translation bureau bandwidth: Can they review quarterly or need more time?
    4. Should we enforce same response length or allow natural variation?
    5. How to measure cultural appropriateness quantitatively?
