    CDD: EVA OS Security Sandbox¶

    Last Updated: 2025-11-19 Status: Backlog Priority: LOW (Moonshot) Epic: EVA OS Integration (Windows Copilot+ Replacement)

    Vision¶

    Enforce strict tenant boundary isolation, Data Loss Prevention (DLP), and comprehensive audit logging for EVA OS integration—ensuring that ambient context capture, cross-app workflows, and always-on AI assistance meet enterprise security standards without compromising user privacy, enabling ESDC to deploy EVA OS features with confidence in regulated environments.

    Business Value: Enable enterprise adoption of OS-integrated AI (meet security requirements), protect sensitive data (prevent leakage across tenants), ensure compliance (PIPEDA, SOC 2, FedRAMP), and establish EVA as the most secure OS-integrated AI assistant for government and regulated industries.

    Core Capabilities¶

    • Tenant Boundary Enforcement: Per-user context isolation (no cross-tenant data access)
    • Data Loss Prevention (DLP): Detect and block sensitive data exposure (PII, SINs, credentials)
    • Content Security Policy (CSP): Sandboxed execution for add-ins, extensions (no XSS, injection)
    • Audit Logging: Comprehensive traceability (who accessed what, when, from where)
    • Encryption: End-to-end encryption for context data (TLS 1.3, AES-256)
    • Role-Based Access Control (RBAC): Granular permissions (admin, user, guest)
    • Anomaly Detection: Behavioral monitoring (detect suspicious activity, auto-block)
    • Compliance Reporting: Pre-built reports (PIPEDA, SOC 2, FedRAMP)

    User Stories¶

    US-1: As a security admin, I want tenant isolation so that users cannot access other tenants' data¶

    Acceptance Criteria:

    • User A (Tenant ESDC) cannot see context from User B (Tenant CRA)
    • Context data tagged with tenant ID (all queries, logs, cached data)
    • API requests include tenant validation (JWT token, Entra ID claim)
    • Cross-tenant queries rejected (403 Forbidden error)
    • Audit log: All cross-tenant access attempts logged (suspicious activity)

    US-2: As a compliance officer, I want DLP enforcement so that sensitive data is not exposed¶

    Acceptance Criteria:

    • EVA detects PII in clipboard (SINs, emails, phone numbers, credit cards)
    • Proactive warning: "Clipboard contains sensitive data. Proceed?" (user confirmation)
    • Admin policy: Block PII copy-paste (no user override, high-security orgs)
    • Audit log: All DLP events logged (user ID, data type, action taken)
    • DLP rules customizable (admin console: define patterns, block/warn/log)

    US-3: As a user, I want my context data encrypted so that my privacy is protected¶

    Acceptance Criteria:

    • Context data encrypted in transit (TLS 1.3, Azure API Management)
    • Context data encrypted at rest (AES-256, Azure Key Vault)
    • No plaintext storage (temporary files, logs, caches)
    • Key rotation: Automatic (90-day rotation, Azure Key Vault managed)
    • Encryption status: Visible in EVA settings ("Data encrypted: ✓")

    US-4: As an auditor, I want comprehensive logs so that I trace all AI interactions¶

    Acceptance Criteria:

    • Audit log includes: user ID, tenant ID, query text, timestamp, IP address, action type
    • Log retention: 1 year (configurable: 1-7 years)
    • Logs immutable (append-only, no deletion except by admin)
    • Export logs: CSV, JSON (compliance reporting)
    • Search logs: Filter by user, date range, action type (admin console)

    US-5: As a security admin, I want anomaly detection so that I detect suspicious activity¶

    Acceptance Criteria:

    • EVA monitors user behavior (query frequency, data access patterns)
    • Anomaly detected: "User accessed 100+ files in 5 minutes" (alert triggered)
    • Auto-block: Temporary account suspension (1 hour cooldown)
    • Admin notification: Email alert with details (user ID, activity, timestamp)
    • False positive handling: Admin can whitelist users (disable anomaly detection per user)

    Functional Requirements¶

    1. FR-1: SHALL enforce tenant boundary isolation (per-user context, no cross-tenant access)
    2. FR-2: SHALL tag all data with tenant ID (queries, logs, cached data)
    3. FR-3: SHALL validate tenant on API requests (JWT token, Entra ID claim)
    4. FR-4: SHALL detect PII in context data (SINs, emails, phone numbers, credit cards)
    5. FR-5: SHALL enforce DLP policies (block, warn, or log sensitive data exposure)
    6. FR-6: SHALL encrypt context data in transit (TLS 1.3) and at rest (AES-256)
    7. FR-7: SHALL store encryption keys in Azure Key Vault (managed service)
    8. FR-8: SHALL rotate encryption keys automatically (90-day rotation)
    9. FR-9: SHALL log all AI interactions to Cosmos DB (append-only, immutable)
    10. FR-10: SHALL retain audit logs for 1 year (configurable: 1-7 years)
    11. FR-11: SHALL provide audit log export (CSV, JSON for compliance reporting)
    12. FR-12: SHALL monitor user behavior for anomalies (query frequency, data access)
    13. FR-13: SHALL auto-block suspicious activity (temporary suspension, cooldown)
    14. FR-14: SHALL provide admin console (configure DLP rules, RBAC, anomaly detection)
    15. FR-15: SHALL generate compliance reports (PIPEDA, SOC 2, FedRAMP)

    • Tenant Validation: <50ms (API request → tenant verified)
    • DLP Detection: <100ms (context capture → PII detected)
    • Encryption Overhead: <5% (performance impact vs. unencrypted)
    • Audit Logging: <10ms (event → log written, async)

    Reliability¶

    • Tenant Isolation: 100% (zero cross-tenant leakage)
    • DLP Accuracy: >99% (PII detection, low false positives)
    • Encryption: 100% (all data encrypted, no exceptions)
    • Audit Log Durability: 99.999% (Cosmos DB SLA)

    Security¶

    • Tenant Boundary: 100% isolation (no cross-tenant data access)
    • DLP Enforcement: >99% (sensitive data blocked/warned)
    • Encryption: TLS 1.3 (transit), AES-256 (rest), Azure Key Vault (keys)
    • Audit Logging: Append-only, immutable, 1-year retention (configurable)

    • Alerts: Cross-tenant access attempt, DLP block rate >5%, encryption failure, anomaly detection triggered
    • Dashboards: Power BI dashboard for security events, DLP stats, audit log summaries

    • Concurrent Users: Support 10K+ users per tenant (multi-tenant scale)
    • Audit Log Volume: Handle 1M+ log entries/day (Cosmos DB partitioning)
    • DLP Throughput: Handle 10K+ context captures/minute (real-time detection)

    Technical Architecture¶

    Tenant Isolation¶

    Tenant Validation Middleware:

    // Azure Functions HTTP trigger
public class TenantValidationMiddleware {
  public async Task<IActionResult> Run(HttpRequest req) {
    // Extract JWT token
    var token = req.Headers["Authorization"].FirstOrDefault()?.Split(" ").Last();
    var claims = DecodeJwtToken(token);

    // Validate tenant ID
    var tenantId = claims["tid"]; // Entra ID tenant claim
    var requestedTenant = req.Query["tenantId"];

    if (tenantId != requestedTenant) {
      // Log cross-tenant access attempt
      await LogSecurityEvent(new SecurityEvent {
        Type = "cross_tenant_access",
        UserId = claims["oid"],
        RequestedTenant = requestedTenant,
        ActualTenant = tenantId,
        Timestamp = DateTime.UtcNow
      });

      return new ForbiddenResult("Cross-tenant access denied");
    }

    // Proceed with request
    return await ProcessRequest(req, tenantId);
  }
}

    Data Loss Prevention¶

    PII Detection:

    // Regex patterns for PII
private static readonly Dictionary<string, Regex> PIIPatterns = new() {
  { "SIN", new Regex(@"\b\d{3}-\d{3}-\d{3}\b") }, // Canadian SIN
  { "Email", new Regex(@"\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b") },
  { "Phone", new Regex(@"\b\d{3}-\d{3}-\d{4}\b") },
  { "CreditCard", new Regex(@"\b\d{4}-\d{4}-\d{4}-\d{4}\b") }
};

public class DLPService {
  public DLPResult DetectPII(string content) {
    var detected = new List<string>();

    foreach (var pattern in PIIPatterns) {
      if (pattern.Value.IsMatch(content)) {
        detected.Add(pattern.Key);
      }
    }

    return new DLPResult {
      HasPII = detected.Any(),
      DetectedTypes = detected,
      Action = GetDLPAction(detected) // Block, Warn, or Log
    };
  }
}

    Implementation Phases¶

    Phase 1: Tenant Isolation (Sprint 27, Week 1-3)¶

    • Tenant validation middleware (JWT token, Entra ID claim)
    • Tenant ID tagging (all data: queries, logs, context)
    • Cross-tenant access prevention (API validation, 403 errors)
    • Audit logging (all tenant validation events)
    • Admin console (view tenant isolation status)

    Phase 2: Data Loss Prevention (Sprint 28, Week 4-6)¶

    • PII detection (SINs, emails, phone numbers, credit cards)
    • DLP policies (block, warn, log actions)
    • Admin console (configure DLP rules, custom patterns)
    • User warnings (clipboard contains sensitive data, proceed?)
    • DLP audit logging (all events: detected, blocked, warned)

    Phase 3: Encryption & Key Management (Sprint 29, Week 7-9)¶

    • TLS 1.3 enforcement (all API traffic)
    • AES-256 encryption (context data at rest)
    • Azure Key Vault integration (key storage, rotation)
    • Automatic key rotation (90-day cycle)
    • Encryption status UI (user settings: "Data encrypted: ✓")

    Phase 4: Anomaly Detection & Compliance (Sprint 30, Week 10-12)¶

    • Behavioral monitoring (query frequency, data access patterns)
    • Anomaly detection (high-frequency queries, suspicious activity)
    • Auto-block (temporary suspension, cooldown)
    • Compliance reports (PIPEDA, SOC 2, FedRAMP)
    • Power BI dashboard for security events, DLP stats, anomalies

    Success Criteria¶

    • ✅ Tenant isolation 100% (zero cross-tenant leakage)

    • ✅ Encryption 100% (all data encrypted)
    • ✅ Audit log durability 99.999% (no data loss)
    • ✅ Anomaly detection >90% (suspicious activity caught)
