    GC Agentic Framework - Comprehensive Summary¶

    Date: December 10, 2025
    Repository: eva-orchestrator
    Status: Documentation Complete

    üéØ Executive Summary¶

    The Government of Canada (GC) Agentic AI Framework is a comprehensive, technology-neutral template system that enables federal departments to design, build, and govern AI agents safely and responsibly. This repository contains extensive documentation across 5 volumes (~2,600+ lines) plus implementation examples through the EVA 2.0 platform.

    üèóÔ∏è Key Concepts from GC Agentic Framework¶

    Autonomy Levels (3 Tiers)¶

    1. Advisory
    2. Retrieves and explains information
    3. Drafts neutral/descriptive content

    4. MUST NOT recommend or decide actions

    5. Assistive

    6. Proposes recommendations and next steps
    7. Creates drafts requiring review

    8. Human MUST review and approve before use

    9. Semi-Autonomous

    10. Performs low-risk, reversible tasks
    11. Operates within clearly defined constraints
    12. MUST NOT make final decisions on rights, benefits, or obligations

    Lifecycle Tiers (3 Stages)¶

    1. Sandbox - Experimental; low-risk data only; clearly labeled
    2. Pilot - Limited audience; documented governance; monitored
    3. Production - Full integration; complete governance evidence

    Project Space Concept¶

    A logical container defining: - Business purpose and objectives - Data sources and security classification - Available tools/APIs and RAG indices - Role-based access control (RBAC) - Governance artefact links (privacy assessments, AI impact assessments)

    Core Platform Components (Abstract)¶

    1. Foundation Services - RAG, tool registry, agent runtime, safety, metering, API gateway
    2. Assistant UIs - General chat, domain-specific assistants
    3. LiveOps & Monitoring - Health dashboards, kill switches, autonomy management
    4. Directory/Registry - Projects, agents, tools, policies, approvals
    5. Shared Services - Auth, i18n, accessibility, notifications, logging, metrics

    üéØ Requirements Tagging System¶

    The framework uses two tags for all requirements:

    • [CORE] - Recommended minimum for any department; foundational capabilities
    • [ADV] - Advanced capabilities for mature teams; future roadmap items

    Example CORE Requirements: - Human-in-the-loop design - Governed & auditable behavior - WCAG 2.1 AA accessibility - Bilingual support (EN/FR) - Kill switch capability - Logging and traceability - Project space boundaries

    Example ADV Requirements: - Multi-deployment faces (web, embedded, API) - Configurable agent memory (user/project level) - Trust posture calculation - Agent catalogue with cloning - Replay of execution traces

    üîí Compliance & Governance Features¶

    Security & Privacy¶

    • Operates within approved GC security classifications
    • Data residency and encryption aligned with GC standards
    • RBAC for all key functions
    • Versioned model tracking per interaction

    Accessibility & Official Languages¶

    • WCAG 2.1 AA minimum (WCAG 2.2 AAA for GC components)
    • Full English and French UI support
    • Personalization controls (contrast, font size, motion)

    Observability & Audit¶

    • Structured logs consumable by departmental monitoring
    • Correlation IDs across requests, agent steps, tool calls
    • Configuration and model version change tracking
    • Exportable reports for oversight bodies

    üìñ How Departments Can Use This¶

    5-Step Quick Start (from Volume 0)¶

    Step 1: Instantiate the Template - Fill instantiation headers (department, platform name, governance bodies) - Decide which [CORE] items to adopt immediately - Identify [ADV] items for future roadmap

    Step 2: Map to Your Governance - Map template roles to real org roles - Connect to existing AI review committees - Link to ITSM/ITIL workflows - Document Sandbox ‚Üí Pilot ‚Üí Production mapping

    Step 3: Choose Starter Use Cases - Select 2-3 low-to-medium risk internal use cases - Examples: Knowledge Explorer, Drafting Assistant, Case Summarizer - Fill Agent Concept Canvas for each - Define autonomy level and lifecycle tier

    Step 4: Stand Up Platform - Map existing components to Volume 1 requirements - Or define minimal viable platform (chat UI + RAG + logging) - Implement directory/registry (even simple database + admin UI) - Enable monitoring and kill switches

    Step 5: Run Pilots & Iterate - Set simple SLOs - Enable monitoring - Define incident workflows - Collect feedback - Decide: scale, adjust, or retire

    üí° Key Insights¶

    Technology-Neutral Design¶

    The GC agentic framework is deliberately cloud/vendor-neutral, allowing departments to: - Choose their own implementation stack - Integrate with existing systems - Avoid platform lock-in - Adapt to evolving technologies

    Human Accountability First¶

    Every aspect emphasizes: - Human-in-the-loop design - Clear autonomy boundaries - Explainable AI decisions - Traceable actions - Kill switch availability

    Risk-Based Approach¶

    Progressive maturity through: - Lifecycle tiers (Sandbox ‚Üí Pilot ‚Üí Production) - Autonomy levels (Advisory ‚Üí Assistive ‚Üí Semi-Autonomous) - Gated promotions with documented evidence - Continuous monitoring and adjustment

    Federal-Grade Standards¶

    Built for Government of Canada requirements: - Security classifications (Protected B capable) - Bilingual operation (EN/FR) - WCAG 2.1 AA accessibility minimum - Privacy and audit compliance - Policy alignment
