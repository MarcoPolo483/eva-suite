    CDD: EVA Vision Accessibility Features¶

    Last Updated: 2025-11-19 Status: Backlog Priority: LOW (Moonshot) Epic: EVA Vision (Power BI Plugin)

    Vision¶

    Transform Power BI dashboards into fully accessible AI-powered experiences for users with disabilities—providing screen reader-optimized narration, voice control, keyboard navigation, high-contrast modes, and customizable accessibility preferences, ensuring EVA Vision achieves WCAG 2.1 AAA compliance and enables equitable access to data insights across all user abilities.

    Business Value: Unlock dashboard access for 15% of workforce with disabilities (WHO estimate), meet legal accessibility requirements (AODA, Section 508), enhance user experience for all (keyboard power users, mobile users), and position EVA as the most accessible enterprise BI assistant.

    Core Capabilities¶

    • Screen Reader Optimization: Structured narration of dashboards (ARIA labels, semantic HTML)
    • Voice Control: Hands-free navigation and data queries (no mouse/touch required)
    • Keyboard Navigation: Full functionality via keyboard shortcuts (no mouse required)
    • High-Contrast Modes: Visual themes for low vision users (dark mode, light mode, custom)
    • Zoom & Magnification: Support for screen magnifiers (ZoomText, Windows Magnifier)
    • Customizable Narration: Adjustable speed, verbosity, voice gender/accent
    • Alternative Text: Auto-generated descriptions for all visual elements (charts, tables, KPIs)
    • Focus Management: Logical tab order, visible focus indicators, skip links

    User Stories¶

    US-1: As a blind user, I want screen reader-optimized narration so that I understand dashboards independently¶

    Acceptance Criteria:

    • Open Power BI dashboard → screen reader announces: "Sales Dashboard loaded, 5 visual elements"
    • Tab through elements → screen reader narrates: "KPI card, Total Sales, $5 million, up 12% year-over-year"
    • Charts described semantically: "Bar chart, Revenue by Region, 4 bars. Canada: $1.5M, highest. USA: $1.2M..."
    • ARIA landmarks for navigation (main, navigation, complementary regions)
    • Tested with JAWS, NVDA, VoiceOver (99% compatibility)

    US-2: As a user with motor disabilities, I want full keyboard control so that I navigate without a mouse¶

    Acceptance Criteria:

    • All EVA Vision features accessible via keyboard (no mouse required)
    • Keyboard shortcuts: Alt+E (open EVA chat), Tab (navigate), Enter (activate), Escape (close)
    • Logical tab order: KPIs → charts → filters → EVA chat widget
    • Visible focus indicators (high-contrast outline, 3px border)
    • Skip links: "Skip to main content", "Skip to EVA chat"

    US-3: As a low vision user, I want high-contrast themes so that I see visuals clearly¶

    Acceptance Criteria:

    • EVA Vision respects Power BI theme (dark mode, light mode)
    • High-contrast mode (Windows High Contrast theme detected, custom colors applied)
    • Text: Minimum 4.5:1 contrast ratio (WCAG AA), 7:1 for AAA compliance
    • Visual elements: High-contrast borders, backgrounds (charts visible in all modes)
    • User preference saved (persists across sessions)

    US-4: As a user with cognitive disabilities, I want simplified narration so that I understand insights easily¶

    Acceptance Criteria:

    • Narration verbosity levels: Brief (key numbers only), Standard (default), Detailed (full context)
    • Plain language: Avoid jargon, use simple sentences (e.g., "Sales went up 12%" vs. "YoY CAGR: 12%")
    • Chunked information: Break long narrations into short segments (max 2 sentences per chunk)
    • Visual indicators: Progress bars for multi-step queries (avoid confusion)
    • Retry/undo: "Repeat that" command (repeat last narration)

    US-5: As an accessibility tester, I want WCAG AAA compliance so that the product meets legal requirements¶

    Acceptance Criteria:

    • WCAG 2.1 AAA compliance verified (automated + manual audit)
    • Accessibility statement published (detailed conformance report)
    • Keyboard-only testing passed (all features accessible without mouse)
    • Screen reader testing passed (JAWS, NVDA, VoiceOver, TalkBack)
    • Color blindness simulation tested (Protanopia, Deuteranopia, Tritanopia)

    Functional Requirements¶

    1. FR-1: SHALL provide screen reader-optimized narration (ARIA labels, semantic HTML, live regions)
    2. FR-2: SHALL support full keyboard navigation (all features accessible via keyboard)
    3. FR-3: SHALL implement logical tab order (KPIs → charts → filters → EVA chat)
    4. FR-4: SHALL provide visible focus indicators (high-contrast outline, 3px border)
    5. FR-5: SHALL support high-contrast themes (Windows High Contrast, custom themes)
    6. FR-6: SHALL ensure text contrast ≥7:1 (WCAG AAA) in high-contrast mode
    7. FR-7: SHALL generate alternative text for all visual elements (charts, tables, KPIs)
    8. FR-8: SHALL support voice control (hands-free navigation, data queries)
    9. FR-9: SHALL allow narration customization (speed, verbosity, voice gender/accent)
    10. FR-10: SHALL support screen magnifiers (ZoomText, Windows Magnifier, browser zoom)
    11. FR-11: SHALL provide skip links ("Skip to main content", "Skip to EVA chat")
    12. FR-12: SHALL implement ARIA landmarks (main, navigation, complementary, search)
    13. FR-13: SHALL announce dynamic content updates (ARIA live regions, polite announcements)
    14. FR-14: SHALL log accessibility usage to Application Insights (feature adoption tracking)
    15. FR-15: SHALL publish accessibility statement (WCAG conformance report)

    Non-Functional Requirements¶

    Performance¶

    • Screen Reader Narration: <500ms (element focus → announcement)
    • Keyboard Navigation: <100ms (keypress → focus change)
    • High-Contrast Mode: <200ms (theme switch → UI update)
    • Alternative Text Generation: <2s (chart → description)

    Reliability¶

    • Screen Reader Compatibility: >99% (JAWS, NVDA, VoiceOver, TalkBack)
    • Keyboard Navigation: 100% feature coverage (no mouse-only features)
    • High-Contrast Themes: Support all Windows High Contrast themes
    • Focus Management: No focus loss (always visible, logical order)

    Security¶

    • Data Privacy: Accessibility logs anonymized (no PII, aggregated metrics only)
    • Authorization: Accessibility features respect RLS (users only access authorized data)
    • Audit Logging: Accessibility feature usage logged to Cosmos DB (feature adoption tracking)

    Accessibility¶

    • WCAG 2.1 AAA: Full compliance (automated + manual audit, accessibility statement published)
    • Screen Readers: Tested with JAWS, NVDA, VoiceOver, TalkBack (99% compatibility)
    • Keyboard Navigation: All features accessible via keyboard (logical tab order, skip links)
    • High-Contrast: Text contrast ≥7:1, visual elements visible in all themes
    • Narration: Adjustable speed (0.5x - 2x), verbosity (brief, standard, detailed)

    Internationalization¶

    • Locales: en-US (default), fr-FR
    • Screen Reader Languages: Support English and French narration
    • Voice Control: Bilingual commands (code-switching supported)

    Observability¶

    • Metrics: Application Insights tracks accessibility feature usage, screen reader sessions, keyboard-only sessions
    • Alerts: Screen reader errors >1%, keyboard navigation failures >0.5%
    • Dashboards: Power BI dashboard for accessibility adoption, feature usage, error trends

    Scalability¶

    • Concurrent Users: Support 100+ simultaneous accessibility sessions (screen reader, keyboard-only)
    • Narration Volume: Handle 1K+ narration requests/day (TTS, alternative text generation)

    Technical Architecture¶

    Frontend¶

    • Semantic HTML: Use native HTML elements (button, input, nav, main) for screen reader compatibility
    • ARIA Attributes: Labels, roles, states, live regions (aria-label, aria-describedby, aria-live)
    • Keyboard Shortcuts: Global hotkeys (Alt+E, Tab, Enter, Escape) using KeyboardEvent API
    • Focus Management: Focus trap in modals, restore focus on close, visible focus indicators
    • High-Contrast CSS: Media queries (@media (prefers-contrast: high)), custom color palettes

    Backend¶

    • Alternative Text Generation: Azure OpenAI GPT-4 Vision (chart descriptions)
    • Text-to-Speech: Azure Speech Services (Neural TTS, adjustable speed/voice)
    • Accessibility Testing: Automated tests (axe-core, Lighthouse), manual audits (screen reader testing)

    Accessibility Features¶

    Screen Reader Optimization:

    <!-- KPI Card Example -->
<div role="region" aria-labelledby="total-sales-label">
  <h2 id="total-sales-label">Total Sales</h2>
  <p aria-live="polite">$5 million, up 12% year-over-year</p>
</div>

<!-- Chart Example -->
<div role="img" aria-labelledby="chart-title" aria-describedby="chart-desc">
  <h3 id="chart-title">Revenue by Region</h3>
  <p id="chart-desc">Bar chart showing revenue for 4 regions. Canada leads with $1.5 million, followed by USA with $1.2 million, UK with $800,000, and France with $600,000.</p>
  <svg><!-- Chart visualization --></svg>
</div>

    Keyboard Navigation:

    // Keyboard shortcuts
document.addEventListener('keydown', (e) => {
  if (e.altKey && e.key === 'e') {
    openEvaChat(); // Alt+E to open EVA chat
  }
  if (e.key === 'Escape') {
    closeEvaChat(); // Escape to close
  }
  if (e.key === 'Tab') {
    manageFocus(); // Logical tab order
  }
});

    Implementation Phases¶

    Phase 1: Core Accessibility (Sprint 13, Week 1-3)¶

    • Semantic HTML refactor (button, input, nav, main elements)
    • ARIA labels and roles (all interactive elements)
    • Keyboard navigation (full feature coverage, logical tab order)
    • Visible focus indicators (high-contrast outline, 3px border)
    • Screen reader testing (JAWS, NVDA, VoiceOver initial compatibility)

    Phase 2: High-Contrast & Themes (Sprint 13, Week 4-6)¶

    • High-contrast mode (Windows High Contrast theme detection)
    • Custom color palettes (dark, light, high-contrast themes)
    • Text contrast ≥7:1 (WCAG AAA compliance)
    • Visual element visibility (charts, borders in high-contrast)
    • User preference persistence (theme saved across sessions)

    Phase 3: Advanced Features (Sprint 14, Week 7-9)¶

    • Alternative text generation (Azure OpenAI GPT-4 Vision for charts)
    • Narration customization (speed, verbosity, voice gender/accent)
    • Voice control integration (hands-free navigation, data queries)
    • ARIA live regions (dynamic content announcements)
    • Skip links ("Skip to main content", "Skip to EVA chat")

    Phase 4: Compliance & Documentation (Sprint 14, Week 10-12)¶

    • WCAG 2.1 AAA audit (automated: axe-core, Lighthouse; manual: screen reader testing)
    • Accessibility statement (detailed conformance report)
    • User guides (accessibility features, keyboard shortcuts)
    • Power BI dashboard for accessibility adoption, feature usage
    • Accessibility training for dev team (best practices, testing)

    Success Criteria¶

    • ✅ WCAG 2.1 AAA compliance (automated + manual audit passed)
    • ✅ Screen reader compatibility >99% (JAWS, NVDA, VoiceOver, TalkBack)
    • ✅ Keyboard-only navigation 100% (all features accessible without mouse)
    • ✅ Text contrast ≥7:1 (high-contrast mode)
    • ✅ User adoption >50% of users with accessibility needs within 6 months
    • ✅ Accessibility statement published (conformance report)
    • ✅ User satisfaction >85% (accessibility user survey)
