    CDD: EVA OS System Service¶

    Last Updated: 2025-11-19 Status: Backlog Priority: LOW (Moonshot) Epic: EVA OS Integration (Windows Copilot+ Replacement)

    Vision¶

    Deploy EVA as a native Windows system service running in the background, continuously capturing user context (active windows, clipboard, files) and providing always-on AI assistance—replacing Windows Copilot with an enterprise-grade, tenant-aware AI that proactively offers insights, suggestions, and automation without requiring explicit user invocation.

    Business Value: Enable ambient AI assistance (reduce time to insight by 90%), capture contextual knowledge passively (no manual input required), establish EVA as the OS-integrated AI layer for Windows 11+ enterprises, and deliver Microsoft's Copilot+ vision with ESDC's data sovereignty and customization.

    Core Capabilities¶

    • Background Process: Windows service running at startup (system-level, not user-level)
    • Context Capture Engine: Monitor active windows, clipboard, file system, browser tabs
    • Always-On Listening: Detect user intent from activity patterns (no explicit commands required)
    • Proactive Suggestions: Offer insights based on context (e.g., "Summarize this email?")
    • Low Resource Footprint: <50MB RAM, <5% CPU usage (idle state)
    • Secure Data Handling: All context data encrypted in transit and at rest
    • Tenant Isolation: Per-user tenant boundaries (no cross-tenant context leakage)
    • Graceful Degradation: Fallback to manual mode if background capture fails

    User Stories¶

    US-1: As a user, I want EVA to run in the background so that I get assistance without opening an app¶

    Acceptance Criteria:

    • Windows startup → EVA system service auto-starts (no user action)
    • Service running: EVA icon appears in system tray (notification area)
    • No visible UI (service runs silently unless user interacts)
    • Resource usage: <50MB RAM, <5% CPU (idle)
    • Service restarts on crash (auto-recovery, Windows Service Control Manager)

    US-2: As a user, I want EVA to understand my current context so that I get relevant suggestions¶

    Acceptance Criteria:

    • User switches to Outlook → EVA detects context: "Reading email from John about Q4 budget"
    • Proactive suggestion appears (notification badge): "Summarize this email?"
    • Click suggestion → EVA provides summary (within 3 seconds)
    • Context persists across app switches (last 5 activities tracked)
    • Privacy: Context not logged (in-memory only, cleared on logout)

    US-3: As an admin, I want to configure EVA's context capture so that I balance utility vs. privacy¶

    Acceptance Criteria:

    • Admin console: Configure capture settings (clipboard: on, file system: off, browser tabs: on)
    • User-level overrides: Users can disable specific capture types (Clipboard: off)
    • Opt-out: Users can disable all context capture (EVA still works, manual queries only)
    • Audit log: All capture settings changes logged (who, when, what changed)
    • Compliance: PIPEDA-compliant data handling (consent, transparency, deletion)

    US-4: As a developer, I want to monitor EVA service health so that I detect issues quickly¶

    Acceptance Criteria:

    • Service status: Windows Event Viewer logs startup, shutdown, errors
    • Health endpoint: /health API returns service status (running, stopped, degraded)
    • Metrics: Application Insights tracks CPU, memory, context capture events, errors
    • Alerts: CPU >10%, memory >100MB, error rate >1%
    • Diagnostic mode: Enable verbose logging (troubleshoot crashes, performance issues)

    US-5: As a user, I want EVA to start only when I log in so that I control when it's active¶

    Acceptance Criteria:

    • User preference: "Start EVA on login" toggle (default: on)
    • If disabled: EVA does not auto-start, user opens EVA manually (desktop shortcut)
    • User logout → EVA stops (context cleared, no background process)
    • Fast user switching: EVA runs per-user (User A's EVA doesn't see User B's context)
    • Resource cleanup: Memory freed on logout (no leaks)

    Functional Requirements¶

    1. FR-1: SHALL run as Windows system service (auto-start on boot, system-level privileges)
    2. FR-2: SHALL capture user context (active windows, clipboard, file system, browser tabs)
    3. FR-3: SHALL consume <50MB RAM and <5% CPU in idle state
    4. FR-4: SHALL encrypt all captured context data in transit (TLS 1.3) and at rest (AES-256)
    5. FR-5: SHALL enforce tenant isolation (per-user context boundaries, no cross-tenant leakage)
    6. FR-6: SHALL provide proactive suggestions based on captured context
    7. FR-7: SHALL support admin-configurable capture settings (clipboard, file system, browser)
    8. FR-8: SHALL allow user-level overrides (opt-out of specific capture types)
    9. FR-9: SHALL log service events to Windows Event Viewer (startup, shutdown, errors)
    10. FR-10: SHALL expose health endpoint (/health) for monitoring
    11. FR-11: SHALL restart automatically on crash (Windows Service Control Manager)
    12. FR-12: SHALL clear context data on user logout (in-memory only, no persistence)
    13. FR-13: SHALL support fast user switching (per-user EVA instances)
    14. FR-14: SHALL log all context capture events to Application Insights (anonymized metadata only)
    15. FR-15: SHALL provide diagnostic mode (verbose logging for troubleshooting)

    Non-Functional Requirements¶

    Performance¶

    • Startup Time: <3s (service start → ready to capture context)
    • Context Capture: <100ms (activity detected → context updated)
    • Proactive Suggestion: <2s (context change → suggestion displayed)
    • Resource Usage: <50MB RAM, <5% CPU (idle), <10% CPU (active)

    Reliability¶

    • Service Uptime: 99.9% (aligned with Windows session uptime)
    • Auto-Recovery: Service restarts on crash (max 3 retries, 5-minute backoff)
    • Graceful Degradation: Manual mode if background capture fails (EVA still usable)
    • Data Integrity: Context data not corrupted (validation on capture)

    Security¶

    • Data Encryption: TLS 1.3 (in transit), AES-256 (at rest, temporary context storage)
    • Tenant Isolation: Per-user context boundaries (no cross-tenant access)
    • Least Privilege: Service runs with minimum required permissions (no admin rights)
    • Audit Logging: All context capture events logged to Cosmos DB (user ID, activity type, timestamp, 1-year retention)

    Accessibility¶

    • WCAG 2.1 AA: System tray icon keyboard-accessible (Alt+Space to open)
    • Screen Reader Support: Service status announced (e.g., "EVA service running")
    • High Contrast: System tray icon respects Windows theme (dark/light mode)

    Internationalization¶

    • Locales: en-US (default), fr-FR
    • Service Name: "EVA Assistant Service" (localized in Windows Services)
    • Error Messages: Translated (e.g., "Service failed to start" → "Échec du démarrage du service")

    Observability¶

    • Metrics: Application Insights tracks CPU, memory, context capture events, errors, suggestion clicks
    • Alerts: CPU >10%, memory >100MB, error rate >1%, crash detected
    • Dashboards: Power BI dashboard for service health, context capture stats, error trends

    Scalability¶

    • Concurrent Users: Support 10K+ users per Windows Server (VDI scenario)
    • Context Volume: Handle 1K+ context captures/hour per user (peak activity)
    • Resource Limits: Auto-throttle if memory >100MB (pause context capture, resume when below 80MB)

    Technical Architecture¶

    Service Implementation¶

    • Language: C# (.NET 8) for Windows service compatibility
    • Service Host: Windows Service Control Manager (SCM)
    • IPC: Named pipes for EVA UI ↔ service communication (low latency, secure)
    • Context Capture: Win32 API hooks (SetWinEventHook for window focus, clipboard monitoring)

    Context Capture¶

    Active Window Monitoring:

    // Hook into window focus changes
SetWinEventHook(
  EVENT_SYSTEM_FOREGROUND,
  EVENT_SYSTEM_FOREGROUND,
  IntPtr.Zero,
  WinEventCallback,
  0, 0,
  WINEVENT_OUTOFCONTEXT
);

void WinEventCallback(IntPtr hWinEventHook, uint eventType, IntPtr hwnd, ...) {
  // Capture window title, process name, application context
  string title = GetWindowText(hwnd);
  string processName = GetProcessName(hwnd);

  // Send context to EVA API (encrypted)
  SendContextUpdate(new Context {
    Type = "window_focus",
    Application = processName,
    Title = title,
    Timestamp = DateTime.UtcNow
  });
}

    Clipboard Monitoring:

    // Monitor clipboard changes
AddClipboardFormatListener(hwnd);

// On clipboard update
string clipboardText = Clipboard.GetText();
if (clipboardText.Length > 0 && clipboardText.Length < 10000) {
  // Send to EVA (PII filtering applied)
  SendContextUpdate(new Context {
    Type = "clipboard",
    Content = FilterPII(clipboardText),
    Timestamp = DateTime.UtcNow
  });
}

    Implementation Phases¶

    Phase 1: Core Service Infrastructure (Sprint 18, Week 1-3)¶

    • Windows service skeleton (C# .NET 8, SCM registration)
    • Service lifecycle management (start, stop, restart, crash recovery)
    • Health endpoint (/health API, Windows Event Viewer logging)
    • Resource monitoring (CPU, memory, throttling logic)
    • IPC channel (named pipes for UI ↔ service communication)

    Phase 2: Context Capture (Sprint 19, Week 4-6)¶

    • Active window monitoring (Win32 API hooks, window focus events)
    • Clipboard monitoring (clipboard format listener, PII filtering)
    • File system monitoring (deferred to Sprint 20, out of scope for MVP)
    • Context encryption (TLS 1.3, AES-256)
    • Tenant isolation (per-user context boundaries)

    Phase 3: Proactive Suggestions (Sprint 20, Week 7-9)¶

    • Context-to-suggestion engine (Azure OpenAI GPT-4o)
    • Proactive notification system (Windows toast notifications)
    • Suggestion relevance scoring (ML model, click-through rate optimization)
    • User feedback loop (thumbs up/down, improve suggestions)
    • Suggestion caching (reduce API calls for common contexts)

    Phase 4: Admin Console & Compliance (Sprint 21, Week 10-12)¶

    • Admin console (configure capture settings, user overrides)
    • User opt-out mechanism (disable context capture per user)
    • Audit logging (all capture events to Cosmos DB)
    • PIPEDA compliance (consent, transparency, data deletion)
    • Power BI dashboard for service health, capture stats, suggestion clicks

    Success Criteria¶

    • ✅ Service startup <3s (service start → ready)
    • ✅ Resource usage <50MB RAM, <5% CPU (idle state)
    • ✅ Context capture <100ms (activity → context updated)
    • ✅ Service uptime 99.9% (aligned with Windows session uptime)
    • ✅ User adoption >40% of Windows users within 6 months
    • ✅ Suggestion click-through rate >20% (proactive suggestions)
    • ✅ PIPEDA compliance (consent, transparency, data deletion)
