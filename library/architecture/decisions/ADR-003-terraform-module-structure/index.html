    ADR-003: Terraform Module Structure¶

    Status¶

    Accepted

    Context¶

    EVA 2.0 infrastructure consists of multiple Azure services (networking, Key Vault, Cosmos DB, Search, OpenAI) that must be: - Reusable across environments (dev, stg, prd) - Testable in isolation - Composable (modules depend on outputs from other modules) - Maintainable by small team (1-2 infrastructure engineers)

    Terraform offers several module organization patterns: 1. Monolithic: Single main.tf with all resources 2. Flat Modules: One module per service (e.g., networking, key-vault) 3. Nested Modules: Sub-modules for granular resources (e.g., networking/vnet, networking/nsg) 4. Module Registry: Publish modules to private Terraform registry

    Decision¶

    Use Flat Module Composition pattern with environment-specific wrappers.

    Directory Structure¶

    eva-infra/
├── modules/
│   ├── networking/
│   │   ├── main.tf
│   │   ├── README.md
│   ├── key-vault/
│   │   ├── main.tf
│   │   ├── README.md
│   ├── azure-monitor/
│   │   ├── main.tf
│   │   ├── README.md
│   └── cosmos-db/ (future)
├── env/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── backend.tf
│   │   ├── variables.tf
│   │   ├── terraform.tfvars
│   ├── stg/
│   └── prd/

    Module Principles¶

    1. Single Responsibility: Each module manages one logical service group
    2. No Nested Modules: Avoid modules/networking/vnet/main.tf complexity
    3. Outputs for Composition: Modules expose outputs consumed by dependent modules
    4. Minimal Variables: Only expose configurable parameters (no over-parameterization)
    5. Self-Contained: Each module includes README.md with usage examples

    Rationale¶

    Why Flat Modules?¶

    • Simplicity: Team of 1-2 can understand entire structure in <30 minutes
    • Testability: Each module can be tested independently with terraform plan
    • Reusability: Environments call same modules with different variables
    • Avoiding Over-Engineering: No need for nested modules until team scales >5 engineers

    Why Not Monolithic?¶

    • Difficult to test individual services (must plan entire stack)
    • No reusability across environments
    • High blast radius for changes (any edit risks entire stack)

    Why Not Nested Modules?¶

    • Over-engineering for current team size
    • Adds indirection (harder to debug)
    • Longer development cycles (more files to manage)

    Why Not Module Registry?¶

    • Overkill for single-team, single-project setup
    • Adds deployment step (publish module before use)
    • No immediate versioning needs (team controls all environments)

    Consequences¶

    Positive¶

    • ✅ Clear separation of concerns (one module = one Azure service)
    • ✅ Easy to test modules in isolation
    • ✅ Simple composition in environment wrappers
    • ✅ Readable code (no deep nesting)
    • ✅ Fast iteration (edit module, apply to dev)

    Negative¶

    • ❌ Potential duplication if modules grow large (mitigate with locals)
    • ❌ No versioning of modules (mitigate with Git tags)
    • ❌ Tight coupling between modules via outputs (mitigate with data sources if needed)

    Mitigation¶

    • Use locals in modules to reduce duplication (e.g., naming patterns)
    • Tag releases in Git (v1.0.0-networking) for module versioning
    • Document module dependencies in README files
    • Use terraform graph to visualize dependencies

    Implementation Examples¶

    Module: Key Vault¶

    # modules/key-vault/main.tf
resource "azurerm_key_vault" "main" {
  name                = "kv-${var.name_prefix}-001"
  location            = var.location
  resource_group_name = var.resource_group_name
  # ...
}

output "key_vault_id" {
  value = azurerm_key_vault.main.id
}

    Environment: dev¶

    # env/dev/main.tf
module "networking" {
  source = "../../modules/networking"
  name_prefix = "eva-dev"
  location    = "canadacentral"
}

module "key_vault" {
  source = "../../modules/key-vault"
  name_prefix         = "eva-dev"
  subnet_id           = module.networking.subnet_ids["pe"]
  private_dns_zone_id = module.networking.private_dns_zone_ids["vault"]
}

module "azure_monitor" {
  source = "../../modules/azure-monitor"
  name_prefix  = "eva-dev"
  key_vault_id = module.key_vault.key_vault_id
}

    Module Dependency Graph¶

    graph TD
    A[networking] --> B[key-vault]
    A --> C[cosmos-db]
    A --> D[azure-search]
    E[azure-monitor] --> B
    E --> C
    E --> D
    B --> F[app-service]
    C --> F
    D --> F

    Module Development Workflow¶

    1. Create new module directory: modules/cosmos-db/
    2. Define main.tf with resources, variables, outputs
    3. Create README.md with usage examples
    4. Test module in dev environment: 
      cd env/dev
terraform init
terraform plan
    5. Update env/dev/main.tf to call new module
    6. Commit module + environment changes together
    7. Open PR, review plan output in GitHub Actions
    8. Merge to main → auto-deploy to dev
