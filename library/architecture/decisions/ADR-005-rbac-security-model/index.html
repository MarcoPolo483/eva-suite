    ADR-005: RBAC Security Model¶

    Status¶

    Accepted

    Context¶

    EVA 2.0 requires fine-grained access control for: - Azure Resources: Who can create, modify, delete infrastructure (Key Vault, Storage, Cosmos) - Secrets Management: Who can read, write, delete secrets in Key Vault - Application Identities: How apps authenticate to Azure services (Managed Identity vs. Service Principal) - User Access: How engineers, operators, and auditors access Azure Portal, CLI, Terraform

    Traditional access control methods: 1. Shared Access Keys: Storage/Cosmos access keys (high risk: key leakage, no audit trail) 2. Access Policies: Key Vault access policies (deprecated, limited granularity) 3. RBAC (Role-Based Access Control): Azure AD-based, role assignments, audit logs

    Decision¶

    Use Azure AD RBAC as the exclusive access control mechanism for all EVA 2.0 resources.

    Core Principles¶

    1. No Access Keys: Disable access keys for Storage, Cosmos DB, Service Bus
    2. No Access Policies: Use Key Vault RBAC, not legacy access policies
    3. Managed Identities: All app-to-service authentication (no service principal keys)
    4. Least Privilege: Grant minimum permissions required for each role
    5. Audit Logs: Track all role assignments and access attempts via Azure Monitor

    Rationale¶

    Why RBAC?¶

    1. Centralized: Single control plane (Azure AD) for all permissions
    2. Granular: Scope permissions to subscription, resource group, or individual resource
    3. Auditable: All access logged to Azure Monitor (who, what, when)
    4. Key-less: No secrets to rotate or leak (Azure AD tokens)
    5. Revocable: Disable user/identity → immediate access revocation across all resources

    Why Not Access Keys?¶

    • ❌ Key leakage risk (logs, CI/CD pipelines, Git commits)
    • ❌ No audit trail (who used key? when?)
    • ❌ Manual rotation required (operational burden)
    • ❌ Blast radius: One key = full access to service

    Why Not Access Policies (Key Vault)?¶

    • ❌ Deprecated by Microsoft (RBAC is recommended)
    • ❌ Limited granularity (20 permissions vs. 100+ RBAC roles)
    • ❌ No resource group/subscription-level inheritance
    • ❌ Difficult to audit (must query each Key Vault individually)

    Consequences¶

    Positive¶

    • ✅ Zero hardcoded secrets (no keys in app config, CI/CD pipelines)
    • ✅ Fine-grained permissions (e.g., "read secrets" vs. "manage secrets")
    • ✅ Centralized audit trail (Azure Monitor tracks all access)
    • ✅ Automatic revocation (disable identity → immediate loss of access)
    • ✅ Easier compliance audits (export RBAC assignments via Azure Policy)

    Negative¶

    • ❌ Complexity: Team must understand Azure AD, role definitions, scope
    • ❌ Debugging: Failed access requires checking role assignments, conditions, scope
    • ❌ Local dev: Engineers need Azure AD authentication (no quick access keys)
    • ❌ CI/CD setup: Must configure OIDC or Managed Identity (more setup vs. access keys)

    Mitigation¶

    • Document RBAC setup in onboarding guide for new engineers
    • Create troubleshooting runbook for "403 Forbidden" errors
    • Use Azure RBAC analyzer tool to visualize permissions
    • Provide pre-configured OIDC setup script for GitHub Actions

    RBAC Role Assignments¶

    Engineers (Platform Team)¶

    Role Scope Justification
    Contributor Resource Group: rg-eva-dev-* Create, modify, delete resources in dev
    Reader Subscription View all resources, quotas, cost reports
    Key Vault Administrator Resource Group: rg-eva-dev-net Manage secrets, keys, certificates in dev
    Storage Blob Data Contributor Storage Account: stevatfstatecac Read/write Terraform state

    Applications (eva-api, eva-rag)¶

    Identity Role Scope Justification
    id-eva-api-dev Key Vault Secrets User Key Vault: kv-eva-dev-001 Read OpenAI API keys, connection strings
    id-eva-api-dev Cosmos DB Built-in Data Contributor Cosmos DB: cosmos-eva-dev-001 Read/write chat history, user context
    id-eva-rag-dev Search Index Data Contributor Search: srch-eva-dev-001 Create/update search indices
    id-eva-rag-dev Storage Blob Data Contributor Storage: steva-dev-001 Read RAG documents, write embeddings

    CI/CD Pipeline (GitHub Actions)¶

    Identity Role Scope Justification
    sp-eva-infra-github Contributor Subscription Terraform plan/apply all resources
    sp-eva-infra-github Storage Blob Data Contributor Storage: stevatfstatecac Read/write Terraform state
    sp-eva-infra-github Key Vault Administrator All Key Vaults Provision initial secrets during deployment

    Auditors (Security Team)¶

    Role Scope Justification
    Reader Subscription View all resources, configurations
    Security Reader Subscription View security alerts, recommendations
    Log Analytics Reader Log Analytics: law-eva-dev-001 Query audit logs, diagnostic logs

    Implementation¶

    Enable RBAC for Key Vault¶

    resource "azurerm_key_vault" "main" {
  name                = "kv-eva-dev-001"
  enable_rbac_authorization = true  # ✅ RBAC mode
  # Do NOT set access_policy blocks
}

    Disable Access Keys for Storage¶

    resource "azurerm_storage_account" "main" {
  name                      = "steva-dev-001"
  shared_access_key_enabled = false  # ✅ Disable access keys
}

    Grant Application Access to Key Vault¶

    resource "azurerm_user_assigned_identity" "eva_api" {
  name                = "id-eva-api-dev"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location
}

resource "azurerm_role_assignment" "api_kv_secrets" {
  scope                = azurerm_key_vault.main.id
  role_definition_name = "Key Vault Secrets User"
  principal_id         = azurerm_user_assigned_identity.eva_api.principal_id
}

    Grant GitHub Actions Access (OIDC)¶

    # Create Azure AD app registration
az ad app create --display-name "eva-infra-github-actions"

# Create service principal
az ad sp create --id <app-id>

# Assign Contributor role
az role assignment create \
  --assignee <app-id> \
  --role Contributor \
  --scope /subscriptions/<subscription-id>

# Configure federated credential for OIDC
az ad app federated-credential create \
  --id <app-id> \
  --parameters '{
    "name": "eva-infra-main",
    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:MarcoPolo483/eva-infra:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
  }'

    Application Code Example (Managed Identity)¶

    // eva-api authenticating to Key Vault
import { DefaultAzureCredential } from "@azure/identity";
import { SecretClient } from "@azure/keyvault-secrets";

const credential = new DefaultAzureCredential();  // ✅ Auto-discovers Managed Identity
const vaultUrl = "https://kv-eva-dev-001.vault.azure.net";
const client = new SecretClient(vaultUrl, credential);

const secret = await client.getSecret("OPENAI_API_KEY");
console.log(secret.value);  // ✅ No hardcoded keys

    Security Best Practices¶

    Least Privilege¶

    • Grant narrowest scope possible (resource-level > resource group > subscription)
    • Use time-bound roles via Azure Privileged Identity Management (PIM) for temporary access
    • Review role assignments quarterly, revoke unused permissions

    Managed Identity Preference¶

    1. User-Assigned Managed Identity (preferred): Reusable across multiple resources
    2. System-Assigned Managed Identity: Tied to single resource (deleted with resource)
    3. Service Principal with OIDC: For CI/CD (no client secrets)
    4. Service Principal with Client Secret: Last resort (rotate every 90 days)

    Audit & Monitoring¶

    • Enable Azure Monitor diagnostic settings for all Key Vaults, Storage, Cosmos
    • Send logs to Log Analytics workspace
    • Create alerts for:
    • New role assignments (unexpected privilege escalation)
    • Failed authentication attempts (brute force, misconfigured identity)
    • Secret access from unexpected IPs (potential compromise)

    Troubleshooting RBAC Issues¶

    Error: "403 Forbidden" when accessing Key Vault¶

    # Verify role assignment
az role assignment list \
  --assignee <identity-object-id> \
  --scope <key-vault-resource-id>

# Expected output: "Key Vault Secrets User" or "Key Vault Administrator"

    Error: "The client does not have permission to perform action"¶

    1. Verify identity has correct role (use az role assignment list)
    2. Check scope (resource-level > resource group > subscription)
    3. Wait 5 minutes (RBAC propagation delay)
    4. Verify identity is active (not deleted, expired, or disabled)

    Error: "Terraform backend authentication failed"¶

    # Verify Storage Blob Data Contributor role
az role assignment list \
  --assignee <sp-object-id> \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-eva-tfstate-cac/providers/Microsoft.Storage/storageAccounts/stevatfstatecac

# Grant role if missing
az role assignment create \
  --assignee <sp-object-id> \
  --role "Storage Blob Data Contributor" \
  --scope <storage-account-resource-id>

    Future Enhancements¶

    Azure Privileged Identity Management (PIM)¶

    • Just-in-time access for production (engineers request temporary Contributor role)
    • Time-bound assignments (auto-expire after 8 hours)
    • Approval workflow for sensitive roles (Key Vault Administrator in prd)

    Conditional Access Policies¶

    • Require MFA for privileged role activations
    • Block access from untrusted locations (non-corporate IPs)
    • Require managed devices (Intune-enrolled laptops only)

    Azure Policy for RBAC Enforcement¶

    • Auto-deny creation of Storage Accounts with access keys enabled
    • Auto-deny Key Vaults without RBAC authorization
    • Alert on role assignments outside approved list
