    CDD: EVA Analytics Command Center Dashboard¶

    Last Updated: 2025-11-19 Status: Backlog Priority: HIGH Epic: EVA Analytics

    Vision¶

    Provide a centralized command center dashboard for platform health monitoring with real-time metrics covering API performance, infrastructure status, error tracking, and SLA compliance—enabling operations teams to proactively identify and resolve issues before they impact users.

    Business Value: Reduce MTTR (Mean Time To Resolution) by 50%, prevent 90% of outages through early warning, maintain 99.9% uptime SLA, and improve operational confidence.

    Core Capabilities¶

    • Platform Health: Overall system status (green/yellow/red), component health checks
    • API Performance: Real-time latency (P50, P95, P99), throughput, error rates
    • Infrastructure Metrics: Azure service status, resource utilization (CPU, memory, storage)
    • Error Tracking: Top errors by frequency, error rate trends, stack traces
    • SLA Compliance: Uptime %, availability zones, incident history
    • Dependency Monitoring: External service health (OpenAI, Cognitive Search, Cosmos DB)
    • Capacity Planning: Resource usage trends, scaling recommendations, forecast alerts
    • Incident Management: Live incident feed, severity classification, response playbooks

    User Stories¶

    US-1: As an ops engineer, I want real-time health status so that I detect issues fast¶

    Acceptance Criteria:

    • Dashboard shows overall status: GREEN (all systems operational), YELLOW (degraded), RED (outage)
    • Component status for each service: eva-api, eva-openai, eva-rag, eva-auth (green/yellow/red)
    • Real-time updates (auto-refresh every 30 seconds)
    • Click component to see detailed metrics
    • Mobile-responsive for on-call alerts

    US-2: As a platform admin, I want API performance metrics so that I identify bottlenecks¶

    Acceptance Criteria:

    • Dashboard shows: API latency (P50, P95, P99), requests/second, error rate %
    • Latency chart (last 24 hours, line graph)
    • Identify slow endpoints (>2 second P95)
    • Drill-down to specific API routes and error logs
    • Set alert thresholds (P95 >3 seconds)

    US-3: As a DevOps lead, I want infrastructure metrics so that I plan capacity¶

    Acceptance Criteria:

    • Dashboard shows: Azure App Service CPU/memory, Cosmos DB RU/s, Azure Search QPS
    • Resource utilization trends (7-day, 30-day)
    • Forecast when capacity limits reached (predict 80% threshold)
    • Suggest scaling actions (scale up, scale out, optimize queries)
    • Export capacity plan report (Excel)

    US-4: As an SRE, I want error tracking so that I prioritize fixes¶

    Acceptance Criteria:

    • Dashboard shows: Top 10 errors by frequency, error rate % trend
    • Group errors by type: 4xx (client errors), 5xx (server errors)
    • Click error to see stack trace, affected users, timestamps
    • Link to GitHub Issues for known bugs
    • Set alert for error rate >5%

    US-5: As a manager, I want SLA compliance so that I report to executives¶

    Acceptance Criteria:

    • Dashboard shows: Monthly uptime % (99.9% target), incident count, MTTR
    • Historical SLA compliance (12-month trend)
    • Identify SLA breaches (downtime >43 minutes/month)
    • Export SLA report (PDF, bilingual)
    • Compare actual vs. SLA commitment

    Functional Requirements¶

    FR-1: Health Checks¶

    System SHALL perform health checks every 30 seconds for all components (eva-api, eva-openai, eva-rag, eva-auth, eva-metering).

    FR-2: API Metrics¶

    System SHALL track API latency (P50, P95, P99), throughput (requests/second), error rates (%) per endpoint using Application Insights.

    FR-3: Infrastructure Metrics¶

    System SHALL collect Azure service metrics: App Service (CPU, memory), Cosmos DB (RU/s, storage), Azure Search (QPS, throttling).

    FR-4: Error Tracking¶

    System SHALL aggregate errors by type (4xx, 5xx), frequency, and affected users using Application Insights.

    FR-5: SLA Calculation¶

    System SHALL calculate monthly uptime %: (total minutes - downtime minutes) / total minutes × 100.

    FR-6: Dependency Monitoring¶

    System SHALL monitor external service health: Azure OpenAI (API status), Cognitive Search (throttling), Cosmos DB (latency).

    FR-7: Capacity Forecasting¶

    System SHALL forecast resource usage (CPU, RU/s, storage) using Azure Monitor Autoscale and ML models (Azure ML).

    FR-8: Incident Feed¶

    System SHALL integrate with incident management (Azure DevOps, PagerDuty) to display live incident feed with severity, owner, status.

    FR-9: Alerts¶

    System SHALL send alerts (Slack, email, SMS) for: health status RED, API latency >3s (P95), error rate >5%, SLA breach risk.

    FR-10: Drill-Down¶

    System SHALL provide drill-down: dashboard → component → endpoint → logs → stack trace.

    FR-11: Playbooks¶

    System SHALL link incidents to runbooks (e.g., "High API Latency Playbook") with step-by-step remediation.

    FR-12: Export Options¶

    System SHALL allow exporting dashboards as PDF, Excel with bilingual summaries.

    FR-13: Access Control¶

    System SHALL enforce RBAC: Ops Engineers see all, Developers see own services, Executives see SLA summary only.

    FR-14: Bilingual Support¶

    System SHALL provide dashboards in en-US and fr-FR with ICU formatting.

    FR-15: Audit Logging¶

    System SHALL log all dashboard views, alert triggers, and incident updates to Application Insights.

    Non-Functional Requirements¶

    Performance¶

    • Dashboard load time: <2 seconds (P95)
    • Real-time data refresh: <30 seconds
    • Health check response: <5 seconds
    • Support 500 concurrent dashboard users
    • Query response time: <1 second for metrics

    Reliability¶

    • Uptime: 99.99% SLA (self-monitoring system)
    • Auto-retry failed health checks (3 attempts)
    • Graceful degradation (static status page if backend down)
    • Backup metrics to Azure Storage (hourly snapshots)

    Security¶

    • RBAC: Ops see all, Developers see own, Executives see summary
    • Encrypt metrics at rest (AES-256) and in transit (TLS 1.3)
    • Audit log all dashboard views and alert triggers
    • Compliance: PIPEDA, TBS Directive on Privacy

    Accessibility¶

    • WCAG 2.1 AA compliance (dashboard UI)
    • Keyboard navigation for all interactive elements
    • Screen reader support (status announcements)
    • High-contrast mode for visuals

    i18n¶

    • Bilingual support (en-US/fr-FR)
    • ICU formatting for dates, numbers
    • Localized alert messages and playbooks
    • Language switcher on dashboard

    Observability¶

    • Application Insights custom events (dashboard viewed, alert sent)
    • Distributed tracing for health checks (OpenTelemetry)
    • Alerts for dashboard failures (meta-monitoring)
    • Metrics: dashboard views/day, alert count, MTTR

    Scalability¶

    • Horizontal scaling (Azure Monitor for metrics collection)
    • Partition storage by /tenantId (HPK)
    • Azure CDN for static dashboard assets
    • Support 100K events/minute (metrics ingestion)

    Technical Architecture¶

    Backend (Monitoring Engine)¶

    • Health Checks: Azure Functions (Timer trigger, 30-second interval)
    • Metrics Collection: Azure Monitor (Azure services), Application Insights (custom metrics)
    • Storage: Azure Data Explorer (ADX) for fast time-series queries
    • Alerting: Azure Monitor Alerts + Logic Apps (Slack, email, SMS)

    Frontend (Dashboard)¶

    • Visualization: Power BI Embedded (real-time dashboards)
    • Status Page: React SPA with WebSocket for real-time updates (SignalR)
    • Mobile: Progressive Web App (PWA) for on-call engineers

    Integration Points¶

    • Azure Monitor: Infrastructure metrics (CPU, memory, RU/s)
    • Application Insights: API metrics, error tracking
    • Azure DevOps: Incident management integration
    • PagerDuty: Alert escalation for critical incidents

    Data Models¶

    Health Check (Azure Data Explorer):

    HealthChecks
| where timestamp > ago(1h)
| where component in ("eva-api", "eva-openai", "eva-rag", "eva-auth")
| summarize status = iff(countif(isHealthy == false) > 0, "RED", "GREEN") by component

    API Metrics (Application Insights):

    requests
| where timestamp > ago(1h)
| summarize
    p50 = percentile(duration, 50),
    p95 = percentile(duration, 95),
    p99 = percentile(duration, 99),
    rps = count() / 3600,
    errorRate = countif(success == false) * 100.0 / count()
by name

    SLA Calculation (Azure Data Explorer):

    let totalMinutes = 44640; // 31 days
let downtime = HealthChecks
| where timestamp > ago(31d)
| where status == "RED"
| summarize sum(durationMinutes);
let uptime = (totalMinutes - downtime) * 100.0 / totalMinutes;
uptime

    Implementation Phases¶

    Phase 1: Health Checks (Sprint 6)¶

    • Azure Functions for health checks (30-second timer)
    • Azure Data Explorer (ADX) for metrics storage
    • Basic dashboard (overall status, component health)
    • WebSocket real-time updates (SignalR)
    • Mobile PWA

    Phase 2: API & Infrastructure Metrics (Sprint 7)¶

    • Application Insights integration (API metrics)
    • Azure Monitor integration (infrastructure metrics)
    • Latency, throughput, error rate dashboards
    • Resource utilization charts (CPU, memory, RU/s)
    • Drill-down to endpoint logs

    Phase 3: Alerts & Capacity (Sprint 8)¶

    • Azure Monitor Alerts (latency, error rate, health)
    • Logic Apps for Slack, email, SMS notifications
    • Capacity forecasting (Azure ML AutoML)
    • Scaling recommendations (Power BI visuals)
    • Incident feed integration (Azure DevOps)

    Phase 4: SLA & Polish (Sprint 9)¶

    • SLA calculation and compliance dashboard
    • Historical trends (12-month)
    • Playbook links for common incidents
    • WCAG 2.1 AA audit (dashboard UI)
    • Load testing (500 concurrent users)

    Success Criteria¶

    • ✅ Dashboard loads in <2 seconds (P95)
    • ✅ Real-time updates every 30 seconds
    • ✅ 50% reduction in MTTR (vs. manual monitoring)
    • ✅ 99.9% uptime SLA maintained (monthly)
    • ✅ Alerts sent within 1 minute of issue detection
    • ✅ WCAG 2.1 AA compliance (axe DevTools passes)
    • ✅ 90% of incidents resolved using playbooks

    Open Questions¶

    Q1: Self-Monitoring (RELIABILITY)¶

    Question: What if monitoring system itself fails? Impact: Blind spots during outages Recommendation: Deploy static status page on separate Azure region, external uptime monitor (Pingdom) Decision Owner: windowsGuy

    Q2: Alert Noise (UX)¶

    Question: Alert on every health check failure or wait for 3 consecutive? Impact: False alarms vs. delayed detection Recommendation: 3 consecutive failures for non-critical, immediate for critical components Decision Owner: ai2

    Q3: Metrics Retention (COST)¶

    Question: Store all metrics indefinitely or archive after 90 days? Impact: Cost vs. historical analysis Recommendation: Hot storage 90 days (ADX), cold storage 24 months (Azure Storage) Decision Owner: windowsGuy

    Q4: Forecast Accuracy (TECH)¶

    Question: Use linear regression or ARIMA for capacity forecasting? Impact: Accuracy vs. complexity Recommendation: Start with Azure ML AutoML (selects best model), manual tuning if needed Decision Owner: ai2

    Q5: Mobile Dashboard (SCOPE)¶

    Question: Full-featured mobile dashboard or status-only view? Impact: Feature parity vs. mobile UX Recommendation: Status-only mobile PWA (MVP), full dashboard Sprint 10 Decision Owner: UX Partner
