    CDD: EVA Portal Navigation & UX¶

    Last Updated: 2025-11-19 Status: Backlog Priority: HIGH Epic: Portal & Frontend

    Vision¶

    Design unified navigation and user experience patterns across all EVA products, ensuring consistent workflows, context preservation, and progressive disclosure. Establish EVA's design system as the foundation for all product interfaces—reducing cognitive load and accelerating user proficiency.

    Business Value: Improve usability by 40%, reduce training time, maintain brand consistency, and enable seamless transitions between EVA products.

    Core Capabilities¶

    • Unified Navigation: Consistent header, sidebar, breadcrumbs across all products
    • Context Preservation: Maintain user state when switching between products
    • Progressive Disclosure: Show advanced features only when needed
    • Design System: Reusable components (buttons, forms, cards, modals)
    • Responsive Breakpoints: Mobile-first design (320px to 1920px+)
    • Dark Mode: System-wide theme toggle
    • Keyboard Shortcuts: Power user productivity (Ctrl+K for search, etc.)
    • Toast Notifications: Non-intrusive feedback for user actions

    User Stories¶

    US-1: As a user, I want consistent navigation so that I can easily switch between EVA products¶

    Acceptance Criteria:

    • All EVA products share same header (logo, product switcher, profile, settings)
    • Sidebar navigation shows current product's features
    • Breadcrumbs display current location (Home > Chat > Session #123)
    • Navigation persists user preferences (collapsed sidebar, dark mode)
    • Transition between products takes <1 second

    US-2: As a user, I want context preservation so that my work isn't lost when switching products¶

    Acceptance Criteria:

    • Chat sessions persist when navigating to other products
    • DA search results remain available after switching to LiveOps
    • Draft content auto-saved every 30 seconds (forms, emails, documents)
    • Back button returns to previous state (not page reload)
    • Session storage tracks last visited pages per product

    US-3: As a designer, I want a design system so that I can build consistent interfaces quickly¶

    Acceptance Criteria:

    • Component library published (Storybook documentation)
    • Reusable components: Button, Card, Modal, Form, Table, Badge, Toast
    • Design tokens for colors, spacing, typography, shadows
    • Figma design files synced with code components
    • Accessibility baked into all components (WCAG 2.1 AA)

    US-4: As a power user, I want keyboard shortcuts so that I can navigate faster¶

    Acceptance Criteria:

    • Ctrl+K opens global search (fuzzy search across products)
    • Ctrl+/ shows keyboard shortcuts help modal
    • Tab navigation works consistently across all forms
    • Escape closes modals, cancels operations
    • Custom shortcuts per product (e.g., Ctrl+N for new chat)

    US-5: As a mobile user, I want responsive design so that I can use EVA on any device¶

    Acceptance Criteria:

    • Navigation collapses to hamburger menu on mobile (<768px)
    • Touch-friendly tap targets (≥44px)
    • Swipe gestures for navigation (left/right to switch tabs)
    • Mobile-optimized forms (large inputs, native date pickers)
    • Offline mode indicators ("No internet connection")

    Functional Requirements¶

    FR-1: Unified Header¶

    All EVA products SHALL share unified header with logo, product switcher, notifications, profile, settings, language toggle.

    FR-2: Sidebar Navigation¶

    Products SHALL use sidebar navigation with collapsible sections, icons, labels, and active state indicators.

    FR-3: Breadcrumbs¶

    Pages SHALL display breadcrumbs showing navigation path (Home > Product > Feature > Detail).

    FR-4: Context Preservation¶

    System SHALL preserve user context (chat sessions, search results, drafts) when switching between products via session storage.

    FR-5: Progressive Disclosure¶

    UI SHALL show basic features by default, with advanced options revealed on demand (e.g., "Show advanced filters").

    FR-6: Design System¶

    System SHALL provide component library (Storybook) with reusable UI components and design tokens.

    FR-7: Responsive Design¶

    UI SHALL adapt to mobile (<768px), tablet (768-1023px), desktop (≥1024px) breakpoints with mobile-first approach.

    FR-8: Dark Mode¶

    System SHALL support dark mode toggle with user preference persisted in profile.

    FR-9: Keyboard Shortcuts¶

    System SHALL provide global keyboard shortcuts (Ctrl+K search, Ctrl+/ help) and product-specific shortcuts.

    FR-10: Toast Notifications¶

    System SHALL display toast notifications for user actions (success, error, info, warning) with auto-dismiss after 5 seconds.

    FR-11: Loading States¶

    UI SHALL show loading spinners, skeleton screens, or progress bars for async operations (>500ms duration).

    FR-12: Empty States¶

    UI SHALL display helpful empty states with illustrations and next actions (e.g., "No chats yet. Start a conversation").

    FR-13: Error Handling¶

    System SHALL display user-friendly error messages (bilingual) with actionable next steps.

    FR-14: Accessibility¶

    All UI components SHALL comply with WCAG 2.1 AA (keyboard navigation, screen reader, color contrast ≥4.5:1).

    FR-15: Performance¶

    UI SHALL achieve Lighthouse performance score ≥90 and Time to Interactive (TTI) <3 seconds.

    Non-Functional Requirements¶

    Performance¶

    • Page load time: <2 seconds (P95)
    • Time to Interactive (TTI): <3 seconds
    • Lighthouse performance score: ≥90
    • Component render time: <16ms (60 FPS)
    • Bundle size: <300KB gzipped

    Reliability¶

    • Uptime: 99.9% SLA (design system hosted on CDN)
    • Graceful degradation if CDN unavailable (inline critical CSS)
    • Auto-save drafts every 30 seconds (prevent data loss)
    • Retry logic for failed API calls (3 attempts)

    Security¶

    • Content Security Policy (CSP) headers
    • XSS protection (sanitize user inputs)
    • HTTPS only (TLS 1.3)
    • OAuth 2.0/OIDC authentication (Entra ID)

    Accessibility¶

    • WCAG 2.1 AA compliance (verified via axe DevTools)
    • Keyboard navigation (Tab, Enter, Esc, Arrow keys)
    • Focus indicators visible (≥3:1 contrast ratio)
    • ARIA labels for dynamic content
    • High-contrast mode support
    • Screen reader tested (NVDA, JAWS)

    i18n¶

    • Bilingual support (en-US/fr-FR)
    • ICU formatting for dates, numbers, currencies
    • Right-to-left (RTL) support not required (French is LTR)
    • Design tokens for text direction (future RTL support)

    Observability¶

    • Application Insights custom events (page views, navigation, shortcuts)
    • Real User Monitoring (RUM) for performance metrics
    • Error tracking with stack traces (Sentry or App Insights)
    • Alerts for slow page loads, high error rates

    Scalability¶

    • CDN for static assets (Azure Front Door)
    • Component lazy loading (code splitting)
    • Image optimization (WebP, lazy loading)
    • Tree shaking (remove unused code)

    Technical Architecture¶

    Frontend Stack¶

    • Framework: React 18+ with TypeScript
    • Styling: Tailwind CSS + HeadlessUI (unstyled accessible components)
    • Component Library: Storybook (documentation + testing)
    • Routing: React Router v6 (nested routes, lazy loading)
    • State Management: React Context + React Query (server state)
    • Build: Vite (fast dev, optimized production)

    Design System¶

    • Tokens: colors, spacing, typography, shadows, radii (CSS custom properties)
    • Components: Button, Card, Modal, Form, Table, Badge, Toast, Dropdown, Tabs
    • Patterns: Navigation, breadcrumbs, empty states, loading states, error states
    • Documentation: Storybook published to Azure Static Web Apps

    Integration Points¶

    • eva-auth: User profile for preferences (dark mode, language)
    • eva-i18n: Bilingual content catalogs
    • eva-api: Gateway for navigation metadata (products, features)

    Implementation Phases¶

    Phase 1: Design System Foundation (Sprint 2)¶

    • Define design tokens (colors, spacing, typography)
    • Build core components (Button, Card, Modal, Form)
    • Set up Storybook documentation
    • Create Figma design files
    • Publish design system to CDN

    Phase 2: Navigation & Layout (Sprint 3)¶

    • Build unified header (logo, product switcher, profile)
    • Implement sidebar navigation (collapsible, icons, labels)
    • Add breadcrumbs component
    • Context preservation (session storage)
    • Dark mode toggle

    Phase 3: Interactions & Feedback (Sprint 4)¶

    • Keyboard shortcuts (Ctrl+K, Ctrl+/, product-specific)
    • Toast notifications (success, error, info, warning)
    • Loading states (spinners, skeleton screens, progress bars)
    • Empty states with illustrations
    • Error handling (user-friendly messages)

    Phase 4: Polish & Launch (Sprint 5)¶

    • Responsive design audit (mobile, tablet, desktop)
    • WCAG 2.1 AA audit (axe DevTools)
    • Performance optimization (Lighthouse ≥90)
    • User acceptance testing (UAT)
    • Documentation and training materials

    Success Criteria¶

    • ✅ Lighthouse performance score ≥90 (all EVA products)
    • ✅ WCAG 2.1 AA compliance (axe DevTools passes)
    • ✅ Component library with 20+ components (Storybook)
    • ✅ Design tokens published and adopted by all products
    • ✅ Time to Interactive (TTI) <3 seconds (P95)
    • ✅ User satisfaction: CSAT >4.0/5.0 (navigation usability)
    • ✅ 80%+ adoption of design system components

    Open Questions¶

    Q1: Design System Framework (TECH CHOICE)¶

    Question: Build custom design system or adopt existing (Fluent UI, Material UI, Chakra UI)? Impact: Control vs. development speed Recommendation: Tailwind CSS + HeadlessUI (unstyled, accessible, customizable) Decision Owner: UX Partner + ai2

    Q2: Storybook Hosting (INFRA)¶

    Question: Host Storybook on Azure Static Web Apps or GitHub Pages? Impact: Cost vs. control Recommendation: Azure Static Web Apps (integrates with Entra ID for private access) Decision Owner: windowsGuy

    Q3: Dark Mode Default (UX)¶

    Question: Light mode default or respect system preference? Impact: User experience vs. personal preference Recommendation: Respect system preference (prefers-color-scheme), allow manual toggle Decision Owner: UX Partner

    Q4: Offline Mode (FEATURES)¶

    Question: Full offline mode (service workers, IndexedDB) or just offline indicators? Impact: UX richness vs. complexity Recommendation: Offline indicators for MVP, full offline mode in Q2 2026 Decision Owner: Marco

    Q5: Animation Strategy (PERFORMANCE)¶

    Question: Use animations (CSS transitions, Framer Motion) or minimize for performance? Impact: Delight vs. performance Recommendation: Minimal animations for feedback (button clicks, toasts), avoid decorative animations Decision Owner: UX Partner + ai2
