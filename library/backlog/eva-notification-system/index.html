    CDD: EVA Notification System¶

    Last Updated: 2025-11-19 Status: Backlog Priority: MEDIUM Epic: EVA Cross-Cutting

    Vision¶

    Provide a unified notification system supporting push notifications, email, in-app alerts, and Teams messagesâ€”enabling timely communication about system events, user actions, and important updates across all EVA touchpoints.

    Business Value: Increase user engagement by 40%, reduce missed updates, improve response times to critical alerts, and enhance platform stickiness.

    Core Capabilities¶

    • Multi-Channel Delivery: Push (web, mobile), email, in-app badges, Teams messages
    • Notification Types: System alerts, user actions, reminders, announcements
    • Personalization: User preferences (frequency, channels, quiet hours)
    • Priority Levels: Critical (immediate), High (hourly digest), Medium (daily), Low (weekly)
    • Rich Content: HTML emails, interactive buttons, deep links
    • Delivery Tracking: Read receipts, click tracking, delivery status
    • Templates: Pre-built notification templates for common scenarios
    • Batching & Digests: Group notifications into daily/weekly digests

    User Stories¶

    US-1: As a user, I want email notifications so that I stay informed about important updates¶

    Acceptance Criteria:

    • Receive email when: query completes (async), space invitation, budget alert
    • Email includes: subject, summary, action button ("View Results"), deep link to EVA
    • Bilingual emails (en-US/fr-FR based on user preference)
    • Unsubscribe link in footer (one-click unsubscribe)
    • Email delivered within 2 minutes of event

    US-2: As a user, I want notification preferences so that I control frequency¶

    Acceptance Criteria:

    • Settings page: toggle channels (email, push, in-app, Teams)
    • Set frequency: Immediate, Hourly digest, Daily digest, Weekly digest
    • Quiet hours: no notifications 8pm-8am (configurable)
    • Per-notification-type settings (e.g., email for critical, in-app for low)
    • Save preferences, apply within 1 minute

    US-3: As a space admin, I want broadcast announcements so that I inform my team¶

    Acceptance Criteria:

    • Create announcement: title, message, target (all space members or specific roles)
    • Schedule announcement (send immediately or future date/time)
    • Track delivery: sent, read, clicked
    • Announcement appears in-app banner (dismissible)
    • Send via email + Teams message

    US-4: As a developer, I want API notifications so that I integrate with external systems¶

    Acceptance Criteria:

    • Webhook endpoint receives POST request when event occurs
    • Payload includes: eventType, userId, tenantId, metadata, timestamp
    • Webhook retries 3 times with exponential backoff if failure
    • Signature verification (HMAC-SHA256)
    • Webhook dashboard shows delivery status, logs

    US-5: As an ops engineer, I want critical alerts so that I respond to incidents fast¶

    Acceptance Criteria:

    • Critical alert: system outage, security breach, budget exceeded 110%
    • Deliver via: SMS, push, email, Teams, PagerDuty
    • Alert includes: severity, description, runbook link, acknowledge button
    • Escalate if not acknowledged within 15 minutes
    • Track MTTR (Mean Time To Response)

    Functional Requirements¶

    FR-1: Multi-Channel Delivery¶

    System SHALL deliver notifications via: email (SendGrid), push (Azure Notification Hubs), in-app (SignalR), Teams (Bot Framework).

    FR-2: Notification Queueing¶

    System SHALL queue notifications in Azure Queue Storage with priority-based processing (Critical > High > Medium > Low).

    FR-3: User Preferences¶

    System SHALL allow users to configure: channels, frequency, quiet hours, per-type settings.

    FR-4: Batching & Digests¶

    System SHALL batch low-priority notifications into hourly, daily, or weekly digests based on user preference.

    FR-5: Template Engine¶

    System SHALL use template engine (Handlebars) for email/message templates with variable substitution (user name, event details).

    FR-6: Delivery Tracking¶

    System SHALL track notification status: queued, sent, delivered, read, clicked, failed.

    FR-7: Webhooks¶

    System SHALL support webhook subscriptions with retry logic (3 attempts, exponential backoff) and HMAC signature verification.

    FR-8: Rich Content¶

    System SHALL support HTML emails, action buttons, deep links, and embedded images.

    FR-9: Unsubscribe¶

    System SHALL provide one-click unsubscribe links in emails (CASL/CAN-SPAM compliance).

    FR-10: Priority Routing¶

    System SHALL route critical notifications through multiple channels simultaneously, lower priorities through user-preferred channel.

    FR-11: Rate Limiting¶

    System SHALL limit notifications per user: max 50/hour, max 200/day to prevent spam.

    FR-12: Audit Logging¶

    System SHALL log all notifications sent, delivery status, user actions (read, clicked, unsubscribed).

    FR-13: RBAC¶

    System SHALL enforce RBAC: Space Admins broadcast to spaces, Platform Admins broadcast to tenants.

    FR-14: Bilingual Support¶

    System SHALL generate bilingual notifications (en-US/fr-FR) based on user language preference.

    FR-15: Performance¶

    System SHALL deliver critical notifications within 30 seconds, non-critical within 5 minutes.

    Non-Functional Requirements¶

    Performance¶

    • Delivery latency: <30 seconds for critical, <5 minutes for non-critical
    • Throughput: 10K notifications/minute
    • In-app notification display: <500ms
    • Email generation: <2 seconds per email
    • Support 100K active notification subscribers

    Reliability¶

    • Uptime: 99.9% SLA
    • Message durability: 99.999999999% (Azure Queue Storage)
    • Auto-retry failed deliveries (3 attempts)
    • Graceful degradation (queue if channel unavailable)

    Security¶

    • Encrypt notification content at rest (AES-256) and in transit (TLS 1.3)
    • HMAC-SHA256 signature for webhooks
    • PII masking in logs (email addresses, names)
    • Compliance: CASL, CAN-SPAM, PIPEDA

    Accessibility¶

    • WCAG 2.1 AA compliance (notification preferences UI)
    • Keyboard navigation for dismissing alerts
    • Screen reader support (in-app notifications)
    • High-contrast mode for notification badges

    i18n¶

    • Bilingual support (en-US/fr-FR)
    • ICU formatting for dates, numbers
    • Localized templates, action buttons
    • Language-aware routing (send in user's preferred language)

    Observability¶

    • Application Insights custom events (notification sent, delivered, read, clicked)
    • Track delivery rates per channel, per type
    • Alerts for: high failure rate (>10%), low delivery rate (<90%)
    • Metrics: notifications/day, avg delivery time, open rate

    Scalability¶

    • Horizontal scaling (Azure Functions for processing)
    • Partition queue by priority (critical, high, medium, low)
    • Azure CDN for static email assets (logos, images)
    • Support 1M notifications/day per tenant

    Technical Architecture¶

    Backend (Notification Engine)¶

    • Queueing: Azure Queue Storage (priority queues)
    • Processing: Azure Functions (Queue trigger, process batches)
    • Email: SendGrid API (transactional emails)
    • Push: Azure Notification Hubs (web push, mobile)
    • In-App: SignalR (real-time WebSocket)
    • Teams: Bot Framework SDK (direct messages)

    Frontend (UI)¶

    • In-App: React notification center (bell icon, badge count)
    • Preferences: React settings page (toggles, sliders)
    • Banner: Dismissible announcement banner (space-wide)

    Integration Points¶

    • eva-api: Trigger notifications from API events
    • eva-auth: RBAC for broadcast permissions
    • eva-i18n: Localized notification templates
    • SendGrid: Email delivery
    • Azure Notification Hubs: Push notifications

    Question: Plain text digest or styled HTML? Impact: Deliverability vs. aesthetics Recommendation: HTML with plain text fallback (accessibility) Decision Owner: UX Partner

    Q3: Webhook Retry (TECH)¶

    Question: 3 retries reasonable or increase to 5? Impact: Reliability vs. delay Recommendation: 3 retries with exponential backoff (30s, 60s, 120s) Decision Owner: ai2
