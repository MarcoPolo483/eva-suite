    CDD: EVA Marketplace (Internal Tools Community Gallery)¶

    Last Updated: 2025-11-19 Status: Backlog Priority: MEDIUM Epic: Cross-Cutting Concerns

    Vision¶

    Enable internal ESDC developers to publish, discover, and share custom EVA agents and tools through a community-driven marketplace with peer review, ratings, version control, and seamless deployment—fostering innovation, collaboration, and best practice sharing across departments.

    Business Value: Accelerate tool development by 50% through code reuse, improve quality with peer review, increase developer satisfaction by 40% with community recognition, and reduce redundant tool creation across teams.

    Core Capabilities¶

    • Community Publishing: Submit custom agents, peer review by 3 approvers, GitHub integration
    • Internal-Only Visibility: ESDC-only access (no external publishing)
    • Ratings & Reviews: 5-star ratings, text reviews, upvote/downvote helpful reviews
    • Version Control: GitHub repo per tool, semantic versioning (semver), changelog
    • One-Click Deployment: Install tool to space <10 seconds, automatic dependency resolution
    • Community Support Forum: Q&A, best practices, troubleshooting (integrated with Learning Community Forum)
    • Usage Analytics: Downloads, active installs, user engagement per tool
    • Category Tagging: Tags (data-analysis, document-processing, automation, integrations)

    User Stories¶

    US-1: As a developer, I want to publish my custom agent so that colleagues can benefit from my work¶

    Acceptance Criteria:

    • Submit tool form: name, description (max 500 chars), GitHub repo URL, category tags, screenshots
    • Peer review workflow: 3 approvers required (any ESDC developer with 100+ reputation)
    • Review criteria: code quality, documentation, test coverage >70%, security scan passed
    • Approval threshold: ⅔ approvals required (submit → approved → published within 5 days average)
    • Published tool appears in marketplace with "New" badge for 7 days

    US-2: As a developer, I want to discover tools so that I don't reinvent the wheel¶

    Acceptance Criteria:

    • Marketplace grid view: tool cards with name, description, rating (5-star), download count
    • Search bar: full-text search (name, description, tags)
    • Filters: category (data-analysis, document-processing), rating (4+ stars), recent (last 30 days)
    • Sort options: most popular (downloads), highest rated, recently updated
    • Click tool card → detail page (README, screenshots, changelog, reviews, install button)

    US-3: As a user, I want to install tools so that I enhance my space with one click¶

    Acceptance Criteria:

    • Click "Install" button → confirm modal (space selection dropdown)
    • Installation progress: downloading → dependency check → deploying → complete (<10 seconds)
    • Tool appears in space's Tools menu with icon and name
    • Post-install: show quick start guide (3-5 steps to use the tool)
    • Automatic updates: opt-in to auto-update (default: manual update)

    US-4: As a user, I want to rate/review tools so that I help others choose quality tools¶

    Acceptance Criteria:

    • Rate tool after 7 days of usage (1-5 stars)
    • Write review (optional, max 1000 chars, markdown support)
    • Upvote/downvote helpful reviews (sort reviews by helpfulness)
    • Edit/delete own review (version history tracked)
    • Developer can respond to reviews (one response per review)

    US-5: As a developer, I want usage analytics so that I understand adoption and improve my tool¶

    Acceptance Criteria:

    • Analytics dashboard: total downloads, active installs, retention rate (30-day)
    • Engagement metrics: daily active users, average session time, feature usage
    • Rating trends: 5-star breakdown (bar chart), average rating over time (line chart)
    • User feedback: top-voted reviews, common feature requests
    • Export analytics as CSV (for reporting to management)

    Functional Requirements¶

    1. FR-1: SHALL allow developers to submit tools (name, description, GitHub URL, tags, screenshots)
    2. FR-2: SHALL enforce peer review (3 approvers, ⅔ approval threshold)
    3. FR-3: SHALL integrate with GitHub (validate repo access, fetch README, check for tests)
    4. FR-4: SHALL provide marketplace search (full-text, filters by category/rating, sort options)
    5. FR-5: SHALL support one-click install (deploy to space, dependency resolution, <10s)
    6. FR-6: SHALL enforce internal-only access (ESDC Entra ID authentication required)
    7. FR-7: SHALL support 5-star ratings and text reviews (after 7-day usage delay)
    8. FR-8: SHALL track usage analytics (downloads, active installs, engagement metrics)
    9. FR-9: SHALL support semantic versioning (major.minor.patch, changelog required)
    10. FR-10: SHALL run security scans (GitHub Advanced Security, block high-severity vulnerabilities)
    11. FR-11: SHALL provide community forum integration (Q&A, best practices)
    12. FR-12: SHALL log all marketplace events to Cosmos DB (publishes, installs, reviews)
    13. FR-13: SHALL support automatic updates (opt-in, notify user before update)
    14. FR-14: SHALL enforce test coverage >70% for published tools (GitHub Actions check)
    15. FR-15: SHALL provide developer dashboard (analytics, reviews, update management)

    Non-Functional Requirements¶

    Performance¶

    • Search Response: <300ms for marketplace search (full-text, filters)
    • Install Time: <10s from click to tool available in space
    • Page Load: <500ms for tool detail page (README rendering)
    • Analytics Refresh: Real-time updates (30-second delay max)

    Reliability¶

    • Marketplace Uptime: 99.9% SLA (43 minutes downtime/month max)
    • Install Success Rate: >98% (automatic rollback on failure)
    • GitHub Integration: Fallback to manual upload if GitHub unavailable
    • Peer Review SLA: Average 5 days from submission to approval

    Security¶

    • Authentication: ESDC Entra ID OAuth 2.0 (internal-only access)
    • Authorization: Developers can publish, all users can install, admins can remove tools
    • Code Scanning: GitHub Advanced Security (SAST, dependency vulnerabilities)
    • Sandboxing: Tools run in isolated containers (no access to other spaces)
    • Audit Logging: All installs, reviews, approvals logged to Cosmos DB (1-year retention)

    Accessibility¶

    • WCAG 2.1 AA: Marketplace UI keyboard-accessible, screen reader support
    • Search Filters: Accessible via keyboard (arrow keys, Enter to apply)
    • Tool Cards: Focus indicators, ARIA labels for ratings
    • Install Modal: Keyboard navigation (Tab, Escape to cancel)

    Internationalization¶

    • Locales: en-US (default), fr-FR
    • Tool Metadata: Bilingual (developers provide en/fr descriptions)
    • Marketplace UI: Bilingual labels (search, filters, sort options)
    • Reviews: Auto-detect language (allow en-US or fr-FR reviews)

    Observability¶

    • Metrics: Application Insights tracks searches, installs, ratings, reviews, page views
    • Alerts: Install failure rate >5%, security scan failures, peer review backlog >10 tools
    • Dashboards: Power BI dashboard for marketplace health, top tools, developer leaderboard

    Scalability¶

    • Tool Catalog: Support 500+ tools (indexed in Azure AI Search)
    • Concurrent Installs: 100+ simultaneous installs without degradation
    • Reviews: Cosmos DB with HPK /tenantId/toolId (scales to 10K+ reviews per tool)

    Technical Architecture¶

    Frontend¶

    • Framework: React 18+ with TypeScript, Tailwind CSS
    • UI Components: Tool cards (grid layout), search bar with autocomplete, filter sidebar
    • State Management: Zustand (marketplace state), React Query (API calls)
    • Routing: React Router v6 (marketplace browse, tool detail, developer dashboard)

    Backend¶

    • API Gateway: Azure APIM (rate limiting, JWT validation)
    • Functions: Azure Functions (Node.js 20) for tool submission, install orchestration, peer review
    • GitHub Integration: GitHub REST API (validate repo, fetch README, run Actions)
    • Search: Azure AI Search (full-text index: tool name, description, tags)
    • Container Registry: Azure Container Registry (store tool images)
    • Deployment: Azure Container Instances (run tools in isolated containers)

    Implementation Phases¶

    Phase 1: Marketplace Core (Sprint 5, Week 1-2)¶

    • Tool submission form (name, description, GitHub URL, tags)
    • Marketplace browse (grid view, search, filters, sort)
    • Tool detail page (README rendering, screenshots, install button)
    • Azure AI Search indexing (tool metadata)
    • Cosmos DB storage (tools, HPK /tenantId/toolId)

    Phase 2: Peer Review & Publishing (Sprint 6, Week 3-4)¶

    • Peer review workflow (assign 3 reviewers, approval voting)
    • GitHub integration (validate repo access, fetch README, check tests)
    • Security scanning (GitHub Advanced Security, block high-severity)
    • Approval dashboard (pending reviews, approve/reject UI)
    • Email notifications (review requests, approval status)

    Phase 3: Install & Deployment (Sprint 7, Week 5-6)¶

    • One-click install (select space, deploy to Azure Container Instances)
    • Dependency resolution (check for required APIs, install dependencies)
    • Install progress indicator (downloading → deploying → complete)
    • Post-install quick start guide (3-5 steps)
    • Automatic update system (opt-in, notify before update)

    Phase 4: Ratings, Reviews & Analytics (Sprint 8, Week 7-8)¶

    • 5-star rating system (after 7-day usage delay)
    • Text reviews (markdown support, upvote/downvote)
    • Developer analytics dashboard (downloads, installs, ratings)
    • Community forum integration (Q&A, best practices)
    • Power BI dashboard (marketplace health, top tools, leaderboard)

    Success Criteria¶

    • ✅ Marketplace search <300ms response time
    • ✅ Install success rate >98% (one-click deployment)
    • ✅ 50+ tools published within 6 months
    • ✅ Average peer review time <5 days
    • ✅ Developer satisfaction >85% (post-publish survey)
    • ✅ Tool reuse rate >40% (vs. building from scratch)
    • ✅ WCAG 2.1 AA compliance (accessibility audit passed)

    Open Questions¶

    1. Should external developers (non-ESDC) be allowed to publish tools (e.g., vendors, contractors)?

    2. Impact: Ecosystem growth vs. security/governance concerns

    3. Recommendation: Internal-only for MVP, external publishing deferred to Q3 2026 with vendor approval process

    4. Decision Owner: Marco (PO)

    5. Should tools support paid licensing (e.g., freemium, subscription)?

    6. Impact: Revenue sharing vs. complexity of payment processing

    7. Recommendation: Free-only for MVP, paid tools deferred to Q4 2026

    8. Decision Owner: Marco (PO)

    9. What is the reputation threshold for becoming a peer reviewer?

    10. Impact: Review quality vs. reviewer pool size

    11. Recommendation: 100+ reputation (earned from publishing, helpful reviews, forum contributions)

    12. Decision Owner: Marco (PO)

    13. Should tools be sandboxed by default (e.g., restricted API access)?

    14. Impact: Security vs. tool functionality

    15. Recommendation: Sandboxed by default, tools can request elevated permissions (admin approval)

    16. Decision Owner: Marco (PO)

    17. Should the marketplace support tool bundles (e.g., install 3 tools as a suite)?

    18. Impact: User convenience vs. dependency management complexity
    19. Recommendation: Individual tools only for MVP, bundles deferred to Sprint 10
    20. Decision Owner: Marco (PO)
