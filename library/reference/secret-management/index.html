    Secret Management Strategy¶

    Last Updated: 2025-11-16

    Overview¶

    EVA Orchestrator requires GitHub Personal Access Token (PAT) to scaffold and manage 17 repositories autonomously. This document outlines the secret management strategy for local, CI/CD, and Azure-hosted scenarios.

    Architecture¶

    Local Dev (Marco's Machine)
   ↓ System Environment Variable
   GITHUB_TOKEN → Orchestrator → GitHub API

GitHub Actions (CI/CD)
   ↓ GitHub Secret
   GH_ORCHESTRATOR_TOKEN → Orchestrator → GitHub API

Azure Key Vault (Optional, Enterprise)
   ↓ Managed Identity
   azure/CLI → Key Vault → Sync to GitHub Secret

    Setup Instructions¶

    1. Local Development (✅ Complete)¶

    You already did this by saving token in system environment variables.

    Verify: 

    $env:GITHUB_TOKEN  # Should show your token
npm run dev  # Should work without prompts

    Persistence (already done): - System-level env var persists across PowerShell sessions - Orchestrator reads automatically via process.env.GITHUB_TOKEN

    2. GitHub Actions (For Autonomous Workflows)¶

    Setup Steps:

    1. Add GitHub Secret:
    2. Go to: https://github.com/MarcoPolo483/eva-orchestrator/settings/secrets/actions
    3. Click "New repository secret"
    4. Name: GH_ORCHESTRATOR_TOKEN
    5. Value: (paste your GitHub PAT)

    6. Click "Add secret"

    7. Workflows Automatically Use It:

    8. .github/workflows/orchestrator-auto.yml - Weekly orchestrator run

    9. .github/workflows/sync-secrets.yml - Azure Key Vault sync (optional)

    10. Test:

    11. Go to: Actions → Orchestrator Auto-Run → Run workflow
    12. Should scaffold all 17 repos without human intervention

    3. Azure Key Vault (Optional, Enterprise-Grade)¶

    Benefits: - Centralized secret management - Automatic rotation - Audit logging - RBAC for secret access

    Setup Steps:

    Step 1: Create Azure Key Vault¶

    # Create resource group
az group create --name eva-secrets-rg --location canadacentral

# Create Key Vault
az keyvault create \
  --name eva-secrets-kv \
  --resource-group eva-secrets-rg \
  --location canadacentral \
  --enable-rbac-authorization false

# Store GitHub token
az keyvault secret set \
  --vault-name eva-secrets-kv \
  --name github-orchestrator-token \
  --value "your_github_pat_here"

    Step 2: Configure Azure OIDC for GitHub Actions¶

    # Create Azure AD App Registration
az ad app create --display-name "EVA Orchestrator GitHub Actions"

# Create service principal
$appId = (az ad app list --display-name "EVA Orchestrator GitHub Actions" --query [0].appId -o tsv)
az ad sp create --id $appId

# Create federated credential (allows GitHub Actions to authenticate)
az ad app federated-credential create \
  --id $appId \
  --parameters '{
    "name": "eva-orchestrator-main",
    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:MarcoPolo483/eva-orchestrator:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
  }'

# Grant Key Vault permissions
$spObjectId = (az ad sp list --display-name "EVA Orchestrator GitHub Actions" --query [0].id -o tsv)
az keyvault set-policy \
  --name eva-secrets-kv \
  --object-id $spObjectId \
  --secret-permissions get list

    Step 3: Add Azure Credentials to GitHub Secrets¶

    Add these 3 secrets to GitHub (same place as GH_ORCHESTRATOR_TOKEN):

    • AZURE_CLIENT_ID - From $appId above
    • AZURE_TENANT_ID - Your Azure AD tenant ID
    • AZURE_SUBSCRIPTION_ID - Your Azure subscription ID

    Step 4: Enable Sync Workflow¶

    • Go to: Actions → Sync Secrets from Azure Key Vault → Enable workflow
    • Run manually first to test
    • Runs weekly to keep GitHub Secret up-to-date with Key Vault

    Security Best Practices¶

    ✅ DO:¶

    • Use system env vars for local dev (you did this)
    • Use GitHub Secrets for CI/CD (encrypted at rest)
    • Use Azure Key Vault for enterprise (centralized, audited)
    • Rotate tokens every 90 days
    • Use minimal scopes (repo + project only)

    ❌ DON'T:¶

    • Commit tokens to repos (we don't)
    • Share tokens in chat (lesson learned!)
    • Use personal accounts for automation (create machine user if possible)
    • Grant admin/delete permissions to automation tokens

    Token Rotation Strategy¶

    Manual Rotation (Current)¶

    1. Generate new GitHub PAT
    2. Update system env var: [System.Environment]::SetEnvironmentVariable('GITHUB_TOKEN', 'new_token', 'User')
    3. Update GitHub Secret: GH_ORCHESTRATOR_TOKEN
    4. (Optional) Update Azure Key Vault: az keyvault secret set ...

    Automated Rotation (Future)¶

    • Use GitHub App instead of PAT (tokens auto-refresh)
    • Azure Key Vault secret rotation policies
    • Notification on expiration (90 days)

    Troubleshooting¶

    Error: "Missing GITHUB_TOKEN"¶

    Local Dev: 

    $env:GITHUB_TOKEN="your_token"  # Set for current session
[System.Environment]::SetEnvironmentVariable('GITHUB_TOKEN', 'your_token', 'User')  # Persist

    GitHub Actions: - Verify secret exists: Settings → Secrets → Actions → GH_ORCHESTRATOR_TOKEN - Check workflow file uses correct secret name

    Error: "Bad credentials"¶

    • Token expired (90-day default)
    • Token revoked
    • Insufficient scopes (need repo + project)

    Solution: Generate new token, update all locations

    Azure Key Vault Access Denied¶

    • Service principal missing Key Vault permissions
    • Federated credential misconfigured
    • Wrong subscription/tenant

    Solution: Re-run Step 2 commands above
