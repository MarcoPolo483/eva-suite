    Chapter 10: Appendices¶

    This appendix provides quick-reference materials for developers, administrators, and end-users working with EVA Chat 2.0. Use these sections to resolve common issues, understand terminology, verify system compatibility, and plan migrations from legacy systems.

    A. Changelog¶

    Version 2.0.0 (Initial Release - December 2025)¶

    Major Features:

    • Government of Canada Compliance: Full WCAG 2.1 AA accessibility compliance, bilingual EN-CA/FR-CA support with instant language switching, GC Design System visual identity integration with Canada wordmark and footer links
    • Enhanced Security Posture: Protected B data classification with customer-managed keys (CMK) in Azure Key Vault, Azure Active Directory authentication with single sign-on (SSO) and multi-factor authentication (MFA), private endpoint deployment (no public IP exposure), SCED network routing via Shared Services Canada infrastructure
    • Retrieval-Augmented Generation (RAG): Multi-index retrieval across SharePoint, Cosmos DB, and Azure Blob Storage, confidence scoring for retrieved documents (0.0-1.0 scale), citation provenance tracking with source document links, domain-specific chunking strategies (legal, policy, technical documentation)
    • GC Agentic Framework Integration: C0-C3 autonomy level indicators displayed in AI response cards, AI-generated content disclaimers with confidence scores, human-in-the-loop workflows for C2 and C3 autonomous operations, audit logging for all AI decisions with 7-year retention
    • EVA Sovereign UI Components: 15 Government of Canada-compliant web components (eva-gc-citation-card, eva-gc-ai-disclaimer, eva-gc-language-switcher, eva-gc-header, eva-gc-footer), keyboard navigation and screen reader support (JAWS, NVDA, VoiceOver), 4.5:1 minimum color contrast ratios
    • Observability and Monitoring: Azure Application Insights integration for real-time performance metrics, conversation analytics dashboards (message volume, user satisfaction, response latency), 7-year audit log retention in Cosmos DB, automated alert thresholds for P95 latency > 500ms

    Infrastructure Architecture:

    • Multi-Region Deployment: Azure Canada Central (primary) + Canada East (disaster recovery), active-active load balancing with Azure Front Door, geo-replicated Cosmos DB (99.999% SLA)
    • Data Sovereignty: All data at rest stored exclusively in Canadian Azure regions, data residency enforced via Azure Policy, no cross-border data transfers
    • Secure Connectivity: SCED network routing (SSC-managed secure channel), Azure Private Link for all service endpoints, TLS 1.2+ encryption in transit, Azure Key Vault for secret management
    • Compute and Storage: Azure App Service Plan P2v3 (8 vCPU, 16 GB RAM) for production tier, Azure Cosmos DB provisioned throughput (10,000 RU/s), Azure Cognitive Search Standard tier (50 GB storage)

    Quality Gates Achieved:

    • Code Coverage: 82% overall coverage (Jest unit tests + Playwright end-to-end tests)
    • Performance: P95 response latency 280ms (target: <300ms), P99 latency 450ms
    • Accessibility: 100% WCAG 2.1 AA pass rate (axe-core automated testing + manual screen reader validation)
    • Security: Zero critical or high-severity vulnerabilities (Snyk security scan), OWASP Top 10 compliance
    • Bilingualism: 100% UI strings translated to French Canadian, dynamic language switching without page reload

    Planned Enhancements:

    B. Migration Guides¶

    Upgrading from EVA Chat v1.0 to v2.0¶

    Pre-Migration Checklist:

    1. Data Export: Navigate to Admin Portal â†’ Data Management â†’ Export Conversations. Select date range and click "Export to JSON". Download will include all conversation histories, user metadata, and citation references.
    2. User Audit: Review Azure AD role assignments in Azure Portal â†’ Enterprise Applications â†’ EVA Chat â†’ Users and Groups. Verify all users have explicit role mappings (v2.0 no longer supports default roles).
    3. API Inventory: Identify all custom integrations using v1.0 REST endpoints. Search codebase for /api/v1/ references. Document endpoint usage for migration.
    4. Browser Compatibility: Survey end-users to confirm browser versions. EVA Chat 2.0 requires Chrome 120+, Edge 120+, Firefox 115+, or Safari 17+. Users on older browsers must upgrade.

    Migration Steps:

    1. Deploy v2.0 Infrastructure in Parallel: Provision new Azure App Service, Cosmos DB account, and Cognitive Search service in Canada Central region. Deploy v2.0 application alongside existing v1.0 infrastructure (no downtime).
    2. Data Migration: Execute eva-migration-tool to convert v1.0 Cosmos DB conversations to v2.0 schema: 
      npx @eva/migration-tool \
  --source cosmosdb://v1-account.documents.azure.com:443/ \
  --target cosmosdb://v2-account.documents.azure.com:443/ \
  --database eva-chat \
  --validate \
  --dry-run
      Review validation report, then run without --dry-run to execute migration.
    3. API Cutover: Update client applications to use v2.0 endpoints. Replace https://api-v1.eva.canada.ca/api/v1/chat with https://api.eva.canada.ca/v2/chat. Update authentication headers.
    4. User Training: Schedule 30-minute virtual training sessions covering: new GC Design System UI, language switcher (top-right header icon), citation cards with confidence scores, AI autonomy level indicators.
    5. Monitoring Period: Enable Application Insights dashboards for 2 weeks. Monitor error rates, latency percentiles, and user feedback. Set alert thresholds: P95 latency > 500ms, error rate > 1%, user satisfaction < 80%.
    6. Decommission v1.0: After 2-week monitoring period with zero critical issues, archive v1.0 conversation data to Azure Blob Storage (cool tier). Terminate v1.0 App Service and Cosmos DB resources. Update DNS to remove v1.0 endpoints.

    Common Migration Issues:

    • Authentication Failures: Ensure Azure AD app registration includes User.Read and Chat.ReadWrite API permissions. Grant admin consent in Azure Portal.
    • Missing Citations: v1.0 conversations lack citation metadata. Citations only appear for new queries in v2.0 (historical messages show "No citations available").
    • Language Preferences: v1.0 stored language preference in browser cookies. v2.0 reads preferredLanguage attribute from Azure AD user profile. Users must set language in Azure AD profile settings.
    • Conversation Formatting: v1.0 used Markdown rendering. v2.0 uses GC Design System typography. Long code blocks may wrap differently. Review conversation exports for formatting issues.

    C. Frequently Asked Questions (FAQ)¶

    For End-Users¶

    Q: How do I switch between English and French? A: Click the language toggle in the top-right header (ðŸ‡¨ðŸ‡¦ Canada flag icon). Your language preference is saved to your Azure Active Directory profile and persists across sessions and devices. No page reload required.

    Q: What does "C2 Supervised Autonomous" mean in the AI disclaimer? A: C2 (Supervised Autonomous) indicates the AI can perform actions automatically (e.g., retrieve documents, generate summaries), but a human reviewer must approve critical operations before they take effect (e.g., policy recommendations submitted to management, data exports containing personal information). See GC Agentic Framework documentation for full definitions.

    Q: How do I verify the accuracy of citations? A: Each citation card displays a confidence score (0.0-1.0) indicating retrieval relevance. Click "View Source Document" to open the original file in SharePoint (requires read permissions). Cross-reference AI-generated summaries with source content. Report citation errors via "Flag Inaccurate Citation" button.

    Q: Can I use EVA Chat on my mobile device? A: Yes. EVA Chat 2.0 is fully responsive and supports iOS Safari 17+ and Chrome for Android 120+. Install as Progressive Web App (PWA) for offline access: Safari â†’ Share â†’ Add to Home Screen. Chrome â†’ Menu â†’ Install App.

    Q: How long are my conversations stored? A: Conversations are retained for 7 years in accordance with Government of Canada records retention policies. You can delete conversations manually via Settings â†’ Privacy â†’ Delete Conversation History. Deleted conversations are purged from production databases within 30 days but remain in backup archives for audit compliance.

    For Developers¶

    Q: How do I extend EVA Chat with custom domain-specific personas? A: Follow the Service Persona Development Guide in docs/agents/personas/README.md. Create a new SPxx persona definition in eva-meta/agents/registry.yaml, implement domain-specific RAG indices in your pipeline repository (e.g., jurispipeline for legal content), and register the persona endpoint in EVA Foundation API Gateway.

    Q: Can I self-host EVA Chat outside of Azure? A: EVA Foundation v2.0 is Azure-native and requires Azure AD, Cosmos DB, and Azure OpenAI. Cloud-agnostic support (AWS, GCP, Cohere Cloud) is planned for v2.2 (Q3 2026). Contact EVA Agile Command for early access to multi-cloud architecture design documents.

    Q: How do I customize the GC Design System theme? A: Override CSS custom properties in your application stylesheet (see Chapter 8: Theming). Example: --eva-color-primary: #26374a (GC blue). Do NOT modify core EVA Sovereign UI component styles directlyâ€”use CSS cascading to override. Changes must maintain 4.5:1 contrast ratio for WCAG compliance.

    Q: How do I debug RAG retrieval issues? A: Enable debug logging in appsettings.json: "Logging": { "LogLevel": { "EVA.RAG": "Debug" } }. Review Application Insights logs for RAG.SearchQuery events. Check Azure Cognitive Search index health in Azure Portal. Verify embedding model consistency (text-embedding-ada-002).

    Q: What authentication methods are supported for API access? A: EVA Chat 2.0 uses Azure AD OAuth 2.0 Bearer tokens. Obtain token via @azure/msal-browser library. Service-to-service authentication uses Managed Identity (no secrets). API keys are deprecated in v2.0 for security compliance.

    D. Troubleshooting¶

    Error Codes¶

    Code Message Root Cause Resolution
    EVA-1001 Authentication token expired Azure AD token TTL exceeded (60 minutes) User automatically redirected to Azure AD login. No action required.
    EVA-2003 Rate limit exceeded (100 requests/minute) User exceeded API throttle threshold Wait 60 seconds for rate limit reset. Contact admin to request quota increase for high-volume users.
    EVA-3005 RAG index unavailable Azure Cognitive Search service offline or index corrupted Check Azure Portal â†’ Cognitive Search â†’ Service Health. Rebuild index via Admin Portal â†’ RAG Management â†’ Rebuild Index.
    EVA-4002 Citation source document not found Referenced document deleted from SharePoint or access revoked Verify document exists in SharePoint library. Check user has read permissions. Citation marked as "Source Unavailable" in UI.
    EVA-5001 WCAG accessibility violation detected Component rendered without proper ARIA labels File bug report in GitHub Issues with axe-core scan results. Include browser version and screenshot.
    EVA-6010 Conversation persistence failed Cosmos DB write throttled (exceeded RU/s quota) Retry conversation save after 5 seconds. Admin: increase Cosmos DB provisioned throughput in Azure Portal.
    EVA-7015 AI response generation timeout Azure OpenAI request exceeded 30-second timeout Simplify user query or reduce conversation context. Admin: check Azure OpenAI quota and throttling limits.

    Known Issues (v2.0.0)¶

    1. Safari 17.0 Language Switcher Persistence: Language toggle preference may not persist on page refresh in Safari 17.0 on macOS. Workaround: Set language in Azure AD profile (preferredLanguage attribute). Fixed in v2.0.1 (January 2026).

    2. Long Conversations (>100 messages) Performance Degradation: Response latency increases beyond 500ms for conversations with more than 100 messages due to context window limitations. Workaround: Start new conversation or use "Summarize Conversation" feature. Redis caching planned for v2.1.0 (Q1 2026).

    3. PDF Complex Table Extraction: Tables with merged cells or nested structures may not render correctly in citation preview cards. Workaround: Click "View Source Document" to open original PDF. Multi-modal RAG with GPT-4 Vision planned for v2.1.0.

    4. Screen Reader Announcement Delays: NVDA users may experience 1-2 second delays before AI responses are announced. Workaround: Use NVDA Browse Mode (Insert+Space) to read response manually. ARIA live region timing improvements in v2.0.2 (February 2026).

    Diagnostic Procedures¶

    Issue: Chat responses are slow (>5 seconds)

    Diagnostic Steps: 1. Check Azure OpenAI quota utilization: Azure Portal â†’ Cognitive Services â†’ EVA OpenAI â†’ Quotas and Usage. Verify tokens-per-minute (TPM) usage is below 80% threshold. 2. Verify Cosmos DB RU/s utilization: Azure Portal â†’ Cosmos DB â†’ Metrics â†’ Normalized RU Consumption. Should be below 80% for optimal performance. 3. Review Application Insights performance telemetry: Application Insights â†’ Performance â†’ Dependencies. Look for external calls exceeding 300ms (e.g., Azure Cognitive Search queries, SharePoint document retrieval). 4. Check user conversation length: Admin Portal â†’ User Analytics â†’ Conversation Length Distribution. Conversations >50 messages may require summarization.

    Resolution: - If Azure OpenAI quota exhausted: Request quota increase via Azure Support ticket (typically approved within 48 hours). - If Cosmos DB throttled: Increase provisioned throughput (RU/s) in Azure Portal. - If slow RAG queries: Optimize Azure Cognitive Search index (reduce field count, disable unused analyzers).

    Issue: Citations missing for known documents

    Diagnostic Steps: 1. Verify document indexed in Azure Cognitive Search: Admin Portal â†’ RAG Management â†’ Search Index Status. Search for document filename. 2. Check embedding model consistency: Application Insights â†’ Custom Events â†’ RAG.EmbeddingModel. All documents must use same model (text-embedding-ada-002). 3. Verify user has RBAC permissions for source library: SharePoint â†’ Library Settings â†’ Permissions. User must have "Read" access. 4. Test RAG retrieval directly: Admin Portal â†’ RAG Management â†’ Test Query. Enter document keywords, review confidence scores.

    Resolution: - If document not indexed: Trigger manual re-index via Admin Portal â†’ RAG Management â†’ Re-index Document Library. - If embedding model mismatch: Rebuild entire search index with consistent model version. - If RBAC permissions issue: Grant user access to SharePoint library or adjust EVA Foundation permission filters.

    E. Glossary¶

    ATO (Authority to Operate): Government of Canada security authorization process required before deploying a system to production environment. EVA Chat 2.0 ATO process includes security assessment, privacy impact assessment (PIA), and threat risk assessment (TRA).

    Autonomy Levels (C0-C3): GC Agentic Framework classification of AI agent decision-making independence: - C0 (Constrained): AI provides informational responses only. No automated actions. Human reviews all outputs before use (e.g., legal advice, financial recommendations). - C1 (Guided Autonomous): AI performs predefined tasks automatically within strict parameters (e.g., search document libraries, format citations). Human approval required for exceptions. - C2 (Supervised Autonomous): AI acts independently for routine operations. Human review required for critical decisions (e.g., policy recommendations, data exports containing personal information). - C3 (Fully Autonomous): AI operates without human intervention. Human oversight via post-audit logs only. Not approved for Protected B environments.

    CMK (Customer-Managed Key): Encryption key stored in Azure Key Vault and controlled by the Government of Canada (versus Microsoft-managed keys). Required for Protected B data classification. Supports automatic 90-day key rotation. Key access audited in Azure Monitor logs.

    Protected B: Canadian government security classification for sensitive information requiring protection against unauthorized disclosure (e.g., Social Insurance Numbers, medical records, policy drafts, procurement documents). Requires encryption at rest (CMK), encryption in transit (TLS 1.2+), and audit logging (7-year retention).

    RAG (Retrieval-Augmented Generation): AI architecture pattern where the language model retrieves relevant documents from a knowledge base before generating responses. Enables factual accuracy, reduces hallucinations, and provides citations for source attribution. EVA Chat 2.0 uses Azure Cognitive Search for vector similarity retrieval.

    SCED (Secure Channel for External Destinations): Shared Services Canada (SSC) managed network infrastructure for secure communication between Government of Canada networks and external cloud providers (e.g., Azure, AWS). All EVA Chat traffic routes through SCED endpointsâ€”no direct public internet exposure.

    WCAG 2.1 AA (Web Content Accessibility Guidelines): W3C international standard for web accessibility. Level AA compliance requires keyboard navigation, screen reader compatibility, 4.5:1 minimum color contrast ratio for body text, and alternative text for images. Mandatory for all Government of Canada web applications per TBS Standard on Web Accessibility.

    Quick Reference¶

    Base URL: https://api.eva.canada.ca/v2

    Authentication: OAuth 2.0 Bearer token (Azure AD) 

    Authorization: Bearer <access_token>

    Key Endpoints:

    • POST /chat/conversations - Create new conversation
    • POST /chat/conversations/{id}/messages - Send message, receive AI response with citations
    • GET /chat/conversations/{id}/messages - Retrieve conversation history (paginated, 20 messages per page)
    • POST /chat/feedback - Submit user satisfaction rating (thumbs-up/down)
    • POST /search - Direct RAG search without conversation context
    • GET /health - Service health check (unauthenticated)
