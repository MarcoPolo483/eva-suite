    Azure Infrastructure Plan - EVA 2.0¶

    Date: November 28, 2025
    Product Owner: Marco Presta
    Infrastructure Owner: Windows Guy
    Scrum Master: Copilot (GitHub Copilot)

    Executive Summary¶

    Decision: Azure shared infrastructure approved by Marco
    Architecture: 1 Static Web App + 1 Cosmos DB (3 logical environments: dev, test, prod)
    Cost: $36/month (64% savings vs $99/month separate deployments)
    Owner: Windows Guy (not Copilot) - IaC creation and Azure provisioning
    Testing Strategy: Terraform validation + dev smoke test → test validation → prod deployment
    Rollback Focus: Patches and minor releases (higher priority than public webpage deployment)

    Architecture Overview¶

    Shared Infrastructure Model¶

    Benefits: - Single DNS zone (eva-suite.marcopolo483.dev or similar) - Unified SSL/TLS certificate management - Shared authentication/authorization layer - Environment routing via path prefixes or subdomains - 64% cost reduction ($99 → $36/month)

    Terraform Testing Strategy¶

    Marco's Concern¶

    "Unless you confirm it can be tested in some terraform best practices workbench."

    Answer: YES - Terraform has robust testing capabilities before applying to Azure.

    Testing Levels¶

    Level 1: Syntax Validation (Pre-commit)¶

    # Run in eva-infra/terraform/
terraform fmt -check    # Check formatting
terraform validate      # Check syntax and references

    Catches: - Typos in resource names - Missing required arguments - Invalid attribute references - Incorrect data types

    Time: ~5 seconds
    Owner: Windows Guy (before committing IaC)

    Level 2: Plan Preview (Pre-apply)¶

    terraform plan -out=tfplan

    Output: 

    Terraform will perform the following actions:

  # azurerm_static_site.eva_suite will be created
  + resource "azurerm_static_site" "eva_suite" {
      + id                = (known after apply)
      + name              = "marcopolo483-eva-suite"
      + location          = "canadacentral"
      + sku_tier          = "Standard"
      ...
    }

  # azurerm_cosmosdb_account.eva_suite will be created
  + resource "azurerm_cosmosdb_account" "eva_suite" {
      + id                = (known after apply)
      + name              = "eva-suite-cosmos"
      + location          = "canadacentral"
      + offer_type        = "Standard"
      + kind              = "GlobalDocumentDB"
      ...
    }

Plan: 2 to add, 0 to change, 0 to destroy.

    Catches: - What resources will be created/modified/deleted - Estimated costs (with Azure Cost Management integration) - Resource dependencies and ordering - Data loss risks (e.g., replacing instead of updating)

    Time: ~10-15 seconds
    Owner: Windows Guy (reviews plan, sends to Marco for approval)

    Approval Gate: Marco reviews plan output, approves via Slack/Teams message

    Level 3: Dev Environment Apply (Safe Testing)¶

    # Apply to dev subscription first (or dev resource group)
terraform apply tfplan

    Validates: - Azure credentials work - Resource names are available (not already taken) - Networking and firewall rules function - Connection strings and secrets are correct

    Time: ~5-10 minutes (Azure provisioning time)
    Owner: Windows Guy (applies to dev, runs smoke tests)

    Level 4: Smoke Tests (Post-deploy Validation)¶

    See Smoke Test Plan section below.

    Validates: - Static Web App is reachable (HTTP 200) - Cosmos DB accepts connections - Authentication works (if Entra ID integrated) - Environment variables are correctly set

    Time: ~5-10 minutes (automated tests)
    Owner: Copilot (runs automated tests), Marco (manual spot-checks)

    Level 5: Test Environment Validation¶

    # Promote same Terraform to test resource group
terraform workspace select test  # or use different tfvars
terraform apply

    Validates: - Configuration works in isolated environment - Data migrations succeed (if any) - Performance under load (if load tests added)

    Time: ~10-15 minutes
    Owner: Windows Guy (applies), Marco (UAT testing)

    Level 6: Production Deployment (Blue/Green)¶

    terraform workspace select prod
terraform apply

    Includes: - Blue/green slot deployment (zero downtime) - Canary release (5% traffic, monitor, then 100%) - Automated health checks (every 30 seconds) - Rollback trigger (5 consecutive failed health checks)

    Time: ~15-20 minutes (includes monitoring window)
    Owner: Windows Guy (applies), Marco (final approval to switch traffic)

    Smoke Test Plan¶

    Objective¶

    Validate that Azure infrastructure is functional BEFORE deploying application code.

    Automated Tests (via GitHub Actions)¶

    Test 1: Static Web App Reachability¶

    #!/bin/bash
# File: eva-orchestrator/scripts/smoke-test-static-web-app.sh

STATIC_WEB_APP_URL="https://marcopolo483-eva-suite.azurestaticapps.net/dev"

echo "Testing Static Web App reachability..."
HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$STATIC_WEB_APP_URL")

if [ "$HTTP_STATUS" -eq 200 ] || [ "$HTTP_STATUS" -eq 404 ]; then
  echo "✅ Static Web App is reachable (HTTP $HTTP_STATUS)"
  exit 0
else
  echo "❌ Static Web App is NOT reachable (HTTP $HTTP_STATUS)"
  exit 1
fi

    Success Criteria: - HTTP 200 (if default placeholder page exists) - HTTP 404 (if no content yet, but server is responding)

    Test 2: Cosmos DB Connectivity¶

    #!/bin/bash
# File: eva-orchestrator/scripts/smoke-test-cosmos-db.sh

COSMOS_ENDPOINT="https://eva-suite-cosmos.documents.azure.com:443/"

echo "Testing Cosmos DB connectivity..."

  --name eva-suite-cosmos \

  exit 1
fi

    Success Criteria: - Provisioning state = "Succeeded" - Can authenticate with primary key - Can list databases/containers

    Test 3: Environment Variables Set¶

    #!/bin/bash
# File: eva-orchestrator/scripts/smoke-test-env-vars.sh

echo "Testing Static Web App environment variables..."

# Check if required env vars are set in Azure
ENV_VARS=$(az staticwebapp appsettings list \
  --name marcopolo483-eva-suite \
  --query "properties" \
  --output json)

REQUIRED_VARS=("COSMOS_ENDPOINT" "COSMOS_KEY" "ENV_NAME")

for VAR in "${REQUIRED_VARS[@]}"; do
  if echo "$ENV_VARS" | grep -q "$VAR"; then
    echo "✅ $VAR is set"
  else
    echo "❌ $VAR is NOT set"
    exit 1
  fi
done

echo "✅ All required environment variables are set"
exit 0

    Success Criteria: - All required env vars present - No secrets logged (redacted in output)

    Test 4: DNS and SSL Certificate¶

    #!/bin/bash
# File: eva-orchestrator/scripts/smoke-test-dns-ssl.sh

DOMAIN="eva-suite.marcopolo483.dev"

echo "Testing DNS resolution..."
IP=$(dig +short "$DOMAIN")
if [ -z "$IP" ]; then
  echo "❌ DNS not resolving for $DOMAIN"
  exit 1
else
  echo "✅ DNS resolves to $IP"
fi

echo "Testing SSL certificate..."
SSL_EXPIRY=$(echo | openssl s_client -servername "$DOMAIN" -connect "$DOMAIN:443" 2>/dev/null | \
  openssl x509 -noout -dates | grep "notAfter" | cut -d= -f2)

echo "✅ SSL certificate valid until $SSL_EXPIRY"
exit 0

    Success Criteria: - DNS resolves to Azure Static Web App IP - SSL certificate is valid and not expired - TLS 1.2+ enforced

    Rollback Strategy¶

    Marco's Primary Concern¶

    "I am more concerned on the possibility to have to rollback patches and minor releases than the possibility of public webpage deployment not working. is that a valid concern?"

    Answer: YES - Valid and important concern. Patches/minor releases are higher risk than UI deployments.

    Why Patches Are Higher Risk¶

    Patch Examples: - Security vulnerability fixes (CVE patches) - Database schema changes (add column, change data type) - API contract changes (add required field, change response format) - Business logic corrections (tax calculation, access control)

    Risks: - Data corruption: Patch may modify data in place (hard to undo) - Breaking changes: Downstream services may depend on old behavior - Performance regression: New code may be slower (not caught in testing) - Security regression: Fix may introduce new vulnerability

    UI Deployment Risks (Lower): - Visual bugs (annoying but not critical) - Layout issues (can be fixed with CSS hotfix) - Broken links (404s, fixable quickly) - Usually no data impact

    Rollback Levels¶

    Level 1: Application Code Rollback (Fast - 2 minutes)¶

    Trigger: Automated health check fails 5× consecutively

    Mechanism: 

    # .github/workflows/deploy-prod.yml
- name: Deploy to Production
  uses: Azure/static-web-apps-deploy@v1
  with:
    slot: blue  # Deploy to blue slot first

- name: Health Check
  run: |
    for i in {1..5}; do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://blue-slot.azurestaticapps.net/api/health)
      if [ "$STATUS" -ne 200 ]; then
        echo "Health check failed ($STATUS), rolling back..."
        az staticwebapp deployment slot swap --name eva-suite --slot blue --target-slot production --resource-group eva-suite-rg
        exit 1
      fi
      sleep 10
    done

    Rollback Command: 

    # Swap back to previous deployment
az staticwebapp deployment rollback \
  --name marcopolo483-eva-suite \
  --resource-group eva-suite-rg

    Time: ~2 minutes (instant traffic switch, 2 min DNS propagation)

    Level 2: Database Schema Rollback (Medium - 10-30 minutes)¶

    Trigger: Manual decision by Marco or Windows Guy after detecting data issues

    Mechanism: 1. Versioned Migrations: All schema changes tracked in eva-suite/migrations/ 2. Rollback Scripts: Every migration has corresponding down script 

    -- migrations/002_add_user_preferences.up.sql
ALTER TABLE users ADD COLUMN preferences JSONB;

-- migrations/002_add_user_preferences.down.sql
ALTER TABLE users DROP COLUMN preferences;

    1. Migration Tool: Use Flyway or Liquibase for Cosmos DB (or custom script)

    Rollback Command: 

    # Rollback last migration
npm run migrate:down  # or dotnet ef migrations remove

    Time: 10-30 minutes (depends on data volume, may require backup restore)

    Level 3: Infrastructure Rollback (Slow - 30-60 minutes)¶

    Trigger: Terraform changes break environment (rare)

    Mechanism: 1. Terraform State Backup: State file backed up before every apply 

    terraform state pull > backup/terraform.tfstate.$(date +%Y%m%d_%H%M%S)

    1. Rollback to Previous State: 

      terraform state push backup/terraform.tfstate.20251128_140000
terraform apply -auto-approve

    2. Re-provision from Git: Checkout previous commit, re-apply Terraform 

      git checkout HEAD~1 -- eva-infra/terraform/
terraform apply

    Time: 30-60 minutes (Azure provisioning time)

    Rollback Decision Matrix¶

    Change Type Risk Level Rollback Method Time to Rollback Requires Marco Approval?
    UI-only deployment Low Slot swap 2 min No (automated)
    Minor release (no schema change) Medium Slot swap + feature flag disable 5 min No (PSM decision)
    Patch (with schema change) High Slot swap + migration rollback 10-30 min Yes (Marco approval)
    Major release (breaking API) Critical Full rollback + comms 30-60 min Yes (Marco + stakeholders)
    Infrastructure change Critical Terraform state restore 30-60 min Yes (Marco + Windows Guy)

    Rollback Testing (Part of Smoke Test)¶

    Test: Simulate failed deployment and verify rollback works

    #!/bin/bash
# File: eva-orchestrator/scripts/test-rollback.sh

echo "Deploying intentionally broken version..."

  exit 1
fi

    Frequency: Run monthly (automated via GitHub Actions scheduled workflow)

    Deployment Pipeline¶

    Branching Strategy¶

    main (prod) ──→ Deploy to /prod environment
  ↑
test (staging) ──→ Deploy to /test environment
  ↑
dev (integration) ──→ Deploy to /dev environment
  ↑
feature/* (local) ──→ No auto-deploy, PR to dev

    CI/CD Workflow¶

    # .github/workflows/deploy-azure.yml
name: Deploy to Azure Static Web Apps

on:
  push:
    branches: [dev, test, main]
  pull_request:
    branches: [dev, test, main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Determine Environment
        id: env
        run: |
          if [[ "${{ github.ref }}" == "refs/heads/main" ]]; then
            echo "environment=prod" >> $GITHUB_OUTPUT
            echo "slot=production" >> $GITHUB_OUTPUT
          elif [[ "${{ github.ref }}" == "refs/heads/test" ]]; then
            echo "environment=test" >> $GITHUB_OUTPUT
            echo "slot=test" >> $GITHUB_OUTPUT
          else
            echo "environment=dev" >> $GITHUB_OUTPUT
            echo "slot=dev" >> $GITHUB_OUTPUT
          fi

      - name: Build
        run: |
          npm ci
          npm run build

      - name: Run Tests
        run: npm test

      - name: Deploy to Azure
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: "upload"
          app_location: "/apps/eva-suite"
          api_location: "/apps/eva-suite/api"
          output_location: "build"
          deployment_environment: ${{ steps.env.outputs.environment }}

      - name: Smoke Tests
        run: |
          bash scripts/smoke-test-static-web-app.sh
          bash scripts/smoke-test-cosmos-db.sh
          bash scripts/smoke-test-env-vars.sh

      - name: Notify on Failure
        if: failure()
        run: |
          # Send Slack/Teams notification
          curl -X POST ${{ secrets.SLACK_WEBHOOK_URL }} \
            -H 'Content-Type: application/json' \
            -d '{"text":"🚨 Azure deployment FAILED: ${{ github.ref }}"}'

    Daily Cost Tracking¶

    #!/bin/bash
# File: eva-orchestrator/scripts/check-azure-costs.sh

TODAY=$(date +%Y-%m-%d)
COST=$(az consumption usage list \
  --start-date $(date -d '1 day ago' +%Y-%m-%d) \
  --end-date $TODAY \
  --query "[].{cost:pretaxCost}" \
  --output tsv | awk '{sum+=$1} END {print sum}')

echo "Azure cost for $TODAY: \$$COST"

if (( $(echo "$COST > 5" | bc -l) )); then
  echo "⚠️ Daily cost exceeds $5, investigating..."
  az consumption usage list --start-date $(date -d '1 day ago' +%Y-%m-%d) --end-date $TODAY
fi

    Schedule: Run daily via GitHub Actions cron job, log to eva-orchestrator/logs/azure-costs.log

    Offline Resilience (Marco's Priority)¶

    Observation from Morning Internet Outage¶

    "I woken up this morning and i had no internet access. That's one of advantages of having the team in the cloud."

    Implication: Autonomous AI crew (Master Scrum Master, PSM-1 to PSM-4, DevMaster, QA) continues working 24/7 even when Marco is offline.

    Architecture Benefits: 1. GitHub Actions run in cloud: No dependency on Marco's machine being online 2. Agent spawns happen in cloud: OpenAI API calls work regardless of Marco's internet 3. Daily standups posted automatically: MSM workflow runs on schedule, Marco sees results when reconnected 4. Deployments happen automatically: Merge to main triggers deployment, no manual intervention

    Trade-off for Local GPU Model: - Local Llama 3.2 70B would run on Marco's machine - If Marco's internet goes down → machine may be powered off → agents can't work - Argues FOR cloud-based OpenAI API (resilience over cost savings)
