    GC Agentic Framework (vNext – Draft)¶

    A Government of Canada Model for Safe, Governed, and Transparent AI Agents

    Section A — Scope & Definitions¶

    A.1 Purpose¶

    This framework establishes a Government of Canada–specific model for agentic AI systems, including definitions, risk classifications, guardrails, and operational controls required for the safe deployment of AI agents in federal environments.

    It is designed as an overlay to:

    • ITSG-33
    • NIST AI RMF 1.0 (Govern–Map–Measure–Manage)
    • ISED’s Voluntary Code of Conduct on AI
    • TBS Responsible Use of AI guidance

    This framework can be applied within ATO processes for any federal AI system, including EVA, Jurisprudence, AssistMe, Knowledge Management, CPP-D, and departmental agentic solutions.

    A.2 What Is an Agentic AI System?¶

    Based on UK Government definitions (AI Insights: Agentic AI):

    An agentic AI system is an AI model or system capable of performing multi-step tasks, invoking tools or APIs, executing workflows, and making context-dependent decisions with varying degrees of autonomy.

    Characteristics include:

    • tool use
    • planning and sequencing
    • memory or state management
    • the ability to act on or transform information
    • optional interaction with external services

    A.3 GC Agentic Levels¶

    To ensure clarity, the Government of Canada adopts a four-level taxonomy:

    Level 0 — Non-Agentic AI¶

    Traditional generative AI (RAG, Q&A, summarization). No tool use or workflow execution. Examples: EVA Chat, EVA DA (basic mode).

    Level 1 — Bounded Tool Helpers¶

    LLM can call a small, fully whitelisted set of tools. No loops, no long-running planning, no external integrations beyond APIM. Examples:

    • “Search jurisprudence” API
    • “Get policy info” API
    • “Summarize document” API

    Risk: Low → Moderate.

    Level 2 — Workflow Agents¶

    Agents can:

    • break tasks into steps
    • call multiple tools
    • perform limited planning
    • produce drafts for human approval
    • operate under strict policy constraints

    All high-impact actions require explicit human approval. Examples:

    • Jurisprudence case analyzer
    • KM auto-indexer agent
    • CPP-D document triage agent

    Risk: Moderate → High.

    Level 3 — Autonomous or Multi-Agent Systems¶

    Agents capable of:

    • open-ended planning
    • learning/adapting across tasks
    • interacting with unknown tools or other agents

    These are restricted to sandbox/lab environments only. Not permitted in production federal systems.

    Risk: High → Prohibited in production.

    Section B — Public Sector Context¶

    B.1 Why Agents Matter for Government¶

    Based on global public-sector research (Agentic State, GovAI, OECD, UK):

    AI agents can support:

    1. Service Delivery

    2. benefits navigation

    3. jurisprudence information retrieval

    4. agricultural support (ex: AgPal Chat)

    5. Policy Development & Evidence Synthesis

    6. case law analysis

    7. literature aggregation

    8. trend detection

    9. Internal Operations

    10. DevOps helpers

    11. document classification
    12. triage routing

    13. ticket analysis

    14. Compliance & Security

    15. log analysis

    16. anomaly detection
    17. pattern recognition

    B.2 Maximum Agent Levels Recommended¶

    Use Case Category Recommended Level Notes
    Public-facing search tools L0–L1 Transparency, low-risk APIs
    Internal analysis (evidence, jurisprudence, CPP-D) L1–L2 Human approval required
    Operational support (DevOps, KM) L1–L2 Tool whitelist mandatory
    Workflow automation L2 only No autonomous loops
    Experimental research L3 Sandbox only

    Section C — Risk Taxonomy for Agentic Systems¶

    Adapted from NIST AI RMF + AI Security Guidelines.

    C.1 Key Agentic Risks¶

    1. Prompt Injection → Tool Misuse
    2. Emergent Workflow Drift
    3. Autonomy Creep (agent exceeding intended scope)
    4. Data Exfiltration via Tools or Memory
    5. Unauthorized Cross-Domain Operations
    6. Model or Tool Supply Chain Risks
    7. Opaque Planning Logic (explainability issues)
    8. Inaccurate or Biased Actions
    9. Interaction with External Agents or Models
    10. Escalation of Privilege Through Tool Orchestration

    C.2 Threat Scenarios¶

    • Model manipulated to call a privileged API inappropriately
    • Workflow planner discovers a novel (unsafe) method to complete a task
    • Agent storing sensitive data in long-term memory unexpectedly
    • External malicious documents triggering harmful tool calls
    • Agent unintentionally triggering downstream actions without human approval

    Section D — Control Requirements & Mapping¶

    The following controls apply per agent level.

    D.1 Control Themes¶

    • Access control (AC)
    • Tool security (SC, SA, SI)
    • Logging & auditability (AU)
    • Human oversight (AT, PE)
    • Boundary protection (SC)
    • Testing & evaluation (CA, RA)
    • Fail-safes (IR, CP)

    D.2 Level-by-Level Controls¶

    Level 0¶

    • Standard ITSG-33
    • RAG guardrails
    • Transparency statement

    Level 1¶

    • Tool whitelist
    • APIM enforcement
    • Per-tool RBAC
    • Full audit logs
    • Cost limits
    • Red-teaming before deployment

    Level 2¶

    Everything in Level 1 plus:

    • Mandatory human approval for high-impact actions
    • Workflow traceability & replay
    • Sandbox planning mode
    • Policy agent verifying decisions
    • Kill-switch at orchestrator level
    • Continuous monitoring & periodic re-certification

    Level 3¶

    • Not permitted in production
    • Sandbox only with isolation + high-assurance logging

    Section E — Operational Patterns (Mandatory)¶

    Inspired by EY, PwC, TR, CISA, Palo Alto, and enterprise AI governance.

    E.1 Run Agents Through a Secure Orchestrator Layer¶

    All agents must operate through a GC-approved orchestration platform such as EVA, enforcing:

    • APIM-mediated tool execution
    • strict RBAC
    • token-level isolation
    • per-action audit logs
    • policy guardrails
    • FinOps tagging & cost controls

    E.2 Mandatory Safe-Operation Patterns¶

    1. Tool Whitelisting No dynamic tool discovery.

    2. Human-in-the-Loop Required for all Level 2 actions with operational impact.

    3. Dry-Run Mode Agents produce suggested plans before execution.

    4. Reversibility & Rollback All downstream actions must be reversible or logged for incident response.

    5. Continuous Monitoring Automated detection of policy violations or anomalous behavior.

    6. Kill-Switch Mechanism Must exist at:

    7. agent level

    8. tool level

    9. platform level

    10. Security Testing & Red-Team Exercises Before promotion to production.

    Section F — Roles, Responsibilities & RACI¶

    F.1 Required Roles¶

    • Agent Orchestrator (Platform Owner – EVA)
    • AI Safety & Compliance Engineer
    • Domain Steward (jurisprudence, CPP-D, etc.)
    • Security Authority (SA)
    • Privacy Lead (ATIP/PIA)
    • Human Approver (domain expert)
    • Model Developer / Prompter
    • Monitoring Operator (DevOps / MLOps)

    F.2 Accountability Model¶

    Design¶

    • Orchestrator + Domain Steward
    • Security & Privacy review (SA/PIA)

    Deployment¶

    • Platform Owner (EVA)
    • Security testing lead
    • Ops owner

    Operation¶

    • Human Approvers
    • Domain Stewards
    • Monitoring Operators

    Incident Response¶

    • Platform Owner + SA
    • Domain Steward
    • Communications / ATIP when applicable

    Section G — Transparency & Documentation¶

    G.1 User-Facing Transparency¶

    • Disclose AI involvement clearly
    • Provide explanation of workflow steps
    • Provide limitations and safe-use guidance

    G.2 Documentation Package (ATO-Ready)¶

    Every agentic system must maintain:

    1. System Card

    2. scope

    3. capabilities
    4. limitations

    5. intended users

    6. Data Card

    7. sources

    8. update frequency

    9. quality thresholds

    10. Risk & Control Register

    11. mapped to ITSG-33

    12. mapped to NIST AI RMF

    13. Testing & Evaluation Report

    14. bias testing

    15. red-teaming

    16. performance evaluation

    17. Operational Logs

    18. full replay of tool actions

    19. model decisions
    20. human approvals

    Section H — GC Production Rules¶

    H.1 Allowed Levels¶

    Environment Allowed Levels Notes
    Production L0–L2 L2 requires ATO + human approval
    Protected B L0–L1 (L2 with conditions) No external APIs
    Internal R&D L0–L3 Full isolation required

    Section I — EVA as Reference Implementation¶

    This framework recommends a centralized GC agentic execution platform. EVA satisfies this role by providing:

    • APIM-controlled tool use
    • secure multi-tenant sandboxes
    • RAG pipelines
    • logging & audit
    • FinOps monitoring
    • bilingual WCAG 2.2 UI
    • policy-governed prompts
    • integration with ATO processes
    • clear cost attribution

    EVA can serve as the reference agentic platform across ESDC — with optional onboarding via IIAID.

    Section J — Conclusion¶

    This GC Agentic Framework provides:

    • a common vocabulary (Levels 0–3),
    • a unified risk model,
    • controls tied to ITSG-33 / NIST AI RMF,
    • operational patterns,
    • required roles,
    • transparency standards,
    • and a reference architecture via EVA.

    It enables departments to safely adopt agentic AI systems while maintaining public trust, auditability, and compliance.
