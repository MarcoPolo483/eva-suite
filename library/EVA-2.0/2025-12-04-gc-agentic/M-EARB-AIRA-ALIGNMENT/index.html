    EARB AI Reference Architecture (AIRA) Alignment¶

    Document Purpose: Show how GC Agentic Framework operationalizes ESDC's EARB-approved AI Reference Architecture
    Audience: Enterprise Architecture Review Board (EARB), Shannon, Todd, Niasha, Directors
    Date: December 4, 2025
    Status: Alignment Analysis Complete

    üìã Executive Summary¶

    The GC Agentic Framework is NOT introducing something new - it operationalizes what EARB already approved.

    ESDC's Enterprise Architecture Review Board (EARB) approved the AI Reference Architecture (AIRA) which explicitly defines Agentic AI as the highest capability layer in ESDC's AI mission journey:

    AI Mission Journey: Data ‚Üí Predictive ‚Üí Generative ‚Üí Agentic Execution

    What AIRA Provides: - ‚úÖ Architectural vision (agentic AI as final capability layer) - ‚úÖ High-level capabilities (AI-specific security, application capabilities) - ‚úÖ Security mapping (aligned to CSRA C1-C7) - ‚úÖ Policy references (ITSG-33, NIST AI RMF, ISO 42001, OWASP LLM, MITRE ATLAS, SAIF)

    What AIRA Does NOT Provide (The Gap This Template Fills): - ‚ùå Agent levels (Level 0-3 maturity model) - ‚ùå Risk categories (agentic-specific threats) - ‚ùå Governance patterns (tool whitelisting, kill switches) - ‚ùå RACI model (roles and responsibilities) - ‚ùå Approval thresholds (when human-in-loop required) - ‚ùå Operational constraints (production rules, sandbox boundaries) - ‚ùå ATO annex (controls mapping to ITSG-33)

    Conclusion: This framework completes AIRA by providing the operational standard the architecture requires.

    üéØ Key Alignment Points¶

    1. Agentic AI Already EARB-Approved¶

    AIRA Position: Agentic AI is the final evolutionary layer in the AI capability stack: - Layer 1: Data & Analytics - Layer 2: Predictive AI (ML models) - Layer 3: Generative AI (LLMs, RAG) - Layer 4: Agentic Execution ‚Üê EARB-approved architectural layer

    GC Template Position: Provides maturity model (Levels 0-3) and operational guidance for Layer 4

    Alignment: Template operationalizes what architecture defines

    2. Security Capabilities Alignment¶

    AIRA Security Framework: AI-specific security capabilities aligned to CSRA (C1-C7): - C1: Threat Intelligence - C2: Security Operations - C3: Identity & Access Management - C4: Data Protection - C5: Infrastructure Security - C6: Endpoint Security - C7: Security Awareness & Training

    GC Template Security Framework: ITSG-33 control families + NIST AI RMF overlay: - AC (Access Control) - maps to C3 - AU (Audit & Accountability) - maps to C2 - SC (System & Communications Protection) - maps to C5 - SI (System & Information Integrity) - maps to C2, C4 - RA (Risk Assessment) - maps to C1 - IR (Incident Response) - maps to C2

    Alignment: Template provides detailed control implementation for AIRA security capabilities

    3. AI Reference Model Alignment¶

    AIRA Components: - AI orchestration - Guardrails & governance - Tool use & external integrations - Multi-step workflow execution - Human oversight mechanisms

    GC Template Components: - Orchestrator layer (EVA as reference implementation) - Policy guardrails (configurable constraints) - Tool registry & APIM-governed calls - Workflow traceability & replay - Human-in-loop approval patterns

    Alignment: Template implements all AIRA architectural components

    4. Policy & Standards Alignment¶

    AIRA References: - ITSG-33 (GC IT security controls) - NIST AI RMF (AI risk management) - ISO 42001 (AI management system) - OWASP LLM (LLM security) - MITRE ATLAS (AI threat matrix) - SAIF (Secure AI Framework)

    GC Template References: - ITSG-33 (complete control mapping) - NIST AI RMF 1.0 (Govern-Map-Measure-Manage) - ISED Voluntary Code (Canadian AI governance) - TBS Responsible AI guidance - UK Gov AI Insights: Agentic AI - StatCan AgPal Chat lessons learned

    Alignment: Template extends AIRA's policy foundation with operational implementation

    üìä Capability Mapping Table¶

    AIRA Capability GC Template Implementation EVA Reference Implementation
    Agentic Layer Definition Maturity Model (Levels 0-3) EVA at Level 2.5-3
    Orchestration Orchestrator patterns, APIM mediation EVA Core orchestrator
    Tool Use Tool registry, whitelisting, RBAC EVA tool registry + APIM
    Guardrails Policy constraints, content filters EVA policy engine
    Workflow Execution Multi-step planning, traceability EVA workflow engine (P01-P15)
    Human Oversight Human-in-loop approval patterns EVA approval workflows
    Audit & Logging Full trace of actions, replay capability EVA audit logs + LiveOps
    Security Controls ITSG-33 + NIST AI RMF mapping EVA security architecture
    Kill Switches 3-level kill-switch pattern EVA emergency stop mechanisms
    Monitoring Continuous monitoring, anomaly detection EVA LiveOps dashboards

    üîí Security Controls Detailed Mapping¶

    AIRA Security Capability ‚Üí GC Template Control ‚Üí ITSG-33 Family¶

    C1: Threat Intelligence - AIRA: Monitor AI-specific threats (prompt injection, data poisoning) - Template: Risk taxonomy (10 agentic-specific risks) - ITSG-33: RA (Risk Assessment), SI (System & Information Integrity)

    C2: Security Operations - AIRA: Detect and respond to AI incidents - Template: Continuous monitoring, kill switches, incident response playbooks - ITSG-33: AU (Audit), IR (Incident Response), CP (Contingency Planning)

    C3: Identity & Access Management - AIRA: Control who can invoke AI systems - Template: RBAC for tools, per-user authentication, approval workflows - ITSG-33: AC (Access Control), IA (Identification & Authentication)

    C4: Data Protection - AIRA: Protect training data, RAG sources, outputs - Template: Data boundary enforcement, PII handling, classification controls - ITSG-33: SC (System & Communications Protection), MP (Media Protection)

    C5: Infrastructure Security - AIRA: Secure AI hosting environments - Template: Orchestrator isolation, sandbox boundaries, network segmentation - ITSG-33: SC (System & Communications Protection), PE (Physical & Environmental)

    C6: Endpoint Security - AIRA: Secure user devices accessing AI - Template: Standard ESDC endpoint controls (not AI-specific) - ITSG-33: SI (System & Information Integrity)

    C7: Security Awareness & Training - AIRA: Train staff on AI risks - Template: User disclosure, limitations documentation, safe-use guidance - ITSG-33: AT (Awareness & Training)

    üß≠ Maturity Model & AIRA Journey Alignment¶

    AIRA Journey: Data ‚Üí Predictive ‚Üí Generative ‚Üí Agentic

    GC Template Maturity Model: - Level 0 (Non-Agentic): Traditional AI, RAG, Q&A - maps to AIRA "Generative" layer - Level 1 (Tool-Bound): Single-tool helpers - entry to AIRA "Agentic" layer - Level 2 (Workflow Agents): Multi-step planning, governed autonomy - full AIRA "Agentic" layer - Level 3 (Multi-Agent): Sandbox only, complex orchestration - advanced AIRA "Agentic" - Level 4 (Autonomous): Prohibited - beyond AIRA scope

    ESDC Current State: - Most departments: Level 0-1 (Q&A chatbots, basic RAG) - EVA Suite: Level 2.5-3 (workflow agents, bounded multi-agent) - Jurisprudence pilot: Targeting Level 2 (workflow agent with human approval)

    Alignment: Template provides clear path from current state (Level 0-1) to AIRA vision (Level 2-3)

    üìù Governance & Decision-Making¶

    AIRA Governance (What Exists)¶

    • EARB approval of architectural vision
    • High-level security capabilities
    • Policy reference framework

    GC Template Governance (What's Added)¶

    • RACI Model: 8 required roles (Orchestrator, Safety Engineer, Domain Steward, Security Authority, Privacy Lead, Human Approver, Model Developer, Monitoring Operator)
    • Approval Thresholds: When PMD/ITAO/Security sign-off required
    • Production Rules: Level 0-2 allowed in production, Level 3 sandbox only, Level 4 prohibited
    • Change Management: Version control for prompts, policies, tool registry
    • Periodic Re-Certification: Ongoing validation of controls

    Alignment: Template provides decision framework AIRA requires for operationalization
