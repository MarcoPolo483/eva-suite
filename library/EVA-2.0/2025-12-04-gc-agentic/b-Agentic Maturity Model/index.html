    **B ‚Äî Government of Canada Agentic Maturity Model (Levels 0‚Äì3)¶

    ‚ö™ LEVEL 0 ‚Äî NO AGENTIC BEHAVIOUR¶

    Description¶

    System has no autonomy. No agents. No tools. Only static, single-turn responses.

    Characteristics¶

    • Simple search
    • Basic FAQ chatbots
    • No reasoning
    • No workflow execution
    • No memory
    • No tool-use

    Governance Requirements¶

    • Only basic privacy and accessibility
    • No agent policies needed
    • Minimal logging

    GC Examples¶

    • Legacy departmental chatbots
    • Search-based intranet widgets

    EVA Mapping¶

    EVA does not operate at Level 0. This is pre-EVA.

    üü¢ LEVEL 1 ‚Äî ASSISTED INTELLIGENCE (Tool-aware, single-step)¶

    Description¶

    Agent can answer questions using approved content and perform one-step actions. No planning. No self-directed workflows.

    Capabilities¶

    • Retrieval from trusted content (RAG)
    • Classification / summarization
    • Drafting assistance
    • One-shot transformations
    • Inline policy guardrails

    Boundaries¶

    • Cannot plan
    • Cannot call multiple tools
    • Cannot chain tasks
    • Cannot act without a user

    Governance Requirements¶

    • Tool restrictions
    • Logging of prompts/responses
    • Clear disclaimers
    • Bilingual outputs
    • A11y compliant UI

    GC Examples¶

    • Copilot in Office apps
    • Basic ‚ÄúAI writing assistant‚Äù
    • Drafting assistants

    EVA Mapping¶

    • EVA Chat
    • Simple Domain Assistant queries
    • Baseline ingestion + RAG

    EVA fully satisfies Level 1 and exceeds it.

    üü° LEVEL 2 ‚Äî COORDINATED AGENT (Multi-step, bounded autonomy)¶

    Description¶

    The agent can plan, execute multi-step workflows, call multiple tools, and refine its own reasoning ‚Äî but only inside guardrails.

    This is where most GC departments stop because they fear Level 3.

    Capabilities¶

    • Bounded planning (task ‚Üí steps ‚Üí execution)
    • Multi-step retrieval + synthesis
    • Iterative refinement
    • Basic memory (session-based)
    • Multiple tool calls
    • Can detect its own failures

    Boundaries¶

    • No cross-agent orchestration
    • No self-repair or self-reconfiguration
    • No external communication
    • Workflow must remain inside approved toolset

    Governance Requirements¶

    • RA/IDE alignment
    • Tool execution logging
    • Workflow explainability
    • Retrieval guardrails
    • Safety/off-switch triggers
    • Testing evidence (bias, hallucination, accuracy)
    • Clear human-in-the-loop checkpoints

    GC Examples¶

    • Planning assistants with bounded steps
    • Agents that retrieve ‚Üí summarize ‚Üí draft
    • Multi-tool copilots

    EVA Mapping¶

    This is where EVA Suite shines:

    • P02 Requirements Engine
    • P03 Reviewer
    • P04 Planner
    • P05 Developer Builder
    • EVA RAG ingestion pipeline
    • EVA Sovereign UI + multimodal workflow UIs

    EVA is Level 2.5 natively.

    This is what solves agentic fear: ‚û°Ô∏è EVA has maturity boundaries that prevent uncontrolled behaviour.

    üîµ LEVEL 3 ‚Äî ORCHESTRATED MULTI-AGENT SYSTEM (GC-governed autonomy)¶

    Description¶

    Multiple agents coordinate, share plans, pass tasks, and correct each other ‚Äî all under strict governance and auditing.

    This is not ‚ÄúAGI-like autonomy.‚Äù It is GC-governed collaborative automation.

    Capabilities¶

    • Multi-agent orchestration
    • Shared memory across agents
    • Automatic handoff between roles
    • Self-critique and self-correction
    • Distributed tool use across roles
    • Execution with audit trails

    Boundaries¶

    • Agents cannot create new agents
    • Agents cannot modify guardrails
    • Agents cannot access new data sources
    • Agents cannot bypass identity or RBAC
    • All orchestration is auditable

    Governance Requirements¶

    • Continuous monitoring (RA/IDE)
    • ATO for sandbox/pilot/production tiers
    • Tool wrapping with role-based permissions
    • Centralized logging and evidence management
    • Explainability for all steps
    • Content provenance
    • Safety policies enforced at runtime

    GC Examples¶

    • Projects in early pilot stages at SSC, AAFC, CRA
    • Tool-based multi-agent copilots in RPA admissions

    EVA Mapping¶

    You have already built a complete Level 3-ready architecture:

    • Jarvis (Scrum Master) = Orchestrator
    • P01‚ÄìP15 agents = Specialists
    • EVA Core = Guardrails + model access
    • EVA Directory = Agent registry
    • EVA Ops / LiveOps = Audit + telemetry
    • Ingestion + RAG + Tools = Foundation layer
    • EVA Sovereign UI = GC-ready interaction layer

    EVA is Level 3, but with policy boundaries that keep it safe.

    This is exactly what GC departments need.

    üî¥ LEVEL 4 ‚Äî (Prohibited for Government of Canada)¶

    GC explicitly forbids systems that:

    • operate without human oversight
    • modify their own policies
    • perform unsupervised autonomous actions
    • access unapproved external networks
    • learn unsupervised from live user prompts
    • act as independent decision-makers

    EVA naturally prevents Level 4 because EVA Core enforces:

    • bounded toolsets
    • human approval
    • external API restrictions
    • RBAC
    • ATO-bound environments
    • kill switches
    • audit trails
