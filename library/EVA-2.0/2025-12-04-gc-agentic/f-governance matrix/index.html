    F — EVA Agentic Governance Matrix¶

    How P01–P15 Agents Map to GC Safety, Policy, Legal, and Ethical Requirements¶

    How to Use This Matrix¶

    For every EVA Project Space, feature, RAG pipeline, or agent workflow, this matrix:

    • Demonstrates compliance
    • Shows mandatory guardrails
    • Defines accountability
    • Enforces GC policy boundaries
    • Eliminates Level 4 (prohibited) autonomy
    • Provides ATO-ready justification
    • Enables cross-department reuse

    LEGEND (for matrix columns)¶

    GC-P1 — Privacy Act & Data Minimization¶

    GC-P2 — Security (ITSG-33 / NIST 800-53)¶

    GC-P3 — Accessibility (WCAG 2.2 AA)¶

    GC-P4 — Official Languages Act (Bilingualism)¶

    GC-P5 — TBS Guide on Generative AI¶

    GC-P6 — DADM (Automated Decision-Making Directive)¶

    GC-P7 — RA/IDE (Continuous Monitoring)¶

    GC-P8 — GC Digital Standards¶

    GC-P9 — Prohibited Autonomy (Level 4 prevention)¶

    Each cell shows how the agent satisfies the requirement.

    1. AGENTIC GOVERNANCE MATRIX (P01–P15)¶

    P01 — Archivist Agent¶

    Role: Retrieves approved content, citations, metadata; enforces evidence-based outputs.

    GC Policy Compliance
    P1 Only approved, non-personal sources; redaction enforced.
    P2 Access controlled via RBAC; vector store classification enforced.
    P3 A11y metadata for all documents included.
    P4 Ensures bilingual retrieval; parallel EN/FR documents maintained.
    P5 Retrieval-bound guardrail to prevent hallucination.
    P6 Human reviewer required; no decisions made.
    P7 Logs all data-source calls for RA/IDE monitoring.
    P8 Open standards and provenance tracking used.
    P9 Cannot fetch from unapproved sources; cannot self-expand scope.

    P02 — Planner (Requirements & Multi-Step Planning Agent)¶

    Role: Converts user requests into structured, safe, bounded plans.

    GC Policy Compliance
    P1 Ensures plan does not involve personal data.
    P2 Applies tool permission scoping before execution.
    P3 Produces A11y-aligned structured plan output.
    P4 Plans generated bilingual-ready; metadata attached.
    P5 Enforces multi-step safety; avoids hallucinated steps.
    P6 Flags high-impact plans for mandatory human approval.
    P7 Logs plans, steps, safety checks.
    P8 Transparent, explainable plans align with Digital Standards.
    P9 Cannot initiate tasks autonomously; stop/go review always required.

    P03 — Reviewer Agent (Safety & Quality)¶

    Role: Audits accuracy, citations, hallucination risk, bias, bilingual parity.

    GC Policy Compliance
    P1 Verifies no personal data appears in output.
    P2 Audit logs, flags anomalies, ensures secure output formatting.
    P3 Validates formatting for screen readers, clear structure.
    P4 Ensures EN/FR equivalence of outputs.
    P5 Enforces Responsible AI guidelines (TBS).
    P6 Marks drafts “non-binding,” preventing DADM risk elevation.
    P7 Outputs a full audit trace with QA metrics.
    P8 Ensures transparency, reproducibility.
    P9 No auto-approve capability; must escalate to human.

    P04 — Orchestrator / Coordinator Agent¶

    Role: Assigns tasks to appropriate agents while respecting guardrails.

    GC Policy Compliance
    P1 Prevents personal data usage across agents.
    P2 Enforces RBAC tool boundaries.
    P3 Ensures workflow outputs meet A11y constraints.
    P4 Maintains bilingual workflow states.
    P5 Applies delegation safety policies.
    P6 Requires human approval for workflow execution.
    P7 Logs each agent interaction as RA/IDE evidence.
    P8 Ensures clear task decomposition and traceability.
    P9 No unsupervised workflows; always bounded.

    P05 — Builder / Composer Agent¶

    Role: Generates multi-document composites, case summaries, reports.

    GC Policy Compliance
    P1 Output sanitized; sensitive metadata removed.
    P2 Managed environment; no file system access.
    P3 Structured headings, ARIA-ready semantics.
    P4 Generates bilingual composite outputs.
    P5 Uses retrieval-only content; no invented data.
    P6 Only produces drafts; human-in-the-loop required.
    P7 Logs all sources used for composite.
    P8 Aligns with GC design standards (semantic structure).
    P9 Cannot autonomously publish or distribute.

    P06 — Research Agent¶

    Role: Conducts thematic or comparative legal/policy research.

    GC Policy Compliance
    P1 Uses sanitized datasets only.
    P2 All research actions within RBAC tools.
    P3 Outputs structured summaries compatible with screen readers.
    P4 Ensures results bilingualized.
    P5 Prevents unsupported inference; citation enforcement.
    P6 Does not make recommendations that lead to decisions.
    P7 Logs queries, sources, retrieved corpus.
    P8 Encourages open standards metadata.
    P9 Restricted to EVA-approved corpora only.

    P07 — Classifier Agent¶

    Role: Categorizes jurisprudence, documents, metadata.

    GC Policy Compliance
    P1 Only processes public case law.
    P2 RBAC tool boundaries for classification outputs.
    P3 Structured outputs for A11y.
    P4 Bilingual schemas enforced.
    P5 Deterministic mapping; avoids hallucinations.
    P6 No impact scores generated (avoiding DADM obligations).
    P7 Logs mappings and confidence.
    P8 Evidence-based, transparent reasoning.
    P9 Cannot create new categories without governance.

    P08 — Ingestion Agent¶

    Role: Extracts text, OCR, metadata, structure; prepares RAG entries.

    GC Policy Compliance
    P1 Redacts names and sensitive details during ingestion.
    P2 Ingestion pipeline runs in secured infra with audit.
    P3 Ensures parsed content retains A11y attributes.
    P4 Detects bilingual pairs and metadata.
    P5 Uses deterministic extraction rules.
    P6 Does not modify decisions or case law content.
    P7 Tracks ingestion jobs (RA/IDE evidence).
    P8 Adheres to open metadata standards.
    P9 Cannot ingest unapproved sources.

    P09 — Summarizer Agent¶

    GC Policy Compliance
    P1 Sanitizes summary; removes personal information.
    P2 Logs transformations; prevents leakage.
    P3 Structured summary headings readable by AT.
    P4 Bilingual summarization.
    P5 Retrieval-bound; no hallucinations.
    P6 Non-binding drafts only.
    P7 Logs summarization steps.
    P8 Encourages brevity and clarity.
    P9 Cannot run without retrieval context.

    P10 — Comparator Agent¶

    Role: Compares rulings or documents.

    GC Policy Compliance
    P1 Scrubs personal data during extraction.
    P2 Safe execution via RBAC tools.
    P3 A11y in structured deltas.
    P4 Produces bilingual comparison tables.
    P5 Uses deterministic extraction.
    P6 Draft comparisons for human validation.
    P7 Logs diff/evidence workflow.
    P8 Traceable reasoning steps.
    P9 Cannot adjudicate or interpret law.

    P11 — Translator Agent¶

    (If used)

    GC Policy Compliance
    P1 No PI included; translation of public content only.
    P2 Sandbox translation models only; no external API.
    P3 Preserves structure/readability.
    P4 EN/FR parity ensured.
    P5 Quality review by P03.
    P6 Human validation required.
    P7 Logs translation versions.
    P8 Upholds clarity and accuracy.
    P9 Cannot auto-publish.

    P12 — Evaluator Agent¶

    Role: Evaluates correctness of another agent’s output.

    GC Policy Compliance
    P1 Checks for prohibited content.
    P2 Enforces security boundaries.
    P3 Ensures evaluation output remains A11y-friendly.
    P4 Evaluates bilingual equivalence.
    P5 Applies TBS Responsible AI criteria.
    P6 Requires human sign-off for major findings.
    P7 Logs evaluation metrics.
    P8 Supports transparency & reproducibility.
    P9 Cannot suppress audit trails.

    P13 — Data Quality Agent¶

    GC Policy Compliance
    P1 Removes sensitive info; flags anomalies.
    P2 Validates dataset classification.
    P3 Ensures accessible formatting.
    P4 Identifies EN/FR mismatches.
    P5 Rejects hallucinated data.
    P6 Ensures datasets non-decisional.
    P7 Tracks corrections and errors.
    P8 Adheres to GC metadata standards.
    P9 Cannot alter source documents.

    P14 — Ops Agent (LiveOps)¶

    Role: Monitoring, telemetry, cost tracking, anomalies.

    GC Policy Compliance
    P1 Ensures no PI appears in telemetry.
    P2 Implements ITSG-33 monitoring controls.
    P3 Accessible dashboard-ready outputs.
    P4 Bilingual observability reporting.
    P5 Detects hallucination spikes, drift.
    P6 Supports human oversight of automation.
    P7 Central RA/IDE evidence store.
    P8 GC observability standards.
    P9 No automated mitigation without human approval.

    P15 — Compliance Agent¶

    Role: Ensures all outputs meet required GC policies.

    GC Policy Compliance
    P1 Verifies privacy rules on outputs.
    P2 Validates RBAC, audit, safety.
    P3 Assesses A11y compliance programmatically.
    P4 Confirms bilingual equivalence.
    P5 Enforces Responsible AI guidance.
    P6 Ensures human-in-the-loop checkpoints.
    P7 Logs compliance verification.
    P8 Encourages transparency & rule-alignment.
    P9 Cannot bypass or rewrite policies.

    2. POLICY COVERAGE SUMMARY (Executive Table)¶

    Policy Area Coverage by P01–P15 Notes
    Privacy Act 100% Redaction at ingestion + output review
    Security (ITSG-33) 100% AC/AU/SC/SI/CM implemented via EVA Core
    WCAG 2.2 AA 100% Sovereign UI + A11y agents
    Official Languages Act 100% Full i18n + bilingual agents
    TBS GenAI Guidance 100% Retrieval-bound, citations, oversight
    DADM 100% No scoring, no automation → low-impact tier
    RA/IDE 100% Full continuous monitoring via P14
    GC Digital Standards 100% Open standards, transparency, modularity
    Prohibited Autonomy 100% Level 4 technically impossible in EVA
