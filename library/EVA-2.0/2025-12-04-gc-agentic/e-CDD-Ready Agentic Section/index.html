    **E — CDD READY SECTION¶

    Agentic Architecture, Governance & Compliance for the Jurisprudence Integration into EVA Suite**

    (Place this under “Solution Design”, “AI Architecture”, or “Non-Functional Requirements” in the CDD.)

    1. Agentic Scope Overview¶

    The Jurisprudence solution integrated into EVA Suite leverages a bounded, GC-compliant agentic framework based on the EVA Agentic Layer (Scrum Master + P01–P15 patterns).

    Agentic behaviour is strictly limited to Level 2 and Level 3 of the Government of Canada Agentic Maturity Model:

    • Level 2 → Multi-step reasoning and tool use within guardrails
    • Level 3 → Multi-agent orchestration controlled by EVA Core

    No component of Jurisprudence is permitted to operate beyond Level 3. Level 4 autonomy is explicitly prohibited in compliance with GC policy.

    2. Agent Archetypes for Jurisprudence¶

    The Jurisprudence solution uses three agent archetypes defined in the GC Agentic Template:

    2.1 Knowledge Explorer Agent¶

    • Retrieves, summarizes, and explains jurisprudence from approved RAG vector stores
    • Bilingual output: English/French
    • Cited, evidence-based responses

    2.2 Drafting & Synthesis Agent¶

    • Produces structured summaries
    • Generates draft decision elements
    • Compares rulings across tribunals (SCC, FCA, FC, SST, etc.)

    2.3 Classification & Structuring Agent¶

    • Extracts metadata (tribunal, docket, date, parties, legal issue)
    • Categorizes case law using configurable schemas
    • Enforces deterministic mappings for ingestion

    No agent performs legal interpretation or binding decisions. All agents operate inside retrieval-bound limits and can only use curated, approved jurisprudence datasets.

    3. Approved Data Sources¶

    All agentic functions must use restricted, department-approved sources, including:

    • Supreme Court of Canada
    • Federal Court / Federal Court of Appeal
    • Social Security Tribunal
    • Tribunal-specific metadata from curated ingestion pipelines
    • Vector stores hosted in the EVA Foundation (Protected B)

    No external web crawling, live internet searching, or unapproved data access is permitted.

    4. EVA Agentic Framework (Level 2–3)¶

    The Jurisprudence integration leverages EVA’s existing agentic framework:

    4.1 Planning Agent (P02 Planner)¶

    • Converts queries into structured plans
    • Identifies required documents, steps, and tools
    • Enforces guardrails before execution

    4.2 Reviewer Agent (P03)¶

    • Audits outputs for correctness, citations, completeness
    • Ensures bilingual and accessibility compliance

    4.3 Retrieval + Evidence Agent (P01 Archivist)¶

    • Enforces retrieval-only evidence
    • Prevents hallucination by requiring citations

    4.4 Builder / Composer Agent (P05)¶

    • Assembles multi-document analyses
    • Builds structured jurisprudence briefs

    All agent execution is mediated through EVA Core, EVA LiveOps, and audit trails.

    5. Guardrails & Safety Engineering¶

    5.1 Policy Guardrails¶

    • Privacy Act (no personal data in RAG corpora)
    • Official Languages Act (bilingual inputs/outputs)
    • GC accessibility (WCAG 2.2 AA UI)
    • TBS Guide on Generative AI (retrieval-based guardrails)
    • DADM risk mitigations

    5.2 Technical Guardrails¶

    • Retrieval-bound answering
    • Vector store scoping (per project space)
    • RBAC controls (AICOE & Program)
    • Prompt injection protections
    • Rate limiting
    • Model refusal conditions
    • Circuit breaker (kill switch)

    5.3 Human-in-the-Loop¶

    All outputs must remain non-binding drafts. Human adjudicators remain fully responsible and accountable.

    6. Access Control & RBAC¶

    Roles¶

    • Administrator (AICOE)
    • Jurisprudence Program Owner
    • Content Contributor (curation team)
    • Read-only User (adjudicators, analysts)

    Controls¶

    • Project Space isolation
    • Per-source ingestion permissions
    • Telemetry per role
    • Fine-grained tool invocation permissions

    EVA Core enforces RBAC at the API, tool, and model layers.

    7. Logging, Audit & Monitoring (RA/IDE Compliance)¶

    Captured Logs¶

    • Prompt + response pairs (with redaction rules)
    • Tool calls
    • Source documents used in answers
    • Workflow plans (P02 artifacts)
    • Errors, exceptions, timeouts
    • Latency, cost, token usage
    • User identifiers (pseudo-anonymized if required)

    Monitoring¶

    • Daily telemetry
    • Retrieval drift detection
    • RAG quality monitoring
    • Tool misuse detection
    • Runtime guardrail violations

    Compliance¶

    Meets all RA/IDE mandatory controls for “continuous monitoring.”

    8. Accessibility, Bilingualism, and UI Standards¶

    Jurisprudence user interfaces will be built using:

    • EVA Sovereign UI (WCAG 2.2 AA+)
    • GC Design System components
    • Full i18n with centralized UI strings service
    • Dark mode and high contrast
    • Keyboard-only navigation
    • ARIA roles, focus trapping, error messaging

    Meets EARB UI expectations for GC intranet applications.

    9. Lifecycle Tier Alignment¶

    Lifecycle Tier Jurisprudence Status Requirements Met
    Sandbox Initial development No PIA, limited data
    Pilot Internal adjudicators, AICOE Safety tests, audit logs
    Production EVA Foundation release ATO, RA/IDE, telemetry

    All deployments follow AICOE Ops and DevSecOps pipelines.

    10. Privacy & Classification¶

    • All data is Unclassified / Protected B
    • No personal information is ingested or processed
    • Vector stores contain only publicly available jurisprudence
    • Privacy Notice is included in the UI
    • Review by departmental Privacy Office before Production

    11. Risk Identification & Mitigation¶

    Risks¶

    • Hallucination of legal concepts
    • Misclassification of case law
    • Outdated sources
    • Bias in language models
    • Over-reliance by users
    • Lack of bilingual parity

    Mitigations¶

    • Retrieval bound to approved corpora
    • Mandatory citations
    • Metadata versioning
    • RAG drift detection
    • Reviewer agent (P03) enforcement
    • Bilingual model prompting + translations
    • Training for adjudicators

    12. Compliance Mapping to GC Frameworks¶

    GC Framework Compliance Evidence
    TBS Guide on Generative AI ✔️ Retrieval-bound, traceable, non-binding outputs
    DADM ✔️ Low-impact category, human oversight
    ITSG-33 ✔️ AC, AU, SC, SI, CM controls via AICOE
    Privacy Act ✔️ No personal info, vetted sources
    OLA ✔️ Full bilingual support
    WCAG 2.2 ✔️ Sovereign UI
    RA/IDE ✔️ Continuous monitoring, logs
    EARB ✔️ GC Design System alignment

    13. Summary of Agentic Compliance for Jurisprudence¶

    • Bound to GC levels 2–3 only
    • Retrieved, cited, verifiable evidence
    • Human approval mandatory
    • Full audit logging
    • Strict RBAC + tool scoping
    • A11y, bilingualism, privacy built-in
    • Reusable EVA patterns for future ESDC use cases
    • Compatible with all EVA Project Spaces
    • Ready for ATO review once integrated

    14. Integration Recommendation for the CDD¶

    Recommended Integration Path (Option 3 modified): Ingest Jurisprudence → Use EVA Domain Assistant + EVA Agentic → Serve through Sovereign UI

    Using the EVA platform ensures:

    • Standardized ingestion
    • Consistent agentic safety
    • Unified audit + monitoring
    • Reuse of Sovereign UI and i18n
    • No duplication with ABRG or external codebases
    • Long-term supportability by AICOE
