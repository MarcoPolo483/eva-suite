    2. MARKDOWN VERSION WITH GC STYLING¶

    (Ready for GitHub, Confluence, or GC documentation sites)

    # Government of Canada  
## Agentic AI Controls Framework (vNext)

> **GC Reference Document**  
> Version: Draft 1  
> Applies to: EVA, Jurisprudence, AssistMe, KM, CPP-D, and departmental AI agentic systems  
> Classification: Unclassified / Protected B

---

# 1. Scope & Purpose
This framework defines the **security, governance, and operational controls** for deploying *agentic AI systems* within the Government of Canada (GC). It acts as an **overlay** to:

- ITSG-33  
- NIST AI RMF  
- ISED Voluntary Code  
- TBS Responsible AI guidance  
- GC Privacy & Data Protection frameworks  

---

# 2. GC Definition of Agentic AI
An **agentic AI system** is:

> *An AI capability able to perform multi-step tasks, invoke tools/APIs, execute workflows, or generate action plans with varying autonomy.*

Characteristics:
- tool use  
- workflow orchestration  
- multi-step planning  
- memory/state  
- autonomous reasoning  

---

# 3. GC Agentic Levels

### **Level 0 — Non-Agentic**
RAG/Q&A only.  
**No tool use.**

### **Level 1 — Bounded Tool Helper**
- Small, whitelisted toolset  
- No looping or open-ended planning  
- Full audit logs and APIM enforcement  
- **Risk: Low–Moderate**

### **Level 2 — Workflow Agent**
- Multi-step plans  
- Workflow execution with **human approval**  
- Policy guardrails + replayability  
- **Risk: Moderate–High**

### **Level 3 — Autonomous / Multi-Agent**
- Open-ended planning  
- Tool discovery  
- Not permitted in production  
- **Risk: High**

---

# 4. Agentic Risk Domains

- Prompt injection → tool misuse  
- Workflow drift / autonomy creep  
- Data leakage via tools/memory  
- Bias amplification  
- Cross-domain policy violations  
- Opaque planning logic  
- Supply chain and model integrity risks  

---

# 5. Required Controls

## **5.1 Level 1 Controls**
- Tool whitelist  
- RBAC per tool  
- APIM-enforced boundaries  
- Audit logs for all actions  
- Red-teaming  
- Cost/usage limits  

## **5.2 Level 2 Controls**
Includes Level 1 +  
- Human-in-the-loop approvals  
- Workflow replayability  
- Planning transparency  
- Memory/state isolation  
- Kill-switch (agent + tool)  
- Policy guardrails  
- Continuous monitoring  

## **5.3 Level 3 Controls**
- Sandbox only  
- Network isolation  
- No sensitive data  

---

# 6. Documentation Requirements

Every Level 1–2 agent requires:

- System Card  
- Data Card  
- Risk & Control Register  
- Architecture Diagram  
- Tool Inventory  
- Prompts & Guardrails  
- Red-Team/Testing Report  
- Audit & Monitoring Plan  
- IR Playbook  

---

# 7. GC Production Rules

| Environment | Allowed Levels |
|------------|----------------|
| Production | L0–L2 |
| Protected B | L0–L1 (L2 with conditions) |
| Sandbox | L0–L3 |

---

# 8. Reference Implementation – EVA

EVA provides:

- APIM-controlled tool execution  
- Multi-tenant isolation  
- Guardrail & policy engine  
- Logging & monitoring  
- WCAG 2.2 bilingual UI  
- DevSecOps integration  
- ATO-aligned architecture  

---

# 9. Conclusion
This framework provides:
- A unified GC agentic taxonomy  
- Risk and control overlays for ITSG-33  
- Operational patterns and transparency obligations  
- A reference platform in EVA  

This forms the basis of a secure, governed, auditable approach to Agentic AI across the Government of Canada.
