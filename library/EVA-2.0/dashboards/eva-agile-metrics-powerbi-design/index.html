    Marco needs a live dashboard to track EVA 2.0 progress without manually reading GitHub issues, sprint plans, or ceremony notes. This Power BI dashboard replaces the GitHub front page concept with automated metrics visualization.

    6. Lessons Learned: How many lessons captured? Applied?
    7. Tools ROI: Time saved by POD Tools Box?

    ğŸ“Š Data Sources¶

    1. GitHub Issues API (Primary Source)¶

    Endpoint: https://api.github.com/repos/MarcoPolo483/eva-orchestrator/issues

    Fields Needed:

    • number (issue ID)
    • title
    • state (open, closed)
    • assignee (owner)
    • labels (status: Ready, In Progress, QA Gate, Review, Done)
    • milestone (sprint)
    • created_at, updated_at, closed_at
    • comments (for blocker tracking)
    • Custom fields:
    • Story points (from issue body or label)
    • Blocker flag (label: blocker)
    • Bug flag (label: bug)

    Refresh Schedule: Every 1 hour (GitHub Actions cron job)

    2. GitHub Projects API (Board State)¶

    Endpoint: https://api.github.com/projects/columns/cards

    Fields Needed:

    • Column name (Ready, In Progress, QA Gate, Review, Done)
    • Card ID (links to issue)
    • Position in column (for WIP limit tracking)

    Refresh Schedule: Every 1 hour

    3. GitHub Actions Logs (CI/CD Metrics)¶

    Endpoint: https://api.github.com/repos/MarcoPolo483/eva-orchestrator/actions/runs

    Fields Needed:

    • workflow_name (e.g., sprint-planning.yml)
    • status (success, failure)
    • conclusion (passed, failed)
    • run_started_at, updated_at
    • jobs[].steps[] (for test coverage parsing)

    Metrics Derived:

    • Test coverage % (from test job logs)
    • Build success rate
    • Deployment frequency

    Refresh Schedule: Every 1 hour

    4. Sprint History Files (Ceremony Notes)¶

    Source: docs/sprint-history/<sprint>/ceremony-notes.md

    Fields Parsed:

    • Sprint name
    • Ceremony dates (planning, stand-ups, QA gate, review, retro)
    • Blockers mentioned
    • Action items
    • Velocity (story points completed)

    Ingestion Method:

    • GitHub Actions workflow reads markdown files
    • Parses structured sections
    • Uploads to Power BI dataset (Azure Analysis Services or Power BI REST API)

    Refresh Schedule: Daily (end of day)

    5. Lessons Learned Files (Knowledge Base)¶

    Source: docs/lessons-learned/LESSON-*.md

    Fields Parsed:

    • Lesson ID (LESSON-001, 002, etc.)
    • Title
    • Date created
    • Pod (Alpha, Beta, Gamma, Delta)
    • Role (Dev Master, QA Master, etc.)
    • Tags (performance, security, tooling, etc.)
    • Status (documented, applied, validated)

    Ingestion Method: Same as sprint history (GitHub Actions â†’ Power BI)

    Refresh Schedule: Daily

    6. Tools Usage Log (POD Tools Box)¶

    Source: Custom telemetry (added to each tool script)

    Example (tools/git-tools/quick-commit.ps1 with telemetry):

    # ... tool code ...

# Telemetry: Log usage to CSV
$logEntry = @{
    Timestamp = Get-Date -Format "o"
    Tool = "quick-commit"
    User = $env:USERNAME
    Duration = (Measure-Command { ... }).TotalSeconds
}
$logEntry | Export-Csv "tools/usage-log.csv" -Append -NoTypeInformation

    Fields Logged:

    • Timestamp
    • Tool name
    • User (agent name)
    • Duration (seconds)
    • Success/Failure

    Refresh Schedule: Real-time (appends to CSV after each tool run)
