    EVA 2.0 – Advanced Services¶

    DevTools & Rules as Code (RaC)¶

    Version: 0.5 Audience: Scrum Master, Agile Crew & Humans Date: 2025-11-23

    This document describes two key advanced service pillars in EVA 2.0:

    1. EVA DevTools – the agentic, AI-powered toolbox for development, modernization, and governance.
    2. EVA Rules as Code (RaC) Integration – the enterprise rules engine capability exposed via the EVA Rules Service.

    Both sit on top of the EVA 2.0 Core and Suite layers, and are meant to be reusable across all EVA Products (EVA Chat, EVA DA, AssistMe successor, Jurisprudence, KM assistants, etc.).

    1. Positioning in the EVA 2.0 Architecture¶

    1. EVA Core Platform (Private Backbone)
    2. APIM-exposed services for:
      • RAG (/rag/retrieve, /rag/answer)
      • Document processing (/doc/summarize, /doc/compare, /doc/extract)
      • Safety & policy guardrails (PII redaction, base prompts, filters)
    3. EVA Rules Service (RaC)

    4. Telemetry, logging, SIEM integration, CI/CD, IaC

    5. EVA Suite Libraries (Shared Libraries)

    6. eva-* libraries encapsulating EVA patterns for:
      • RAG (chunking, indexing, retrieval)
      • Safety & policy
      • Metering & telemetry
      • LiveOps & configuration
      • UI / i18n / accessibility
      • Agent tooling (DevTools, RaC helpers, etc.)

    2. EVA DevTools – Agentic Landscape & Feature Catalog¶

    Product name: EVA DevTools
    Motto: Developer’s Tools Box that can be created and maintained at AI speed.

    EVA DevTools is EVA’s internal delivery optimization layer for developers, BAs, architects, testers, and governance teams. It takes the three-tier agentic concept (Orchestrator, Super, Utility agents) and reimplements it inside EVA 2.0 for software delivery, modernization, and governance.

    2.1 Agent Taxonomy¶

    We use a three-tier agent design:

    1. Orchestrator Agents
    2. Primary entry points for humans.
    3. Understand user intent (“what is this person trying to do?”).
    4. Route to the right Super Agents.

    5. Manage multi-step flows, tool selection, and user interaction.

    6. Super Agents

    7. Own a domain or capability area (e.g., backlog quality, testing, legacy analysis).
    8. Orchestrate multiple Utility Agents and tools to accomplish complex tasks.

    9. Provide opinionated workflows and domain-specific logic.

    10. Utility Agents

    11. Small, reusable building blocks.
    12. Provide focused capabilities (parse code, call ADO, generate tests, summarize logs, etc.).
    13. Reused across Orchestrators and Super Agents.

    Runtime model

    All agents are implemented as EVA 2.0 “tools”:

    • APIs exposed through APIM.
    • RAG/LLM workflows where needed.
    • Callable by:
    • EVA DevTools web UI,
    • EVA Chat (e.g., /devtools … commands),
    • Future agent frameworks (MCP, Semantic Kernel, etc.).

    2.2 EVA DevTools Orchestrator¶

    Purpose
    Main entry point for developers and delivery teams.

    Primary users

    • Developers and Tech Leads
    • Testers and QA
    • Business Analysts
    • Scrum Masters and Release Train Engineers
    • Architects and Governance teams

    Example intents

    • “Clean up my ADO backlog for the next PI.”
    • “Generate test scripts for this feature branch.”
    • “Summarize mainframe module X and generate documentation.”
    • “Create a first-cut process map for this feature.”
    • “Give me a sprint health report for the last 3 sprints.”

    Responsibilities

    • Classify incoming requests into capability areas:
    • Backlog, Testing, Legacy, Governance, Planning, Skills, Observability.
    • Select the appropriate Super Agent(s).
    • Orchestrate multi-step flows (e.g., backlog → impact analysis → release planning).
    • Maintain conversational state so users can iterate (“refine this”, “show me only risks”, etc.).

    2.3.1 EVA Backlog Guardian¶

    Goal
    Improve backlog quality, consistency, and traceability.

    Key functions

    • Analyze ADO boards for:
    • INVEST-quality of user stories.
    • Missing or weak acceptance criteria.
    • Inconsistent naming, tags, or taxonomy.
    • Detect dependencies between features/stories:
    • Cross-team impacts.
    • Potential blockers and sequencing issues.
    • Suggest improvements:
    • Merge or split stories.
    • Add missing links and tags.
    • Standardize wording using agreed vocabularies.

    Inputs

    • ADO project / team / query (e.g., “current PI backlog”).
    • Optional domain-specific vocab/ontology (EI, BDM, EVA).

    Outputs

    • Backlog quality report (score and findings).
    • Suggested changes (per-item recommendations).
    • Optional machine-readable export (JSON/CSV) for bulk updates.

    2.3.2 EVA Process Mapper¶

    Goal
    Accelerate creation of product process maps, user journeys, and system flows.

    Key functions

    • Generate candidate process maps from:
    • Natural language descriptions.
    • Existing user stories and features.
    • Auto-structure flows:
    • Steps, decisions, outcomes, actors, systems.
    • Produce dual artifacts:
    • Diagram spec (Mermaid/BPMN JSON or similar).
    • Human-readable narrative for documentation.

    Inputs

    • Free-form text describing the process.
    • Optional links to ADO items or policy documents.

    Outputs

    • Machine-readable process map definition.
    • Text description (for Confluence, SharePoint, EVA DA, etc.).
    • Suggested swimlanes (actors, systems) and boundaries.

    2.3.3 EVA Test Architect¶

    Goal
    Improve test coverage and reduce redundant / low-value tests.

    Key functions

    • Analyze existing test suites:
    • Redundancy and overlaps.
    • Coverage gaps vs. code or requirements.
    • Generate new tests from:
    • Legacy or modern code.
    • User stories and acceptance criteria.
    • API contracts (OpenAPI/Swagger).
    • Propose optimized suite execution:
    • Prioritization by risk, frequency, and impact.

    Inputs

    • Test repository / configuration.
    • Target module or service.
    • Reference stories / features.

    Outputs

    • Test coverage summary & heatmap.
    • Suggested new test cases (ready-to-adapt code stubs).
    • Optimized regression suite definition.

    2.3.4 EVA Legacy Analyzer¶

    Goal
    Demystify legacy systems and help teams modernize them.

    Key functions

    • Parse legacy code and configuration:
    • Programs, data structures, key entry points.
    • Produce:
    • Plain-language system summaries.
    • Rule descriptions and decision paths.
    • Assist modernization:
    • Identify candidate user stories.
    • Suggest “quick-win” extractions into modern services.

    Inputs

    • Source code repositories or extracts.
    • Optional platform metadata (mainframe, Cúram, etc.).

    Outputs

    • High-level architecture overview.
    • Module-level documentation for critical components.
    • Candidate backlog items for modernization.

    2.3.5 EVA Impact Analyzer¶

    Goal
    Assess the impact of new or changed policies/requirements on systems and flows.

    Key functions

    • Compare:
    • Old vs. new text (policy, regulations, requirements).
    • Identify:
    • Affected rules, services, UIs, data structures, test cases.
    • Suggest:
    • Change tasks.
    • Impacted documentation and artefacts.

    Inputs

    • Old and new policy/requirement documents (or marked sections).
    • Optional mapping to existing systems or code.

    Outputs

    • Impact analysis report.
    • List of impacted components, flows, and tests.
    • Suggested change backlog.

    2.3.6 EVA Release Navigator¶

    3. EVA 2.0 – Rules as Code (RaC) Integration¶

    Short version:
    EVA should treat Rules as Code as another first-class backend service (alongside RAG, safety, and metering) and expose it via APIM so all EVA “faces” (EVA DA, EVA Chat, AssistMe, Jurisprudence, KM, etc.) can call the same rules engine for decisions, explanations, and simulations.

    3.1 What Rules as Code Gives EVA¶

    Rules as Code (RaC) is:

    Legislation and policy encoded as machine-readable logic, exposed as an API so any channel can consume the same rules.

    EVA’s role:

    • Host and expose RaC engines (e.g., OpenFisca-based, others) via a standard EVA Rules Service.
    • Ensure rules are:
    • Versioned, testable, and auditable.
    • Reused across channels and products.
    • Combine RaC with LLMs:
    • RaC for decisions.
    • RAG + LLM for explanations and guidance.

    4. EVA Rules Service – New Core Capability¶

    Define a dedicated EVA Rules Service as a backend capability alongside RAG:

    EVA Rules Service = RaC engine(s) exposed via APIM with consistent contracts, logging, and governance.

    4.1 Proposed API Surface¶

    Exposed via APIM, for example:

    • POST /rules/evaluate
    • Input:
      • program (e.g., EI, CERB, CPP-D)
      • ruleSetVersion (or “latest”)
      • variables (client situation: income, hours worked, family composition, etc.)

    • Output:

      • Eligibility decisions
      • Calculated amounts / entitlements
      • Reasons and trace metadata

    • POST /rules/trace

    • Input: same as /rules/evaluate

    • Output:

      • Decision tree / trace showing:
      • Which conditions evaluated to true/false
      • Which variables were explicit vs defaulted

    • POST /rules/scenario

    • Input:
      • Baseline scenario
      • List of “what-if” changes (proposed rule changes or client variations)

    • Output:

      • Comparative results (current vs proposed rules, or scenario A vs B vs C)

    • GET /rules/metadata

    • Output:
      • Available programs and rule sets
      • Versions, owners, and provenance (Act, regulation, policy instruments)

    4.2 Backend Engines & Project Spaces¶

    Behind the EVA Rules Service, EVA can:

    • Wrap OpenFisca-style engines for EI / CERB-style experiments.
    • Add additional engines later (CPP/OAS, GIS, Cúram-derived rules, etc.).
    • Register each rule set as a Project Space artifact with its own:
    • Versioning
    • Tests
    • Ownership and governance records

    This promotes EVA to the role of enterprise RaC gateway for ESDC.

    5. Neurosymbolic Pattern – EVA DA & EVA Chat¶

    5.1 EVA DA: RAG + RaC = Neurosymbolic Assistant¶

    Example user question:

    “Am I eligible for benefit X if I worked 600 hours and was laid off due to shortage of work?”

    EVA DA pipeline:

    1. RAG
      Retrieve relevant policy / guidance / jurisprudence documents that describe Benefit X:
    2. Eligibility criteria

    3. Definitions of “insurable hours”, “shortage of work”, etc.

    4. RaC (EVA Rules Service)
      Call /rules/evaluate with the scenario (e.g., 600 hours, separation = shortage_of_work).
      The result provides:

    5. The decision (benefit_x_eligible).
    6. The amount (benefit_x_amount).

    7. A trace of how the decision was reached.

    8. LLM composition
      The LLM composes the final answer using:

    9. The decision (from RaC).
    10. The explanation and references (from RAG).
    11. Optional trace snippet, translated into plain language.

    Result: EVA DA uses symbolic rules for correctness and probabilistic LLMs for explanation and context.

    5.2 EVA Chat – Policy Copilot Backed by RaC¶

    EVA Chat exposes RaC as a callable tool:

    • RaC Query tool
    • User: “What are the hours thresholds for EI maternity benefits?”

    • Chat:

      • Calls GET /rules/metadata and/or POST /rules/trace.
      • Uses RAG only to add nuance and operational examples.

    • For caseworkers:

    • They paste a client scenario.
    • EVA Chat normalizes into variables.
    • Calls /rules/evaluate.
    • Returns:
      • “According to EI rules, this scenario is eligible/ineligible because …”
      • Optional trace details.

    This keeps hard decisions in the RaC layer, with the LLM acting as a translator and helper.

    6. RaC in EVA 2.0 Governance & Tooling¶

    6.1 Project Spaces + RaC¶

    For each EVA Project Space, manage RaC alongside RAG:

    • Rule Sets per program / product.
    • Test Suites:
    • Unit-style tests for rules.
    • Policy-driven scenario tests.
    • Scenario Libraries:
    • “What-if” scenario sets for program review.
    • Policy-to-Service Trace:
    • Chain from Act → Regulation → Rules → Services → UI flows.

    This lines up with NIST-style requirements:

    • REQ-TRC – Trace from rule to outcome.
    • REQ-XAI – Ability to show why a decision was made.
    • REQ-GOV – Governance of rule versions, reviews, and approvals.

    6.2 EVA IDE Tools – Rules Workbench¶

    EVA IDE Tools can include RaC-specific capabilities:

    • Rules Workbench tab:
    • Visualize:
      • Variables, parameters, and rule topology from RaC engines.
      • Decision trees and calculation flows.
    • Enable:
      • Concept modeling workshops with live scenarios.
      • Management of gold scenario sets (for Test Bench and program review).

    Future enhancements:

    • Guidance for authoring rule fragments.
    • Automatic test generation from examples.
    • Impact analysis when rules change (which services, flows, and cohorts are affected).

    6.3 EVA Command Center – RaC Views¶

    Extend EVA Command Center with RaC widgets:

    • Platform-level view:
    • Which rule sets exist and where they are used.
    • Recent rule-change events (who changed what, when).
    • Space-level view:
    • Current rule version and planned next version.
    • Key scenario outcomes and fairness indicators (e.g., by region, age group).

    This makes RaC visible as operational infrastructure, not just as an experiment.

    7. DevTools + RaC – How They Complement Each Other¶

    EVA DevTools and RaC integration reinforce each other:

    • DevTools can:
    • Use RaC metadata to understand where rules are applied.

    • Provide:

      • “Rules impact analysis” (via EVA Impact Analyzer).
      • Automated test generation targeting RaC scenarios (via EVA Test Architect).
      • Governance reports summarizing rules usage and changes (via EVA Governance Reporter).

    • RaC benefits from DevTools by:

    • Having:
      • Better test coverage (DevTools-generated scenarios).
      • Clearer documentation and diagrams (Process Mapper + Legacy Analyzer).
      • Automated governance artifacts for rule changes.

    Together, they help EVA deliver:

    • Faster, higher-quality AI + rules solutions.
    • Stronger governance, explainability, and program review capabilities.
    • A reusable, enterprise-grade platform that can support many ESDC programs.
