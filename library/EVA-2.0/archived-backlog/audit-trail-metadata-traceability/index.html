    CDD: Audit Trail and Metadata Traceability¶

    Last updated: 2025-11-15
    Source: docs/EVA-2.0/ESDC/EVA Backlog/EVA Backlog/CDD Audit Trail and Metadata Traceabi.docx
    Status: Backlog
    Priority: HIGH

    Vision¶

    Implement comprehensive audit trail and metadata tracking across all EVA interactions to support:

    • Regulatory compliance (Protected B requirements)
    • Security incident investigation
    • User activity monitoring
    • Session replay and debugging
    • Cost attribution and chargeback

    Key Requirements¶

    Audit Events to Capture¶

    • User authentication and authorization events
    • Session start/end with duration
    • All user prompts and AI responses
    • Tool invocations and results
    • RAG document citations and retrieval scores
    • Safety filter triggers (moderation, injection detection)
    • API calls with latency and status codes
    • Error events with stack traces
    • Configuration changes

    Metadata Schema¶

    • Session: sessionId, userId, tenantId, startTime, endTime, language
    • Interaction: interactionId, timestamp, promptTokens, completionTokens, model, latency
    • Security: clientIP, userAgent, authMethod, permissionLevel
    • Cost: computeCost, storageCost, apiCost, totalCost
    • Provenance: documentIds cited, retrievalScores, groundednessScore

    Technical Approach¶

    Storage Strategy¶

    • Hot Path: Azure Cosmos DB (HPK: tenantId/userId/sessionId) for last 30 days
    • Warm Path: Azure Storage (Parquet) for 1-3 years
    • Cold Path: Archive tier for 3-7 years (regulatory retention)

    Pipeline Architecture¶

    EVA API/Agent
   ↓
Event Pipeline (Azure Event Hubs)
   ↓
Stream Analytics → Cosmos DB (real-time queries)
   ↓
Archive Function → Storage (Parquet for analytics)

    Query Patterns¶

    • Real-time session lookup (Cosmos DB point reads)
    • User activity timeline (Cosmos DB range queries)
    • Cross-user analytics (Synapse on Parquet files)
    • Cost showback reports (Power BI on Parquet)

    User Stories¶

    Story 1: Compliance Officer Audit Investigation¶

    As a compliance officer investigating a security incident
    I want to retrieve complete audit logs for a specific user/session with all interactions and metadata
    So that I can reconstruct what happened and provide evidence for regulatory reporting

    Acceptance: Can query audit logs by user ID, session ID, date range; retrieve all prompts, responses, citations, and security events; export to CSV for evidence submission

    Story 2: Platform Operator Cost Attribution¶

    As a platform operator managing tenant budgets
    I want accurate cost attribution for every interaction (compute, storage, API calls)
    So that I can charge back costs to tenant cost centers and track budget burn

    Acceptance: Every interaction has cost metadata (tokens × pricing); costs aggregated per tenant/user/session; Power BI dashboard shows monthly spend trends; cost accuracy >99%

    Story 3: Security Analyst Threat Detection¶

    As a security analyst monitoring for malicious activity
    I want real-time alerts on suspicious patterns (injection attempts, data exfiltration, unusual volumes)
    So that I can respond to threats before damage occurs

    Acceptance: Anomaly detection runs on event stream; alerts triggered within 1 minute; security team receives notifications via email/SMS; audit logs capture incident timeline

    Story 4: Data Privacy Officer GDPR Compliance¶

    As a data privacy officer handling GDPR delete requests
    I want to identify and redact all personal data for a given user across hot/warm/cold storage
    So that we comply with right-to-erasure regulations

    Acceptance: Can query all audit logs containing user PII; redaction process completes within 30 days; audit trail preserved (user deleted but interactions remain anonymized)

    Story 5: Developer Debugging Session Issues¶

    As a developer troubleshooting a reported bug
    I want to replay a user's session with full context (prompts, responses, tool calls, errors)
    So that I can reproduce the issue and identify root cause

    Acceptance: Session replay UI shows chronological timeline; includes all metadata (latency, model, tokens, citations); can export session as JSON for analysis

    Functional Requirements¶

    FR-1: Universal Event Emission¶

    ALL EVA components (eva-api, eva-agent, eva-rag, eva-safety) MUST emit audit events for every user interaction, tool invocation, and system event.

    FR-2: Event Schema Standardization¶

    All audit events MUST follow standardized JSON schema with required fields: eventId, timestamp, tenantId, userId, sessionId, eventType, payload.

    FR-3: Event Hubs Ingestion¶

    All audit events MUST be sent to Azure Event Hubs with guaranteed delivery (at-least-once semantics, retry with exponential backoff).

    FR-4: Real-Time Cosmos DB Streaming¶

    Event Hubs data MUST stream to Cosmos DB via Azure Stream Analytics within <10 seconds of emission (real-time queries).

    FR-5: Hierarchical Partition Key¶

    Cosmos DB MUST use 3-level HPK (tenantId/userId/sessionId) to support efficient session lookups and avoid hot partitions.

    FR-6: Security Metadata Capture¶

    Every audit event MUST capture security context: client IP, user agent, auth method (OAuth/API key), permission level, geo-location.

    FR-7: Cost Calculation¶

    Every interaction event MUST include cost metadata: prompt tokens × $X, completion tokens × $Y, API calls × $Z, total cost.

    FR-8: RAG Provenance Tracking¶

    RAG interactions MUST capture cited document IDs, retrieval scores, grounding scores, and chunk metadata for audit trail.

    FR-9: PII Masking¶

    Audit logs MUST mask sensitive PII patterns (SSN, credit cards, email) automatically before storage using regex-based detection.

    FR-10: Immutable Audit Trail¶

    Audit logs in Cosmos DB and Storage MUST be write-only (no updates/deletes allowed except via compliance workflows).

    FR-11: Warm Path Archival¶

    Cosmos DB data older than 30 days MUST be archived to Azure Storage in Parquet format (monthly batch jobs).

    FR-12: Cold Path Lifecycle¶

    Parquet files older than 1 year MUST move to Archive tier automatically (cost optimization, 7-year retention).

    FR-13: Query API¶

    EVA Ops MUST expose audit log query API: GET /v1/audit/sessions/{sessionId}, GET /v1/audit/users/{userId}, POST /v1/audit/query (KQL-style queries).

    FR-14: Export Functionality¶

    Users MUST be able to export audit logs in CSV/JSON format for compliance reporting (with proper RBAC: admins only).

    FR-15: GDPR Right-to-Erasure¶

    Platform MUST support user deletion workflow: redact PII from all audit logs (hot/warm/cold), preserve anonymized interaction data for compliance.

    Non-Functional Requirements (Expanded)¶

    Performance¶

    • Event Emission Overhead: <10ms added latency to API requests
    • Real-Time Query: P95 <100ms for session lookup (Cosmos DB point read)
    • Timeline Query: P95 <500ms for user activity timeline (30 days, Cosmos DB range query)
    • Throughput: Support 100,000 events/second ingestion (Event Hubs capacity)

    Reliability¶

    • Event Delivery: 100% delivery guarantee (Event Hubs → Cosmos DB, at-least-once with deduplication)
    • Data Durability: 99.999% durability (Cosmos DB replication + Storage redundancy)
    • Zero Data Loss: No audit events dropped under failure scenarios

    Security¶

    • Encryption: All audit data encrypted at rest (AES-256) and in transit (TLS 1.2+)
    • Access Control: RBAC enforced (only authorized users query audit logs)
    • PII Protection: Automated PII masking before storage (regex-based detection)
    • Immutability: Write-only storage (prevent tampering/deletion)

    Compliance¶

    • Retention: 30 days hot (Cosmos DB), 3 years warm (Storage), 7 years cold (Archive tier)
    • Protected B: Meets ESDC Protected B security requirements
    • GDPR: Right-to-erasure supported within 30 days
    • Auditability: Audit trail for audit log access (who queried what, when)

    Scalability¶

    • Horizontal Scaling: Event Hubs partitions scale to 100+ (support high-volume tenants)
    • Storage Growth: Parquet archival reduces Cosmos DB costs (10x cheaper for cold data)
    • Query Performance: Synapse Analytics for cross-session queries (billions of events)

    Observability¶

    • Pipeline Monitoring: Track Event Hubs lag, Stream Analytics failures, archival job status
    • Cost Tracking: Measure audit infrastructure costs (Cosmos DB, Storage, Event Hubs)
    • Alerting: Automated alerts for ingestion delays >1 minute, query errors, archival failures

    Acceptance Criteria¶

    AC-1: Universal Event Coverage¶

    ✅ All user interactions logged (chat, RAG, tool calls, errors)
    ✅ All system events logged (auth, config changes, safety triggers)
    ✅ Event schema validation passes (100% conformance)

    AC-2: Real-Time Ingestion¶

    ✅ Event Hubs → Cosmos DB lag <10 seconds (P95)
    ✅ Zero events dropped during ingestion
    ✅ Duplicate detection active (idempotency keys)

    AC-3: Query Performance Met¶

    ✅ Session lookup <100ms (P95, Cosmos DB)
    ✅ User timeline <500ms (P95, 30 days)
    ✅ Cross-session analytics functional (Synapse)

    AC-4: Cost Attribution Accurate¶

    ✅ Every interaction has cost metadata
    ✅ Cost accuracy >99% (validated against Azure billing)
    ✅ Power BI dashboard shows tenant spend

    AC-5: Security Hardened¶

    ✅ PII automatically masked before storage
    ✅ Audit logs encrypted at rest and in transit
    ✅ RBAC enforced (only admins query logs)
    ✅ Immutability enforced (no updates/deletes)

    AC-6: Compliance Ready¶

    ✅ 30-day hot, 3-year warm, 7-year cold retention active
    ✅ GDPR deletion workflow functional (<30 days)
    ✅ Audit trail for audit log access captured

    AC-7: Archival Functional¶

    ✅ Monthly Parquet archival jobs complete successfully
    ✅ Archive tier lifecycle policy active (1-year threshold)
    ✅ Synapse queries work on archived data

    AC-8: Observability Complete¶

    ✅ Dashboard shows ingestion lag, throughput, errors
    ✅ Alerts configured for failures and delays
    ✅ Cost monitoring dashboard operational

    Technical Design Notes¶

    Event Schema (JSON)¶

    {
  "eventId": "uuid-v4",
  "timestamp": "2025-11-19T14:32:10.123Z",
  "tenantId": "tenant-001",
  "userId": "user-123",
  "sessionId": "session-abc",
  "eventType": "interaction.completion",
  "payload": {
    "prompt": "What are my benefits?",
    "response": "You are eligible for...",
    "model": "gpt-4o",
    "promptTokens": 45,
    "completionTokens": 120,
    "latency": 1250,
    "cost": 0.00234,
    "citations": ["doc-123", "doc-456"],
    "groundednessScore": 0.92
  },
  "security": {
    "clientIP": "10.20.30.40",
    "userAgent": "Mozilla/5.0...",
    "authMethod": "OAuth2",
    "permissionLevel": "user"
  }
}

    Cosmos DB Schema¶

    Container: audit-events
    Partition Key: /tenantId/userId/sessionId (3-level HPK)
    TTL: 30 days (auto-delete after archival)
    Indexing: Timestamp (range), eventType (hash)

    Archival Function (Pseudo-code)¶

    // Monthly job: Cosmos DB → Parquet → Storage
async function archiveAuditLogs() {
  const cutoffDate = DateTime.now().minus({ days: 30 });
  const events = await cosmos.query(`
    SELECT * FROM c WHERE c.timestamp < '${cutoffDate}'
  `);

  const parquet = convertToParquet(events);
  await storage.upload(`audit-logs/2025-11-01.parquet`, parquet);
  await cosmos.bulkDelete(events); // Free up space
}

    Anomaly Detection (Stream Analytics Query)¶

    -- Detect injection attempts (high safety filter trigger rate)
SELECT tenantId, userId, COUNT(*) as triggerCount
INTO [anomaly-alerts]
FROM [audit-events]
WHERE eventType = 'safety.filter.trigger'
GROUP BY tenantId, userId, TumblingWindow(minute, 5)
HAVING COUNT(*) > 10

    Authentication (eva-auth)¶

    • Requirement: User/tenant context extraction from JWT tokens
    • Timing: Sprint 2 (security metadata)
    • Owner: ai2

    Operations (eva-ops)¶

    • Requirement: Audit log query API and export functionality
    • Timing: Sprint 4 (query interface)
    • Owner: windowsGuy

    All EVA Services¶

    • Requirement: Event emission instrumentation (eva-api, eva-agent, eva-rag, eva-safety)
    • Timing: Incremental across Sprint 2-5
    • Owner: Dev Masters (per service)
    • Risk: Consistent event schema adoption across teams (need code gen or SDK)

    Open Questions (Updated)¶

    Q1: PII Masking Strategy (PRIVACY DECISION)¶

    Question: Mask PII automatically (regex) or require explicit opt-in per tenant?
    Impact: False positives (over-masking) vs. compliance risk (under-masking)
    Recommendation: Automatic masking with configurable patterns per tenant (default: SSN, credit cards, emails)
    Decision Owner: Security Sentinel + Marco

    Q2: GDPR Right-to-Erasure Scope (LEGAL DECISION)¶

    Question: Delete all user data (anonymize interactions) or retain anonymized aggregates?
    Impact: Compliance vs. analytics value
    Recommendation: Anonymize (replace userId with "deleted-user-{hash}"), retain interaction data for platform analytics
    Decision Owner: Legal team + Marco

    Q3: Retention Policy Alignment (COMPLIANCE)¶

    Question: Confirm 7-year retention aligns with ESDC Protected B guidelines?
    Impact: Storage costs vs. regulatory requirements
    Recommendation: Validate with ESDC compliance team before Q1 2026 demo
    Decision Owner: Marco + Compliance Officer

    Q4: Cross-Tenant Analytics Privacy (POLICY)¶

    Question: Allow aggregated metrics across tenants (e.g., "average session cost") or strict isolation?
    Impact: Platform insights vs. tenant privacy concerns
    Recommendation: Aggregated metrics OK if anonymized (no tenant identification possible)
    Decision Owner: Marco + Privacy Officer

    Q5: Session Replay UI Scope (FEATURE SCOPE)¶

    Question: Build full session replay UI (Sprint 5) or defer to v2?
    Impact: Developer productivity vs. MVP timeline
    Recommendation: Defer to v2; provide JSON export API for Sprint 1 (devs can use Postman/curl)
    Decision Owner: Marco

    Q6: Anomaly Detection Algorithms (TECHNICAL)¶

    Question: Use simple threshold-based rules (Stream Analytics) or ML-based models (Azure ML)?
    Impact: Accuracy vs. complexity/cost
    Recommendation: Start with threshold rules (Sprint 3), add ML models if false positive rate >5%
    Decision Owner: ai2 (Security Sentinel)

    Success Criteria¶

    Phase 1: Core Audit Events¶

    • Define audit event schema (JSON/Avro)
    • Implement event emission in eva-api
    • Wire Event Hubs ingestion pipeline
    • Stream to Cosmos DB with HPK

    Phase 2: Metadata Enrichment¶

    • Add security context (IP, auth method)
    • Calculate interaction costs (tokens × pricing)
    • Capture RAG provenance (cited docs)
    • Track safety filter triggers

    Phase 3: Analytics & Compliance¶

    • Archive to Parquet (monthly batches)
    • Build Power BI cost showback dashboard
    • Implement retention policies (30/365/2555 days)
    • Create audit log export API (CSV/JSON)

    Phase 4: Advanced Features¶

    • Session replay UI (reconstruct conversations)
    • Anomaly detection (unusual activity patterns)
    • Real-time cost alerts (budget thresholds)
    • Compliance report generation (audit logs for regulators)

    Success Criteria¶

    • All user interactions logged with <10ms overhead
    • 30-day hot data queryable in <100ms (P95)
    • 100% event delivery (Event Hubs → Cosmos DB)
    • Audit log retention meets Protected B requirements
    • Cost attribution accuracy >99%
    • GDPR/privacy compliance (PII handling, right to erasure)

    Dependencies¶

    • eva-metering: Cost calculation and pricing tables
    • eva-auth: User identity and tenant context
    • eva-api: Event emission points
    • eva-ops: Audit log query and export APIs
    • Azure Event Hubs: Event ingestion pipeline
    • Azure Cosmos DB: Hot path storage (HPK)
    • Azure Storage: Warm/cold path archival
    • Azure Synapse: Cross-session analytics
