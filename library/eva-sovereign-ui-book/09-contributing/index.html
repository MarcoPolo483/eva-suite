    Chapter 09: Contributing¶

    EVA Sovereign UI is an open-source project built for the Government of Canada enterprise ecosystem. We welcome contributions from developers, designers, accessibility experts, and translators who want to improve the component library.

    9.1 Getting Started¶

    Prerequisites¶

    Before contributing, ensure your development environment has:

    • Node.js 18 or 20 (LTS versions) - Download from nodejs.org
    • npm 9+ (included with Node.js)
    • Git 2.40+ - Download from git-scm.com
    • VS Code (recommended) - Download from code.visualstudio.com

    Recommended VS Code extensions:

    • Lit Plugin - Syntax highlighting and IntelliSense for Lit templates
    • ESLint - Real-time linting feedback
    • Prettier - Code formatting on save
    • axe Accessibility Linter - Accessibility checking in editor

    First-Time Setup¶

    # Fork the repository on GitHub, then clone your fork
git clone https://github.com/YOUR_USERNAME/eva-sovereign-ui.git
cd eva-sovereign-ui

# Install dependencies
npm install

# Build components
npm run build

# Start Storybook development server
npm run storybook

    Storybook will open at http://localhost:6006 with live component previews.

    9.2 Development Workflow¶

    Branching Strategy¶

    1. Create feature branches from main: 

      git checkout main
git pull upstream main
git checkout -b feature/eva-gc-tooltip

    2. Name branches descriptively:

    3. feature/eva-gc-tooltip - New component
    4. fix/button-focus-ring - Bug fix
    5. docs/theming-guide - Documentation update

    6. chore/upgrade-lit-3.2 - Dependency update

    7. Commit frequently with clear messages: 

      git commit -m "feat(tooltip): Add eva-gc-tooltip with keyboard nav"
git commit -m "fix(button): Restore focus ring on keyboard nav"
git commit -m "docs: Add theming chapter to UI book"

    8. Push to your fork and open a pull request: 

      git push origin feature/eva-gc-tooltip

    9.3 Component Development Standards¶

    File Structure¶

    Every component requires these files:

    src/components/eva-gc-button/
├── eva-gc-button.ts          # Component implementation
├── eva-gc-button.styles.ts   # Scoped CSS
├── eva-gc-button.test.ts     # Unit tests (Vitest)
├── eva-gc-button.stories.ts  # Storybook stories
└── README.md                  # Component documentation

    Bilingual Template¶

    All components MUST support English and French text via properties:

    @property({ type: String }) label: string = 'Submit';
@property({ type: String }) labelFr: string = 'Soumettre';

render() {
  const lang = document.documentElement.lang || 'en';
  const text = lang === 'fr' ? this.labelFr : this.label;
  return html`<button>${text}</button>`;
}

    Alternatively, use slots for flexible content:

    render() {
  return html`
    <button>
      <slot name="label-en" ?hidden="${lang === 'fr'}"></slot>
      <slot name="label-fr" ?hidden="${lang === 'en'}"></slot>
    </button>
  `;
}

    Accessibility Requirements¶

    • Keyboard navigation - All interactive elements must be keyboard accessible
    • ARIA attributes - Use semantic HTML first, ARIA when needed
    • Focus management - Visible focus indicators, logical tab order
    • Screen reader support - Descriptive labels, roles, states
    • Color contrast - WCAG 2.1 AA minimum (4.5:1 for text, 3:1 for UI)

    9.4 Testing Requirements¶

    All components MUST pass three testing gates before merge:

    Unit Tests (Vitest)¶

    Target: 80%+ code coverage

    import { expect, test } from 'vitest';
import { fixture, html } from '@open-wc/testing';
import './eva-gc-button';

test('renders with English label by default', async () => {
  const el = await fixture(html`<eva-gc-button label="Submit"></eva-gc-button>`);
  expect(el.shadowRoot.querySelector('button').textContent).toBe('Submit');
});

test('renders French label when lang=fr', async () => {
  document.documentElement.lang = 'fr';
  const el = await fixture(html`<eva-gc-button label="Submit" label-fr="Soumettre"></eva-gc-button>`);
  expect(el.shadowRoot.querySelector('button').textContent).toBe('Soumettre');
});

    Run tests: 

    npm test                  # Run all tests
npm run test:coverage     # Generate coverage report

    Visual Regression (Percy)¶

    Percy captures screenshots of Storybook stories and compares them to baseline images. Visual changes require manual approval.

    npm run percy             # Run visual regression tests

    Accessibility Tests (axe-core)¶

    All components MUST pass axe-core WCAG 2.1 AA checks:

    import { expect, test } from 'vitest';
import { fixture, html } from '@open-wc/testing';
import { axe, toHaveNoViolations } from 'jest-axe';

expect.extend(toHaveNoViolations);

test('has no accessibility violations', async () => {
  const el = await fixture(html`<eva-gc-button label="Submit"></eva-gc-button>`);
  const results = await axe(el);
  expect(results).toHaveNoViolations();
});

    9.5 Code Style¶

    ESLint¶

    Lint TypeScript code: 

    npm run lint              # Check for issues
npm run lint:fix          # Auto-fix issues

    ESLint enforces: - No unused variables - Consistent spacing and indentation - Proper TypeScript types - Import order

    Prettier¶

    Format code: 

    npm run format            # Format all files

    Prettier enforces: - 2-space indentation - Single quotes - 100-character line length - Trailing commas

    Pre-Commit Hooks¶

    Husky runs linting and formatting automatically before commits:

    git commit -m "feat: Add tooltip component"
# → Runs lint-staged
# → Lints and formats changed files
# → Aborts commit if errors found

    9.6 Pull Request Process¶

    PR Checklist¶

    Before opening a PR, ensure:

    • All tests pass (npm test)
    • Code coverage ≥80% (npm run test:coverage)
    • No linting errors (npm run lint)
    • Code is formatted (npm run format)
    • Storybook stories added (npm run storybook)
    • Component README updated
    • Accessibility tests pass (axe-core)
    • Bilingual support verified (EN + FR)

    PR Template¶

    Use this template when opening a pull request:

    ## Summary
Brief description of changes.

## Type of Change
- [ ] New component
- [ ] Bug fix
- [ ] Feature enhancement
- [ ] Documentation
- [ ] Refactoring

## Testing
How was this tested? Include steps to verify.

## Screenshots
Before/after screenshots for UI changes.

## Accessibility
- [ ] Keyboard navigation tested
- [ ] Screen reader tested (NVDA/JAWS)
- [ ] axe-core violations: 0

## Checklist
- [ ] Tests added/updated
- [ ] Storybook stories added
- [ ] README updated
- [ ] Bilingual (EN/FR) support

    Review Process¶

    1. Automated checks run on every PR (tests, lint, Percy)
    2. Maintainer review - At least one maintainer approval required
    3. Accessibility review - For new components or UX changes
    4. Squash-merge - All commits squashed into one when merged

    9.7 Documentation¶

    Component README¶

    Every component needs a README with:

    • Overview - What the component does
    • Usage - Code examples
    • Properties - All @property attributes with types and defaults
    • Events - All @event dispatches with payload types
    • Slots - All named slots with descriptions
    • CSS Custom Properties - All --eva-* theming variables
    • Accessibility - Keyboard shortcuts, ARIA attributes, screen reader behavior

    Storybook Stories¶

    Every component needs Storybook stories demonstrating:

    • Default - Basic usage
    • Variants - All visual variants (primary, secondary, danger, etc.)
    • States - Disabled, loading, error, success
    • Bilingual - English and French text
    • Accessibility - Keyboard navigation, screen reader labels
