    DUA Format Specification v1.0¶

    Document Type: Technical Standard Version: 1.0.0 Status: OFFICIAL Effective Date: December 8, 2025 Authority: EVA Suite Project Standards Scope: All EVA repositories and components

    1. Purpose¶

    The DUA (Detailed Update Archive) format provides a standardized structure for documenting significant project events, milestones, and changes across the EVA Suite ecosystem. It ensures consistent, traceable, and machine-readable records for audit, compliance, and project management purposes.

    2. When to Use DUA Format¶

    2.1 Required DUA Events¶

    Create a DUA document for these mandatory events:

    • ‚úÖ Sprint completions - Every sprint closure (Agile sprints)
    • ‚úÖ Production deployments - Any release to production environment
    • ‚úÖ Data migrations - Schema changes, data movement between systems
    • ‚úÖ Security incidents - Security breaches, vulnerabilities resolved
    • ‚úÖ Breaking changes - API changes, backward-incompatible updates
    • ‚úÖ Service migrations - Moving services between infrastructure/platforms

    2.2 Recommended DUA Events¶

    Create a DUA document for these important events:

    • ‚≠ê Major feature releases - Significant new functionality (v1.0, v2.0)
    • ‚≠ê Architecture changes - Major refactoring, system redesigns
    • ‚≠ê Performance milestones - Achieving significant performance targets
    • ‚≠ê Technical debt paydown - Completing debt reduction initiatives
    • ‚≠ê Integration completions - New service integrations
    • ‚≠ê Compliance certifications - Security audits, compliance approvals

    2.3 Optional DUA Events¶

    Consider creating a DUA for:

    • üí° Proof of concept completions - Successful POC validations
    • üí° Research findings - Important technical investigations
    • üí° Capacity planning milestones - Scaling achievements
    • üí° Disaster recovery tests - DR drill completions

    2.4 Do NOT Use DUA For¶

    • ‚ùå Minor bug fixes - Individual bug fixes (use commit messages)
    • ‚ùå Daily commits - Routine code commits
    • ‚ùå Work-in-progress updates - Incomplete work
    • ‚ùå Draft documentation - Documentation in progress
    • ‚ùå Feature flags toggles - Simple configuration changes
    • ‚ùå Dependency updates - Routine package updates (unless critical security fix)
    • ‚ùå Code formatting changes - Style/linting updates

    2.5 Triggering Criteria¶

    Create a DUA when any of these conditions are met:

    Criterion Threshold Example
    Impact Affects >100 users or >3 services Multi-tenant feature launch
    Risk HIGH or CRITICAL severity Security patch deployment
    Effort >40 story points or >2 weeks Major refactoring initiative
    Compliance Regulatory requirement GDPR compliance implementation
    Rollback Complexity Cannot roll back easily Database schema migration
    Team Size >3 team members involved Cross-team integration project

    2.6 When in Doubt¶

    Ask yourself: 1. Will stakeholders need a formal record of this event in 6 months? 2. Does this event require approval or sign-off? 3. Could this event impact compliance or audits? 4. Would this event be discussed in a retrospective? 5. Does this event establish a new baseline or milestone?

    If YES to 2 or more ‚Üí Create a DUA If NO to all ‚Üí Skip DUA, use commit/PR description

    3. DUA Document Structure¶

    3.1 Header (REQUIRED)¶

    # DUA (Detailed Update Archive) - [Event Title]

**DUA-ID**: [COMPONENT]-[EVENT_TYPE]-[YYYYMMDD]
**Timestamp**: [ISO 8601 timestamp with timezone]
**Event Type**: [EVENT_TYPE]
**Severity**: [INFO|WARNING|CRITICAL]
**Component**: [component-name]
**Version**: [semantic version]

    3.2 Event Summary (REQUIRED)¶

    Brief description of the event, status, duration, and key dates.

    ## Event Summary

**Event**: [Description]
**Status**: [COMPLETE|IN_PROGRESS|BLOCKED|CANCELLED]
**Duration**: [Start date - End date]
**Completion Date**: [Date]
**Archive Date**: [Date]

    3.3 Quantitative Metrics (REQUIRED)¶

    Measurable outcomes in tabular format.

    ## Quantitative Metrics

| Metric | Value | Unit |
|--------|-------|------|
| [Metric Name] | [Value] | [Unit] |

    3.4 Deliverables Inventory (REQUIRED for completions)¶

    Detailed breakdown of artifacts with completion timestamps.

    ## Deliverables Inventory

### [Deliverable Name] ([Story Points])
- **Status**: [Status]
- **Files**: [Count]
- **LOC**: [Lines of Code]
- **Artifacts**:
  - `path/to/file.ext` ([lines] lines)
- **Completion**: [ISO 8601 timestamp]

    3.5 Risk Assessment (RECOMMENDED)¶

    ## Risk Assessment

### Risks Mitigated
| Risk | Impact | Mitigation | Status |
|------|--------|------------|--------|

### New Risks Identified
| Risk | Impact | Probability | Mitigation Plan |
|------|--------|-------------|-----------------|

    3.6 Quality Metrics (RECOMMENDED)¶

    ## Quality Metrics

### Code Quality
- **Test Coverage**: [percentage]%
- **Code Style**: [description]
- **Documentation**: [description]

    3.7 Dependencies & Integration (OPTIONAL)¶

    ## Dependencies & Integration

### External Dependencies Added
- [dependency] - [purpose]

### Services Integrated
- [service] - [purpose]

    3.8 Approval & Sign-off (REQUIRED)¶

    ## Approval & Sign-off

**Prepared By**: [Author/Team]
**Reviewed By**: [Reviewer]
**Approved By**: [Approver]
**Status**: [DRAFT|PENDING|APPROVED|REJECTED]
**Archive Date**: [ISO 8601 timestamp]

    3.9 Metadata (REQUIRED)¶

    Machine-readable JSON metadata block.

    ## Metadata

\`\`\`json
{
  "dua_id": "COMPONENT-EVENTTYPE-YYYYMMDD",
  "timestamp": "YYYY-MM-DDTHH:MM:SSZ",
  "event_type": "EVENT_TYPE",
  "component": "component-name",
  "version": "x.y.z",
  "status": "STATUS",
  "metrics": {
    "key": "value"
  },
  "tags": ["tag1", "tag2"]

    4. DUA-ID Format¶

    [COMPONENT]-[EVENT_TYPE]-[YYYYMMDD]

    Components¶

    • EVA-RAG - RAG engine
    • EVA-ORCHESTRATOR - Orchestration service
    • EVA-AUTH - Authentication service
    • EVA-UX - User experience layer
    • EVA-SUITE - Cross-component events

    Event Types¶

    • SPRINT - Sprint completion
    • RELEASE - Version release
    • DEPLOY - Production deployment
    • MIGRATION - Data/service migration
    • INCIDENT - Security/operational incident
    • MILESTONE - Major milestone achievement
    • REFACTOR - Architecture refactoring
    • DEPRECATION - Feature deprecation

    Examples¶

    • EVA-RAG-SPRINT2-20251208
    • EVA-ORCHESTRATOR-RELEASE-20251215
    • EVA-AUTH-MIGRATION-20251220
    • EVA-SUITE-MILESTONE-20251231

    5. Severity Levels¶

    Severity Description Examples
    INFO Routine updates, completions Sprint completions, feature releases
    WARNING Significant changes requiring attention Breaking changes, deprecations
    CRITICAL High-impact events requiring immediate action Security incidents, data loss, service outages

    6. Status Values¶

    Status Description Usage
    COMPLETE Successfully finished Sprint completions, migrations done
    IN_PROGRESS Currently active Ongoing sprints, partial deployments
    BLOCKED Waiting on external factors Dependency issues, approvals pending
    CANCELLED Abandoned/cancelled Deprecated features, cancelled work
    DRAFT Document in progress Preliminary documentation
    PENDING Awaiting approval Review pending
    APPROVED Officially approved Final archive state
    REJECTED Not approved Did not meet criteria

    7. File Naming Convention¶

    DUA-[EVENT_TYPE]-[YYYYMMDD].md

    Examples¶

    • DUA-SPRINT2-20251208.md
    • DUA-RELEASE-20251215.md
    • DUA-MIGRATION-20251220.md

    8. File Location¶

    Store DUA documents in component-specific locations:

    [repository]/docs/dua/YYYY/DUA-[EVENT_TYPE]-[YYYYMMDD].md

    Examples¶

    eva-rag/docs/dua/2025/DUA-SPRINT2-20251208.md
eva-orchestrator/docs/dua/2025/DUA-RELEASE-20251215.md

    Alternative (for feature-specific DUAs): 

    [repository]/docs/features/[feature-name]/DUA-[EVENT_TYPE]-[YYYYMMDD].md

    9. Automation Integration¶

    9.1 Parsing DUA Metadata¶

    DUA metadata blocks are JSON-formatted for machine parsing:

    import json
import re

def parse_dua_metadata(dua_content: str) -> dict:
    """Extract metadata from DUA document."""
    match = re.search(r'```json\n({.*?})\n```', dua_content, re.DOTALL)
    if match:
        return json.loads(match.group(1))
    return {}

    9.2 DUA Registry¶

    Maintain a central registry of all DUAs:

    {
  "registry_version": "1.0",
  "last_updated": "2025-12-08T23:45:00Z",
  "duas": [
    {
      "dua_id": "EVA-RAG-SPRINT2-20251208",
      "path": "eva-rag/docs/features/eva-data-model-faster/DUA-SPRINT2-20251208.md",
      "timestamp": "2025-12-08T23:45:00Z",
      "event_type": "SPRINT",
      "status": "APPROVED"
    }
  ]
}

    9.3 CI/CD Integration¶

    Validate DUA format in CI/CD pipelines:

    # .github/workflows/validate-dua.yml
name: Validate DUA Format
on:
  pull_request:
    paths:
      - 'docs/dua/**/*.md'
      - 'docs/features/**/DUA-*.md'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Validate DUA Format
        run: python scripts/validate_dua.py ${{ github.event.pull_request.changed_files }}

    11. Best Practices¶

    11.1 Writing DUAs¶

    • ‚úÖ Use clear, concise language
    • ‚úÖ Include quantitative metrics whenever possible
    • ‚úÖ Add timestamps in ISO 8601 format (UTC)
    • ‚úÖ Link to related artifacts (PRs, issues, docs)
    • ‚úÖ Include lessons learned for major events
    • ‚úÖ Ensure JSON metadata is valid

    11.2 Review Process¶

    1. Author creates DUA document
    2. Technical lead reviews for accuracy
    3. Product owner approves metrics
    4. Update status to APPROVED
    5. Archive in appropriate location

    11.3 Retention Policy¶

    • Keep DUAs indefinitely (they're lightweight)
    • Maintain registry for quick lookup
    • Annual review of DUA format updates
