    EVA 2.0 – MASTER CONTEXT¶

    One file to onboard AI collaborators (ChatGPT, Copilot, others) into the EVA 2.0 ecosystem.

    Author: Marco + AI advisors
    Status: Living document (keep updated)
    Target audience: ChatGPT, GitHub Copilot, and human collaborators

    1. EVA 2.0 – High-Level Vision¶

    EVA is not just a chatbot. It is:

    • An Enterprise Virtual Assistant platform for organizations ready to transform with AI.
    • A multi-tenant ecosystem where many AI-powered solutions can plug into a shared backbone.
    • A business and operating model that supports:
    • Sandboxes for experimentation
    • Production spaces for real workloads
    • Cost recovery and governance

    At a high level, EVA 2.0 has three layers:

    1. EVA Core Platform – Private, Azure-hosted backbone: RAG, safety, telemetry, deployment.
    2. EVA Matrix Libraries – Public/open libraries that encode EVA’s patterns and best practices.
    3. EVA Products – Concrete apps (EVA Chat, EVA Domain Assistant, Jurisprudence, etc.) that sit on top.

    This document is the master context to help AI tools understand:

    • Architecture & repo structure
    • Build sequence & milestones
    • Agile operating model
    • Quality bar (Definition of Done)
    • Governance, roles, and responsibilities
    • My (Marco’s) working style and preferences

    2. Architecture – The 3-Layer Model¶

    2.1 Layer 1 – EVA Core Platform (Private Backbone)¶

    Purpose:
    Provide a secure, governed backbone for all enterprise AI workloads.

    Key capabilities:

    • RAG services exposed via APIs:
    • /rag/retrieve
    • /rag/answer
    • Document services:
    • /doc/summarize
    • /doc/compare
    • /doc/extract
    • Safety & policy guardrails:
    • PII redaction
    • Base prompts & policy enforcement
    • Logging and auditing hooks
    • Ops & infra:
    • Azure-based deployment (Protected B)
    • APIM for traffic & cost attribution
    • Telemetry, logging, SIEM integration

    Non-goal:
    This layer is faceless – no user-facing UI here. It’s the “brain and nerves”, not the face.

    2.2 Layer 2 – EVA Matrix Libraries (Public/Open)¶

    Purpose:
    Turn EVA patterns into reusable, modular libraries that can be used by:

    • EVA Products
    • Other enterprise projects
    • The broader ecosystem (open source)

    Examples of eva-* repos (conceptual):

    • eva-core – Core domain models, shared utilities
    • eva-openai – LLM clients & model routing
    • eva-rag – Chunking, indexing, retrieval, ranking
    • eva-api – Common API contracts and handlers
    • eva-safety – Guardrails, filters, base prompts
    • eva-metering – Cost attribution, usage telemetry
    • eva-ops – LiveOps backend (config, flags, status)
    • eva-ui – Shared React components (WCAG + i18n ready)
    • eva-i18n – Internationalization helpers and resources
    • eva-i11y – Accessibility and personalization helpers
    • eva-seed – Seed data / demo configs
    • eva-enterprise – Connectors and integrations

    The Command Matrix Dashboard repo is the public portfolio view of this layer.

    2.3 Layer 3 – EVA Products (Faces of EVA)¶

    Examples:

    • EVA Chat – ChatGPT-style assistant for employees.
    • EVA Domain Assistant (EVA DA) – RAG UI for specific domains/projects (e.g., Jurisprudence).
    • AssistMe successor – Case-processing assist tools powered by EVA.
    • AI Knowledge Management solutions – KM assistants, content search tools, etc.

    These products:

    • Call EVA Core services through APIM.
    • Use EVA Matrix libraries to avoid re-inventing RAG, safety, telemetry, etc.
    • Are multi-tenant, each tenant being a Project Space.

    3. EVA Project Spaces, Sandboxes & Cost Model¶

    3.1 Project Spaces¶

    A Project Space is a logical “box” for a client solution:

    • Ex: “Jurisprudence Research Assistant”, “BDM KM Assistant”, “Canada Life Pilot”, etc.
    • Each box is isolated: its own RAG configuration, indexes, workflows, and UI.

    Each Project Space has at least two modes:

    1. Accelerator Sandbox
    2. For experimentation, proof-of-concept, and UAT.
    3. Funded centrally (e.g., IITB) during evaluation.
    4. Production Space
    5. For live workloads, with cost recovery via client cost center.

    3.2 Roles Around a Project Space¶

    • EVA Project Space Administrator (EVA PS-A)
    • Owns the box creation, promotion, cost bindings, and termination.
    • Can:
      • Spin up Accelerator Sandboxes (standard templates).
      • Promote sandbox → production (“turn key to Production”).
      • Export/import configs, literals, i18n/a11y packs, and logs.
    • EVA Production Space Business Analyst (EVA PS-BA)
    • Designs RAG workflows, prompts, UI texts, ingestion pipelines.
    • Works very closely with the client SMEs.
    • Client Production Space Admin
    • Owns the runtime of their solution:
      • Start/stop instances (e.g., overnight).
      • Manage tenant users (Project Admin, Contributor, Reader).
      • See their own usage/cost dashboards.

    3.3 Lifecycle: Accelerator Sandbox → Production¶

    1. PS-A creates a standard EVA DA Accelerator Sandbox.
    2. PS-BA + client:
    3. Load documents
    4. Configure ingestion & RAG parameters
    5. Customize UI text, i18n, a11y via the EVA IDE Tools.
    6. They run PS Test Bench with ~100 gold questions/answers.
    7. Client decides if the solution is acceptable (accuracy, cost, behavior).
    8. PS-A “turns the key” to create a Production EVA Project Space:
    9. Clone configs/workflows.
    10. Reset financial balance and associate with client cost center.
    11. Sandbox can be paused, recycled, or re-used.

    4. EVA Command Centers & EVA Directory¶

    4.1 EVA Command Center (Platform-level)¶

    Audience: AICOE platform owners / leadership.

    Responsibilities:

    • See global health:
    • RAG services
    • Safety/filtering
    • Ingestion pipelines
    • LLM usage
    • See all Project Spaces:
    • Count of sandboxes & productions
    • Status (Green/Yellow/Red/Waiting)
    • Track cost recovery:
    • Sandbox spend vs Production spend.
    • Drill into subordinate Command Centers at the Project Space level.

    4.2 Subordinate Command Centers¶

    Each Project Space has its own Command Center:

    • For Production:
    • Uptime, usage, cost.
    • Active workflow/version.
    • Start/stop controls.
    • For Sandbox:
    • Evaluation metrics.
    • Test Bench results.
    • Spend (funded by IITB / internal).

    4.3 EVA Directory & “Key Keeper” Role¶

    EVA Directory manages:

    • Roles:
    • PS-A, PS-BA, PS-PE (Prompt Engineer), Client Admin, etc.
    • Assignments:
    • Who can manage which Project Space.
    • Who can access which consoles.
    • Traceability:
    • Every change (who/when/what/before/after).

    The Directory Admin (“Key Keeper”):

    • Has power over authorization & scope, not contents of each space.
    • Is the single source of truth for “who can touch what”.

    5. EVA IDE Tools – Siebel-style Visual Builder¶

    The EVA IDE Tools are inspired by Siebel Tools:

    • Visual interface for:
    • Designing RAG workflows (pipelines).
    • Configuring ingestion & connectors.
    • Adjusting prompts, system prompts, and safety.
    • Customizing UI elements (texts, colors, layout, i18n/a11y tokens).

    Core pieces:

    • Palette (components):
    • Ingestion (files, SharePoint, APIs)
    • Transform (OCR, chunking, enrichment)
    • Indexing (semantic, metadata)
    • Retrieval (search, rerank)
    • Answering (LLM chain, safety)
    • Agents/workflows (for MCP/agentic features later)
    • Canvas:
    • Workflow diagram (start → steps → end).
    • Properties panel:
    • Per component parameters and bindings.
    • Test Bench integration:
    • Launch RAG workflow versions into Test Bench jobs.

    MVP goal: let a PS-BA configure RAG pipelines and test them visually, without hand-writing back-end code.

    Aim: Demo EVA 2.0 with:

    • EVA Core services running in Azure.
    • EVA Matrix libraries backing services and UI utilities.
    • EVA IDE Tools showing RAG workflow design.
    • EVA Live Ops showing platform + Space-level monitoring.
    • An end-to-end story:
    • Create sandbox → configure → test → promote → operate.

      • Global dashboard

    7. Agile Operating Model (AGILE-OPERATING-MODE)¶

    (Summarize so AI tools know how to speak the same language.)

    • Product Owner (PO) – Sets vision, backlog priorities. (This is effectively Marco for EVA 2.0.)
    • Scrum Master (SM) – Facilitates sprints, removes impediments.
    • Dev Masters / Tech Leads – Own quality and architecture within repos.
    • Sprints / Cadence:
    • Time-boxed iterations (e.g., 2 weeks).
    • Each sprint aims for demonstrable progress: UI prototypes, service endpoints, tests, etc.

    AI tools should:

    • Respect role boundaries in recommendations (e.g., don’t bypass PO decisions).
    • Frame suggestions in terms of backlog items, epics, and stories.

    8. Definition of Done (DoD) – Quality Bar¶

    When AI suggests code, architecture, or requirements, it should assume:

    • Testing:
    • High test coverage target (e.g., around 80% where reasonable).
    • Meaningful unit + integration tests.
    • Performance:
    • P95 latency targets (e.g., <300ms for internal APIs where feasible, excluding heavy LLM calls).
    • Accessibility (A11y):
    • WCAG 2.1 AA compliance for all UIs.
    • Keyboard navigation, screen reader support, proper color contrast.
    • Bilingual (EN/FR):
    • All user-facing strings must be externalized and translatable.
    • UI ready for EN/FR toggles by design.
    • Security & Privacy:
    • No logging of sensitive content in plain text.
    • Role-based access control enforced at all relevant layers.
    • Docs:
    • README and inline docs up to date with main workflows.
    • ADRs for major architecture decisions.

    AI should generate suggestions that are aligned with this DoD by default.

    9. Repo Registry (registry.yaml)¶

    (High-level description so AI understands scope – details live in registry.yaml.)

    The EVA ecosystem consists of ~17 repos, grouped roughly as:

    • eva-* libraries (Matrix)
    • Product UIs (EVA Chat, EVA DA, Live Ops, IDE)
    • Infra & IaC
    • Supporting tools (command matrix dashboard, test bench, etc.)

    registry.yaml (not reproduced here) lists:

    • Repo name
    • Purpose
    • Owner / lead
    • Dependencies

    AI tools should use this registry to:

    • Suggest where a new feature should live.
    • Identify cross-repo impacts.
    • Avoid duplicating functionality.

    10. Key Decisions & ADRs¶

    Major architecture decisions should be recorded as ADRs (Architecture Decision Records) in the relevant repos.

    Examples of decisions AI should respect:

    • EVA Core is APIM-first – all product calls go through an API gateway.
    • EVA prefers modular libraries over monolithic apps.
    • EVA UIs must be accessible, bilingual, and intranet-friendly (no external SPA hosts).
    • EVA is faceless as a platform:
    • Products & agents are “faces” using the same backbone.

    11. Marco’s Preferences & Working Style¶

    This section is here so AI tools “work with Marco”, not against him.

    • Big-picture first, then details
      I like having the conceptual map before diving into code.
    • Siebel Tools nostalgia
      I am comfortable with model-driven and metadata-driven design (Siebel, workflows, config, etc.).
      Visual builders and configs make sense to me.
    • Accessibility & bilingualism aren’t optional
      They are first-class constraints, not afterthoughts.
    • Human + AI collaboration
      GitHub Copilot Pro+ and ChatGPT are teammates:
    • Copilot: inline code, refactors, local context.
    • ChatGPT: architecture, documents, PRDs, design reviews, and “thinking partner”.
    • Personal context
      I’m balancing a demanding job, long-term life partnership (Chinese partner), and a lot of ideas.
      I value tools that help me:
    • Reduce cognitive load
    • See dependencies clearly
    • Avoid rework

    AI tools should:

    • Minimize noise.
    • Avoid re-asking questions already answered in this master context.
    • Give concrete, actionable outputs (docs, code, templates) that I can paste into repos.

    12. How to Use This Document (For AI Tools)¶

    When you (ChatGPT, Copilot, others) are asked to help with EVA 2.0:

    1. Load this file first to understand the ecosystem.
    2. When asked for help, always anchor responses to:
    3. The 3-layer architecture
    4. Project Spaces & roles
    5. DoD and governance constraints
    6. Prefer:
    7. Suggestions that fit the existing repo structure.
    8. Designs that are modular, testable, and ready for enterprise realities.
    9. Be explicit when:
    10. Proposing new repos, new roles, or breaking changes.
    11. There is a trade-off (performance vs. cost, flexibility vs. complexity).
